Section 1: Introduction to Performance Testing & JMeter Installation

What is Performance Testing?

Definition: Performance testing is a type of non-functional testing performed to ensure that software applications perform well under their expected workload. It validates speed, responsiveness, stability, and scalability of a system.

Key Difference from Functional Testing

AspectFunctional TestingPerformance Testing
FocusWhat the system does (features & behavior)How well the system performs under load
UsersSingle user at a timeThousands of concurrent users
GoalVerify correct functionalityMeasure speed, throughput, stability
OutputPass / FailMetrics (response time, throughput, error rate)
ToolsSelenium, Cypress, PlaywrightJMeter, LoadRunner, NeoLoad

Functional Testing Example — Amazon Purchase Flow

A functional tester verifies that a single user can: search for a product, add it to cart, proceed to checkout, enter payment details, and receive an order confirmation. Each step is validated for correctness.

Performance Testing Example — Thousands of Users Simultaneously

A performance tester simulates 10,000 users simultaneously searching, adding to cart, and checking out on Amazon during a sale event. The goal is to measure whether the site remains responsive, how long pages take to load, and whether any requests fail under load.

Types of Performance Testing

Test TypeDescription
Load TestingTesting with the expected number of users to verify the system handles normal anticipated traffic
Stress TestingTesting beyond normal capacity to find the breaking point of the system
Volume TestingTesting with large amounts of data to verify the system handles data-intensive operations
Endurance TestingTesting under sustained load for an extended duration to detect memory leaks and degradation

Why Performance Testing is Necessary

Real-World Scenario: Independence Day Sale

Imagine an e-commerce website launching a massive Independence Day sale with 50% discounts. Marketing campaigns drive millions of visitors within hours. Without performance testing, the site may crash, transactions may fail, and revenue is lost along with customer trust.

Consequences of Not Doing Performance Testing

  • Website crashes during peak traffic events
  • Slow page load times causing user abandonment
  • Failed transactions and lost revenue
  • Negative brand reputation and social media backlash
  • Database overload and server downtime

Partial vs Complete Failure

Partial Failure: Some features work while others fail — e.g., users can browse products but checkout fails. This is harder to detect and equally damaging.

Complete Failure: The entire website becomes unavailable — users see error pages or timeouts across all functionality.

Key Takeaway: Performance testing is mandatory for any website or application that receives traffic from real users. Skipping it is a business risk, not just a technical oversight.

How Load Testing Tools Help

Record and Playback Concept

Baseline Process (4 Steps)

  1. Record — Use a proxy or browser plugin to capture HTTP requests while browsing
  2. Playback — Replay the captured requests using JMeter thread groups
  3. Enhance — Add assertions, timers, cookie managers, and parameterization
  4. Execute at Scale — Run with multiple virtual users to simulate load

How It Works

When you browse a website, your browser sends HTTP requests to the server. The JMeter proxy sits between your browser and the server, intercepting every request and response. These intercepted requests are converted into JMeter samplers (HTTP Request elements) that can be replayed programmatically.

Important Limitation: Recorded scripts capture requests as-is. Dynamic values (session tokens, CSRF tokens, timestamps, unique IDs) will cause failures during playback unless handled with correlation (Regular Expression Extractor, JSON Extractor, etc.).

HTTP(S) Test Script Recorder Setup

Port Number: 8888

The default port for JMeter's HTTP(S) Test Script Recorder is 8888. This must match the proxy port configured in your browser.

Request Filtering — Add Suggested Excludes

Click "Add suggested excludes" to filter out static resources (CSS, JS, images, fonts) and third-party analytics. This keeps your script clean and focused on business-critical requests.

Step-by-Step Setup

  1. Open Firefox → Settings → General → Network Settings → Settings
  2. Select "Manual proxy configuration"
  3. HTTP Proxy: localhost, Port: 8888
  4. Check "Also use this proxy for HTTPS"
  5. Click OK and start recording in JMeter

Reverting Proxy Settings

After recording, change Firefox proxy back to "No proxy" or "Use system proxy settings" to restore normal browsing.

HTTPS Certificate Import

Why Needed

HTTPS traffic is encrypted. JMeter acts as a man-in-the-middle proxy, decrypting and re-encrypting traffic. The browser must trust JMeter's self-signed CA certificate to allow this interception.

Step-by-Step Import into Firefox

  1. In JMeter, click "Start" on the HTTP(S) Test Script Recorder
  2. Click "CA Certificate" button to locate the certificate file
  3. In Firefox: Settings → Privacy & Security → Certificates → View Certificates → Authorities → Import
  4. Select ApacheJMeterTemporaryRootCA (located in JMeter's bin folder)
  5. Check "Trust this CA to identify websites" and confirm

Certificate Location: <jmeter-install-dir>/bin/ApacheJMeterTemporaryRootCA

Section 3: Thread Groups, Listeners & Performance Metrics

Thread Group

Definition: A Thread Group is the starting point of any JMeter test plan. Each thread represents one virtual user. Threads execute the test plan independently and concurrently.

Parameters

Number of Threads (Users): Total virtual users to simulate.

Ramp-Up Period (seconds): Time over which all threads are started. If 100 threads with 100-second ramp-up, one new thread starts every second.

ThreadsRamp-UpBehavior
1010 sec1 thread starts every second
10050 sec2 threads start every second
50100 sec1 thread every 2 seconds
1000 secAll 100 threads start immediately (spike test)

Loop Count: Number of times each thread executes the test. Set to "Forever" for continuous execution combined with a Duration.

Forever + Duration: When Loop Count is Forever, specify Duration (seconds) to control how long the test runs.

Action on Sampler Error

OptionBehavior
ContinueContinue to next sampler in the loop (default)
Start Next Thread LoopSkip remaining samplers in current iteration; start next loop
Stop ThreadStop current thread; other threads continue
Stop TestStop all threads immediately after current sampler completes
Stop Test NowHard stop — abort all threads immediately

Moving Recorded Script into Thread Group

  1. Create a Thread Group under the Test Plan
  2. Drag the Recording Controller (with captured samplers) into the Thread Group
  3. Disable or remove the HTTP(S) Test Script Recorder (not needed for execution)
  4. Add Listeners to the Thread Group or Test Plan level
  5. Configure thread count, ramp-up, and loop count

Listeners

View Results Tree

  • Green icon = sampler passed (no assertion failures)
  • Red icon = sampler failed (assertion failure or error)
  • Request tab — shows headers, body, and cookies sent
  • Response tab — shows response data, headers, and HTML rendered view
Green 200 Problem: A request can show green with HTTP 200 status but still be a functional failure — e.g., login returns 200 but response body contains the login page again. Always validate response content, not just status codes.

Best Practice: Run with 1 user first using View Results Tree to debug. Disable or remove it before running large load tests — it consumes significant memory.

Section 5: HTTP Cookie Manager

The Problem

You record a login flow, run the test, and all requests show green (success). However, when you inspect the response data, the login page HTML is returned instead of the dashboard. The test appears to pass but functionally fails.

What are Session Cookies?

Facebook Example: When you log into Facebook, the server creates a session and sends a cookie (e.g., session_id=abc123) to your browser. On every subsequent request, your browser automatically sends this cookie. The server reads it and knows who you are — without re-entering credentials.

Cookie Flow

  • Stateless HTTP: HTTP is inherently stateless — each request is independent with no memory of previous requests
  • Unique Identification: Cookies provide a mechanism for the server to identify returning users across multiple requests
  • Expiration: Cookies have expiration times — session cookies expire when the browser closes; persistent cookies have a set expiry date

The Solution: HTTP Cookie Manager

How to Add

  1. Right-click Thread Group → Add → Config Element → HTTP Cookie Manager
  2. Place it at the Thread Group level (applies to all samplers in the group)
  3. No additional configuration needed for basic usage

How It Works Step by Step

  1. User sends login request → Server responds with Set-Cookie: session_id=abc123
  2. HTTP Cookie Manager stores the cookie automatically
  3. Next request → Cookie Manager attaches Cookie: session_id=abc123 to the request header
  4. Server recognizes the session → Returns dashboard instead of login page

Section 6: Assertions

Why Assertions?

HTTP status codes alone are insufficient. A server can return 200 OK with wrong content (login page instead of dashboard). Assertions validate response content, size, and timing to determine true pass/fail beyond status codes.

Size Assertion

Why Size Matters

An error page is often much smaller than a full dashboard page. If you expect a dashboard (~50KB) but receive 2KB, the response is likely an error page even with HTTP 200.

Section 8: Timers

Why Timers?

Real-World Problem

Without timers, JMeter sends requests as fast as possible — far faster than any real user. A human takes 3–10 seconds to read a page before clicking the next link. Without think time, your test creates unrealistic load patterns and inaccurate performance results.

DB Update Latency

Real users also wait for operations to complete. After submitting a form that triggers a database update, users typically wait before proceeding. Timers simulate this natural pause.

Timer Behavior Observation

ScenarioBehaviorResult
Without TimerRequests fire immediately one after anotherUnrealistic load; inflated throughput; inaccurate response time patterns
With TimerPause inserted before each sampler (or after, depending on placement)Realistic user think time; accurate load simulation
Important: Timers are scoped — a timer applies to all samplers at the same level or below it in the test plan tree. Timer at Thread Group level pauses before every sampler in the group.

Constant Timer

Fixed Delay: Pauses for the exact same duration before every sampler execution.

Configuration

Thread Delay (milliseconds): e.g., 3000 = 3-second pause before each request.

Demo

Constant Timer set to 2000ms → every request waits exactly 2 seconds before executing. Predictable and simple.

Gaussian Random Timer

Provides variable delay centered around a mean value with Gaussian (normal) distribution — most delays cluster near the mean, with occasional shorter/longer pauses.

ThreadDeviation (ms)Constant Delay (ms)Calculated Delay (ms)
Thread 15002000~2100 (2000 + random Gaussian × 500)
Thread 25002000~1850
Thread 35002000~2300
Thread 45002000~1950

Formula: Delay = Constant Delay + (Random Gaussian × Deviation). Most values fall within ±1 deviation of the mean.

Poisson Random Timer

Similar to Gaussian but uses Poisson distribution. Lambda (mean delay) determines the average pause. Better for modeling events that occur independently over time (e.g., time between customer arrivals).

Constant Throughput Timer

Petrol Bunk Analogy

Imagine a petrol station that can serve 60 cars per hour (1 per minute). Three scenarios:

  1. Scenario 1 — 30 cars arrive in 30 minutes: Station serves smoothly, no queue. Timer ensures even spacing.
  2. Scenario 2 — 60 cars arrive in 30 minutes: Station at capacity, slight delays. Timer throttles to maintain target rate.
  3. Scenario 3 — 120 cars arrive in 30 minutes: Queue builds up, system overloaded. Timer cannot compensate — server is the bottleneck.

Key Throughput Insight

ConceptExplanation
Target ThroughputDesired requests per minute (set in timer config)
Timer RoleDelays threads to spread requests evenly across time
Server BottleneckIf server can't handle target rate, timer can't fix it — response times increase
Calculation ModeAll active threads vs. current thread only

Configuration

  • Target throughput (samples per minute): e.g., 120 = 2 requests per second
  • Calculate Throughput based on: All active threads in current thread group

Petrol Station Analogy for Timer Behavior

The Constant Throughput Timer is the traffic controller at the petrol station entrance — it spaces out cars (requests) to maintain the target service rate. If cars arrive faster than the target, the controller makes them wait. If the station (server) is slow, the queue grows regardless.

Difference from Constant Timer

AspectConstant TimerConstant Throughput Timer
ControlsFixed delay per requestTarget request rate (throughput)

GoalSimulate user think timeMaintain specific load rate regardless of thread count
Thread ScalingMore threads = more total requests (no rate control)More threads = timer delays more to maintain same rate

When to Use / When NOT to Use

Key Takeaway: Use Constant/Gaussian timers to simulate realistic user behavior. Use Constant Throughput Timer to control load rate. Never use Constant Throughput Timer when your goal is to find the server's maximum capacity — it artificially limits the load.

The Problem: Hard-Coded Values

  • When recording, JMeter captures exact values (e.g., username = "tomsmith")

Section 14: Distributed Mode (Master-Slave Architecture)

Why Distributed Mode is Needed

The Problem: Single Machine Limitations

  • Running 5000+ users on one machine → memory fills up, heap overflow, inaccurate results
  • Limited CPU, RAM, heap configuration cannot handle thousands of threads
  • No point running 5000 users if results are inaccurate

The Real-World Need

  • E-commerce sites (Amazon, Flipkart) get millions of users on sale days
  • Must test with 5000, 10000, 15000+ users
  • Single machine CANNOT handle this realistically

What is Distributed Mode?

  • Definition: Distributing JMeter test across multiple machines
  • One Master controls the test
  • Multiple Slaves execute portions of the load
  • All slaves hit the target simultaneously
  • Results collected back at Master

Architecture

                    ┌─── Slave 1 (1000 users) ───┐
                    │                              │
Master (JMeter) ───├─── Slave 2 (1000 users) ───├───→ Target Application
  (5000 users      │                              │      (e.g., Amazon)
   configured)     ├─── Slave 3 (1000 users) ───┤
                    │                              │
                    ├─── Slave 4 (1000 users) ───┤
                    │                              │
                    └─── Slave 5 (1000 users) ───┘

Setup Requirements

  • All machines must have same JMeter version
  • All on same network/subnet
  • Same or similar operating systems
  • No firewall/antivirus blocking connections

Step 1: Configure Slave Machine

  1. Navigate to JMeter's bin directory
  2. Open jmeter.properties
  3. Find server.rmi.ssl.disable → set to true (uncomment)
  4. Save the file
  5. Run jmeter-server.bat
  6. Wait for: "Created remote object endpoint"
  7. Note the Slave's IP address

Step 2: Configure Master Machine

  1. Open jmeter.properties in Master's bin directory
  2. Set server.rmi.ssl.disable to true
  3. Find remote_hosts → replace 127.0.0.1 with Slave's IP
  4. Multiple slaves: remote_hosts=192.168.1.10,192.168.1.11,192.168.1.12
  5. Save and restart JMeter

Step 3: Run Test on Slave(s)

  1. Open JMeter on Master → Load test plan
  2. Go to Run → Start Remote → Select Slave IP
  3. Test executes on Slave (not Master)
  4. Results appear back in Master's JMeter

Workload Modeling

  • Transaction Mix: Distribution of different user actions (e.g., 40% browse, 30% search, 20% add-to-cart, 10% checkout)
  • Think Time: Pause between user actions simulating real human behavior (typically 3-10 seconds)
  • Pacing: Time between iteration start times to control throughput
  • Concurrency vs Total Users: Concurrent = active at same time; Total = all users over time

Little's Law — Calculating Concurrent Users

Concurrent Users = Throughput × Average Response Time

Example:
- Throughput = 10 transactions/second
- Average Response Time = 5 seconds
- Concurrent Users = 10 × 5 = 50 users active at any moment

Interview Questions — 4-Year Experience Level

Conceptual Questions

  1. Explain the Performance Testing Lifecycle.
    Answer: Requirements → Planning → Script Dev → Env Setup → Execution → Analysis → Bottleneck ID → Tuning & Retest
  2. What is the difference between Load, Stress, Spike, and Endurance testing?
    Load = expected users; Stress = beyond capacity; Spike = sudden surge; Endurance = sustained load over hours
  3. How do you determine the number of virtual users for a test?
    Use Little's Law, production access logs, Google Analytics, or business requirements. Concurrent Users = TPS × Avg Response Time
  4. What is correlation and why is it needed?
    Extracting dynamic server-generated values (session IDs, tokens) from responses and passing them to subsequent requests. Without it, scripts fail on replay.
  5. What is Think Time and why is it important?
    Simulated pause between user actions. Without it, virtual users fire requests unrealistically fast, generating higher load than real users would.
  6. Explain Apdex Score.
    Application Performance Index — measures user satisfaction. Apdex = (Satisfied + Tolerating/2) / Total. T = threshold (e.g., 3 sec). Score: 0 to 1.
  7. What are the exit criteria for a performance test?
    All SLAs met, error rate within threshold, no memory leaks, resource utilization within limits, sign-off from stakeholders.

JMeter-Specific Questions

  1. BeanShell vs Groovy (JSR223)?
    Groovy is compiled → significantly faster. BeanShell is interpreted → slower. Always prefer JSR223/Groovy in production tests.
  2. How to handle dynamic values?
    Regular Expression Extractor, JSON Extractor, CSS/JQuery Extractor, XPath Extractor, Boundary Extractor
  3. How to run JMeter in CI/CD?
    Non-GUI mode: jmeter -n -t test.jmx -l results.jtl -e -o report/. Integrate via Jenkins JMeter plugin or shell step.
  4. How to generate HTML reports?
    jmeter -g results.jtl -o report_folder/ or use -e -o flags during test execution.
  5. What is JMeter Backend Listener?
    Sends real-time test results to InfluxDB/Graphite for live Grafana dashboards during test execution.
  6. How to handle file uploads in JMeter?
    HTTP Request → Files Upload tab → specify file path, parameter name, and MIME type.
  7. What is the difference between Throughput and Hits per Second?
    Throughput = successful transactions per second. Hits/sec = all HTTP requests (including embedded resources). Throughput is the meaningful business metric.

Real-World Scenario Questions

  1. Your test shows 0% errors but client says app is slow. Why?
    Response times may be high (meeting SLA but barely). Check 90th/95th percentile, not just average. Check server-side resource utilization.
  2. How do you handle a test where scripts pass locally but fail in CI?
    Check environment differences, data availability, network connectivity, file paths, JMeter version compatibility.
  3. Client says "test with 10,000 users." Your laptop can't handle it. What do you do?
    Use distributed testing (master-slave), cloud platforms (BlazeMeter, AWS), or optimize scripts (remove listeners, use CSV output).
  4. Memory keeps growing during an 8-hour endurance test. What's happening?
    Likely a memory leak in the application. Capture heap dumps, analyze with MAT (Memory Analyzer Tool), identify leaked objects.

  1. Always run a baseline test (1 user) before load testing

  7. Use CSV output format instead of XML for better performance
  8. Monitor server resources (CPU, memory, DB) during test execution
  9. Use JSR223/Groovy instead of BeanShell for better performance
  10. Always compare with baseline to measure improvement
  11. Generate HTML Dashboard reports for stakeholders
  12. Store scripts in version control (Git) and integrate with CI/CD
  13. Test on an environment that mirrors production as closely as possible
  14. Document your workload model with transaction mix and think times
