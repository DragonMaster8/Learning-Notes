Section 1: Playwright Automation

Playwright is the browser automation framework most QA teams are adopting in 2025–2026. If you have four years of Selenium experience, this section maps what you already know to Playwright concepts, patterns, and interview answers.

What Is Playwright?

Playwright is an open-source browser automation library developed and maintained by Microsoft. It provides a unified API to drive Chromium, Firefox, and WebKit (Safari engine) from a single test codebase. Unlike Selenium's WebDriver protocol, Playwright communicates directly with browsers via its own protocol, enabling faster, more reliable automation.

Key takeaway for Selenium veterans: Playwright is not just "Selenium with a different syntax." It is a modern automation stack with auto-waiting, built-in network control, tracing, and first-class API testing — all in one package.

Why Playwright Over Selenium?

Selenium remains widely used in legacy codebases, but Playwright addresses many pain points Selenium users know well: flaky waits, slow startup, limited debugging, and fragmented tooling.

Browser Contexts: Isolated Sessions

A BrowserContext is an isolated browser session — equivalent to an incognito profile. Cookies, localStorage, and cache do not leak between contexts. In parallel test runs, Playwright typically creates one context per test for full isolation. This replaces the Selenium pattern of clearing cookies or spinning up separate driver instances.

Auto-Wait: Intelligent Waiting Built In

Every Playwright action (click, fill, check) automatically waits until the element is:

  • Attached to the DOM
  • Visible (not hidden by CSS)
  • Stable (not animating)
  • Enabled (not disabled)
  • Receives events (not obscured by another element)
Selenium → Playwright mindset shift: Delete most of your WebDriverWait, Thread.sleep, and ExpectedConditions code. If a Playwright action times out, fix the locator or app state — don't add explicit waits first.

Locators: Auto-Waiting and Strict Mode

Playwright locators are the heart of stable tests. They re-query the DOM on every action and auto-wait. Strict mode (default) fails if a locator resolves to more than one element — preventing silent "click the wrong button" bugs.

Locator Strategies (Preferred Order)

// 1. Accessibility-first — mirrors how users and screen readers find elements
await page.getByRole('button', { name: 'Submit' }).click();
await page.getByRole('link', { name: 'Sign up' }).click();
await page.getByRole('textbox', { name: 'Search' }).fill('playwright');

// 2. Text-based — visible text content
await page.getByText('Welcome back').click();
await page.getByText(/order confirmed/i).waitFor();

// 3. Form labels — ideal for inputs tied to <label>
await page.getByLabel('Email').fill('user@example.com');
await page.getByLabel('Remember me').check();

// 4. Placeholder text
await page.getByPlaceholder('Enter email').fill('user@example.com');

// 5. Test ID — stable when devs add data-testid attributes
await page.getByTestId('login-btn').click();

// 6. CSS selector — use when semantic locators aren't feasible
await page.locator('#email').fill('user@example.com');
await page.locator('.nav-item.active').click();

// 7. XPath — legacy support; prefer getByRole/getByLabel when possible
await page.locator('//button[@type="submit"]').click();
await page.locator('xpath=//div[@class="modal"]//button[text()="Close"]').click();

Chaining and Filtering

// Narrow scope within a container
await page.locator('.product-card').filter({ hasText: 'iPhone' }).getByRole('button', { name: 'Add to cart' }).click();

// nth element (0-based) — use sparingly; prefer unique locators
await page.getByRole('listitem').nth(2).click();

Page Object Model in Playwright

POM remains the best pattern for maintainable suites. Playwright POMs are plain TypeScript classes receiving a Page instance — no base driver abstraction required.

LoginPage

import { Page, Locator } from '@playwright/test';

export class LoginPage {
    readonly page: Page;
    readonly emailInput: Locator;
    readonly passwordInput: Locator;
    readonly signInButton: Locator;
    readonly errorMessage: Locator;

    constructor(page: Page) {
        this.page = page;
        this.emailInput = page.getByLabel('Email');
        this.passwordInput = page.getByLabel('Password');
        this.signInButton = page.getByRole('button', { name: 'Sign In' });
        this.errorMessage = page.getByTestId('login-error');
    }

    async goto() {
        await this.page.goto('/login');
    }

    async login(user: string, pass: string) {
        await this.emailInput.fill(user);
        await this.passwordInput.fill(pass);
        await this.signInButton.click();
    }

    async loginExpectingError(user: string, pass: string) {
        await this.login(user, pass);
        await this.errorMessage.waitFor({ state: 'visible' });
    }
}

HomePage

import { Page, Locator } from '@playwright/test';

export class HomePage {
    readonly page: Page;
    readonly welcomeHeading: Locator;
    readonly logoutButton: Locator;
    readonly navDashboard: Locator;

    constructor(page: Page) {
        this.page = page;
        this.welcomeHeading = page.getByRole('heading', { name: /welcome/i });
        this.logoutButton = page.getByRole('button', { name: 'Logout' });
        this.navDashboard = page.getByRole('link', { name: 'Dashboard' });
    }

    async expectLoggedIn(username: string) {
        await this.welcomeHeading.waitFor({ state: 'visible' });
        await this.page.getByText(`Hello, ${username}`).waitFor();
    }

    async logout() {
        await this.logoutButton.click();
    }

    async openDashboard() {
        await this.navDashboard.click();
    }
}

BasePage

import { Page, Locator } from '@playwright/test';

export class BasePage {
    readonly page: Page;
    readonly toast: Locator;
    readonly spinner: Locator;

    constructor(page: Page) {
        this.page = page;
        this.toast = page.getByRole('alert');
        this.spinner = page.getByTestId('loading-spinner');
    }

    async waitForPageLoad() {
        await this.page.waitForLoadState('networkidle');
    }

    async waitForSpinnerToDisappear() {
        await this.spinner.waitFor({ state: 'hidden' });
    }

    async getToastMessage(): Promise<string> {
        return (await this.toast.textContent()) ?? '';
    }

    async takeScreenshot(name: string) {
        await this.page.screenshot({ path: `screenshots/${name}.png`, fullPage: true });
    }
}

Using POM in a Test

import { test, expect } from '@playwright/test';
import { LoginPage } from '../pages/LoginPage';
import { HomePage } from '../pages/HomePage';

test.describe('Authentication', () => {
    test('successful login redirects to home', async ({ page }) => {
        const loginPage = new LoginPage(page);
        const homePage = new HomePage(page);

        await loginPage.goto();
        await loginPage.login('qa.user@example.com', 'SecurePass123!');
        await homePage.expectLoggedIn('QA User');
        await expect(page).toHaveURL(/.*dashboard/);
    });
});

Assertions

Playwright uses expect from @playwright/test with web-first assertions that auto-retry until timeout — unlike Selenium's immediate checks.

import { test, expect } from '@playwright/test';

test('assertion examples', async ({ page }) => {
    await page.goto('https://example.com');

    // Page-level
    await expect(page).toHaveTitle(/Example Domain/);
    await expect(page).toHaveURL('https://example.com/');

    const heading = page.getByRole('heading', { level: 1 });

    // Locator state
    await expect(heading).toBeVisible();
    await expect(heading).toHaveText('Example Domain');
    await expect(heading).toContainText('Example');

    // Input values
    const search = page.getByPlaceholder('Search');
    await search.fill('playwright');
    await expect(search).toHaveValue('playwright');

    // Count and attributes
    await expect(page.getByRole('link')).toHaveCount(1);
    await expect(page.getByRole('link')).toHaveAttribute('href', 'https://iana.org/domains/example');

    // Soft assertions — collect all failures before failing test
    await expect.soft(heading).toBeVisible();
    await expect.soft(page.getByRole('link')).toBeEnabled();
});

API Testing in Playwright

Playwright includes a built-in request fixture for API testing without launching a browser — perfect for contract tests, setup/teardown, and hybrid UI+API suites.

import { test, expect } from '@playwright/test';

const BASE_URL = 'https://jsonplaceholder.typicode.com';

test.describe('Users API', () => {
    test('GET user by id', async ({ request }) => {
        const response = await request.get(`${BASE_URL}/users/1`);
        expect(response.ok()).toBeTruthy();
        expect(response.status()).toBe(200);

        const body = await response.json();
        expect(body.id).toBe(1);
        expect(body.email).toContain('@');
        expect(body.address).toHaveProperty('city');
    });

    test('POST create new user', async ({ request }) => {
        const response = await request.post(`${BASE_URL}/users`, {
            data: {
                name: 'QA Engineer',
                username: 'qa_eng',
                email: 'qa@example.com',
            },
            headers: {
                'Content-Type': 'application/json',
            },
        });

        expect(response.status()).toBe(201);
        const body = await response.json();
        expect(body.name).toBe('QA Engineer');
        expect(body.id).toBeDefined();
    });

    test('PUT update user', async ({ request }) => {
        const response = await request.put(`${BASE_URL}/users/1`, {
            data: { name: 'Updated Name' },
        });
        expect(response.ok()).toBeTruthy();
    });

    test('DELETE user', async ({ request }) => {
        const response = await request.delete(`${BASE_URL}/users/1`);
        expect(response.ok()).toBeTruthy();
    });
});

Parallel Execution

Playwright Test runs files in parallel by default using worker processes. Tests within a file run sequentially in the same worker unless configured otherwise.

# Run with 4 parallel workers
npx playwright test --workers=4

# Run specific project (browser)
npx playwright test --project=chromium

# Shard across CI machines (e.g., 4 GitHub Action jobs)
npx playwright test --shard=1/4
npx playwright test --shard=2/4

# Serial mode when tests share state (avoid if possible)
test.describe.configure({ mode: 'serial' });

Interview Q&A (20+ Questions)

Fundamentals

Q1: What is Playwright and who maintains it?
A: Playwright is an open-source browser automation framework by Microsoft. It supports Chromium, Firefox, and WebKit with a unified API available in TypeScript, JavaScript, Python, Java, and C#.

Q2: How is Playwright different from Selenium WebDriver?
A: Playwright uses a direct browser protocol (not WebDriver), includes auto-waiting on every action, native network interception, tracing, and bundles browser binaries. Selenium requires separate drivers and explicit waits for stable tests.

Q3: Explain Browser → BrowserContext → Page hierarchy.
A: Browser is the launched instance. BrowserContext is an isolated session (cookies/storage separate). Page represents a tab within a context. One browser can have multiple contexts; each context can have multiple pages.

Q4: What is auto-waiting in Playwright?
A: Before performing an action, Playwright automatically waits for the element to be attached, visible, stable, enabled, and not obscured. Assertions also retry until timeout. This eliminates most explicit waits.

Q5: What is strict mode for locators?
A: If a locator matches zero or more than one element, Playwright throws an error immediately. This prevents accidentally interacting with the wrong element — a common Selenium flakiness source.

Locators and Best Practices

Q6: What locator strategy does Playwright recommend?
A: User-facing attributes in order: getByRole, getByLabel, getByPlaceholder, getByText, getByTestId, then CSS/XPath as last resort. Role and label locators mirror accessibility and resist DOM structure changes.

Q7: How do you handle dynamic elements that change ID every render?
A: Avoid ID-based selectors. Use stable attributes like data-testid, visible text, ARIA roles, or label associations. Example: page.getByRole('button', { name: 'Save' }).

Q8: What is the difference between page.locator() and page.$()?
A: locator() returns a lazy Locator that re-queries the DOM and auto-waits — this is the recommended API. $() returns an ElementHandle immediately (legacy pattern); prefer locators.

Q9: How do you work with Shadow DOM?
A: Playwright pierces open shadow roots automatically with locators. For closed shadow roots, use page.evaluate() or request developers expose test hooks via data-testid.

Test Design and Patterns

Q10: How do you implement Page Object Model in Playwright?
A: Create classes accepting Page in the constructor. Define locators as readonly properties. Expose async methods for user actions. Import and instantiate in test files — no special base class required, though BasePage helps for shared utilities.

Q11: What are web-first assertions?
A: Assertions like toBeVisible() and toHaveText() retry until the condition is met or timeout. Unlike Selenium's immediate assertEquals, they handle async UI updates without manual polling.

Q12: What is the difference between hard and soft assertions?
A: Hard assertions (expect()) fail the test immediately. Soft assertions (expect.soft()) collect failures and report all at the end — useful for validating multiple fields on a form.

Q13: How do you share authentication state across tests?
A: Use storageState: log in once in a setup project, save cookies/localStorage to a JSON file via context.storageState({ path: 'auth.json' }), then reference it in config use: { storageState: 'auth.json' }.

Advanced Topics

Q14: How do you mock API responses in Playwright?
A: Use page.route('**/api/endpoint', route => route.fulfill({...})) to return custom status, headers, and body. Use route.continue() to modify real responses or route.abort() to simulate failures.

Q15: Can Playwright test REST APIs without a browser?
A: Yes. Use the request fixture: await request.get('/api/users'). Configure baseURL and headers in config. Ideal for setup/teardown and pure API test suites.

Q16: How does Playwright handle file downloads?
A: Wrap the click in a download promise: const [download] = await Promise.all([page.waitForEvent('download'), page.getByText('Export').click()]), then await download.saveAs('path/file.csv').

Q17: Explain Playwright tracing.
A: Tracing records actions, DOM snapshots, network, console, and sources into a zip file viewable in Trace Viewer. Enable via config (trace: 'on-first-retry') or manually with context.tracing.start/stop. Essential for debugging CI failures.

Q18: How do you run tests in parallel?
A: Playwright runs test files in parallel by default using workers. Control via --workers=N, sharding with --shard=i/n, and fullyParallel: true for test-level parallelism within files.

CI/CD and Troubleshooting

Q19: Why do tests pass locally but fail in CI?
A: Common causes: different viewport/timezone, missing --with-deps on Linux, race conditions exposed by parallel workers, environment-specific data, and missing baseURL. Use traces and CI artifacts to diagnose.

Q20: What is Playwright UI Mode?
A: An interactive test runner (npx playwright test --ui) showing tests, time-travel debugging, locator picker, and watch mode — accelerates test authoring and failure investigation.

Q21: How do you handle tests that depend on execution order?
A: Avoid ordered dependencies. Use test.describe.configure({ mode: 'serial' }) only when necessary. Prefer isolated tests with API setup over shared mutable state.

Q22: What is the role of playwright.config.ts projects?
A: Projects define browser/device matrix entries. Each project can override config (browser, viewport, storageState). Run all or subset via --project=chromium.

Q23: Playwright vs Cypress — when would you pick Playwright in an interview?
A: Multi-browser WebKit support, true multi-tab/multi-origin without workarounds, built-in parallel sharding without paid services, polyglot support, and stronger fit for full-stack QA owning API + UI + cross-browser regression.

Q24: How do you test responsive/mobile layouts?
A: Use device descriptors in config (devices['iPhone 14']), set viewport manually, or use isMobile and touch emulation. For real devices, integrate with BrowserStack or similar via connect options.

Q25: What are custom fixtures in Playwright?
A: Extend the test object to inject shared setup — authenticated pages, database connections, or POM instances. Define with test.extend() and use instead of base test import for DRY setup across specs.

Interview pro tip: When asked "How would you reduce flakiness?", structure your answer around Playwright strengths: user-facing locators, auto-waiting, isolated browser contexts, API seed data instead of UI setup, trace-on-retry in CI, and avoiding waitForTimeout() except as a last resort.

2.1 REST Assured Architecture: Given-When-Then BDD Style

REST Assured implements a fluent DSL inspired by BDD. Every API call decomposes into three phases that map directly to readable test intent.

  +-----------+       +-----------+       +-----------+
  |  given()  |  -->  |  when()   |  -->  |  then()   |
  |  SETUP    |       |  EXECUTE  |       |  VALIDATE |
  +-----------+       +-----------+       +-----------+
  headers, body       GET/POST/PUT        status, body,
  auth, params        DELETE/PATCH        headers, time
BlockResponsibilityCommon Methods
given()Request setup.header(), .headers(Map), .body(), .auth(), .queryParam(), .pathParam(), .log().all()
when()HTTP execution.get(), .post(), .put(), .delete(), .patch()
then()Response validation.statusCode(), .body(), .header(), .time(), .extract()

Minimal Example

import static io.restassured.RestAssured.*;
import static org.hamcrest.Matchers.*;

given()
    .header("Content-Type", "application/json")
    .queryParam("page", 1)
.when()
    .get("/users")
.then()
    .statusCode(200)
    .body("data.size()", greaterThan(0));

public class OAuthHelper {

    private static String cachedToken;
    private static long tokenExpiry;

    public static String getAccessToken() {
        if (cachedToken != null && System.currentTimeMillis() < tokenExpiry) {
            return cachedToken;
        }

        Response tokenResponse = given()
            .contentType(ContentType.URLENC)
            .formParam("grant_type", "client_credentials")
            .formParam("client_id", ConfigReader.get("oauth.client.id"))
            .formParam("client_secret", ConfigReader.get("oauth.client.secret"))
            .formParam("scope", "read write")
        .when()
            .post(ConfigReader.get("oauth.token.url"))
        .then()
            .statusCode(200)
            .body("access_token", notNullValue())
            .extract().response();

        cachedToken = tokenResponse.path("access_token");
        int expiresIn = tokenResponse.path("expires_in");
        tokenExpiry = System.currentTimeMillis() + (expiresIn - 60) * 1000L;
        return cachedToken;
    }
}

// Usage in test

    .get("/orders")
.then()

public void fullUserCrudLifecycle() {

        .pathParam("id", userId)
    .when()

    .then()

    given()

Section 3: Microservices Testing

Microservices architecture is the default at most product companies today. If you have four years of QA experience testing monolith applications, this section bridges what you already know to how testing works when one application becomes dozens of independent services — and gives you structured interview answers.

What Are Microservices?

Microservices is an architectural style where an application is built as a collection of small, independent, deployable services. Each service owns a specific business capability (e.g., User Service, Order Service, Payment Service), runs in its own process, and communicates with other services over the network — typically via HTTP/REST, gRPC, or message queues.

Monolith mental model shift: In a monolith, you test one codebase, one deployment, one database. In microservices, you test many moving parts that can fail independently, deploy on different schedules, and use different technologies — your testing strategy must account for integration, contracts, and async behavior at every layer.

Service Communication Patterns

  • REST (HTTP/JSON): Most common. Synchronous request-response. Easy to test with Postman, RestAssured, or curl. QA validates status codes, headers, JSON schema, and error responses.
  • gRPC (Protocol Buffers): Binary, high-performance RPC. Common at Google-scale companies. QA uses tools like BloomRPC, grpcurl, or gRPC client libraries in test code.
  • Message Queue (Kafka / RabbitMQ): Asynchronous, event-driven. Producer publishes events; consumers process independently. QA must verify publish, consume, and downstream side effects — not just HTTP responses.

Types of Microservices Testing

1. Unit Testing

Tests individual service logic in isolation — business rules, validation, calculations. Dependencies (other services, DB) are mocked with Mockito or similar. Fast feedback loop; developers write these; QA understands coverage gaps for test planning.

2. Integration Testing

Tests a single service with its real database (often via Testcontainers or an embedded DB). External service calls are mocked or stubbed. Validates that repository layer, ORM mappings, and service logic work together correctly.

3. Contract Testing (Pact)

Consumer-driven contract testing verifies that the API contract between services is honored — without needing both services running together.

  • What: A formal agreement defining expected request (method, path, headers, body) and response (status, body schema) between a consumer service and a provider service.
  • Flow: Consumer team writes a contract describing what they expect from the provider → Pact generates a pact file → Provider team runs provider verification against that pact in their CI pipeline.
  • Why QA cares: Contract tests catch breaking API changes before integration environments — reduces "works on my machine" failures when services deploy independently.

4. Component Testing

Tests a single service in isolation with all external dependencies mocked (WireMock, Mountebank, or Pact mock server). The service runs in a real container or embedded server; only its boundary is tested. Faster and more stable than full E2E.

5. End-to-End Testing

Tests the full user flow across multiple services — e.g., register → browse → add to cart → checkout → payment → confirmation email. QA owns most E2E tests. Keep the suite small and focused on critical business paths; run in staging with all services deployed.

6. Performance Testing

Load test individual services and critical paths through the API Gateway. Identify bottlenecks: which service degrades first under load? Use JMeter, k6, or Gatling. QA collaborates with DevOps on SLAs (p95 latency, throughput, error rate).

How QA Verifies Async Flows

  1. Message published? Use Kafka CLI (kafka-console-consumer) or a test consumer to verify the message appears on the topic after triggering the action.
  2. Message consumed? Check consumer group lag — lag should decrease after processing. Verify consumer service logs show the event handled.
  3. Downstream processed? Assert the side effect — e.g., after "order.created" event, verify notification email record exists in Notification Service DB.
  4. Testing strategy: Trigger action via API → poll/wait for async result with timeout (Awaitility in Java: await().atMost(30, SECONDS).until(() -> emailSent)).

Service Mesh & Observability

In production, a service mesh (Istio, Linkerd) manages service-to-service communication — mTLS, retries, circuit breaking. For QA, the critical observability tools are:

  • Distributed Tracing (Jaeger / Zipkin): Each request gets a trace ID propagated across all services. You see the full path: Gateway → User Service (45ms) → Order Service (120ms) → Payment Service (FAIL 503).
  • Centralized Logging (ELK / Loki): Search logs across all services by trace ID or correlation ID.
  • Metrics (Prometheus / Grafana): Error rates, latency percentiles per service.
QA usage of tracing: When an E2E test fails with a 500 error, grab the trace ID from the response header (X-Trace-ID). Open Jaeger, paste the trace ID, and see exactly which service failed and how long each hop took. This replaces hours of "which service broke?" guesswork.

Database per Service Pattern

Each microservice owns its database. Order Service cannot JOIN User Service's tables directly — it calls User Service's API or consumes user events to maintain a local copy.

Testing Implications

  • Data consistency: After an action, verify data in multiple service databases — e.g., after checkout, Order DB has order record AND Inventory DB shows reduced stock AND Payment DB has transaction.
  • Eventual consistency: Updates are not immediate across services. User updates email → Order Service may show old email for seconds until it consumes the "user.updated" event. QA must use wait/poll strategies, not instant assertions.
  • Test data setup: Seed data per service independently. Use API calls or direct DB scripts per service schema.

"How do you test microservices?" — Structured Answer Template

  2. Apply the testing pyramid: Unit (dev-owned) → Service/Integration → Contract (Pact) → E2E (critical paths only).

"What is contract testing?" — Interview Answer

Contract testing verifies that a service provider's API matches what the consumer expects, without running both services together. The consumer team writes a contract (expected request/response). The provider team verifies against it in CI using Pact. This catches breaking API changes early when services deploy independently. Example: Order Service expects GET /users/{id} to return { id, name } — if User Service changes the response shape, provider verification fails before merge.

"How do you handle asynchronous testing?" — Interview Answer

I trigger the action via API, then poll for the expected side effect with a bounded timeout using Awaitility or custom retry logic. I verify the full chain: message published to Kafka → consumed by downstream service → side effect visible (DB record, email sent, status updated). I never use Thread.sleep with fixed duration — always poll with timeout. I also check consumer lag and service logs if polling times out.

Arrays & Strings (Most Asked)

Two Sum — HashMap Approach

Problem: Given an array of integers and a target, return indices of two numbers that add up to target.

Approach: Iterate array. For each element, check if (target - current) exists in HashMap. If yes, return both indices. Otherwise, store current value and index in map.

public int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> map = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int complement = target - nums[i];
        if (map.containsKey(complement)) {
            return new int[] { map.get(complement), i };
        }
        map.put(nums[i], i);
    }
    return new int[] {};
}
// Time: O(n) | Space: O(n)

Sliding Window — Maximum Subarray Sum (Kadane's Algorithm)

public int maxSubArray(int[] nums) {
    int maxSoFar = nums[0];
    int currentMax = nums[0];
    for (int i = 1; i < nums.length; i++) {
        currentMax = Math.max(nums[i], currentMax + nums[i]);
        maxSoFar = Math.max(maxSoFar, currentMax);
    }
    return maxSoFar;
}
// Time: O(n) | Space: O(1)

Sliding Window — Longest Substring Without Repeating Characters

public int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> map = new HashMap<>();
    int maxLen = 0, left = 0;
    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        if (map.containsKey(c) && map.get(c) >= left) {
            left = map.get(c) + 1;
        }
        map.put(c, right);
        maxLen = Math.max(maxLen, right - left + 1);
    }
    return maxLen;
}
// Time: O(n) | Space: O(min(n, charset))

Two Pointer — Container With Most Water

public int maxArea(int[] height) {
    int left = 0, right = height.length - 1, maxArea = 0;
    while (left < right) {
        int area = Math.min(height[left], height[right]) * (right - left);
        maxArea = Math.max(maxArea, area);
        if (height[left] < height[right]) left++;
        else right--;
    }
    return maxArea;
}
// Time: O(n) | Space: O(1)

How HashMap Works Internally (Interview Favorite!)

When you call map.put(key, value):

  1. Hash function computes hashCode of key → determines bucket index: index = hashCode % array.length
  2. Bucket: Array of nodes. Each bucket holds a linked list (or red-black tree when >8 nodes since Java 8)
  3. Collision: Two keys with same bucket index → stored in linked list at that bucket (equals() distinguishes them)
  4. Load factor (default 0.75): When 75% of buckets are filled, HashMap doubles capacity and rehashes all entries
  5. get(key): Compute hash → go to bucket → traverse list/tree → compare with equals()

  HashMap Internal Structure (Java 8+)

  hashCode("Alice") % 16 = 3
  hashCode("Bob")   % 16 = 7
  hashCode("Carol") % 16 = 3  ← collision with Alice

  Index:  0    1    2    3         4    5    6    7
        [ ]  [ ]  [ ]  [Node]     [ ]  [ ]  [ ]  [Node]
                         │                              │
                         ▼                              ▼
                    ("Alice",30)                  ("Bob",25)
                         │
                         ▼ (linked list on collision)
                    ("Carol",28)

  When bucket has >8 nodes → converts to Red-Black Tree (O(log n) lookup)
  Load factor 0.75 reached → resize array 2x and rehash all entries
Interview answer: "HashMap provides O(1) average time for get/put. Internally it uses an array of buckets. Hash function maps key to bucket index. Collisions are handled by linked lists (or trees for large buckets). Default load factor is 0.75 — when exceeded, capacity doubles and rehashing occurs."

Linked List (Occasionally Asked)

A linked list is a chain of nodes, each containing data and a pointer (next) to the next node. Unlike arrays, elements are not stored contiguously.

Reverse Linked List — Iterative

public ListNode reverseList(ListNode head) {
    ListNode prev = null, curr = head;
    while (curr != null) {
        ListNode next = curr.next;
        curr.next = prev;
        prev = curr;
        curr = next;
    }
    return prev;
}
// Time: O(n) | Space: O(1)

Detect Cycle — Floyd's Algorithm (Tortoise & Hare)

public boolean hasCycle(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
        if (slow == fast) return true;
    }
    return false;
}
// Time: O(n) | Space: O(1)

Stack & Queue (Sometimes Asked)

Stack — Valid Parentheses

public boolean isValid(String s) {
    Deque<Character> stack = new ArrayDeque<>();
    for (char c : s.toCharArray()) {
        if (c == '(' || c == '[' || c == '{') {
            stack.push(c);
        } else {
            if (stack.isEmpty()) return false;
            char top = stack.pop();
            if (c == ')' && top != '(') return false;
            if (c == ']' && top != '[') return false;
            if (c == '}' && top != '{') return false;
        }
    }
    return stack.isEmpty();
}
// Time: O(n) | Space: O(n)

Queue — BFS Level Order (Tree)

public List<List<Integer>> levelOrder(TreeNode root) {
    List<List<Integer>> result = new ArrayList<>();
    if (root == null) return result;
    Queue<TreeNode> queue = new LinkedList<>();
    queue.offer(root);
    while (!queue.isEmpty()) {
        int size = queue.size();
        List<Integer> level = new ArrayList<>();
        for (int i = 0; i < size; i++) {
            TreeNode node = queue.poll();
            level.add(node.val);
            if (node.left != null) queue.offer(node.left);
            if (node.right != null) queue.offer(node.right);
        }
        result.add(level);
    }
    return result;
}

Binary Search — O(log n)

public int binarySearch(int[] nums, int target) {
    int left = 0, right = nums.length - 1;
    while (left <= right) {
        int mid = left + (right - left) / 2;
        if (nums[mid] == target) return mid;
        if (nums[mid] < target) left = mid + 1;
        else right = mid - 1;
    }
    return -1;
}

Big O Notation

Big O describes how time or space grows as input size (n) increases. SDET interviews expect you to state complexity after every solution.

ComplexityNameExamplePerformance (n=1M)
O(1)ConstantHashMap get/put, array index accessInstant
O(log n)LogarithmicBinary search, BST search~20 operations
O(n)LinearSingle loop, linear scan1,000,000 operations

How to Analyze Complexity

  1. Single loop over n elements → O(n)
  2. Nested loops → O(n²) (or O(n × m) if different sizes)
  3. Divide problem in half each step → O(log n)
  4. Sort then scan → O(n log n)
  5. HashMap operations inside one loop → O(n) total (each get/put is O(1) average)

1. Two Sum

Problem: Find two indices whose values sum to target.

Approach: HashMap stores value → index; look up complement.

public int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> map = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int comp = target - nums[i];
        if (map.containsKey(comp)) return new int[]{map.get(comp), i};
        map.put(nums[i], i);
    }
    return new int[]{};
}

2. Reverse String

Problem: Reverse a char array in-place.

Approach: Two pointers swap from both ends.

public void reverseString(char[] s) {
    int l = 0, r = s.length - 1;
    while (l < r) { char t = s[l]; s[l] = s[r]; s[r] = t; l++; r--; }
}

3. Valid Parentheses

Problem: Determine if brackets are balanced.

Approach: Stack — push opening, pop and match closing.

public boolean isValid(String s) {
    Deque<Character> st = new ArrayDeque<>();
    for (char c : s.toCharArray()) {
        if ("([{".indexOf(c) >= 0) st.push(c);
        else {
            if (st.isEmpty()) return false;
            char o = st.pop();
            if (c==')' && o!='(' || c==']' && o!='[' || c=='}' && o!='{') return false;
        }
    }
    return st.isEmpty();
}

5. Palindrome Check

public boolean isPalindrome(String s) {

6. Anagram Check

Problem: Check if two strings are anagrams.

Approach: Sort both and compare, or count character frequencies.

public boolean isAnagram(String s, String t) {
    if (s.length() != t.length()) return false;
    int[] count = new int[26];
    for (int i = 0; i < s.length(); i++) {
        count[s.charAt(i) - 'a']++;
        count[t.charAt(i) - 'a']--;
    }
    for (int c : count) if (c != 0) return false;
    return true;
}

8. Maximum Subarray (Kadane's)

public int maxSubArray(int[] nums) {

9. Merge Two Sorted Arrays

Problem: Merge two sorted arrays into one sorted array.

Approach: Two pointers compare and append smaller element.

public int[] mergeSorted(int[] a, int[] b) {
    int[] result = new int[a.length + b.length];
    int i = 0, j = 0, k = 0;
    while (i < a.length && j < b.length)
        result[k++] = a[i] <= b[j] ? a[i++] : b[j++];
    while (i < a.length) result[k++] = a[i++];
    while (j < b.length) result[k++] = b[j++];
    return result;
}

10. Binary Search

Problem: Find target in sorted array.

Approach: Compare with middle, eliminate half each step.

public int binarySearch(int[] nums, int target) {

11. String Compression

Problem: Compress "aabcccccaaa" → "a2b1c5a3". Return compressed if shorter.

Approach: Count consecutive chars and build result.

public String compress(String s) {
    if (s.isEmpty()) return s;
    StringBuilder sb = new StringBuilder();
    int count = 1;
    for (int i = 1; i <= s.length(); i++) {
        if (i < s.length() && s.charAt(i) == s.charAt(i-1)) count++;
        else { sb.append(s.charAt(i-1)).append(count); count = 1; }
    }
    return sb.length() < s.length() ? sb.toString() : s;
}

12. Roman to Integer

14. Longest Common Prefix

Approach: Compare characters column by column.

public String longestCommonPrefix(String[] strs) {
    if (strs.length == 0) return "";
    for (int i = 0; i < strs[0].length(); i++) {

17. Fibonacci (Iterative + Recursive)

Problem: Return nth Fibonacci number.

// Iterative — preferred
public int fibIterative(int n) {
    if (n <= 1) return n;
    int a = 0, b = 1;
    for (int i = 2; i <= n; i++) { int t = a + b; a = b; b = t; }
    return b;
}
// Time: O(n) | Space: O(1)

// Recursive — mention exponential without memoization
public int fibRecursive(int n) {
    if (n <= 1) return n;
    return fibRecursive(n-1) + fibRecursive(n-2);
}
// Time: O(2^n) | Space: O(n) call stack

18. Power of Two

Problem: Check if n is a power of 2.

Approach: Bit trick — powers of 2 have exactly one bit set: n & (n-1) == 0.

public boolean isPowerOfTwo(int n) {
    return n > 0 && (n & (n - 1)) == 0;
}

Time: O(1) | Space: O(1)

19. Missing Number (XOR / Sum)

Problem: Array contains n distinct numbers from 0 to n — find the missing one.

Approach: XOR all indices and values; duplicates cancel out.

public int missingNumber(int[] nums) {
    int xor = nums.length;
    for (int i = 0; i < nums.length; i++) xor ^= i ^ nums[i];
    return xor;
}
// Alternative: expected sum n*(n+1)/2 minus actual sum

Time: O(n) | Space: O(1)

20. Intersection of Two Arrays

Problem: Return common elements (unique) between two arrays.

Approach: HashSet from first array; check second array membership.

public int[] intersection(int[] nums1, int[] nums2) {
    Set<Integer> set = new HashSet<>();
    for (int n : nums1) set.add(n);
    Set<Integer> result = new HashSet<>();
    for (int n : nums2) if (set.contains(n)) result.add(n);
    return result.stream().mapToInt(Integer::intValue).toArray();
}

Time: O(n + m) | Space: O(n)

6When to use HashMap vs HashSet?HashMap: key-value pairs (count frequency, two sum). HashSet: unique elements only (find duplicates, membership check).
7What is two-pointer technique?Two indices moving toward each other or same direction to solve problems in O(n) instead of O(n²) — palindrome, two sum on sorted array, container with water.
8What is sliding window?Maintain a window [left, right] that expands/shrinks to find optimal subarray/substring in O(n) — max subarray, longest substring without repeat.
9Stack vs Queue?Stack: LIFO — undo, parentheses, DFS. Queue: FIFO — BFS, task scheduling. Deque: double-ended, useful for both.
10What is binary search prerequisite?Array must be sorted (or monotonic). O(log n) by eliminating half the search space each iteration.
11Recursive vs iterative — when to prefer which?Iterative preferred for interviews (no stack overflow, usually easier to optimize). Recursion good for tree/graph DFS when depth is bounded.
12What is Kadane's algorithm?O(n) solution for maximum subarray sum. At each index, decide: extend current subarray or start fresh from current element.
13How do you detect a cycle in a linked list?Floyd's tortoise and hare: slow moves 1 step, fast moves 2. If they meet, cycle exists. O(n) time, O(1) space.
14What is the difference between BFS and DFS?BFS: level-by-level with Queue — shortest path in unweighted graph. DFS: depth-first with Stack/recursion — path finding, cycle detection.
15What is stable sort?Sort that preserves relative order of equal elements. Merge sort and Timsort (Java's default) are stable. Quick sort is not.
16What is space complexity of merge sort?O(n) — requires auxiliary array. In-place sorts like quick sort use O(log n) for recursion stack.
17How would you find duplicate in array without extra space (sorted)?Compare adjacent elements after sorting — O(n log n). Or Floyd's cycle detection if array values map to indices (special case).
18What problems should an SDET focus on for Amazon/Google?Arrays, strings, HashMap, two pointer, sliding window, stack (parentheses), binary search, basic tree traversal. Medium difficulty, clean code, explain test cases and edge cases.
Final interview tip: Always talk through your approach before coding. State time/space complexity after writing the solution. Mention edge cases (empty input, single element, null, duplicates). Write clean, readable Java — interviewers at SDET roles value code quality as much as correctness.

What Is Appium?

Appium is an open-source mobile automation framework that lets you write tests for native, hybrid, and mobile web applications on Android and iOS using the same WebDriver protocol as Selenium. You write tests in Java, Python, JavaScript, C#, or Ruby — Appium translates commands to platform-specific automation engines.

  • Cross-platform: One API for Android and iOS (with platform-specific capabilities)
  • No app modification: Tests run against production builds — no SDK injection required
  • WebDriver-compatible: Reuses Selenium concepts: drivers, locators, waits, Page Objects
  • Open source: Maintained by the Appium community; no licensing cost

Appium Architecture

Appium follows a client-server model. Your test code (client) sends HTTP requests to the Appium Server, which routes commands to the appropriate platform driver.

Mobile locators differ from web CSS selectors. Prefer Accessibility ID and resource ID over XPath for stability and performance.

Page Object Model for Mobile

The POM pattern is identical to Selenium web automation: encapsulate locators and actions in page classes. Only the locator types and gesture methods differ.

Complete LoginPage for Mobile

package pages.mobile;

import io.appium.java_client.AppiumDriver;
import io.appium.java_client.pagefactory.AndroidFindBy;
import io.appium.java_client.pagefactory.AppiumFieldDecorator;
import org.openqa.selenium.WebElement;
import org.openqa.selenium.support.PageFactory;
import org.openqa.selenium.support.ui.ExpectedConditions;
import org.openqa.selenium.support.ui.WebDriverWait;

import java.time.Duration;

public class LoginPage {

    private final AppiumDriver driver;
    private final WebDriverWait wait;

    public LoginPage(AppiumDriver driver) {
        this.driver = driver;
        this.wait = new WebDriverWait(driver, Duration.ofSeconds(15));
        PageFactory.initElements(new AppiumFieldDecorator(driver), this);
    }

    @AndroidFindBy(id = "com.example.myapp:id/username")
    private WebElement usernameField;

    @AndroidFindBy(id = "com.example.myapp:id/password")
    private WebElement passwordField;

    @AndroidFindBy(accessibility = "login_btn")
    private WebElement loginButton;

    @AndroidFindBy(id = "com.example.myapp:id/error_message")
    private WebElement errorMessage;

    public LoginPage enterUsername(String username) {
        wait.until(ExpectedConditions.visibilityOf(usernameField));
        usernameField.clear();
        usernameField.sendKeys(username);
        return this;
    }

    public LoginPage enterPassword(String password) {
        passwordField.clear();
        passwordField.sendKeys(password);
        return this;
    }

    public HomePage tapLogin() {
        loginButton.click();
        return new HomePage(driver);
    }

    public LoginPage login(String username, String password) {
        enterUsername(username);
        enterPassword(password);
        return tapLogin() != null ? this : this;
    }

    public String getErrorMessage() {
        wait.until(ExpectedConditions.visibilityOf(errorMessage));
        return errorMessage.getText();
    }

    public boolean isLoginButtonDisplayed() {
        return loginButton.isDisplayed();
    }
}

Hybrid App Testing: NATIVE ↔ WEBVIEW Context Switching

Hybrid apps combine native screens and embedded web views (React Native WebView, Cordova, Ionic). Your test must switch contexts to interact with each layer.

public class HybridAppHelper {

    private final AppiumDriver driver;

    public HybridAppHelper(AppiumDriver driver) {
        this.driver = driver;
    }

    public void switchToWebView() {
        Set<String> contexts = driver.getContextHandles();
        for (String context : contexts) {
            if (context.contains("WEBVIEW")) {
                driver.context(context);
                return;
            }
        }
        throw new RuntimeException("No WEBVIEW context found. Available: " + contexts);
    }

    public void switchToNative() {
        driver.context("NATIVE_APP");
    }

    public void loginHybridFlow(String user, String pass) {
        // Native login screen
        switchToNative();
        driver.findElement(MobileBy.id("com.app:id/username")).sendKeys(user);
        driver.findElement(MobileBy.id("com.app:id/password")).sendKeys(pass);
        driver.findElement(MobileBy.AccessibilityId("login_btn")).click();

        // WebView dashboard after login
        switchToWebView();
        driver.findElement(By.cssSelector(".dashboard-title")).isDisplayed();

        // Back to native settings
        switchToNative();
        driver.findElement(MobileBy.AccessibilityId("Settings")).click();
    }
}

Interview Q&A (15+ Questions)

Fundamentals

Q1: What is Appium and why is it popular?
A: Appium is an open-source mobile automation framework using the WebDriver protocol. It's popular because tests can be written in multiple languages, no app recompilation is needed, and the same API works for Android and iOS native, hybrid, and mobile web apps.

Q2: Explain Appium architecture.
A: Test client sends HTTP WebDriver commands to Appium Server (Node.js). The server routes commands to UiAutomator2 (Android) or XCUITest (iOS), which interact with the device/emulator. The client never talks directly to the device.

Q3: How is Appium different from Selenium?
A: Selenium automates web browsers via browser drivers. Appium automates mobile apps via Appium Server and platform drivers (UiAutomator2/XCUITest). Appium extends WebDriver with mobile gestures, context switching, and mobile-specific locators.

Q4: What are Desired Capabilities?
A: Key-value pairs passed at session creation telling Appium the platform, device, app package/activity, automation engine, and other settings. Example: platformName=Android, appPackage=com.app, automationName=UiAutomator2.

Q5: What is the difference between appPackage and appActivity?
A: appPackage is the Android application package name (e.g., com.example.app). appActivity is the specific screen/activity to launch (e.g., .MainActivity). Together they define which app and screen to open.

Locators and Actions

Q6: What locator strategy do you prefer for mobile and why?
A: Accessibility ID first — cross-platform, fast, and aligns with accessibility standards. Resource ID second for Android. Avoid XPath unless necessary — it's slow and brittle on mobile UI hierarchies.

Q7: How do you handle scrolling to an element not visible on screen?
A: Use UiAutomator2 scrollable selector: new UiScrollable(new UiSelector().scrollable(true)).scrollIntoView(new UiSelector().text("Settings")). Alternatively, perform swipe gestures with W3C Actions API.

Q8: What is context switching in Appium?
A: Hybrid apps have NATIVE_APP and WEBVIEW contexts. Use driver.getContextHandles() to list contexts and driver.context("WEBVIEW_...") to switch. Web locators work only inside WEBVIEW context.

Q9: How do you perform a long press?
A: Use W3C Actions API: create a Sequence with pointer down, pause (1–2 seconds), pointer up. Legacy code uses TouchAction with press().waitAction().release().

Setup and Execution

Q10: What tools do you need to set up Appium for Android?
A: Java JDK, Android SDK (ANDROID_HOME), Node.js, Appium Server, UiAutomator2 driver, an emulator or real device with USB debugging, and the Appium Java client library in your project.

Q11: How do you find appPackage and appActivity?
A: Launch the app, then run adb shell dumpsys window | find "mCurrentFocus" (Windows) or use Appium Inspector after starting a session. For installed apps, aapt dump badging app.apk also reveals package and launch activity.

Q12: What is Appium Inspector used for?
A: Visual element inspection — shows the UI hierarchy, element attributes, and generates locators. Equivalent to browser DevTools for mobile apps.

Framework and Cloud

Q13: Can you use Page Object Model with Appium?
A: Yes — same pattern as Selenium. Use @AndroidFindBy / @iOSXCUITFindBy annotations with AppiumFieldDecorator for lazy initialization. Encapsulate locators and user actions in page classes.

Q14: Real device vs emulator — when do you use each?
A: Emulators for development, smoke tests, and CI cost savings. Real devices (or cloud farms) for release regression, hardware-dependent features (camera, GPS, biometrics), and accurate performance/UX validation.

Q15: How do you run Appium tests on BrowserStack?
A: Upload the .apk/.ipa, configure cloud hub URL (hub-cloud.browserstack.com), set BrowserStack credentials in capabilities (bstack:options), and point your driver to the cloud hub instead of localhost.

Q16: What causes "Session not created" errors in Appium?
A: Common causes: Appium Server not running, wrong capabilities, device not connected (adb devices empty), incompatible driver version, app not installed, or port 4723 blocked.

Q17: How do you handle flaky mobile tests?
A: Use explicit waits instead of Thread.sleep, prefer stable locators (accessibility ID), reset app state between tests, ensure network stability, capture screenshots on failure, and run on real devices for final validation.

Q18: What is the W3C Actions API in Appium 2.x?
A: Standard WebDriver actions replacing deprecated TouchAction. Uses PointerInput, KeyInput, and Sequence classes for tap, swipe, and multi-touch gestures — aligned with Selenium 4 actions.

Why Security Matters for QA

  • Shift-left security: Find vulnerabilities early in development — cheaper than production fixes
  • DevSecOps: Security integrated into CI/CD pipelines alongside functional tests
  • Everyone's responsibility: QA validates security requirements just like functional requirements
  • Compliance: GDPR, PCI-DSS, HIPAA require evidence of security testing
  • Customer trust: Data breaches destroy reputation — QA is a gatekeeper before release

OWASP Top 10 (2021)

The OWASP Top 10 lists the most critical web application security risks. Know each category, a real-world example, and what QA can verify.

1. Broken Access Control

What: Users can access resources or perform actions outside their authorized permissions — e.g., a regular user accessing admin pages or another user's data.

Example: Change URL from /user/profile/123 to /user/profile/456 and view another user's profile.

QA verification:

  • Test role-based access — login as each role (user, manager, admin) and verify permitted pages/actions
  • Attempt to access admin URLs directly as a normal user — expect 403 Forbidden
  • Test horizontal privilege escalation (access peer's data) and vertical escalation (access admin functions)
  • Verify API returns 403 when accessing other users' resources with valid but unauthorized token

2. Cryptographic Failures

What: Sensitive data exposed due to weak or missing encryption — passwords in plain text, HTTP instead of HTTPS, unencrypted database fields.

Example: Password reset email contains password in plain text; API returns credit card numbers unmasked.

QA verification:

  • Verify all pages use HTTPS — check for mixed content warnings
  • Inspect network traffic (browser DevTools) — sensitive data should not appear in URLs or unencrypted payloads
  • Check logs and error reports — passwords, tokens, and PII should never appear
  • Verify password fields are masked; confirm sensitive DB columns are encrypted at rest (coordinate with dev/DBA)

3. Injection (SQL, XSS, Command)

What: Untrusted input sent to an interpreter (database, browser, OS) causes unintended commands to execute.

SQL Injection example:

' OR '1'='1
admin'--
'; DROP TABLE users;--

XSS (Cross-Site Scripting) example:

<script>alert('XSS')</script>
<img src=x onerror=alert('XSS')>
javascript:alert(document.cookie)

QA verification:

  • Test special characters in ALL input fields: ' " < > ; -- /* */
  • SQL injection in login, search, filters — expect graceful error, not database error messages
  • XSS in text fields, comments, profile names — script should NOT execute (should be escaped/sanitized)
  • Verify output encoding — user-supplied HTML displayed as text, not rendered

4. Insecure Design

What: Fundamental architecture flaws — missing security controls by design, not implementation bugs.

Example: No rate limiting on login allows unlimited brute force; password recovery reveals whether an email exists.

QA verification:

  • Review threat models and security requirements during story refinement
  • Ask: "What happens if a malicious user does X?" during test design
  • Verify security user stories have acceptance criteria and test cases

5. Security Misconfiguration

What: Default credentials, unnecessary features enabled, verbose error messages, open cloud storage buckets.

Example: Stack trace shown to users on 500 error; default admin/admin credentials still work.

QA verification:

  • Verify error messages are user-friendly — no stack traces, SQL queries, or internal paths exposed
  • Check default accounts are disabled or passwords changed
  • Verify security headers (CSP, X-Frame-Options, HSTS) in response headers
  • Confirm debug mode is OFF in staging/production environments

6. Vulnerable and Outdated Components

What: Using libraries/frameworks with known CVEs — outdated Spring, Log4j, jQuery versions.

Example: Log4Shell (CVE-2021-44228) in Log4j 2.x — remote code execution via crafted log input.

QA verification:

  • Run dependency scans: mvn dependency:tree to review library versions
  • Integrate OWASP Dependency-Check or Snyk in CI pipeline
  • Verify no critical/high CVEs in release dependency report

7. Identification and Authentication Failures

What: Weak password policies, missing MFA, session not invalidated on logout, credential stuffing succeeds.

Example: Password "123456" accepted; session token still valid after logout; no account lockout after 100 failed logins.

QA verification:

  • Test password policy — minimum length, complexity, common password rejection
  • Verify session expires after inactivity timeout
  • Confirm logout invalidates session server-side (old token rejected)
  • Test brute force protection — account lockout or CAPTCHA after N failed attempts
  • Verify MFA flow if implemented

8. Software and Data Integrity Failures

What: Code or data modified without verification — unsigned updates, compromised CI/CD pipeline, insecure deserialization.

Example: Application downloads plugins without signature verification; CI pipeline allows unsigned artifact deployment.

QA verification:

  • Verify CI/CD pipeline has approval gates for production deployment
  • Confirm artifacts are signed and checksums validated
  • Test that tampered update packages are rejected

9. Security Logging and Monitoring Failures

What: Security events not logged or monitored — failed logins, access control violations, injection attempts go unnoticed.

Example: 10,000 failed login attempts produce no alert; admin privilege changes not audited.

QA verification:

  • Trigger security events (failed login, unauthorized access) and verify they appear in audit logs
  • Confirm logs include timestamp, user ID, IP address, action, and outcome
  • Verify sensitive data is NOT logged (passwords, full credit card numbers)

10. Server-Side Request Forgery (SSRF)

What: Attacker tricks the server into making requests to internal resources — metadata endpoints, internal APIs, localhost services.

Example: URL input field accepts http://169.254.169.254/latest/meta-data/ (AWS metadata) and returns cloud credentials.

QA verification:

  • Test URL/file import fields with internal URLs: http://localhost, http://127.0.0.1, http://169.254.169.254
  • Verify server rejects or blocks requests to private IP ranges
  • Test webhook/callback URL fields for SSRF vectors

API Security Testing

APIs are the primary attack surface in modern applications. QA should include security scenarios in API test suites alongside functional tests.

Key API Security Test Scenarios

  • Authentication bypass: Call protected endpoints without token — expect 401 Unauthorized
  • Authorization: Access another user's data with valid token — expect 403 Forbidden
  • Rate limiting: Send 1000 requests in 1 second — expect 429 Too Many Requests
  • Input validation: SQL injection and XSS payloads in JSON body fields
  • Mass assignment: Send extra fields (e.g., "role": "admin") — should be ignored

Complete REST Assured Security Test Examples

import io.restassured.http.ContentType;
import org.testng.annotations.Test;

import static io.restassured.RestAssured.*;
import static org.hamcrest.Matchers.*;

public class ApiSecurityTests {

    private static final String BASE_URI = "https://api.example.com";

    @Test(description = "Access protected endpoint without token — expect 401")
    public void testAuthenticationBypass() {
        given()
            .baseUri(BASE_URI)
            .contentType(ContentType.JSON)
        .when()
            .get("/users/me")
        .then()
            .statusCode(401)
            .body("error", equalTo("Unauthorized"));
    }

    @Test(description = "Access another user's data — expect 403")
    public void testAuthorizationViolation() {
        String userToken = AuthHelper.getToken("user_a@test.com", "password");

        given()
            .baseUri(BASE_URI)
            .header("Authorization", "Bearer " + userToken)
        .when()
            .get("/users/user_b_id/orders")
        .then()
            .statusCode(anyOf(is(403), is(404)));  // 403 Forbidden or 404 Not Found
    }

    @Test(description = "SQL injection in login API")
    public void testSqlInjectionInLogin() {
        String sqlPayload = "' OR '1'='1";

        given()
            .baseUri(BASE_URI)
            .contentType(ContentType.JSON)
            .body("""
                {"email": "%s", "password": "anything"}
                """.formatted(sqlPayload))
        .when()
            .post("/auth/login")
        .then()
            .statusCode(anyOf(is(400), is(401)))  // Should NOT return 200
            .body("token", nullValue());
    }

    @Test(description = "XSS payload in user profile update")
    public void testXssInProfileUpdate() {
        String token = AuthHelper.getToken("user@test.com", "password");
        String xssPayload = "<script>alert('XSS')</script>";

        given()
            .baseUri(BASE_URI)
            .header("Authorization", "Bearer " + token)
            .contentType(ContentType.JSON)
            .body("""
                {"displayName": "%s"}
                """.formatted(xssPayload))
        .when()
            .put("/users/me/profile")
        .then()
            .statusCode(200);

        // Verify sanitized on retrieval
        given()
            .header("Authorization", "Bearer " + token)
        .when()
            .get("/users/me/profile")
        .then()
            .body("displayName", not(containsString("<script>")));
    }

    @Test(description = "Rate limiting — expect 429 after burst")
    public void testRateLimiting() {
        String token = AuthHelper.getToken("user@test.com", "password");
        int rateLimitHit = 0;

        for (int i = 0; i < 100; i++) {
            int status = given()
                .baseUri(BASE_URI)
                .header("Authorization", "Bearer " + token)
            .when()
                .get("/users/me")
            .then()
                .extract().statusCode();

            if (status == 429) {
                rateLimitHit = status;
                break;
            }
        }
        // Assert rate limiting kicks in (adjust threshold per API docs)
        assert rateLimitHit == 429 || true : "Verify rate limit threshold with API team";
    }
}

Interview Q&A (15+ Questions)

Fundamentals

Q1: Why should QA care about security testing?
A: Security defects reach production if untested. QA validates security requirements alongside functional ones. Shift-left security and DevSecOps expect QA to run basic checks — OWASP Top 10 verification, dependency scans, and API auth tests — before release.

Q2: What is the OWASP Top 10?
A: A standard awareness document listing the 10 most critical web application security risks, updated periodically (2021 is current). It guides what to test: injection, broken access control, cryptographic failures, etc.

Q3: What is the difference between SAST and DAST?
A: SAST analyzes source code statically without running the app (SonarQube). DAST tests a running application by sending attack payloads (OWASP ZAP, Burp Suite). SAST finds code-level flaws early; DAST finds runtime and configuration issues.

Q4: Explain SQL injection and how you test for it.
A: SQL injection occurs when user input is concatenated into SQL queries. Test by entering payloads like ' OR '1'='1 in input fields. Expect the app to reject or sanitize input — never return database errors or unauthorized data.

Q5: What is XSS and how do you test for it?
A: Cross-Site Scripting injects malicious scripts into pages viewed by other users. Test by entering <script>alert('XSS')</script> in input fields. The script should be escaped and displayed as text, not executed.

Practical Testing

Q6: How do you test for broken access control?
A: Login as a low-privilege user and attempt to access admin URLs, API endpoints, and other users' data. Verify 403 Forbidden responses. Test both UI navigation and direct URL/API access.

Q7: How do you test API authentication and authorization?
A: Authentication: call protected endpoints without token (expect 401). Authorization: call with valid token but access another user's resource (expect 403). Test role-based access for admin vs user endpoints.

Q8: What security tests can you automate in CI/CD?
A: SAST (SonarQube on every build), dependency scanning (OWASP Dependency-Check), container scanning (Trivy), DAST baseline scan (OWASP ZAP API), and API security tests (Rest Assured auth/injection tests).

Q9: What is SSRF and how would you test it?
A: Server-Side Request Forgery tricks the server into requesting internal resources. Test URL input fields with http://localhost, http://169.254.169.254 (cloud metadata). Server should block private IP ranges.

Tools and Process

Q10: What is OWASP ZAP and when do you use it?
A: Free DAST tool that spiders your app and sends attack payloads. Use against staging environments for automated vulnerability scanning. Integrate into CI via ZAP API for baseline scans on every deployment.

Q11: What is the difference between Burp Suite Community and Pro?
A: Community is free with manual intercept/repeater — good for learning and ad-hoc testing. Pro adds automated scanning, advanced intrusion tools, and reporting — used by professional penetration testers.

Q12: How do you verify security headers?
A: Check HTTP response headers in browser DevTools or Rest Assured: Strict-Transport-Security, Content-Security-Policy, X-Frame-Options, X-Content-Type-Options. Missing headers indicate misconfiguration.

Q13: What is a CVE and how does it relate to QA?
A: Common Vulnerabilities and Exposures — publicly cataloged security flaws with severity scores. QA ensures dependency scans flag critical CVEs and block releases until patched (e.g., Log4Shell CVE-2021-44228).

Q14: How do you test session management?
A: Verify session timeout after inactivity, confirm logout invalidates the token server-side, test concurrent session limits, check session cookies have HttpOnly and Secure flags, and ensure session IDs are not in URLs.

Q15: What is shift-left security?
A: Moving security testing earlier in the development lifecycle — during design, coding, and PR review — rather than only before release. QA participates by reviewing security requirements, running SAST/DAST in CI, and testing security stories.

Q16: Can QA perform penetration testing?
A: QA performs basic security testing (OWASP checks, ZAP scans, manual payload testing). Professional penetration testing requires specialized skills and is done by security experts before major releases. Know the difference in interviews.

Q17: How do you test password policies?
A: Attempt passwords below minimum length, without required character types, using common passwords from breach lists, and verify rejection messages. Confirm passwords are never returned in API responses or displayed in logs.

Interview pro tip: Frame security answers around OWASP Top 10 + your QA workflow: "For every user input field, I test functional behavior AND injection payloads. For every API endpoint, I test auth, authorization, and rate limiting alongside happy path."

Q10: What Azure services are most relevant to QA?
A: App Service (web app hosting), Azure DevOps (CI/CD pipelines), Azure Monitor (logging), Blob Storage (artifacts), and Container Instances (Docker-based test environments).

Q11: What is Azure DevOps and how does QA use it?
A: Microsoft's CI/CD and project management platform. QA configures test stages in pipelines, reviews test results and trends, manages test plans linked to user stories, and triggers automated suites on deployment.

IaC and Certifications

Q12: What is Infrastructure as Code and why should QA care?
A: IaC (Terraform, CloudFormation) defines infrastructure in version-controlled files. QA cares because environment inconsistencies cause test failures — IaC ensures dev, QA, and staging are provisioned identically.

Q13: Which cloud certification is best for QA engineers?
A: AWS Cloud Practitioner (CLF-C02) is most recommended — foundational, covers core services QA interacts with (EC2, S3, Lambda, CloudWatch). Azure AZ-900 if targeting Azure-heavy organizations.

Q14: How do you run load tests against cloud applications?
A: Use JMeter or k6 targeting staging URL (with team approval). Monitor CloudWatch metrics (CPU, latency, error rate) during the test. Coordinate with DevOps to avoid triggering production alerts or excessive costs.

Q15: What is a bastion host and why does QA need it?
A: A bastion (jump host) is a secure entry point to access resources in private VPC subnets. QA connects to bastion via SSH, then to internal test servers that aren't publicly accessible.

Q16: How do CI/CD pipelines integrate with cloud testing?
A: Pipeline builds app → deploys to cloud staging (EC2/ECS/App Service) → triggers automated test suite against staging URL → uploads reports to S3/Blob → notifies team on pass/fail. Cloud platforms like BrowserStack integrate as parallel test stages.

Q17: What is the difference between IaaS, PaaS, and SaaS?
A: IaaS (EC2 — you manage OS and app), PaaS (App Service/Heroku — platform manages OS, you deploy code), SaaS (Gmail, Jira — fully managed). QA tests apps on all three — know where responsibility shifts.

Interview pro tip: You don't need to architect cloud solutions. Demonstrate practical knowledge: "After deployment, I hit the staging health endpoint, run my regression suite, and if tests fail, I check CloudWatch logs filtered to the test time window for exceptions."
