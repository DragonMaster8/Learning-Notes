Complete foundation for API testing — what APIs are, REST architecture, HTTP methods, status codes, URL anatomy, headers, request/response structure, and why API testing matters. These concepts apply to ALL API testing tools.

What is an API?

API (Application Programming Interface) is a messenger that takes a request from one software application, delivers it to another, and brings back the response. It acts as a bridge between different systems, allowing them to communicate even when built with different programming languages or running on different platforms.

Definition

An API defines how two systems talk to each other: which endpoints to call, what data format to use (usually JSON), and what responses to expect. Without APIs, a frontend (Angular/React) could not call a backend written in Java, Python, or C#.

Real-World Analogy — The Waiter

Think of an API as a restaurant waiter. The kitchen (backend) speaks one language. Customers (clients) speak another. The waiter takes orders (requests), communicates with the kitchen, and brings back food (responses) — without either side understanding the other’s internal details.

Why APIs Exist

  • Different technology stacks: A company may have a modern frontend (TypeScript/Angular) but a legacy backend (Java). Direct method calls between languages are impossible. The frontend sends HTTP + JSON; the API translates to backend method calls and returns JSON.
  • Third-party integration: Marriott cannot share its entire backend with Hotels.com. Marriott exposes an API; Hotels.com calls it with customer details and receives booking confirmations — no code sharing required.
  • Mobile apps: iOS (Swift) and Android (Kotlin) both need the same server data. One API serves both platforms using HTTP and JSON.

REST Architecture

REST (Representational State Transfer) is a set of design rules (not a protocol) that makes APIs lightweight, scalable, and easy to maintain. Most modern websites use REST APIs. Older systems may use SOAP (rigid, XML-based).

6 REST Constraints

ConstraintDescriptionTesting Impact
StatelessServer does not store client context between requests. Each request must contain all necessary information.Every request needs auth tokens, IDs, and params — no reliance on server memory.
Client-ServerSeparation of concerns: UI and data storage evolve independently.Test API independently of UI; contract-driven testing.
CacheableResponses must define themselves as cacheable or not; clients can reuse cached responses.Verify Cache-Control headers; test idempotent GET caching.
Uniform InterfaceStandard endpoints using HTTP methods (GET, POST, PUT, DELETE) with consistent resource naming.Same CRUD patterns across all resources; predictable test design.
Layered SystemClient cannot tell if connected directly to end server or through intermediaries (load balancer, gateway).Tests should work through proxies/CDNs; don’t assume direct server access.
Code on Demand (optional)Servers can send executable code (e.g., JavaScript) to extend client functionality.Rare in API testing; mainly relevant for web clients.

REST vs SOAP Comparison

AspectRESTSOAP
FormatPrimarily JSON (lightweight, easy to parse)XML exclusively for requests and responses
ProtocolFollows HTTP protocols (GET, POST, PUT, DELETE)SOAP protocol with its own standards; all requests typically POST

Introduction

REST Assured is a Java library for automating REST API testing. It provides a domain-specific language (DSL) for writing tests using Given-When-Then BDD-style syntax, making API tests readable and maintainable.

Why Use REST Assured?

  • Java DSL: Fluent, readable syntax integrated with Java ecosystem
  • BDD-style: Given-When-Then structure mirrors natural language
  • JsonPath: Built-in JSON parsing and field extraction
  • TestNG/JUnit integration: Works seamlessly with test frameworks, Maven, and CI/CD
  • Hamcrest matchers: Powerful response body assertions

API Chaining (End-to-End)

Step 1: Add Place (POST) ? Step 2: Extract place_id from response ? Step 3: Update Place (PUT) using extracted place_id ? Step 4: Get Place (GET) using same place_id ? Step 5: Validate address was updated correctly
public class EndToEndTest {
    @Test
    public void addUpdateGetPlace() {
        RestAssured.baseURI = "https://rahulshettyacademy.com";

        // Step 1: Add Place
        String addResponse = given()
            .queryParam("key", "qaclick123")
            .header("Content-Type", "application/json")
            .body(payload.AddPlace())
        .when()
            .post("maps/api/place/add/json")
        .then()
            .statusCode(200)
            .extract().asString();

        // Step 2: Extract place_id
        JsonPath js = new JsonPath(addResponse);
        String placeId = js.getString("place_id");

        // Step 3: Update Place
        String newAddress = "70 Summer walk, USA";
        String updateBody = "{\n" +
            "\"place_id\": \"" + placeId + "\",\n" +
            "\"address\": \"" + newAddress + "\",\n" +
            "\"key\": \"qaclick123\"\n" +
            "}";

        given()
            .queryParam("key", "qaclick123")
            .header("Content-Type", "application/json")
            .body(updateBody)
        .when()
            .put("maps/api/place/update/json")
        .then()
            .statusCode(200)
            .body("msg", equalTo("Address successfully updated"));

        // Step 4: Get Place and validate
        String getResponse = given()
            .queryParam("key", "qaclick123")
            .queryParam("place_id", placeId)
        .when()
            .get("maps/api/place/get/json")
        .then()
            .statusCode(200)
            .extract().asString();

        JsonPath jsGet = new JsonPath(getResponse);
        String actualAddress = jsGet.getString("address");
        Assert.assertEquals(actualAddress, newAddress);
    }
}

1. Postman Mock Servers

What Is a Mock Server?

A mock server simulates a real API by returning predefined responses to incoming HTTP requests. Postman hosts the mock at a unique URL (e.g., https://4e612d63-31bb-41f3-9430-d30ca851391e.mock.pstmn.io). The client sends the same request path and method it would send to production; the mock matches the request against saved examples and returns the configured body and status code.

Tester Request → Mock Server (Postman) → Predefined Mock Response ↓ (When real API is ready) ↓ Tester Request → Real API Server → Actual Response

Why Use Mock Servers in Agile?

In a typical sprint, backend development may take 2 weeks while QA has automation tasks in the same sprint. Without mocks, testers wait idle. With mocks:

  • Developers build the real API in parallel
  • Testers create collections, write assertions, and chain requests against mock data immediately
  • When the real API deploys, change one environment variable (baseUrl or mockURL) — automation code stays unchanged
  • Sprint deadlines (e.g., 10-day sprints) are met because automation is done before backend completion
Real-world result: Course-management automation completed in 2 days instead of waiting 2 weeks for backend completion — only the URL needed to change when the real API went live.

function getNestedValue(obj, path) {
    return path.split(".").reduce(function (current, key) {

                ? current[arrayMatch[1]][parseInt(arrayMatch[2], 10)]

        }
        return current ? current[key] : undefined;
    }, obj);
}

pm.sendRequest() sends an HTTP request programmatically from a Pre-request or Tests script. Use it when you need to:

}, function (err, res) {

        console.error("GET check failed:", err);

                console.log("Delete status:", delRes.code);
            }
        });

OAuth Retry Pattern — exchangeCodeWithRetry()

let retryCount = 0;
const maxRetries = 3;

function exchangeCodeWithRetry(code) {
    if (retryCount >= maxRetries) {
        throw new Error("Max retries exceeded for OAuth code exchange");
    }

    pm.sendRequest({
        url: "https://www.googleapis.com/oauth2/v4/token",
        method: "POST",
        header: { "Content-Type": "application/x-www-form-urlencoded" },
        body: {
            mode: "urlencoded",
            urlencoded: [
                { key: "code", value: code },
                { key: "client_id", value: pm.variables.get("client_id") },
                { key: "client_secret", value: pm.variables.get("client_secret") },
                { key: "redirect_uri", value: pm.variables.get("redirect_uri") },
                { key: "grant_type", value: "authorization_code" }
            ]
        }
    }, function (err, response) {
        if (err) {
            console.error("Token exchange error:", err);
            return;
        }

        const body = response.json();

        if (response.code === 400 && body.error === "invalid_grant") {
            retryCount++;
            console.log("Retry " + retryCount + "/" + maxRetries + " for invalid_grant");
            exchangeCodeWithRetry(code);
        } else if (response.code === 200) {
            pm.globals.set("access_token", body.access_token);
            console.log("Token stored successfully");
        } else {
            console.error("Token exchange failed:", response.code, body);
        }
    });
}

const authCode = pm.globals.get("authorization_code");
exchangeCodeWithRetry(authCode);

OAuth 2.0 Deep Dive

Why OAuth?

In microservices, backend services communicate securely without exposing user credentials.

Scenario: User logs into banking app once. Homepage calls Account Overview, which internally calls Balance API, Credit Card API, Profile API. Passing credentials to every internal API is insecure. Solution: OAuth 2.0 — services use access tokens instead of credentials.

Authorization Code Flow (7 Steps)

  1. User Initiates Login: User clicks “Login with Google” on client application (e.g., BookMyShow). Application prepares authorization request URL.
  2. Redirect to Authorization Server: Browser redirects to Google’s authorization endpoint with client_id, redirect_uri, response_type=code, scope, state.
  3. User Authenticates: User enters credentials directly on Google’s page. Password never shared with client. User sees scope permissions requested.
  4. Authorization Code Returned: Google redirects to redirect_uri with code as query parameter: ?code=4/0AY0e-g6... State parameter returned for validation.
  5. Client Exchanges Code for Token: Client makes POST to token endpoint with code, client_id, client_secret, redirect_uri, grant_type=authorization_code. Back-channel call (server-to-server).
  6. Access Token Received: Authorization server validates code and credentials. Returns access_token (and optionally refresh_token, id_token). Client stores token.
  7. Access Protected Resources: Client includes access_token in API requests. Resource server validates token. Sent as Authorization: Bearer ya29... header.

Token Handling Best Practices

Security Checklist

  • Always use HTTPS for all OAuth endpoints
  • Never expose client_secret in client-side code (browser/mobile apps)
  • Validate state parameter to prevent CSRF attacks
  • Keep access tokens confidential; transmit only over secure channels
  • Use short-lived access tokens (typically 1 hour or less)
  • Implement refresh token rotation for better security
  • Store credentials in environment variables, never hardcode
  • Request minimal scopes (least privilege principle)

Testing Checklist

  • Get client_id, client_secret, endpoints from development team
  • Test token expiration and refresh scenarios
  • Validate error responses (invalid_grant, invalid_client, etc.)
  • Create mock OAuth servers for integration testing
  • Log OAuth failures for debugging (without exposing secrets)
  • Register separate clients for dev, staging, and production

OAuth vs Authentication

ConceptQuestionExample

OAuth 2.0 handles authorization, not authentication. User authenticates separately on the authorization server, then delegates permission to the application. Password never leaves the authorization server.

Google OAuth Automation Challenges

Problem: Google blocks automated login attempts to Gmail/Google accounts. Neither Selenium, Cypress, nor any tool can automate Google login.

Why: Google implements anti-bot mechanisms to prevent credential stuffing and unauthorized automated access.

Industry Workaround

  1. Manual code once daily: Developer/tester manually generates authorization code through browser login
  2. Shared token: Store access token in environment/global variables for all automated tests
  3. 12–13 hour expiry: Set token expiration to cover full workday so one token serves all test runs
  4. Run automation suite: Execute all OAuth-dependent tests using the shared token until expiry

PKCE, OpenID Connect, JWT

PKCE (Proof Key for Code Exchange)

Extension for public clients (mobile/native apps). Prevents authorization code interception attacks using code_verifier and code_challenge. Required for OAuth 2.0 in mobile applications.

OpenID Connect (OIDC)

Identity layer built on top of OAuth 2.0. Adds authentication to OAuth’s authorization. Provides id_token (JWT) with user identity claims and standard UserInfo endpoint for profile data.

JWT (JSON Web Token)

Common format for access tokens. Self-contained with claims (expiration, issuer, audience). Can be validated without calling authorization server. Signed to prevent tampering. Structure: header.payload.signature

  10. Existence: containsKey() on Map; absence: assertNull on missing path
  11. Java streams: filter list by attribute, then assert properties
  12. SoapUI: JSONPath Match, Existence, Count assertion types
  13. JSON Schema keywords: type, properties, required, items, $schema, enum, min/max
  14. REST Assured: matchesJsonSchemaInClasspath("file.json")
  15. Postman: pm.response.to.have.jsonSchema(schema)
  16. Common mistakes: index on object, missing @, type mismatch, no null check
  17. Dynamic strategy: filter → assert existence → schema → parameterize

getValueByCondition() Reusable Helper

import io.restassured.path.json.JsonPath;
import java.util.List;
import java.util.Map;

public class JsonUtils {

    public static String getValueByCondition(
            JsonPath jp,
            String path,
            String key,
            String value,
            String target) {

        List<Map<String, Object>> items = jp.getList(path);
        return items.stream()
            .filter(it -> value.equals(it.get(key)))
            .map(it -> (String) it.get(target))
            .findFirst()
            .orElse(null);
    }
}

What is a Web Service?

Definition: A software system designed for machine-to-machine interaction over a network (typically HTTP/HTTPS). It allows applications built on different programming languages and platforms to communicate using structured messages (XML or JSON).

Real-World Example: Airline + Hotel Integration

British Airways (Java website) wants to show Paris hotels after a flight booking. Hotels.com (.NET) cannot share code directly due to platform differences. A web service acts as a bridge: British Airways sends only the destination ("Paris"), the service talks to Hotels.com backend, and returns the hotel list — no direct code integration needed.

Why Web Services Exist

  • Cross-platform communication (Java, .NET, Python, mobile)
  • Loose coupling between systems — consumers need not know internal implementation
  • Reusable business logic exposed as standardized endpoints
  • Integration with third-party APIs (PayPal, Google Maps, Twitter feeds)

SOAP Protocol

SOAP (Simple Object Access Protocol) is a protocol-based web service standard with strict rules for exchanging structured information exclusively in XML format.

SOAP Message Structure

SOAP Message Components: +-----------------------------------+ | SOAP Envelope (Required) | | +-----------------------------+ | | | SOAP Header (Optional) | | | | (auth tokens, session IDs) | | | +-----------------------------+ | | +-----------------------------+ | | | SOAP Body (Required) | | | | +-----------------------+ | | | | | Operation + Parameters| | | | | +-----------------------+ | | | +-----------------------------+ | | SOAP Fault (in response only) | +-----------------------------------+

Example SOAP Request (Calculator Add)

<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Body>
    <Add xmlns="http://tempuri.org/">
      <intA>2</intA>
      <intB>3</intB>
    </Add>
  </soap:Body>

Example SOAP Response

<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Body>
    <AddResponse xmlns="http://tempuri.org/">
      <AddResult>5</AddResult>
    </AddResponse>
  </soap:Body>
</soap:Envelope>

SOAP Fault Explanation

When an error occurs, SOAP returns a Fault element inside the Body instead of normal data:

<soap:Fault>
  <faultcode>soap:Client</faultcode>
  <faultstring>Invalid input parameters</faultstring>
  <detail>intA must be a positive integer</detail>
</soap:Fault>

Faults provide structured error codes and messages — a built-in advantage over REST where errors vary by implementation.

WSDL (Web Services Description Language)

What is WSDL? An XML document that describes network services as a set of endpoints operating on messages. It acts as the contract or blueprint for interacting with a SOAP web service. WSDL is mandatory for SOAP because the rigid protocol requires a formal contract defining operations, inputs, outputs, and endpoint URLs.

ElementDescription

operationIndividual action within a portType

Restaurant Analogy: WSDL = the menu (lists all dishes, ingredients, prices). SOAP = the order slip (structured request). Service = the kitchen (processes the order and returns the meal).

XML Namespaces

Why they exist: XML allows elements from different vocabularies to coexist in one document without name collisions. A colon prefix (e.g., soap:Envelope) identifies which namespace an element belongs to.

How they work: Declared via xmlns:prefix="URI" on an element. The URI is a unique identifier (not necessarily a fetchable URL).

3. Advanced Assertions (VERY DETAILED)

Module 7 introduced basic assertions. This section covers namespace-aware XPath, uniqueness checks, regex patterns, SLA performance gates, and security scanning — the full assertion stack for production-grade SOAP testing.

3.1 XPath Match with Namespace

SOAP responses use XML namespaces. Without declaring them, XPath cannot locate elements.

Scenario: Verify employee John’s age is 26 in a namespaced response.

<!-- Sample response fragment -->
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Body>
    <GetEmployeeResponse xmlns:emp="http://tempuri.org/EmployeeSchema">
      <emp:Employee>
        <emp:Name>John</emp:Name>
        <emp:Age>26</emp:Age>
      </emp:Employee>
    </GetEmployeeResponse>
  </soap:Body>
</soap:Envelope>

XPath expression:

//emp:Employee[emp:Name='John']/emp:Age/text()

Expected Result: 26

3.2 Namespace Declaration Steps

  1. In the request panel, open the Assertions tab.
  2. Click + (Add Assertion) → select XPath Match.
  3. Paste the XPath expression using your prefix (e.g., emp:).
  4. Click the Declare button in the assertion dialog.
  5. SoapUI shows namespaces detected in the last response.
  6. Add prefix emp mapped to xmlns:emp=http://tempuri.org/EmployeeSchema
  7. Alternatively, prefix ns1 works — the prefix name is your choice; the URI must match exactly.
  8. Enter expected value and save.
Interview tip: If XPath returns empty, the #1 cause is missing namespace declaration. Always click Declare after receiving a response, then re-run the request before configuring the assertion.

3.3 XPath Not Exists Assertion

Passes when no node matches the XPath query. Essential for verifying deletion.

  • Use case: After deleteEmployee, confirm the employee node is gone.
  • XPath: //emp:Employee[emp:Name='Josh']
  • Assertion type: XPath Not Exists
  • If any matching node is found, the assertion fails.

3.4 count() for Uniqueness

Ensure exactly one record is returned — not zero, not duplicates.

  • XPath: count(//emp:Employee)
  • Expected Result: 1
  • If result is 0: employee not found.
  • If result is >1: duplicate data or flawed query — test fails.

3.6 Valid SOAP Response Assertion

Confirms the response is structurally valid SOAP, not an HTML error page or plain text.

  • Root element is Envelope
  • Contains a Body element
  • Follows standard SOAP namespace structure
  • Failure indicates: server crash, proxy error, or malformed XML

3.7 Response SLA Assertion (< 200ms)

  1. Assertions → Add → Response SLA
  2. Set maximum response time: 200 milliseconds
  3. Every test run automatically measures round-trip time
  4. Failure means performance regression — slow DB query, network issue, or overloaded server

3.8 Security Assertion: No Session Tokens in Body

Functional tests can pass while leaking sensitive data. Always scan responses for tokens, internal IDs, and stack traces.

// Script Assertion — security check for session tokens in response body
def response = messageExchange.response.responseContent

// Fail if a 32-char hex token pattern appears in the body
def tokenPattern = ~/[a-f0-9]{32}/
assert !(response =~ tokenPattern) : "SECURITY FAIL: Session token found in response body"

// Fail if common sensitive keywords appear
def forbidden = ["sessionToken", "password", "stackTrace", "SQLException"]
forbidden.each { keyword ->
    assert !response.contains(keyword) : "SECURITY FAIL: '${keyword}' found in response body"
}
log.info "Security assertion passed — no sensitive data in body"
Real-world example: After login, the session token should arrive in an HTTP-only cookie, not in the SOAP/JSON response body. A security assertion prevents this regression from reaching production.

6. WsdlTestCaseRunner (Cross Test Case Execution)

Sometimes one Groovy script must run test steps that live in a different TestCase. The current script’s context only works within its own TestCase — you need WsdlTestCaseRunner to create a valid RunContext for the target.

6.1 Why It Is Needed

  • Organize tests into separate TestCases (Add, Get, Validate) for modularity and reuse.
  • A master orchestration script in one TestCase must trigger steps in others.
  • testRunner.runTestStepByName() only works for steps in the current TestCase.
  • Cross-TestCase execution requires a dedicated RunContext via WsdlTestCaseRunner.

What is GraphQL?

  • Created by Facebook in 2012, open-sourced in 2015
  • A query language and server-side runtime — NOT a database
  • Acts as an intermediary layer between frontend and backend data sources (REST APIs, SQL, NoSQL, legacy systems)
  • Clients request exactly the data they need — nothing more, nothing less

GraphQL Explorer (GraphiQL)

GraphiQL (GraphQL + IDE) is a built-in web interface provided by most GraphQL servers. Open the GraphQL endpoint URL in a browser to access it.

  • Left panel: Query editor with syntax highlighting and auto-completion (Ctrl+Space)
  • Right panel: Formatted JSON response viewer
  • Docs panel: Complete schema documentation with types, fields, and arguments
  • Variables panel: Define query variables as JSON below the editor
  • Use Explorer to learn schema before writing Postman tests or automation code

Best Practices

  • Request only needed fields — core benefit of GraphQL; avoid copy-pasting full schemas
  • Use named queries — query GetUserProfile helps debugging and server logs
  • Use variables for all dynamic values — never hardcode IDs in query strings
  • Use fragments when the same field set appears in multiple places
  • Set depth limits on server to prevent expensive deeply nested queries
  • Use DataLoader to batch database calls and solve the N+1 problem
  • Disable introspection in production to prevent schema exposure
  • Always check the errors array in responses — GraphQL returns HTTP 200 even with errors

Common Pitfalls

  1. Requesting too many fields — large responses and slow performance; request only what the UI needs
  2. Not using variables — no query reuse, injection risks, poor caching; always parameterize dynamic values
  3. Forgetting nested field selections — requesting an object field without sub-fields causes validation error
  4. Overusing fragments — makes queries harder to read; use only for genuinely repeated field sets
  5. Not checking schema first — guessing field names wastes time; use Docs/Explorer before writing queries
  6. Assuming GraphQL is always faster — complex queries without DataLoader can be slower than REST
  7. Ignoring error handling — partial data with errors in errors array can crash frontends if not handled

Over-Fetching Deep Dive

Character homepage example: A web series platform homepage needs four pieces of data for one character. The REST approach requires 4 separate API calls:

  1. GET /character/{id} → returns name, status, gender, species, age, origin, and 10+ more fields (need only name & status)
  2. GET /location/{id} → returns name, population, coordinates, creation date (need only name & coordinates)
  3. GET /episode/latest/{charId} → returns full episode object (need only airDate)
  4. GET /episodes/{charId} → returns complete episode list (need only titles or count)

Problem 1 — Over-fetching

  • Each API returns 10–20 fields; the frontend needs 1–2 per call
  • Wastes bandwidth and client-side parsing effort
  • Mobile apps on limited data plans suffer disproportionately

Problem 2 — Performance

  • Four separate network round-trips to load one page
  • Page load time = sum of all four latencies (not parallel in many implementations)
  • Each call adds failure points; one slow endpoint blocks the entire homepage

Problem 3 — Complexity

  • Frontend needs a separate parser for each API response shape
  • Error handling must be implemented four times
  • Coordinating four asynchronous responses adds state management overhead

GraphQL solves this with one POST to /graphql requesting only the four needed field groups in a single query.

Field-Level Authorization

GraphQL exposes a single endpoint, so authorization must happen at the field level, not just the endpoint level:

  • Token validation — Authentication middleware validates JWT/OAuth token before any resolver executes
  • Permissions in resolvers — Each field resolver checks role/permission before returning data (e.g., salary visible only to managers)
  • Schema directives — Use @auth or @requiresRole(role: ADMIN) directives to declaratively protect fields
  • Context propagation — Pass authenticated user context from middleware into every resolver
// Pseudocode — field-level auth in resolver
function resolveSalary(parent, args, context) {
    if (!context.user || context.user.role !== "MANAGER") {
        throw new ForbiddenError("Insufficient permissions for salary field");
    }
    return parent.salary;
}

Query Complexity Limits

Prevent expensive queries that could overload the server:

  • Complexity scoring — Assign cost to each field; sum total complexity before execution
  • Depth limiting — Reject queries nested deeper than N levels (e.g., character → location → residents → location → ...)
  • Reject threshold — Return error if complexity score exceeds limit (e.g., 1000 points)
  • Rate limiting by complexity — Simple queries and expensive queries consume different quota amounts
# Dangerous deep query (should be rejected)
query {
  characterDetails(id: 8) {
    location {
      residents {
        location {
          residents {
            name
          }
        }
      }
    }
  }
}
Testing implication: Include negative tests that send over-complex queries and assert the server returns a GraphQL error with HTTP 200 (GraphQL always returns 200 with errors array for validation failures).

import static io.restassured.RestAssured.*;
import io.restassured.path.json.JsonPath;
import java.io.File;

public class JiraBugTest {

    // Store securely via env vars in real projects
    private static final String BASE64_TOKEN = System.getenv("JIRA_BASE64_TOKEN");

    public static void main(String[] args) {
        baseURI = "https://your-domain.atlassian.net/";

        // Step 1: Create Issue
        String createResponse = given()
            .header("Content-Type", "application/json")
            .header("Authorization", "Basic " + BASE64_TOKEN)
            .body("{\n" +
                  "    \"fields\": {\n" +
                  "        \"project\": {\"key\": \"SCRUM\"},\n" +
                  "        \"summary\": \"Button not working - automation failure\",\n" +
                  "        \"description\": {\n" +
                  "            \"type\": \"doc\",\n" +
                  "            \"version\": 1,\n" +
                  "            \"content\": [{\n" +
                  "                \"type\": \"paragraph\",\n" +
                  "                \"content\": [{\"type\": \"text\", \"text\": \"Test: LoginTest.testSubmit\\nError: Element not clickable\"}]\n" +
                  "            }]\n" +
                  "        },\n" +
                  "        \"issuetype\": {\"name\": \"Bug\"}\n" +
                  "    }\n" +

            .post("rest/api/3/issue")
            .then()
            .assertThat()
            .statusCode(201)
            .extract()
            .response()
            .asString();

        JsonPath js = new JsonPath(createResponse);
        String issueId = js.getString("id");

Token Management Pattern

Centralized token caching with auto-refresh prevents mid-suite 401 failures:

import java.time.Instant;
import static io.restassured.RestAssured.*;

public class TokenManager {
    private static String cachedToken;
    private static Instant expiryTime;

    public static synchronized String getToken() {
        if (cachedToken == null || Instant.now().isAfter(expiryTime)) {
            refreshToken();
        }
        return cachedToken;
    }

    private static void refreshToken() {
        LoginRequest login = new LoginRequest("testuser", "Test@123");
        Response response = given()
            .body(login)
            .post("/auth/login");

        cachedToken = response.jsonPath().getString("accessToken");
        int expiresInSeconds = response.jsonPath().getInt("expiresIn");
        // Refresh 60 seconds before actual expiry
        expiryTime = Instant.now().plusSeconds(expiresInSeconds - 60);
    }
}

// Usage in tests
given()
    .header("Authorization", "Bearer " + TokenManager.getToken())
    .get("/profile");

Validation Strategy (6 Categories)

  1. Status codes: Assert expected HTTP status (200, 201, 400, 401, 404). Most basic but necessary check.
  2. Response structure: Verify required fields exist, correct data types (string, int, array). Use JSON Schema validation.
  3. Data values: Assert business data matches expectations (name, price, email format). Compare against request input for write operations.
  4. Headers: Validate Content-Type, caching headers, Set-Cookie, custom API headers.
  5. Error messages: For 4xx/5xx responses, verify user-friendly error codes and descriptive validation messages.
  6. Performance: Assert response time within SLA: .time(lessThan(2000L))

Best Practices Checklist

  • Never hardcode URLs, credentials, tokens, or dynamic IDs
  • Always validate response body — not just status code
  • Use properties files or environment variables for configuration
  • Generate unique test data (timestamps, UUIDs) to avoid conflicts
  • Cleanup created resources in @AfterMethod (DELETE operations)
  • Test positive AND negative scenarios (400, 401, 403, 404, 405)
  • Chain operations for end-to-end coverage (create → read → update → delete)
  • Version control everything — tests, configs, test data in Git
  • Separate test data from test logic (JSON files, factories, builders)
  • Implement JSON Schema validation for contract testing
  • Log full request/response on test failure (.log().ifValidationFails())
  • Keep tests independent — no shared state between test methods

REST Assured Interview Questions

Q1: What is REST Assured?

Answer: REST Assured is a Java DSL (Domain-Specific Language) library for automating REST API testing. It provides a readable Given-When-Then syntax that makes HTTP requests and validates responses using Hamcrest matchers. It integrates with TestNG/JUnit and supports JSON/XML parsing, authentication, file uploads, and schema validation. It eliminates boilerplate code needed with raw HttpClient or OkHttp.

Q2: Explain Given-When-Then

Answer: This is REST Assured’s BDD-style structure:

  • Given — Setup/preconditions: base URI, headers, query params, request body, authentication
  • When — Action: HTTP method (GET, POST, PUT, DELETE) + endpoint path
  • Then — Validation: status code, response body assertions, headers, response time
given().header("Authorization", "Bearer " + token).body(payload)
.when().post("/users")
.then().statusCode(201).body("id", notNullValue());

Q3: Difference between PUT and PATCH

Answer: PUT performs a full resource replacement — you send the entire entity and missing fields may be set to null/default. PATCH performs a partial update — only changed fields are sent. PUT is idempotent (same request twice = same result); PATCH may not be idempotent depending on implementation. Example: updating a user’s email only → use PATCH; replacing entire user profile → use PUT.

Q4: What is JsonPath?

Answer: JsonPath is a query language for extracting data from JSON documents, similar to XPath for XML. In REST Assured, use it to parse responses and extract values for chaining or assertions.

JsonPath js = new JsonPath(response);
String placeId = js.getString("place_id");
int count = js.getInt("courses.size()");
String title = js.getString("courses[0].title");

Q5: How to perform API chaining?

Answer: Extract data from one API response and pass it to subsequent requests. Store in a variable (local, static, or class field) and use in the next call’s body, query param, or path param.

// Step 1: Create resource
Response addResp = given().body(addPayload).post("/place/add/json");
String placeId = addResp.jsonPath().getString("place_id");

// Step 2: Use extracted ID
given().queryParam("place_id", placeId).get("/place/get/json")
    .then().statusCode(200);

Q6: Serialization vs Deserialization

Answer:

  • Serialization: Java Object → JSON string. Used when sending POJOs as request body. mapper.writeValueAsString(userObject)
  • Deserialization: JSON string → Java Object. Used when mapping API responses to POJOs. mapper.readValue(json, User.class)

REST Assured can auto-serialize/deserialize when you pass POJOs to .body() or use .as(User.class).

Q7: What is TestNG DataProvider?

Answer: A TestNG annotation that returns a 2D Object array. The test method runs once for each inner array, receiving elements as parameters. Enables data-driven testing without duplicating test methods.

@DataProvider(name = "BooksData")
public Object[][] getData() {
    return new Object[][]{{"isbn1","aisle1"}, {"isbn2","aisle2"}};
}
@Test(dataProvider = "BooksData")
public void addBook(String isbn, String aisle) { ... }

Q8: How to upload file with REST Assured?

Answer: Use .multiPart(key, fileObject). REST Assured automatically sets multipart/form-data content type.

given()
    .multiPart("file", new File("C:\\screenshot.png"))
    .multiPart("description", "Test attachment")
.when()
    .post("/upload")
.then()
    .statusCode(200);

Q9: queryParam vs formParam vs pathParam

Answer:

  • queryParam: Appended to URL after ? (e.g., ?place_id=abc&key=123). Used for filtering, pagination.
  • formParam: Sent in request body as application/x-www-form-urlencoded. Used in OAuth token requests, HTML form submissions.
  • pathParam: Replaces {variable} placeholders in URL path (e.g., /users/{userId}). Used to identify specific resources.

Q10: How to validate response headers?

Answer: Use .header() with Hamcrest matchers in the then() block:

.then()
    .header("Content-Type", "application/json")
    .header("server", containsString("Apache"))
    .header("X-RateLimit-Remaining", notNullValue());

Postman Interview Questions

Q1: How pass data between requests?

Answer: In the test script of Request A, extract a value from the response and store it in a variable using pm.environment.set(), pm.collectionVariables.set(), or pm.globals.set(). In Request B, reference it as {{variableName}} in URL, headers, or body.

const jsonData = pm.response.json();
pm.collectionVariables.set("bookId", jsonData.bookId);

Q2: Environment vs Global variables?

Answer: Environment variables are scoped to the currently selected environment (QA, UAT) and change when you switch environments. Global variables are available across all environments and workspaces. Use environments for environment-specific values (baseUrl, API keys); globals for shared constants.

Q3: How validate JSON Schema?

Answer: In the Tests tab, use Postman’s built-in schema validation:

const schema = { /* JSON Schema definition */ };
pm.test("Schema is valid", () => {
    pm.response.to.have.jsonSchema(schema);
});

Or use Ajv library in the test script for custom schema files.

Q4: How handle "already exists" error?

Answer: In the pre-request script, check if the resource exists (GET call). If it exists, delete it first (DELETE call), then proceed with the Add request. This ensures a clean state before each test run.

Q5: Variable scope precedence?

Answer: When the same variable name exists at multiple scopes, Postman resolves in this order (highest priority first): Local → Data → Environment → Collection → Global. Use pm.variables.get("key") to get the resolved value regardless of scope.

Q6: How access request body in test script?

Answer: Use pm.request.body.raw to get the raw request body string. Parse it if JSON: JSON.parse(pm.request.body.raw). Useful for validating that dynamic variables were substituted correctly.

Q7: What is postman.setNextRequest()?

Answer: A Collection Runner function that controls execution flow by jumping to a specific request by name. Used for conditional branching (e.g., if login fails, skip remaining tests and go to cleanup request). Syntax: postman.setNextRequest("Request Name").

Q8: Data-driven testing with 100 datasets?

Answer: Create a CSV or JSON file with 100 rows. In Collection Runner or Newman, attach the data file (-d data.csv). Each row becomes one iteration; column headers map to variable names referenced as {{columnName}} in requests.

OAuth 2.0 Interview Questions

Q1: Authentication vs Authorization

Answer: Authentication verifies who you are (login with credentials). Authorization determines what you can do (permissions to access resources). OAuth 2.0 is an authorization framework — it grants access tokens, not identity verification. OpenID Connect (OIDC) extends OAuth 2.0 to add authentication.

Q2: Authorization Code vs Client Credentials

Answer: Authorization Code grant involves a user login flow — user authenticates, grants permission, app receives a code exchanged for tokens. Used for third-party apps and web apps with backend. Client Credentials grant sends client_id + client_secret directly to the token endpoint with no user interaction. Used for server-to-server (microservices, batch jobs).

Q3: What is the state parameter?

Answer: A random value generated by the client and sent in the authorization request. The authorization server returns the same value in the redirect. The client validates they match to prevent CSRF (Cross-Site Request Forgery) attacks — ensuring the response corresponds to the actual request the client initiated.

Q4: Token expiration handling

Answer: Access tokens expire (typically 30–60 minutes). Use the refresh_token to obtain a new access token without user re-login. Implement token refresh logic: check expiry before each API call, or catch 401 responses and refresh automatically. Store token + expiry timestamp in a TokenManager class.

Q5: Can you fully automate OAuth?

Answer: Client Credentials grant can be fully automated (no browser needed). Authorization Code grant Step 1 requires manual browser login by the user — cannot be fully automated in Postman without workarounds. Steps 2 and 3 (code exchange and API calls) can be automated. For CI/CD, use Client Credentials or pre-generated long-lived tokens in secure vaults.

Q6: What is PKCE?

Answer: Proof Key for Code Exchange — an OAuth 2.0 extension for public clients (mobile/native apps) that cannot securely store client_secret. Uses a code_verifier (random string) and code_challenge (hash of verifier). Prevents authorization code interception attacks. Required for mobile OAuth implementations.

SOAP / SoapUI Interview Questions

Q1: SOAP message structure

Answer: Every SOAP message has an Envelope (mandatory root container), optional Header (metadata, auth tokens), mandatory Body (actual request/response data), and optional Fault element (error details in responses). All content is XML.

Q2: What is WSDL?

Answer: Web Services Description Language — an XML document describing a SOAP service’s operations, parameters, data types, and endpoint URL. It is the contract between provider and consumer. Without WSDL, you cannot know what operations exist or what parameters to send. Obtain it by appending ?wsdl to the service URL.

Q3: SOAP vs REST — when choose each

Answer: Choose SOAP for banking, healthcare, payment gateways, and legacy enterprise systems requiring strict contracts, WS-Security, ACID transactions, and formal WSDL. Choose REST for modern web/mobile apps, microservices, and public APIs where JSON, speed, and flexibility matter. Many organizations use both.

Q4: Contains vs XPath Match assertions

Answer: Contains assertion checks if the response contains a simple string (easy but fragile — breaks if response format changes). XPath Match extracts a value via XPath and compares to an expected value (recommended — precise and maintainable). XPath Match requires namespace declaration for SOAP responses.

Q5: Property Transfer purpose

Answer: A SoapUI feature that extracts a value from one response (via XPath or JSONPath) and injects it into a subsequent request. Enables chaining: Add Employee → extract employeeId → use in Get/Delete requests. Alternative to scripting with Groovy.

Q6: How handle namespaces

Answer: SOAP responses define XML namespaces in the root element. Without registering namespaces, XPath returns nothing. In SoapUI: use “Declare Namespace” in XPath dialog. In Groovy: xml.declareNamespace('ns1':'http://example.com/') before accessing elements.

Q7: What are SOAP Faults?

Answer: SOAP’s built-in error element appearing in the response body when something goes wrong. Contains faultcode, faultstring, and optional detail. Unlike REST (which uses HTTP status codes), SOAP can return HTTP 200 with a Fault in the body. Always check for <fault> element in SOAP responses.

Q8: Mocking vs Stubbing

Answer: Stubbing provides predefined responses to specific requests without verifying call patterns. Mocking additionally verifies that expected calls were made with correct parameters. SoapUI primarily does stubbing (Mock Services). Tools like WireMock and Mockito support full mocking. In API testing, “mocking” is often used loosely to mean stubbing.

GraphQL Interview Questions

Q1: What is GraphQL?

Answer: A query language and server-side runtime (created by Facebook) that allows clients to request exactly the data they need from a single endpoint. It is not a database — it acts as a layer above existing data sources (REST APIs, databases, legacy systems).

Q2: Over-fetching vs under-fetching

Answer: Over-fetching: REST returns more data than needed (20 fields when you need 2). Under-fetching: one REST call doesn’t return enough data, requiring multiple calls. GraphQL solves both by letting clients specify exact fields in one request to one endpoint.

Q3: Main operations

Answer: Three operation types: Query (read data, like GET), Mutation (create/update/delete, like POST/PUT/DELETE), and Subscription (real-time data via WebSockets).

Q4: N+1 problem and solution

Answer: N+1 occurs when a resolver makes a separate database call for each item in a list (1 query for list + N queries for related data = N+1 total). Solution: use DataLoader which batches and caches requests, reducing N+1 calls to 2 calls (one for list, one batched for related data).

Q5: How handle errors?

Answer: GraphQL returns HTTP 200 even when errors occur. Errors appear in an errors array in the response body. Partial data is possible — successful fields return data even if other fields fail. Clients must always check the errors array, not just HTTP status.

Q6: How to version GraphQL?

Answer: GraphQL APIs typically avoid URL versioning (/v1/, /v2/). Instead, add new fields and deprecate old ones using the @deprecated directive. Existing queries continue working because fields are never removed. This is a key difference from REST versioning strategies.

Scenario-Based Interview Questions

1. API returns correct data but takes 10 seconds

Answer: Log a performance bug against the SLA. Add timeout assertions in tests: .time(lessThan(2000L)). Work with dev team to profile the endpoint (slow DB query, missing index, external API call, no caching). Suggest async processing if the operation is inherently long-running. Monitor with APM tools (Datadog, New Relic).

2. Test passes locally but fails on Jenkins

Answer: Check: environment differences (URLs, credentials in properties files), network latency (increase timeouts), authentication token freshness, parallel execution interference (tests modifying shared data), Java/library version mismatches, missing environment variables in Jenkins, and timezone/locale differences affecting date assertions.

3. API returns 200 but error in body

Answer: This is an API design anti-pattern. HTTP status codes should indicate result category (400 for validation errors, 422 for semantic errors). In tests, validate the response body status/error field in addition to HTTP status. Report to dev team. Implement monitoring that checks body-level error indicators, not just HTTP 200.

4. Third-party API is down during testing

Answer: Mock the third-party API using WireMock, SoapUI Mock Services, or recorded responses. Separate integration tests (real service, run nightly) from component tests (mocked, run on every commit). Use contract testing to ensure mocks stay in sync with real API behavior.

5. Manager wants single-click test dashboard

Answer: Set up Jenkins Declarative Pipeline: checkout code → run tests → publish HTML report via publishHTML or TestNG/Allure reports. Add quality gate (fail build if pass rate < 95%). Integrate Allure or ExtentReports for rich dashboards with trends, categories, and failure details.

6. API team changed response structure breaking tests

Answer: Implement JSON Schema validation in every build to catch breaking changes early. Use versioned APIs (/v1/, /v2/). Encourage consumer-driven contracts (Pact). Design tests to validate structure via schema rather than hardcoding every field path. Maintain backward-compatible test utilities.

7. 401 vs 403 difference

Answer: 401 Unauthorized = “Who are you?” — authentication failed (missing/invalid/expired credentials). 403 Forbidden = “I know who you are, but you cannot do this” — authenticated but lacks permission for the resource/action.

8. Can POST be idempotent?

Answer: By HTTP specification, POST is not idempotent — each call may create a new resource with a different ID. However, some APIs implement idempotency keys (send same key → same result). PUT and DELETE are idempotent. GET is safe and idempotent.

9. How test rate limiting?

Answer: Write a test that sends many requests in a loop (multi-threaded). Assert that after a threshold, the API returns 429 Too Many Requests. Verify the Retry-After header. Test that after waiting, requests succeed again. Document the rate limit for the team.

10. How test undocumented API?

Answer: Use browser DevTools Network tab to capture requests while using the application. Import into Postman via Import → Raw text. Try common CRUD patterns. Check for Swagger/OpenAPI at /swagger-ui or /api-docs. Analyze error messages for parameter hints. Ask developers for examples or Postman collections.

Framework Interview Questions

Q1: Explain your API automation framework architecture

Answer: We use REST Assured + TestNG + Maven + ExtentReports + Log4j2. The framework is layered: configuration layer (property files per environment), client layer (wrapper classes around REST Assured), POJO layer (request/response models), test data layer (JSON files and factories), and test layer (TestNG classes organized by smoke/regression/integration). TokenManager handles auth. Jenkins runs smoke on every commit and full regression nightly.

Q2: How handle multiple environments?

Answer: Externalized configuration via properties files (dev.properties, qa.properties). ConfigManager loads the appropriate file based on a system property (-Denvironment=qa). In Jenkins, use Choice Parameter to select environment. All URLs, credentials, and test data paths come from config — zero hardcoding.

Q3: How handle token expiration in long suites?

Answer: TokenManager class stores token + expiry timestamp. Before each test (or in @BeforeMethod), check if token is still valid. If expired or within 60 seconds of expiry, call login endpoint to refresh. All tests call TokenManager.getToken() instead of managing tokens individually.

Q4: API dependencies — one response needed for another?

Answer: Extract values from the first response using JsonPath and pass to subsequent requests via variables. In REST Assured: store in local variables or class fields. In Postman: pm.collectionVariables.set(). In SoapUI: Property Transfer or Groovy scripts. Design E2E test methods that chain operations within a single test rather than across separate tests.

Q5: How handle flaky API tests?

Answer: Implement TestNG IRetryAnalyzer for automatic retry (max 2–3 attempts). Add detailed logging on failure (.log().ifValidationFails()). Use explicit waits/polling for async APIs. Ensure tests are independent with their own data. In Jenkins, archive logs as build artifacts. Track flaky test metrics and fix root causes (timing, shared data, external dependencies).
