Section 15: Reporting in Jenkins

As an automation engineer, your tests are only as valuable as the reports stakeholders can read. Jenkins reporting turns raw test execution into actionable dashboards, trend graphs, and shareable artifacts that QA leads, developers, and managers rely on daily.

TestNG Default Reports

When you run TestNG via Maven Surefire, reports are auto-generated without any Jenkins plugin configuration.

test-output/ Folder

After every TestNG run, Maven creates target/surefire-reports/ (JUnit XML) and TestNG creates test-output/ with HTML files. Key files:

  • index.html — main dashboard with suite summary
  • emailable-report.html — lightweight summary designed for email sharing
  • testng-results.xml — machine-readable results for Jenkins plugins
  • testng-failed.xml — failed test methods for re-run
Automation Engineer Tip: Always configure Surefire to output JUnit XML format. Jenkins natively understands JUnit XML via the JUnit plugin — this is your baseline reporting layer even before Allure or Extent.

emailable-report.html

TestNG generates this automatically. Share it via Jenkins Email Extension Plugin using attachmentsPattern: 'test-output/emailable-report.html'. It contains pass/fail counts, duration, and grouped results — perfect for quick stakeholder updates.

Jenkins TestNG Results Plugin

Install TestNG Results Plugin from Manage Jenkins → Plugins. Configure as a post-build action:

  1. Go to job configuration → Post-build Actions
  2. Add Publish TestNG Results
  3. Set report path: test-output/testng-results.xml or **/testng-results.xml

Benefits for automation engineers:

  • Trend graphs showing pass/fail history over builds
  • Test result diff between builds (new failures, fixed tests)
  • Drill-down into failed test methods with stack traces
  • Build stability metrics for flaky test identification
// Declarative Pipeline — TestNG Results
post {
    always {
        step([$class: 'Publisher',
              reportFilenamePattern: '**/testng-results.xml'])

Report Archiving Strategy

Automation engineers must define retention policies so Jenkins disk does not fill up while keeping enough history for trend analysis.

Keep Last N Builds

// Jenkinsfile — discard old builds
options {
    buildDiscarder(logRotator(
        numToKeepStr: '30',           // keep last 30 builds
        artifactNumToKeepStr: '10'    // keep artifacts for last 10
    ))
}

Archive Artifacts

post {
    always {
        archiveArtifacts artifacts: '''
            target/allure-results/**,
            reports/ExtentReport.html,
            test-output/emailable-report.html,
            screenshots/**,
            logs/**
        ''', allowEmptyArchive: true, fingerprint: true

        junit '**/surefire-reports/*.xml'
    }
}

Recommended Strategy for Automation Projects

  • Allure raw results — archived per build; Allure plugin generates report on-the-fly
  • Extent HTML — archive + HTML Publisher for inline viewing
  • Screenshots/logs — archive on failure only to save disk
  • External storage — push reports to S3/Azure Blob for long-term retention beyond Jenkins
Production Pattern: Use Allure for Jenkins dashboard trends + archive screenshots/logs as artifacts. Send emailable-report.html or Allure summary link via email on failure. This gives managers visibility without bloating Jenkins storage.

Interview Questions

  1. Q: What is the difference between TestNG's emailable-report.html and Allure report?
    A: emailable-report is a static HTML summary auto-generated by TestNG with basic pass/fail tables. Allure is a rich interactive report with steps, attachments, severity, trends, and categories — requires framework integration and the Allure Jenkins plugin.
  2. Q: How do you publish Allure reports in a Jenkins pipeline?
    A: Install Allure Commandline in Global Tool Configuration, add allure-testng dependency, generate results in target/allure-results during tests, then use the allure pipeline step with the results path.
  3. Q: What file format does Jenkins JUnit plugin consume?
    A: JUnit XML format. TestNG's testng-results.xml and Surefire's TEST-*.xml both work with the junit pipeline step.
  4. Q: How do you attach screenshots to Extent Reports on test failure?
    A: Implement an ITestListener, capture screenshot in onTestFailure(), and call test.addScreenCaptureFromPath(path) before flushing the report.
  5. Q: What is the purpose of the Cucumber JSON report?
    A: It is a machine-readable report consumed by the Jenkins Cucumber Reports plugin to render feature/scenario dashboards with step-level failure details and trend graphs.
  6. Q: How do you prevent Jenkins disk from filling up with reports?
    A: Configure buildDiscarder to keep last N builds, limit artifact retention, archive screenshots only on failure, and optionally offload long-term storage to S3 or similar.
  7. Q: Can you use multiple reporting tools in the same project?
    A: Yes. Common pattern: Allure for detailed Jenkins dashboard + Extent for standalone HTML + JUnit XML for Jenkins test result trends. They serve different audiences.
  8. Q: What Allure annotations do automation engineers use most?
    A: @Epic, @Feature (organization), @Severity (priority), @Description (test purpose), @Step (granular step logging in page objects).
  9. Q: Why is publishHTML allowMissing set to false?
    A: To fail the build if the report was not generated — this catches reporting pipeline breaks instead of silently showing empty report links.
  10. Q: How do you share test reports with non-Jenkins stakeholders?
    A: Email emailable-report.html as attachment, share archived Extent HTML via shared drive, or send Allure report zip exported from Jenkins build artifacts.

Notification Strategy for Automation Engineers

Best Practice: Route notifications by severity. Smoke test failure → Slack #qa-alerts immediately. Nightly regression failure → email QA lead + Slack. Success → Slack summary only (avoid email noise). Production deployment gate failure → email + Teams + PagerDuty.
  • Do not email on every success — causes alert fatigue; use Slack or weekly digest
  • Always include BUILD_URL — one-click access to console and reports
  • Attach reports on failure only — keeps email size manageable
  • Use parameterized recipient lists — different emails per environment (QA vs prod)

Interview Questions

  1. Q: What is the difference between default Jenkins email and Email Extension Plugin?
    A: Default email sends plain text on fixed triggers with limited customization. Email-ext supports HTML bodies, attachments, per-trigger templates, recipient tokens, and pipeline emailext step with rich content.
  2. Q: How do you attach test reports to Jenkins notification emails?
    A: Use attachmentsPattern in emailext step, e.g., attachmentsPattern: 'reports/*.html, test-output/emailable-report.html'. Files must exist in workspace at post-build time.
  3. Q: What SMTP settings are required for Gmail in Jenkins?
    A: smtp.gmail.com, port 465, SSL enabled, valid username, and Google App Password (not regular password if 2FA is on).
  4. Q: When would you use Slack instead of email?
    A: For real-time team alerts where immediate visibility matters — failed smoke tests, broken CI pipeline. Slack reduces response time; email suits formal stakeholder reporting.
  5. Q: How do you send different emails for success vs failure in a pipeline?
    A: Use post { success { emailext(...) } failure { emailext(...) } unstable { emailext(...) } } blocks with different subjects, recipients, and bodies.
  6. Q: What is an unstable build in Jenkins and how do you notify for it?
    A: Unstable means build completed but tests failed (or quality gate degraded). Use the unstable post condition with emailext to alert QA with test report links.
  7. Q: How do you configure Teams notifications without a dedicated plugin?
    A: Create an Incoming Webhook in Teams channel, then POST a MessageCard JSON payload via httpRequest step or use Office 365 Connector plugin.
  8. Q: How do you prevent notification spam from frequent builds?
    A: Email only on failure/unstable, use Slack for success summaries, configure fix-failure-only triggers, and set quiet period on jobs triggered by every commit.
  9. Q: What environment variables are commonly used in email body templates?
    A: JOB_NAME, BUILD_NUMBER, BUILD_URL, BUILD_URL/console, BRANCH_NAME, currentBuild.durationString, currentBuild.result.

Why Parameterize Pipelines?

Parameter Best Practices for Automation

  • Provide sensible defaultValue — job should work with zero input on first run
  • Add description to every parameter — self-documenting for QA self-service
  • Validate dangerous combos (prod + full regression) with input step
  • Never store secrets in string parameters — use Jenkins Credentials binding
  • Log parameter values at build start for debugging: echo "Params: ${params}"
  • Keep parameter count manageable — more than 8 parameters suggests job should be split

Interview Questions

  1. Q: Why parameterize Jenkins pipelines instead of creating separate jobs?
    A: One pipeline reduces maintenance — same Jenkinsfile, same plugins, same reporting. Parameters let teams run different configs without duplicating jobs or changing code.
  2. Q: What is the difference between params.X and env.X?
    A: params.X are user-supplied build parameters. env.X are environment variables (built-in or custom). Parameters must be explicitly mapped to env vars if needed in shell export context.
  3. Q: How do you run a parameterized pipeline manually?
    A: Click "Build with Parameters" on the job page. Jenkins renders the parameter form defined in the parameters block.
  4. Q: Can you add new parameters to an existing pipeline without losing history?
    A: Yes. Add parameters block to Jenkinsfile. First run after change may require "Build with Parameters" to initialize defaults. Existing build history is preserved.
  5. Q: How do you pass Cucumber tags as a parameter?
    A: Define string parameter TAGS with default '@smoke', pass to Maven: -Dcucumber.filter.tags="${params.TAGS}".
  6. Q: What is Active Choices Plugin used for?
    A: Dynamic parameter values that change based on other parameter selections — e.g., suite list changes when environment changes.
  7. Q: How do you share data between pipeline stages on different agents?
    A: Use stash/unstash for files, or store values in env variables (master only), or use external storage (S3, database).
  8. Q: How do you prevent accidental production test execution?
    A: Restrict prod in choice parameter, add input confirmation step when ENV=prod, use Jenkins RBAC to limit who can select prod.
  9. Q: What happens if you reference params before parameters block is defined?
    A: On first job run, parameters may be null until "Build with Parameters" initializes them. Use defaultValue in parameter definition to avoid null issues.
  10. Q: How do booleanParam values appear in Maven command line?
    A: As string 'true' or 'false' — pass directly: -Dheadless=${params.HEADLESS}. Java/TestNG reads as boolean system property.
  11. Q: Can file parameters be used in pipeline?
    A: Yes. Uploaded file lands in workspace. Reference path in test framework config. Useful for ad-hoc test data uploads by QA.

Complete Production-Grade Jenkinsfile

pipeline {
    agent any

    tools {
        maven 'Maven-3.9'
        jdk 'JDK-17'
    }

    parameters {
        choice(name: 'ENV', choices: ['qa', 'staging'], description: 'Target environment')
        choice(name: 'SUITE', choices: ['smoke', 'regression', 'full'], description: 'Test suite')
        choice(name: 'BROWSER', choices: ['chrome', 'firefox'], description: 'Primary browser')
        booleanParam(name: 'HEADLESS', defaultValue: true, description: 'Headless mode')
        string(name: 'TAGS', defaultValue: '@smoke', description: 'Cucumber tags (optional)')
    }

    environment {
        REPORT_DIR = 'target/reports'
        ALLURE_RESULTS = 'target/allure-results'
        TEST_ENV = "${params.ENV}"
    }

    options {
        buildDiscarder(logRotator(numToKeepStr: '30', artifactNumToKeepStr: '10'))
        timeout(time: 120, unit: 'MINUTES')
        timestamps()
        disableConcurrentBuilds()
    }

    triggers {
        // Nightly regression at ~2 AM
        cron('H 2 * * *')
        // Poll SCM fallback (prefer webhook)
        // pollSCM('H/15 * * * *')
    }

    stages {
        stage('Checkout') {
            steps {
                checkout scm
                script {
                    echo "Building ${env.JOB_NAME} #${env.BUILD_NUMBER} on ${env.NODE_NAME}"
                    echo "Parameters: ENV=${params.ENV}, SUITE=${params.SUITE}, BROWSER=${params.BROWSER}"
                }
            }
        }

        stage('Build') {
            steps {
                bat 'mvn clean compile -DskipTests'
            }
        }

        stage('Unit Tests') {
            steps {
                bat 'mvn test -Dsurefire.suiteXmlFiles=unit-tests.xml'
            }
            post {
                always {
                    junit '**/surefire-reports/TEST-*.xml'
                }
            }
        }

        stage('Smoke Tests') {
            when {
                expression { params.SUITE != 'regression' }
            }
            steps {
                bat """
                    mvn test ^
                      -Dsurefire.suiteXmlFiles=smoke.xml ^
                      -Dbrowser=${params.BROWSER} ^
                      -Dheadless=${params.HEADLESS} ^
                      -Denv=${params.ENV}
                """
            }
        }

        stage('Parallel Browser Tests') {
            when {
                expression { params.SUITE == 'full' }
            }
            parallel {
                stage('Chrome Regression') {
                    steps {
                        bat 'mvn test -Dbrowser=chrome -Dheadless=true -Dsurefire.suiteXmlFiles=regression.xml -Denv=${params.ENV}'
                    }
                }
                stage('Firefox Regression') {
                    steps {
                        bat 'mvn test -Dbrowser=firefox -Dheadless=true -Dsurefire.suiteXmlFiles=regression.xml -Denv=${params.ENV}'
                    }
                }
            }
        }

        stage('Regression') {
            when {
                expression { params.SUITE == 'regression' }
            }
            steps {
                bat """
                    mvn test ^
                      -Dsurefire.suiteXmlFiles=regression.xml ^
                      -Dbrowser=${params.BROWSER} ^
                      -Dheadless=${params.HEADLESS} ^
                      -Denv=${params.ENV}
                """
            }
        }

        stage('API Tests') {
            when {
                expression { params.SUITE == 'full' || params.SUITE == 'regression' }
            }
            steps {
                bat "mvn test -Dsurefire.suiteXmlFiles=api.xml -Denv=${params.ENV}"
            }
        }

        stage('Cucumber BDD') {
            when {
                expression { params.TAGS?.trim() && params.SUITE != 'smoke' }
            }
            steps {
                bat "mvn test -Dcucumber.filter.tags=\"${params.TAGS}\" -Denv=${params.ENV}"
            }
        }

        stage('Reports') {
            steps {
                allure includeProperties: false,
                      jdk: '',
                      results: [[path: "${ALLURE_RESULTS}"]]
                publishHTML([
                    allowMissing: false,
                    alwaysLinkToLastBuild: true,
                    keepAll: true,
                    reportDir: "${REPORT_DIR}",
                    reportFiles: 'ExtentReport.html',
                    reportName: 'Test Report'
                ])
            }
        }

        stage('Deploy to QA') {
            when {
                allOf {
                    branch 'main'
                    expression { params.ENV == 'qa' }
                    expression { currentBuild.result == null || currentBuild.result == 'SUCCESS' }
                }
            }
            steps {
                input message: 'Deploy to QA environment?', ok: 'Deploy'
                bat 'mvn deploy -DskipTests -Denv=qa'
                echo "Deployed build ${env.BUILD_NUMBER} to QA"
            }
        }
    }

    post {
        always {
            junit '**/surefire-reports/*.xml'
            archiveArtifacts artifacts: 'screenshots/**, logs/**', allowEmptyArchive: true
            cleanWs(cleanWhenNotBuilt: false, deleteDirs: true, disableDeferredWipeout: true)
        }
        success {
            emailext(
                subject: "PASSED: ${env.JOB_NAME} #${env.BUILD_NUMBER}",
                body: """<h2>Build Passed</h2>
                         <p>Env: ${params.ENV} | Suite: ${params.SUITE}</p>
                         <p><a href='${env.BUILD_URL}allure'>Allure Report</a></p>""",
                to: 'team@company.com',
                mimeType: 'text/html'
            )
        }
        failure {
            emailext(
                subject: "FAILED: ${env.JOB_NAME} #${env.BUILD_NUMBER}",
                body: """<h2>Build Failed</h2>
                         <p><a href='${env.BUILD_URL}console'>Console Log</a></p>""",
                to: 'team@company.com',
                mimeType: 'text/html',
                attachmentsPattern: 'test-output/emailable-report.html'
            )
            slackSend channel: '#qa-alerts', color: 'danger',
                      message: "Build Failed: ${env.JOB_NAME} #${env.BUILD_NUMBER}"
        }
        unstable {
            slackSend channel: '#qa-alerts', color: 'warning',
                      message: "Tests Failed: ${env.JOB_NAME} #${env.BUILD_NUMBER} — <${env.BUILD_URL}testReport|Report>"
        }
    }
}

Pipeline Design Principles for Automation Engineers

  • Fail fast — smoke before regression; compile before tests
  • Parallelize wisely — browser parallel for full suite; avoid overloading agents
  • Conditional stages — use when {} to skip irrelevant stages per parameter
  • Always publish results — post { always { junit ... } } even on failure
  • Clean workspace — prevent stale artifacts causing false passes
  • Timeout everything — hung Selenium tests should not block agents for hours
  • Human gates for prod — input step before production deploy or test

Interview Questions

  1. Q: Describe a production-grade CI pipeline for Selenium automation.
    A: Webhook-triggered checkout → compile → unit tests → smoke → parallel browser regression → API tests → Allure/Extent reports → JUnit publish → email/Slack on failure → optional deploy with input gate.
  2. Q: How do you run Chrome and Firefox tests in parallel in Jenkins?
    A: Use parallel {} block with separate stages, each running mvn test with -Dbrowser=chrome or firefox. Ensure sufficient agent executors.
  3. Q: What is the purpose of the when block in declarative pipeline?
    A: Conditionally execute stages based on branch, parameters, build result, or custom expression — e.g., skip smoke when SUITE=regression.
  4. Q: How do you configure GitHub webhook to trigger Jenkins?
    A: Add webhook URL https://jenkins/github-webhook/ in GitHub repo settings. Enable "GitHub hook trigger" in Jenkins job Build Triggers.
  5. Q: What does cron('H 2 * * *') mean?
    A: Run approximately at 2 AM daily. H hash spreads load across the hour to prevent all jobs firing simultaneously.
  6. Q: What are Jenkins shared libraries and why use them?
    A: Reusable Groovy code (vars/ and src/) shared across pipelines — DRY principle for reporting, notifications, and test execution steps.
  7. Q: How do you prevent concurrent builds from conflicting?
    A: options { disableConcurrentBuilds() } or throttle concurrent builds per agent label.
  8. Q: What goes in post { always {} } vs post { failure {} }?
    A: always runs regardless of result — JUnit publish, artifact archive, workspace cleanup. failure runs only on failure — alert emails, Slack.
  9. Q: How do you add a manual approval before deployment?
    A: Use input step: input message: 'Deploy?', ok: 'Proceed' — pipeline pauses until authorized user approves.
  10. Q: Why use buildDiscarder in options?
    A: Automatically delete old builds and artifacts to prevent Jenkins disk exhaustion while keeping recent history for trends.
  11. Q: How do you handle different test suites from one pipeline?
    A: Parameterize SUITE choice and use when {} blocks or dynamic surefire.suiteXmlFiles=${params.SUITE}.xml.

    A: Wipe workspace, configure buildDiscarder, delete old artifacts, archive to external storage, and monitor agent disk capacity.
  3. Q: Jenkins agent is offline — what steps do you take?
    A: Check agent log, verify network connectivity from master, confirm Java installed on agent, restart agent service, regenerate SSH/JNLP credentials if expired.
  4. Q: How do you investigate Maven dependency download failures?
    A: Run mvn dependency:tree, check .m2/repository for corrupt jars, verify proxy/settings.xml, test connectivity to Maven Central or internal Nexus.
  5. Q: What causes Jenkins OutOfMemoryError and how do you fix it?
    A: Default JVM heap too small for workload. Increase JAVA_OPTS -Xmx, limit parallel builds, tune Surefire forkCount, and investigate memory leaks via GC logs.
  6. Q: Git checkout fails with authentication error — what do you check?
    A: Jenkins Credentials for expired token/password, SSH key validity, repo URL correctness, and service account permissions on the repository.
  7. Q: Where do you look first when a Jenkins build fails?

  8. Q: How do you handle plugin incompatibility after Jenkins upgrade?

  9. Q: How do you prevent workspace-related failures?

  10. Q: What is In-process Script Approval in Jenkins?
    A: Security sandbox that blocks unapproved Groovy methods in pipelines. Admin must approve scripts at Manage Jenkins → In-process Script Approval when pipelines use restricted APIs.

What is Docker?

Interview Questions

  1. Q: What is the difference between a Docker image and container?
    A: Image is a read-only template (blueprint). Container is a running instance of that image. You build an image once, run many containers from it.
  2. Q: Why do automation engineers use Docker?
    A: Consistent test environments across local, CI, and staging. Eliminates version mismatches for Java, browsers, and drivers. Enables Selenium Grid and clean CI agents per build.
  3. Q: How do you run Selenium Grid with Docker?
    A: docker-compose with selenium/hub and selenium/node-chrome/node-firefox services. Connect tests via webdriver.remote.url=http://localhost:4444.
  4. Q: What is shm_size and why is it needed for Chrome in Docker?
    A: Shared memory allocation. Chrome requires significant shared memory; default 64MB causes crashes. Set shm_size: '2gb' in docker-compose for Chrome nodes.
  5. Q: How do you persist test reports from a Docker container?
    A: Volume mount: docker run -v $(pwd)/reports:/app/reports. Reports written inside container appear on host filesystem.
  6. Q: How do you use Docker as a Jenkins pipeline agent?
    A: agent { docker { image 'maven:3.9-eclipse-temurin-17' } } — Jenkins starts container, runs steps inside, destroys after build.
  7. Q: What is the purpose of COPY pom.xml before COPY src in Dockerfile?
    A: Docker layer caching. Dependencies resolve in a separate layer that only rebuilds when pom.xml changes — much faster subsequent builds.
  8. Q: docker run vs docker-compose up — when to use each?
    A: docker run for single containers (one-off test runs). docker-compose for multi-container setups (Selenium Grid hub + nodes).
  9. Q: How do you debug a failing test inside a Docker container?
    A: docker exec -it container-name bash to get shell access; docker logs container-name for output; run with interactive mode and override CMD.
  10. Q: What is docker-compose scale used for in Selenium Grid?
    A: Run multiple browser node instances: docker-compose up --scale chrome=3 — scales parallel browser capacity for faster test execution.
  11. Q: Why pin Docker image versions instead of using :latest?
    A: :latest changes unpredictably — browser version drift causes test failures. Pinning ensures reproducible builds across time and environments.

    A: kubectl get pods (check STATUS), kubectl logs (app errors), kubectl describe pod (Events). If pods healthy and logs clean, it's likely a test or app logic issue.

    A: Basic: kubectl get/describe/logs/exec, understand pod/deployment/service/namespace concepts. Not expected: cluster admin, writing operators, network policy configuration.

Common Mistakes

  • Running tests against pod IP instead of Service URL — pod restarts change IP
  • Ignoring CrashLoopBackOff and re-running tests hoping they pass
  • Not specifying namespace (-n qa) — checking wrong environment's pods
  • Assuming test failure is always application bug — missing K8s infrastructure check
  • Not checking application logs before escalating test failures to developers
  • Over-provisioning Grid nodes without KEDA — paying for idle browser pods 24/7

Troubleshooting Tips

  • Pod CrashLoopBackOff: kubectl logs pod --previous for crash reason; check describe Events
  • ImagePullBackOff: Wrong image tag or missing registry credentials — check deployment YAML
  • OOMKilled: Pod exceeded memory limit — increase resources.limits.memory in deployment
  • Tests timeout connecting to app: Verify service exists (kubectl get svc); check Ingress rules
  • Grid sessions queue indefinitely: kubectl get pods for grid nodes; scale up if pending
  • Local access to cluster app: kubectl port-forward svc/app-service 8080:80 -n qa
