Section 8: Jenkins Jobs

As an automation engineer, Jenkins jobs are the entry point where your TestNG suites, Cucumber features, and Selenium/Playwright tests get scheduled and executed. Understanding job types helps you choose the right approach for test automation—not just "how to deploy apps."

Freestyle Project

A Freestyle job is the classic Jenkins job type configured entirely through the UI. Most automation teams start here before migrating to Pipeline as Code.

Configuration Areas

  • Source Code Management (Git): Connect your automation repo URL, credentials, and branch (e.g., develop or main).
  • Build Triggers:
    • SCM polling — Jenkins checks Git every N minutes for changes (H/5 * * * * = every 5 min).
    • Webhook — GitHub/GitLab/Bitbucket pushes trigger builds instantly (preferred for PR validation).
    • Cron — Schedule nightly regression (0 2 * * * = 2 AM daily).
  • Build Steps (Maven): Execute mvn clean test -Dsurefire.suiteXmlFiles=smoke.xml or invoke a batch/shell script.
  • Post-build Actions: Publish TestNG/HTML reports, archive screenshots, send email on failure, mark build unstable on test failure.

Step-by-Step: Create Freestyle Job for TestNG Suite

  1. Jenkins Dashboard → New Item → enter name automation-smoke-tests → select Freestyle project → OK.
  2. General: Add description: "Runs smoke TestNG suite on every push to develop."
  3. Source Code Management: Select Git → Repository URL → add credentials → Branch Specifier: */develop.
  4. Build Triggers: Check "GitHub hook trigger for GITScm polling" (after configuring webhook in GitHub).
  5. Build Environment: Optionally add environment variables: BROWSER=chrome, ENV=qa.
  6. Build Steps: Add "Invoke top-level Maven targets" → Goals: clean test -Dsurefire.suiteXmlFiles=src/test/resources/smoke.xml -Dbrowser=chrome.
  7. Post-build Actions:
    • Publish JUnit test result report → target/surefire-reports/*.xml
    • Publish HTML reports (HTML Publisher Plugin) → report dir: target/surefire-reports
    • Editable Email Notification → send on failure to qa-team@company.com
  8. Save → Build Now → verify console output shows TestNG execution.
Automation Engineer Tip: Freestyle jobs are fine for a single smoke suite on one branch. Once you need parameterized browser selection, parallel stages, or different suites per branch, move to Pipeline.

Advantages

  • Easy setup—no Groovy knowledge required
  • UI-based configuration; quick for POC and demos
  • Works well for one-off scheduled regression runs

Limitations

  • Configuration is not version-controlled—changes live only in Jenkins
  • Hard to maintain across multiple environments (QA, staging, prod)
  • No pipeline logic: no parallel browser runs, conditional stages, or approval gates
  • Difficult to replicate across Jenkins instances (dev Jenkins vs prod Jenkins)

Pipeline Project

Pipeline projects use a Jenkinsfile stored in your automation Git repo. The Jenkinsfile defines checkout, build, test execution, reporting, and notifications as code.

Why Pipeline for Test Automation?

  • Version-controlled: Jenkinsfile changes go through PR review like test code
  • Scriptable: Parameterized browser, environment, suite selection
  • Reusable: Shared libraries for common test steps
  • Auditable: Git history shows who changed the pipeline and when

When to Use Pipeline Over Freestyle

  • Multiple test stages (smoke → regression → report)
  • Parameterized execution (browser, environment, suite)
  • Parallel browser execution (Chrome + Firefox simultaneously)
  • PR validation with branch-specific behavior
  • Team needs pipeline config reviewed in Git PRs

Multibranch Pipeline

Multibranch Pipeline automatically discovers branches and PRs in your Git repo and creates a Jenkins job for each one. Each branch uses its own Jenkinsfile.

Key Features for QA

  • Auto-discovers branches — feature branches get their own pipeline runs
  • Jenkinsfile per branch — feature branch can run smoke only; develop runs regression
  • PR validation: Auto-trigger on PR creation/update; block merge if smoke fails
  • Branch-specific suites:
    • feature/* → smoke suite (~5 min)
    • develop → smoke + API regression (~20 min)
    • release/* → full regression + cross-browser (~2 hrs)
Setup: New Item → Multibranch Pipeline → Branch Sources → add Git repo → Build Configuration: "by Jenkinsfile" → Scan Multibranch Pipeline Triggers: "Periodically if not otherwise run" or webhook.

Interview Questions

  1. Q: What is the difference between Freestyle and Pipeline jobs?
    A: Freestyle is UI-configured with no version control; Pipeline uses a Jenkinsfile in Git with stages, parameters, and parallel execution. Pipeline is preferred for maintainable test automation.
  2. Q: How do you trigger a Jenkins job on Git push?
    A: Configure a webhook in Git (GitHub/GitLab) pointing to Jenkins, enable "GitHub hook trigger" in the job, or use SCM polling as fallback.
  3. Q: What is a Multibranch Pipeline and why use it for QA?
    A: It auto-creates jobs per branch/PR. QA uses it to run smoke on feature branches and full regression on develop/release without manual job creation.
  4. Q: How do you run a specific TestNG suite in a Freestyle job?
    A: Maven build step: mvn test -Dsurefire.suiteXmlFiles=smoke.xml. Post-build: publish JUnit results from target/surefire-reports/*.xml.
  5. Q: What are limitations of Freestyle jobs for automation teams?
    A: Not version-controlled, no parallel stages, hard to parameterize, difficult to replicate across environments, no PR branch auto-discovery.
  6. Q: When would you choose Scripted over Declarative Pipeline?
    A: When you need dynamic stage generation (e.g., loop over 10 environments), complex Groovy logic, or non-standard flow that Declarative syntax cannot express.
  7. Q: How do you organize multiple test jobs in Jenkins?
    A: Use Folders (Smoke, Regression, Nightly) or a single Pipeline with parameters for suite selection. Multibranch for branch-based execution.
  8. Q: What post-build actions are essential for test jobs?
    A: Publish JUnit/TestNG results, archive HTML reports and failure screenshots, email notification on failure, optionally mark unstable vs failed.

Real Project Example

An e-commerce automation team started with 3 Freestyle jobs (smoke, regression, nightly). Maintenance became painful when they added 4 environments and 3 browsers. They migrated to a Multibranch Pipeline: PRs triggered smoke (10 tests, 3 min), merges to develop ran API + UI regression (150 tests, 25 min), and nightly cron ran full cross-browser suite. Jenkinsfile lived in the automation repo; QA reviewed pipeline changes alongside test code.

Common Mistakes

  • Hardcoding credentials in Freestyle job config instead of Jenkins Credentials
  • Using SCM polling (every minute) instead of webhooks—wastes resources and delays feedback
  • Not publishing test results—team only sees "BUILD FAILURE" without knowing which tests failed
  • Creating separate Freestyle jobs per browser instead of parameterized Pipeline
  • Forgetting to archive screenshots/logs on failure—impossible to debug CI-only failures

Troubleshooting Tips

  • Job not triggering on push: Verify webhook delivery in GitHub → Settings → Webhooks; check Jenkins URL is reachable from Git server.
  • Git checkout fails: Verify credentials ID in job matches Jenkins Credentials; check branch name pattern.
  • Maven not found: Configure Maven in Global Tool Configuration; reference in Pipeline with tools { maven 'Maven-3.9' }.
  • Tests pass locally but fail in Jenkins: Check env vars, headless browser config, and different JDK/Maven versions.
  • Multibranch not discovering branches: Run "Scan Multibranch Pipeline Now"; verify Jenkinsfile exists on branch root.

Quick Revision Notes

  • Freestyle = UI config, no version control, good for POC
  • Pipeline = Jenkinsfile in Git, parameterized, parallel, reusable
  • Multibranch = auto branch/PR jobs, ideal for PR smoke validation
  • Triggers: webhook (best) > SCM polling > cron (nightly)
  • Always publish JUnit results + archive reports/screenshots

Section 9: Jenkins Pipelines — Deep Dive

Pipelines are how automation engineers define test execution as code. Your Jenkinsfile is as important as your Page Object Model—it's the orchestration layer for every test run in CI/CD.

Pipeline as Code

Store the Jenkinsfile in your project root (same repo as tests). Jenkins checks out the repo and executes the Jenkinsfile. Changes to pipeline logic go through the same PR review process as test code.

Convention: Place Jenkinsfile at repo root. For Multibranch, Jenkins auto-detects it. For standalone Pipeline job, specify "Pipeline script from SCM."

Declarative Pipeline — Full Example

pipeline {
    agent any
    tools { maven 'Maven-3.9' }
    parameters {
        choice(name: 'BROWSER', choices: ['chrome', 'firefox', 'edge'], description: 'Browser')
        choice(name: 'ENV', choices: ['qa', 'staging', 'prod'], description: 'Environment')
        choice(name: 'SUITE', choices: ['smoke', 'regression', 'full'], description: 'Test Suite')
    }
    stages {
        stage('Checkout') {
            steps { git branch: 'main', url: 'https://github.com/team/automation.git' }
        }
        stage('Build') {
            steps { bat 'mvn clean compile' }
        }
        stage('Execute Tests') {
            steps {
                bat "mvn test -Dbrowser=${params.BROWSER} -Denv=${params.ENV} -Dsurefire.suiteXmlFiles=${params.SUITE}.xml"
            }
        }
        stage('Generate Reports') {
            steps {
                publishHTML([
                    allowMissing: false,
                    alwaysLinkToLastBuild: true,
                    keepAll: true,
                    reportDir: 'target/surefire-reports',
                    reportFiles: 'index.html',
                    reportName: 'Test Report'
                ])
            }
        }
    }
    post {
        always {
            archiveArtifacts artifacts: 'target/**', allowEmptyArchive: true
        }
        success {
            emailext subject: 'Build Passed',
                     body: 'All tests passed for ${params.SUITE} on ${params.ENV}.',
                     to: 'team@company.com'
        }
        failure {
            emailext subject: 'Build Failed',
                     body: 'Tests failed. Check Jenkins: ${BUILD_URL}',
                     to: 'team@company.com'
        }
    }
}

Scripted Pipeline

node {
    stage('Checkout') { checkout scm }
    stage('Build')     { sh 'mvn clean compile' }
    stage('Test')      { sh 'mvn test -Dsurefire.suiteXmlFiles=smoke.xml' }
}

Scripted Pipeline uses imperative Groovy. Use it when:

  • Dynamic stage generation (loop over environments or test modules)
  • Complex conditional logic that Declarative when cannot express
  • Shared libraries with advanced Groovy functions
  • Legacy pipelines migrated from Scripted before Declarative existed
Recommendation: Start with Declarative. Only switch to Scripted when you hit a genuine limitation. Most test automation pipelines never need Scripted.

Pipeline Syntax Elements

agent

agent any                              // any available node
agent { label 'windows-selenium' }     // specific agent with browsers installed
agent { docker { image 'maven:3.9-eclipse-temurin-17' } }  // containerized
agent none                             // no global agent; set per stage

environment

environment {
    TEST_ENV = 'qa'
    BASE_URL = 'https://qa.example.com'
    BROWSER  = "${params.BROWSER}"
}

when — Conditional Stage Execution

stage('Regression') {
    when {
        expression { params.SUITE == 'regression' }
    }
    steps { bat 'mvn test -Dsurefire.suiteXmlFiles=regression.xml' }
}

stage('Prod Tests') {
    when { branch 'release/*' }
    steps { bat 'mvn test -Denv=prod' }
}

parallel — Simultaneous Stages

stage('Cross-Browser') {
    parallel {
        stage('Chrome') {
            steps { bat 'mvn test -Dbrowser=chrome -Dsurefire.suiteXmlFiles=smoke.xml' }
        }
        stage('Firefox') {
            steps { bat 'mvn test -Dbrowser=firefox -Dsurefire.suiteXmlFiles=smoke.xml' }
        }
    }
}

input — Manual Approval Gate

stage('Deploy to Staging') {
    steps {
        input message: 'Smoke passed. Deploy to staging and run regression?', ok: 'Deploy'
        bat 'mvn test -Denv=staging -Dsurefire.suiteXmlFiles=regression.xml'
    }
}

timeout — Stage Timeout

stage('Regression') {
    options { timeout(time: 60, unit: 'MINUTES') }
    steps { bat 'mvn test -Dsurefire.suiteXmlFiles=regression.xml' }
}

retry — Retry Failed Stage

stage('Smoke') {
    options { retry(2) }
    steps { bat 'mvn test -Dsurefire.suiteXmlFiles=smoke.xml' }
}

Complete Automation Pipeline Example

Checkout → Build → Smoke (parallel Chrome+Firefox) → Regression → Report → Email → Deploy

pipeline {
    agent any
    tools { maven 'Maven-3.9' }
    parameters {
        choice(name: 'ENV', choices: ['qa', 'staging'], description: 'Target Environment')
        booleanParam(name: 'RUN_REGRESSION', defaultValue: true, description: 'Run full regression?')
    }
    environment {
        BASE_URL = "${params.ENV == 'qa' ? 'https://qa.example.com' : 'https://staging.example.com'}"
    }
    stages {
        stage('Checkout') {
            steps { checkout scm }
        }
        stage('Build') {
            steps { bat 'mvn clean compile -DskipTests' }
        }
        stage('Smoke Tests') {
            parallel {
                stage('Smoke - Chrome') {
                    steps {
                        bat 'mvn test -Dbrowser=chrome -Dheadless=true -Dsurefire.suiteXmlFiles=smoke.xml'
                    }
                }
                stage('Smoke - Firefox') {
                    steps {
                        bat 'mvn test -Dbrowser=firefox -Dheadless=true -Dsurefire.suiteXmlFiles=smoke.xml'
                    }
                }
            }
        }
        stage('Regression') {
            when { expression { params.RUN_REGRESSION } }
            options { timeout(time: 90, unit: 'MINUTES') }
            steps {
                bat "mvn test -Denv=${params.ENV} -Dsurefire.suiteXmlFiles=regression.xml"
            }
        }
        stage('Generate Reports') {
            steps {
                junit 'target/surefire-reports/*.xml'
                publishHTML([
                    reportDir: 'target/surefire-reports',
                    reportFiles: 'emailable-report.html',
                    reportName: 'TestNG Report',
                    keepAll: true
                ])
            }
        }
        stage('Deploy to QA') {
            when { branch 'develop' }
            steps {
                input message: 'Tests passed. Proceed with QA deployment?', ok: 'Deploy'
                bat 'echo Deploying build to QA environment...'
            }
        }
    }
    post {
        always {
            archiveArtifacts artifacts: 'target/surefire-reports/**, screenshots/**',
                               allowEmptyArchive: true
        }
        success {
            emailext subject: "PASS: ${env.JOB_NAME} #${env.BUILD_NUMBER}",
                     body: "All stages passed.\nBuild: ${env.BUILD_URL}",
                     to: 'qa-automation@company.com'
        }
        failure {
            emailext subject: "FAIL: ${env.JOB_NAME} #${env.BUILD_NUMBER}",
                     body: "Pipeline failed. Check: ${env.BUILD_URL}",
                     to: 'qa-automation@company.com',
                     attachLog: true
        }
    }
}

Pipeline Best Practices (10 Points)

  1. Keep Jenkinsfile in Git — never edit pipeline in Jenkins UI (Cheat option breaks version control).
  2. Use parameters for browser, environment, suite — avoid hardcoding values.
  3. Always publish test results — junit 'target/surefire-reports/*.xml' in a dedicated Report stage or post { always }.
  4. Archive artifacts on failure — screenshots, logs, HTML reports for debugging CI-only failures.
  5. Set stage timeouts — prevent hung browser sessions from blocking agents for hours.
  6. Use parallel for independent suites — cross-browser smoke, API + UI simultaneously.
  7. Separate smoke from regression — fast feedback on PR; heavy suite on merge/nightly.
  8. Use credentials binding — withCredentials for passwords/tokens, never plain text in Jenkinsfile.
  9. Add retry for flaky smoke — options { retry(1) } on smoke stage only, not regression.
  10. Keep stages focused — one responsibility per stage; easier to identify failure point in Blue Ocean/graph.

Interview Questions

  1. Q: What is Pipeline as Code?
    A: Defining CI/CD pipeline in a Jenkinsfile stored in Git. Benefits: version control, code review, reproducibility across Jenkins instances.
  2. Q: Explain Declarative vs Scripted Pipeline.
    A: Declarative uses structured syntax (pipeline { stages { } }) with built-in validation and post conditions. Scripted uses free Groovy in node { } for maximum flexibility.
  3. Q: What does agent any mean?
    A: Run the pipeline on any available Jenkins agent/node. Use agent { label 'windows' } when tests need a specific OS or installed browsers.
  4. Q: How do you run Chrome and Firefox tests in parallel?
    A: Use parallel { stage('Chrome'){...} stage('Firefox'){...} } inside a parent stage. Each parallel branch needs its own workspace or isolated execution.
  5. Q: What is the post block?
    A: Runs after all stages complete. Conditions: always, success, failure, unstable, changed. Used for archiving, notifications, cleanup.
  6. Q: How do you add a manual approval before regression?
    A: Use input message: 'Approve regression run?', ok: 'Proceed' as a step before the regression stage.
  7. Q: What is when directive?
    A: Conditionally executes a stage based on branch, parameter value, environment, or custom expression. E.g., run regression only when params.SUITE == 'regression'.
  8. Q: How do you pass Maven properties from Jenkins parameters?
    A: Interpolate in shell step: bat "mvn test -Dbrowser=${params.BROWSER}". Parameters are available via params.PARAM_NAME.
  9. Q: What happens if a stage times out?
    A: Jenkins aborts the stage and marks the build as failed. Configure with options { timeout(time: 30, unit: 'MINUTES') }.
  10. Q: How do you skip a stage based on a boolean parameter?
    A: when { expression { params.RUN_REGRESSION } } on the stage block.
  11. Q: Difference between bat and sh?
    A: bat for Windows agents (cmd.exe); sh for Linux/Mac agents (bash). Match your Jenkins agent OS.
  12. Q: How do you publish TestNG HTML report in Pipeline?
    A: Use HTML Publisher Plugin: publishHTML([reportDir: '...', reportFiles: 'index.html', reportName: '...']) plus junit for trend graphs.

Real Project Example

A banking QA team replaced 6 Freestyle jobs with one Declarative Pipeline. Parameters: ENV (dev/qa/staging), BROWSER, SUITE (smoke/regression). PR builds ran smoke in parallel on Chrome+Firefox (4 min). Merge to develop triggered regression (45 min) with 90-min timeout. Post block archived ExtentReports and emailed team with pass/fail counts. Pipeline changes required PR approval—eliminating "someone changed the Jenkins job and broke nightly" incidents.

Common Mistakes

  • Editing pipeline via "Replay" or UI instead of committing Jenkinsfile changes to Git
  • Not setting timeouts— hung WebDriver sessions block agents indefinitely
  • Running regression on every PR (slow feedback); should be smoke only on PR
  • Using sh on Windows agents or bat on Linux agents
  • Hardcoding URLs/credentials instead of parameters and credentials binding
  • Parallel stages writing to same workspace without isolation—causes test interference

Troubleshooting Tips

  • Pipeline syntax error: Use Declarative Pipeline Linter (Workflow CPS plugin) or "Pipeline Syntax" generator in job config.
  • Parameter not recognized: Run job once after adding parameters (Jenkins needs initial build to register them).
  • Parallel stage fails randomly: Ensure ThreadLocal WebDriver and separate test data per thread.
  • post block not running: Check for uncaught exception before stages complete; use post { always }.
  • Maven command works in console but not pipeline: Add tools { maven '...' } or use full path to mvn.

Quick Revision Notes

  • Declarative = structured, validated, preferred for test pipelines
  • Key blocks: pipeline → agent → parameters → stages → post
  • parallel = cross-browser; when = conditional; input = approval gate
  • timeout + retry = control flaky/hung test runs
  • Always: junit + archiveArtifacts + email in post block

Section 10: Jenkins + TestNG Integration

TestNG is the most common test framework integrated with Jenkins in Java/Selenium automation projects. This section covers Maven execution, suite XML design, pipeline integration, parallel runs, and failure handling—from an automation engineer's workflow perspective.

TestNG Execution via Maven

Run TestNG suites through Maven Surefire plugin:

mvn test -Dsurefire.suiteXmlFiles=src/test/resources/testng/smoke.xml
mvn test -Dsurefire.suiteXmlFiles=smoke.xml -Dbrowser=chrome -Denv=qa

Surefire Plugin Configuration in pom.xml

<build>
  <plugins>
    <plugin>
      <groupId>org.apache.maven.plugins</groupId>
      <artifactId>maven-surefire-plugin</artifactId>
      <version>3.2.5</version>
      <configuration>
        <suiteXmlFiles>
          <suiteXmlFile>${suiteXmlFile}</suiteXmlFile>
        </suiteXmlFiles>
        <systemPropertyVariables>
          <browser>${browser}</browser>
          <env>${env}</env>
        </systemPropertyVariables>
        <testFailureIgnore>false</testFailureIgnore>
      </configuration>
    </plugin>
  </plugins>
</build>
Pass suite at runtime: -Dsurefire.suiteXmlFiles=smoke.xml overrides pom.xml default. Pass browser/env as Maven properties consumed by your framework's config reader.

Parallel Execution Setup

testng.xml Configuration

<suite name="Parallel Suite" parallel="methods" thread-count="5">
  <test name="Parallel Tests">
    <classes>
      <class name="com.app.tests.AccountTests"/>
    </classes>

</suite>

Requirements

  • Jenkins agent executors: Set # executors ≥ thread-count on the agent label
  • ThreadLocal WebDriver: Each thread must have its own driver instance—never share static WebDriver across parallel tests
  • Independent test data: Use unique user accounts or data factories per thread to avoid collisions
  • Grid for heavy parallel: Local machine may handle 5 threads; 20+ threads need Selenium Grid

    A: Runs @Test methods concurrently across threads. Requires ThreadLocal WebDriver and thread-safe test data.
  5. Q: Difference between failed and unstable build for tests?
    A: FAILURE blocks pipeline/deploy; UNSTABLE indicates test failures but allows pipeline to continue (use for non-gating suites).
  6. Q: How do you show TestNG trend graph in Jenkins?
    A: Publish JUnit results consistently every build. Jenkins auto-generates "Test Result Trend" on job page.
  7. Q: What happens if surefire reports directory is empty?
    A: Build may fail at junit step. Use allowEmptyResults: true only when tests might legitimately not run.
  8. Q: How to run different suites on different branches?
    A: Multibranch Pipeline with when { branch 'feature/*' } for smoke and when { branch 'develop' } for regression.
  9. Q: Can you run multiple suite XML files in one Maven command?
    A: Yes: -Dsurefire.suiteXmlFiles=smoke.xml,api.xml or define multiple suites in pom.xml.
  10. Q: How do you archive TestNG emailable report?
    A: HTML Publisher Plugin pointing to target/surefire-reports/emailable-report.html plus archiveArtifacts in post block.

Real Project Example

A SaaS company ran 180 TestNG tests across 4 suite files. PR pipeline: smoke.xml (12 tests, 3 min). Nightly: regression.xml (180 tests, 40 min) with parallel="methods" thread-count=5. Jenkins published JUnit trends—when failures spiked after a release, QA caught a regression within hours. Failure emails included ${FAILED_TESTS} token listing exact test names and stack traces.

Common Mistakes

  • Static shared WebDriver with parallel="methods"—causes random failures and session conflicts
  • Wrong path to suite XML—use repo-relative path matching pom.xml structure
  • Not publishing JUnit results—team cannot see which @Test methods failed
  • Running full regression on every commit—kills developer feedback loop
  • Setting testFailureIgnore=true to "make pipeline green"—hides real failures

Troubleshooting Tips

  • 0 tests run: Check suite XML class names match package; verify Surefire includes **/*Test.java.
  • Tests pass locally, fail in Jenkins: Compare JDK version, env properties, headless mode, test data availability.
  • Parallel tests flaky only in CI: Reduce thread-count; verify ThreadLocal driver; check for shared test accounts.
  • No trend graph: Ensure junit step runs every build with same report path pattern.
  • Maven property not reaching TestNG: Define in surefire systemPropertyVariables or pass via -D flag.

Quick Revision Notes

  • Execute: mvn test -Dsurefire.suiteXmlFiles=<suite>.xml
  • Suites: smoke (PR), regression (nightly), sanity (post-deploy), crossbrowser (weekly)
  • Jenkins: junit step + publishHTML + archiveArtifacts on failure
  • Parallel: thread-count in XML + ThreadLocal WebDriver + enough executors
  • Failure handling: FAILURE blocks merge; UNSTABLE for non-gating runs

Section 12: Jenkins + Selenium

Running Selenium in Jenkins requires understanding local vs headless vs Grid execution, browser driver management, and artifact collection on failure. This section focuses on what automation engineers configure—not server administration.

Local Execution

Jenkins agent runs on the same machine where ChromeDriver/GeckoDriver are installed. Suitable for small teams or dedicated automation VMs with browsers pre-installed.

  • Install browsers and matching driver versions on the Jenkins agent
  • Set driver path via system property or WebDriverManager in framework
  • Limitation: single machine resources; cannot scale parallel browsers easily

Headless Execution

CI servers typically have no display (GUI). Headless mode runs browsers without a visible window.

Why Headless in CI

  • No GUI/display server required on Linux CI agents
  • Faster execution—no rendering overhead
  • Standard approach for Jenkins, GitHub Actions, Azure DevOps agents

Configuration Code

ChromeOptions options = new ChromeOptions();
if (Boolean.parseBoolean(System.getProperty("headless", "true"))) {
    options.addArguments("--headless=new");
    options.addArguments("--window-size=1920,1080");
    options.addArguments("--disable-gpu");
    options.addArguments("--no-sandbox");
    options.addArguments("--disable-dev-shm-usage");
}
WebDriver driver = new ChromeDriver(options);
Pass from Jenkins: mvn test -Dheadless=true -Dbrowser=chrome. Always set window-size in headless—default 800×600 causes element visibility failures.

Remote Execution with Selenium Grid

Hub-Node Architecture

  • Hub: Central router receiving test requests (port 4444)
  • Nodes: Worker machines/containers with browsers; hub assigns sessions
  • Tests use RemoteWebDriver pointing to hub URL
DesiredCapabilities caps = new DesiredCapabilities();
caps.setBrowserName("chrome");
WebDriver driver = new RemoteWebDriver(
    new URL("http://selenium-grid:4444/wd/hub"), caps);

Screenshot Archiving on Failure

@AfterMethod
public void captureScreenshot(ITestResult result) {
    if (result.getStatus() == ITestResult.FAILURE) {
        File src = ((TakesScreenshot) driver.get())
                     .getScreenshotAs(OutputType.FILE);
        Files.copy(src.toPath(),
            Paths.get("screenshots/" + result.getName() + ".png"),
            StandardCopyOption.REPLACE_EXISTING);
    }
}

Jenkins post { failure { archiveArtifacts 'screenshots/**' } } makes screenshots downloadable from build page.

Selenium Grid 4 with Docker in Jenkins

  1. Install Docker on Jenkins agent (or use dedicated Grid server)
  2. Add stage to start Grid: sh 'docker-compose -f docker/grid-compose.yml up -d'
  3. Wait for hub ready: sh 'curl --retry 10 --retry-delay 5 http://localhost:4444/status'
  4. Run tests pointing to Grid URL
  5. Post always: sh 'docker-compose down' to free resources

Interview Questions

  1. Q: Why run Selenium headless in Jenkins?
    A: CI agents lack display/GUI. Headless runs Chrome/Firefox without a visible window, faster and no Xvfb needed (with --headless=new).
  2. Q: What ChromeOptions are essential for CI?
    A: --headless=new, --window-size=1920,1080, --no-sandbox, --disable-dev-shm-usage (Docker).
  3. Q: Explain Selenium Grid hub-node architecture.
    A: Hub routes test requests; nodes execute browsers. Enables parallel cross-browser execution across machines/containers.
  4. Q: How do tests connect to Grid from Jenkins?
    A: RemoteWebDriver with hub URL passed as Maven property: -Dgrid.url=http://grid:4444/wd/hub.
  5. Q: Why is shm_size: 2gb needed in Docker Grid?
    A: Default /dev/shm (64MB) is too small for Chrome; causes crashes. Increase shared memory in docker-compose.
  6. Q: How to archive failure screenshots in Jenkins?
    A: Test listener saves to screenshots/ on @AfterMethod failure; pipeline archiveArtifacts 'screenshots/**' in post failure block.
  7. Q: Local vs Grid execution—when to use each?
    A: Local for dev/single-browser smoke. Grid when parallel sessions exceed agent capacity or cross-browser scaling needed.
  8. Q: Tests pass headless locally but fail in Jenkins—common causes?
    A: Different screen resolution, missing --no-sandbox in Docker, driver/browser version mismatch, timing issues (need explicit waits).
  9. Q: How to run Chrome and Firefox in parallel on Grid?
    A: Jenkins parallel stages each invoking mvn test with different -Dbrowser value, both pointing to same Grid hub.
  10. Q: What is WebDriverManager role in CI?
    A: Auto-downloads matching driver for installed browser version—reduces "driver version mismatch" failures on agents.

Common Mistakes

  • Missing --window-size in headless—elements reported as not visible/interactable
  • Not increasing Docker shm_size—Chrome crashes with "session deleted" errors
  • Forgetting to tear down Grid containers—memory exhaustion on agent over time
  • Hardcoding Grid URL instead of Jenkins parameter—breaks when Grid moves
  • Not archiving screenshots—CI-only failures impossible to debug

Troubleshooting Tips

  • Session not created: Browser/driver version mismatch—use WebDriverManager or Selenium 4 built-in driver management.
  • Timeout connecting to Grid: Verify hub is up (/status endpoint); check firewall between Jenkins agent and Grid.
  • Chrome crashed in Docker: Add --no-sandbox, increase shm_size, ensure enough RAM for node containers.
  • Tests slower in CI: Normal—add explicit waits; avoid Thread.sleep; consider Grid scaling.
  • Screenshots empty/black: Headless screenshot timing—take screenshot after page load complete.

Section 13: Jenkins + Playwright

Playwright is increasingly chosen for CI/CD due to built-in auto-waiting, native parallelism, and multi-browser support. Automation engineers using TypeScript/JavaScript/Python can integrate Playwright into Jenkins with Node.js tooling.

Why Playwright in CI/CD

  • Fast: Auto-wait eliminates most flaky timing issues
  • Reliable: Built-in retry, trace viewer for debugging CI failures
  • Built-in parallelism: --workers=N without Grid setup
  • Multi-browser: Chromium, Firefox, WebKit from single test code
  • Headless by default in CI—no extra configuration needed

Playwright Setup in Jenkins

  1. Install NodeJS Plugin in Jenkins → Global Tool Configuration → add Node.js 18+
  2. Pipeline references: tools { nodejs 'Node-18' }
  3. Install dependencies: npm ci (uses package-lock.json—reproducible)
  4. Install browsers: npx playwright install --with-deps
  5. Run tests: npx playwright test

Jenkins Pipeline for Playwright

pipeline {
    agent any
    tools { nodejs 'Node-18' }

            }
        }

            steps {
                script {
                    if (params.PROJECT == 'all') {
                        bat "npx playwright test --workers=${params.WORKERS} --reporter=html,list"
                    } else {
                        bat "npx playwright test --project=${params.PROJECT} --workers=${params.WORKERS} --reporter=html,list"
                    }
                }
            }
        }

            steps {

                    reportDir: 'playwright-report',

        failure {
            archiveArtifacts artifacts: 'playwright-report/**, test-results/**',
                               allowEmptyArchive: true

        }
        always {
            junit allowEmptyResults: true,
                  testResults: 'test-results/junit.xml'
        }
    }
}

stage('Quality Gate') {
    steps {
        script {
            def testResults = junit testResults: 'target/surefire-reports/*.xml'
            def passRate = (testResults.passCount / testResults.totalCount) * 100
            if (passRate < 95) {
                error("Pass rate ${passRate}% below 95% threshold")
            }
        }
    }
}

Flaky Test Management

RetryAnalyzer in TestNG

public class RetryAnalyzer implements IRetryAnalyzer {
    private int count = 0;
    private static final int MAX_RETRY = 2;

    @Override
    public boolean retry(ITestResult result) {
        if (count < MAX_RETRY) { count++; return true; }
        return false;
    }
}

// Usage: @Test(retryAnalyzer = RetryAnalyzer.class)

Quarantine Strategy

  • Tag flaky tests: @Tag("flaky") or TestNG group flaky
  • Exclude from blocking suites: @smoke and not @flaky
  • Run quarantined tests in separate non-blocking nightly job
  • Track fix progress—flaky tests must have assigned owner and deadline

Monitoring

  • Track flaky rate: (tests that passed on retry) / (total tests) per build
  • Jenkins test trend + custom dashboard (Grafana, ReportPortal)
  • Alert when flaky rate exceeds 2%—indicates infrastructure or test design problem
Never permanently leave retry on smoke tests to "keep CI green." Fix root cause or quarantine.

Interview Questions

  1. Q: What tests should run on every PR?
    A: Smoke suite only—critical paths, under 10 min, blocks merge on failure. Never full regression on PR.
  2. Q: Explain build promotion with quality gates.
    A: Dev → smoke pass → deploy QA → regression pass → staging → UAT pass → production. Each gate has pass criteria.
  3. Q: What is a quality gate?
    A: Automated checkpoint (e.g., 95% pass rate, all smoke pass, 80% coverage) that must pass before promoting build.
  4. Q: How do you handle flaky tests in CI?
    A: Quarantine with @flaky tag, exclude from blocking suites, run in separate job, track flaky rate, fix root cause—never permanently retry on smoke.
  5. Q: Difference between smoke, sanity, and regression?
    A: Smoke = minimal critical paths (PR). Sanity = module/post-deploy checks. Regression = full coverage (nightly).
  6. Q: How to manage test data in parallel CI runs?
    A: Dynamic data factories (Faker), unique accounts per thread, DB reset before suite, or isolated Docker DB per pipeline.
  7. Q: When should CI block vs notify on test failure?
    A: Block: PR smoke, release UAT. Notify: nightly regression, weekly cross-browser—still important but not merge/release blocking.
  8. Q: How to design suites for fast feedback?
    A: Tier tests: smoke (20 tests, 5 min) → API (50 tests, 10 min) → regression (500 tests, 2 hr). Match trigger to tier.
  9. Q: What is test quarantine?
    A: Moving flaky tests to a separate non-blocking suite until fixed; excluded from smoke/regression gates via tags.
  10. Q: How do you enforce 95% pass rate in Jenkins?
    A: Parse JUnit results in pipeline script; calculate passCount/totalCount; call error() if below threshold.

Real Project Example

An ERP product team implemented a 5-tier strategy: PR smoke (15 tests, 4 min, blocks merge), post-merge API+smoke (15 min), nightly regression (420 tests, 90 min), weekly cross-browser (3 browsers), release sanity (30 tests before prod). Flaky tests tagged @flaky moved to quarantine job—flaky rate dropped from 8% to 1.5% in 6 weeks. Test data seeded via API before regression; static JSON for smoke. Quality gate: 100% smoke + 95% regression pass rate required for staging promotion.

Common Mistakes

  • Running full regression on every PR—developers ignore CI due to 1-hour wait
  • No test tier differentiation—same suite for PR and nightly
  • Ignoring flaky tests by adding retries everywhere—masks real issues
  • Shared test accounts across parallel runs—random login failures
  • No quality gates on staging/production promotion—bugs reach users
  • Static test data never refreshed—tests pass but validate outdated behavior

Troubleshooting Tips

  • Smoke suite too slow: Audit test count; move non-critical tests to post-merge or API suite.
  • High flaky rate: Identify top 10 flaky tests from retry logs; quarantine and assign owners.
  • Data collisions in parallel: Switch to dynamic data factory; never share user accounts across threads.
  • Quality gate false positives: Exclude quarantined @flaky tests from pass rate calculation.
  • Nightly always red: Team stops caring—fix top failures first; split environment issues from real bugs.

Quick Revision Notes

  • PR = smoke (5 min, block) | Nightly = regression (2 hr, notify)
  • Suites: smoke.xml, api.xml, regression.xml, crossbrowser.xml, sanity.xml
  • Promotion: smoke → QA → regression → staging → UAT → prod
  • Gates: 100% smoke, 95% pass rate, 80% coverage, 0 P1/P2 failures
  • Flaky: quarantine @flaky, separate job, track rate, fix root cause
  • Test data: seed before run, reset DB, env-specific files, dynamic factories for parallel
