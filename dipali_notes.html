SDLC — Software Development Life Cycle

SDLC is a structured process for planning, creating, testing, and deploying software. It defines phases from idea to maintenance ensuring quality at every step.

7 Stages of SDLC

StageDescriptionQA Involvement
1. Requirement CollectionGather functional & non-functional requirements from stakeholders via BRD, FRD, user storiesReview requirements for testability, clarity, completeness
2. Feasibility StudyAnalyze technical, economic, legal, operational feasibilityEstimate testing effort, tools, environment needs
3. DesignSystem architecture, HLD, LLD, database design, UI mockupsReview design docs; identify test scenarios early
4. CodingDevelopers write code per design specificationsUnit testing support; static code review participation
5. TestingExecute test cases, report defects, regression, UATPrimary QA phase — STLC execution
6. DeploymentRelease to production/staging; smoke testing post-deploySmoke tests, deployment verification, rollback validation
7. MaintenanceBug fixes, enhancements, patches after releaseRegression testing, impact analysis for changes

SDLC Models

Waterfall Model

Sequential linear approach — each phase must complete before the next begins. Requirements are fixed upfront.

AspectDetails
AdvantagesSimple, well-documented, easy to manage, clear milestones, stable requirements
DisadvantagesNo working software until late, difficult to accommodate changes, testing starts late, high risk
When to UseShort projects, fixed requirements, regulatory/compliance projects, well-understood domains

Spiral Model

Risk-driven iterative model combining waterfall with prototyping. Each spiral: Plan → Risk Analysis → Engineering → Evaluation.

  • Best for: Large, complex, high-risk projects (e.g., banking, aerospace)
  • Advantage: Early risk identification; prototyping reduces uncertainty
  • Disadvantage: Expensive; requires risk assessment expertise

V-Model (Verification & Validation)

Extension of waterfall where each development phase has a corresponding testing phase.

Requirements —————————→ Acceptance Testing ↓ ↓ System Design ————————→ System Testing ↓ ↓ Architecture Design ——————→ Integration Testing ↓ ↓ Module Design ————————→ Unit Testing ↓ ↓ Coding / Implementation

Verification = "Are we building the product right?" (reviews, inspections). Validation = "Are we building the right product?" (actual testing).

Prototype Model

Build a throwaway/exploratory prototype to clarify requirements before full development. User feedback drives refinement.

  • Types: Throwaway (discarded after feedback) vs Evolutionary (becomes final product)
  • Best for: UI-heavy apps, unclear requirements, new technology

Agile Model

Iterative, incremental approach delivering working software in short sprints (1–4 weeks). Embraces change; continuous feedback.

PrincipleAgile Value
Individuals & interactionsOver processes & tools
Working softwareOver comprehensive documentation
Customer collaborationOver contract negotiation
Responding to changeOver following a plan

Software Testing Fundamentals

7 Principles of Software Testing

  1. Testing shows presence of defects — not their absence
  2. Exhaustive testing is impossible — use risk-based prioritization
  3. Early testing — find defects in requirements/design phase (shift-left)
  4. Defect clustering — 80% bugs in 20% modules (Pareto principle)
  5. Pesticide paradox — repeating same tests won't find new bugs; update test cases
  6. Testing is context-dependent — e-commerce vs medical device need different approaches
  7. Absence-of-errors fallacy — bug-free software may still fail if requirements are wrong

Black Box Testing Types (12 Types)

1. Functionality Testing

Verify each function of the application works per requirements. Test all features, fields, buttons, links, APIs.

Example: Login with valid credentials → dashboard loads; invalid credentials → error message displayed.

2. Integration Testing

Test interfaces between integrated modules/components.

ApproachDescriptionStubs/Drivers
Top-DownTest from top modules downward; lower modules stubbedStubs simulate lower modules
Bottom-UpTest from lowest modules upward; upper modules use driversDrivers simulate upper modules
Non-Incremental (Big Bang)All modules integrated at once then testedHard to isolate defects
Incremental (Sandwich/Hybrid)Combination of top-down and bottom-upBoth stubs and drivers

3. System Testing

End-to-end testing of complete integrated system against requirements. Includes functional + non-functional testing in production-like environment.

4. Acceptance Testing

TypeFull FormDescription
UATUser Acceptance TestingEnd users validate system meets business needs
BATBusiness Acceptance TestingBusiness stakeholders verify business processes
CATContract Acceptance TestingVerify contract/agreement criteria met
RATRegulation Acceptance TestingCompliance with regulatory standards (FDA, GDPR)
OATOperational Acceptance TestingBackup/recovery, maintenance, failover readiness
Alpha Testing—In-house testing at developer site before release
Beta Testing—Real users test in real environment before final release

5. Smoke Testing

Quick shallow test of critical functionalities after a build to decide if build is stable enough for further testing. Also called Build Verification Testing (BVT).

Example: App launches → login works → main page loads → logout works. If any fails, reject build.

6. Adhoc / Monkey Testing

Adhoc: Unplanned, random testing without documentation — relies on tester experience.
Monkey Testing: Random inputs/actions to crash the application — no predefined test cases.

7. Performance Testing

TypeDescriptionExample Scenario
Load TestingTest under expected user load1000 concurrent users for 30 min
Stress TestingTest beyond normal capacity to find breaking point5000 users until server crashes
Volume TestingLarge amounts of data handled by applicationInsert 10 million records in DB
Soak/Endurance TestingSustained load over extended period500 users for 48 hours — check memory leaks

8. Regression Testing

TypeScope
Unit RegressionRe-test specific unit/module after code change
Regional RegressionTest affected area + adjacent modules
Full RegressionComplete test suite re-executed — before major release

9. Exploratory Testing

Simultaneous learning, test design, and test execution. Tester explores application without predefined scripts — finds edge cases through intuition.

10. Usability Testing

Evaluate user-friendliness: navigation, layout, accessibility, learnability. Real users perform tasks while observers note difficulties.

11. Compatibility Testing

Verify application works across browsers (Chrome, Firefox, Safari, Edge), OS (Windows, Mac, Linux), devices (mobile, tablet, desktop), screen resolutions.

12. Globalization Testing

TypeDescription
I18N (Internationalization)Design app to support multiple languages/regions without code changes. Unicode, date/currency formats
L10N (Localization)Adapt app for specific locale — translate UI, adjust formats, cultural adaptations

Test Cases

Why Write Test Cases?

  • Ensure complete test coverage of requirements
  • Provide repeatable, consistent testing
  • Enable traceability (requirement → test case → result)
  • Facilitate knowledge transfer and onboarding
  • Support regression testing and automation
  • Document evidence for audits and compliance

Interview Questions

  1. What are the 7 stages of SDLC?
    Requirement Collection, Feasibility Study, Design, Coding, Testing, Deployment, Maintenance.
  2. Difference between SDLC and STLC?
    SDLC covers entire software development; STLC is the testing subset within SDLC's Testing phase with its own phases (planning, design, execution, closure).
  3. What is the difference between Severity and Priority?
    Severity = impact on application functionality. Priority = urgency of fixing. High severity bug in rarely-used feature may have low priority.
  4. Explain ECP and BVA with example.
    Password field (6–12 chars): ECP partitions — valid (6–12), too short (<6), too long (>12). BVA tests: 5, 6, 7, 11, 12, 13.
  5. What is defect leakage?
    Defects that escape to production/customer. Formula: (Defects found in production / Total defects) — 100.
  6. Difference between Smoke and Sanity testing?
    Smoke = broad shallow test of critical features on new build. Sanity = narrow deep test of specific changed functionality.
  7. What is RTM and why is it important?
    Requirement Traceability Matrix maps requirements to test cases ensuring 100% coverage and no orphan tests.
  8. Explain V-Model.
    Each development phase has corresponding test phase: Requirements→Acceptance, Design→System, Architecture→Integration, Module Design→Unit Testing.
  9. What are the 7 principles of testing?
    Testing shows defects exist; exhaustive testing impossible; early testing; defect clustering; pesticide paradox; context-dependent; absence-of-errors fallacy.
  10. Difference between Alpha and Beta testing?
    Alpha = in-house at developer site. Beta = real users in real environment before final release.

What is Automation Testing?

Automation Testing uses tools and scripts to execute pre-defined test cases automatically, compare actual vs expected results, and generate reports without manual intervention.

Benefits of Automation Testing

  • Speed: Execute thousands of tests in minutes vs days manually
  • Repeatability: Same tests run identically every time
  • Regression: Re-run full suite after every code change
  • Data-driven: Same test with multiple data sets (CSV, Excel, JSON)
  • Parallel execution: Run tests on multiple browsers/machines simultaneously
  • 24/7 execution: CI/CD pipelines run tests overnight
  • Cost-effective long-term: Reduces manual effort for stable features

Selenium Introduction

Selenium is an open-source suite of tools for automating web browsers. Supports multiple languages (Java, Python, C#, JavaScript) and browsers (Chrome, Firefox, Edge, Safari).

Advantages

  • Open source and free
  • Cross-browser and cross-platform support
  • Large community and extensive documentation
  • Integrates with TestNG, JUnit, Maven, Jenkins, Cucumber
  • Supports headless browser execution

Drawbacks

  • Only for web applications (not desktop/mobile native apps)
  • No built-in reporting — needs TestNG/Jenkins/ExtentReports
  • Dynamic elements and flaky tests require robust wait strategies
  • Cannot automate captcha, barcode scanning, or OTP without workarounds
  • Maintenance overhead when UI changes frequently

    IDE (record/playback), RC (deprecated), WebDriver (primary), Grid (parallel execution).
  2. Difference between findElement and findElements?
    findElement returns first matching WebElement (throws exception if none). findElements returns List (empty list if none).
  3. Which locator is fastest?
    ID is fastest and most reliable. CSS Selector is second fastest. XPath is most flexible but slowest.
  4. When to use Actions class?
    Mouse operations: hover, drag-drop, double-click, right-click, click-and-hold when standard click() doesn't work.
  5. Implicit vs Explicit wait?
    Implicit = global timeout for all findElement. Explicit = wait for specific condition on specific element with WebDriverWait.
  6. How to handle dropdown?
    Use Select class: selectByValue(), selectByVisibleText(), selectByIndex().
  7. get() vs navigate().to()?
    Both open URL. navigate().to() additionally supports back(), forward(), refresh() via browser history.
  8. quit() vs close()?
    quit() closes ALL windows and ends WebDriver session. close() closes only current window.

Page Object Model (POM)

Design pattern where each web page has a corresponding class containing locators and methods for that page.

Why POM?

  • Maintainability: If UI changes, update only one class (not all tests)
  • Reusability: Same page methods used across multiple tests
  • Readability: Tests read like user actions, not locator details
  • Separation of concerns: Test logic separated from page structure

public class LoginPage {
    WebDriver driver;

    @FindBy(id = "username") WebElement usernameField;
    @FindBy(id = "password") WebElement passwordField;
    @FindBy(id = "loginBtn") WebElement loginButton;

        usernameField.sendKeys(user);
        passwordField.sendKeys(pass);
        loginButton.click();
        return new DashboardPage(driver);
    }
}

16. String Permutations

public class Permutations {
    public static void permute(String str, int start, int end) {
        if (start == end) System.out.println(str);
        for (int i = start; i <= end; i++) {
            str = swap(str, start, i);
            permute(str, start + 1, end);
            str = swap(str, start, i);
        }
    }
    static String swap(String s, int i, int j) {
        char[] ch = s.toCharArray();
        char temp = ch[i]; ch[i] = ch[j]; ch[j] = temp;
        return String.valueOf(ch);
    }
    public static void main(String[] args) { permute("ABC", 0, 2); }
}
// Output: ABC, ACB, BAC, BCA, CAB, CBA

public class LongestSubstring {
    public static void main(String[] args) {
        String str = "abcabcbb";
        Set<Character> set = new HashSet<>();
        int max = 0, left = 0;
        for (int right = 0; right < str.length(); right++) {
            while (set.contains(str.charAt(right))) {
                set.remove(str.charAt(left++));
            }
            set.add(str.charAt(right));

}

41. hashCode() and equals()

class Person {
    String name;
    Person(String name) { this.name = name; }
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Person)) return false;
        return name.equals(((Person) o).name);
    }
    @Override
    public int hashCode() { return name.hashCode(); }
    public static void main(String[] args) {
        Person p1 = new Person("John"), p2 = new Person("John");

        System.out.println("hashCode same: " + (p1.hashCode() == p2.hashCode()));

}

Interview Questions

  1. How to reverse a string without StringBuilder?
    Convert to char array, swap characters from start and end using two pointers.
  2. How to find duplicate characters in a string?
    Use HashMap to count frequency of each character; print entries with count > 1.
  3. Difference between == and equals()?
    == compares references (memory address). equals() compares content (overridden in String, custom objects).
  4. Why override hashCode() when overriding equals()?
    Contract: equal objects must have same hashCode. Required for HashMap/HashSet to work correctly.
  5. How to find second largest element in array?
    Track largest and second largest in single pass through array.

Interview Questions

  1. Difference between DELETE and TRUNCATE?
    DELETE removes rows one by one (can use WHERE, rollback possible). TRUNCATE removes all rows at once (faster, resets identity, minimal logging).
  2. Difference between WHERE and HAVING?
    WHERE filters rows before grouping. HAVING filters groups after GROUP BY with aggregate functions.
  3. What is a PRIMARY KEY vs FOREIGN KEY?
    PRIMARY KEY uniquely identifies each row (unique + NOT NULL). FOREIGN KEY references PRIMARY KEY in another table for referential integrity.
  4. INNER JOIN vs LEFT JOIN?
    INNER returns only matching rows from both tables. LEFT returns all rows from left table + matched rows from right (NULL if no match).
  5. How to find duplicate records?
    SELECT email, COUNT(*) FROM employees GROUP BY email HAVING COUNT(*) > 1;
  6. UNION vs UNION ALL?
    UNION removes duplicates. UNION ALL keeps all rows including duplicates (better performance).
  7. What is a VIEW?
    Virtual table based on SQL query result. Simplifies complex queries; provides security by hiding columns.

  2. Explain Bug Life Cycle.
    Bug Life Cycle is the journey of a defect from discovery to closure:
    New → Tester logs the defect
    Assign/Open → Assigned to developer
    Fixed → Developer fixes the bug
    Retest → Tester verifies the fix
    Closed → Fix confirmed, defect resolved
    Reopen → Fix didn't work, sent back to developer

  3. Explain XPath types: Absolute, Relative, Dynamic.
    Absolute XPath: Complete path from root element — /html/body/div[1]/form/input[1]. Fragile; breaks when DOM structure changes.
    Relative XPath: Starts from anywhere using // — //input[@id='username']. Preferred; more maintainable.
    Dynamic XPath: Handles changing attributes — //button[contains(@class,'submit')], //input[starts-with(@id,'user_')], //span[text()='Login'].
  4. What are Annotations in Selenium?

    @Test — marks method as test case
    @BeforeMethod — runs before each test (e.g., open browser)
    @AfterMethod — runs after each test (e.g., close browser)
    @BeforeClass — runs once before all tests in class
    @AfterClass — runs once after all tests in class
    @BeforeSuite / @AfterSuite — runs before/after entire test suite
    @DataProvider — supplies data for data-driven tests
    @Parameters — passes parameters from testng.xml
    @Ignore / @Test(enabled=false) — skip test execution

  8. What are Selenium Locators?

SoftAssert soft = new SoftAssert();
soft.assertEquals(actual1, expected1);
soft.assertEquals(actual2, expected2);
soft.assertAll();  // Reports all failures at end

Test Design Techniques

Boundary Value Analysis (BVA)

Test at boundaries of input domains. For range 18-30:

  • Valid: 18 (min), 19 (min+1), 29 (max-1), 30 (max)
  • Invalid: 17 (min-1), 31 (max+1)

Equivalence Partitioning (EP)

Divide input into valid/invalid partitions; test one value from each.

Example: Username accepts a-z, length 4-10. Valid partition: alphabets. Invalid: numbers, special chars, blank.

Decision Table Testing

Capture input combinations and expected outputs in tabular form (Cause-Effect table).

State Transition Testing

Test transitions between states. Example: ATM — 3 wrong PINs locks account.

API Interview Questions with Answers

Q1. What is Idempotent? Which HTTP methods are idempotent?

Idempotent means making the same request multiple times produces the same result. GET, PUT, DELETE, HEAD, OPTIONS are idempotent. POST and PATCH are NOT idempotent (POST creates new resource each time).

Q2. What is the difference between PUT and PATCH?

PUT: Replaces the entire resource. Must send complete object. If fields are missing, they become null.
PATCH: Updates only specified fields. Partial update. Other fields remain unchanged.

// PUT - replaces entire user
PUT /users/1 {"name": "John", "email": "john@test.com", "age": 30}

// PATCH - updates only email
PATCH /users/1 {"email": "newemail@test.com"}

Q3. Explain OAuth 2.0 flow.

  1. Client requests authorization from Resource Owner
  2. Resource Owner grants authorization code
  3. Client exchanges code for Access Token from Auth Server
  4. Client uses Access Token to access Resource Server
  5. Token expires → use Refresh Token to get new Access Token

Q4. What is API Versioning? Why is it important?

Maintaining multiple API versions to support backward compatibility. Methods:

  • URL Path: /api/v1/users, /api/v2/users
  • Query Parameter: /api/users?version=2
  • Header: Accept: application/vnd.api.v2+json

Q5. How do you handle API testing when backend is not ready?

Use Mock Servers (Postman Mock, WireMock, MockServer). Define expected requests/responses. Allows frontend and QA to proceed independently. Also useful for testing error scenarios that are hard to reproduce.

Q6. What is API Chaining? Give an example.

Using the response of one API call as input to the next:

1. POST /users → Response: {"id": 123}
2. GET /users/123 → Verify user created
3. PUT /users/123 → Update user
4. DELETE /users/123 → Remove user

Q7. How do you test API Rate Limiting?

  • Send requests exceeding the limit (e.g., 101 requests when limit is 100/min)
  • Verify 429 (Too Many Requests) status code returned
  • Check Retry-After header in response
  • Verify rate limit resets after specified window

Q8. What is Contract Testing?

Verifying that the API provider and consumer agree on the interface (contract). Tools: Pact, Spring Cloud Contract. Ensures API changes don't break consumers without running full integration tests.

git bisect start

# Git checks out middle commit - you test it
git bisect good/bad     # tell git if this commit has the bug

Section 11: UAT — User Acceptance Testing

Complete guide to UAT — the final testing checkpoint before deployment where real users validate business requirements.

What is UAT?

  • Final phase of testing in SDLC
  • Validates whether software meets business needs and user expectations
  • NOT about finding technical bugs — about ensuring usability and business value
  • Fact: Over 60% of post-launch issues could be caught during UAT
  • Fact: Fixing a bug after release costs 6x more than during UAT

Who Performs UAT?

  • Business Users / End Users
  • Clients or Customers
  • Business Analysts (guide the process)
  • Sometimes dedicated UAT testers

UAT Process (6 Steps)

  1. Requirement Analysis: Review business requirements, define scope
  2. Test Planning: Who, when, how; prepare test environment
  3. Test Case Preparation: Simple business scenario language
  4. Test Execution: Users execute in UAT environment
  5. Defect Reporting: Mismatches reported to dev team
  6. Sign Off: Formal approval for production deployment

Challenges in UAT

  • Limited time — users busy with daily work
  • Unclear requirements make validation difficult
  • Environment must mirror production (costly)
  • Communication gaps between technical and business teams
  • Users may not know how to write proper defect reports
  • Scope creep — users request new features during UAT

  • Before UAT: Help prepare test cases, set up environment, prepare test data
  • During UAT: Support users, help reproduce issues, triage defects, track progress
  • After UAT: Consolidate feedback, verify fixes, prepare sign-off documentation

UAT Best Practices

  • Involve business users early in requirement gathering
  • Keep test cases in business language (not technical)
  • Use production-like data (anonymized if needed)
  • Set clear timelines and milestones
  • Have a dedicated UAT coordinator
  • Document all feedback formally (not verbal)
  • Define clear acceptance criteria upfront

UAT Interview Questions

Q1. What happens if UAT fails?

Defects are logged and sent back to development. Fixes are made and re-deployed to UAT environment. Users re-test the fixed scenarios. If critical requirements fail repeatedly, the release may be postponed. A "No-Go" decision is documented.

Q2. Can QA testers do UAT?

Technically, QA can assist but should NOT perform UAT alone. UAT must involve actual business users/end users because they validate from a business perspective. QA validates technical functionality; UAT validates business value and user acceptance.

Q3. How do you handle scope creep during UAT?

Document new requests separately as "enhancement requests" or new user stories. Do not mix with UAT defects. Communicate that new features will be addressed in the next sprint/release. Keep UAT focused on agreed acceptance criteria only.

Formal written approval from business stakeholders confirming that the software meets their requirements and is ready for production deployment. It includes: features validated, outstanding issues (with agreed workarounds), deployment date, rollback plan, and responsible parties.

Section 12: Tricky QA Interview Scenarios

20 high-signal QA interview questions that test how you think, problem-solve, and communicate — no single right answer.

Scenario-Based Questions

1. How would you test an application with no documentation?

Approach:

  1. Start with exploratory testing — navigate the app, understand flows
  2. Check similar applications in the market for expected behavior
  3. Talk to developers, BAs, POs for domain knowledge
  4. Create your own documentation/mind maps as you explore
  5. Use Session-Based Test Management (SBTM) for structured exploration
  6. Focus on critical business flows first
  7. Use browser dev tools to understand API calls and data flow

Key point for interview: Show that you're proactive, not dependent on documentation.

2. A developer says your bug isn't valid. What do you do?

Approach:

  1. Stay professional — never make it personal
  2. Provide clear evidence: screenshots, videos, logs, steps to reproduce
  3. Reference the requirement/user story and acceptance criteria
  4. Ask: "What is the expected behavior from your perspective?"
  5. If still disagreed, escalate to BA/PO for clarification
  6. Document the discussion and decision

Key point: Goal is shared understanding. If requirement is ambiguous, it's a documentation issue, not a win/lose situation.

3. How do you decide what NOT to test?

Risk-Based Testing approach:

  • Identify risk = (probability of failure) × (business impact)
  • Skip: stable modules with no changes, low-impact features, thoroughly tested legacy code
  • Always test: payment flows, user data, authentication, new features, recent bug fixes
  • Discuss with PO/stakeholders and document decisions in test plan
  • Use 80/20 rule: 80% of bugs found in 20% of modules

4. A bug is found in production. What do you check first?

Systematic approach:

  1. Assess impact: How many users affected? Is it a showstopper?
  2. Check application logs for error messages and stack traces
  3. Review recent deployments — what changed?
  4. Check monitoring tools (Grafana, Datadog, New Relic) for anomalies
  5. Reproduce in lower environment with same data/config
  6. Communicate with team, stakeholders about impact and ETA
  7. Post-fix: Root Cause Analysis + add regression test

5. What if you're the only QA on the team?

Strategy:

  • Prioritize: Critical business flows > integration points > edge cases
  • Automate smoke/regression tests ASAP (saves time long-term)
  • Promote quality culture: involve devs in testing (code reviews, unit tests)
  • Use risk-based approach — you can't test everything, so be smart
  • Shift-left: review requirements and designs early
  • Create lightweight processes (checklists, not heavy documentation)
  • Communicate clearly what's tested and what's not

6. Tests taking too long in CI. What would you do?

Optimization strategies:

  1. Parallel execution: Split tests across multiple agents/threads
  2. Test pyramid: More unit tests, fewer E2E tests
  3. Smart tagging: Run smoke on every PR, full suite nightly
  4. Optimize test setup/teardown: Use test fixtures efficiently
  5. Remove redundant tests: Audit for overlap
  6. Use headless browser for UI tests (faster than headed)
  7. Data setup: Use API calls instead of UI for preconditions

7. How would you test an API with no documentation?

Approach:

  • Use browser DevTools (Network tab) to capture API calls while using the app
  • Use Postman/Insomnia to replay captured requests
  • Analyze URL patterns to guess other endpoints
  • Try common CRUD operations (GET, POST, PUT, DELETE)
  • Check for Swagger/OpenAPI endpoints (/swagger-ui, /api-docs)
  • Look at error messages for clues about expected parameters
  • Test authentication by removing/modifying tokens

8. Bug found day before release. What now?

Decision framework:

SeverityAction
Critical (blocks core function)STOP release. Fix immediately. Retest. Communicate delay.
High (significant impact)Assess risk with PO. Fix if time allows, else delay or hotfix plan.
Medium (workaround exists)Document workaround. Release with known issue. Fix in next sprint.
Low (cosmetic)Log it. Release. Fix later.

Always: Communicate transparently with stakeholders. Document the decision.

9. How would you test a third-party integration?

  • Validate data exchange format (request/response contracts)
  • Test failure scenarios: timeout, invalid response, service down
  • Verify retry logic and fallback mechanisms
  • Test with edge case data (special chars, large payloads)
  • Monitor response times — third party might be slow
  • Use mocks when third party is unavailable/unstable
  • Verify error handling: what happens when 3rd party returns error?
  • Check data privacy: what data is sent to third party?

10. App is stable but users keep reporting bugs. What do you do?

Investigation steps:

  1. Check if users are on different environment/version than QA tested
  2. Analyze user data — real data often differs from test data
  3. Check device/browser combinations users are on
  4. Review user workflows — they may use app differently than expected
  5. Look for intermittent issues (network, timing, load)
  6. Set up error monitoring (Sentry, Bugsnag) for real-time crash reports
  7. Add the missed scenarios to regression suite

11. How do you know when testing is "done"?

Criteria for test completion:

  • All planned test cases executed (coverage ≥ 95%)
  • No critical/high severity defects open
  • Exit criteria from test plan met
  • Defect discovery rate declining (defect convergence)
  • Stakeholder confidence achieved
  • Risk level acceptable to business

Key point: Testing is never truly "done" — you stop when the risk of continuing testing is lower than the risk of releasing.

12. Multiple bugs in same area. What does that tell you?

Analysis:

  • Indicates defect clustering (Pareto principle: 80% bugs in 20% modules)
  • The module is likely complex, poorly designed, or recently changed
  • Increase test coverage in that area
  • Recommend code review or refactoring to developers
  • Add the area to regression priority list
  • Root cause: possibly unclear requirements, tight deadlines, or new team members

13. How handle duplicate bug reports?

Process:

  1. Mark the newer one as "Duplicate" and link to the original
  2. Combine additional details from the duplicate into the original
  3. Communicate with the reporter (acknowledgment, not dismissal)
  4. Multiple duplicates may indicate: unclear bug titles, poor searchability, or widespread visibility of the issue

14. Joining team with no automation. What's your plan?

90-day plan:

  1. Week 1-2: Understand the application, manual test flows, identify automation candidates
  2. Week 3-4: Set up framework (tool selection, project structure, CI/CD integration)
  3. Month 2: Automate smoke tests and top 10 critical flows
  4. Month 3: Expand to regression suite, create reporting, document standards

Selection criteria for automation: Repetitive, stable, high-risk, data-driven, time-consuming manually.

15. How deal with flaky tests?

Root causes and fixes:

Root CauseFix
Timing issuesUse explicit waits, not Thread.sleep
Test data dependencyCreate fresh data per test, cleanup after
Environment instabilityUse stable test environment, health checks
Race conditionsAdd synchronization, avoid shared state
Element not loadedWait for specific conditions, not fixed time
Order dependencyMake tests independent, use setup/teardown

Never ignore flaky tests — they erode trust in the automation suite.

16. Developer wants you to skip a test for faster release?

Response:

  1. Ask WHY — understand the pressure
  2. Communicate the RISK of skipping (what could break?)
  3. Offer alternatives: reduced test scope, parallel testing, automated quick check
  4. If the decision is to skip, ensure it's documented (who approved, what's the risk)
  5. Escalate to management if risk is unacceptable

Key: QA doesn't block releases; QA provides information for informed decisions.

17. New feature to test with only a Jira title?

Steps:

  1. Request acceptance criteria from PO/BA immediately
  2. Ask developer for a quick demo or walkthrough
  3. Check design mockups or wireframes if available
  4. Look at similar existing features for expected behavior
  5. Start exploratory testing while waiting for clarification
  6. Document your assumptions and get them validated

18. Bug suspected but not consistently reproducible?

Strategy for intermittent bugs:

  • Document all instances: time, data, environment, steps
  • Look for patterns: specific time of day? After certain actions? Under load?
  • Add detailed logging around the suspected area
  • Try to reproduce under different conditions (load, data volume, concurrent users)
  • Check for race conditions, memory leaks, or timeout issues
  • File the bug with all findings, label as "intermittent", assign medium-high priority
  • Set up monitoring/alerts for the specific error

19. How communicate QA value to product team?

Metrics and communication:

  • Share defect metrics: bugs caught before vs after release
  • Show cost savings: defect found in QA costs 10x less to fix than in production
  • Highlight customer satisfaction improvements
  • Report automation ROI (time saved per sprint)
  • Share examples: "This bug would have affected X users and cost $Y"
  • Quality dashboards visible to all stakeholders

20. Handed a legacy test suite you didn't write?

Assessment and action plan:

  1. Audit: Run the entire suite, note pass/fail/flaky rates
  2. Understand: Read code, map to features, identify patterns
  3. Categorize: Working (keep), Flaky (fix), Redundant (remove), Missing (add)
  4. Quick wins: Fix obvious failures, add documentation/comments
  5. Long-term: Refactor gradually, don't rewrite everything at once
  6. Standards: Establish naming conventions, structure for new tests
