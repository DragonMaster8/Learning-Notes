Section 1: What is Git & Why It Matters

Concept Explanation

Git is a Distributed Version Control System (DVCS) created by Linus Torvalds in 2005. It tracks every change to your files, allows you to revert to any previous state, and enables seamless collaboration among teams. Unlike centralized systems (SVN, TFS), every developer/tester has a full copy of the repository history locally.

Why Git is the Industry Standard

  • Collaboration — Multiple developers and testers work on the same project without overwriting each other's work
  • Safety — Full "time machine" capability; revert any mistake instantly
  • Full History — Who changed what, when, and why (with commit messages)
  • Branching & Merging — Experiment in isolated branches, then merge safely
  • Offline Work — Commit, branch, merge all work without internet
  • Speed — Operations are local; no network latency for commits/diffs/logs

Over 70% of developers use Git — it's a non-negotiable skill for automation testers.

Real-Time Example

You're an automation tester working on a Selenium framework. A developer pushes code that breaks the login page. Without Git, you'd have no way to know which change caused the failure. With Git, you can:

  1. Run git log --oneline to see recent changes
  2. Use git bisect to find the exact commit that introduced the bug
  3. Use git revert to undo the problematic commit
  4. Share the fix via a Pull Request for team review

Project-Based Example

In a typical e-commerce automation project with 5 team members:

  • Tester A writes login tests on feature/login-tests branch
  • Tester B writes cart tests on feature/cart-tests branch
  • Both push to remote, create PRs, CI runs tests automatically
  • After review and CI pass, both merge to main
  • No conflicts, no overwriting, full audit trail

Best Practices

  1. Commit often with small, logical changes — not giant blobs
  2. Write meaningful commit messages using Conventional Commits format
  3. Always git pull --rebase before pushing
  4. Use .gitignore from day one — never commit secrets or build artifacts
  5. Create branches for every feature/fix — never commit directly to main

Common Mistakes

  • Committing .env files with API keys or passwords
  • Force pushing to shared branches without communication
  • Vague commit messages like "fix", "update", "test"
  • Accumulating massive uncommitted changes for days
  • Ignoring merge conflicts instead of resolving them properly

Interview Questions

  1. What is Git and how is it different from SVN?
    Git is distributed (full repo on each machine), SVN is centralized (single server). Git works offline; SVN requires network for most operations.
  2. Why is Git important for an automation tester?
    Version control for test scripts, collaboration in teams, CI/CD integration, debugging with bisect, branching for parallel test development.
  3. What does "distributed" mean in DVCS?
    Every user has a complete copy of the repository including full history. No single point of failure.

Section 2: Git vs GitHub — The Core Distinction

Concept Explanation

Git and GitHub are often confused but serve different purposes. Understanding the distinction is critical for professional QA engineers.

Git — The Tool

  • A distributed version control system
  • Runs locally on your computer
  • Installed as software (CLI or GUI tools like GitKraken, SourceTree)
  • Works completely offline (commit, branch, merge, log)
  • Tracks file changes and manages versions
  • Handles branching and merging of code
  • No built-in collaboration interface
  • Open-source tool created by Linus Torvalds

GitHub — The Platform

  • A cloud-based hosting platform for Git repositories
  • Runs online (cloud servers)
  • Accessed via web browser, desktop apps, or Git commands
  • Requires internet for push, pull, and sync
  • Stores and backs up your repositories remotely
  • Adds collaboration features: Pull Requests, Issues, Actions (CI/CD), Discussions, Code Review
  • Owned by Microsoft

Real-Time Example

When you run git commit -m "test: add login tests" — that's Git (local).
When you run git push origin main — that's sending your local Git commits to GitHub (remote).
When a teammate opens a Pull Request for review — that's a GitHub feature (not Git).

  • Thinking you need internet to commit — commits are local Git operations
  • Confusing GitHub Desktop (GUI app) with Git (the VCS tool)
  • Assuming GitHub is the only way to use Git — it works perfectly standalone

  2. Can you use Git without GitHub?
    Yes. Git is fully functional locally. GitHub is just one of many hosting options.

Section 3: Git Fundamentals — The Three States Workflow

Concept Explanation

Git manages files through three distinct states. Understanding this workflow is fundamental to everything else in Git.

Architecture / Flow Diagram

┌───────────────┐ git add ┌───────────────┐ git commit ┌───────────────┐ │ Working │ ──────────────► │ Staging │ ──────────────► │ Repository │ │ Directory │ │ Area │ │ (.git) │ │ │ ◄────────────── │ (Index) │ │ │ │ (Modified) │ git restore │ (Staged) │ │ (Committed) │ └───────────────┘ └───────────────┘ └───────────────┘ │ │ │ git checkout / git restore │ ◄────────────────────────────────────────────────────────────────────┘

The Three States Explained

StateLocationDescriptionCommand to Move Forward
ModifiedWorking DirectoryFile is changed but not marked for commitgit add <file>
StagedStaging Area (Index)File is marked to be included in next commitgit commit -m "msg"
CommittedRepository (.git folder)Changes permanently saved in Git historyDone — snapshot created

The Four File Statuses

StatusMeaningExample
UntrackedGit sees the file but has never tracked itNew test file just created
TrackedAlready added/committed; can be modified, staged, or unmodifiedExisting test file in repo
IgnoredListed in .gitignore — Git completely ignores ittest-reports/, .env
StagedAdded to staging area, ready for commitAfter git add

Real-Time Example

# You edit a test file
vim tests/test_login.py          # File is now MODIFIED

# Check status
git status                       # Shows: modified: tests/test_login.py (red)

# Stage the file
git add tests/test_login.py      # File is now STAGED

# Check again
git status                       # Shows: modified: tests/test_login.py (green)

# Commit
git commit -m "test: add negative login scenarios"  # File is COMMITTED

# Check again
git status                       # Clean working tree — nothing to commit

Project-Based Example

In a test automation project, your daily workflow looks like:

  1. Pull latest changes: git pull --rebase
  2. Create/edit test files (Modified state)
  3. Review what changed: git diff
  4. Stage specific files: git add tests/login/
  5. Commit with meaningful message: git commit -m "test(login): add SSO validation tests"
  6. Push to remote: git push

Best Practices

  1. Always use git status before committing to verify what's staged
  2. Use git add -p (interactive staging) to stage only specific hunks of a file
  3. Never use git add . blindly — check git status first
  4. Stage related changes together; unrelated changes in separate commits
  5. Use git diff --staged to review exactly what will be committed

Common Mistakes

  • Using git add . and accidentally staging unwanted files (test reports, logs)
  • Committing without checking git status first
  • Not understanding that staging area is separate from working directory
  • Thinking git add is the same as committing

Hands-On Exercise

# Exercise: Practice the three states
mkdir git-states-practice && cd git-states-practice
git init

# Create a file (Untracked)
echo "def test_login(): pass" > test_login.py
git status   # Untracked file

# Stage it (Staged)
git add test_login.py
git status   # Staged for commit

# Commit it (Committed)
git commit -m "test: add initial login test"
git status   # Clean working tree

# Modify it (Modified)
echo "def test_logout(): pass" >> test_login.py
git status   # Modified (red)
git diff     # See the actual change
Quick Revision — Section 3 (12 items)
  1. Three states: Modified → Staged → Committed
  2. Working Directory = where you edit files
  3. Staging Area (Index) = prepared changes for next commit
  4. Repository (.git) = permanent history
  5. git add moves from Modified to Staged
  6. git commit moves from Staged to Committed
  7. Four file statuses: Untracked, Tracked, Ignored, Staged
  8. git status = always check before commit
  9. git diff = see unstaged changes
  10. git diff --staged = see what will be committed
  11. git add -p = interactive partial staging
  12. Staging area lets you craft precise, logical commits

Section 5: Branching & Merging Mastery

Concept Explanation

A branch is a lightweight movable pointer to a commit. It lets you work on different features/tests in isolation without affecting the main codebase. Branching in Git is extremely fast and cheap (just creates a 41-byte file).

Branch Commands

CommandAction
git branchList all local branches (* marks current)
git branch <name>Create new branch (stay on current)
git checkout <name>Switch to existing branch
git checkout -b <name>Create AND switch to new branch
git switch <name>Modern way to switch branches (Git 2.23+)
git switch -c <name>Modern way to create + switch
git branch -d <name>Delete branch (only if merged)
git branch -D <name>Force delete (even if unmerged)
git branch -aList all branches (local + remote)

Merging

# Standard merge workflow
git checkout main              # Switch to target branch
git merge feature/add-tests    # Merge feature into main

# Create merge commit even for fast-forward
git merge --no-ff feature/add-tests

Merge vs Rebase — Two Integration Strategies

AspectMergeRebase
What it doesCreates a new merge commit combining branchesReplays your commits on top of another branch
HistoryNon-linear (preserves true timeline with branches)Linear (clean, straight line)
Merge commitYes (extra commit created)No (commits are rewritten)
SafetySafe for shared branchesDANGEROUS for shared branches
Commit hashesPreservedChanged (rewritten)
Best forIntegrating into main / shared branchesCleaning up feature branches before merge
Golden Rule: Rebase YOUR private feature branch onto main. NEVER rebase main or any shared branch. Merge for integration into shared branches.

Architecture / Flow Diagram

MERGE: REBASE: A─B─C─────F (main) A─B─C─D─E' (main + feature rebased) \ / \ D─E (feature) D─E (old feature, abandoned) Merge creates commit F Rebase replays D,E as D',E' on top of C (combines C and E) (linear history, no merge commit)

Handling Merge Conflicts

# When merge conflicts occur:
git merge feature/tests
# CONFLICT: Merge conflict in tests/test_login.py

# 1. Open the conflicted file — look for markers:
# <<<<<<< HEAD
# (your changes)
# =======
# (their changes)
# >>>>>>> feature/tests

# 2. Manually resolve (keep one, both, or combine)
# 3. Stage the resolved file
git add tests/test_login.py

# 4. Complete the merge
git commit -m "merge: resolve conflict in login tests"

Real-Time Example (QA Team)

Two testers work on the same test file:

  1. Tester A on feature/login-negative adds negative test cases to test_login.py
  2. Tester B on feature/login-sso adds SSO tests to the same file
  3. Tester A merges first — no conflict
  4. Tester B merges — conflict in test_login.py (both added to same location)
  5. Tester B resolves: keeps both sets of tests in proper order

Common Mistakes

  • Rebasing main branch — rewrites shared history, breaks teammates' repos
  • Deleting branches before ensuring merge was successful
  • Not pulling latest main before creating a new branch
  • Working directly on main without creating a feature branch
  • Ignoring merge conflict markers (committing with <<<<<<< still in file)

Interview Questions

  1. What is the difference between git merge and git rebase?
    Merge creates a merge commit and preserves history. Rebase replays commits on top of target branch for linear history. Use merge for shared branches; rebase for private feature branches.
  2. When would you use --no-ff with merge?
    When you want to always create a merge commit (even if fast-forward is possible). This preserves branch topology in history — you can see where a feature branch was merged.
  3. How do you resolve a merge conflict?
    Open conflicted file, find <<<<<<< markers, manually choose correct code, remove markers, git add, git commit.
Quick Revision — Section 5 (15 items)
  1. Branch = lightweight pointer to a commit
  2. git checkout -b name = create + switch (classic)
  3. git switch -c name = create + switch (modern, Git 2.23+)
  4. Always create branches from updated main
  5. git merge = combine branches with merge commit
  6. git rebase = replay commits for linear history
  7. Merge is safe for shared branches; rebase only for private
  8. NEVER rebase main or any public branch
  9. --no-ff = always create merge commit (preserves topology)
  10. Conflicts show with <<<<<<< / ======= / >>>>>>> markers
  11. Resolve → add → commit to complete conflicted merge
  12. git branch -d = safe delete | -D = force delete
  13. git branch -a = see all branches (local + remote)
  14. Feature branch workflow: create → work → PR → merge → delete
  15. Pull latest main before creating new branch

Common Mistakes

  • Pushing without pulling first — results in rejected push
  • Using git push --force on shared branches — destroys teammates' work
  • Not setting upstream with -u on first push — subsequent git push fails
  • Cloning with HTTPS when SSH is configured (or vice versa) — authentication errors

Interview Questions

  1. What is the difference between git fetch and git pull?
    fetch downloads remote changes without merging. pull = fetch + merge. Use fetch when you want to inspect changes first.
  2. What does git push -u origin main do?
    Pushes the main branch to origin AND sets upstream tracking. After this, you can just use git push without specifying remote and branch.
  3. What is --force-with-lease?
    A safer alternative to --force. It only force-pushes if no one else has pushed to the branch since your last fetch. Prevents overwriting others' work.

Section 7: Advanced Git — Undo, Recover, Rewrite History

Undoing Changes — Complete Guide

CommandWhat It DoesChanges StateSafe for Shared?Use Case

git reset --soft HEAD~1Removes last commit, keeps changes stagedStagedNo (private only)Re-commit with different message/content

Architecture / Flow Diagram

RESET Modes Comparison: Commit History: A ← B ← C (HEAD) git reset --soft HEAD~1: A ← B (HEAD) Changes from C are STAGED git reset --mixed HEAD~1: A ← B (HEAD) Changes from C are UNSTAGED git reset --hard HEAD~1: A ← B (HEAD) Changes from C are DELETED ⚠️ git revert HEAD: A ← B ← C ← C' (HEAD) C' undoes C (safe, new commit)

Stashing — Temporary Shelf

Stash saves your uncommitted work temporarily so you can switch branches cleanly.

CommandAction
git stashSave uncommitted changes to stash stack
git stash listView all stashes
git stash popApply latest stash AND remove from stack
git stash applyApply latest stash but keep in stack
git stash drop stash@{1}Delete specific stash
git stash -m "description"Stash with meaningful name
git stash branch <name>Create branch from stash

Real-Time Example

# Scenario: You're mid-work on login tests when an urgent bug fix is needed

# Save current unfinished work
git stash -m "WIP: login negative tests incomplete"

# Switch to hotfix branch
git checkout -b hotfix/broken-checkout
# ... fix the bug, commit, push ...
git checkout main && git merge hotfix/broken-checkout

# Return to your work
git checkout feature/login-tests
git stash pop   # Restore your WIP changes

Recovering Lost Commits — Git Reflog

# git reflog shows ALL HEAD movements (even after reset --hard)
git reflog

# Example output:
# abc1234 HEAD@{0}: reset: moving to HEAD~1
# def5678 HEAD@{1}: commit: test: add payment tests    ← the "lost" commit

# Recover the lost commit
git reset --hard def5678   # or
git cherry-pick def5678    # Apply it as new commit
Lifesaver: git reflog keeps records for 90 days by default. If you accidentally reset --hard, you can STILL recover using reflog. Nothing in Git is truly lost (within 90 days).

Interview Questions

  1. How would you undo a public commit that broke CI?
    Use git revert <hash> — it creates a new commit that undoes the changes without rewriting history.
  2. Explain git reset --soft, --mixed, --hard.
    Soft: uncommit, keep staged. Mixed: uncommit, keep unstaged. Hard: uncommit AND delete changes. Only use on private/local branches.
  3. You accidentally ran git reset --hard and lost work. How to recover?
    Use git reflog to find the lost commit hash, then git reset --hard <hash> or git cherry-pick <hash> to recover.
  4. What is git stash used for?
    Temporarily shelving uncommitted changes to work on something else (like an urgent bugfix), then restoring later with git stash pop.
Quick Revision — Section 7 (15 items)

  2. git reset --soft HEAD~1 = uncommit, keep changes staged
  3. git reset --mixed HEAD~1 = uncommit, keep changes unstaged
  4. git reset --hard HEAD~1 = DELETE last commit + changes (destructive)
  5. git commit --amend = fix last commit (message or content)
  6. Use revert for public branches; reset for private only
  7. git stash = temporary shelf for WIP changes
  8. git stash pop = restore + remove | apply = restore + keep
  9. git stash -m "desc" = name your stashes
  10. git reflog = shows ALL HEAD movements (recovery tool)
  11. Reflog keeps data for 90 days — nothing truly lost
  12. Never reset --hard on shared branches
  13. Never amend commits already pushed to remote
  14. --force-with-lease safer than --force if you must force push
  15. Recovery: reflog → find hash → reset/cherry-pick

Section 9: Git Hooks — Shift-Left Testing

Concept Explanation

Git hooks are scripts that run automatically at certain Git events. They enforce quality gates locally before code ever reaches the remote repository. This is the essence of Shift-Left Testing — catching issues as early as possible.

Hook scripts live in .git/hooks/ directory (remove .sample extension to activate).

Hook Types for QA

HookWhen It RunsQA Use Case
pre-commitBefore a commit is createdLint test code, check for hardcoded secrets, enforce formatting
pre-pushBefore git pushRun smoke/unit tests — block push if tests fail
commit-msgAfter user enters commit messageValidate format (Conventional Commits), require Jira ticket ID
post-checkoutAfter git checkoutAuto-install dependencies, set env vars for test framework
post-mergeAfter a merge completesRun npm install if package.json changed

Example: pre-push Hook (Block Push if Tests Fail)

#!/bin/bash
# File: .git/hooks/pre-push (must be executable: chmod +x)
echo "🧪 Running smoke tests before push..."
pytest tests/smoke/ --maxfail=1 -q
if [ $? -ne 0 ]; then
    echo "❌ Tests failed. Push aborted."
    echo "Fix failing tests before pushing."
    exit 1
fi
echo "✅ All smoke tests passed. Pushing..."
exit 0

Example: commit-msg Hook (Enforce Conventional Commits)

#!/bin/bash
# File: .git/hooks/commit-msg
MSG=$(cat "$1")
PATTERN="^(feat|fix|test|refactor|chore|docs|style|ci)(\(.+\))?: .{10,}"
if ! echo "$MSG" | grep -qE "$PATTERN"; then
    echo "❌ Invalid commit message format!"
    echo "Expected: type(scope): description (min 10 chars)"
    echo "Example: test(login): add negative scenarios for empty password"
    exit 1
fi

Best Practices

  1. Keep pre-commit hooks FAST (<5 seconds) — linting and formatting only
  2. Run heavier tests (unit/integration) in pre-push, not pre-commit
  3. Use shared hook tools (Husky/pre-commit) instead of manual .git/hooks
  4. Always allow bypass with --no-verify for emergencies (but document why)
  5. Don't run full test suites in hooks — that belongs in CI/CD

  • Forgetting chmod +x on hook scripts (won't execute on Unix)
  • Not sharing hooks with team — only one person has quality gates
  • Blocking push with flaky tests — use only reliable smoke tests in hooks

Section 11: git bisect — Automated Bug Hunter

Concept Explanation

git bisect performs a binary search through commit history to find the exact commit that introduced a bug. Instead of checking every commit linearly (O(n)), it halves the range each step (O(log n)). For a QA tester, this is invaluable for identifying when a test started failing.

How Binary Search Works

100 commits to search: Linear search: check 1, 2, 3, ... → up to 100 checks Binary search: check 50, 25, 12, 6, 3, 1 → only 7 checks! Commits: [GOOD] ● ● ● ● ● ● ● ● [BAD] ● ● ● ● ● ● ● ● [CURRENT - BAD] ▲ bisect checks here first (middle) If middle is GOOD → bug is in right half If middle is BAD → bug is in left half Repeat until found!

Manual Bisect

# Start bisect session
git bisect start

# Mark current commit as bad (test is failing here)
git bisect bad

# Mark a known good commit (test was passing here)
git bisect good v1.0    # or a commit hash

# Git checks out a middle commit — you test manually:
# Run your test, then mark:
git bisect good   # if test passes on this commit
git bisect bad    # if test fails on this commit

# Repeat until Git identifies the first bad commit
# Output: "abc1234 is the first bad commit"

# End bisect session
git bisect reset

Automated Bisect (Most Powerful for QA)

# Fully automated — Git runs your test at each step
git bisect start
git bisect bad                          # Current is broken
git bisect good v1.0                    # This version was working
git bisect run pytest tests/test_login.py -x   # Run this test automatically

# Git will:
# 1. Checkout middle commit
# 2. Run pytest tests/test_login.py
# 3. If exit code 0 (pass) → mark as good
# 4. If exit code non-0 (fail) → mark as bad
# 5. Repeat until faulty commit found
# 6. Output: "abc1234 is the first bad commit"

git bisect reset   # Clean up

Handling Flaky Tests with --skip

# If a test is flaky (intermittent pass/fail), use skip:
git bisect start
git bisect bad
git bisect good v1.0

# During manual bisect, if result is ambiguous:
git bisect skip   # Skip this commit, try another

# For automated bisect with flaky tests:
# Exit code 125 = skip this commit
# Your script should return 125 for inconclusive results

Real-Time Example

Scenario: Login test test_login_valid_credentials started failing 3 days ago. 50 commits have been made since it last passed.

  1. Find the last known good commit: git log --oneline --since="3 days ago"
  2. Start automated bisect: git bisect start && git bisect bad && git bisect good <hash>
  3. git bisect run pytest tests/test_login.py::test_login_valid_credentials -x
  4. In ~6 steps (log2(50)), Git finds the exact commit that broke the test
  5. Share the commit hash with the developer: "This commit broke login tests"

Interview Questions

  1. What is git bisect and how does a tester use it?
    Binary search through commits to find which one introduced a bug. QA can automate it with a test script — Git runs the test at each step and identifies the faulty commit.
  2. How many steps does bisect take for 1000 commits?
    log2(1000) ≈ 10 steps. Binary search is extremely efficient.
  3. What exit code should a bisect script return for skip?
    Exit code 125 tells Git to skip the current commit (useful for flaky tests or build failures).
Quick Revision — Section 11 (10 items)
  1. git bisect = binary search for bug-introducing commit
  2. O(log n) complexity — 7 steps for 100 commits, 10 for 1000
  3. git bisect start → bad → good → test → mark → repeat
  4. git bisect run <script> = fully automated
  5. Exit 0 = good, non-0 = bad, 125 = skip
  6. git bisect reset = end session, return to original state
  7. Perfect for finding "when did this test start failing?"
  8. git bisect skip = skip ambiguous/flaky commits
  9. Share faulty commit with dev team for quick resolution
  10. Most underused yet most powerful QA debugging tool

Section 12: Cherry-Pick, Git LFS & Submodules

git cherry-pick — Selective Commit Deployment

Apply a specific commit from one branch to another without merging the entire branch.

# Apply a specific commit to current branch
git checkout qa
git cherry-pick <commit-hash>   # Bring just that test fix into qa

# Cherry-pick multiple commits
git cherry-pick abc123 def456 ghi789

# Cherry-pick without committing (stage only)
git cherry-pick --no-commit <hash>

Use Cases for QA

  • Critical test fix needed on release branch, but develop has unstable tests — cherry-pick only the fix
  • Backport a test improvement to an older version branch
  • Apply a specific bug fix from one team member's branch

Git LFS — Large File Storage

Git is not designed for large binary files. Git LFS replaces large files with text pointers in the repo, storing actual files on a separate server.

# Install Git LFS (one-time)
git lfs install

# Track large files by pattern
git lfs track "*.iso"
git lfs track "test-data/*.csv"
git lfs track "*.mp4"          # Recorded test videos

# Commit the .gitattributes (tracking config)
git add .gitattributes
git commit -m "chore: track large test files with LFS"

# Then use git normally — LFS handles the rest
git add test-data/large-dataset.csv
git commit -m "test: add performance test dataset"
git push

Git Submodules — Shared Test Libraries

When multiple projects share a common test utility library, use submodules to include it.

# Add shared library as submodule

on:

        python-version: ['3.10', '3.11', '3.12']

      - name: Set up Python
        uses: actions/setup-python@v5
        with:

        run: |

      - name: Run tests
        run: |
          pytest tests/ --junitxml=test-results.xml --html=report.html

      - name: Upload test report

        with:
          name: test-report

Scenario-Based Questions (4-Year Experience Level)

Q1: You accidentally committed a config file with an API key. What do you do?

  1. Immediately revoke the key (security first)
  2. git reset --soft HEAD~1 to undo the commit, keep files staged
  3. Add the file to .gitignore
  4. Remove the key from the file, recommit
  5. If already pushed: git push --force-with-lease + notify team to re-pull
  6. Consider using git filter-branch or BFG Repo Cleaner to remove from ALL history

Q2: Test suite takes 30 minutes. How to avoid blocking developers during push?

  1. Use pre-push hook for fast smoke tests only (<2 minutes)
  2. Run full suite in CI/CD triggered by push (non-blocking)
  3. Implement parallel test execution in CI
  4. Use test caching — only run tests affected by changed files
  5. Tag critical tests for pre-push; rest for CI only

Q3: A flaky test fails nondeterministically. How to find the culprit commit?

  1. Use git bisect run with a script that runs the test multiple times
  2. Script returns exit 0 if passes 3/3 times, exit 1 if fails, exit 125 if ambiguous
  3. Alternatively, use git bisect skip for ambiguous commits
  4. Isolate the test and check environment-dependent factors

Q4: Share a common test utility library across 5 microservice repos?

  1. Option 1: Git Submodules — include repo within repo at specific commit
  2. Option 2: Package Manager — publish as npm/pip package with versioned tags
  3. Option 3: Monorepo — all services + shared libs in one repo
  4. Recommendation: Package manager for mature teams (proper versioning, dependency management)

Q5: Two testers modify the same test file. How to handle the merge conflict?

  1. Use feature branches — each tester on separate branch
  2. First merger faces no conflict; second tester resolves
  3. Open conflicted file, look for <<<<<<< markers
  4. Usually both changes should be kept (additive test cases)
  5. After resolving: git add → git commit
  6. Prevention: split test files by feature area (login_tests.py, cart_tests.py)

Q6: How do you ensure code quality in your test automation repo?

  • Pre-commit hooks: linting (pylint/eslint), formatting (black/prettier)
  • Pre-push hooks: smoke test execution
  • Branch protection: require CI pass + 1 reviewer approval
  • Conventional Commits enforced via commit-msg hook
  • PR templates with checklist (tests pass, docs updated, no secrets)
  • Automated code coverage reporting on PRs

FAQs — Frequently Asked Questions

Q: Can I undo a git reset --hard?

Yes! Use git reflog to find the commit hash before the reset, then git reset --hard <hash>. Reflog keeps data for 90 days.

Q: Should I use merge or rebase?

Rebase for YOUR private feature branch (clean history). Merge for integrating into shared branches (main, develop). Never rebase public/shared branches.

Q: How often should I commit?

Every logical unit of work. Completed a test case? Commit. Fixed a bug? Commit. Don't accumulate a full day's work in one commit.

Q: What if I accidentally pushed secrets to GitHub?

1) Revoke the secret immediately. 2) Use BFG Repo Cleaner or git filter-branch to remove from ALL history. 3) Force push. 4) Notify team. GitHub also has secret scanning alerts.

Q: How do I keep my fork in sync with the original repo?

git remote add upstream <original-repo-url>
git fetch upstream
git rebase upstream/main
git push origin main

Q: What's the difference between git switch and git checkout?

git switch (Git 2.23+) is dedicated to branch switching. git checkout does many things (switch branch, restore files, detach HEAD). Use switch for clarity.

Q: How to see who wrote a specific line of code?

git blame <file>              # Every line with author + date
git log -p -S "search text"   # Find commits that added/removed specific text

Q: How to squash multiple commits into one before merging?

# Interactive rebase — squash last 3 commits
git rebase -i HEAD~3
# In editor: change 'pick' to 'squash' for commits to combine
