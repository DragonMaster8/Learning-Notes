Section 11: File Handling

Why File Handling in Automation?

In test automation frameworks, almost every layer touches files. Configuration drives browser selection, URLs, and timeouts. Test data lives in Excel, JSON, or CSV. Reports and logs capture execution results. Screenshots attach to failure reports. A robust framework abstracts file I/O behind utility classes so tests stay clean and maintainable.

Framework Rule: Never hardcode paths or read files directly inside test methods. Centralize all file access in dedicated reader/writer utilities under a utils or config package.

Common File Types in Automation

  • Properties — environment config (browser, URL, timeouts)
  • JSON — API payloads, REST test data, schema validation
  • Excel — data-driven UI tests (TestNG DataProvider)
  • CSV — lightweight test data exports
  • Text/Log — log analysis, baseline comparisons

Properties File — Most Used in Automation

The .properties file is the backbone of framework configuration. Key-value pairs are simple, human-readable, and require no external library.

Complete ConfigReader Utility (Singleton Pattern)

import java.io.FileInputStream;
import java.io.IOException;
import java.util.Properties;

public class ConfigReader {

    private static Properties prop;

    static {
        try {
            prop = new Properties();
            prop.load(new FileInputStream("src/test/resources/config/config.properties"));
        } catch (IOException e) {
            throw new RuntimeException("Failed to load config.properties", e);
        }
    }

    public static String get(String key) {
        return prop.getProperty(key);
    }

    public static String get(String key, String defaultValue) {
        return prop.getProperty(key, defaultValue);
    }

    public static int getInt(String key) {
        return Integer.parseInt(prop.getProperty(key));
    }

    public static boolean getBoolean(String key) {
        return Boolean.parseBoolean(prop.getProperty(key));
    }
}

public class BaseTest {
    protected WebDriver driver;

    @BeforeMethod
    public void setup() {
        String browser = ConfigReader.get("browser");
        String url = ConfigReader.get("url");
        driver = DriverFactory.createDriver(browser);
        driver.get(url);

}

Environment-Specific Configs

Maintain separate property files per environment and load dynamically:

  • qa.properties — QA URLs, test credentials
  • staging.properties — pre-production settings
  • prod.properties — production smoke test config (read-only)
public class ConfigReader {

    private static Properties prop;

    static {
        String env = System.getProperty("env", System.getenv("ENV"));
        if (env == null) env = "qa";

        String filePath = "src/test/resources/config/" + env + ".properties";
        try {
            prop = new Properties();
            prop.load(new FileInputStream(filePath));
        } catch (IOException e) {
            throw new RuntimeException("Failed to load: " + filePath, e);
        }
    }

    public static String get(String key) {
        return prop.getProperty(key);
    }
}

Run with: mvn test -Denv=staging or set ENV=staging in CI pipeline.

JSON File Handling

JSON is standard for API automation — request bodies, response validation, and structured test data. Use Jackson (ObjectMapper) in Maven/Gradle projects.

Reading Test Data from JSON

// testdata/login_users.json
// [
//   {"username": "admin", "password": "admin123", "expectedUrl": "/dashboard"},
//   {"username": "guest", "password": "guest", "expectedUrl": "/home"}
// ]

import com.fasterxml.jackson.core.type.TypeReference;
import com.fasterxml.jackson.databind.ObjectMapper;
import java.io.File;
import java.util.List;

public class JsonDataReader {

    private static final ObjectMapper mapper = new ObjectMapper();

    public static <T> T read(String filePath, Class<T> clazz) {
        try {
            return mapper.readValue(new File(filePath), clazz);
        } catch (Exception e) {
            throw new RuntimeException("Failed to read JSON: " + filePath, e);
        }
    }

    public static <T> List<T> readList(String filePath, Class<T> clazz) {
        try {
            return mapper.readValue(new File(filePath),
                mapper.getTypeFactory().constructCollectionType(List.class, clazz));
        } catch (Exception e) {
            throw new RuntimeException("Failed to read JSON list: " + filePath, e);
        }
    }

    public static String toJson(Object obj) {
        try {
            return mapper.writeValueAsString(obj);
        } catch (Exception e) {
            throw new RuntimeException("Failed to serialize object", e);
        }
    }
}

Excel File Handling (Apache POI)

Excel remains the most common format for business-authored test data. Apache POI provides XSSFWorkbook for .xlsx files.

Complete ExcelReader Class

import org.apache.poi.ss.usermodel.*;
import org.apache.poi.xssf.usermodel.XSSFWorkbook;
import java.io.FileInputStream;
import java.util.ArrayList;
import java.util.HashMap;
import java.util.List;
import java.util.Map;

public class ExcelReader {

    private String filePath;
    private String sheetName;

    public ExcelReader(String filePath, String sheetName) {
        this.filePath = filePath;
        this.sheetName = sheetName;
    }

    public int getRowCount() {
        try (FileInputStream fis = new FileInputStream(filePath);
             Workbook workbook = new XSSFWorkbook(fis)) {
            Sheet sheet = workbook.getSheet(sheetName);
            return sheet.getLastRowNum();
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }

    public String getCellData(int rowNum, int colNum) {
        try (FileInputStream fis = new FileInputStream(filePath);
             Workbook workbook = new XSSFWorkbook(fis)) {
            Sheet sheet = workbook.getSheet(sheetName);
            Row row = sheet.getRow(rowNum);
            Cell cell = row.getCell(colNum);

            switch (cell.getCellType()) {
                case STRING:  return cell.getStringCellValue();
                case NUMERIC: return String.valueOf((int) cell.getNumericCellValue());
                case BOOLEAN: return String.valueOf(cell.getBooleanCellValue());
                default:      return "";
            }
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }

    public Object[][] getTestData() {
        int rowCount = getRowCount();
        int colCount = getColumnCount();
        Object[][] data = new Object[rowCount][colCount];

        for (int i = 1; i <= rowCount; i++) { // skip header row 0
            for (int j = 0; j < colCount; j++) {
                data[i - 1][j] = getCellData(i, j);
            }
        }
        return data;
    }

    public List<Map<String, String>> getTestDataAsMaps() {
        List<Map<String, String>> rows = new ArrayList<>();
        int colCount = getColumnCount();

        // Build header map
        List<String> headers = new ArrayList<>();
        for (int j = 0; j < colCount; j++) {
            headers.add(getCellData(0, j));
        }

        for (int i = 1; i <= getRowCount(); i++) {
            Map<String, String> row = new HashMap<>();
            for (int j = 0; j < colCount; j++) {
                row.put(headers.get(j), getCellData(i, j));
            }
            rows.add(row);
        }
        return rows;
    }

    private int getColumnCount() {
        try (FileInputStream fis = new FileInputStream(filePath);
             Workbook workbook = new XSSFWorkbook(fis)) {
            return workbook.getSheet(sheetName).getRow(0).getLastCellNum();
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }
}

import com.opencsv.CSVReader;

public class CsvReader {

    public static List<String[]> readAll(String filePath) {
        try (CSVReader reader = new CSVReader(new FileReader(filePath))) {
            List<String[]> rows = new ArrayList<>();
            String[] line;
            reader.readNext(); // skip header
            while ((line = reader.readNext()) != null) {
                rows.add(line);
            }
            return rows;

            throw new RuntimeException(e);
        }
    }
}

Interview Q&A — File Handling

  1. Q: Why use properties files instead of hardcoding values?
    A: Separation of config from code; change environment without recompiling; CI can inject different configs per pipeline stage.
  2. Q: How do you load different configs for QA vs Prod?
    A: Pass -Denv=qa or read ENV environment variable; load corresponding {env}.properties file.
  3. Q: What is the Singleton pattern in ConfigReader?
    A: Static block loads properties once at class load time; single shared Properties instance across all tests.
  4. Q: Jackson vs Gson — which do you prefer?
    A: Jackson is more feature-rich (annotations, streaming); Gson is simpler. Most enterprise frameworks use Jackson.
  5. Q: How do you handle numeric cells in Excel?
    A: Check CellType.NUMERIC; cast or use DataFormatter to avoid decimal formatting issues.
  6. Q: Why skip row 0 in Excel DataProvider?
    A: Row 0 typically contains column headers, not test data.
  7. Q: How do you read JSON array into List of objects?
    A: Use ObjectMapper.readValue() with TypeReference<List<MyClass>> or constructCollectionType.
  8. Q: When to use CSV over Excel?
    A: Simple data, version-control friendly (text diffs), no POI dependency, faster parsing.
  9. Q: Files.readAllLines vs BufferedReader?
    A: readAllLines loads entire file into memory — fine for small files. BufferedReader streams line-by-line — required for large log files.
  10. Q: How do you write test results to a file?
    A: Use Files.write() with StandardOpenOption.APPEND for incremental logging, or a reporting library like ExtentReports.
  11. Q: Where should config files live in a Maven project?
    A: src/test/resources/config/ — available on classpath at runtime.
  12. Q: How do you handle file-not-found in framework utilities?
    A: Wrap in RuntimeException with descriptive message; fail fast during test setup rather than mid-test.

Section 12: Java 8 Features — Very Important for SDET Interviews

Why Java 8 Matters for Automation

Java 8 introduced lambda expressions, functional interfaces, and the Streams API — transforming how automation code is written. Explicit waits become one-liners. Element filtering replaces verbose loops. API response processing becomes declarative. Interviewers expect SDET candidates to fluently use these features.

Interview Tip: Be ready to rewrite a traditional for-loop as a Stream pipeline. This is the #1 Java 8 coding question in SDET interviews.

Lambda Expressions

Syntax: (params) -> expression or (params) -> { statements; }

Coding Questions

1. Filter Even Numbers from List

// Stream solution — O(n)
List<Integer> evens = numbers.stream()
    .filter(n -> n % 2 == 0)
    .collect(Collectors.toList());

// Traditional — O(n)
List<Integer> evens = new ArrayList<>();
for (int n : numbers) {
    if (n % 2 == 0) evens.add(n);
}

2. Find Duplicates in List

// Stream solution — O(n)
List<String> duplicates = list.stream()
    .collect(Collectors.groupingBy(e -> e, Collectors.counting()))
    .entrySet().stream()
    .filter(e -> e.getValue() > 1)
    .map(Map.Entry::getKey)
    .collect(Collectors.toList());

// Traditional — O(n²) with nested loops, O(n) with HashSet
Set<String> seen = new HashSet<>();
Set<String> dups = new HashSet<>();
for (String s : list) {
    if (!seen.add(s)) dups.add(s);
}

3. Sort Employees by Salary

// Stream solution — O(n log n)
List<Employee> sorted = employees.stream()
    .sorted(Comparator.comparing(Employee::getSalary).reversed())
    .collect(Collectors.toList());

// Traditional — O(n log n)
Collections.sort(employees, (e1, e2) -> e2.getSalary() - e1.getSalary());

4. Count Occurrences of Each Word

// Stream solution — O(n)
Map<String, Long> wordCount = words.stream()
    .collect(Collectors.groupingBy(w -> w, Collectors.counting()));

// Traditional — O(n)
Map<String, Integer> wordCount = new HashMap<>();
for (String w : words) {
    wordCount.merge(w, 1, Integer::sum);
}

5. Find Highest Salary

// Stream solution — O(n)
Optional<Employee> highest = employees.stream()
    .max(Comparator.comparing(Employee::getSalary));

// Traditional — O(n)
Employee highest = employees.get(0);
for (Employee e : employees) {
    if (e.getSalary() > highest.getSalary()) highest = e;
}

Interview Q&A — Java 8

  1. Q: What is a lambda expression?
    A: Anonymous function implementing a functional interface. Syntax: (params) -> body.
  2. Q: Difference between intermediate and terminal stream operations?
    A: Intermediate ops (filter, map) return a Stream and are lazy. Terminal ops (collect, forEach) produce a result and trigger execution.
  3. Q: Is Stream reusable?
    A: No. Once a terminal operation is called, the stream is consumed. Create a new stream for another pipeline.
  4. Q: parallelStream() vs stream()?
    A: parallelStream uses ForkJoinPool for concurrent processing. Use for large datasets with CPU-bound operations; avoid with I/O-bound tasks like Selenium.
  5. Q: What is Optional and why use it?
    A: Container that may or may not contain a value. Prevents NullPointerException; forces explicit null handling.
  6. Q: map() vs flatMap()?
    A: map transforms 1-to-1. flatMap transforms 1-to-many and flattens the result (e.g., nested lists to single list).
  7. Q: What is a method reference?
    A: Shorthand for lambda calling an existing method: WebElement::getText instead of e -> e.getText().
  8. Q: Collectors.toList() vs toSet()?
    A: toList preserves order and allows duplicates. toSet removes duplicates; order not guaranteed (unless LinkedHashSet).
  9. Q: Can you use streams with WebDriver findElements?
    A: Yes — findElements returns List<WebElement>, which has .stream() in Java 8+.
  10. Q: What is @FunctionalInterface?
    A: Annotation ensuring interface has exactly one abstract method. Compiler error if violated. Allows lambda usage.
  11. Q: Predicate vs Function difference?
    A: Predicate returns boolean (test). Function returns any type (apply). Predicate filters; Function transforms.
  12. Q: How does groupingBy work?
    A: Groups stream elements into Map by classifier function. Optional downstream collector for aggregation (counting, toList).
  13. Q: findFirst() vs findAny()?
    A: findFirst returns first element in encounter order. findAny returns any element — useful in parallel streams.
  14. Q: What is peek() used for?
    A: Debugging — perform side-effect (logging) without modifying stream. Elements pass through unchanged.
  15. Q: Comparator.comparing() vs lambda comparator?
    A: comparing() is cleaner, composable (thenComparing, reversed). Preferred in Java 8+ code.
  16. Q: Why avoid get() on Optional?
    A: Throws NoSuchElementException if empty. Prefer orElse(), orElseGet(), or ifPresent().

  • TestNG parallel execution — parallel="methods" runs tests in separate threads
  • Selenium Grid — distributes tests across nodes/threads
  • API load testing — multiple concurrent requests
  • Background tasks — screenshot capture, report generation
Danger: Shared resources (WebDriver, report writers, counters) become race-condition targets when tests run in parallel. Thread safety is non-negotiable in parallel frameworks.

Synchronization

public class ReportWriter {
    private static int passCount = 0;

    // Method-level synchronization
    public static synchronized void incrementPass() {
        passCount++; // Only one thread at a time
    }

    // Block-level synchronization
    public static void writeToFile(String result) {
        synchronized (ReportWriter.class) {
            Files.write(path, result.getBytes(), StandardOpenOption.APPEND);
        }
    }
}

Deadlock

Deadlock occurs when Thread A holds Lock 1 and waits for Lock 2, while Thread B holds Lock 2 and waits for Lock 1. Both threads block forever.

Prevention: Always acquire locks in the same order. Use timeout locks. Prefer ThreadLocal over shared synchronized resources.

Thread Safety in Automation

// THREAD-UNSAFE — DO NOT DO THIS
public class BadBaseTest {
    protected static WebDriver driver; // Shared across all threads!

    @BeforeMethod
    public void setup() {
        driver = new ChromeDriver(); // Thread B overwrites Thread A's driver
    }
}

// THREAD-SAFE — use ThreadLocal (see Section 14)
public class GoodBaseTest {
    @BeforeMethod
    public void setup() {
        DriverFactory.setDriver("chrome"); // Each thread gets own driver
    }

    @AfterMethod
    public void teardown() {
        DriverFactory.quitDriver();
    }
}

ExecutorService

import java.util.concurrent.*;

public class ParallelApiRunner {

    public void runParallelHealthChecks(List<String> endpoints) {
        ExecutorService pool = Executors.newFixedThreadPool(5);

        for (String endpoint : endpoints) {
            pool.submit(() -> {
                int status = RestAssured.get(endpoint).statusCode();
                System.out.println(endpoint + " -> " + status);
            });
        }

        pool.shutdown();
        try {
            if (!pool.awaitTermination(60, TimeUnit.SECONDS)) {
                pool.shutdownNow();
            }
        } catch (InterruptedException e) {
            pool.shutdownNow();
        }
    }
}

Interview Q&A — Multithreading

  1. Q: Thread vs Runnable — which to use?
    A: Runnable — allows extending another class; better design separation. Thread is acceptable for simple, standalone tasks.
  2. Q: start() vs run()?
    A: start() creates new thread and JVM calls run(). Calling run() directly executes in current thread — no parallelism.
  3. Q: Why is static WebDriver dangerous in parallel tests?
    A: All threads share one instance. Thread A's navigation affects Thread B's test. Causes random failures.
  4. Q: What is a race condition?
    A: Outcome depends on timing of thread execution. Two threads modifying shared counter simultaneously may lose updates.
  5. Q: synchronized method vs synchronized block?
    A: Method locks on object (instance) or class (static). Block locks on specified object — finer-grained control.
  6. Q: How does TestNG parallel execution work?
    A: TestNG creates thread pool; each test method runs in separate thread. Requires thread-safe driver management.
  7. Q: What is ExecutorService?
    A: Thread pool manager. Submits tasks, reuses threads, provides shutdown/awaitTermination for clean completion.
  8. Q: What is deadlock and how to detect it?
    A: Circular lock dependency. Detect with thread dumps (jstack), VisualVM, or avoid nested lock acquisition.
  9. Q: Can you parallelize Selenium UI tests on one machine?
    A: Yes with TestNG parallel + ThreadLocal drivers. Each thread needs its own browser instance.
  10. Q: sleep() vs wait()?
    A: sleep() is Thread class method — pauses without releasing lock. wait() is Object method — releases lock, waits for notify().
  11. Q: What is thread pool size for API tests?
    A: Depends on target server capacity. Start with 5-10; monitor response times and errors; tune accordingly.
  12. Q: How does Selenium Grid handle threading?
    A: Hub receives requests; nodes run browsers. Each session is isolated — Grid handles thread/session mapping.

Section 14: ThreadLocal — Critical for Parallel Execution

What is ThreadLocal?

ThreadLocal provides thread-confined storage — each thread accessing the variable gets its own independent copy. Changes by one thread are invisible to others. It is the cornerstone of parallel test automation.

Senior SDET Must-Know: ThreadLocal is asked in virtually every senior automation interview. If you run parallel tests, you MUST understand ThreadLocal.

Why ThreadLocal is the Most Important Concept

Problem: Shared Static WebDriver

Thread-1 (testLogin)  ──> static WebDriver ──> Chrome Browser
Thread-2 (testCheckout) ──> static WebDriver ──> SAME Chrome Browser  ← CHAOS!
Thread-3 (testSearch)   ──> static WebDriver ──> SAME Chrome Browser

Thread-1 navigates to /login
Thread-2 navigates to /checkout  ← overwrites Thread-1's page!
Thread-1 assertion fails randomly

Solution: ThreadLocal Isolation

Thread-1 ──> ThreadLocal copy ──> Chrome Browser A  (isolated)
Thread-2 ──> ThreadLocal copy ──> Chrome Browser B  (isolated)
Thread-3 ──> ThreadLocal copy ──> Chrome Browser C  (isolated)

Each thread operates on its own browser — no interference

ThreadLocal API

// Create ThreadLocal variable
ThreadLocal<WebDriver> driverThread = new ThreadLocal<>();

// withInitial — lazy initialization with default
ThreadLocal<WebDriver> driverThread = ThreadLocal.withInitial(() -> null);

// set(T value) — store value for CURRENT thread
driverThread.set(new ChromeDriver());

// get() — retrieve value for CURRENT thread
WebDriver driver = driverThread.get();

// remove() — CRITICAL! Cleanup when thread is done
driverThread.remove(); // Prevents memory leak in thread pools

Complete DriverFactory with ThreadLocal

import org.openqa.selenium.WebDriver;
import org.openqa.selenium.chrome.ChromeDriver;
import org.openqa.selenium.firefox.FirefoxDriver;
import org.openqa.selenium.edge.EdgeDriver;

public class DriverFactory {

    private static ThreadLocal<WebDriver> driver = new ThreadLocal<>();

    public static WebDriver getDriver() {
        return driver.get();
    }

    public static void setDriver(String browser) {
        switch (browser.toLowerCase()) {
            case "chrome":
                driver.set(new ChromeDriver());
                break;
            case "firefox":
                driver.set(new FirefoxDriver());
                break;
            case "edge":
                driver.set(new EdgeDriver());
                break;
            default:
                driver.set(new ChromeDriver());
        }
        getDriver().manage().window().maximize();
    }

    public static void quitDriver() {
        if (driver.get() != null) {
            driver.get().quit();
            driver.remove(); // Prevent memory leak!
        }
    }
}

Interview Q&A — ThreadLocal

  1. Q: What is ThreadLocal?
    A: Class providing thread-confined variables. Each thread has its own copy; no sharing between threads.
  2. Q: Why is ThreadLocal essential for parallel Selenium?
    A: WebDriver is not thread-safe. ThreadLocal gives each test thread its own driver instance.
  3. Q: What happens if you forget remove()?
    A: Memory leak. Thread pool reuses threads; old WebDriver references remain, preventing GC. Causes OutOfMemoryError in long CI runs.
  4. Q: ThreadLocal vs synchronized — which is better for WebDriver?
    A: ThreadLocal — no locking overhead, true isolation. Synchronized serializes access — defeats purpose of parallel execution.
  5. Q: Can two threads see each other's ThreadLocal values?
    A: No. Each thread has completely independent storage.
  6. Q: Where do you call set() and remove()?
    A: set() in @BeforeMethod. remove() in @AfterMethod after quit().
  7. Q: Is ThreadLocal used only for WebDriver?
    A: No. Also used for ExtentTest reporting, RestAssured RequestSpecification, scenario context, soft assertions.
  8. Q: ThreadLocal.withInitial() vs manual set()?
    A: withInitial provides default value on first get() without set(). Manual set() gives explicit control — preferred for WebDriver.
  9. Q: How does TestNG thread pool interact with ThreadLocal?
    A: TestNG reuses threads from pool. Without remove(), previous test's data leaks into next test on same thread.
  10. Q: ThreadLocal in inheritance — do child threads inherit?
    A: No by default. Child threads get null unless using InheritableThreadLocal.
  11. Q: Static ThreadLocal variable — is it shared?
    A: The ThreadLocal object is static (one instance), but each thread's stored value is separate. Static refers to the container, not the value.
  12. Q: How to debug ThreadLocal issues?
    A: Log thread name + driver hashCode in @BeforeMethod/@AfterMethod. Verify each thread gets unique driver.
  13. Q: ThreadLocal vs dependency injection for driver?
    A: ThreadLocal is simpler for TestNG. DI (Spring, PicoContainer) works but adds complexity. ThreadLocal is industry standard for TestNG frameworks.
  14. Q: Can ThreadLocal cause memory leak with parallel=methods?
    A: Yes, if remove() is skipped. Thread pool keeps threads alive; unreleased WebDriver instances accumulate.
  15. Q: What is InheritableThreadLocal?
    A: Subclass where child threads inherit parent's value. Rarely needed in test automation.

Section 15: equals() and hashCode() Contract

Why Important for Automation

When comparing test data objects, using objects as HashMap keys, or performing Set operations on custom classes, correct equals() and hashCode() implementations are critical. Incorrect overrides cause silent bugs — tests pass/fail unpredictably.

equals() — Default vs Override

// Default equals() — reference comparison (same as ==)
Employee e1 = new Employee("John", 50000);
Employee e2 = new Employee("John", 50000);
e1.equals(e2); // false — different objects in memory

// Override equals() — value comparison
@Override
public boolean equals(Object obj) {
    if (this == obj) return true;
    if (obj == null || getClass() != obj.getClass()) return false;
    Employee other = (Employee) obj;
    return salary == other.salary && Objects.equals(name, other.name);
}

equals() Contract

  • Reflexive: x.equals(x) is true
  • Symmetric: x.equals(y) ↔ y.equals(x)
  • Transitive: x.equals(y) and y.equals(z) → x.equals(z)
  • Consistent: Multiple calls return same result
  • Null: x.equals(null) is false

hashCode()

// Default hashCode — based on memory address
e1.hashCode() != e2.hashCode(); // even if fields are identical

// Override — consistent with equals()
@Override
public int hashCode() {
    return Objects.hash(name, salary);
}

The Contract

Golden Rule: If you override equals(), you MUST override hashCode(). If a.equals(b) is true, then a.hashCode() == b.hashCode() must be true. The reverse is NOT required — hash collision is allowed.

Bug Demo — Missing hashCode()

public class BadEmployee {
    private String name;
    // equals() overridden but hashCode() NOT overridden

    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (!(obj instanceof BadEmployee)) return false;
        return Objects.equals(name, ((BadEmployee) obj).name);
    }
    // hashCode() uses default — memory address!
}

// THE BUG:
BadEmployee e1 = new BadEmployee("John");
BadEmployee e2 = new BadEmployee("John");
e1.equals(e2);           // true — values match
e1.hashCode() == e2.hashCode(); // FALSE — different addresses!

Set<BadEmployee> set = new HashSet<>();
set.add(e1);
set.contains(e2);        // FALSE! Set can't find e2 even though equals says they're same

Map<BadEmployee, String> map = new HashMap<>();
map.put(e1, "Engineering");
map.get(e2);             // NULL! Wrong bucket due to hash mismatch

Complete Employee Class

import java.util.Objects;

public class Employee {
    private String name;
    private int salary;
    private String department;

    public Employee(String name, int salary, String department) {
        this.name = name;
        this.salary = salary;
        this.department = department;
    }

    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (obj == null || getClass() != obj.getClass()) return false;
        Employee employee = (Employee) obj;
        return salary == employee.salary
            && Objects.equals(name, employee.name)
            && Objects.equals(department, employee.department);
    }

    @Override
    public int hashCode() {
        return Objects.hash(name, salary, department);
    }

    // getters, toString...
}

Java 7+ utility methods simplify implementation and handle null safely:

Objects.hash(field1, field2, field3)  // generates hash from fields

Interview Q&A — equals() and hashCode()

  1. Q: What is the equals() and hashCode() contract?
    A: If two objects are equal per equals(), they must have the same hashCode(). Violating this breaks HashMap/HashSet.
  2. Q: What happens if you override equals() but not hashCode()?
    A: HashMap/HashSet break — equal objects may be stored/retrieved from different buckets.
  3. Q: Does same hashCode mean objects are equal?
    A: No. Hash collision is allowed. equals() must be used for true equality check.
  4. Q: Why use getClass() instead of instanceof in equals()?
    A: getClass() ensures exact type match. instanceof allows subclass equality which can violate symmetry.
  5. Q: What fields should be included in equals/hashCode?
    A: All fields that define object identity/equality. Exclude transient or computed fields.
  6. Q: Can you use @EqualsAndHashCode from Lombok?
    A: Yes — generates correct implementations. Specify fields with @EqualsAndHashCode.Include/Exclude.
  7. Q: How does Assert.assertEquals use equals()?
    A: Calls expected.equals(actual) for objects. Incorrect equals() causes false pass/fail.
  8. Q: == vs equals()?
    A: == compares references (memory address). equals() compares values (when overridden).
  9. Q: Why is hashCode used in HashMap?
    A: Determines bucket index. Faster lookup — check hash first, then equals() only within bucket.
  10. Q: Is hashCode() performance-critical?
    A: Yes for collections. Poor hashCode (always returns 1) degrades HashMap to linked list — O(n) lookup.

Section 16: Garbage Collection

What is Garbage Collection?

Garbage Collection (GC) is the JVM's automatic process of reclaiming memory from objects that are no longer reachable. Automation engineers don't manage memory manually, but GC behavior affects test suite performance and stability.

When Does an Object Become Eligible for GC?

  • No references point to the object
  • All references are nullified
  • Object goes out of scope (local variable in method ends)
  • Weak/Soft/Phantom references collected per reference type rules

JVM Memory Model

┌─────────────────────────────────────────────────────┐
│                    JVM Memory                        │
├─────────────────────┬───────────────────────────────┤
│   Young Generation  │       Old Generation           │
│  ┌───────┬────────┐ │  Long-lived objects promoted   │
│  │ Eden  │S0 │ S1 │ │  from Young Gen after surviving│
│  │ (new  │   │    │ │  multiple GC cycles            │
│  │ objs) │   │    │ │                                │
│  └───────┴────────┘ │                                │
├─────────────────────┴───────────────────────────────┤
│  Metaspace (Java 8+) — class metadata, static vars  │
│  (Replaced PermGen in Java 8)                       │
└─────────────────────────────────────────────────────┘

Flow: new object → Eden → (Minor GC) → Survivor → ... → Old Gen → (Major GC)

GC Types

CollectorDescriptionUse Case
Serial GCSingle thread, stop-the-worldSmall apps, client mode
Parallel GCMultiple threads for throughputBatch processing, default for server
G1 GCRegion-based, predictable pausesDefault since Java 9, large heaps
ZGCUltra-low latency (<10ms pauses)Large heap, latency-sensitive apps

Automation Relevance

  • WebDriver not quit — browser process + driver object leak memory
  • Large test suites — hundreds of page objects created/destroyed per run
  • Parallel execution — 5 threads = 5 browsers = 5× memory pressure + GC overhead
  • Long CI jobs — accumulated leaks cause OutOfMemoryError after hours
  • ExtentReports/screenshots — large objects held in memory if not flushed
Real Incident: A team running 500 parallel tests saw CI failures after 2 hours. Root cause: ThreadLocal WebDriver without remove(). Each thread retained quit-but-not-removed driver references. Fix: add driver.remove() in @AfterMethod.

How to Prevent Memory Issues

// 1. Always quit WebDriver
@AfterMethod
public void tearDown() {
    DriverFactory.quitDriver(); // quit() + ThreadLocal.remove()
}

// 2. Close resources with try-with-resources
try (FileInputStream fis = new FileInputStream("data.xlsx");
     Workbook workbook = new XSSFWorkbook(fis)) {
    // use workbook
} // automatically closed

// 3. Nullify large collections after use

// process elements...

// 4. JVM flags for CI pipelines
// -Xmx2g          Max heap size 2GB
// -Xms1g          Initial heap size 1GB
// -XX:+UseG1GC    Use G1 garbage collector
// -XX:+HeapDumpOnOutOfMemoryError  Dump heap for analysis

System.gc(); // JVM may ignore this — do not rely on it

// finalize() — DEPRECATED since Java 9, removed in Java 18
// Do NOT use — unpredictable timing, performance penalty
// Use try-with-resources or explicit cleanup instead

Interview Q&A — Garbage Collection

  1. Q: What is garbage collection?
    A: Automatic reclamation of memory from unreachable objects by the JVM. No manual free() like C/C++.
  2. Q: When is an object eligible for GC?
    A: When no live reference points to it — nullified, out of scope, or reference reassigned.
  3. Q: Young Gen vs Old Gen?
    A: Young Gen (Eden + Survivor) holds new objects. Objects surviving multiple GC cycles are promoted to Old Gen.
  4. Q: What is Metaspace?
    A: Stores class metadata (replaced PermGen in Java 8). Grows dynamically; OutOfMemoryError if class loading is excessive.
  5. Q: How does GC relate to WebDriver leaks?
    A: If driver.quit() isn't called and ThreadLocal.remove() is skipped, driver objects remain reachable — GC cannot reclaim them.
  6. Q: What JVM flags do you set for CI?
    A: -Xmx (max heap), -Xms (initial heap), -XX:+UseG1GC, -XX:+HeapDumpOnOutOfMemoryError for debugging.
  7. Q: Minor GC vs Major GC?
    A: Minor GC cleans Young Gen (frequent, fast). Major/Full GC cleans entire heap including Old Gen (less frequent, slower pauses).
  8. Q: Should you call System.gc()?
    A: No. It's a hint, not a command. JVM decides when to GC. Calling it can cause unnecessary stop-the-world pauses.
  9. Q: Why was finalize() deprecated?
    A: Unpredictable execution timing, performance overhead, security issues. Use AutoCloseable/try-with-resources instead.
