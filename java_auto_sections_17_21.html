Section 17: Design Patterns for Automation

Design patterns are reusable solutions to common software design problems. In test automation frameworks, they solve recurring challenges: managing WebDriver instances, creating browser objects, building test configurations, switching wait strategies, and reacting to test events. Mastering these five patterns is essential for framework design interviews and real-world Selenium/TestNG projects.

Singleton Pattern

What

The Singleton pattern ensures a class has only ONE instance throughout the application lifecycle and provides a global access point to that instance. The constructor is private; creation happens via a static getInstance() method.

Why in Automation

  • ConfigReader — Read config.properties once at startup; every test class accesses the same configuration without re-parsing the file.
  • ExtentReports — One report instance accumulates all test results into a single HTML report.
  • DriverManager — Centralized driver lifecycle (often combined with ThreadLocal for parallel runs).
Framework Usage: Place Singleton classes in src/main/java/config/ or driver/. Initialize in @BeforeSuite and access anywhere via ConfigReader.getInstance().

Implementation Variants

  • Eager Singleton — Instance created at class load time.
  • Lazy Singleton — Instance created on first getInstance() call.
  • Thread-Safe Singleton — synchronized getInstance(), double-checked locking, or Bill Pugh (static inner holder class).

Complete Code: Singleton ConfigReader

package config;

import java.io.FileInputStream;
import java.io.IOException;
import java.util.Properties;

public class ConfigReader {

    private static ConfigReader instance;
    private Properties properties;

    // Private constructor — prevents external instantiation
    private ConfigReader() {
        properties = new Properties();
        try {
            FileInputStream fis = new FileInputStream("src/test/resources/config.properties");
            properties.load(fis);
            fis.close();
        } catch (IOException e) {
            throw new RuntimeException("Failed to load config.properties", e);
        }
    }

    // Thread-safe lazy initialization (synchronized)
    public static synchronized ConfigReader getInstance() {
        if (instance == null) {
            instance = new ConfigReader();
        }
        return instance;
    }

    public String getProperty(String key) {
        return properties.getProperty(key);
    }

    public String getBrowser() {
        return properties.getProperty("browser", "chrome");
    }

    public String getBaseUrl() {
        return properties.getProperty("baseUrl", "https://example.com");
    }

    public int getTimeout() {
        return Integer.parseInt(properties.getProperty("timeout", "10"));
    }
}

Factory Pattern

What

The Factory pattern creates objects without exposing the instantiation logic to the client. The client requests an object by type/name; the factory decides which concrete class to instantiate.

Why in Automation

  • DriverFactory — Create ChromeDriver, FirefoxDriver, or EdgeDriver based on a config parameter or TestNG parameter.
  • PageFactory — Dynamically create page object instances bound to the current WebDriver.
  • BrowserType enum — Type-safe browser selection: BrowserType.CHROME → ChromeDriver.

Complete Code: BrowserFactory with Switch/Enum

package driver;

import org.openqa.selenium.WebDriver;
import org.openqa.selenium.chrome.ChromeDriver;
import org.openqa.selenium.chrome.ChromeOptions;
import org.openqa.selenium.edge.EdgeDriver;
import org.openqa.selenium.firefox.FirefoxDriver;
import org.openqa.selenium.firefox.FirefoxOptions;
import io.github.bonigarcia.wdm.WebDriverManager;

public enum BrowserType {
    CHROME, FIREFOX, EDGE, HEADLESS_CHROME
}

public class BrowserFactory {

    public static WebDriver createDriver(BrowserType browser) {
        WebDriver driver;
        switch (browser) {
            case CHROME:
                WebDriverManager.chromedriver().setup();
                driver = new ChromeDriver();
                break;
            case FIREFOX:
                WebDriverManager.firefoxdriver().setup();
                driver = new FirefoxDriver();
                break;
            case EDGE:
                WebDriverManager.edgedriver().setup();
                driver = new EdgeDriver();
                break;
            case HEADLESS_CHROME:
                WebDriverManager.chromedriver().setup();
                ChromeOptions options = new ChromeOptions();
                options.addArguments("--headless", "--disable-gpu", "--window-size=1920,1080");
                driver = new ChromeDriver(options);
                break;
            default:
                throw new IllegalArgumentException("Unsupported browser: " + browser);
        }
        driver.manage().window().maximize();
        driver.manage().timeouts().implicitlyWait(java.time.Duration.ofSeconds(10));
        return driver;
    }

    // Factory method reading from config
    public static WebDriver createDriverFromConfig() {
        String browserName = config.ConfigReader.getInstance().getBrowser().toUpperCase();
        BrowserType type = BrowserType.valueOf(browserName);
        return createDriver(type);
    }
}

import org.openqa.selenium.WebDriver;
import java.lang.reflect.Constructor;

public class PageFactory {

    @SuppressWarnings("unchecked")
    public static <T extends BasePage> T createPage(WebDriver driver, Class<T> pageClass) {

        }
    }

Builder Pattern

What

The Builder pattern constructs complex objects step by step. Instead of a constructor with 10 parameters, you chain setter-like methods and call build() at the end.

Why in Automation

  • REST Assured — RequestSpecBuilder().setBaseUri().addHeader().addQueryParam().build() creates a RequestSpecification.
  • TestConfig — Build test configuration with browser, URL, timeout, headless mode, parallel threads.
  • APIRequest — Construct API test requests with method, headers, body, and auth token.

package api;

import io.restassured.specification.RequestSpecification;
import io.restassured.builder.RequestSpecBuilder;
import java.util.HashMap;
import java.util.Map;

public class APIRequest {
    private final String method;
    private final String endpoint;
    private final Map<String, String> headers;
    private final Map<String, String> queryParams;
    private final String body;
    private final String baseUri;

    private APIRequest(Builder builder) {
        this.method = builder.method;
        this.endpoint = builder.endpoint;
        this.headers = builder.headers;
        this.queryParams = builder.queryParams;
        this.body = builder.body;
        this.baseUri = builder.baseUri;
    }

    public RequestSpecification toRequestSpec() {
        RequestSpecBuilder specBuilder = new RequestSpecBuilder()
            .setBaseUri(baseUri)
            .setContentType("application/json");
        headers.forEach(specBuilder::addHeader);
        queryParams.forEach(specBuilder::addQueryParam);
        return specBuilder.build();
    }

    public String getMethod() { return method; }
    public String getEndpoint() { return endpoint; }
    public String getBody() { return body; }

    public static class Builder {
        private String method = "GET";
        private String endpoint = "/";
        private Map<String, String> headers = new HashMap<>();
        private Map<String, String> queryParams = new HashMap<>();
        private String body = "";
        private String baseUri = "https://api.example.com";

        public Builder method(String method) { this.method = method; return this; }
        public Builder endpoint(String endpoint) { this.endpoint = endpoint; return this; }
        public Builder header(String key, String value) { this.headers.put(key, value); return this; }
        public Builder queryParam(String key, String value) { this.queryParams.put(key, value); return this; }
        public Builder body(String body) { this.body = body; return this; }
        public Builder baseUri(String baseUri) { this.baseUri = baseUri; return this; }

        public APIRequest build() { return new APIRequest(this); }
    }
}

Strategy Pattern

What

The Strategy pattern defines a family of algorithms, encapsulates each one, and makes them interchangeable. The client chooses the strategy at runtime without changing client code.

Why in Automation

  • Wait Strategies — Switch between ExplicitWait, FluentWait, and ImplicitWait based on element behavior.
  • Locator Strategies — Swap between ID, CSS, XPath strategies for element finding.
  • Reporting Strategies — Switch between ExtentReports and Allure without changing test code.

Complete Code: WaitStrategy

package utils;

import org.openqa.selenium.By;
import org.openqa.selenium.WebDriver;
import org.openqa.selenium.WebElement;
import org.openqa.selenium.support.ui.WebDriverWait;
import org.openqa.selenium.support.ui.FluentWait;
import java.time.Duration;

// Strategy interface
public interface WaitStrategy {
    WebElement waitForElement(WebDriver driver, By locator);
}

// Concrete strategy: Explicit Wait
public class ExplicitWaitStrategy implements WaitStrategy {
    private final int timeoutSeconds;

    public ExplicitWaitStrategy(int timeoutSeconds) {
        this.timeoutSeconds = timeoutSeconds;
    }

    @Override
    public WebElement waitForElement(WebDriver driver, By locator) {
        WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(timeoutSeconds));
        return wait.until(d -> d.findElement(locator).isDisplayed()
            ? d.findElement(locator) : null);
    }
}

// Concrete strategy: Fluent Wait
public class FluentWaitStrategy implements WaitStrategy {
    private final int timeoutSeconds;
    private final int pollingMillis;

    public FluentWaitStrategy(int timeoutSeconds, int pollingMillis) {
        this.timeoutSeconds = timeoutSeconds;
        this.pollingMillis = pollingMillis;
    }

    @Override
    public WebElement waitForElement(WebDriver driver, By locator) {
        FluentWait<WebDriver> wait = new FluentWait<>(driver)
            .withTimeout(Duration.ofSeconds(timeoutSeconds))
            .pollingEvery(Duration.ofMillis(pollingMillis))
            .ignoring(org.openqa.selenium.NoSuchElementException.class);
        return wait.until(d -> d.findElement(locator));
    }
}

// Context class — uses strategy
public class WaitContext {
    private WaitStrategy strategy;

    public WaitContext(WaitStrategy strategy) {
        this.strategy = strategy;
    }

    public void setStrategy(WaitStrategy strategy) {
        this.strategy = strategy;
    }

    public WebElement findElement(WebDriver driver, By locator) {
        return strategy.waitForElement(driver, locator);
    }
}

Observer Pattern

What

The Observer pattern defines a one-to-many dependency: when one object (subject) changes state, all dependent objects (observers) are notified and updated automatically.

Why in Automation

TestNG Listeners ARE the Observer pattern. The TestNG framework (subject) fires test events; your listener classes (observers) react to those events — logging results, taking screenshots, updating reports, sending Slack notifications.

How ITestListener Works as Observer

package listeners;

import org.testng.ITestListener;
import org.testng.ITestResult;
import org.testng.ITestContext;
import reports.ExtentReportManager;
import com.aventstack.extentreports.ExtentReports;
import com.aventstack.extentreports.ExtentTest;

// Observer: reacts to test lifecycle events from TestNG (Subject)
public class TestListener implements ITestListener {

    private ExtentReports extent = ExtentReportManager.getInstance().getReport();
    private ThreadLocal<ExtentTest> extentTest = new ThreadLocal<>();

    // Observer method: notified when test STARTS
    @Override
    public void onTestStart(ITestResult result) {
        ExtentTest test = extent.createTest(result.getMethod().getMethodName());
        extentTest.set(test);
        System.out.println("TEST STARTED: " + result.getName());
    }

    // Observer method: notified when test PASSES
    @Override
    public void onTestSuccess(ITestResult result) {
        extentTest.get().pass("Test passed: " + result.getName());
        System.out.println("TEST PASSED: " + result.getName());
    }

    // Observer method: notified when test FAILS
    @Override
    public void onTestFailure(ITestResult result) {
        extentTest.get().fail("Test failed: " + result.getThrowable());
        captureScreenshot(result);
        System.out.println("TEST FAILED: " + result.getName());
    }

    // Observer method: notified when test is SKIPPED
    @Override
    public void onTestSkipped(ITestResult result) {
        extentTest.get().skip("Test skipped: " + result.getName());
    }

    // Observer method: notified when entire SUITE finishes
    @Override
    public void onFinish(ITestContext context) {
        extent.flush();
        System.out.println("SUITE FINISHED — Report generated");
    }

    private void captureScreenshot(ITestResult result) {
        // Screenshot logic on failure
        extentTest.get().addScreenCaptureFromPath("screenshots/" + result.getName() + ".png");
    }
}

// Register in testng.xml:
// <listeners>
//     <listener class-name="listeners.TestListener"/>
// </listeners>

Interview Q&A — Design Patterns for Automation

Q1: Which design pattern is used in TestNG listeners?

A: Observer pattern. TestNG is the subject that fires events (onTestStart, onTestFailure); listener classes are observers that react to those events.

Q2: Why use Singleton for ConfigReader instead of static variables?

A: Singleton provides lazy initialization, thread safety, and encapsulation. Static variables are initialized at class load and harder to reset between test suites.

Q3: Singleton vs Factory — when to use which?

A: Singleton when you need exactly ONE instance (config, report). Factory when you need to CREATE different types of objects (browsers, page objects).

Q4: Is PageFactory (Selenium) the same as Factory pattern?

A: Selenium's PageFactory is a hybrid — it uses Factory to initialize page elements via reflection and proxy, but the core Factory pattern creates entire page object instances.

Q5: How does REST Assured use Builder pattern?

A: RequestSpecBuilder and ResponseSpecBuilder let you chain setBaseUri(), addHeader(), addQueryParam() and call build() to create immutable specification objects.

Q6: Can you combine Singleton with ThreadLocal?

A: Yes. Singleton DriverManager holds a ThreadLocal<WebDriver> — one manager instance, but each thread gets its own driver. Common in parallel TestNG execution.

Q7: What is double-checked locking in Singleton?

A: Check if instance is null twice — once without synchronization (fast path), once inside synchronized block — to avoid creating multiple instances in multi-threaded environments while minimizing lock overhead.

Q8: Strategy vs Factory — what's the difference?

A: Factory creates objects; Strategy defines interchangeable behavior/algorithms. Factory answers "which object?"; Strategy answers "which algorithm?"

Q9: How would you add a new browser to BrowserFactory?

A: Add enum value to BrowserType, add case in switch statement, configure WebDriverManager for that browser. Tests remain unchanged — they call createDriverFromConfig().

Q10: What are disadvantages of Singleton in test automation?

A: Global state makes unit testing difficult, hidden dependencies between classes, and state can leak between test suites if not properly reset in @AfterSuite.

Q11: How does Builder pattern improve test readability?

A: Instead of new TestConfig("chrome", "url", 10, true, 4), you write fluent chains: new TestConfig.Builder().browser("chrome").headless(true).build() — self-documenting and order-independent.

Q12: Give a real example of Observer in your framework.

A: TestListener implements ITestListener. On test failure, it captures screenshot, logs to ExtentReport, and sends Slack notification — all triggered automatically by TestNG without modifying test classes.

Q13: Is Enum Singleton better than synchronized getInstance()?

A: Joshua Bloch recommends enum singleton — inherently serializable, reflection-proof, and thread-safe without any synchronization code. Best for simple singletons like ConfigReader.

Q14: How do design patterns make a framework scalable?

A: Factory adds new browsers/pages without changing tests. Strategy swaps wait/report implementations via config. Builder handles growing config complexity. Observer adds new listeners without modifying TestNG or test classes.

Section 18: Page Object Model — Framework Design

What is POM?

The Page Object Model (POM) is a design pattern that separates page UI elements and actions from test logic. Each web page (or component) gets its own class containing locators (private) and action methods (public). Tests interact with pages through these methods — never directly with locators.

Why POM? — The #1 Framework Design Question

Without POM — Maintenance Nightmare

// BAD: Locators and actions mixed directly in test class
@Test
public void testLogin() {
    driver.findElement(By.id("username")).sendKeys("admin");
    driver.findElement(By.id("password")).sendKeys("pass123");
    driver.findElement(By.xpath("//button[@type='submit']")).click();
    Assert.assertTrue(driver.findElement(By.cssSelector(".welcome-msg"))
        .getText().contains("Welcome"));
    // If login button ID changes → update EVERY test file that logs in!
}

With POM — Change Once, All Tests Work

// GOOD: Test uses page object methods — no locators in test
@Test
public void testLogin() {
    LoginPage loginPage = new LoginPage(driver);
    HomePage homePage = loginPage.login("admin", "pass123");
    Assert.assertTrue(homePage.getWelcomeText().contains("Welcome"));
    // If login button ID changes → update ONLY LoginPage.java
}

POM Rules

  1. One class per page/component — LoginPage, HomePage, SearchPage each get their own class.
  2. Locators: private — Never expose By locators outside the page class.
  3. Actions: public methods — enterUsername(), clickLogin() encapsulate interactions.
  4. Return type: next page object — clickLogin() returns HomePage for method chaining.
  5. NO assertions in page objects — Assertions belong in test classes only.
  6. NO test data in page objects — Pass data as method parameters from tests.

package base;

import org.openqa.selenium.*;
import org.openqa.selenium.support.ui.WebDriverWait;
import org.openqa.selenium.support.ui.ExpectedConditions;
import org.openqa.selenium.interactions.Actions;
import java.time.Duration;

public class BasePage {
    protected WebDriver driver;
    protected WebDriverWait wait;
    protected Actions actions;

    public BasePage(WebDriver driver) {
        this.driver = driver;
        this.wait = new WebDriverWait(driver, Duration.ofSeconds(15));
        this.actions = new Actions(driver);
    }

    protected void click(By locator) {
        waitForElement(locator);
        driver.findElement(locator).click();
    }

    protected void type(By locator, String text) {

PageFactory Limitations

  • StaleElementReferenceException — Using @CacheLookup on dynamic elements (AJAX-loaded content) returns stale references after DOM refresh.
  • No dynamic locator support — Cannot parameterize locators: @FindBy(id = dynamicId) does not work with runtime values.
  • Hidden initialization — Elements initialized via proxy; debugging is harder than explicit driver.findElement(By).
  • Tied to WebElement — Cannot easily return By locators for custom wait strategies.

Interview Q&A — Page Object Model

Q1: What is Page Object Model?

A: A design pattern separating page UI elements/actions from test logic. Each page is a class with private locators and public action methods. Tests call page methods, never interact with locators directly.

Q2: Why should page objects NOT contain assertions?

A: Page objects represent UI behavior (actions/state), not test validation. Assertions in page objects violate Single Responsibility and make pages unusable for both positive and negative test scenarios.

Q3: What should clickLogin() return?

A: The next page object — HomePage. This enables method chaining and makes navigation explicit: HomePage home = loginPage.clickLogin().

Q4: POM vs PageFactory — which do you prefer?

A: POM with By locators. It supports dynamic locators, integrates with explicit waits, avoids stale element issues, and is easier to debug. PageFactory is acceptable for static pages only.

Q5: How do you handle dynamic elements in POM?

A: Use parameterized methods: By rowLocator = By.xpath("//tr[td='" + name + "']"). By locators allow runtime construction; PageFactory @FindBy does not.

Q6: What is component-based POM?

A: Splitting shared UI sections (header, footer, nav) into separate component classes. Pages compose components via HAS-A relationship instead of duplicating locators.

Q7: Where do you store test data in a POM framework?

A: External sources — JSON, Excel, CSV, properties files, or TestNG DataProvider. Never hardcode data inside page objects; pass as method parameters from tests.

Q8: How does BasePage reduce code duplication?

A: Common actions (click, type, wait, scroll, isDisplayed) are implemented once in BasePage. All page classes extend BasePage and inherit these utilities.

Q9: Can a page object represent a modal dialog?

A: Yes. Treat modals as components: ConfirmationDialogComponent with clickConfirm() and clickCancel() methods. Pages return dialog components when modals appear.

Q10: How do you handle iFrames in POM?

A: Create a method in BasePage: switchToFrame(By locator) and switchToDefault(). Page objects for iframe content call switch methods before interacting with elements.

Q11: What is the difference between page object and page factory?

A: Page Object is the design pattern (concept). Page Factory is Selenium's implementation using @FindBy annotations and initElements. You can implement POM without PageFactory.

Q12: How do you organize page objects for a SPA (Single Page Application)?

A: Use component-based POM. SPA pages don't fully reload — create components for each UI section that may appear/disappear dynamically rather than one class per URL.

Q13: Should page objects handle waits internally?

A: Yes. Page action methods should wait for elements before interacting. Tests should not contain wait logic — that's the page object's responsibility via BasePage.waitForElement().

Q14: How do you test a page object in isolation?

A: Use a mock WebDriver or a lightweight page load in @BeforeMethod. Verify action methods find correct elements and return expected page objects. Keep page objects free of assertions for easy unit testing.

Q15: What is lazy initialization in POM context?

A: With By locators, elements are found only when an action method is called — not at page object construction. This avoids StaleElementReferenceException on dynamic pages.

Q3: Why use ThreadLocal in automation?

Answer: ThreadLocal provides each thread its own isolated copy of a variable. In parallel TestNG execution, each test thread gets its own WebDriver instance — preventing shared state corruption where one test quits another test's browser.

WITHOUT ThreadLocal (Parallel = BROKEN):
Thread-1 ──→ shared WebDriver ←── Thread-2
              (Thread-2 quits → Thread-1 fails!)

WITH ThreadLocal (Parallel = SAFE):
Thread-1 ──→ ThreadLocal ──→ WebDriver-1
Thread-2 ──→ ThreadLocal ──→ WebDriver-2
Thread-3 ──→ ThreadLocal ──→ WebDriver-3

public class DriverManager {
    private static ThreadLocal<WebDriver> driverThread = new ThreadLocal<>();

    public static WebDriver getDriver() {
        if (driverThread.get() == null) {
            driverThread.set(BrowserFactory.createDriverFromConfig());
        }
        return driverThread.get();
    }

    public static void setDriver(WebDriver driver) {
        driverThread.set(driver);
    }

    public static void quitDriver() {
        if (driverThread.get() != null) {
            driverThread.get().quit();
            driverThread.remove(); // CRITICAL: prevent memory leak
        }
    }
}

Interviewer expects: You know ThreadLocal + remove() to prevent memory leaks in thread pools.

// Retry pattern for flaky dynamic elements
public WebElement findWithRetry(By locator, int maxAttempts) {
    for (int i = 0; i < maxAttempts; i++) {
        try {
            return wait.until(ExpectedConditions.visibilityOfElementLocated(locator));
        } catch (TimeoutException e) {
            if (i == maxAttempts - 1) throw e;
            driver.navigate().refresh();
        }
    }
    return null;

Additional Interview Q&A

Q11: Why use Maven in automation projects?

A: Dependency management (Selenium, TestNG, ExtentReports versions in pom.xml), standardized project structure, build lifecycle (compile → test → report), and CI/CD integration via mvn test.

Q12: What is the difference between implicit and explicit wait?

A: Implicit wait is a global timeout for findElement across the entire session. Explicit wait is a targeted condition (visibility, clickability) for a specific element. Always prefer explicit waits in frameworks.

Q13: How do you run tests in parallel with TestNG?

A: Set parallel="methods" and thread-count="4" in testng.xml. Use ThreadLocal for WebDriver isolation. Each thread gets its own driver instance via DriverManager.

Q14: What is PageFactory.initElements() doing internally?

A: It scans the class for @FindBy annotations, creates proxy WebElement objects, and binds them to the WebDriver. Elements are lazily located on first interaction via proxy invocation handler.

Q15: How do you integrate ExtentReports with TestNG?

A: Create ExtentReports instance in @BeforeSuite. Implement ITestListener to call extent.createTest() on start, pass/fail on result, and extent.flush() on finish. Register listener in testng.xml.

Q16: What is the difference between @BeforeMethod and @BeforeClass?

A: @BeforeMethod runs before EACH test method — use for fresh browser launch. @BeforeClass runs once per class — use for expensive setup like DB connection or API auth token generation.

Q17: How do you handle authentication in API automation?

A: Store token in a Singleton AuthManager. Use REST Assured RequestSpecBuilder to add Authorization header. Refresh token in @BeforeSuite if it expires during long test runs.

Q18: What is Log4j and why use it over System.out.println?

A: Log4j provides log levels (DEBUG, INFO, WARN, ERROR), file/console output, timestamp formatting, and runtime configuration. Essential for debugging failed CI/CD runs where console output is the only evidence.

Q19: How do you version-control test data?

A: Store JSON/CSV/Properties in src/test/resources/ alongside code in Git. Data changes are tracked in pull requests. Sensitive data (passwords) goes in encrypted properties or CI/CD secrets, never in Git.

Q20: What is the role of BaseTest in a framework?

A: BaseTest is the parent of all test classes. It handles @BeforeMethod (launch browser, navigate to URL), @AfterMethod (screenshot on fail, quit driver), and @BeforeSuite (init reports). Tests inherit this setup/teardown automatically.

Q21: How do you handle file uploads in Selenium?

A: Use sendKeys(absoluteFilePath) on the file input element — Selenium bypasses the OS file dialog. Store test files in src/test/resources/testfiles/.

Q22: What is Selenium Grid and when do you use it?

A: Grid distributes tests across multiple machines/browsers. Hub receives tests; Nodes execute on specific browsers/OS. Use when you need parallel cross-browser execution at scale beyond local ThreadLocal parallelism.

1. Reverse a String

Problem Statement: Reverse a given string. Automation use: reverse test case IDs for uniqueness validation, reverse URL paths for breadcrumb testing.

    }
    public static void main(String[] args) {

        System.out.println("Original: " + testId);
        System.out.println("Reversed: " + reverse(testId));

        //         Reversed: 100_NIGOL_CT
    }

Optimized — Two Pointer:

public static String reverseTwoPointer(String input) {
    char[] chars = input.toCharArray();
    int left = 0, right = chars.length - 1;
    while (left < right) {
        char temp = chars[left];
        chars[left++] = chars[right];
        chars[right--] = temp;
    }
    return new String(chars);
}

Time Complexity: O(n) | Space Complexity: O(n) for char array

2. Reverse Words in a Sentence

        String[] words = sentence.trim().split("\\s+");
        StringBuilder result = new StringBuilder();
        for (int i = words.length - 1; i >= 0; i--) {
            result.append(words[i]);
            if (i > 0) result.append(" ");
        }
        return result.toString();
    }

public static String reverseWordsStack(String sentence) {
    java.util.Stack<String> stack = new java.util.Stack<>();

        stack.push(word);

    while (!stack.isEmpty()) {
        result.append(stack.pop());
        if (!stack.isEmpty()) result.append(" ");

}

7. First Non-Repeated Character

Problem Statement: Find the first character that appears only once. Automation use: find unique test identifier, first unmatched error code in log parsing.

Approach: LinkedHashMap maintains insertion order — iterate and find first count==1.

Brute Force — LinkedHashMap:

public class FirstNonRepeated {
    public static char firstNonRepeated(String input) {
        java.util.Map<Character, Integer> freq = new java.util.LinkedHashMap<>();
        for (char c : input.toCharArray()) {
            freq.merge(c, 1, Integer::sum);
        }
        for (java.util.Map.Entry<Character, Integer> e : freq.entrySet()) {
            if (e.getValue() == 1) return e.getKey();
        }

            c -> c, java.util.LinkedHashMap::new,
            java.util.stream.Collectors.counting()))
        .entrySet().stream()
        .filter(e -> e.getValue() == 1)

8. Longest Substring Without Repeating Characters

Problem Statement: Find length of longest substring without repeating characters. Automation use: find longest unique test name sequence, sliding window for log analysis.

Approach: Sliding window with HashMap tracking last seen index.

Brute Force — Check All Substrings:

public class LongestSubstring {
    public static int longestUnique(String input) {
        int maxLen = 0;
        for (int i = 0; i < input.length(); i++) {
            java.util.Set<Character> seen = new java.util.HashSet<>();
            for (int j = i; j < input.length(); j++) {
                if (seen.contains(input.charAt(j))) break;
                seen.add(input.charAt(j));
                maxLen = Math.max(maxLen, j - i + 1);
            }
        }
        return maxLen;
    }
    public static void main(String[] args) {
        System.out.println(longestUnique("testcase")); // 4 (tesc)
        System.out.println(longestUnique("automation")); // 7 (automti)
        System.out.println(longestUnique("abcabcbb")); // 3 (abc)
    }
}

Optimized — Sliding Window:

public static int longestUniqueSliding(String input) {
    java.util.Map<Character, Integer> lastSeen = new java.util.HashMap<>();
    int maxLen = 0, start = 0;
    for (int end = 0; end < input.length(); end++) {
        char c = input.charAt(end);
        if (lastSeen.containsKey(c) && lastSeen.get(c) >= start) {
            start = lastSeen.get(c) + 1;
        }
        lastSeen.put(c, end);
        maxLen = Math.max(maxLen, end - start + 1);
    }
    return maxLen;
}

Time Complexity: O(n²) brute force, O(n) sliding window | Space Complexity: O(k)

Interview Discussion: Sliding window is the expected optimized answer. Explain window shrink when duplicate found. Classic Google/Amazon coding question adapted for automation context.

3. Missing Number in 1-N

Problem Statement: Array contains N-1 numbers from 1 to N. Find the missing number. Automation use: find missing test case ID in sequential test suite.

Approach: Sum formula n*(n+1)/2 minus array sum, or XOR all numbers.

public class MissingNumber {
    public static int findMissing(int[] arr, int n) {
        int expectedSum = n * (n + 1) / 2;
        int actualSum = 0;
        for (int num : arr) actualSum += num;
        return expectedSum - actualSum;
    }
    public static void main(String[] args) {
        int[] testIds = {1, 2, 3, 5, 6, 7, 8}; // N=8, missing 4

}

public static int findMissingXOR(int[] arr, int n) {
    int xor = 0;
    for (int i = 1; i <= n; i++) xor ^= i;
    for (int num : arr) xor ^= num;
    return xor;
}

Time Complexity: O(n) | Space Complexity: O(1)

Interview Discussion: XOR avoids integer overflow for large N. Sum formula simpler but overflow risk with large arrays. Both are O(n).

6. Maximum Difference (Buy-Sell Stock)

        for (int i = 0; i < arr.length; i++) {

        minVal = Math.min(minVal, arr[i]);
    }
    return maxDiff;
}

7. Rotate Array by K Positions

Problem Statement: Rotate array right by K positions. Automation use: rotate test execution order, circular buffer for test retry queue.

Approach: Reverse entire array, then reverse first K and rest separately.

Brute Force — One by One:

public class RotateArray {
    public static void rotate(int[] arr, int k) {
        k = k % arr.length;
        for (int i = 0; i < k; i++) {
            int last = arr[arr.length - 1];
            for (int j = arr.length - 1; j > 0; j--) {
                arr[j] = arr[j - 1];
            }
            arr[0] = last;
        }
    }
    public static void main(String[] args) {
        int[] testOrder = {1, 2, 3, 4, 5, 6, 7};
        rotate(testOrder, 3);
        System.out.println(java.util.Arrays.toString(testOrder));
        // Output: [5, 6, 7, 1, 2, 3, 4]
    }
}

Optimized — Reverse Approach:

public static void rotateReverse(int[] arr, int k) {
    k = k % arr.length;
    reverse(arr, 0, arr.length - 1);
    reverse(arr, 0, k - 1);
    reverse(arr, k, arr.length - 1);
}
private static void reverse(int[] arr, int start, int end) {
    while (start < end) {
        int temp = arr[start];
        arr[start++] = arr[end];
        arr[end--] = temp;
    }
}

Time Complexity: O(n*k) brute, O(n) reverse | Space Complexity: O(1)

Interview Discussion: Reverse approach is elegant — explain three reverses. Handle k > array.length with k % n. Follow-up: rotate left instead of right?

                Map.Entry::getKey, Map.Entry::getValue,
                (a, b) -> a, LinkedHashMap::new));

        System.out.println("Pages ranked by failures (least first):");
        sorted.forEach((page, count) ->
            System.out.println("  " + page + " → " + count + " failures"));
        // Output: SearchPage→3, ProfilePage→7, LoginPage→12, CheckoutPage→25
    }
}

Time Complexity: O(n log n) | Space Complexity: O(n)

Interview Discussion: LinkedHashMap preserves sorted order. reversed() for descending. TreeMap natural order alternative if keys should be sorted.

3. Word Count in Paragraph

Problem Statement: Count word frequencies in a paragraph. Automation use: analyze test log output, count error message occurrences.

Approach: Split by whitespace, HashMap frequency count.

Complete Code:

import java.util.*;

public class WordCount {
    public static Map<String, Integer> countWords(String paragraph) {
        Map<String, Integer> wordCount = new HashMap<>();
        for (String word : paragraph.toLowerCase().split("\\s+")) {
            word = word.replaceAll("[^a-zA-Z0-9]", "");
            if (!word.isEmpty()) {
                wordCount.merge(word, 1, Integer::sum);
            }
        }
        return wordCount;
    }
    public static void main(String[] args) {
        String log = "Test FAILED on LoginPage Test FAILED on CheckoutPage " +
                     "Test PASSED on SearchPage Test FAILED on LoginPage";
        countWords(log).entrySet().stream()
            .sorted(Map.Entry.<String, Integer>comparingByValue().reversed())
            .forEach(e -> System.out.println(e.getKey() + " → " + e.getValue()));
        // Output: test→4, failed→3, on→4, loginpage→2, ...
    }
}

Time Complexity: O(n) | Space Complexity: O(k) unique words

Interview Discussion: Handle punctuation with regex cleanup. Case-insensitive with toLowerCase(). Stream groupingBy as alternative one-liner.

1. Stream Sorting (Multiple Criteria)

Problem Statement: Sort test results by status (FAIL first) then by duration (longest first). Automation use: prioritize failed tests in report, then slowest.

Approach: Comparator chaining with thenComparing().

Complete Code:

import java.util.*;
import java.util.stream.*;

public class StreamMultiSort {
    static class TestResult {
        String name; String status; long durationMs;
        TestResult(String n, String s, long d) { name=n; status=s; durationMs=d; }
        public String toString() { return name + " [" + status + ", " + durationMs + "ms]"; }
    }

    public static void main(String[] args) {
        List<TestResult> results = Arrays.asList(
            new TestResult("LoginTest", "PASS", 1200),
            new TestResult("CheckoutTest", "FAIL", 3200),
            new TestResult("SearchTest", "PASS", 450),
            new TestResult("ProfileTest", "FAIL", 890),
            new TestResult("CartTest", "FAIL", 1500)
        );

        results.stream()
            .sorted(Comparator
                .comparing((TestResult r) -> r.status.equals("PASS") ? 1 : 0)
                .thenComparing(r -> r.durationMs, Comparator.reverseOrder()))
            .forEach(System.out::println);
        // Output: CheckoutTest [FAIL, 3200ms] → CartTest [FAIL, 1500ms] →
        //         ProfileTest [FAIL, 890ms] → LoginTest [PASS, 1200ms] → SearchTest [PASS, 450ms]
    }
}

Time Complexity: O(n log n) | Space Complexity: O(n)

Interview Discussion: Comparator.comparing() + thenComparing() for multi-field sort. reversed() for descending. Method references vs lambdas in comparators.

2. Stream Filtering (Complex Conditions)

Problem Statement: Filter test results that FAILED with duration > 1000ms and name contains "Test". Automation use: identify slow failing tests for investigation.

Approach: Predicate composition with and/or/negate.

import java.util.*;
import java.util.stream.*;
import java.util.function.Predicate;

public class StreamFilter {
    static class TestResult {
        String name; String status; long durationMs;
        TestResult(String n, String s, long d) { name=n; status=s; durationMs=d; }
        public String toString() { return name + " [" + status + ", " + durationMs + "ms]"; }
    }

    public static void main(String[] args) {
        List<TestResult> results = Arrays.asList(
            new TestResult("LoginTest", "FAIL", 1200),
            new TestResult("SearchTest", "PASS", 450),
            new TestResult("CheckoutTest", "FAIL", 3200),
            new TestResult("APIHealth", "FAIL", 200),
            new TestResult("ProfileTest", "FAIL", 890)
        );

        Predicate<TestResult> isFailed = r -> r.status.equals("FAIL");
        Predicate<TestResult> isSlow = r -> r.durationMs > 1000;
        Predicate<TestResult> hasTest = r -> r.name.contains("Test");

        System.out.println("Slow failing tests:");
        results.stream()
            .filter(isFailed.and(isSlow).and(hasTest))
            .forEach(System.out::println);
        // Output: LoginTest [FAIL, 1200ms], CheckoutTest [FAIL, 3200ms]
    }
}

Time Complexity: O(n) | Space Complexity: O(1)

Interview Discussion: Predicate.and/or/negate for readable complex filters. Extract predicates as named variables for clarity. Alternative: single lambda with && operators.

import java.util.*;
import java.util.stream.*;

public class StreamGrouping {
    static class TestCase {
        String name; String module; String status;
        TestCase(String n, String m, String s) { name=n; module=m; status=s; }
    }

    public static void main(String[] args) {
        List<TestCase> tests = Arrays.asList(
            new TestCase("testLogin", "Auth", "PASS"),
            new TestCase("testLogout", "Auth", "PASS"),
            new TestCase("testSearch", "Search", "FAIL"),
            new TestCase("testFilter", "Search", "PASS"),
            new TestCase("testCheckout", "Cart", "FAIL"),
            new TestCase("testAddToCart", "Cart", "PASS")
        );

        Map<String, List<TestCase>> byModule =
            tests.stream().collect(Collectors.groupingBy(t -> t.module));

        byModule.forEach((module, cases) -> {
            long failed = cases.stream().filter(t -> t.status.equals("FAIL")).count();
            System.out.println(module + ": " + cases.size() + " tests, " + failed + " failed");
        });
        // Output: Auth: 2 tests, 0 failed
        //         Search: 2 tests, 1 failed
        //         Cart: 2 tests, 1 failed
    }
}

Time Complexity: O(n) | Space Complexity: O(n)

4. Counting Elements by Category

Problem Statement: Count test results by status category (PASS/FAIL/SKIP). Automation use: generate test summary dashboard counts.

Approach: groupingBy + counting downstream collector.

Complete Code:

import java.util.*;
import java.util.stream.*;

public class StreamCounting {
    public static void main(String[] args) {
        List<String> testStatuses = Arrays.asList(
            "PASS", "PASS", "FAIL", "PASS", "SKIP",
            "FAIL", "PASS", "FAIL", "SKIP", "PASS",
            "PASS", "FAIL", "PASS", "PASS", "SKIP"
        );

        Map<String, Long> statusCounts = testStatuses.stream()
            .collect(Collectors.groupingBy(s -> s, Collectors.counting()));

        System.out.println("=== Test Execution Summary ===");
        statusCounts.forEach((status, count) ->
            System.out.printf("  %-6s: %d (%.1f%%)%n", status, count,
                count * 100.0 / testStatuses.size()));

        long totalFailed = statusCounts.getOrDefault("FAIL", 0L);
        System.out.println("\nPass rate: " +
            String.format("%.1f%%", (1 - totalFailed * 1.0 / testStatuses.size()) * 100));
        // Output: PASS: 8 (53.3%), FAIL: 4 (26.7%), SKIP: 3 (20.0%)
        //         Pass rate: 73.3%
    }
}

Time Complexity: O(n) | Space Complexity: O(k) categories

Interview Discussion: groupingBy + counting is the idiomatic Java 8 approach. Alternative: Collectors.counting() vs manual increment. Follow-up: count by module AND status (multi-level grouping).
