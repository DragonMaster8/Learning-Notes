Section 1: Java Fundamentals

Java Overview

Java is a high-level, object-oriented programming language developed by Sun Microsystems (now Oracle) in 1995. It is designed for building platform-independent applications — from enterprise backends to mobile apps (Android) to test automation frameworks like Selenium, TestNG, and Cucumber.

What is Java?

Java is a compiled-then-interpreted language. Source code (.java) is compiled into platform-neutral bytecode (.class), which the JVM executes on any operating system. This makes Java ideal for cross-platform test suites that run on Windows CI agents, macOS developer machines, and Linux Docker containers without code changes.

Why Java for QA Automation?

  • Industry standard — Selenium WebDriver, TestNG, JUnit, Rest Assured, and Cucumber all have first-class Java support.
  • Rich ecosystem — Maven/Gradle, ExtentReports, Log4j, Jackson, and Apache POI for data-driven testing.
  • Strong typing — catches errors at compile time, reducing flaky test code.
  • OOP design — Page Object Model, Factory patterns, and layered frameworks map naturally to Java.
  • Enterprise adoption — most SDET/Senior QA roles expect Java proficiency.

Platform Independence & "Write Once, Run Anywhere" (WORA)

Java achieves platform independence through bytecode and the JVM. You compile once; the JVM on each platform translates bytecode to native machine instructions. Your Selenium tests compiled on Windows run identically on a Linux Jenkins agent — only the browser/driver binaries differ.

Interview Tip: WORA means bytecode portability, not absolute zero configuration. You still need a JVM and OS-specific native drivers (ChromeDriver, geckodriver) on each machine.

Garbage Collection Basics

GC automatically reclaims heap memory from objects with no live references. When your test finishes and the WebDriver reference goes out of scope (or is set to null), GC eventually frees that memory.

  • Minor GC — cleans Young Generation (Eden/Survivor); frequent, fast
  • Major/Full GC — cleans Old Generation; less frequent, can cause pauses
  • Unreachable objects — no reference chain from GC roots (stack refs, static fields, JNI refs)

GC Algorithms

  • Serial GC (-XX:+UseSerialGC) — single thread; small apps, low memory
  • Parallel GC (-XX:+UseParallelGC) — multi-threaded; throughput-focused; default for server-class machines
  • G1 GC (-XX:+UseG1GC) — default since JDK 9; divides heap into regions; predictable pauses; ideal for large parallel test suites
CI Tip: Parallel test runs consume heap fast. Set -Xmx2g -XX:+UseG1GC in your Maven Surefire or Jenkins JVM args to avoid OOM during large suites.

Java Features

Interview Q&A — Section 1

Q1: What is the difference between JDK, JRE, and JVM?

A: JVM executes bytecode. JRE = JVM + runtime libraries (enough to run apps). JDK = JRE + development tools like javac (needed to write and compile code). SDETs need JDK; CI runners executing pre-built JARs may only need JRE.

Q2: Explain "Write Once, Run Anywhere."

A: Java source compiles to platform-neutral bytecode. Any OS with a compatible JVM can run that bytecode. The JVM handles OS-specific details. In automation, the same test JAR runs on Windows, macOS, and Linux agents.

Q3: What is bytecode?

A: Intermediate instruction set produced by javac, stored in .class files. It is neither source code nor native machine code — the JVM interprets or JIT-compiles it to native instructions at runtime.

Q4: Stack vs Heap — where is a WebDriver object stored?

A: The reference variable driver is on the stack (local variable in a method). The actual ChromeDriver object lives on the heap. When the method ends, the stack reference is popped; the heap object becomes eligible for GC.

Q5: What is Garbage Collection?

A: Automatic memory management that reclaims heap objects no longer reachable from GC roots. Prevents memory leaks in long-running test suites. Triggered automatically; can be hinted with System.gc() but never guaranteed.

Q6: What is the main method signature and why is it required?

A: public static void main(String[] args) — JVM entry point. public for JVM access, static so JVM can call without creating an object, void returns nothing, String[] args for command-line params. TestNG/JUnit don't need main — their runners invoke tests directly.

Q7: What is JIT compilation?

A: Just-In-Time compiler converts frequently executed bytecode to native machine code at runtime for performance. Hot methods in your framework (waits, element lookups) benefit from JIT after warm-up runs.

Q8: Can Java run without main method?

A: Yes. Applets (deprecated), servlets, TestNG test classes, and Spring Boot apps use other entry points. TestNG uses annotations like @Test discovered by the TestNG runner.

Q9: What is the role of ClassLoader?

A: Loads .class files into Method Area. Bootstrap loads core JDK classes, Extension loads extension libs, Application loads classpath classes (your test framework). Custom ClassLoaders exist for plugin architectures.

Q10: Why is Java popular for test automation?

A: Mature Selenium bindings, strong typing, excellent IDE support (IntelliJ), Maven/Gradle dependency management, enterprise adoption, OOP for Page Object Model, and vast hiring demand for Java SDETs.

Q11: What happens when you run java LoginTest?

A: JVM starts → ClassLoader loads LoginTest.class → bytecode verified → main (or TestNG runner) invoked → Execution Engine interprets/JIT-compiles → native code runs on OS.

Q12: What is -Xmx and -Xms?

A: JVM heap flags. -Xms = initial heap size, -Xmx = maximum heap size. Example: -Xms512m -Xmx2g. Critical for parallel test execution to avoid OutOfMemoryError.

Q13: Is Java compiled or interpreted?

A: Both. Source is compiled to bytecode by javac (compile-time). JVM interprets bytecode at runtime and JIT-compiles hot code to native machine code. Hence "compiled-then-interpreted."

Q14: What is MetaSpace (Java 8+)?

A: Replaced PermGen. Stores class metadata in native memory (not heap). Grows dynamically. PermGen OutOfMemoryErrors from loading too many classes (dynamic proxies in frameworks) are largely resolved.

Q15: Difference between .java and .class files?

A: .java is human-readable source code you write. .class is compiled bytecode the JVM executes. One .java file typically produces one .class file (inner classes produce additional .class files).

Q16: What JVM do you use in production CI?

A: HotSpot (default Oracle/OpenJDK JVM). Amazon Corretto, Eclipse Temurin, and Azul Zulu are popular distributions. All implement the same JVM spec — choose based on org licensing and support policies.

Section 2: OOPS Concepts

Class

Definition

A class is a blueprint or template that defines the structure (fields) and behavior (methods) of objects. It encapsulates data and operations into a single unit.

Real-World Analogy

A house blueprint defines rooms, doors, and layout. You don't live in the blueprint — you build actual houses from it. Similarly, a LoginPage class defines locators and actions; each test creates a real instance to interact with the browser.

Java Code Example

public class LoginPage {
    // Fields (state)
    private WebDriver driver;
    private By usernameField = By.id("username");
    private By passwordField = By.id("password");
    private By loginButton   = By.id("loginBtn");

    // Default constructor (no-args)
    public LoginPage() {
        this.driver = DriverFactory.getDriver();
    }

    // Parameterized constructor
    public LoginPage(WebDriver driver) {
        this.driver = driver;
    }

    // Copy constructor
    public LoginPage(LoginPage other) {
        this.driver = other.driver;
    }

    // Methods (behavior)
    public void enterUsername(String username) {
        driver.findElement(usernameField).sendKeys(username);
    }

    public void enterPassword(String password) {
        driver.findElement(passwordField).sendKeys(password);
    }

    public void clickLogin() {
        driver.findElement(loginButton).click();
    }

    public void login(String user, String pass) {
        enterUsername(user);
        enterPassword(pass);
        clickLogin();
    }
}

Constructor Types

  • Default — no-args; JVM provides one if none defined
  • Parameterized — accepts arguments to initialize fields
  • Copy — creates new object from existing object's state

this Keyword

Refers to the current object instance. Used to: differentiate fields from parameters, call other constructors (this()), return current instance for method chaining.

Selenium Usage

Page Object classes are the cornerstone of maintainable automation. Each page/component gets its own class with locators as private fields and user actions as public methods.

Interview Q&A — Class

Q: Can a class have multiple constructors?
A: Yes — constructor overloading. Same class name, different parameter lists. JVM selects based on arguments passed at instantiation.

Object

Definition

An object is an instance of a class — a concrete entity in memory with its own state (field values) and behavior (methods).

Real-World Analogy

If Car is a class, your specific Toyota Camry is an object — it has actual color, mileage, and license plate values.

Java Code Example

// Class definition
public class DashboardPage {
    private WebDriver driver;

    public DashboardPage(WebDriver driver) {
        this.driver = driver;
    }

    public String getWelcomeMessage() {
        return driver.findElement(By.id("welcome")).getText();
    }
}

// Object creation
public class LoginTest {
    @Test
    public void testSuccessfulLogin() {
        WebDriver driver = new ChromeDriver();          // Object 1
        LoginPage loginPage = new LoginPage(driver);      // Object 2
        loginPage.login("admin", "pass123");

        DashboardPage dashboard = new DashboardPage(driver); // Object 3
        Assert.assertTrue(dashboard.getWelcomeMessage().contains("Welcome"));

  • new keyword allocates memory on heap and invokes constructor
  • Reference variable (e.g., loginPage) on stack points to heap object
  • Multiple references can point to same object
  • null means reference points to nothing — calling methods causes NullPointerException

Selenium Usage

loginPage.login("testuser", "password");
// loginPage is a reference; the LoginPage object holds driver state and locators

Encapsulation

Definition

Encapsulation bundles data (fields) and methods that operate on that data within a class, restricting direct access via access modifiers. Data hiding through private fields + public getters/setters.

Real-World Analogy

A TV remote exposes buttons (public interface) but hides internal circuitry (private). You press "power" without knowing how the transistor switches work.

Complete Code Example

public class ConfigReader {
    // Private fields — hidden from outside
    private static Properties properties = new Properties();
    private static final String CONFIG_PATH = "src/test/resources/config.properties";

    // Private constructor — prevent instantiation (utility class pattern)
    private ConfigReader() {}

    // Controlled public access
    public static String getProperty(String key) {
        loadPropertiesIfNeeded();
        return properties.getProperty(key);
    }

    public static String getBaseUrl() {
        return getProperty("base.url");
    }

    public static int getTimeout() {
        return Integer.parseInt(getProperty("timeout.seconds"));
    }

    private static void loadPropertiesIfNeeded() {
        if (properties.isEmpty()) {
            try (FileInputStream fis = new FileInputStream(CONFIG_PATH)) {
                properties.load(fis);
            } catch (IOException e) {
                throw new FrameworkException("Failed to load config", e);
            }
        }
    }
}

// Encapsulated WebDriver in BaseTest
public class BaseTest {
    private WebDriver driver;  // private — tests can't accidentally reassign

    protected WebDriver getDriver() {
        return driver;
    }

    @BeforeMethod
    public void setUp() {
        driver = DriverFactory.createDriver("chrome");
        driver.manage().timeouts().implicitlyWait(Duration.ofSeconds(10));
    }

    @AfterMethod
    public void tearDown() {
        if (driver != null) {
            driver.quit();
        }
    }
}

Selenium Usage

Page Objects hide locator details (private By fields). Tests call loginPage.login(user, pass) without knowing element IDs changed from #user to #username.

Interview Q&A — Encapsulation

Q: How does encapsulation help test maintenance?
A: Locator changes happen in one class. Tests remain unchanged. Reduces duplication and prevents tests from directly manipulating WebDriver internals.

Inheritance

Definition

Inheritance allows a class (child/subclass) to acquire fields and methods of another class (parent/superclass) using extends. Establishes an IS-A relationship.

Real-World Analogy

A ElectricCar IS-A Car — it inherits wheels, engine concept, but adds battery-specific behavior.

Types of Inheritance in Java

  • Single — one parent, one child: LoginTest extends BaseTest
  • Multilevel — chain: SmokeTest → RegressionTest → BaseTest
  • Hierarchical — multiple children, one parent: LoginTest and SearchTest both extend BaseTest
  • Multiple inheritance of classes — NOT supported in Java

Complete Inheritance Hierarchy

// Parent / Super class
public class BaseTest {
    protected WebDriver driver;
    protected Logger logger = LogManager.getLogger(this.getClass());

    @BeforeMethod
    @Parameters({"browser"})
    public void setUp(@Optional("chrome") String browser) {
        driver = DriverFactory.createDriver(browser);
        driver.manage().window().maximize();
        logger.info("Browser launched: " + browser);
    }

    @AfterMethod
    public void tearDown() {
        if (driver != null) {
            driver.quit();
            logger.info("Browser closed");
        }
    }

    protected void takeScreenshot(String name) {
        // shared utility for all tests
    }
}

// Child / Sub class
public class LoginTest extends BaseTest {

    private LoginPage loginPage;

    @BeforeMethod
    public void initPage() {
        loginPage = new LoginPage(driver);  // driver inherited from BaseTest
    }

    @Test
    public void verifyValidLogin() {
        loginPage.login("admin", "admin123");
        Assert.assertTrue(driver.getCurrentUrl().contains("dashboard"));
    }

    @Test
    public void verifyInvalidLogin() {
        loginPage.login("wrong", "wrong");
        Assert.assertTrue(loginPage.isErrorDisplayed());
    }
}

public class BasePage {
    public void waitForPageLoad() {
        new WebDriverWait(driver, Duration.ofSeconds(30))
            .until(d -> ((JavascriptExecutor) d)

super Keyword

Refers to immediate parent class. Used to call parent constructor (super()), access parent fields (super.driver), or invoke overridden parent methods (super.waitForPageLoad()).

Why Multiple Inheritance Not Supported — Diamond Problem

        ClassA
       /      \
   method()  method()
     /          \
 ClassB        ClassC
     \          /
      \        /
       ClassD  ← Which method() does D inherit?

If ClassB and ClassC both override method() differently, ClassD inheriting from both creates ambiguity. Java avoids this by allowing single class inheritance + multiple interface implementation.

Polymorphism

Definition

"Many forms" — same interface, different implementations. Enables flexible, extensible code.

Compile-Time Polymorphism (Method Overloading)

Same method name, different parameter lists. Resolved at compile time.

public class ElementHelper {
    // Overloaded click methods
    public void click(By locator) {
        driver.findElement(locator).click();
    }

    public void click(WebElement element) {
        element.click();
    }

    public void click(By locator, int timeoutSeconds) {
        new WebDriverWait(driver, Duration.ofSeconds(timeoutSeconds))
            .until(ExpectedConditions.elementToBeClickable(locator)).click();
    }
}

// Usage — compiler picks method based on arguments
helper.click(By.id("submit"));                    // By version
helper.click(driver.findElement(By.id("submit"))); // WebElement version
helper.click(By.id("submit"), 20);                 // By + timeout version

Runtime Polymorphism (Method Overriding)

Parent reference holds child object. Method invoked depends on actual object type at runtime.

// THE most important polymorphism in Selenium:
WebDriver driver = new ChromeDriver();   // parent ref → child object
driver = new FirefoxDriver();            // same ref, different implementation
driver = new EdgeDriver();               // swap at runtime

driver.get("https://example.com");       // each browser executes its own get()

// Another example
BaseTest test = new LoginTest();  // reference type BaseTest, actual object LoginTest
test.setUp();                     // calls LoginTest's setUp if overridden

// Interface polymorphism
List<WebElement> elements = driver.findElements(By.tagName("a"));
// ArrayList or other implementation returned — you use List interface

Real-World Analogy

A "Remote Control" button (interface) works on any TV brand (implementation). Press "volume up" — Samsung or LG each handle it their way.

Abstraction

Definition

Abstraction hides complex implementation details and exposes only essential features. Achieved via abstract classes and interfaces.

Real-World Analogy

Driving a car — you use steering wheel and pedals (interface) without understanding fuel injection internals (implementation).

Abstract Class Example

public abstract class BasePage {
    protected WebDriver driver;
    protected WebDriverWait wait;

    public BasePage(WebDriver driver) {
        this.driver = driver;
        this.wait = new WebDriverWait(driver, Duration.ofSeconds(15));
    }

    // Abstract method — must be implemented by subclasses
    public abstract boolean isPageLoaded();

    // Concrete method — shared implementation
    public String getPageTitle() {
        return driver.getTitle();
    }

    protected void click(By locator) {
        wait.until(ExpectedConditions.elementToBeClickable(locator)).click();
    }
}

public class HomePage extends BasePage {
    private By logo = By.id("logo");

    public HomePage(WebDriver driver) {
        super(driver);
    }

    @Override
    public boolean isPageLoaded() {
        return wait.until(ExpectedConditions.visibilityOfElementLocated(logo)).isDisplayed();
    }
}

Interface Example

public interface Retryable {
    int MAX_RETRIES = 3;  // public static final by default

    boolean execute();     // public abstract by default

    default void onFailure(int attempt) {  // Java 8+ default method
        System.out.println("Attempt " + attempt + " failed, retrying...");
    }
}

public class FlakyElementClick implements Retryable {
    private WebDriver driver;
    private By locator;

    public FlakyElementClick(WebDriver driver, By locator) {
        this.driver = driver;
        this.locator = locator;
    }

    @Override
    public boolean execute() {
        for (int i = 1; i <= MAX_RETRIES; i++) {
            try {
                driver.findElement(locator).click();
                return true;
            } catch (ElementClickInterceptedException e) {
                onFailure(i);
            }
        }
        return false;
    }
}

Interview Q&A — Section 2 (20+ Questions)

Q1: What are the four pillars of OOP?

A: Encapsulation, Inheritance, Polymorphism, Abstraction.

Q2: Difference between method overloading and overriding?

A: Overloading: same class, same name, different params, compile-time. Overriding: parent-child, same signature, runtime dispatch based on object type.

Q3: Can you override static methods?

A: No — static methods belong to the class, not instance. Subclass can define same signature (method hiding), but it's not true overriding. No dynamic dispatch.

Q4: Can you override private methods?

A: No — private methods are not visible to subclasses. Subclass can define same signature but it's a new method, not an override.

Q5: Why use Page Object Model?

A: Encapsulates locators and actions (encapsulation), enables reuse via inheritance (BasePage), supports polymorphism (different page implementations), and abstracts UI complexity from test logic.

Q6: IS-A vs HAS-A — which is LoginPage and WebDriver?

A: LoginPage HAS-A WebDriver (composition/association). LoginTest IS-A BaseTest (inheritance).

Q7: When to use abstract class vs interface?

A: Abstract class when subclasses share state and common implementation (BasePage). Interface when defining a contract multiple unrelated classes implement (Retryable, WebDriver).

Q8: Can an abstract class have a constructor?

A: Yes. Called when subclass object is created via super(). Initializes abstract class fields.

Q9: Can an interface extend another interface?

A: Yes. Interface inheritance is supported: interface B extends A.

Q10: What is the Diamond Problem?

A: Ambiguity when a class inherits two classes with same method. Java prevents multiple class inheritance. Interfaces with default methods require explicit override if both define same method.

Q11: What is composition over inheritance?

A: Design principle: prefer HAS-A (composition) over IS-A (inheritance) for flexibility. Example: inject WebDriverHelper into Page instead of extending it — easier to test and swap.

Q12: Explain @Override annotation.

A: Tells compiler to verify method actually overrides a parent method. Catches typos in method signatures at compile time.

Q13: Can a constructor be private?

A: Yes — Singleton pattern, utility classes (ConfigReader). Prevents external instantiation.

Q14: What is final keyword in OOP context?

A: final class — cannot extend. final method — cannot override. final variable — cannot reassign (constant reference).

Q15: Runtime polymorphism example with collections?

A: List<String> data = new ArrayList<>(); — reference is List, implementation is ArrayList. Methods like add(), get() dispatch to ArrayList at runtime.

Q16: How does WebDriver demonstrate abstraction?

A: WebDriver interface defines get(), findElement(), quit() without browser-specific code. ChromeDriver, FirefoxDriver provide implementations hidden from test code.

Q17: Can you instantiate an abstract class?

A: No. Must create concrete subclass that implements all abstract methods.

Q18: Difference between aggregation and composition?

A: Aggregation: weak ownership, parts survive independently (TestSuite ↔ Tests). Composition: strong ownership, parts destroyed with whole (Order ↔ OrderItems).

Q19: Why is BaseTest usually abstract?

A: Provides shared setup/teardown but isn't a test itself. Abstract prevents accidental instantiation while forcing subclasses to implement test methods.

Q20: How do default interface methods affect multiple inheritance?

A: A class implementing two interfaces with conflicting default methods must override and specify which to use: InterfaceA.super.method().

Q21: What is coupling and how does OOP reduce it?

A: Coupling = dependency between classes. Encapsulation and interfaces reduce coupling — tests depend on Page Object public methods, not internal locators or WebDriver details.

Q22: Explain Object class role in inheritance.

A: Every class implicitly extends java.lang.Object. Inherits equals(), hashCode(), toString(), getClass(). Override toString() in Page Objects for meaningful debug output.

Interview Q&A — Section 3

Q1: How many primitive types does Java have?

A: 8 — byte, short, int, long, float, double, char, boolean.

Q2: Difference between primitive and reference types?

A: Primitives store actual values on stack (when local). References store memory addresses pointing to heap objects. Primitives have default values; references default to null.

Q3: What is the default value of an instance int variable?

A: 0. Local int variables have no default — compiler error if used uninitialized.

Q4: Can static variables be accessed without creating an object?

A: Yes. TestConfig.BASE_URL — belongs to class, not instance.

Q5: What happens if you unbox a null Integer?

Q7: When to use long vs int for timestamps?

A: System.currentTimeMillis() returns long — exceeds int range. Always use long for timestamps and large counts.

Q8: What is var and its limitations?

A: Local type inference (Java 10+). Only for local variables with initializer. Not for fields, method parameters, or return types. Improves readability for complex generics.

Q9: Difference between float and double?

A: double has double precision (8 bytes, ~15 digits). float has single precision (4 bytes, ~6 digits). Use double by default; float only when memory-critical.

Q10: Can boolean be converted to int?

A: No implicit conversion. Use ternary: int val = flag ? 1 : 0; Unlike C/C++.

Q11: What is scope of a local variable?

A: From declaration to end of block (method, loop, if-block). Cannot access outside declaring block.

Q12: Why use Integer instead of int in collections?

A: Generics don't support primitives. List<Integer> not List<int>. Autoboxing handles conversion transparently.

Interview Q&A — Section 4

Q1: Difference between == and .equals() for Strings?

A: == compares references (memory addresses). .equals() compares content. Always use .equals() for String value comparison in assertions.

Q2: When to use switch vs if-else?

A: switch for discrete values of one variable (browser, env). if-else for complex boolean conditions (element visible AND enabled).

Q3: What is short-circuit evaluation?

A: && doesn't evaluate second operand if first is false. || skips second if first is true. Prevents NPE: if (driver != null && driver.getTitle().equals("X")).

Q4: Can switch work with String?

A: Yes, since Java 7. Common in browser/env selection. Use .toLowerCase() for case-insensitive matching.

Q5: Difference between while and do-while?

A: while checks condition first (may never execute). do-while executes at least once, then checks. Use do-while when first attempt must always run.

Q6: How does instanceof help in frameworks?

A: Safely check capabilities before casting — TakesScreenshot for screenshots, JavascriptExecutor for JS clicks, RemoteWebDriver for Grid sessions.

Q7: What is break vs continue?

A: break exits the loop entirely. continue skips remaining code in current iteration and moves to next. break in nested loops only exits innermost loop unless labeled.

Q8: Can you use String in switch with null?

A: No — NullPointerException if switch variable is null. Null-check before switch.

Q9: Enhanced for vs traditional for?

A: Enhanced for is cleaner, read-only (no index modification), works with arrays and Iterable. Traditional for needed when you need index or modify list during iteration.

Q10: What is ternary operator advantage?

A: Concise conditional assignment. Useful for inline config: timeout values, driver paths, pass/fail status strings.

Q11: How to iterate Map in automation?

A: for (Map.Entry<String, String> entry : configMap.entrySet()) or keySet()/values() with enhanced for.

Q12: What happens with missing break in switch?

A: Fall-through — execution continues into next case. Sometimes intentional, usually a bug. Use break or switch expressions (Java 14+) to prevent.

String — Immutable

Once created, a String object cannot be modified. Any "modification" creates a new String object on the heap. This ensures thread safety and enables the String pool optimization.

    int left = 0, right = chars.length - 1;
    while (left < right) {
        char temp = chars[left];
        chars[left++] = chars[right];
        chars[right--] = temp;
    }
    return new String(chars);
}

Interview Q&A — Section 5

Q1: Why is String immutable?

A: Security (class loading, network connections), thread safety, String pool optimization, and hashCode caching. Immutable objects are inherently thread-safe.

Q2: How many objects created: String s = new String("Hello");?

A: Potentially 2 — "Hello" in pool (if not already) + new heap object from new keyword.

Q3: String vs StringBuilder performance in loops?

A: String concatenation in loop creates O(n²) objects. StringBuilder creates O(n) — always use StringBuilder for dynamic string building.

Q4: Is String thread-safe?

A: Yes — immutability guarantees thread safety. StringBuffer is mutable but synchronized. StringBuilder is mutable and not thread-safe.

Q5: What is String.intern()?

A: Moves String to pool and returns pool reference. s.intern() — rarely needed in automation; useful for memory optimization with many duplicate strings.

Q6: replace() vs replaceAll()?

A: replace() takes CharSequence literals. replaceAll() takes regex patterns. replaceAll("[0-9]", "") removes all digits.

Q7: How to compare strings ignoring whitespace?

A: s1.strip().equals(s2.strip()) or normalize both before comparison.

Q8: What happens with String concatenation and null?

A: "Hello " + null produces "Hello null" (null converted to "null" string). But null.equals("x") throws NPE — always put literal first: "x".equals(variable).

Q9: charAt() vs indexOf()?

A: charAt(i) returns char at index i (throws IndexOutOfBounds). indexOf(c) returns first index of char/substring (-1 if not found).

Q10: How to validate email in test assertions?

A: email.matches("^[\\w.-]+@[\\w.-]+\\.[a-zA-Z]{2,}$") or Apache Commons Validator for production frameworks.

Q11: String pool location in Java 7+?

A: Moved from PermGen to Heap. Reduces PermGen OOM issues. Pool still deduplicates string literals.

Q12: Difference between strip() and trim()?

A: trim() removes chars ≤ U+0020 (ASCII whitespace). strip() (Java 11) removes all Unicode whitespace including non-breaking spaces — preferred for modern apps.

Interview Q&A — Section 6

Q1: Array vs ArrayList?

A: Array: fixed size, primitives allowed, faster. ArrayList: dynamic, objects only (autoboxing), rich API (add, remove, sort). Use ArrayList in automation for flexible test data.

Q2: How does HashMap handle collisions?

A: Same bucket → linked list of entries. Java 8+: when bucket exceeds 8 nodes, converts to red-black tree for O(log n) lookup. Rehashes when load factor exceeds 0.75.

Q3: Why override equals() and hashCode() together?

A: HashMap/HashSet contract: equal objects must have same hashCode. Violating this breaks HashSet deduplication and HashMap lookups.

Q4: ArrayList vs LinkedList for storing WebElements?

A: ArrayList — you iterate and occasionally access by index. LinkedList only if frequent head/tail insertions. ArrayList is default choice.

Q5: HashSet how does it prevent duplicates?

A: Uses hashCode() to find bucket, then equals() to check existing entries. If equals match found, add() returns false.

Q6: ConcurrentHashMap vs Hashtable?

A: ConcurrentHashMap uses finer-grained locking (segments/CAS), much better throughput in parallel tests. Hashtable locks entire table — legacy, avoid.

Q7: Can you add null to HashMap?

A: Yes — one null key, multiple null values. TreeMap and ConcurrentHashMap reject null keys.

Q8: Fail-fast vs fail-safe iterator?

A: Fail-fast (ArrayList, HashMap) throws ConcurrentModificationException if collection modified during iteration (except via iterator.remove()). Fail-safe (ConcurrentHashMap) works on clone, no exception.

Q9: How to sort HashMap by values?

A: Stream API: map.entrySet().stream().sorted(Map.Entry.comparingByValue()).collect(toMap(...)) or TreeMap with custom Comparator.

Q10: What is load factor in HashMap?

A: Default 0.75 — when 75% of buckets are filled, HashMap doubles capacity and rehashes all entries.

Q11: LinkedHashMap use case in automation?

A: Maintaining test step execution order, breadcrumb navigation validation, ordered config loading for reproducible test runs.

Q12: Difference between Collection and Collections?

A: Collection is the root interface. Collections is a utility class with static methods (sort, reverse, synchronized wrappers).

Q13: How to convert List to Array and vice versa?

A: list.toArray(new String[0]) and Arrays.asList(array) or new ArrayList<>(Arrays.asList(array)).

Q14: Why Properties extends Hashtable?

A: Legacy design. Properties is Hashtable<Object, Object> specialized for String key-value config files. Used extensively in config.properties loading.

Q15: TreeSet vs HashSet when to use?

A: HashSet for fast unique collection (test IDs). TreeSet when you need sorted iteration (priority-ordered test cases).

NoSuchElementException

Cause: Locator doesn't match any DOM element.
Example: driver.findElement(By.id("nonExistent"))

Cause: Explicit wait condition not met within timeout.
Example: Waiting for element visibility that never appears.
Fix: Increase timeout, verify condition, check app state.

StaleElementReferenceException

Cause: Element was found but DOM refreshed/re-rendered (AJAX, React re-render).
Example: Found element, page updates, then click stale reference.
Fix: Re-find element before interaction; use retry wrapper.

public void clickWithRetry(By locator, int maxAttempts) {
    for (int i = 0; i < maxAttempts; i++) {
        try {
            driver.findElement(locator).click();
            return;
        } catch (StaleElementReferenceException e) {
            if (i == maxAttempts - 1) throw e;
        }
    }
}

Exception Handling Best Practices for Frameworks

  1. Use explicit waits over implicit waits — reduces TimeoutException and NoSuchElementException flakiness.
  2. Never swallow exceptions silently — always log with context (page, locator, test name).
  3. Re-throw after logging — catch for screenshot/reporting, then re-throw or fail test explicitly.
  4. Create custom exceptions — FrameworkException hierarchy for meaningful error messages.
  5. Use @AfterMethod for cleanup — finally-equivalent for driver.quit() regardless of test outcome.
  6. Retry only recoverable exceptions — StaleElementReference, ElementClickIntercepted. Don't retry NoSuchElement (locator is wrong).
  7. try-with-resources for I/O — config files, test data readers, report streams.
  8. Fail fast on config errors — throw InvalidConfigException at startup, not mid-suite.
  9. Capture diagnostics on failure — screenshot, page source, browser logs in catch blocks.
  10. Avoid generic catch (Exception e) — catch specific exceptions; generic catch hides bugs.
SDET Pattern — Retry Utility: Wrap flaky actions in a retry mechanism that catches StaleElementReferenceException and ElementClickInterceptedException, re-finds the element, and retries up to N times with exponential backoff.

Interview Q&A — Section 7

Q1: Difference between Error and Exception?

A: Error — serious JVM problems (OutOfMemoryError, StackOverflowError), not recoverable, don't catch. Exception — recoverable conditions, should be handled.

Q2: Checked vs Unchecked — which are Selenium exceptions?

A: All Selenium WebDriver exceptions extend RuntimeException — unchecked. No compile-time handling required.

Q3: Can you have try without catch?

A: Yes — try with finally only, or try-with-resources. try-catch-finally and try-finally are valid.

Q4: Does finally always execute?

A: Almost always. Exceptions: System.exit(), JVM crash, daemon thread killed, infinite loop in try. Even with return in try, finally runs first.

Q5: What is exception propagation?

A: If not caught, exception propagates up the call stack. TestNG marks test as failed when uncaught exception reaches the test runner.

Q6: throw vs throws?

A: throw — explicitly throws an exception instance in method body. throws — declares checked exceptions a method may throw, delegating handling to caller.

Q7: Why create custom exceptions?

A: Meaningful error context (PageLoadException with page name), centralized handling, framework-specific recovery logic, cleaner test failure reports.

Q8: How to handle StaleElementReferenceException in POM?

A: Don't cache WebElements as instance fields. Re-find elements in each method call, or implement retry utility that re-locates on stale.

Q9: What is try-with-resources?

A: Java 7 feature. Resources implementing AutoCloseable declared in try parentheses are auto-closed. Suppresses close exceptions properly. Eliminates boilerplate finally blocks.

Q10: Can finally block throw exception?

A: Yes — if finally throws, it suppresses any exception from try/catch. Avoid throwing from finally; keep it for cleanup only.

Q11: Multi-catch block rules?

A: Java 7+. Catch multiple exceptions in one block if handling is identical. Exceptions must not have inheritance relationship. Caught as implicitly final.

Q12: How to fail a TestNG test with message?

A: Assert.fail("reason"), throw AssertionError, or throw RuntimeException. Uncaught exceptions automatically fail the test.

Q13: ElementClickInterceptedException vs ElementNotInteractableException?

A: Intercepted — element visible but another element overlays it. NotInteractable — element not clickable (hidden, disabled, off-screen). Different fixes: JS click/close overlay vs scroll/wait for enabled.

Q14: Should you catch Throwable?

A: No in test code — catches Errors (OutOfMemoryError) which should propagate. Catch Exception or specific runtime exceptions.

Q15: How to log and re-throw pattern?

A: catch (SpecificException e) { logger.error("Context: {}", locator, e); throw e; } — preserves stack trace, adds logging context.

Q16: What is suppressed exception (Java 7+)?

A: When try and finally both throw, finally's exception is thrown and try's is suppressed. Access via e.getSuppressed().

Q17: How does TestNG handle exceptions in @Test?

A: Uncaught exception → test FAILED. Expected exceptions: @Test(expectedExceptions = NoSuchElementException.class). Soft assertions collect multiple failures before throwing.

Q18: Best way to handle optional elements?

A: Use findElements (returns empty list, no exception) or catch TimeoutException/NoSuchElementException for optional popups/cookie banners.
