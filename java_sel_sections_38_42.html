Section 40: Interview Preparation

Comprehensive Q&A bank for Java, Selenium, framework design, coding, and scenario-based interviews. Every entry includes what the interviewer expects.

Top 150 Java Questions

OOP (30 Questions)

Q1: What is a class?

Answer: A class is a blueprint that defines fields (state) and methods (behavior) for objects. It does not consume memory until instantiated. In automation, each Page Object is typically a class encapsulating locators and actions. Example: public class LoginPage { private By user = By.id("user"); }

Interviewer expects: Mention blueprint vs instance and give a Page Object example.

Q2: What is an object?

Answer: An object is a runtime instance of a class created with new, stored on the heap with its own state. Two objects of the same class have independent field values. In tests, WebDriver driver = new ChromeDriver() creates a driver object managing a browser session.

Interviewer expects: Distinguish reference variable from actual heap object.

Q3: What is a constructor?

Answer: A constructor initializes object state when new is called; it has no return type and matches the class name. Default constructor is provided if none defined. Page objects use constructors to receive WebDriver: public LoginPage(WebDriver driver) { this.driver = driver; }

Interviewer expects: Know default vs custom constructor and this assignment pattern.

Q4: Default vs parameterized constructor?

Answer: Default constructor takes no arguments (compiler-generated if no constructor defined). Parameterized constructor accepts arguments to initialize fields at creation time. Use parameterized constructors in page objects to inject driver and config. Cannot have return type; overloading allows multiple constructors.

Interviewer expects: Explain when you inject dependencies via constructor in framework code.

Q5: What is encapsulation?

Answer: Encapsulation hides internal state behind private fields and exposes controlled access via public getters/setters. It protects data integrity and allows validation before modification. In frameworks, ConfigReader encapsulates file path and parsing logic — callers use getProperty() without knowing internals.

Interviewer expects: Link data hiding to maintainable test utilities.

Q6: What is inheritance?

Answer: Inheritance (extends) lets a subclass reuse and extend superclass fields and methods — IS-A relationship. BasePage can hold common wait/click helpers; LoginPage extends BasePage. Java supports single class inheritance only. Use when true specialization exists, not just code reuse.

Interviewer expects: Give BasePage inheritance example; mention single inheritance limit.

Q7: What is polymorphism?

Answer: Polymorphism allows a parent reference to point to child objects and invoke overridden methods at runtime. WebDriver driver = new ChromeDriver() — driver reference type is WebDriver, actual object is ChromeDriver. Method overriding enables this; overloading is compile-time polymorphism.

Interviewer expects: Distinguish runtime (overriding) from compile-time (overloading) polymorphism.

Q8: What is abstraction?

Answer: Abstraction hides complex implementation and exposes only essential behavior. Achieved via abstract classes (partial implementation) and interfaces (pure contract). WebDriver interface abstracts browser-specific details — your tests call get() and click() without knowing Chrome vs Firefox internals.

Interviewer expects: Connect abstraction to WebDriver interface as real-world example.

Q9: IS-A vs HAS-A relationship?

Answer: IS-A is inheritance: Dog IS-A Animal (extends). HAS-A is composition: Car HAS-A Engine (field reference). Favor HAS-A when you need flexibility — a TestContext HAS-A WebDriver rather than forcing all tests to inherit from a mega base class. Composition reduces tight coupling.

Interviewer expects: Prefer composition for framework utilities; inheritance for true IS-A hierarchies.

Q10: Method overloading?

Answer: Overloading defines multiple methods with the same name but different parameter lists (type, count, or order) in the same class. Resolved at compile time. Example: click(By locator) and click(WebElement element) in a utility class. Return type alone does not distinguish overloads.

Interviewer expects: Compile-time resolution; different parameter lists required.

Q11: Method overriding?

Answer: Overriding redefines a superclass method in a subclass with the same signature and compatible return type. Resolved at runtime via dynamic dispatch. @Override annotation catches errors. Cannot override static, final, or private methods. Used when BasePage defines click() and MobilePage overrides for touch events.

Interviewer expects: Runtime polymorphism; same signature; @Override annotation.

Q12: Can we override static methods?

Answer: No — static methods belong to the class, not instances. A subclass method with the same signature hides (shadows) the parent static method but does not override it. Calling via parent reference invokes parent version. This is method hiding, not polymorphic overriding.

Interviewer expects: Use "method hiding" terminology — not overriding.

Q13: Can we override private methods?

Answer: No — private methods are not visible to subclasses. If a subclass defines a method with the same signature, it is a new method, not an override. The @Override annotation will cause a compile error if applied. Encapsulation prevents subclass access to private implementation details.

Interviewer expects: Private methods are not part of inheritance contract.

Q14: What is super keyword?

Answer: super refers to the immediate parent class. super() calls parent constructor (must be first statement in constructor). super.method() invokes overridden parent method from subclass. Useful when extending BasePage and calling super.waitForPageLoad() before child-specific actions.

Interviewer expects: Mention super() as first line in constructor rule.

Q15: What is this keyword?

Answer: this refers to the current object instance. Used to disambiguate field vs parameter (this.driver = driver), call other constructors (this(args)), or pass current object. In page objects, this is commonly used for fluent interface method chaining: return this;

Interviewer expects: Give fluent page object return this example.

Q16: Abstract class vs interface?

Answer: Abstract class can have instance fields, constructors, concrete and abstract methods; single inheritance. Interface defines contract (methods); supports multiple implementation; Java 8+ allows default/static methods. Use abstract class for shared state/behavior (BasePage); interface for capability contracts (Retryable, Reportable).

Interviewer expects: Trade-off table: state, inheritance, use case.

Q17: What is final class?

Answer: A final class cannot be extended — String and Integer are final. Prevents subclassing that could break invariants. Use final on utility classes that should not be subclassed (e.g., DriverFactory with private constructor). Declared with final keyword before class keyword.

Interviewer expects: String is final — classic interview example.

Q18: What is final method?

Answer: A final method cannot be overridden by subclasses. Prevents altering critical behavior. Useful for template method pattern where base defines algorithm skeleton and final method enforces step order. Rare in test code but important for security-sensitive or framework-core methods.

Interviewer expects: Prevents overriding — different from final variable.

Q19: What is final variable?

Answer: A final variable can be assigned once — for primitives, value cannot change; for references, reference cannot be reassigned (object contents may still mutate). Use final for locators: private final By loginBtn = By.id("login"); signals immutability of reference.

Interviewer expects: Distinguish final reference from immutable object (String is immutable, StringBuilder is not).

Q20: Composition vs inheritance?

Answer: Inheritance (IS-A) couples subclass to parent hierarchy; composition (HAS-A) embeds objects as fields for flexibility. Prefer composition when behavior can be swapped — a TestRunner HAS-A RetryStrategy rather than extending RetryTest. Composition follows "favor composition over inheritance" principle from Effective Java.

Interviewer expects: Effective Java principle — composition for flexibility.

Q21: What is coupling?

Answer: Coupling measures dependency between classes. Tight coupling (tests directly new ChromeDriver()) makes change hard. Loose coupling (tests use WebDriver interface injected via factory) improves maintainability. Dependency injection and interfaces reduce coupling in automation frameworks.

Interviewer expects: Show tight vs loose coupling with driver creation example.

Q22: What is cohesion?

Answer: Cohesion measures how closely a class's methods and fields relate to a single purpose. High cohesion: LoginPage only handles login. Low cohesion: UtilityClass with email, DB, and screenshot methods. High cohesion makes classes easier to test, maintain, and reuse.

Interviewer expects: Single Responsibility — one class, one purpose.

Q23: What is static block?

Answer: Static block runs once when class is first loaded by JVM, before any constructor. Used for static field initialization (loading config.properties into static Map). Order: static fields → static block → main/constructor. Cannot access instance fields inside static block.

Interviewer expects: Runs once at class loading — good for config initialization.

Q24: What is instance block?

Answer: Instance initialization block { } runs before constructor each time an object is created. Used for common initialization shared across constructors. Execution order: static block → instance block → constructor. Rare in test code but shows JVM initialization understanding.

Interviewer expects: Know initialization order: static → instance → constructor.

Q25: What is object cloning?

Answer: Cloning creates a copy of an object. Shallow clone (clone()) copies field values — references are shared. Deep clone copies referenced objects too. Test data objects may need clone to avoid cross-test contamination in parallel runs. Implement Cloneable and override clone() for shallow copy.

Interviewer expects: Shallow vs deep clone distinction with parallel test data example.

Q26: Shallow vs deep copy?

Answer: Shallow copy duplicates object fields but shared references point to same nested objects. Deep copy recursively clones nested objects. In parallel tests, shallow copy of a test data Map still shares inner lists — deep copy isolates each thread. Serialization or manual copy constructors achieve deep copy.

Interviewer expects: Parallel test data isolation is the practical hook.

Q27: What is marker interface?

Answer: Marker interface has no methods — tags a class for JVM/framework treatment. Serializable enables object serialization; Cloneable enables clone(). TestNG uses annotations (@Test) rather than marker interfaces, but concept shows metadata-driven behavior in Java ecosystem.

Interviewer expects: Serializable and Cloneable as classic examples.

Q28: What is functional interface?

Answer: Functional interface has exactly one abstract method (SAM), enabling lambda expressions. @FunctionalInterface annotation is optional but recommended. Examples: Runnable, Callable, Predicate, Function. In tests: Predicate<WebElement> isVisible = e -> e.isDisplayed();

Interviewer expects: SAM rule and link to lambda usage in test utilities.

Q29: Diamond problem in Java?

Answer: Diamond problem occurs when a class inherits conflicting default methods from two interfaces. Java 8 allows multiple interface inheritance but class must override conflicting defaults. Example: if A and B both define default log(), class C implements both and must override log() to resolve ambiguity.

Interviewer expects: Java 8 default methods created this — override required to resolve.

Q30: Multiple inheritance in Java?

Answer: Java does not support multiple class inheritance (one extends only). Supports multiple interface implementation. This avoids diamond problem with classes. A test class can implement ITestListener and IRetryAnalyzer simultaneously while extending BaseTest once.

Interviewer expects: Classes: single; interfaces: multiple.

Strings (20 Questions)

Q31: Why is String immutable?

Answer: String is final with internal char array not exposed for mutation. Immutability enables String pool sharing, thread safety without synchronization, and security (SQL/param safety). Once created, "admin" literal is reused. Use StringBuilder for loop concatenation in test data builders.

Interviewer expects: Mention String pool, thread safety, and security benefits.

Q32: What is String pool?

Answer: String pool (intern pool) stores unique string literals in heap for reuse. String s = "hello" uses pool; new String("hello") creates heap object outside pool. s.intern() adds to pool. Saves memory when same literals appear across framework config keys and test data.

Interviewer expects: Literal vs new String() — pool behavior difference.

Q33: new String() vs literal?

Answer: Literal "test" may reuse pooled instance; new String("test") always creates new heap object. == may differ: literal1 == literal2 is true (same pool ref); new String("test") == "test" is false. Always use equals() for content comparison in assertions.

Interviewer expects: Never use == for String content comparison.

Q34: == vs equals() for String?

Answer: == compares memory references (same object?). equals() compares character content (when properly overridden — String does). "a" == new String("a") is false; "a".equals(new String("a")) is true. In test assertions, always use assertEquals(expected, actual) which uses equals().

Interviewer expects: Reference vs content — assertEquals uses equals().

Q35: String vs StringBuilder?

Answer: String is immutable — each modification creates new object ( costly in loops). StringBuilder is mutable, not thread-safe, efficient for concatenation. Build dynamic XPath or JSON in loops with StringBuilder. Use String for constants and final values like locator templates.

Interviewer expects: Performance in loops — StringBuilder wins.

Q36: StringBuilder vs StringBuffer?

Answer: Both mutable. StringBuffer methods are synchronized (thread-safe, slower). StringBuilder is not synchronized (faster, use in single-threaded test code). In parallel test generation, prefer StringBuilder per thread or StringBuffer if sharing one builder (rare).

Interviewer expects: Thread safety trade-off — StringBuilder for single-threaded tests.

Q37: Important String methods?

Answer: Key methods: length(), charAt(), substring(), toLowerCase/toUpperCase(), trim(), replace(), split(), contains(), startsWith/endsWith(), indexOf(), equals/equalsIgnoreCase(), isEmpty/isBlank(), format(), join(). In automation: split CSV data, contains for soft assertions, trim for input cleanup.

Interviewer expects: Name 5-6 methods with automation use cases.

Q38: String concatenation performance?

Answer: Using + in loops creates many intermediate String objects (O(n²) total copying). Compiler optimizes compile-time constants but not loop concatenation. Use StringBuilder.append() in loops building dynamic locators or SQL. For few concatenations, + is readable and fine.

Interviewer expects: Loop concatenation — StringBuilder mandatory.

Q39: How to reverse a String?

Answer: StringBuilder sb = new StringBuilder(str); return sb.reverse().toString(); Or two-pointer char array swap. String has no reverse method. In interviews, mention immutability requires new object. Time O(n), space O(n).

Interviewer expects: StringBuilder reverse or two-pointer approach.

Q40: How to check palindrome?

Answer: Two pointers at start and end, compare chars moving inward. Or reverse string and compare with equals(). Ignore case/non-alphanumeric for full variant. Time O(n). Common coding question — write clean loop, handle empty string edge case.

Interviewer expects: Two-pointer O(n) without extra space preferred.

Q41: String to char array?

Answer: char[] chars = str.toCharArray(); creates copy of internal characters. Modify array without affecting original String (immutability). Reverse array in-place for palindrome check. Prefer toCharArray over manual charAt loop for clarity.

Interviewer expects: toCharArray() returns copy — String unchanged.

Q42: char array to String?

Answer: new String(charArray) or String.valueOf(charArray). After in-place array manipulation, wrap back to String for assertions. String.copyValueOf(charArray, offset, count) for partial conversion.

Interviewer expects: valueOf and constructor both valid.

Q43: String.format usage?

Answer: String.format("User %s logged in at %d", name, timestamp) or System.out.printf. Format specifiers: %s string, %d int, %f float, %n newline. Useful for dynamic log messages and test report templates: String.format("FAIL: %s expected [%s] but got [%s]", testName, expected, actual).

Interviewer expects: Format specifiers and logging/report use case.

Q44: String.join usage?

Answer: String.join(", ", list) joins iterable with delimiter (Java 8). Cleaner than manual loop for CSV output or building tag strings. Arrays: String.join("-", "a","b","c"). Null delimiter throws NPE; elements can be null (prints "null").

Interviewer expects: Java 8 join — cleaner than StringBuilder for simple joins.

Q45: Split vs tokenizer?

Answer: split(regex) returns String[] using regex (handles empty trailing strings differently). StringTokenizer legacy, not regex-based. Prefer split for CSV: line.split(",", -1) preserves trailing empties. split uses regex — escape dot: split("\\.").

Interviewer expects: split is modern; mention regex escaping.

Q46: Replace vs replaceAll?

Answer: replace(old, new) literal string replacement. replaceAll(regex, replacement) uses regex — special chars need escape. replaceFirst replaces first match only. For dynamic IDs in locators, replace("{{id}}", actualId) is safer than replaceAll without regex intent.

Interviewer expects: Literal replace vs regex replaceAll.

Q47: Intern method?

Answer: str.intern() returns pooled canonical reference. If pool has equal string, returns pool copy; else adds and returns. Rarely needed in test code. Useful for memory optimization with many duplicate literals. "hello".intern() == "hello" is true.

Interviewer expects: Returns canonical pool reference.

Q48: CompareTo vs equals?
