private — Maximum Encapsulation

The most restrictive modifier. Accessible only within the declaring class. Use it to hide implementation details and protect internal state.

public class DriverFactory {
    private WebDriver driver;

    public WebDriver createDriver(String browser) {
        driver = initializeDriver(browser);
        return driver;
    }

    private WebDriver initializeDriver(String browser) {
        switch (browser.toLowerCase()) {
            case "chrome": return new ChromeDriver();
            case "firefox": return new FirefoxDriver();
            default: throw new IllegalArgumentException("Unsupported browser");
        }
    }

    private void logDriverInfo() {
        System.out.println("Driver initialized: " + driver.getClass().getSimpleName());
    }
}
Framework Usage: Keep WebDriver instances, locators, and internal helper methods private. Expose only public action methods that tests need.

default (Package-Private) — Package-Level Access

No keyword required. Accessible within the same package only. Useful for utilities shared among framework classes in one package.

// package com.automation.utils;

class StringUtils {
    static String trimAndLower(String input) {
        return input == null ? "" : input.trim().toLowerCase();
    }
}

class DateUtils {
    static String getTimestamp() {
        return StringUtils.trimAndLower(
            java.time.LocalDateTime.now().toString()
        );
    }
}
Framework Usage: Package-level utilities like internal formatters, log helpers, or shared validators that should not be exposed outside the framework package.

protected — Subclass Access

Accessible in the same class, same package, and by subclasses (even in different packages).

public abstract class BaseTest {
    protected WebDriver driver;
    protected Properties config;

    protected void setUpDriver() {
        driver = DriverManager.getDriver();
        config = ConfigReader.loadProperties();
    }

    protected void tearDownDriver() {
        if (driver != null) {
            driver.quit();
            DriverManager.removeDriver();
        }
    }

    protected void takeScreenshotOnFailure(String testName) {
        // shared screenshot logic for all child test classes
    }
}

public class LoginTest extends BaseTest {
    @Test
    public void testValidLogin() {
        setUpDriver();
        // can access protected methods and fields from BaseTest
        driver.get(config.getProperty("url"));
        tearDownDriver();
    }
}
Framework Usage: Base test class setup/teardown methods, shared WebDriver references, and common assertion helpers accessible to all child test classes.

public — Universal Access

Accessible from anywhere. Use for the public API of your framework — what test classes and external consumers interact with.

public class LoginPage {

Common Mistakes

Mistake 1: Making Everything Public

Exposing all fields and methods breaks encapsulation. Tests become tightly coupled to implementation. If you change a locator strategy, every test that accessed the field directly breaks.

Mistake 2: Not Using Encapsulation

Public WebDriver driver fields in test classes allow any code to manipulate the driver directly. Use private fields with controlled access through Page Objects and DriverManager.

Mistake 3: Using Protected When Private Suffices

If a helper method is only used within one class, make it private, not protected. Wider access increases coupling surface.

Interview Questions & Answers

Q1: What are the four access modifiers in Java?

Answer: private, default (package-private), protected, and public. They control visibility from most restrictive to least restrictive. Only public and protected are explicit keywords; default requires no keyword.

Interviewer expects: Correct ordering by visibility scope and knowing default has no keyword.

Q2: Can a top-level class be private or protected?

Answer: No. Top-level classes can only be public or package-private (default). Inner/nested classes can use all four modifiers. A private top-level class would be meaningless since nothing outside the file could reference it.

Interviewer expects: Understanding that modifier rules differ for top-level vs nested classes.

Q3: What is the difference between default and protected?

Answer: Both allow access within the same package. However, protected also allows access from subclasses in different packages. Default does not grant cross-package subclass access. Protected is strictly broader than default.

Interviewer expects: The subclass-in-different-package distinction.

Q4: Why encapsulate WebDriver in a Page Object?

Answer: Making the driver private prevents tests from calling driver.findElement() directly. All interactions go through well-named public methods, improving readability and maintainability. Locator changes happen in one place.

Interviewer expects: Connection between access modifiers and Page Object pattern.

Q5: Can you reduce the visibility of an overridden method?

Answer: No. When overriding, you cannot make a method more restrictive. If the parent has public void click(), the child must also be public. You can widen access (protected → public) but never narrow it.

Interviewer expects: Knowledge of overriding visibility rules.

Q6: What access modifier is used for interface methods by default?

Answer: Interface methods are implicitly public abstract (before Java 8). Default methods and static methods in interfaces are also implicitly public. Interface fields are implicitly public static final.

Interviewer expects: Awareness of implicit modifiers in interfaces.

Q7: How do access modifiers relate to the Single Responsibility Principle?

Answer: Using private for internal logic keeps each class focused on its public API. External code only sees what it needs. This enforces boundaries between components and prevents classes from becoming god-objects with everything public.

Interviewer expects: Linking OOP design principles with practical modifier usage.

Q8: Can a constructor be private?

Answer: Yes. Private constructors prevent external instantiation, used in Singleton pattern, utility classes (all static methods), and factory classes. Example: private ConfigReader() {} prevents creating instances of a utility class.

Interviewer expects: Singleton/utility class pattern as a real-world use case.

Q9: What happens if you try to access a private field from a subclass?

Answer: Compilation error. Private members are not inherited in terms of access — the subclass cannot see or use them directly. The subclass can only use public/protected methods of the parent that internally use the private field.

Interviewer expects: Clear understanding that private ≠ inherited access.

Q10: In a Selenium framework, should test data fields be public?

Answer: No. Test data should be loaded via private fields or methods and accessed through public getters or parameterized test methods. Public mutable test data fields invite accidental modification across test methods.

Interviewer expects: Practical framework design thinking, not just textbook definitions.

Section 9: Static Keyword

The static keyword belongs to the class itself rather than any instance. In test automation, static members are widely used for utilities, constants, and thread-safe driver management — but misuse causes serious parallel execution problems.

Static Variables — Class-Level, Shared State

A static variable is created once when the class is loaded and shared by all instances. Only one copy exists in memory regardless of how many objects you create.

public class BrowserSession {
    static WebDriver driver;  // ONE driver shared by ALL tests

    public void launchBrowser() {
        driver = new ChromeDriver();
    }
}
Why static WebDriver driver is Dangerous:

In parallel TestNG execution, multiple test threads share the same static driver. Test A navigates to URL X while Test B navigates to URL Y — both overwrite the same driver reference. Results: flaky tests, wrong assertions, browser crashes.

ThreadLocal Alternative for Parallel Execution

public class DriverManager {
    private static ThreadLocal<WebDriver> driverThread = new ThreadLocal<>();

    public static WebDriver getDriver() {
        if (driverThread.get() == null) {
            driverThread.set(new ChromeDriver());
        }
        return driverThread.get();
    }

    public static void removeDriver() {
        driverThread.remove();
    }
}

Each thread gets its own WebDriver instance. No cross-test interference during parallel runs.

Static Methods — Called Without an Object

Static methods belong to the class. Invoke them as ClassName.methodName() without creating an instance.

public class ConfigReader {
    private static Properties properties = new Properties();

    public static String getProperty(String key) {
        return properties.getProperty(key);
    }

    public static void loadConfig(String filePath) throws IOException {
        properties.load(new FileInputStream(filePath));
    }
}

// Usage — no object needed
ConfigReader.loadConfig("config.properties");
String url = ConfigReader.getProperty("url");

Static Method Restrictions

  • Cannot access non-static (instance) members directly
  • Cannot use this or super — there is no instance context
  • Can only call other static methods or access static fields directly
public class Example {
    int instanceVar = 10;
    static int staticVar = 20;

    static void staticMethod() {
        // System.out.println(instanceVar);  // COMPILE ERROR
        System.out.println(staticVar);       // OK
        // this.instanceVar;                  // COMPILE ERROR
    }

    void instanceMethod() {
        System.out.println(instanceVar);       // OK
        System.out.println(staticVar);         // OK
        staticMethod();                          // OK
    }
}

Static Block — Executes Once at Class Load

Runs exactly once when the JVM loads the class, before any constructor or static method is called. Ideal for one-time initialization.

public class ConfigReader {

    static {

        } catch (IOException e) {
            throw new RuntimeException("Failed to load config", e);
        }
    }

    public static String getProperty(String key) {
        return properties.getProperty(key);
    }
}

Interview Questions & Answers

Q1: What does the static keyword mean in Java?

Answer: Static members belong to the class, not to any instance. They are loaded when the class is loaded and exist for the entire program lifecycle. Access via class name without creating an object.

Interviewer expects: Class-level vs instance-level distinction.

Q2: Can static methods be overridden?

Answer: No. Static methods are hidden, not overridden. Method hiding occurs when a subclass defines a static method with the same signature — the method called depends on the reference type, not the object type. This is static binding (compile-time).

Interviewer expects: Method hiding vs overriding distinction.

Q3: Why is static WebDriver dangerous in parallel testing?

Answer: A static driver is shared across all test threads. Parallel tests overwrite each other's browser state, causing race conditions, wrong page assertions, and intermittent failures. Use ThreadLocal to give each thread its own driver instance.

Interviewer expects: Practical parallel execution knowledge.

Q4: What is a static block and when does it execute?

Answer: A static block runs once when the class is first loaded by the JVM, before any static method or constructor. Used for complex static initialization like loading config files or registering JDBC drivers.

Interviewer expects: One-time initialization use case.

Q5: Can a static method access instance variables?

Answer: No. Static methods have no this reference and no instance context. They can only directly access static fields and call other static methods. To access instance data, you need an object reference passed as a parameter.

Interviewer expects: Clear compile-time restriction understanding.

Q6: What is static import?

Answer: import static ClassName.member; allows using static members without the class prefix. Common in tests: import static org.testng.Assert.*; enables calling assertEquals() directly instead of Assert.assertEquals().

Interviewer expects: Practical test code usage example.

Q7: Can you create objects of a class with a private constructor?

Answer: Not from outside the class. Private constructor + static factory method or static getInstance() is the Singleton pattern. Utility classes use private constructors to prevent instantiation since all methods are static.

Interviewer expects: Singleton and utility class patterns.

Q8: Difference between static and final?

Answer: Static means class-level (one copy). Final means value cannot change after assignment. static final combines both — a class-level constant. Static without final can be reassigned; final without static is an instance constant.

Interviewer expects: Independent concepts that combine for constants.

Q9: Are static methods thread-safe?

Answer: Only if they don't modify shared static state. A static method that only uses local variables and parameters is thread-safe. If it reads/writes static fields without synchronization, race conditions occur in multi-threaded environments.

Interviewer expects: Thread safety depends on shared mutable state, not static alone.

Q10: What happens to static variables when the program ends?

Answer: Static variables live in the Method Area (Metaspace) and persist for the JVM's lifetime. They are garbage collected only when the class loader that loaded the class is garbage collected, which typically happens at program shutdown.

Interviewer expects: Memory area awareness (Metaspace vs Heap).

Q11: Can a class be static?

Answer: Top-level classes cannot be static. Only nested (inner) classes can be declared static — they become static nested classes, which don't require an outer class instance. Example: Map.Entry is a static nested class.

Interviewer expects: Nested class context, not top-level.

Q12: Why use ThreadLocal with static wrapper instead of instance WebDriver?

Answer: ThreadLocal provides thread-confined storage accessible via static methods (no need to pass driver everywhere). Each thread gets its own driver via get(), set(), remove(). Combines centralized access with thread isolation.

Interviewer expects: DriverManager pattern as the gold standard for Selenium frameworks.

Section 10: Interfaces & Abstract Classes

Interfaces and abstract classes are the two primary mechanisms for achieving abstraction in Java. Test automation frameworks rely heavily on both — interfaces define contracts, abstract classes provide shared implementation.

Interface — Pure Contract

An interface defines a contract: what a class must do, without specifying how. Before Java 8, interfaces provided 100% abstraction (only abstract methods).

Basic Interface Syntax

public interface Searchable {
    void search(String query);
    int getResultCount();
}

public class GoogleSearch implements Searchable {
    @Override
    public void search(String query) {
        System.out.println("Searching Google for: " + query);
    }

    @Override
    public int getResultCount() {
        return 1000000;
    }
}

Default Methods (Java 8+)

Add new behavior to interfaces without breaking existing implementations.

public interface Searchable {
    void search(String query);

    default void clearSearch() {
        System.out.println("Clearing search field...");
    }

    default void logSearch(String query) {
        System.out.println("[LOG] Search performed: " + query);
        search(query);
    }
}

An abstract class can have both abstract methods (no body) and concrete methods (with implementation). Cannot be instantiated directly.

public abstract class BasePage {
    protected WebDriver driver;
    protected WebDriverWait wait;

        this.driver = driver;
        this.wait = new WebDriverWait(driver, Duration.ofSeconds(10));

        wait.until(d -> ((JavascriptExecutor) d)
            .executeScript("return document.readyState").equals("complete"));
    }

    public abstract void verifyPageLoaded();

    protected void clickElement(By locator) {
        wait.until(ExpectedConditions.elementToBeClickable(locator)).click();
    }
}

public class DashboardPage extends BasePage {

        wait.until(ExpectedConditions.visibilityOfElementLocated(By.id("dashboard-header")));

}

Interview Questions & Answers

Q1: What is the difference between an interface and an abstract class?

Answer: Interface defines a contract (what), abstract class provides partial implementation (what + some how). A class can implement multiple interfaces but extend only one abstract class. Abstract classes can have constructors and instance fields; interfaces cannot (except constants).

Interviewer expects: Multiple inheritance via interfaces as the key differentiator.

Q2: Can an interface have a constructor?

Answer: No. Interfaces cannot have constructors because they cannot be instantiated and have no instance state to initialize. Abstract classes can have constructors because subclasses need to initialize inherited fields.

Interviewer expects: Brief, definitive answer.

Q3: What is a functional interface?

Answer: An interface with exactly one abstract method (SAM). Annotated with @FunctionalInterface. Can be implemented via lambda expressions. Examples: Runnable, Callable, Comparator, Predicate.

Interviewer expects: SAM definition and lambda connection.

Q4: Why does Selenium use WebDriver as an interface?

Answer: Interface allows polymorphism — write code against WebDriver, swap ChromeDriver/FirefoxDriver/EdgeDriver at runtime. New browser implementations can be added without changing test code. Follows Open/Closed Principle.

Interviewer expects: Polymorphism and framework extensibility reasoning.

Q5: What are default methods and why were they added?

Answer: Default methods provide implementation in interfaces (Java 8). Added for backward compatibility — adding a new abstract method to an interface breaks all existing implementations. Default methods allow evolution without breaking code.

Interviewer expects: Backward compatibility as the primary motivation.

Q6: Can an abstract class implement an interface?

Answer: Yes. An abstract class can implement an interface and optionally override its methods. It can leave some interface methods abstract, forcing concrete subclasses to implement them. Common pattern: abstract BasePage implements IPageActions.

Interviewer expects: Combined usage pattern awareness.

Q7: Can you instantiate an abstract class?

Answer: No. Abstract classes cannot be instantiated directly. You must create a concrete subclass that implements all abstract methods, then instantiate the subclass. Attempting new BasePage() causes a compile error.

Interviewer expects: Subclass requirement for instantiation.

Q8: What is the diamond problem and how does Java handle it?

Answer: Diamond problem occurs when a class inherits the same method from two interfaces. Java 8 resolves this by requiring the class to override the conflicting default method. If not overridden, compile error. Java avoids multiple class inheritance entirely.

Interviewer expects: Default method conflict resolution.

Q9: When would you choose abstract class over interface?

Answer: When classes share significant common code and state (BasePage with driver, wait, common click/type methods). Use abstract class for Template Method pattern. Use interface when classes share a capability but not implementation (Searchable, Clickable).

Interviewer expects: "is-a" with shared code → abstract class; "can-do" → interface.

Q10: Can interface methods be private?

Answer: Yes, since Java 9. Private methods in interfaces help share code between default methods without exposing it. They cannot be abstract — must have a body. Only called from within the interface itself.

Interviewer expects: Java 9+ feature awareness.

Q11: What happens if a class implements two interfaces with the same default method?

Answer: The class must override the method to resolve ambiguity, optionally calling InterfaceName.super.methodName() to delegate to a specific interface's implementation. Without override, the compiler throws an error.

Interviewer expects: Explicit override with super syntax.

Q12: Explain Template Method pattern with abstract class.

Answer: Abstract class defines the skeleton algorithm in a concrete method, calling abstract methods for steps that subclasses customize. Example: BaseTest defines test flow (setup → run → teardown) with abstract runTest() implemented by each test class.

Interviewer expects: BaseTest/BasePage as concrete framework examples.

Interview Questions & Answers

Q1: What is a lambda expression?

Answer: A concise anonymous function that implements a functional interface's single abstract method. Syntax: (parameters) -> expression. Eliminates boilerplate anonymous inner classes. Used heavily in streams, comparators, and Selenium waits.

Interviewer expects: Connection to functional interfaces.

Q2: What is the difference between map() and flatMap()?

Answer: map() transforms each element 1:1 (element → element). flatMap() transforms each element and flattens the result (element → stream → flattened). Use flatMap when the mapping function returns a stream or collection.

Interviewer expects: 1:1 vs 1:many transformation distinction.

Q3: What is Optional and why use it?

Answer: Optional is a container that may or may not contain a value. It forces explicit null handling instead of returning null. Prevents NullPointerException. Use orElse/orElseGet for defaults, ifPresent for conditional actions.

Interviewer expects: Null safety as the primary motivation.

Q4: Difference between intermediate and terminal stream operations?

Answer: Intermediate operations (filter, map, sorted) return a new stream and are lazy — not executed until a terminal operation is called. Terminal operations (collect, forEach, count) produce a result and trigger pipeline execution.

Interviewer expects: Lazy evaluation concept.

Q5: Can you reuse a Java stream?

Answer: No. Once a terminal operation is called, the stream is consumed and cannot be reused. Attempting to call another terminal operation throws IllegalStateException. Create a new stream from the source collection if needed.

Interviewer expects: One-time consumption rule.

Q6: What is a method reference?

Answer: Shorthand notation for lambda expressions that call an existing method. Four types: static (Class::method), instance of particular object (obj::method), instance of arbitrary object (Class::instanceMethod), constructor (Class::new).

Interviewer expects: All four types with examples.

Q7: What is the @FunctionalInterface annotation?

Answer: Marks an interface as functional (one abstract method). Compiler validates the constraint. Not required but recommended — prevents accidentally adding a second abstract method. Examples: Runnable, Callable, Comparator, Predicate.

Interviewer expects: Compiler enforcement benefit.

Q8: How do lambdas improve Selenium wait conditions?

Answer: ExpectedCondition is a functional interface. Lambdas replace verbose anonymous classes: wait.until(d -> d.findElement(By.id("x")).isDisplayed()). Cleaner, readable, and inline custom wait logic without separate classes.

Interviewer expects: Concrete Selenium code example.

Q9: What is the difference between orElse() and orElseGet()?

Answer: orElse() always evaluates the default value argument, even if Optional has a value. orElseGet() takes a Supplier and only evaluates it when Optional is empty. Use orElseGet for expensive default computations.

Interviewer expects: Eager vs lazy default evaluation.

Q10: Explain parallel streams.

Answer: Calling parallelStream() splits data across multiple threads using ForkJoinPool. Good for CPU-intensive operations on large datasets. Not recommended for I/O-bound Selenium operations. Can cause thread safety issues with shared state.

Interviewer expects: When NOT to use parallel streams in testing.

Q11: What is a record in Java?

Answer: A compact class for immutable data carriers (Java 16). Auto-generates constructor, getters, equals(), hashCode(), toString(). Ideal for test data objects: record User(String name, String email) {}.

Interviewer expects: Test data POJO replacement use case.

Q12: What is the var keyword limitation?

Answer: var enables local type inference — compiler deduces type from initializer. Cannot use for fields, method parameters, or return types. Variable must be initialized at declaration. Cannot be null without explicit type context.

Interviewer expects: Local variables only restriction.

Q13: How does Collectors.groupingBy work?

Answer: Groups stream elements into a Map by a classifier function. Example: group test results by status — .collect(Collectors.groupingBy(TestResult::getStatus)) produces Map<String, List<TestResult>>.

Interviewer expects: Practical aggregation example.

Q14: What is the difference between forEach and forEachOrdered?

Answer: forEach may process elements in any order in parallel streams. forEachOrdered guarantees encounter order even in parallel streams. Use forEachOrdered when order matters; forEach for better parallel performance when order is irrelevant.

Interviewer expects: Parallel stream ordering awareness.

Q15: Can lambda expressions access local variables?

Answer: Yes, but local variables must be effectively final (not reassigned after initialization). Lambdas capture the variable value, not the reference. This prevents concurrency issues and ensures consistent behavior.

Interviewer expects: Effectively final constraint.

public class ExcelReader {
    public static Object[][] getSheetData(String filePath, String sheetName) {
        Object[][] data = null;
        try (FileInputStream fis = new FileInputStream(filePath);
             XSSFWorkbook workbook = new XSSFWorkbook(fis)) {

            XSSFSheet sheet = workbook.getSheet(sheetName);
            int rowCount = sheet.getLastRowNum();
            int colCount = sheet.getRow(0).getLastCellNum();
            data = new Object[rowCount][colCount];

            for (int i = 1; i <= rowCount; i++) {
                XSSFRow row = sheet.getRow(i);
                for (int j = 0; j < colCount; j++) {
                    XSSFCell cell = row.getCell(j);
                    switch (cell.getCellType()) {
                        case STRING:  data[i-1][j] = cell.getStringCellValue(); break;
                        case NUMERIC: data[i-1][j] = (int) cell.getNumericCellValue(); break;
                        case BOOLEAN: data[i-1][j] = cell.getBooleanCellValue(); break;
                        default:      data[i-1][j] = "";
                    }
                }
            }
        } catch (IOException e) {
            throw new RuntimeException("Failed to read Excel: " + filePath, e);
        }
        return data;
    }
}

ThreadLocal — Thread-Confined Storage

ThreadLocal provides each thread its own independent copy of a variable. The most important pattern for Selenium parallel execution.

public class DriverManager {
    private static ThreadLocal<WebDriver> driverThread = new ThreadLocal<>();
    private static ThreadLocal<Properties> configThread = new ThreadLocal<>();

    public static WebDriver getDriver() {
        return driverThread.get();
    }

    public static void setDriver(WebDriver driver) {
        driverThread.set(driver);
    }

    public static void removeDriver() {
        if (driverThread.get() != null) {
            driverThread.get().quit();
            driverThread.remove();
        }
    }

    public static Properties getConfig() {
        return configThread.get();
    }

    public static void setConfig(Properties config) {
        configThread.set(config);
    }
}

ThreadLocal Methods

  • get() — retrieves value for current thread

Interview Questions & Answers

Q1: What is the difference between Thread and Runnable?

Answer: Thread is a class you extend; Runnable is an interface you implement. Runnable is preferred because Java supports single inheritance — implementing Runnable leaves extends available for another class. Runnable works naturally with ExecutorService thread pools.

Interviewer expects: Runnable preferred due to single inheritance.

Q2: What is ThreadLocal and why is it essential for Selenium?

Answer: ThreadLocal gives each thread its own copy of a variable. In parallel testing, each test thread gets its own WebDriver via ThreadLocal, preventing cross-test interference. Without it, static WebDriver is shared and tests fail randomly.

Interviewer expects: DriverManager pattern as the canonical answer.

Q3: What is a race condition?

Answer: A race condition occurs when multiple threads access shared mutable state concurrently and the outcome depends on timing. Example: two tests incrementing a shared counter simultaneously may lose an update. Fixed by synchronization or ThreadLocal.

Interviewer expects: Shared mutable state as the root cause.

Q4: Difference between start() and run()?

Answer: start() creates a new thread and JVM calls run() on that new thread. Calling run() directly executes the method on the current thread — no new thread is created. start() enables parallelism; run() is just a regular method call.

Interviewer expects: Common mistake of calling run() instead of start().

Q5: Why must you call ThreadLocal.remove()?

Answer: In thread pool environments (TestNG parallel, ExecutorService), threads are reused. Without remove(), the previous test's WebDriver persists in the thread, causing stale references and memory leaks. Always remove in @AfterMethod.

Interviewer expects: Thread pool reuse as the reason.

Q6: What is synchronization and when to use it?

Answer: Synchronization ensures only one thread executes a critical section at a time. Use synchronized keyword on methods or blocks when multiple threads modify shared data (counters, report lists). ThreadLocal is preferred when each thread needs its own copy.

Interviewer expects: synchronized vs ThreadLocal decision.

Q7: What is deadlock?

Answer: Deadlock occurs when two or more threads wait for each other to release locks, creating a circular dependency. Thread A holds Lock 1 and waits for Lock 2; Thread B holds Lock 2 and waits for Lock 1. Neither can proceed. Prevent by acquiring locks in consistent order.

Interviewer expects: Circular wait condition explanation.

Q8: How does TestNG parallel execution work?

Answer: Configure parallel attribute in testng.xml: methods, tests, or classes. Set thread-count for max concurrent threads. Each thread needs its own WebDriver via ThreadLocal. TestNG creates threads and assigns test methods to them from a thread pool.

Interviewer expects: testng.xml configuration + ThreadLocal requirement.

Q9: What is ExecutorService?

Answer: A higher-level thread management API from java.util.concurrent. Provides thread pools (newFixedThreadPool, newCachedThreadPool), task submission (submit, execute), and graceful shutdown. Preferred over manually creating Thread objects for managing multiple concurrent tasks.

Interviewer expects: Thread pool concept over raw Thread creation.

Q10: Is WebDriver thread-safe?

Answer: No. WebDriver implementations are not thread-safe. A single WebDriver instance must not be shared across threads. Each thread must have its own WebDriver instance, typically managed via ThreadLocal in a DriverManager class.

Interviewer expects: Definitive "no" with ThreadLocal solution.

Q1: What is a class and object in Java?

Q2: What are the four pillars of OOP?

Interviewer expects: All four named with brief definitions.

Q3: What is encapsulation?

Answer: Bundling data and methods together while restricting direct access to fields using private modifiers and public getters/setters. In Page Objects, WebDriver and locators are private; action methods are public.

Interviewer expects: Data hiding with access modifiers example.

Q4: What is inheritance?

Answer: A mechanism where a child class acquires fields and methods from a parent class using extends. LoginTest extends BaseTest inherits setup/teardown methods. Java supports single inheritance for classes, multiple for interfaces.

Interviewer expects: extends keyword and single inheritance rule.

Q5: What is polymorphism?

Answer: Same method name, different behavior based on object type. Compile-time: method overloading. Runtime: method overriding. WebDriver driver = new ChromeDriver() — driver.get() calls ChromeDriver's implementation at runtime.

Interviewer expects: Compile-time vs runtime polymorphism distinction.

Q6: What is abstraction?

Answer: Hiding implementation details and exposing only essential features. Achieved via abstract classes and interfaces. WebDriver interface abstracts browser operations — tests don't know if Chrome or Firefox is running.

Interviewer expects: Interface/abstract class as abstraction tools.

Q7: What is a constructor?

Answer: A special method called when an object is created. Same name as class, no return type. Initializes object state. Page objects use constructors to receive WebDriver: public LoginPage(WebDriver driver) { this.driver = driver; }.

Interviewer expects: No return type, same name as class.

Q8: Constructor overloading vs method overloading?

Answer: Constructor overloading: multiple constructors with different parameter lists in the same class. Method overloading: multiple methods with same name but different parameters. Both are compile-time polymorphism. Constructors cannot be overridden.

Interviewer expects: Constructors are not inherited or overridden.

Q9: What is method overloading?

Answer: Same method name, different parameter list (type, number, or order) in the same class. Resolved at compile time. Example: findElement(By id) and findElement(String css) — different parameter types.

Interviewer expects: Parameter list must differ; return type alone is insufficient.

Q10: What is method overriding?

Answer: Subclass provides specific implementation of a method already defined in parent class. Same signature, resolved at runtime. @Override annotation recommended. Cannot reduce visibility. BaseTest defines tearDown(); child can override with extra cleanup.

Interviewer expects: Runtime polymorphism, @Override, visibility rules.

Q11: Overloading vs overriding?

Answer: Overloading: same class, different parameters, compile-time binding. Overriding: parent-child classes, same signature, runtime binding. Overloading is static polymorphism; overriding is dynamic polymorphism.

Interviewer expects: Side-by-side comparison with binding time.

Q12: Can you override static methods?

Answer: No. Static methods belong to the class, not instances. Subclass can define a static method with same signature — this is method hiding, not overriding. Resolution happens at compile time based on reference type.

Interviewer expects: Method hiding, not overriding.

Q13: Can you override private methods?

Answer: No. Private methods are not visible to subclasses. A subclass method with the same signature is a completely new method, not an override. Compiler treats them as unrelated methods.

Interviewer expects: Private methods are not part of inheritance hierarchy.

Q14: What is the diamond problem?

Answer: Multiple inheritance causes ambiguity when two parent interfaces provide the same default method. Java avoids it by not allowing multiple class inheritance. For interfaces, the implementing class must override the conflicting method.

Interviewer expects: Java's interface-based solution to diamond problem.

Q15: What are interface default methods?

Q16: What is the super keyword?

Q19: What is an abstract class?

Answer: A contract defining method signatures without implementation (pre-Java 8). A class implements an interface using implements keyword. Supports multiple interface implementation. WebDriver is the classic Selenium interface example.

Q21: Abstract class vs interface — when to use which?

Q27: What is static binding vs dynamic binding?

Q31: Why is String immutable in Java?

Answer: Strings are final with internal char array that cannot change after creation. Immutability enables String pool optimization, thread safety, and security (passwords, URLs). Any modification creates a new String object.

Interviewer expects: String pool and thread safety as reasons.

Q32: What is the String pool?

Answer: A special memory area in the heap where String literals are stored. JVM reuses existing literals instead of creating duplicates. "hello" == "hello" is true (same pool reference). new String("hello") creates a separate heap object.

Interviewer expects: Literal vs new String() difference.

Q33: == vs equals() for Strings?

Answer: == compares memory references (same object?). equals() compares actual character content. Always use equals() for String comparison. "hello" == new String("hello") is false; "hello".equals(new String("hello")) is true.

Interviewer expects: Always equals() for content comparison.

Q34: StringBuilder vs StringBuffer?

Answer: Both are mutable string builders. StringBuilder is not synchronized — faster, use in single-threaded code. StringBuffer is synchronized — thread-safe, slower. For test report building in single thread, use StringBuilder.

Interviewer expects: Thread safety as the key difference.

Q35: String vs StringBuilder vs StringBuffer?

Answer: String: immutable, concatenation creates new objects (slow in loops). StringBuilder: mutable, fast, not thread-safe. StringBuffer: mutable, thread-safe, slower. Rule: String for constants; StringBuilder for building strings in loops.

Interviewer expects: Performance in loop concatenation scenario.

Q36: What does intern() do?

Answer: Returns a String from the pool. If the literal exists in the pool, returns that reference; otherwise adds it. new String("test").intern() == "test" becomes true. Rarely needed in application code; JVM handles literals automatically.

Interviewer expects: Pool reference retrieval purpose.

Q37: What is String.format()?

Answer: Creates formatted strings using placeholders: String.format("Test %s - %s", name, status). Supports %s (string), %d (int), %f (float), %n (newline). Cleaner than concatenation for dynamic messages in test logs and reports.

Interviewer expects: Practical formatting use case.

Q38: How do you compare strings ignoring case?

Answer: Use equalsIgnoreCase(): "ADMIN".equalsIgnoreCase("admin") returns true. Also regionMatches() for partial comparison. Never use toLowerCase() before equals() — locale issues can cause unexpected results (Turkish I problem).

Interviewer expects: equalsIgnoreCase(), not toLowerCase hack.

Q39: What is the difference between charAt() and substring()?

Answer: charAt(index) returns a single char at the given index. substring(start, end) returns a new String from start (inclusive) to end (exclusive). Both throw StringIndexOutOfBoundsException for invalid indices.

Interviewer expects: Return type difference: char vs String.

Q40: How do you split a String in Java?

Answer: String.split(regex) returns String array. "chrome,firefox,edge".split(",") → ["chrome", "firefox", "edge"]. split() uses regex, so special chars need escaping: split("\\."). For CSV parsing, consider OpenCSV library.

Interviewer expects: Regex-based splitting awareness.

Q41: What is a text block (Java 15)?

Answer: Multi-line string literal using triple quotes: String json = """ { "key": "value" } """; Preserves formatting, eliminates escape characters and concatenation. Useful for embedding JSON/XML test payloads directly in code.

Interviewer expects: Multi-line string without escape sequences.

Q42: How do you check if a String is empty or blank?

Answer: isEmpty() — true if length is 0. isBlank() (Java 11) — true if empty or only whitespace. " ".isEmpty() is false; " ".isBlank() is true. Use isBlank() for validating user input fields in tests.

Interviewer expects: isBlank() vs isEmpty() distinction.

Q43: What is StringJoiner?

Answer: Java 8 utility for joining strings with delimiter, prefix, suffix. new StringJoiner(", ").add("a").add("b").toString() → "a, b". Also String.join(", ", list) as simpler alternative. Useful for building comma-separated test result summaries.

Interviewer expects: Alternative to manual concatenation with delimiters.

Q44: Can String be used in switch statements?

Answer: Yes, since Java 7. switch(browser) { case "chrome": ... case "firefox": ... }. Compiler checks for null — NullPointerException if switch variable is null. Useful for browser selection in DriverFactory.

Interviewer expects: Browser switch example from frameworks.

Q45: What happens with String concatenation in a loop?

Answer: Each += creates a new String object — O(n²) time and memory. With 1000 iterations, thousands of intermediate objects are created and garbage collected. Use StringBuilder inside the loop instead for O(n) performance.

Interviewer expects: Performance anti-pattern awareness.

Q46: What is the valueOf() method?

Answer: Static method that converts primitives and objects to String. String.valueOf(42) → "42". Returns "null" string for null objects (doesn't throw NPE). Preferred over concatenation for converting numbers to strings in test assertions.

Interviewer expects: Null-safe conversion.

Q47: How do you reverse a String?

Answer: StringBuilder has reverse(): new StringBuilder(str).reverse().toString(). Or convert to char array, swap characters in loop. String itself has no reverse method due to immutability.

Interviewer expects: StringBuilder.reverse() as the idiomatic approach.

Q48: What is the difference between trim() and strip()?

Answer: trim() removes characters ≤ U+0020 (space). strip() (Java 11) removes all Unicode whitespace including tabs, non-breaking spaces. strip() is more comprehensive. Use strip() for modern Unicode-aware trimming of web element text.

Interviewer expects: Unicode whitespace handling difference.

Q49: How do you extract numbers from a String?

Answer: Regex: str.replaceAll("[^0-9]", "") removes non-digits. Pattern/Matcher: Pattern.compile("\\d+").matcher(str).find(). Integer.parseInt() converts extracted digits. Useful for parsing price text from web elements: "$29.99" → 29.

Interviewer expects: Regex-based extraction approach.

Q50: What is the hashCode of a String?

Answer: String overrides hashCode() based on character content. Same content produces same hash code. Two equal strings (by equals()) must have same hashCode. Important when using Strings as HashMap keys for test data lookup.

Interviewer expects: Content-based hashing, HashMap key usage.

Collections — 25 Questions

Q51: List vs Set vs Map?

Answer: List: ordered, allows duplicates (ArrayList). Set: no duplicates (HashSet). Map: key-value pairs, no duplicate keys (HashMap). Use List for test steps, Set for unique locators, Map for config key-value pairs.

Interviewer expects: Duplicates and ordering behavior for each.

Q52: ArrayList vs LinkedList?

Answer: ArrayList: dynamic array, fast random access O(1), slow insert/delete in middle O(n). LinkedList: doubly linked list, slow access O(n), fast insert/delete O(1). Use ArrayList for most cases including storing WebElements.

Interviewer expects: ArrayList as default choice with performance reason.

Q53: How does HashMap work internally?

Answer: Uses array of buckets with hash function to compute bucket index. Collisions handled by linked list (Java 7) or tree (Java 8+, when bucket > 8 entries). get/put average O(1). Key must implement hashCode() and equals() correctly.

Interviewer expects: Buckets, hashing, collision handling.

Q54: HashMap vs Hashtable?

Answer: HashMap is not synchronized, allows one null key and null values, faster. Hashtable is synchronized (thread-safe), no null keys/values, legacy class. Use HashMap for single-threaded; ConcurrentHashMap for multi-threaded.

Interviewer expects: ConcurrentHashMap as modern alternative.

Q55: HashMap vs LinkedHashMap vs TreeMap?

Answer: HashMap: no order guarantee. LinkedHashMap: insertion order (or access order). TreeMap: sorted by key (Red-Black tree). Use LinkedHashMap when iteration order matters; TreeMap for sorted test data.

Interviewer expects: Ordering behavior as the differentiator.

Q56: What is the default capacity of ArrayList?

Answer: Default initial capacity is 10. When full, grows by 50% (new capacity = old + old/2). Internal array is copied to new larger array. Pre-size with new ArrayList<>(expectedSize) if you know the count to avoid resizing.

Interviewer expects: Growth strategy and pre-sizing tip.

Q57: What is ConcurrentHashMap?

Answer: Thread-safe Map without locking the entire map. Uses segment/bucket-level locking (Java 7) or CAS operations (Java 8+). Better performance than Hashtable in concurrent scenarios. Use for shared test result maps in parallel execution.

Interviewer expects: Thread-safe alternative to HashMap.

Q58: What is fail-fast iterator?

Answer: Iterator that throws ConcurrentModificationException if collection is modified during iteration (except via iterator's own remove()). ArrayList, HashMap use fail-fast iterators. Detects modification via modCount check.

Interviewer expects: ConcurrentModificationException scenario.

Q59: What is fail-safe iterator?

Answer: Iterator that works on a clone/snapshot of the collection. Does not throw exception on concurrent modification. CopyOnWriteArrayList, ConcurrentHashMap iterators are fail-safe. Trade-off: may not reflect latest data.

Interviewer expects: Snapshot-based iteration concept.

Q60: ArrayList vs Array?

Answer: Array: fixed size, primitive or object types, faster. ArrayList: dynamic size, objects only (autoboxing for primitives), rich API (add, remove, sort). Use ArrayList in frameworks for flexibility; arrays for performance-critical fixed-size data.

Interviewer expects: Dynamic vs fixed size as key difference.

Q61: How do you sort a List?

Answer: Collections.sort(list) or list.sort(comparator). With lambdas: list.sort((a, b) -> a.compareTo(b)) or list.sort(Comparator.naturalOrder()). For custom objects, implement Comparable or pass Comparator. Stream: list.stream().sorted().collect(Collectors.toList()).

Interviewer expects: Comparable vs Comparator distinction.

Q62: Comparable vs Comparator?

Answer: Comparable: natural ordering, implemented by the class itself (compareTo()). Comparator: external comparison logic (compare()). One class, one Comparable; multiple Comparators possible. Use Comparator for sorting WebElements by text length.

Interviewer expects: Internal vs external comparison logic.

Q63: What is Collections.unmodifiableList()?

Answer: Returns a read-only view of a list. Any modification attempt throws UnsupportedOperationException. Does not prevent modification of the backing list. Since Java 9, use List.of() for truly immutable lists.

Interviewer expects: View vs truly immutable distinction.

Q64: What is HashSet internal working?

Answer: HashSet internally uses HashMap. Elements stored as keys with a constant dummy value. Uniqueness enforced by hashCode() and equals(). add() returns false if element already exists. No ordering guarantee.

Interviewer expects: HashMap-backed implementation detail.

Q65: HashSet vs LinkedHashSet vs TreeSet?

Answer: HashSet: no order, fastest. LinkedHashSet: insertion order, slight overhead. TreeSet: sorted order (Red-Black tree), O(log n) operations. Use HashSet for unique test IDs; TreeSet for sorted unique browser names.

Interviewer expects: Ordering and performance trade-offs.

Q66: What is the Iterator interface?

Answer: Provides hasNext(), next(), remove() for traversing collections. Enhanced for loop uses Iterator internally. ListIterator adds previous(), set(), add() for bidirectional traversal. Never modify collection directly while iterating.

Interviewer expects: Safe removal via iterator.remove().

Q67: What is Queue in Java?

Answer: FIFO data structure. LinkedList implements Queue. PriorityQueue for priority ordering. Methods: offer() (add), poll() (remove head), peek() (view head). Useful for managing ordered test execution queues.

Interviewer expects: FIFO principle and key methods.

Q68: What is PriorityQueue?

Answer: Queue where elements are ordered by priority (natural ordering or Comparator). Head is the least element. Not thread-safe. Internally uses heap data structure. O(log n) for insert/remove. Use for prioritizing test retries.

Interviewer expects: Heap-based priority ordering.

Q69: What are Java 9 immutable collections?

Answer: List.of(), Set.of(), Map.of() create immutable collections. Cannot add, remove, or modify. Null elements not allowed. Map.of() supports up to 10 key-value pairs; Map.ofEntries() for more. Preferred over Collections.unmodifiableList().

Interviewer expects: Factory methods and immutability guarantee.

Q70: How does HashMap handle null keys?

Answer: HashMap allows one null key (stored at bucket 0) and multiple null values. Hashtable does not allow null keys or values. Null key's hashCode is 0. Important for default value patterns in config maps.

Interviewer expects: Single null key allowed at bucket 0.

Q71: What is the load factor in HashMap?

Answer: Default load factor is 0.75 — when 75% of buckets are filled, HashMap resizes (doubles capacity) and rehashes all entries. Higher load factor = less memory but more collisions. Lower = more memory, fewer collisions.

Interviewer expects: Resize/rehash trigger mechanism.

Q72: What is CopyOnWriteArrayList?

Answer: Thread-safe ArrayList where every write operation creates a new copy of the underlying array. Iterators never throw ConcurrentModificationException. Best for read-heavy, write-rare scenarios like listener lists in test reporting.

Interviewer expects: Read-heavy, write-rare use case.

Q73: How do you convert List to Array and vice versa?

Answer: List to Array: list.toArray(new String[0]). Array to List: Arrays.asList(array) — fixed size, not resizable. For resizable: new ArrayList<>(Arrays.asList(array)). Stream: Arrays.stream(array).collect(Collectors.toList()).

Interviewer expects: Arrays.asList() fixed-size limitation.

Q74: What is Stream API's relationship with Collections?

Answer: Streams process collection data functionally without modifying the source. collection.stream() creates a stream. Operations: filter, map, sorted (intermediate); collect, forEach, count (terminal). Streams are one-time use; collections are reusable.

Interviewer expects: Non-destructive functional processing.

Q75: What is the difference between Collection and Collections?

Answer: Collection is the root interface for List, Set, Queue. Collections is a utility class with static methods: sort(), reverse(), shuffle(), synchronizedList(), unmodifiableList(). Collections adds functionality; Collection defines contract.

Interviewer expects: Interface vs utility class distinction.

Exception Handling — 15 Questions

Q76: Checked vs unchecked exceptions?

Answer: Checked: compile-time verification required (IOException, SQLException) — must handle or declare. Unchecked: runtime exceptions (NullPointerException, IllegalArgumentException) — no compile-time requirement. Selenium throws mostly unchecked exceptions.

Interviewer expects: Compile-time checking as the difference.

Q77: What is try-catch-finally?

Answer: try: code that may throw exception. catch: handles specific exception types. finally: always executes (cleanup). In tests: try { run test } catch (AssertionError e) { screenshot } finally { driver.quit() }. finally runs even if return is in try/catch.

Interviewer expects: finally always executes (except System.exit).

Q78: What is try-with-resources?

Answer: Auto-closes resources implementing AutoCloseable. try (FileReader fr = new FileReader("file.txt")) { ... } — no finally needed for close(). Java 9+ allows effectively final variables. Preferred over manual close() in finally blocks.

Interviewer expects: AutoCloseable and resource leak prevention.

Q79: Can you have multiple catch blocks?

Answer: Yes. Multiple catch blocks handle different exception types. Catch more specific exceptions before general ones. Java 7+ multi-catch: catch (IOException | SQLException e). Only one catch block executes per exception.

Interviewer expects: Specific before general ordering rule.

Q80: What is exception propagation?

Answer: When an exception is not caught, it propagates up the call stack to the caller. If no caller catches it, JVM terminates the thread. TestNG catches test exceptions and marks test as failed. Use throws keyword to declare propagated checked exceptions.

Interviewer expects: Call stack propagation concept.

Q81: What is a custom exception?

Answer: User-defined exception extending Exception (checked) or RuntimeException (unchecked). Example: ElementNotFoundException extends RuntimeException. Use for domain-specific errors: FrameworkException, ConfigLoadException with meaningful messages.

Interviewer expects: When to create custom vs use standard exceptions.

Q82: finally vs finalize?

Answer: finally: block associated with try-catch, always runs for cleanup, part of exception handling. finalize(): method called by garbage collector before object destruction — deprecated since Java 9, unreliable timing. Never use finalize(); use try-with-resources or Cleaner.

Interviewer expects: finalize is deprecated and unreliable.

Q83: What is the exception hierarchy?

Answer: Throwable → Error (OutOfMemoryError — don't catch) and Exception → RuntimeException (unchecked: NPE, IllegalArgument) and checked exceptions (IOException). Catch specific types, not Throwable or Exception broadly.

Interviewer expects: Error vs Exception, don't catch Error.

Q84: What is throw vs throws?

Answer: throw: explicitly throws an exception instance inside a method — throw new IllegalArgumentException("Invalid browser"). throws: declares exceptions a method may throw — void read() throws IOException. throw is the action; throws is the declaration.

Interviewer expects: Action vs declaration distinction.

Q85: How do you handle exceptions in Selenium?

Answer: Catch specific exceptions: NoSuchElementException, TimeoutException, StaleElementReferenceException. Use WebDriverWait instead of catching. Retry on StaleElementReferenceException. Take screenshot in catch block. Log exception details for debugging.

Interviewer expects: Specific Selenium exception types.

Q86: What is suppressed exception?

Answer: In try-with-resources, if both try block and close() throw exceptions, the try block exception is primary and close() exception is suppressed. Access via getSuppressed(). Ensures no exception information is lost during resource cleanup.

Interviewer expects: try-with-resources multi-exception handling.

Q87: What is StaleElementReferenceException?

Answer: Occurs when a WebElement reference becomes invalid because the DOM was refreshed or the element was removed/re-added. Common after page navigation, AJAX updates, or form submission. Fix: re-find the element or use ExpectedConditions.

Interviewer expects: DOM refresh as root cause, re-find solution.

Q88: Can finally block override return value?

Answer: Yes. If finally contains a return statement, it overrides the return value from try or catch. This is an anti-pattern — avoid returning from finally. Exception in finally can also mask the original exception from try block.

Interviewer expects: Anti-pattern awareness.

Q89: What is exception chaining?

Answer: Wrapping one exception in another to preserve the original cause. throw new FrameworkException("Failed to load config", originalIOException). Access cause via getCause(). Maintains full stack trace for debugging in test frameworks.

Interviewer expects: Cause preservation for debugging.

Q90: What happens if main method throws exception?

Answer: JVM catches uncaught exceptions from main, prints stack trace, and terminates the program. Non-zero exit code. In TestNG, the test runner catches test exceptions, reports them, and continues with remaining tests (depending on configuration).

Interviewer expects: JVM termination vs TestNG continuation.

Multithreading — 15 Questions

Q111: What is the difference between Thread and Runnable?

Answer: Thread is a class; extending it limits inheritance (Java single inheritance). Runnable is a functional interface with run() method — preferred approach. Implement Runnable and pass to Thread: new Thread(() -> runTests()).start(). Separates task from thread execution.

Interviewer expects: Runnable preferred for flexibility.

Q112: What is ThreadLocal and why is it essential for Selenium?

Answer: ThreadLocal provides thread-confined variable storage. Each thread gets its own copy of the variable. Essential for parallel TestNG execution: ThreadLocal<WebDriver> gives each test thread its own browser instance, preventing cross-test interference.

Interviewer expects: Parallel WebDriver isolation solution.

Q113: What is a race condition?

Answer: Occurs when multiple threads access shared mutable data concurrently and the outcome depends on timing. Example: two parallel tests sharing a static WebDriver variable — both may navigate to different URLs simultaneously, causing unpredictable failures.

Interviewer expects: Shared mutable state as root cause.

Q114: Difference between start() and run()?

Answer: start() creates a new thread and calls run() asynchronously in that thread. Calling run() directly executes the method in the current thread — no new thread is created. Always use start() for concurrent execution.

Interviewer expects: run() does not create a new thread.

Q115: Why must you call ThreadLocal.remove()?

Answer: ThreadLocal values persist for the thread's lifetime. In thread pools (TestNG parallel, ExecutorService), threads are reused. Without remove(), old WebDriver references leak into new test executions, causing stale driver bugs and memory leaks.

Interviewer expects: Thread pool reuse and memory leak prevention.

Q116: What is synchronization and when to use it?

Answer: synchronized keyword ensures only one thread executes a method/block at a time on the same object monitor. Prevents race conditions on shared resources. Use for shared counters, report writers, or file append operations in parallel test suites.

Interviewer expects: Mutual exclusion concept with use case.

Q117: What is deadlock?

Answer: Two or more threads permanently blocked, each waiting for a lock held by the other. Classic scenario: Thread A holds Lock1, wants Lock2; Thread B holds Lock2, wants Lock1. Prevention: acquire locks in consistent order, use timeout locks.

Interviewer expects: Circular wait condition awareness.

Q118: How does TestNG parallel execution work?

Answer: Configure parallel="methods|tests|classes" and thread-count in testng.xml. TestNG creates a thread pool and assigns tests to threads. Each thread needs its own WebDriver via ThreadLocal. Use @BeforeMethod/@AfterMethod for per-test setup/teardown.

Interviewer expects: testng.xml parallel attributes and ThreadLocal pairing.

Q119: What is ExecutorService?

Answer: Higher-level API for managing thread pools. Executors.newFixedThreadPool(n) creates n reusable threads. submit(Runnable) queues tasks. shutdown() stops accepting new tasks. More efficient than creating new Thread objects for each task.

Interviewer expects: Thread pool vs raw Thread advantage.

Q120: Is WebDriver thread-safe?

Answer: No. WebDriver implementations are not thread-safe. Multiple threads calling methods on the same WebDriver instance cause unpredictable behavior. Each thread must have its own WebDriver instance, typically managed via ThreadLocal in DriverManager.

Interviewer expects: One driver per thread rule.

Q121: What are the thread states in Java?

Answer: NEW (created), RUNNABLE (ready/running), BLOCKED (waiting for monitor lock), WAITING (wait/join with no timeout), TIMED_WAITING (sleep/join with timeout), TERMINATED (completed). Monitor via Thread.getState().

Interviewer expects: At least NEW, RUNNABLE, TERMINATED states.

Q122: What is volatile keyword?

Answer: Ensures visibility of variable changes across threads. Without volatile, threads may cache variable values locally. Does not provide atomicity for compound operations (i++). Use for simple flags like boolean stopRequested in test runner shutdown.

Interviewer expects: Visibility vs atomicity distinction.

Q123: synchronized method vs synchronized block?

Answer: synchronized method locks on the object instance (or class for static methods). synchronized block locks on a specific object: synchronized(lockObject) { ... }. Blocks offer finer-grained locking — lock only the critical section, not the entire method.

Interviewer expects: Finer-grained control with blocks.

Q124: What is thread-safe reporting in parallel tests?

Answer: Shared report objects (ExtentReports, log files) must handle concurrent writes. Use synchronized blocks, thread-safe collections, or per-thread report instances merged at the end. ExtentReports is thread-safe by design; custom loggers may need synchronization.

Interviewer expects: Shared resource protection in parallel context.

Q125: What is the difference between wait() and sleep()?

Answer: wait() releases the object lock and waits until notify()/notifyAll() or timeout — must be called inside synchronized block. sleep() pauses current thread for specified milliseconds without releasing any lock. Selenium Thread.sleep() is different from Object.wait().

Interviewer expects: Lock release behavior difference.

Q126: Reverse a string without using reverse API.

Q127: Find duplicate elements in an array.

Q128: Check if a string is a palindrome.

Interviewer expects: Two-pointer O(1) space solution.

Q129: Find the second largest element in an array.

Q130: Sort an array without using sort API.

Answer: Bubble sort: compare adjacent elements, swap if out of order, repeat n-1 passes. O(n²) but easy to implement. Better: implement quicksort or mergesort for O(n log n). For interviews, bubble sort is acceptable if you explain trade-offs.

Interviewer expects: Any correct sorting algorithm with complexity stated.

Q131: Count occurrences of each character in a string.

Q132: Find missing number in array of 1 to n.

Interviewer expects: Sum formula or XOR trick.

Q133: Remove duplicates from an ArrayList.

Q134: Swap two numbers without a temp variable.

Q135: Print Fibonacci series up to n terms.
