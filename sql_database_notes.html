Section 1: Database Fundamentals

What is Data?

Data is raw, unorganized facts that represent real-world entities — a user's email address, an order total, a product SKU, a payment timestamp. In QA automation, data is the foundation of every test assertion. When you verify that "Order #1042 shows status SHIPPED on the UI," you are ultimately validating that a specific row in the orders table contains the expected value.

Data exists in many forms:

  • Structured data — rows and columns in relational tables (e-commerce orders, user profiles)
  • Semi-structured data — JSON payloads in API responses, XML config files
  • Unstructured data — product images, PDF invoices, log files

What is a Database?

A database is an organized collection of structured data stored electronically, designed for efficient retrieval, insertion, update, and deletion. Think of it as the persistent memory of your application — unlike in-memory variables, database data survives server restarts, deployments, and crashes.

In a typical e-commerce QA project, the database might contain:

  • users — 50,000 registered customers
  • orders — 200,000 order records spanning 3 years
  • products — 5,000 SKUs across categories
  • payments — transaction records linked to orders
  • addresses — shipping and billing addresses per user

DBMS vs RDBMS

A Database Management System (DBMS) is software that creates, manages, and controls access to databases. An RDBMS (Relational DBMS) is a DBMS that stores data in related tables using rows and columns, enforcing relationships through keys and constraints.

Feature DBMS RDBMS
Data Model File-based or hierarchical; data may be stored in files Relational model — data in tables with defined relationships
Relationships No enforced relationships between data entities Foreign keys enforce referential integrity between tables
ACID Compliance Not guaranteed Full ACID support (Atomicity, Consistency, Isolation, Durability)
Normalization Not required Supports 1NF, 2NF, 3NF to reduce redundancy

Database Architecture

Client-Server Architecture

In the client-server model, the client (your application or test script) sends SQL requests over a network connection to the database server, which processes the query and returns results.

│  Client Layer   │  ────────────────────────► │  Database Server │

        │                                              │

QA Example: After your Selenium test clicks "Place Order," you connect via JDBC to verify the orders table has a new row with status = 'PENDING' and the payments table has a matching charge record.

Most enterprise applications use a three-tier architecture that separates concerns:

Database Objects

Tables

The fundamental storage unit. Each table represents an entity (users, orders, products) with rows (records) and columns (attributes).

CREATE TABLE users (
    user_id       INT PRIMARY KEY,
    email         VARCHAR(255) NOT NULL UNIQUE,
    first_name    VARCHAR(100),
    last_name     VARCHAR(100),
    created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    is_active     BOOLEAN DEFAULT TRUE
);

Views

Virtual tables defined by a SQL query. They do not store data themselves — they present a filtered or joined perspective of underlying tables.

CREATE VIEW active_user_orders AS
SELECT u.user_id, u.email, o.order_id, o.total_amount, o.status
FROM users u
INNER JOIN orders o ON u.user_id = o.user_id
WHERE u.is_active = TRUE;

QA use case: If the UI reads from a view, your test query should also query the view — not the base tables — to match what the application sees.

Indexes

Data structures that speed up data retrieval at the cost of slower writes and additional storage. Like a book's index — you find pages faster without scanning every page.

CREATE INDEX idx_orders_user_id ON orders(user_id);
CREATE INDEX idx_orders_status ON orders(status);
CREATE UNIQUE INDEX idx_users_email ON users(email);

QA use case: After a performance test, verify that slow queries on orders.status are resolved by checking the query execution plan uses idx_orders_status.

Stored Procedures

Precompiled SQL code stored in the database, called with parameters. Encapsulate business logic at the database level.

CREATE PROCEDURE sp_cancel_order(IN p_order_id INT)
BEGIN
    UPDATE orders SET status = 'CANCELLED', updated_at = NOW()
    WHERE order_id = p_order_id AND status = 'PENDING';
    
    UPDATE payments SET status = 'REFUND_PENDING'
    WHERE order_id = p_order_id AND status = 'AUTHORIZED';
END;

QA use case: Test the stored procedure directly with boundary values — cancel a PENDING order (should succeed), cancel a SHIPPED order (should fail/no-op), cancel a non-existent order.

Triggers

Automatic actions fired when INSERT, UPDATE, or DELETE occurs on a table. Used for audit logging, data validation, cascading updates.

CREATE TRIGGER trg_audit_order_status
AFTER UPDATE ON orders
FOR EACH ROW
BEGIN
    IF OLD.status != NEW.status THEN
        INSERT INTO order_audit_log (order_id, old_status, new_status, changed_at)
        VALUES (NEW.order_id, OLD.status, NEW.status, NOW());
    END IF;
END;

QA use case: After updating an order status via the API, verify the trigger created an audit log entry with correct old/new status values.

Sequences

Database objects that generate unique sequential numbers, commonly used for auto-incrementing primary keys.

CREATE SEQUENCE seq_order_id START WITH 10001 INCREMENT BY 1;

INSERT INTO orders (order_id, user_id, total_amount, status)
VALUES (NEXTVAL('seq_order_id'), 42, 149.99, 'PENDING');

QA use case: Verify that concurrent order creation (load test with 100 simultaneous requests) produces unique, gap-free order IDs without duplicates.

Why QA Engineers Need Database Knowledge

Bottom line: UI tests verify what the user sees. API tests verify what the service returns. Database tests verify what actually happened. All three together provide complete test coverage.
  1. Backend validation without UI dependency — When the checkout UI is broken but the API works, SQL queries confirm whether the order was actually created.
  2. Test data setup and teardown — Insert specific user/order states directly instead of clicking through 15 UI screens to reach an edge case.
  3. Data integrity verification — After a refund API call, verify both payments.status = 'REFUNDED' AND orders.status = 'REFUNDED' AND inventory was restored.
  4. Bug investigation — "User says they were charged twice" → query SELECT * FROM payments WHERE user_id = 789 AND order_id = 1042 to find duplicate charge records.
  5. Performance testing analysis — Identify slow queries, missing indexes, and table scan issues from execution plans.
  6. Environment parity checks — Compare staging vs production schema: SELECT column_name, data_type FROM information_schema.columns WHERE table_name = 'orders'
  7. Reporting and reconciliation — Validate that dashboard revenue figures match raw SQL: SELECT SUM(total_amount) FROM orders WHERE status = 'COMPLETED'

Real QA scenario — Order lifecycle validation:

-- Step 1: Verify order created after checkout API call
SELECT order_id, user_id, total_amount, status, created_at
FROM orders
WHERE user_id = 101 AND created_at >= '2026-07-11 00:00:00'
ORDER BY created_at DESC LIMIT 1;
-- Expected: 1 row, status = 'PENDING', total_amount = 249.97

-- Step 2: Verify payment record exists
SELECT payment_id, order_id, amount, status, payment_method
FROM payments
WHERE order_id = (SELECT order_id FROM orders WHERE user_id = 101 ORDER BY created_at DESC LIMIT 1);
-- Expected: 1 row, status = 'AUTHORIZED', amount = 249.97

-- Step 3: Verify order items match cart
SELECT oi.product_id, oi.quantity, oi.unit_price, p.product_name
FROM order_items oi
JOIN products p ON oi.product_id = p.product_id
WHERE oi.order_id = 5001;
-- Expected: 3 rows matching the products added to cart

Interview Questions & Answers

Q1: What is the difference between a database and a DBMS?

Answer: A database is the organized collection of data itself (the files, tables, records). A DBMS is the software system that manages that data — providing tools to create, read, update, delete, and secure it. Analogy: the database is the library's book collection; the DBMS is the librarian and catalog system that helps you find and manage books.

Q2: What is ACID? Why does it matter for testing?

Answer: ACID stands for Atomicity (all-or-nothing transactions), Consistency (data meets all rules/constraints), Isolation (concurrent transactions don't interfere), Durability (committed data survives crashes). For QA: test that a failed payment doesn't leave a half-created order (Atomicity), that negative quantities are rejected (Consistency), that two users buying the last item don't both succeed (Isolation).

Q3: Explain the difference between DBMS and RDBMS with an example.

Answer: DBMS stores data without enforced relationships — like storing user info in one file and orders in another with no link validation. RDBMS stores data in related tables with foreign keys — the orders.user_id must reference a valid users.user_id. MySQL and PostgreSQL are RDBMS; they enforce referential integrity, support JOINs, and provide ACID transactions.

Q4: What is a primary key vs a foreign key?

Answer: A primary key uniquely identifies each row in a table (e.g., users.user_id). A foreign key is a column in one table that references the primary key of another table (e.g., orders.user_id references users.user_id). Foreign keys enforce referential integrity — you cannot insert an order with user_id = 999 if user 999 doesn't exist.

Q5: When would you choose NoSQL over a relational database?

Answer: Choose NoSQL when you need horizontal scaling (millions of IoT events/sec), flexible/evolving schema (user profiles with varying attributes), or specific data models (graph for social networks, document for CMS). Choose SQL when you need complex transactions (banking, e-commerce checkout), strict data integrity, or ad-hoc reporting with JOINs across entities.

Q6: What is normalization? Why is it important?

Answer: Normalization is organizing data to reduce redundancy. 1NF: atomic values (no comma-separated lists in a column). 2NF: no partial dependencies on composite keys. 3NF: no transitive dependencies. Instead of storing user name/email in every order row, you store user_id and JOIN to the users table. This prevents update anomalies — changing a user's email only requires one UPDATE.

Q7: What is denormalization and when is it used?

Answer: Denormalization intentionally adds redundancy to improve read performance. Example: storing product_name in the order_items table even though it exists in products — so order history shows the name at time of purchase even if the product is later renamed. QA must verify denormalized copies stay in sync or are intentionally snapshot values.

Q8: Explain client-server vs file-based database.

Answer: File-based (like SQLite) stores the entire database in a single file on disk — good for mobile apps and local testing. Client-server (like PostgreSQL) runs a dedicated server process handling multiple concurrent clients over TCP/IP — required for production web applications with hundreds of simultaneous users.

Q9: What database objects would you verify after a schema migration?

Answer: Verify tables exist with correct columns and data types, primary keys and foreign keys are intact, indexes are recreated, views compile without errors, stored procedures and triggers are present, sequences have correct current values, and grants/permissions are applied. Run: SELECT * FROM information_schema.tables WHERE table_schema = 'public' and compare against expected schema.

Q10: How would you test a database trigger?

Answer: (1) Identify the trigger's table and firing event (BEFORE/AFTER INSERT/UPDATE/DELETE). (2) Perform the triggering action. (3) Query the affected/audit table to verify the trigger's side effect. (4) Test edge cases: NULL values, bulk updates, rollback scenarios. (5) Verify the trigger does NOT fire when it shouldn't (e.g., updating a non-status column shouldn't create an audit log).

Q11: What is a schema in a database?

Answer: A schema is a logical container/grouping of database objects (tables, views, procedures). In PostgreSQL, schemas namespace objects — public.users vs audit.users. In MySQL, schema is synonymous with database. QA uses schema comparison tools to ensure staging matches production structure before release.

Q12: How do you connect to a database in an automation framework?

Answer: Common approaches: Java — JDBC with connection strings and PreparedStatement; Python — psycopg2/sqlalchemy for PostgreSQL, pymysql for MySQL; Test frameworks — DBUnit (Java) for test fixtures, Flyway/Liquibase for migration testing. Always use connection pooling, parameterized queries (never string concatenation), and close connections in finally blocks/teardown methods.

Section 2: SQL Basics (DDL, DML, DCL, TCL)

SQL commands are categorized into four groups. As a QA engineer, you'll primarily use DML (querying and modifying test data) and DDL (setting up test schemas). Know DCL and TCL for understanding permissions and transaction behavior.

Category Full Name Commands QA Usage
DDL Data Definition Language CREATE, ALTER, DROP, TRUNCATE, RENAME Setting up test schemas, modifying table structure

SELECT Interview Questions

Q: What is the difference between SELECT * and selecting specific columns?
A: SELECT * returns all columns, which is convenient for exploration but bad for production — it breaks if columns are added/removed, transfers unnecessary data over the network, and prevents index-only scans. Always select only needed columns in automation scripts.

INSERT Interview Questions

Q: What happens if you INSERT without specifying column names?
A: Values must be provided for ALL columns in table definition order. If the table has an auto-increment ID and 6 other columns, you must provide all 7 values in exact order. Always specify column names to avoid breakage when schema changes.

Q: Can INSERT cause a transaction rollback?
A: Yes. If an INSERT violates a constraint (UNIQUE, NOT NULL, FOREIGN KEY, CHECK), the database raises an error. In a transaction block, this typically aborts the entire transaction unless error handling catches it. Test by inserting duplicate emails, invalid foreign keys, and NULL into NOT NULL columns.

UPDATE Statement

Syntax

UPDATE table_name
SET column1 = value1, column2 = value2, ...
[WHERE condition];

Example 1: Update single column

UPDATE products
SET stock_quantity = stock_quantity - 1
WHERE product_id = 15 AND stock_quantity > 0;

Example 2: Update multiple columns

UPDATE orders
SET status = 'SHIPPED',
    shipped_at = CURRENT_TIMESTAMP,
    tracking_number = 'TRK-789456123'
WHERE order_id = 5001;

Example 3: UPDATE with subquery

UPDATE payments
SET status = 'REFUNDED', refunded_at = CURRENT_TIMESTAMP
WHERE order_id IN (
    SELECT order_id FROM orders WHERE status = 'CANCELLED' AND updated_at >= '2026-07-01'
);
CRITICAL QA WARNING: An UPDATE without a WHERE clause modifies EVERY row in the table. In 2016, a developer at a UK company ran UPDATE users SET status = 'inactive' without WHERE on production — wiping 870GB of data. Always include WHERE, and in test environments, wrap updates in transactions: BEGIN; UPDATE ...; SELECT * FROM ...; ROLLBACK;

QA Testing Scenario: Verify that cancelling an order via API updates all related records:

-- Before API call
SELECT status FROM orders WHERE order_id = 5002;        -- PENDING
SELECT status FROM payments WHERE order_id = 5002;      -- AUTHORIZED

-- After API call: POST /orders/5002/cancel

-- Verification
SELECT status FROM orders WHERE order_id = 5002;        -- Expected: CANCELLED
SELECT status FROM payments WHERE order_id = 5002;      -- Expected: REFUND_PENDING
SELECT stock_quantity FROM products WHERE product_id = 8; -- Expected: restored by +1

Common Mistakes Testers Make

  1. Forgetting WHERE in UPDATE/DELETE — Always double-check before executing. Use SELECT with the same WHERE first to preview affected rows.
  2. Not using transactions for test data changes — Wrap in BEGIN/ROLLBACK during exploration to avoid polluting shared test environments.
  3. Hardcoding IDs that change between environments — Use dynamic lookups: WHERE email = 'test@qa.com' instead of WHERE user_id = 42.
  4. Ignoring soft deletes — Many apps use is_deleted = TRUE instead of DELETE. Querying without this filter returns "deleted" records.
  5. Not accounting for timezone differences — created_at stored in UTC but compared against local time causes off-by-one-day failures.
  6. Running DDL in shared environments — CREATE/ALTER/DROP auto-commit and affect all teams. Never run DDL in staging without coordination.
  7. String concatenation in queries — "SELECT * FROM users WHERE email = '" + email + "'" is a SQL injection vulnerability. Always use parameterized queries.

Section 2 Interview Questions

Q1: What is the difference between DELETE and TRUNCATE?

Answer: DELETE is DML — removes rows one by one, can use WHERE, fires triggers, can be rolled back, and does not reset auto-increment. TRUNCATE is DDL — removes all rows by deallocating pages, cannot use WHERE, typically doesn't fire triggers, is faster, and resets auto-increment counters.

Q2: What happens if you run UPDATE without WHERE?

Answer: Every row in the table is updated. This is one of the most dangerous SQL mistakes. In testing, always preview with SELECT first. In automation, never construct UPDATE statements without a WHERE clause programmatically.

Q3: Explain PRIMARY KEY vs UNIQUE constraint.

Answer: Both enforce uniqueness. PRIMARY KEY additionally: cannot be NULL, only one per table, automatically creates a clustered index (in SQL Server). UNIQUE allows one NULL (or multiple NULLs depending on DB), multiple UNIQUE constraints per table.

Q4: What is a CHECK constraint? Give a QA test example.

Answer: CHECK validates data against a condition before insert/update. Example: CHECK (total_amount >= 0). QA test: attempt to insert an order with total_amount = -50 via API — expect 400 error and no database row created.

Q5: How do you safely test an UPDATE in a shared environment?

Answer: Use a transaction: BEGIN; UPDATE orders SET status = 'CANCELLED' WHERE order_id = 9999; SELECT status FROM orders WHERE order_id = 9999; ROLLBACK; This verifies the update logic without persisting changes.

Q6: What is the difference between DDL and DML?

Answer: DDL (CREATE, ALTER, DROP) defines/modifies structure and auto-commits. DML (SELECT, INSERT, UPDATE, DELETE) manipulates data and can be rolled back within transactions. QA primarily uses DML but needs DDL awareness for schema validation.

Q7: Can you INSERT into a table with an auto-increment primary key without specifying the ID?

Answer: Yes. Omit the auto-increment column and the database generates the next value automatically. Verify with: INSERT INTO users (email, first_name) VALUES ('test@test.com', 'Test'); SELECT LAST_INSERT_ID(); (MySQL) or RETURNING user_id (PostgreSQL).

Q8: What is a foreign key constraint and how do you test it?

Answer: Ensures referential integrity — orders.user_id must exist in users.user_id. Test by attempting to insert an order with a non-existent user_id — expect constraint violation error. Also test ON DELETE CASCADE: deleting a user should cascade-delete their orders if configured.

Section 3 Interview Questions

Q1: What is the difference between WHERE and HAVING?

Answer: WHERE filters rows before grouping. HAVING filters groups after GROUP BY. Example: WHERE status = 'COMPLETED' filters individual orders; HAVING COUNT(*) > 5 filters users with more than 5 orders.

Q2: Why does WHERE column = NULL not work?

Answer: NULL represents unknown, so comparing anything to NULL yields NULL (unknown), not TRUE. Use IS NULL or IS NOT NULL instead.

Q3: What is the difference between IN and EXISTS?

Answer: IN checks if a value matches any value in a list/subquery result. EXISTS checks if a subquery returns at least one row. EXISTS often performs better with large datasets because it short-circuits on first match. Use EXISTS when checking for related records: WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.user_id).

Q4: How does LIKE differ from exact match (=)?

Answer: = requires exact match of the entire string. LIKE performs pattern matching with wildcards (% for any sequence, _ for single character). LIKE 'Mouse%' matches "Mouse", "Mouse Pad", "Mousepad Pro". = 'Mouse' matches only "Mouse". LIKE cannot use indexes efficiently (leading wildcard prevents index use).

Q5: Write a query to find duplicate emails in the users table.

Answer: SELECT email, COUNT(*) AS cnt FROM users GROUP BY email HAVING COUNT(*) > 1;

Q6: What happens with NOT IN when the subquery returns NULL?

Answer: NOT IN returns zero rows if any value in the subquery result is NULL. This is because x NOT IN (1, 2, NULL) evaluates to x != 1 AND x != 2 AND x != NULL, and x != NULL is always NULL/false. Use NOT EXISTS to avoid this trap.

Q7: How do you paginate results in SQL?

Answer: Use LIMIT/OFFSET (MySQL/PostgreSQL) or FETCH FIRST/OFFSET (SQL standard). Page N with size S: LIMIT S OFFSET (N-1)*S. Always include ORDER BY for deterministic pagination.

Q8: What is the default sort order of ORDER BY?

Answer: ASC (ascending). Numbers: 1, 2, 3. Strings: A, B, C. Dates: oldest first. NULL placement varies by database — PostgreSQL: NULLS LAST in ASC, NULLS FIRST in DESC.

Q9: How would you test a date range filter on a reporting page?

Answer: Create test data with orders on: start date, end date, day before start, day after end, and NULL dates. Apply the filter and verify only in-range orders appear. Cross-check count with: SELECT COUNT(*) FROM orders WHERE created_at BETWEEN 'start' AND 'end'.

Q10: Difference between DISTINCT and GROUP BY?

Answer: Both eliminate duplicates. DISTINCT is simpler — removes duplicate row combinations. GROUP BY groups rows and allows aggregate functions (COUNT, SUM). Use DISTINCT for unique values; GROUP BY when you need aggregations per group.

Q11: How do you filter for records updated in the last hour?

Answer: SELECT * FROM orders WHERE updated_at >= CURRENT_TIMESTAMP - INTERVAL '1 hour'; (PostgreSQL). MySQL: WHERE updated_at >= NOW() - INTERVAL 1 HOUR. Always verify timezone handling.

Q12: Write a query to find products that have never been ordered.

Answer: SELECT p.product_id, p.product_name FROM products p LEFT JOIN order_items oi ON p.product_id = oi.product_id WHERE oi.product_id IS NULL;

Section 4 Interview Questions

Q1: What is the difference between COUNT(*) and COUNT(column)?

Answer: COUNT(*) counts all rows including those with NULL values in any column. COUNT(column) counts only rows where the specified column is NOT NULL. If 7 orders exist but 1 has NULL total_amount, COUNT(*) = 7 and COUNT(total_amount) = 6.

Q2: How does SUM handle NULL values?

Answer: SUM ignores NULLs entirely. SUM(100, NULL, 200) = 300. If all values are NULL, SUM returns NULL (not 0). Use COALESCE(SUM column), 0) when you need zero instead of NULL.

Q3: Can you use aggregate functions in WHERE clause?

Answer: No. WHERE filters individual rows before aggregation. Use HAVING to filter after GROUP BY. Example: HAVING COUNT(*) > 5 not WHERE COUNT(*) > 5.

Q4: What does COUNT(DISTINCT column) do?

Answer: Counts the number of unique non-NULL values. COUNT(DISTINCT user_id) from 7 orders where 4 unique users exist returns 4. Useful for "how many unique customers ordered" metrics.

Q5: Write a query to find users who spent more than $500 total.

Answer: SELECT user_id, SUM(total_amount) AS total FROM orders WHERE status = 'COMPLETED' GROUP BY user_id HAVING SUM(total_amount) > 500;

Q6: Why might SUM(order_total) differ from the admin dashboard revenue figure?

Answer: Possible causes: dashboard excludes cancelled/refunded orders, uses a different date range, applies currency conversion, includes/excludes tax, caches stale data, or reads from a materialized view not yet refreshed. QA should replicate the exact dashboard query logic.

Q7: What is the difference between AVG and calculating SUM/COUNT manually?

Answer: They are equivalent when no NULLs exist: AVG(x) = SUM(x)/COUNT(x). With NULLs, AVG excludes them from both numerator and denominator. Manual SUM/COUNT(*) would divide by total rows including NULL rows, giving a different result.

Q8: How do you validate that a "Top 10 Products" widget is correct?

Answer: Run the same query the widget uses: SELECT product_id, SUM(quantity) AS sold FROM order_items GROUP BY product_id ORDER BY sold DESC LIMIT 10. Compare each product and count with the UI. Also verify the time filter matches (last 30 days vs all time).

Q9: Can MIN/MAX work on non-numeric columns?

Answer: Yes. MIN/MAX work on dates (earliest/latest), strings (alphabetically first/last), and any comparable type. MIN(created_at) gives the earliest date. MIN(first_name) gives alphabetically first name.

Q10: Write a query to calculate conversion rate (completed orders / total orders).

Answer: SELECT ROUND(100.0 * COUNT(CASE WHEN status = 'COMPLETED' THEN 1 END) / COUNT(*), 2) AS conversion_rate FROM orders;

Common JOIN Mistakes

  1. Missing JOIN condition → CROSS JOIN effect: Forgetting ON clause produces Cartesian product. 1000 orders × 1000 users = 1,000,000 rows.
  2. Joining on wrong columns: ON u.user_id = o.order_id instead of ON u.user_id = o.user_id — silently wrong results.
  3. Using INNER JOIN when LEFT JOIN is needed: Missing records that should appear with NULLs (users without orders).
  4. Not aliasing tables: Ambiguous column names cause errors or wrong results. Always alias: users u, orders o.
  5. Filtering on right table in WHERE instead of ON: LEFT JOIN orders o ON u.user_id = o.user_id WHERE o.status = 'COMPLETED' converts LEFT JOIN to INNER JOIN effectively. Move filter to ON clause to preserve left rows.
  6. Counting duplicates after JOIN: Joining orders to order_items inflates order count. Use COUNT(DISTINCT o.order_id) or subquery.

Section 5 Interview Questions (15+)

Q1: What is the difference between INNER JOIN and LEFT JOIN?

Answer: INNER JOIN returns only matching rows from both tables. LEFT JOIN returns all rows from the left table plus matches from the right; unmatched right columns are NULL. Use INNER when you only want records with relationships; LEFT when you need all records from the primary table regardless of matches.

Q2: How do you find records in table A that have no match in table B?

Answer: SELECT a.* FROM A a LEFT JOIN B b ON a.key = b.key WHERE b.key IS NULL; Or: SELECT * FROM A WHERE key NOT IN (SELECT key FROM B) (careful with NULLs). Or: SELECT * FROM A a WHERE NOT EXISTS (SELECT 1 FROM B b WHERE b.key = a.key).

Q3: What happens if you forget the ON clause in a JOIN?

Answer: You get a CROSS JOIN (Cartesian product). Every row in table A pairs with every row in table B. With 1000 rows each, that's 1,000,000 rows — likely causing performance issues or incorrect results.

Q4: Can you join more than two tables?

Answer: Yes. Chain JOINs: FROM A JOIN B ON ... JOIN C ON ... JOIN D ON .... Each additional JOIN adds another table to the result set. QA order validation typically joins 4-6 tables.

Q5: What is a SELF JOIN? Give an example.

Answer: Joining a table to itself using aliases. Example: employee-manager hierarchy where employees.manager_id references employees.emp_id. SELECT e.name, m.name AS manager FROM employees e LEFT JOIN employees m ON e.manager_id = m.emp_id.

Q6: LEFT JOIN vs RIGHT JOIN — are they different?

Answer: A LEFT JOIN B is equivalent to B RIGHT JOIN A. They produce the same result — just swap which table is considered "left" vs "right." Most developers prefer LEFT JOIN and swap table order rather than using RIGHT JOIN.

Q7: Why does LEFT JOIN with WHERE on the right table exclude NULL rows?

Answer: WHERE filters AFTER the join. WHERE o.status = 'COMPLETED' removes rows where o.status is NULL (unmatched left rows). To filter right table while keeping unmatched left rows, put the condition in ON: LEFT JOIN orders o ON u.user_id = o.user_id AND o.status = 'COMPLETED'.

Q8: How do you detect orphaned records using JOINs?

Answer: Orphan = child record without valid parent. SELECT oi.* FROM order_items oi LEFT JOIN orders o ON oi.order_id = o.order_id WHERE o.order_id IS NULL — finds order items referencing non-existent orders.

Q9: What is the difference between JOIN and UNION?

Answer: JOIN combines columns from different tables horizontally (adding columns). UNION combines rows from different queries vertically (stacking rows). JOIN needs related columns; UNION needs same number and compatible types of columns.

Q10: How would you find users who ordered every product in the Electronics category?

Answer: SELECT u.user_id, u.email FROM users u WHERE NOT EXISTS (SELECT p.product_id FROM products p WHERE p.category = 'Electronics' AND NOT EXISTS (SELECT 1 FROM orders o JOIN order_items oi ON o.order_id = oi.order_id WHERE o.user_id = u.user_id AND oi.product_id = p.product_id)) — uses relational division pattern.

Q11: Explain CROSS JOIN with a QA example.

Answer: CROSS JOIN produces every combination of rows from both tables. QA use: generate all browser × OS test configurations from a browsers table (5 rows) and os table (4 rows) = 20 test combinations automatically.

Q12: You join orders to order_items and get inflated revenue. Why?

Answer: One order with 3 line items produces 3 joined rows. SUM(o.total_amount) counts the order total 3 times. Fix: SUM from a subquery, use SUM(DISTINCT o.order_id) with a different approach, or aggregate order_items separately.

Q13: Write a query showing each user and their most recent order.

Answer: SELECT u.user_id, u.email, o.order_id, o.created_at FROM users u LEFT JOIN orders o ON u.user_id = o.user_id AND o.created_at = (SELECT MAX(o2.created_at) FROM orders o2 WHERE o2.user_id = u.user_id);

Q14: Does MySQL support FULL OUTER JOIN?

Answer: No. Simulate with UNION of LEFT and RIGHT JOIN: SELECT ... FROM A LEFT JOIN B ... UNION SELECT ... FROM A RIGHT JOIN B WHERE A.key IS NULL.

Q15: How do JOINs help in API test validation?

Answer: After POST /orders API returns 201, a single JOIN query verifies: order exists with correct user_id, line items match request payload, payment record created with matching amount, inventory decremented — all in one query instead of 4 separate SELECTs.

Q16: What is the difference between ON and WHERE in a JOIN?

Answer: ON specifies the join condition (evaluated during join). WHERE filters the final result set (evaluated after join). For INNER JOIN, they're interchangeable. For OUTER JOINs, conditions in ON preserve unmatched rows; conditions in WHERE filter them out.

Q17: Find pairs of products ordered together frequently.

Answer: SELECT oi1.product_id AS p1, oi2.product_id AS p2, COUNT(*) AS times_together FROM order_items oi1 INNER JOIN order_items oi2 ON oi1.order_id = oi2.order_id AND oi1.product_id < oi2.product_id GROUP BY oi1.product_id, oi2.product_id HAVING COUNT(*) > 5 ORDER BY times_together DESC;

Section 6: Subqueries

A subquery (inner query/nested query) is a SQL query embedded within another query. The outer query uses the result of the inner query. Subqueries are powerful for QA validation, especially for existence checks, comparisons, and data reconciliation.

Single-Row Subqueries

Return exactly one row and one column. Used with comparison operators: =, !=, <, >, <=, >=

Example 1: Orders above average order value

SELECT order_id, user_id, total_amount, status
FROM orders
WHERE total_amount > (SELECT AVG(total_amount) FROM orders WHERE status = 'COMPLETED');

-- Inner query returns single value: e.g., 183.865
-- Outer query finds orders exceeding that average

Example 2: User who placed the highest-value order

SELECT u.user_id, u.email, o.order_id, o.total_amount
FROM users u
JOIN orders o ON u.user_id = o.user_id
WHERE o.total_amount = (SELECT MAX(total_amount) FROM orders);
-- Expected: Dave, order 5004, $320.50

Example 3: Products priced above category average

SELECT product_id, product_name, price, category
FROM products p
WHERE price > (
    SELECT AVG(price) FROM products WHERE category = p.category
);

Note: This is actually a correlated subquery (references outer query's p.category). See correlated section below.

QA Testing Scenario: Verify the "Premium Orders" dashboard filter shows orders above average:

-- Get the threshold the UI should use
SELECT AVG(total_amount) AS threshold FROM orders WHERE status = 'COMPLETED';

-- Get orders that should appear
SELECT order_id, total_amount FROM orders
WHERE total_amount > (SELECT AVG(total_amount) FROM orders WHERE status = 'COMPLETED')
  AND status = 'COMPLETED';

Multi-Row Subqueries

Return multiple rows. Used with IN, ANY, ALL operators.

IN — Match any value in the set

-- Users who have placed at least one order
SELECT user_id, email, first_name
FROM users
WHERE user_id IN (SELECT DISTINCT user_id FROM orders);

-- Products that have been ordered
SELECT product_id, product_name, price
FROM products
WHERE product_id IN (SELECT DISTINCT product_id FROM order_items);

-- Orders paid via credit card
SELECT order_id, total_amount, status
FROM orders
WHERE order_id IN (
    SELECT order_id FROM payments WHERE payment_method = 'CREDIT_CARD'
);

ANY — Compare against any value (OR logic)

-- Products cheaper than ANY product in the Furniture category
SELECT product_id, product_name, price
FROM products
WHERE price < ANY (SELECT price FROM products WHERE category = 'Furniture');
-- Returns products cheaper than at least one furniture item (i.e., cheaper than the most expensive furniture)

ALL — Compare against all values (AND logic)

-- Products cheaper than ALL products in the Furniture category
SELECT product_id, product_name, price
FROM products
WHERE price < ALL (SELECT price FROM products WHERE category = 'Furniture');
-- Returns products cheaper than even the cheapest furniture item

Correlated Subqueries

The inner query references columns from the outer query. Executed once per row of the outer query — like a nested loop.

Correlated Subquery Execution:

Outer Query Row 1 (user_id=101)
  └── Inner Query runs: SELECT COUNT(*) FROM orders WHERE user_id = 101 → 2
Outer Query Row 2 (user_id=102)
  └── Inner Query runs: SELECT COUNT(*) FROM orders WHERE user_id = 102 → 1
Outer Query Row 3 (user_id=103)
  └── Inner Query runs: SELECT COUNT(*) FROM orders WHERE user_id = 103 → 0
... and so on for each outer row

Example 1: Users with more than 1 order

SELECT u.user_id, u.email, u.first_name
FROM users u
WHERE (
    SELECT COUNT(*) FROM orders o WHERE o.user_id = u.user_id
) > 1;
-- Expected: Alice (user 101) — she has 2 orders

Example 2: Each user's most recent order

SELECT u.user_id, u.email, o.order_id, o.total_amount, o.created_at
FROM users u
JOIN orders o ON u.user_id = o.user_id
WHERE o.created_at = (
    SELECT MAX(o2.created_at)
    FROM orders o2
    WHERE o2.user_id = u.user_id
);

Example 3: Products never ordered (correlated NOT EXISTS)

SELECT p.product_id, p.product_name, p.price
FROM products p
WHERE NOT EXISTS (
    SELECT 1 FROM order_items oi WHERE oi.product_id = p.product_id
);

Nested Subqueries (Multiple Levels)

-- Users who ordered products in the most popular category
SELECT DISTINCT u.user_id, u.email
FROM users u
WHERE u.user_id IN (
    SELECT o.user_id FROM orders o
    WHERE o.order_id IN (
        SELECT oi.order_id FROM order_items oi
        WHERE oi.product_id IN (
            SELECT product_id FROM products
            WHERE category = (
                SELECT category FROM products
                GROUP BY category
                ORDER BY COUNT(*) DESC
                LIMIT 1
            )
        )
    )
);

Deep nesting hurts readability. Refactor to JOINs or CTEs (Common Table Expressions) for maintainability.

Subquery in Different Clauses

Subquery in SELECT (Scalar subquery as column)

SELECT
    u.user_id,
    u.email,
    (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.user_id) AS order_count,
    (SELECT COALESCE(SUM(total_amount), 0) FROM orders o WHERE o.user_id = u.user_id AND o.status = 'COMPLETED') AS total_spent
FROM users u;

Subquery in FROM (Derived table / inline view)

SELECT user_stats.user_id, user_stats.order_count, user_stats.total_spent
FROM (
    SELECT
        user_id,
        COUNT(*) AS order_count,
        SUM(total_amount) AS total_spent
    FROM orders
    WHERE status = 'COMPLETED'
    GROUP BY user_id
) AS user_stats
WHERE user_stats.total_spent > 200;

Subquery in WHERE (Filtering)

-- Orders where payment amount doesn't match order total (data integrity bug)
SELECT order_id, total_amount
FROM orders o
WHERE total_amount != (
    SELECT amount FROM payments p WHERE p.order_id = o.order_id
);

Subquery in HAVING (Filter grouped results)

-- Categories with average product price above overall average
SELECT category, AVG(price) AS avg_category_price
FROM products
GROUP BY category
HAVING AVG(price) > (SELECT AVG(price) FROM products);

Performance Considerations

  1. Correlated subqueries are expensive — They execute once per outer row. For 10,000 users, the inner query runs 10,000 times. Rewrite as JOIN when possible.
  2. IN with large subquery results — Materializes the entire subquery. Use EXISTS for better performance with large datasets.
  3. Subquery in SELECT — Runs for every row in the result. Consider JOIN with GROUP BY instead.
  4. Use EXPLAIN/EXPLAIN ANALYZE — Check query execution plan. Look for "SubPlan" or "Nested Loop" which indicate correlated subquery overhead.
-- Slow: correlated subquery in SELECT
SELECT u.email, (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.user_id) AS cnt
FROM users u;

-- Faster: JOIN with GROUP BY
SELECT u.email, COUNT(o.order_id) AS cnt
FROM users u
LEFT JOIN orders o ON u.user_id = o.user_id
GROUP BY u.user_id, u.email;

Section 6 Interview Questions

Q1: What is a subquery? Where can it be used?

Answer: A query nested inside another query. Can appear in SELECT (scalar column), FROM (derived table), WHERE (filtering), and HAVING (group filtering) clauses. The outer query uses the inner query's result.

Q2: What is the difference between single-row and multi-row subqueries?

Answer: Single-row returns one value — use with =, <, >. Multi-row returns multiple values — use with IN, ANY, ALL. Using = with a multi-row subquery causes a runtime error.

Q3: What is a correlated subquery?

Answer: A subquery that references a column from the outer query. It executes once for each row of the outer query. Example: finding users with above-average order counts where the inner query references outer user_id.

Q4: EXISTS vs IN — when to use which?

Answer: Use EXISTS for existence checks ("has this user placed an order?") — it's NULL-safe and short-circuits. Use IN for matching against a small set of values. Avoid NOT IN when subquery might return NULLs; use NOT EXISTS instead.

Q5: What happens if a single-row subquery returns multiple rows?

Answer: Database raises an error: "subquery returns more than one row." This is a common bug when the inner query isn't properly filtered to guarantee a single result. Use LIMIT 1, MAX(), or IN depending on intent.

Q6: How do you find orphan records using subqueries?

Answer: SELECT * FROM child_table c WHERE NOT EXISTS (SELECT 1 FROM parent_table p WHERE p.id = c.parent_id) — finds child records with no matching parent. Safe with NULLs unlike NOT IN.

Q7: Can a subquery return multiple columns?

Answer: In the FROM clause (derived table), yes — it can return any number of columns. In WHERE with IN, only one column. In SELECT as scalar subquery, exactly one column and one row.

Q8: Write a subquery to find the second highest order amount.

Answer: SELECT MAX(total_amount) FROM orders WHERE total_amount < (SELECT MAX(total_amount) FROM orders); Or using OFFSET: SELECT total_amount FROM orders ORDER BY total_amount DESC LIMIT 1 OFFSET 1;

Q9: What is the difference between ANY and ALL?

Answer: ANY: condition true if it holds for at least one value in the set (OR logic). ALL: condition true only if it holds for every value in the set (AND logic). price < ANY(100,200,300) means price < 100 OR price < 200 OR price < 300, which simplifies to price < 300.

Q10: How would you rewrite a correlated subquery as a JOIN?

Answer: SELECT u.email FROM users u WHERE (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.user_id) > 1 becomes SELECT u.email FROM users u JOIN (SELECT user_id, COUNT(*) AS cnt FROM orders GROUP BY user_id HAVING COUNT(*) > 1) o ON u.user_id = o.user_id.

Q11: Why is NOT IN dangerous with NULLs?

Answer: If the subquery returns ANY NULL value, NOT IN returns zero rows for all inputs. Because x NOT IN (1, 2, NULL) evaluates to x != 1 AND x != 2 AND x != NULL, and the last comparison is always UNKNOWN/false. Always use NOT EXISTS instead.

Q12: How do you use a subquery in the FROM clause?

Answer: Wrap the subquery in parentheses with an alias: SELECT * FROM (SELECT user_id, COUNT(*) AS cnt FROM orders GROUP BY user_id) AS order_counts WHERE cnt > 5. The alias is mandatory in most databases.

GROUP BY Fundamentals

GROUP BY collapses rows that share the same value(s) in the specified column(s) into a single group. Aggregate functions (COUNT, SUM, AVG, MIN, MAX) then operate on each group independently.

Golden Rule for QA: Every column in the SELECT list must either appear in GROUP BY or be wrapped in an aggregate function. Violating this causes SQL errors in strict databases (PostgreSQL, SQL Server) or unpredictable results in lenient ones (MySQL with ONLY_FULL_GROUP_BY disabled).

FROM orders
WHERE status NOT IN ('CANCELLED', 'FAILED')
GROUP BY customer_id
ORDER BY order_count DESC;

       COUNT(DISTINCT oi.order_id) AS orders_with_category,

JOIN products p ON oi.product_id = p.product_id
JOIN orders o ON oi.order_id = o.order_id
WHERE o.status = 'COMPLETED'
GROUP BY p.category_id
ORDER BY category_revenue DESC;

ROLLUP and CUBE — Brief Overview

ROLLUP generates subtotal and grand total rows. CUBE generates all possible grouping combinations — powerful for ad-hoc QA dashboards but expensive on large datasets.

-- ROLLUP: subtotals per category, then grand total
SELECT category_id,
       SUM(quantity * unit_price) AS revenue
FROM order_items oi
JOIN products p ON oi.product_id = p.product_id
GROUP BY ROLLUP (category_id);
-- Result includes: per-category rows + one row with category_id = NULL (grand total)

-- CUBE: all combinations of category and payment method
SELECT p.category_id, pay.payment_method, SUM(o.total_amount) AS revenue
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
JOIN payments pay ON o.order_id = pay.order_id
WHERE o.status = 'COMPLETED'
GROUP BY CUBE (p.category_id, pay.payment_method);

Interview Questions & Answers (12 Questions)

  1. What is the difference between GROUP BY and DISTINCT?
    DISTINCT removes duplicate row combinations. GROUP BY groups rows for aggregate calculations. SELECT DISTINCT customer_id FROM orders lists unique customers; SELECT customer_id, COUNT(*) FROM orders GROUP BY customer_id counts orders per customer.
  2. Can you use WHERE and HAVING together?
    Yes. WHERE filters rows before grouping; HAVING filters groups after. Example: find customers with more than 5 completed orders in 2025 — filter completed orders in WHERE, then HAVING COUNT(*) > 5.
  3. Why can't you write HAVING without GROUP BY?
    In standard SQL, HAVING without GROUP BY treats the entire result as one group (valid but rare). Most real queries pair HAVING with GROUP BY. Some databases allow HAVING COUNT(*) > 0 without explicit GROUP BY.
  4. What happens if you GROUP BY a nullable column?
    All NULL values form a single group. QA implication: if region is NULL for many customers, you get one group with NULL — verify whether NULL means "unknown" or a data defect.
  5. COUNT(*) vs COUNT(column) vs COUNT(DISTINCT column)?
    COUNT(*) counts all rows in the group including NULLs. COUNT(column) counts non-NULL values only. COUNT(DISTINCT column) counts unique non-NULL values — use for unique customer counts per day.
  6. How do you find duplicate records in SQL?
    GROUP BY the business key, HAVING COUNT(*) > 1. For emails: normalize with LOWER(TRIM(email)) first. Always drill into full rows for remediation.
  7. Write a query to find customers who placed orders in every month of 2025.
    SELECT customer_id
FROM orders
WHERE order_date >= '2025-01-01' AND order_date < '2026-01-01'
GROUP BY customer_id
HAVING COUNT(DISTINCT DATE_TRUNC('month', order_date)) = 12;
  8. What is the execution order of SQL clauses?
    FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. This explains why WHERE cannot reference SELECT aliases or aggregates.
  9. How would you validate monthly sales report accuracy as QA?
    Reconcile report totals against raw SUM(total_amount) with same filters. Cross-check order count. Spot-check random months manually. Compare staging vs production reports for same date range after deployment.
  10. GROUP BY with JOINs — common pitfall?
    Joining before GROUP BY can inflate counts if the join is one-to-many. Example: joining orders to order_items then COUNT(*) counts line items, not orders. Use COUNT(DISTINCT o.order_id) instead.
  11. What is ROLLUP used for in testing?
    Quick subtotal validation — verify category subtotals sum to grand total in financial reports. If subtotals don't reconcile, aggregation logic or data duplication in joins is suspect.
  12. How do you test for orphan records after a migration?
    Run LEFT JOIN queries from child to parent; assert zero rows where parent key IS NULL. Automate as post-migration smoke tests. Example: order_items LEFT JOIN orders WHERE orders.order_id IS NULL.

Section 8: CTEs & Window Functions

Common Table Expressions (CTEs) and window functions are among the most frequently tested SQL topics in SDET interviews. They enable readable complex queries, pagination, deduplication, running totals, and trend analysis — all critical for API-DB validation and data quality automation.

CTE (Common Table Expression) — Syntax and Advantages

A CTE is a temporary named result set defined with the WITH clause, referenced in the main query. It improves readability and allows query reuse within a single statement.

-- Basic CTE syntax
WITH completed_orders AS (
    SELECT order_id, customer_id, total_amount, order_date
    FROM orders
    WHERE status = 'COMPLETED'
)
SELECT customer_id,
       COUNT(*) AS order_count,
       SUM(total_amount) AS lifetime_value
FROM completed_orders
GROUP BY customer_id
HAVING SUM(total_amount) > 1000;

Recursive CTE — Hierarchical Data

Recursive CTEs traverse tree structures: org charts, category trees, bill-of-materials. Essential for validating nested product categories or manager hierarchies.

-- Category tree: find all subcategories under 'Electronics' (category_id = 1)
WITH RECURSIVE category_tree AS (
    -- Anchor: root category
    SELECT category_id, name, parent_category_id, 0 AS depth
    FROM product_categories
    WHERE category_id = 1

    UNION ALL

    -- Recursive: children
    SELECT pc.category_id, pc.name, pc.parent_category_id, ct.depth + 1
    FROM product_categories pc
    JOIN category_tree ct ON pc.parent_category_id = ct.category_id
)
SELECT category_id, name, depth
FROM category_tree
ORDER BY depth, name;
-- Org chart: all reports under manager_id = 100
WITH RECURSIVE org_hierarchy AS (
    SELECT employee_id, name, manager_id, 1 AS level
    FROM employees
    WHERE manager_id = 100

    UNION ALL

    SELECT e.employee_id, e.name, e.manager_id, oh.level + 1
    FROM employees e
    JOIN org_hierarchy oh ON e.manager_id = oh.employee_id
)
SELECT * FROM org_hierarchy ORDER BY level, name;

    SELECT customer_id, email, created_at,
           ROW_NUMBER() OVER (
               PARTITION BY LOWER(TRIM(email))
               ORDER BY created_at DESC

    FROM customers
)
SELECT customer_id, email, created_at
FROM ranked
WHERE rn = 1;

WITH product_revenue AS (

WITH daily_orders AS (

PARTITION BY — Detailed Explanation

PARTITION BY divides rows into independent groups for window function calculation — like GROUP BY but without collapsing rows. Each row retains its identity while getting a computed value within its partition.

Without PARTITION BY:     one window for ALL rows
  ROW_NUMBER() → 1, 2, 3, 4, 5, 6, 7, 8 ...

With PARTITION BY category_id:
  Category A: 1, 2, 3        (resets per category)
  Category B: 1, 2, 3, 4
  Category C: 1, 2

-- Running balance per customer (cumulative spend)
SELECT order_id, customer_id, order_date, total_amount,
       SUM(total_amount) OVER (
           PARTITION BY customer_id
           ORDER BY order_date
           ROWS UNBOUNDED PRECEDING
       ) AS running_balance
FROM orders
WHERE status = 'COMPLETED'
ORDER BY customer_id, order_date;

Interview Questions & Answers (16 Questions)

  1. What is a CTE and why use it over a subquery?
    A named temporary result set via WITH. Improves readability, allows multiple references, supports recursion. Subqueries work for simple cases; CTEs scale better for complex QA validation pipelines.
  2. Can a CTE be referenced multiple times?
    Yes — one major advantage. Example: compare current month revenue CTE against prior month CTE without duplicating logic.
  3. What is the difference between ROW_NUMBER, RANK, and DENSE_RANK?
    All assign ranks; ROW_NUMBER is always unique (needs tiebreaker). RANK skips ranks after ties. DENSE_RANK does not skip. Use ROW_NUMBER for deduplication; RANK/DENSE_RANK for leaderboard reporting.
  4. Write a query to find the 3rd highest order value.
    SELECT total_amount FROM (
    SELECT total_amount, DENSE_RANK() OVER (ORDER BY total_amount DESC) AS dr
    FROM orders
) t WHERE dr = 3;
  5. What does PARTITION BY do?
    Creates independent windows per group value. Resets window calculations for each partition. Like GROUP BY but keeps all rows.
  6. LEAD vs LAG — which looks forward?
    LEAD accesses the next row; LAG accesses the previous row. Both require ORDER BY in the OVER clause.
  7. How do you implement pagination with ROW_NUMBER?
    Wrap query in subquery with ROW_NUMBER, filter WHERE rn BETWEEN start AND end. Page N with size S: start = (N-1)*S+1, end = N*S.
  8. What is a running total query pattern?
    SUM(amount) OVER (ORDER BY date ROWS UNBOUNDED PRECEDING). Add PARTITION BY for per-customer running totals.
  9. How do you remove duplicates keeping the latest record?
    ROW_NUMBER partitioned by business key, ordered by timestamp DESC, filter rn = 1.
  10. What is NTILE used for in testing?
    Split data into equal buckets for load testing (simulate 10% traffic), stratified sampling, or quartile-based test case selection.
  11. Recursive CTE — what are anchor and recursive members?
    Anchor is the base case (non-recursive SELECT). Recursive member references the CTE itself via UNION ALL. Terminates when recursive member returns no rows.
  12. Window function vs GROUP BY aggregate?
    GROUP BY collapses rows into one per group. Window functions compute aggregates while preserving each row — essential for running totals and row-level comparisons.
  13. How to find employees earning more than their department average?
    SELECT employee_id, salary, department_id,
       AVG(salary) OVER (PARTITION BY department_id) AS dept_avg
FROM employees
-- filter in outer query: WHERE salary > dept_avg
  14. What is ROWS vs RANGE in window frame?
    ROWS counts physical rows (6 PRECEDING = 6 prior rows). RANGE uses logical value range (same date values). Use ROWS for moving averages by row count.
  15. How would you validate YoY dashboard data as SDET?
    Run independent queries for each year-month, compare against dashboard API response. Use LAG(12) query as source of truth. Assert growth percentage calculation handles division by zero.
  16. Performance concern with window functions?
    Full table scans without indexes on PARTITION BY / ORDER BY columns. For large tables, filter with WHERE first, then apply window function in subquery.

Interview Questions & Answers (16 Questions)

  1. What is the difference between frontend and database testing?
    Frontend validates presentation and user flows. Database testing validates persisted state, constraints, and data transformations directly — catches bugs UI may hide due to caching or computed views.
  2. How do you test referential integrity?
    Attempt to insert child row with invalid FK (expect failure). Query for orphan child rows via LEFT JOIN. Verify ON DELETE CASCADE behavior if applicable.
  3. What checks do you run after a data migration?
    Row counts, checksums on numeric columns, sample field-by-field comparison, FK constraint validation, application smoke tests, orphan detection queries.
  4. How do you test a stored procedure?
    Execute with known inputs, assert output parameters/result sets, verify side effects on related tables, test invalid inputs for expected error handling.
  5. What is ETL testing?
    Validating data movement from source to target through transformation rules — row counts, data types, business rule application, incremental load correctness, error handling for bad records.
  6. How do you validate CRUD at the database level?
    Pre-query confirms state, perform operation via API/UI, post-query asserts expected changes. Include negative cases (duplicate insert, update non-existent row).
  7. What is data reconciliation?
    Comparing source and target datasets to find missing, extra, or mismatched records — essential after ETL runs and migrations.
  8. How do you test triggers?
    Perform triggering action (INSERT/UPDATE/DELETE), query affected/audit tables, verify trigger logic including edge cases (bulk operations, NULL values).
  9. What tools do you use for database testing?
    DBeaver/pgAdmin for ad-hoc queries; JDBC in automation frameworks; schema compare tools (Liquibase, Flyway) for migration validation; CI pipelines running SQL assertion scripts.
  10. How do you manage test data in automation?
    Create data via API/factory before test, tag with unique identifier, assert, cleanup in teardown. Use transactions with rollback where supported for isolation.
  11. Structural vs functional database testing?
    Structural = schema, constraints, indexes. Functional = CRUD correctness, business rules, procedure/trigger behavior.
  12. How do you test soft delete?
    After DELETE API call, verify row still exists with is_deleted = true and deleted_at populated. Confirm row excluded from normal SELECT queries but present in admin/audit queries.
  13. What is domain integrity?
    Column values conform to defined domain — data type, range, allowed enum values. Test by querying for out-of-domain values.
  14. How do you automate database assertions in CI?
    JDBC/Node pg client runs SQL after API tests; assert row counts and field values; fail build on mismatch. Store expected values in test data files or compute dynamically.
  15. Common data quality issues you check for?
    Duplicates, NULLs in required fields, orphan FKs, amount mismatches between header and line items, future-dated records, inactive users with recent orders.
  16. How do you test without affecting production data?
    Dedicated test environment, test schema isolation, transaction rollback pattern, never run destructive queries against shared/staging without approval.

public Map<String, Object> getOrderById(long orderId) throws SQLException {
    String sql = "SELECT order_id, customer_id, status, total_amount " +
                 "FROM orders WHERE order_id = ?";
    try (Connection conn = DatabaseHelper.getConnection();
         PreparedStatement ps = conn.prepareStatement(sql)) {
        ps.setLong(1, orderId);
        try (ResultSet rs = ps.executeQuery()) {
            if (!rs.next()) return null;
            Map<String, Object> row = new HashMap<>();
            row.put("order_id", rs.getLong("order_id"));
            row.put("customer_id", rs.getLong("customer_id"));
            row.put("status", rs.getString("status"));
            row.put("total_amount", rs.getBigDecimal("total_amount"));
            return row;
        }
    }
}

Section 11: Indexing & Performance

QA engineers don't just verify functional correctness — they identify slow queries that degrade user experience and catch missing indexes that cause production incidents. Understanding indexing and execution plans helps you file actionable performance bugs with evidence.

What Is an Index?

An index is a separate data structure that speeds up row lookup — like a book's index pointing to page numbers instead of reading every page sequentially.

Without Index (Full Table Scan):     With Index (Seek):
  Row 1  ─┐                            Index: customer_id → row location
  Row 2   │ Scan ALL rows              Lookup: customer_id = 1001 → Row 847
  Row 3   │ O(n) time                  Read ONLY Row 847
  ...     │
  Row 1M ─┘                            O(log n) time with B-Tree
QA Analogy: Testing a search feature without an index is like verifying the search box works while the backend reads every product row — functionally correct but unusable at scale. Performance is a test requirement.

Clustered Index

  • Determines the physical order of data rows on disk
  • Only one clustered index per table (usually the primary key)
  • Data rows are stored in the index leaf pages — no separate heap lookup needed
  • Range queries on clustered key are fast (e.g., orders by order_date if clustered on date)
-- SQL Server: view clustered index
SELECT i.name AS index_name, i.type_desc
FROM sys.indexes i
WHERE i.object_id = OBJECT_ID('orders') AND i.type = 1;

-- PostgreSQL: tables are heap-organized by default;
-- CLUSTER command temporarily reorders by index (not maintained automatically)

Non-Clustered Index

  • Separate structure from data rows — contains key columns + pointer to row
  • Multiple non-clustered indexes allowed per table
  • Extra lookup step (key lookup / bookmark lookup) to fetch non-indexed columns
  • Ideal for frequently filtered/joined columns not in clustered key
-- Create non-clustered index on frequently searched column
CREATE INDEX idx_orders_customer_id ON orders (customer_id);
CREATE INDEX idx_orders_status ON orders (status);
CREATE INDEX idx_orders_order_date ON orders (order_date);

Composite Index — Column Order Matters

A composite index on (customer_id, order_date) supports queries filtering by customer_id alone or by both columns. It does not efficiently support queries filtering only by order_date (leftmost prefix rule).

CREATE INDEX idx_orders_customer_date ON orders (customer_id, order_date);

-- USES index efficiently:
SELECT * FROM orders WHERE customer_id = 1001;
SELECT * FROM orders WHERE customer_id = 1001 AND order_date > '2025-01-01';

-- May NOT use index efficiently (order_date alone):
SELECT * FROM orders WHERE order_date > '2025-01-01';

Covering Index

Includes all columns needed by a query in the index itself — avoids key lookup to the table (index-only scan).

-- Query always needs customer_id, status, total_amount
CREATE INDEX idx_orders_covering
ON orders (customer_id)
INCLUDE (status, total_amount);  -- SQL Server syntax

-- PostgreSQL equivalent: include columns in index definition
CREATE INDEX idx_orders_covering_pg

Interview Questions & Answers (12 Questions)

  1. What is a database index?
    A data structure (usually B-Tree) that speeds up row retrieval by key — trades storage and write speed for read speed.
  2. Clustered vs non-clustered index?
    Clustered defines physical row order; one per table. Non-clustered is separate structure with pointer to row; multiple allowed.
  3. Why does composite index column order matter?
    Leftmost prefix rule: index on (A, B) supports WHERE A and WHERE A,B but not efficiently WHERE B alone.
  4. What is a covering index?
    Index containing all columns a query needs — enables index-only scan without table lookup.
  5. When should you NOT add an index?
    Small tables, low-selectivity columns, write-heavy tables with rare reads, or when too many indexes degrade INSERT/UPDATE performance.
  6. What is EXPLAIN used for?
    Shows how the database executes a query — scan types, join methods, estimated cost. Essential for diagnosing slow queries.
  7. Seq Scan vs Index Scan vs Index Seek?
    Seq Scan reads all rows. Index Scan reads all index entries. Index Seek directly locates specific entries — fastest for point lookups.
  8. What is the N+1 query problem?
    Loading a collection then querying related data one row at a time. Fix with JOINs or batch IN queries. Detect via SQL logging during UI tests.
  9. What is a sargable query?
    Search ARGument ABLE — predicate written so index can be used. Avoid functions on indexed columns in WHERE clause.
  10. How does indexing affect INSERT performance?
    Each index must be updated on insert. More indexes = slower writes. Balance read vs write patterns.
  11. How would you test query performance as SDET?
    Define SLA thresholds, run EXPLAIN ANALYZE in CI for critical queries, compare plans across releases, monitor query logs during load tests.
  12. Unique index vs unique constraint?
    Unique constraint is a logical rule; unique index is the physical implementation enforcing it. Both prevent duplicates.

Section 12: Transactions & ACID

Transactions guarantee that multi-step database operations (checkout, payment, inventory update) either complete fully or leave no partial state. Every SDET testing e-commerce order flows, financial transfers, or concurrent user actions must understand ACID properties and isolation levels.

What Is a Transaction?

A transaction is a logical unit of work comprising one or more SQL statements that execute as a single indivisible operation. It begins with BEGIN (or implicit start), ends with COMMIT (persist changes) or ROLLBACK (undo all changes).

BEGIN;
    INSERT INTO orders (customer_id, total_amount, status)
    VALUES (1001, 599.98, 'PENDING');

    INSERT INTO order_items (order_id, product_id, quantity, unit_price)
    VALUES (currval('orders_order_id_seq'), 101, 2, 299.99);

    UPDATE inventory SET stock_quantity = stock_quantity - 2
    WHERE product_id = 101;

COMMIT;  -- all three statements persist together

ACID Properties — Detailed Explanation

A — Atomicity

All statements in a transaction succeed together or none persist. Partial completion is impossible from the caller's perspective.

Banking Transfer Example: Debit $100 from Account A, credit $100 to Account B. If credit fails after debit, atomicity requires both operations roll back — no lost money.

BEGIN;
    UPDATE accounts SET balance = balance - 100 WHERE account_id = 'A';
    UPDATE accounts SET balance = balance + 100 WHERE account_id = 'B';
    -- If any statement fails:
ROLLBACK;  -- both updates undone

C — Consistency

Transaction moves database from one valid state to another, enforcing all constraints (PK, FK, CHECK, UNIQUE). Invalid data never commits.

BEGIN;
    INSERT INTO orders (customer_id, total_amount) VALUES (99999, -50);
    -- CHECK constraint total_amount > 0 fails → entire transaction rolls back
ROLLBACK;

I — Isolation

Concurrent transactions don't interfere with each other's intermediate state. Each transaction appears to run in isolation — though physical execution may be interleaved.

Scenario: User A reads inventory = 5. User B purchases 3 (inventory = 2). User A still sees 5 unless isolation level permits dirty reads.

D — Durability

Once COMMIT succeeds, changes survive system crash, power failure, or restart. Database writes to persistent storage (WAL/redo log) before acknowledging commit.

QA Scenario — Checkout Failure: Simulate payment gateway timeout after order insert. Verify: either order + payment + inventory update all exist, or none exist. Query for orders with status PENDING and no payment record — indicates atomicity violation. 
-- Detect partial checkout state (atomicity bug)
SELECT o.order_id, o.status, p.payment_id
FROM orders o
LEFT JOIN payments p ON o.order_id = p.order_id
WHERE o.status = 'PENDING'

  AND p.payment_id IS NULL
  AND o.total_amount > 0;

    WHERE product_id = 101;

    -- Order and items remain; only inventory update undone

Phenomena Explained

  • Dirty Read: Reading uncommitted data from another transaction that may rollback
  • Non-Repeatable Read: Same query in one transaction returns different rows because another transaction committed an update
  • Phantom Read: Same query returns different number of rows because another transaction inserted/deleted matching rows
-- Set isolation level (PostgreSQL)
BEGIN ISOLATION LEVEL REPEATABLE READ;
    SELECT stock_quantity FROM inventory WHERE product_id = 101;  -- returns 10
    -- Another session updates stock to 7 and commits
    SELECT stock_quantity FROM inventory WHERE product_id = 101;
    -- READ COMMITTED: returns 7 (non-repeatable read)
    -- REPEATABLE READ: returns 10 (snapshot)
COMMIT;

Deadlocks — Detection and Prevention

A deadlock occurs when Transaction A holds lock on Row 1 waiting for Row 2, while Transaction B holds lock on Row 2 waiting for Row 1. Database detects and aborts one transaction.

Transaction A:  LOCK orders(1001)  →  WAIT order_items(5001)
Transaction B:  LOCK order_items(5001)  →  WAIT orders(1001)
                     ↓
              DEADLOCK detected → one transaction ROLLBACK

-- PostgreSQL: view recent deadlocks in log
-- Enable: log_lock_waits = on, deadlock_timeout = 1s

-- SQL Server: system health session extended events
-- MySQL: SHOW ENGINE INNODB STATUS; (look for LATEST DETECTED DEADLOCK)

Prevention Strategies

  • Access tables/rows in consistent order across all transactions
  • Keep transactions short — minimize lock hold time
  • Use appropriate isolation level (don't use SERIALIZABLE unless required)
  • Implement retry logic in application for deadlock victims
  • Use row-level locking hints only when necessary

Interview Questions & Answers (12 Questions)

  1. What is a database transaction?
    A group of SQL operations treated as a single unit — all succeed (COMMIT) or all fail (ROLLBACK).
  2. Explain ACID properties.
    Atomicity (all or nothing), Consistency (valid state to valid state), Isolation (concurrent transactions don't interfere), Durability (committed data survives crashes).
  3. What is the difference between COMMIT and ROLLBACK?
    COMMIT permanently saves changes. ROLLBACK undoes all changes since BEGIN.
  4. What is a SAVEPOINT?
    Marker within a transaction allowing partial rollback to that point without undoing earlier statements in the same transaction.
  5. Explain isolation levels.
    READ UNCOMMITTED (least strict) through SERIALIZABLE (strictest). Higher isolation prevents more concurrency anomalies but reduces performance and increases deadlock risk.
  6. What is a dirty read?
    Reading uncommitted data from another transaction that may still rollback — possible only at READ UNCOMMITTED level.
  7. Non-repeatable read vs phantom read?
    Non-repeatable read: same row returns different values (updated by another txn). Phantom read: same query returns different number of rows (inserts/deletes by another txn).
  8. What is a deadlock?
    Circular wait — two transactions each hold a lock the other needs. DB kills one transaction; application should retry.
  9. How do you test transaction rollback?
    Begin transaction, insert/update data, force failure, rollback, verify data absent or unchanged. Automate with JDBC connection.setAutoCommit(false).
  10. How do you test concurrent inventory updates?
    Parallel API calls purchasing same product with stock = 1. Assert exactly one succeeds, inventory = 0, no negative stock.
  11. Default isolation level in PostgreSQL vs MySQL?
    PostgreSQL: READ COMMITTED. MySQL InnoDB: REPEATABLE READ.
  12. How would you detect a partial transaction bug in production?
    Query for orphaned/inconsistent states: orders without payments, payments without orders, inventory not decremented after completed orders. Monitor counts after deployment.

Section 13: Normalization

Normalization is the process of organizing relational database tables to reduce redundancy and improve data integrity. For QA/SDET engineers, understanding normalization helps you design better test data, write accurate validation queries, and spot schema defects during migration or API testing.

What Is Normalization? Why Is It Needed?

Normalization applies a series of rules (normal forms) to split data across tables so each fact is stored in exactly one place. Without it, the same data appears in multiple rows, causing update anomalies (change in one place, stale data elsewhere), insert anomalies (cannot add data without unrelated data), and delete anomalies (deleting one record removes unrelated data).

QA Insight: When you see the same column (e.g., customer_email) duplicated across orders, shipments, and invoices, expect update bugs — fix an email in one table and UI shows old email elsewhere. Your test plan should include cross-table consistency checks.

BCNF (Boyce-Codd Normal Form)

BCNF is a stricter version of 3NF. Every determinant (column that determines another) must be a candidate key. Example violation: a table with columns (student_id, course_id, instructor) where each instructor teaches only one course — instructor → course_id, but instructor is not a key. Fix by splitting into separate tables. In practice, most OLTP systems target 3NF; BCNF matters in academic design and complex dependency scenarios.

Denormalization — When and Why

Denormalization intentionally introduces redundancy for performance. Common in data warehouses, reporting dashboards, and read-heavy systems where JOIN cost is too high.

  • Data warehouses: Star/snowflake schemas with pre-aggregated fact tables
  • Reporting: Flattened tables for BI tools (Power BI, Tableau)
  • Caching layers: Materialized views, read replicas with denormalized snapshots
  • Event logs: Store user_name alongside user_id to avoid JOIN on billions of rows

Interview Questions & Answers (Normalization)

Q1. What is database normalization?

Answer: A design technique to organize tables, eliminate redundancy, and ensure data dependencies make sense — achieved through normal forms (1NF through BCNF).

Q3. What is a partial dependency?

Q4. What is a transitive dependency?

Answer: Column A → Column B → Column C, where B and C are non-key. Example: emp_id → dept_id → dept_name. dept_name transitively depends on emp_id via dept_id.

Interviewer expects: Link to 3NF and ability to decompose tables.

Q5. Difference between 3NF and BCNF?

Answer: 3NF allows non-key determinants in edge cases; BCNF requires every determinant to be a candidate key. BCNF is stricter.

Interviewer expects: Awareness that most production schemas target 3NF.

Q6. When would you denormalize a database?

Answer: When read performance is critical and JOINs are expensive — reporting, data warehouses, caching. Accept redundancy with sync/ETL validation.

Interviewer expects: Trade-off awareness, not "always normalize."

Q7. How does normalization affect your test strategy?

Answer: More tables = more INSERT/UPDATE/DELETE validation points. Need multi-table test data setup, FK orphan checks, and cross-table consistency queries after API calls.

Interviewer expects: Practical QA perspective, not just theory.

Q8. What anomalies does normalization prevent?

Answer: Update anomaly (inconsistent copies), insert anomaly (cannot insert without unrelated data), delete anomaly (deleting one fact removes another).

Interviewer expects: All three named with a brief example each.

Q9. Is a fully normalized database always better?

Answer: No. Over-normalization causes excessive JOINs and hurts read performance. Balance based on workload — OLTP favors normalization, OLAP favors denormalization.

Interviewer expects: Pragmatic engineering judgment.

Q10. How do you test after a normalization migration?

Answer: Compare logical entity counts pre/post, validate no data loss via checksums, verify FK relationships, run application regression, and compare aggregated totals (sum of orders, revenue).

Interviewer expects: Concrete validation approach beyond "run tests."

Top 100 SQL Interview Questions by Difficulty

Beginner Level (30 Questions)

BQ1. What is SQL and what is it used for?

Answer: SQL (Structured Query Language) is the standard language for managing relational databases — querying, inserting, updating, deleting data, and defining schema.

Interviewer expects: Basic definition; mention RDBMS (MySQL, PostgreSQL, Oracle, SQL Server).

BQ2. Difference between DDL, DML, DCL, and TCL?

Answer: DDL defines structure (CREATE, ALTER, DROP). DML manipulates data (INSERT, UPDATE, DELETE, SELECT). DCL controls access (GRANT, REVOKE). TCL manages transactions (COMMIT, ROLLBACK, SAVEPOINT).

Interviewer expects: Correct categorization with at least 2 commands per type.

BQ3. Write a query to select all columns from the employees table.

Answer: SELECT * FROM employees;

Interviewer expects: Correct syntax; bonus if they mention avoiding * in production.

BQ4. How do you select specific columns?

Answer: SELECT first_name, last_name, salary FROM employees;

Interviewer expects: Column list comma-separated from table.

BQ5. What does the WHERE clause do?

Answer: Filters rows based on a condition before returning results. Example: SELECT * FROM employees WHERE dept_id = 10;

Interviewer expects: Filtering concept with example.

BQ6. Difference between WHERE and HAVING?

Answer: WHERE filters rows before grouping. HAVING filters groups after GROUP BY. Example: SELECT dept_id, AVG(salary) FROM employees GROUP BY dept_id HAVING AVG(salary) > 50000;

Interviewer expects: Pre-aggregation vs post-aggregation filter distinction.

BQ7. How do you sort results in ascending and descending order?

Answer: ORDER BY salary ASC (default, lowest first); ORDER BY salary DESC (highest first). Multi-column: ORDER BY dept_id ASC, salary DESC;

Interviewer expects: ASC/DESC keywords and multi-column sort.

BQ8. What is the DISTINCT keyword?

Answer: Removes duplicate rows from result set. SELECT DISTINCT dept_id FROM employees; returns unique department IDs.

Interviewer expects: Uniqueness concept; not same as UNIQUE constraint.

BQ9. How does LIKE work with wildcards?

Answer: % matches zero or more chars; _ matches exactly one. Examples: WHERE name LIKE 'J%' (starts with J), WHERE name LIKE '_ohn' (4-char names ending in ohn).

Interviewer expects: Both wildcards with examples.

BQ10. What is the difference between COUNT(*), COUNT(column), and COUNT(DISTINCT column)?

Answer: COUNT(*) counts all rows including NULLs. COUNT(column) counts non-NULL values only. COUNT(DISTINCT column) counts unique non-NULL values.

Interviewer expects: NULL handling difference — critical for QA count validation.

BQ11. How do you handle NULL values in WHERE clause?

Answer: Use IS NULL or IS NOT NULL. Never use = NULL (always unknown). Example: WHERE email IS NULL;

Interviewer expects: IS NULL syntax; knows = NULL fails.

BQ12. What does COALESCE do?

Answer: Returns first non-NULL value from arguments. SELECT COALESCE(phone, 'N/A') FROM users; replaces NULL phone with 'N/A'.

Interviewer expects: NULL replacement; mention ISNULL (SQL Server) / NVL (Oracle) equivalents.

BQ13. Write an INSERT statement with specific columns.

Answer: INSERT INTO employees (emp_id, first_name, salary) VALUES (101, 'John', 50000);

Interviewer expects: Column list + VALUES; knows order matters if columns specified.

BQ14. How do you update multiple columns?

Answer: UPDATE employees SET salary = 55000, dept_id = 20 WHERE emp_id = 101;

Interviewer expects: SET with comma-separated columns; always mention WHERE to avoid updating all rows.

BQ15. Difference between DELETE and TRUNCATE?

Answer: DELETE removes specific rows (can use WHERE), is DML, can rollback, fires triggers. TRUNCATE removes all rows, is DDL, faster, resets identity, cannot rollback in some DBs.

Interviewer expects: DML vs DDL, rollback, WHERE clause difference.

BQ16. What is a PRIMARY KEY?

Answer: Column(s) that uniquely identify each row. NOT NULL + UNIQUE. One per table. Automatically indexed.

Interviewer expects: Uniqueness + NOT NULL; one PK per table.

BQ17. What is a FOREIGN KEY?

Answer: Column that references PRIMARY KEY of another table. Enforces referential integrity — prevents orphan records.

Interviewer expects: Parent-child relationship; orphan prevention.

BQ18. What is the IN operator?

Answer: Shorthand for multiple OR conditions. WHERE dept_id IN (10, 20, 30); equivalent to dept_id = 10 OR dept_id = 20 OR dept_id = 30.

Interviewer expects: Syntax and equivalence to OR.

BQ19. What is BETWEEN used for?

Answer: Range filter inclusive of boundaries. WHERE salary BETWEEN 40000 AND 60000; same as salary >= 40000 AND salary <= 60000.

Interviewer expects: Inclusive range; date ranges too.

BQ20. How do you limit rows returned (TOP/LIMIT)?

Answer: SQL Server: SELECT TOP 10 * FROM employees ORDER BY salary DESC; MySQL/PostgreSQL: SELECT * FROM employees ORDER BY salary DESC LIMIT 10;

Interviewer expects: Dialect awareness; ORDER BY with LIMIT for meaningful top-N.

BQ21. What are aggregate functions?

Answer: Functions that compute a single value from multiple rows: COUNT, SUM, AVG, MIN, MAX.

Interviewer expects: Name all five with brief purpose.

BQ22. What is GROUP BY?

Answer: Groups rows with same values in specified columns; used with aggregates. SELECT dept_id, COUNT(*) FROM employees GROUP BY dept_id;

Interviewer expects: Every non-aggregated SELECT column must be in GROUP BY.

BQ23. Can you use column alias in WHERE clause?

Answer: No. WHERE is evaluated before SELECT, so aliases aren't available. Use HAVING for aggregate aliases, or repeat the expression.

Interviewer expects: SQL execution order awareness.

BQ24. What is an alias and why use it?

Answer: Temporary name for column or table. SELECT first_name AS fname FROM employees e; Improves readability and shortens queries.

Interviewer expects: AS keyword (optional) for columns and tables.

BQ25. Difference between CHAR and VARCHAR?

Answer: CHAR is fixed-length (padded with spaces). VARCHAR is variable-length (stores only actual chars + length). Use CHAR for fixed codes (state abbreviations); VARCHAR for names, emails.

BQ26. What is a VIEW?

Advanced Level (35 Questions)

AQ1. Explain ROW_NUMBER, RANK, and DENSE_RANK.

Answer: ROW_NUMBER: unique sequential number per partition. RANK: same rank for ties, skips next (1,2,2,4). DENSE_RANK: same rank for ties, no skip (1,2,2,3). ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC)

Interviewer expects: Tie behavior difference; PARTITION BY + ORDER BY.

AQ2. What are window functions?

Answer: Perform calculations across related rows without collapsing like GROUP BY. Use OVER clause. Examples: ROW_NUMBER, SUM() OVER, LAG, LEAD.

Interviewer expects: Difference from aggregates — rows preserved.

AQ3. Explain LAG and LEAD with use case.

Answer: LAG accesses previous row; LEAD accesses next row. LAG(salary, 1) OVER (ORDER BY hire_date) compares current salary to previous employee's. Use for period-over-period analysis.

Interviewer expects: Offset parameter; QA use for audit trail comparison.

AQ4. What is a recursive CTE?

Answer: CTE that references itself. Anchor member + recursive member united by UNION ALL. Use for org hierarchies, bill of materials. Must have termination condition.

Interviewer expects: Anchor/recursive parts; avoid infinite loops.

AQ5. Explain transaction ACID properties.

Answer: Atomicity (all or nothing), Consistency (valid state to valid state), Isolation (concurrent transactions don't interfere), Durability (committed data survives crash).

Interviewer expects: All four defined; link to COMMIT/ROLLBACK.

AQ6. What are isolation levels?

Answer: READ UNCOMMITTED, READ COMMITTED (default most DBs), REPEATABLE READ, SERIALIZABLE. Higher isolation = less concurrency, fewer anomalies (dirty read, non-repeatable read, phantom read).

Interviewer expects: Name levels; dirty read vs phantom read.

AQ7. What is a deadlock and how do you detect it?

Answer: Two transactions waiting for each other's locks. DB kills one victim. Detect via error message or monitoring. Prevent by consistent lock ordering.

Interviewer expects: Practical awareness; QA may hit deadlocks in parallel tests.

AQ8. Write a query using SUM() OVER for running total.

Answer: SELECT order_date, amount, SUM(amount) OVER (ORDER BY order_date) AS running_total FROM orders;

Interviewer expects: OVER ORDER BY for cumulative sum.

AQ9. How do you optimize a slow query?

AQ10. What is EXPLAIN PLAN used for?

Interviewer expects: Has used EXPLAIN in real debugging.

AQ11. Difference between WHERE and ON in JOIN?

Answer: ON defines join condition (before outer join fill). WHERE filters after join. For LEFT JOIN, condition in ON preserves unmatched left rows; in WHERE converts to INNER behavior.

Interviewer expects: LEFT JOIN filter placement — common trick question.

AQ12. What is a materialized view?

AQ13. Explain MERGE/UPSERT statement.

Top 10 JOIN Questions (Quick Revision)

  1. INNER JOIN — only matching rows from both tables.
  2. LEFT JOIN — all left rows + matches; use for orphan detection.
  3. RIGHT JOIN — mirror of LEFT; rarely used (swap tables instead).
  4. FULL OUTER JOIN — all rows both sides; NULL for gaps.
  5. Self join — same table twice with aliases (employee-manager).
  6. Cross join — Cartesian product; M × N rows.
  7. ON vs WHERE in LEFT JOIN — filter in ON preserves unmatched left rows.
  8. Multiple JOINs — chain: users → orders → order_items → products.
  9. JOIN vs subquery — JOIN usually faster; subquery more readable.
  10. Anti-join — LEFT JOIN IS NULL or NOT EXISTS for "rows without match."

Top 10 Window Function Questions

  1. ROW_NUMBER — unique sequential number; use for dedup and top-N per group.
  2. RANK — ties get same rank, gaps after (1,2,2,4).
  3. DENSE_RANK — ties same rank, no gaps (1,2,2,3).
  4. LEAD/LAG — access next/previous row for period comparison.
  5. SUM() OVER — running total without GROUP BY collapse.
  6. PARTITION BY — resets window per group (per dept, per customer).
  7. Window vs GROUP BY — window keeps all rows; GROUP BY collapses.
  8. NTILE — divide into N buckets for sampling/quartiles.
  9. FIRST_VALUE — first value in window frame.
  10. Top-N per group pattern — ROW_NUMBER + WHERE rn <= N in subquery/CTE.

Top 50 Quick Revision Points

  1. SQL = Structured Query Language for RDBMS.
  2. DDL: CREATE, ALTER, DROP, TRUNCATE.
  3. DML: INSERT, UPDATE, DELETE, SELECT.
  4. TCL: COMMIT, ROLLBACK, SAVEPOINT.
  5. PRIMARY KEY = unique + NOT NULL; one per table.
  6. FOREIGN KEY enforces referential integrity.
  7. UNIQUE allows one NULL (most DBs); multiple per table.
  8. CHECK validates column values against condition.
  9. DEFAULT auto-fills omitted columns on INSERT.
  10. Never use = NULL; use IS NULL.
  11. COUNT(*) counts all rows including NULLs.
  12. WHERE filters before grouping; HAVING after.
  13. Every non-aggregated SELECT column must be in GROUP BY.
  14. INNER JOIN returns only matching rows.
  15. LEFT JOIN preserves all left table rows.
  16. Orphan check = LEFT JOIN + WHERE parent.key IS NULL.
  17. Self join requires table aliases.
  18. Subquery in WHERE must return scalar for comparison operators.
  19. Correlated subquery runs once per outer row.
  20. EXISTS returns TRUE if subquery has any row.
  21. UNION removes duplicates; UNION ALL does not.
  22. CTE: WITH name AS (SELECT ...) SELECT ... FROM name.
  23. ROW_NUMBER always unique; RANK/DENSE_RANK handle ties differently.
  24. Window functions use OVER (PARTITION BY ... ORDER BY ...).
  25. DELETE can use WHERE; TRUNCATE removes all rows.
  26. TRUNCATE is DDL; may not rollback; resets identity.
  27. Index speeds reads, slows writes; don't over-index.
  28. EXPLAIN shows query execution plan.
  29. ACID: Atomicity, Consistency, Isolation, Durability.
  30. PreparedStatement prevents SQL injection.
  31. Normalization reduces redundancy (1NF→3NF).
  32. Denormalization improves read performance in DWH.
  33. 1NF: atomic values, no repeating groups.
  34. 2NF: no partial dependency on composite key.
  35. 3NF: no transitive dependency.
  36. Soft delete: is_deleted flag; row still in DB.
  37. Hard delete: row physically removed.
  38. API POST → verify INSERT in DB.
  39. API PUT → verify UPDATE in DB.
  40. API DELETE → verify DELETE or soft delete flag.
  41. ETL: validate source count = target count.
  42. Migration: compare checksums pre and post.
  43. DECIMAL for money; never FLOAT.
  44. utf8mb4 for emoji and international chars.
  45. Test isolation: unique data per test run.
  46. Async: poll DB with retry, don't assert immediately.
  47. Read replica may lag behind primary.
  48. ON DELETE CASCADE: deleting parent deletes children.
  49. Chopstick testing: Create → Read → Update → Delete.
  50. Always SELECT before UPDATE/DELETE in manual testing.

Common Mistakes to Avoid in SQL Interviews

  • Using = NULL instead of IS NULL — always wrong.
  • Forgetting WHERE in UPDATE/DELETE — updates/deletes all rows.
  • Using WHERE instead of HAVING for aggregate filters.
  • Not knowing LEFT JOIN for orphan detection — most common QA pattern.
  • Confusing RANK with ROW_NUMBER — ties behave differently.
  • Putting filter on LEFT JOIN in WHERE — accidentally converts to INNER JOIN.
  • Using SELECT * in production/testing — specify columns explicitly.
  • Comparing FLOAT/DECIMAL without tolerance — rounding failures.
  • Ignoring NULL in COUNT/GROUP BY — wrong counts.
  • Not mentioning PreparedStatement when discussing SQL in automation.
  • Saying "DELETE removes table" — that's DROP TABLE.
  • Unable to write second highest salary — prepare 2-3 approaches.
  • No QA context — always tie SQL to validation, not just syntax.
  • Ignoring transaction/testing rollback — show atomicity awareness.
  • Not asking which database dialect — TOP vs LIMIT vs ROWNUM matters.

Section 19: Stored Procedures, Triggers, Views & Functions

Database objects beyond tables and indexes—views, stored procedures, functions, triggers, cursors, and temporary tables—form the programmable layer that QA engineers must understand to validate business logic at the data tier. This section covers creation syntax, testing strategies, and real-world automation patterns.

19.1 Views

CREATE VIEW Syntax

A view is a virtual table defined by a stored query. It does not store data itself (unless materialized); it presents a filtered or joined projection of underlying tables.

-- Simple view: active customers only
CREATE VIEW vw_active_customers AS
SELECT customer_id, first_name, last_name, email, created_at
FROM customers
WHERE status = 'ACTIVE'
  AND deleted_at IS NULL;

-- View with joins and aggregations
CREATE VIEW vw_order_summary AS
SELECT
    o.order_id,
    o.customer_id,
    c.email,
    COUNT(oi.line_id)   AS line_count,
    SUM(oi.quantity * oi.unit_price) AS order_total,
    o.order_status,
    o.created_at
FROM orders o
JOIN customers c ON c.customer_id = o.customer_id
JOIN order_items oi ON oi.order_id = o.order_id
GROUP BY o.order_id, o.customer_id, c.email, o.order_status, o.created_at;

-- Query the view like a table
SELECT * FROM vw_order_summary WHERE order_total > 500;

Updatable Views

Some views support INSERT, UPDATE, and DELETE when they map cleanly to a single base table without aggregations, DISTINCT, GROUP BY, or subqueries in the SELECT list.

CREATE VIEW vw_customer_contact AS
SELECT customer_id, first_name, last_name, email, phone
FROM customers
WHERE status = 'ACTIVE';

-- Valid update through view (maps to customers table)
UPDATE vw_customer_contact
SET phone = '+1-555-0199'
WHERE customer_id = 10042;

-- Non-updatable: aggregated view
CREATE VIEW vw_monthly_revenue AS
SELECT DATE_TRUNC('month', order_date) AS month, SUM(total) AS revenue
FROM orders GROUP BY 1;
-- UPDATE vw_monthly_revenue SET revenue = 0;  -- ERROR in most engines
QA Note: Always verify whether a view is updatable before writing tests that mutate data through it. PostgreSQL exposes information_schema.views.is_updatable; SQL Server uses sp_helptext and catalog views.

Materialized Views

Materialized views persist query results physically and must be refreshed. Common in PostgreSQL and Oracle; SQL Server uses indexed views with similar semantics.

-- PostgreSQL
CREATE MATERIALIZED VIEW mv_daily_sales AS
SELECT order_date, product_id, SUM(quantity) AS units_sold, SUM(line_total) AS revenue
FROM order_items oi
JOIN orders o ON o.order_id = oi.order_id
GROUP BY order_date, product_id;

CREATE UNIQUE INDEX idx_mv_daily_sales ON mv_daily_sales (order_date, product_id);

-- Refresh after ETL load
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_daily_sales;

QA Use Case: Validation Views

QA teams create validation views that flag data anomalies. These become living test oracles runnable in CI pipelines.

CREATE VIEW qa_v_data_integrity AS
SELECT 'orphan_order_items' AS issue, oi.line_id AS record_id
FROM order_items oi
LEFT JOIN orders o ON o.order_id = oi.order_id
WHERE o.order_id IS NULL
UNION ALL
SELECT 'negative_inventory', product_id
FROM inventory WHERE quantity_on_hand < 0
UNION ALL
SELECT 'order_total_mismatch', o.order_id
FROM orders o
JOIN (
    SELECT order_id, SUM(quantity * unit_price) AS calc_total
    FROM order_items GROUP BY order_id
) x ON x.order_id = o.order_id
WHERE ABS(o.total_amount - x.calc_total) > 0.01;

-- CI assertion: expect zero rows
SELECT COUNT(*) AS integrity_violations FROM qa_v_data_integrity;

19.2 Stored Procedures

CREATE PROCEDURE Syntax

-- SQL Server / MySQL style
DELIMITER //
CREATE PROCEDURE sp_process_order(
    IN  p_order_id      INT,
    IN  p_payment_method VARCHAR(20),
    OUT p_result_code   INT,
    OUT p_result_msg    VARCHAR(255)
)
BEGIN
    DECLARE v_customer_id INT;
    DECLARE v_total       DECIMAL(12,2);
    DECLARE v_stock_ok    BOOLEAN DEFAULT TRUE;

    DECLARE EXIT HANDLER FOR SQLEXCEPTION
    BEGIN
        ROLLBACK;
        SET p_result_code = -1;
        SET p_result_msg = 'Unexpected error during order processing';
    END;

    START TRANSACTION;

    SELECT customer_id, total_amount INTO v_customer_id, v_total
    FROM orders WHERE order_id = p_order_id FOR UPDATE;

    IF v_customer_id IS NULL THEN
        SET p_result_code = 404;
        SET p_result_msg = CONCAT('Order not found: ', p_order_id);
        ROLLBACK;
    ELSE
        -- Check inventory for all line items
        SELECT COUNT(*) = 0 INTO v_stock_ok
        FROM order_items oi
        JOIN inventory inv ON inv.product_id = oi.product_id
        WHERE oi.order_id = p_order_id
          AND inv.quantity_on_hand < oi.quantity;

        IF NOT v_stock_ok THEN
            SET p_result_code = 409;
            SET p_result_msg = 'Insufficient stock';
            ROLLBACK;
        ELSE
            UPDATE inventory inv
            JOIN order_items oi ON oi.product_id = inv.product_id
            SET inv.quantity_on_hand = inv.quantity_on_hand - oi.quantity
            WHERE oi.order_id = p_order_id;

            UPDATE orders
            SET order_status = 'CONFIRMED',
                payment_method = p_payment_method,
                confirmed_at = NOW()
            WHERE order_id = p_order_id;

            INSERT INTO order_audit (order_id, action, actor, created_at)
            VALUES (p_order_id, 'CONFIRMED', CURRENT_USER(), NOW());

            COMMIT;
            SET p_result_code = 0;
            SET p_result_msg = 'Order processed successfully';
        END IF;
    END IF;
END //
DELIMITER ;

Control Flow: IF/ELSE and LOOP

CREATE PROCEDURE sp_bulk_discount(IN p_min_order_value DECIMAL(10,2))
BEGIN
    DECLARE done INT DEFAULT FALSE;
    DECLARE v_order_id INT;
    DECLARE cur CURSOR FOR
        SELECT order_id FROM orders
        WHERE order_status = 'PENDING' AND total_amount >= p_min_order_value;
    DECLARE CONTINUE HANDLER FOR NOT FOUND SET done = TRUE;

    OPEN cur;
    read_loop: LOOP
        FETCH cur INTO v_order_id;
        IF done THEN
            LEAVE read_loop;
        END IF;
        UPDATE orders SET discount_pct = 10 WHERE order_id = v_order_id;
    END LOOP;
    CLOSE cur;
END;

             CallableStatement cs = conn.prepareCall("{CALL sp_process_order(?, ?, ?, ?)}")) {

Custom Validation Functions

CREATE FUNCTION fn_validate_order_total(p_order_id INT)
RETURNS VARCHAR(20)
BEGIN
    DECLARE v_header_total DECIMAL(12,2);
    DECLARE v_line_total   DECIMAL(12,2);

    SELECT total_amount INTO v_header_total FROM orders WHERE order_id = p_order_id;
    SELECT SUM(quantity * unit_price) INTO v_line_total
    FROM order_items WHERE order_id = p_order_id;

    IF v_header_total IS NULL THEN RETURN 'NOT_FOUND';
    ELSEIF ABS(v_header_total - IFNULL(v_line_total, 0)) > 0.01 THEN RETURN 'MISMATCH';
    ELSE RETURN 'VALID';
    END IF;
END;

-- Use in QA query
SELECT order_id, fn_validate_order_total(order_id) AS validation_status
FROM orders WHERE created_at >= CURDATE();

Audit Trail Example

CREATE TABLE customers_audit (
    audit_id      BIGINT AUTO_INCREMENT PRIMARY KEY,
    customer_id   INT NOT NULL,
    action_type   ENUM('INSERT','UPDATE','DELETE') NOT NULL,
    old_email     VARCHAR(255),
    new_email     VARCHAR(255),
    changed_by    VARCHAR(100),
    changed_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

DELIMITER //
CREATE TRIGGER trg_customers_audit_update
AFTER UPDATE ON customers
FOR EACH ROW
BEGIN
    IF OLD.email <> NEW.email OR OLD.status <> NEW.status THEN
        INSERT INTO customers_audit (customer_id, action_type, old_email, new_email, changed_by)
        VALUES (OLD.customer_id, 'UPDATE', OLD.email, NEW.email, CURRENT_USER());
    END IF;
END //
DELIMITER ;

How to Test Triggers

  1. Baseline count: Record row count in audit table before action.
  2. Perform DML: INSERT/UPDATE/DELETE the target row with known values.
  3. Assert audit row: Verify exactly one audit record with correct old/new values.
  4. Negative case: Update non-tracked columns; assert no audit row created.
  5. Rollback isolation: Use transactions in test fixtures to avoid polluting shared DB.
  6. Disable check: Confirm trigger exists via SHOW TRIGGERS or catalog queries before blaming application code.
START TRANSACTION;
SELECT COUNT(*) INTO @before FROM customers_audit WHERE customer_id = 10042;
UPDATE customers SET email = 'new@test.com' WHERE customer_id = 10042;
SELECT COUNT(*) INTO @after FROM customers_audit WHERE customer_id = 10042;
SELECT @after - @before AS new_audit_rows;  -- expect 1
SELECT action_type, old_email, new_email FROM customers_audit
WHERE customer_id = 10042 ORDER BY audit_id DESC LIMIT 1;
ROLLBACK;

19.5 Cursors

When to Use Cursors

  • Row-by-row processing when set-based SQL cannot express the logic (rare in modern SQL).
  • Calling external procedures or complex conditional logic per row in legacy systems.
  • Generating sequential identifiers or custom reports in stored procedures without window functions.
  • Migrating data with row-level validation and error logging.

Performance Implications

Warning: Cursors are iteratively slow compared to set-based operations. A cursor processing 100,000 rows may take minutes where a single UPDATE/JOIN completes in seconds. Always attempt set-based rewrite first; use cursors only when necessary.

19.8 Interview Questions & Answers

Q1. What is the difference between a view and a table?

A: A table stores physical rows on disk; a view is a saved query that presents data from underlying tables at query time. Views provide abstraction without duplicating storage (unless materialized).

Q2. When is a view updatable?

A: When it maps to a single base table, has no DISTINCT/GROUP BY/aggregates in the SELECT list, and all NOT NULL columns without defaults are included. Engine-specific rules may further restrict updatability.

Q3. What is a materialized view and when would QA care?

A: A materialized view stores precomputed results. QA must verify refresh jobs run on schedule and that stale data windows are acceptable for reporting tests.

Q4. IN vs OUT vs INOUT parameters?

A: IN passes values into the procedure; OUT returns values to the caller; INOUT serves both directions—useful for counters or flags modified during execution.

Q5. How do you test a stored procedure that modifies multiple tables?

A: Seed known fixtures, call the SP within a transaction, assert all affected tables, verify OUT parameters, then ROLLBACK. Include negative and concurrency scenarios.

Q6. Scalar function vs stored procedure?

A: Functions return a value and are callable from SELECT; procedures perform actions (DML, multi-result) and are invoked via CALL/EXEC. Functions should avoid side effects in most engines.

Q7. What is a table-valued function?

A: A function returning a result set usable in FROM clauses—essentially a parameterized view with procedural logic optional.

Q8. BEFORE vs AFTER trigger?

A: BEFORE fires prior to row persistence and can modify incoming data; AFTER fires post-write and is ideal for audit logging and cascading effects.

Q9. How do triggers affect QA test design?

A: Hidden side effects (audit rows, derived updates) must be accounted for in assertions. Tests should verify both application behavior and trigger-generated data.

Q10. Why avoid cursors in production SQL?

A: Row-by-row processing is orders of magnitude slower than set-based operations, holds locks longer, and increases blocking risk.

Q11. #temp vs ##temp in SQL Server?

A: #temp is session-scoped; ##temp is global across sessions until the last session referencing it disconnects.

Q12. CTE vs subquery?

A: CTEs improve readability and allow recursion; subqueries inline. Performance is often identical—optimizer may treat them similarly.

Q13. Can you INSERT into a view?

A: Only if the view is updatable per engine rules. Inserting into a joined view typically fails unless INSTEAD OF triggers handle it.

Q14. How call a stored procedure from JDBC?

A: Use CallableStatement with {CALL proc(?, ?, ?)}, set IN params, registerOutParameter for OUT params, then execute().

Q15. What is an INSTEAD OF trigger?

A: Fires in place of INSERT/UPDATE/DELETE on a view, allowing DML on otherwise non-updatable views by redirecting to base tables.

Q16. How validate data integrity using views in CI?

A: Create qa_v_* views that UNION anomaly queries; pipeline fails if SELECT COUNT(*) > 0.

Q17. DETERMINISTIC vs NOT DETERMINISTIC functions?

A: Deterministic functions always return the same output for the same inputs (e.g., SHA256). Non-deterministic may vary (NOW(), RAND()). Affects indexing and replication.

Q18. What happens if a trigger raises an error?

A: The entire triggering statement rolls back (in most engines), preventing partial updates—critical for QA negative testing.

Q19. How do you find all triggers on a table?

A: SQL Server: SELECT * FROM sys.triggers joined to parent_id. MySQL: SHOW TRIGGERS LIKE 'customers'. PostgreSQL: pg_trigger catalog.

Q20. Explain recursive CTE with QA example.

A: Recursive CTEs traverse hierarchies (categories, org trees). QA validates depth limits, cycle detection, and correct parent-child counts at each level.

Section 19 Summary: Views simplify validation; stored procedures encapsulate business logic; functions compute reusable values; triggers enforce rules and audit changes. QA engineers should test these objects directly—not only through the application layer—to catch data-tier regressions early.

20.10 Microservices Database Patterns

Database per Service

Each microservice owns its database schema. No direct cross-service SQL joins. QA validates via API contracts and eventual consistency—not shared table assertions.

Event Sourcing

State changes stored as immutable events. Current state is derived by replaying events. QA verifies event ordering, idempotency, and snapshot accuracy.

CQRS (Command Query Responsibility Segregation)

Write model (commands) separated from read model (queries/projections). QA tests write path and read path independently, then validates projection lag bounds.

20.12 GraphQL + Database

How GraphQL Queries Translate to SQL

GraphQL resolvers map fields to database queries. A query requesting user { orders { items { product } } } may generate nested SELECTs or JOINs. DataLoader patterns batch N+1 queries into single IN-clause lookups.

# GraphQL query
query {
  customer(id: "501") {
    name
    orders(status: PENDING) {
      orderId
      total
      items { sku quantity }
    }
  }
}

# Equivalent SQL (simplified)
SELECT c.name, o.order_id, o.total, oi.sku, oi.quantity
FROM customers c
JOIN orders o ON o.customer_id = c.customer_id AND o.status = 'PENDING'
JOIN order_items oi ON oi.order_id = o.order_id
WHERE c.customer_id = 501;

The N+1 Problem

Naive resolvers fetch orders for each customer in a list query—1 query for customers plus N queries for orders. QA detects this via query logging: assert query count does not scale linearly with list size.

20.13 Interview Questions & Answers

Q1. What are the four main types of NoSQL databases?

A: Document, Key-Value, Column-Family, and Graph—each optimized for different access patterns and data shapes.

Q2. When choose MongoDB over PostgreSQL?

A: When schema flexibility, nested documents, or horizontal sharding matter more than complex multi-table ACID joins. Many teams use both.

Q3. How do you validate MongoDB aggregation pipelines?

A: Seed known documents, run pipeline, compare output to manually computed expected results. Test edge cases: empty arrays, missing fields, null coalescing.

Q4. What is Redis used for in modern apps?

A: Caching, session storage, rate limiting, pub/sub, leaderboards—ephemeral or fast-access data where milliseconds matter.

Q5. How test cache invalidation?

A: Update source data via API, assert Redis key deleted or updated, verify next read reflects new data and repopulates cache.

Q6. Difference between TTL and EXPIRE in Redis?

A: TTL reads remaining seconds; EXPIRE sets expiration on an existing key. SET with EX combines set + TTL atomically.

Q7. Why avoid KEYS * in production Redis?

A: KEYS blocks the server scanning all keys—O(N). Use SCAN for iterative non-blocking iteration in tests and prod.

Q8. What is database-per-service?

A: Microservices pattern where each service owns its datastore; cross-service data accessed only via APIs or events, not shared SQL.

Q9. Explain event sourcing for QA.

A: Testers verify every state change produces an event, events are immutable, replay produces correct current state, and idempotent consumers handle duplicates.

Q10. What is CQRS?

A: Separates command (write) and query (read) models. QA validates writes persist correctly and read projections update within acceptable lag.

Q11. Flyway vs Liquibase for QA pipelines?

A: Both run migrations in CI before tests. Flyway uses versioned SQL files; Liquibase uses changelogs with richer rollback. Choose based on team preference and rollback needs.

Q12. How validate AWS RDS read replica lag?

A: Write to primary, query replica, measure delay until record appears. Assert lag stays below SLA (e.g., < 1 second).

Q13. What is the N+1 problem in GraphQL?

A: Resolver executes one query per parent row (N children queries + 1 parent). Detect via query count logging; fix with DataLoader batching or JOIN queries.

Q14. How mask email addresses for QA?

A: Replace with deterministic fake emails (user{id}@qa-test.local) preserving uniqueness and format for login tests without real PII.

Q15. MongoDB $lookup vs SQL JOIN?

A: $lookup performs a left outer join in aggregation pipeline—similar semantics but syntax differs; QA validates unmatched documents produce empty arrays.

Q16. What consistency levels exist in Cosmos DB?

A: Strong, Bounded Staleness, Session, Consistent Prefix, Eventual. QA tests read-your-writes under Session, global consistency under Strong.

Q17. How test Redis TTL expiration?

A: SET with short EX, poll TTL until -2 (key gone), or use mocked time in unit tests. Integration tests may use sleep with timeout guards.

Q18. CAP theorem—what does it mean for QA?

A: Distributed systems trade Consistency, Availability, Partition tolerance. QA validates behavior during network partitions matches chosen guarantees (e.g., Cassandra tunable consistency).

Q19. What is negative caching?

A: Caching "not found" responses to prevent repeated DB hits for invalid keys. QA ensures TTL prevents indefinite false negatives after data is created.

Q20. How automate MongoDB tests in CI?

A: Spin up MongoDB via Docker service container, run pymongo/Java tests, tear down. Use unique test prefixes to avoid collisions in shared instances.

Q21. GraphQL vs REST for database testing?

A: GraphQL returns exactly requested fields—test field selection does not over-fetch. REST endpoints may map 1:1 to tables; GraphQL may trigger multiple resolver queries.

Q22. What is tombstone in Cassandra?

A: Deletion marker persisting until compaction. QA may see "deleted" rows briefly during consistency window—understand eventual deletion semantics.

Section 20 Summary: Modern QA spans relational, document, cache, and cloud-native stores. Validate not only data correctness but consistency models, cache behavior, migration integrity, PII masking, and API-to-database query patterns—including GraphQL N+1 risks.
