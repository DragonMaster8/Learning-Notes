Section 13: Normalization

Normalization is the process of organizing relational database tables to reduce redundancy and improve data integrity. For QA/SDET engineers, understanding normalization helps you design better test data, write accurate validation queries, and spot schema defects during migration or API testing.

What Is Normalization? Why Is It Needed?

Normalization applies a series of rules (normal forms) to split data across tables so each fact is stored in exactly one place. Without it, the same data appears in multiple rows, causing update anomalies (change in one place, stale data elsewhere), insert anomalies (cannot add data without unrelated data), and delete anomalies (deleting one record removes unrelated data).

QA Insight: When you see the same column (e.g., customer_email) duplicated across orders, shipments, and invoices, expect update bugs — fix an email in one table and UI shows old email elsewhere. Your test plan should include cross-table consistency checks.

BCNF (Boyce-Codd Normal Form)

BCNF is a stricter version of 3NF. Every determinant (column that determines another) must be a candidate key. Example violation: a table with columns (student_id, course_id, instructor) where each instructor teaches only one course — instructor → course_id, but instructor is not a key. Fix by splitting into separate tables. In practice, most OLTP systems target 3NF; BCNF matters in academic design and complex dependency scenarios.

Denormalization — When and Why

Denormalization intentionally introduces redundancy for performance. Common in data warehouses, reporting dashboards, and read-heavy systems where JOIN cost is too high.

  • Data warehouses: Star/snowflake schemas with pre-aggregated fact tables
  • Reporting: Flattened tables for BI tools (Power BI, Tableau)
  • Caching layers: Materialized views, read replicas with denormalized snapshots
  • Event logs: Store user_name alongside user_id to avoid JOIN on billions of rows

QA Perspective — How Normalization Affects Testing

  • Test data setup: Normalized schemas require INSERT order (parent before child). Test fixtures must respect FK dependencies.
  • API validation: A single POST may INSERT into 3–4 tables — validate all, not just the primary response table.
  • UI vs DB: UI may show denormalized/cached data; DB is normalized — compare via JOIN queries, not single-table SELECT.
  • Migration testing: Normalization changes during migration mean row counts per table change (1 table → 3 tables). Compare logical record counts, not raw table counts.
  • Regression: Update to products.product_name should reflect everywhere if denormalized copies exist — test propagation.

Interview Questions & Answers (Normalization)

Q1. What is database normalization?

Answer: A design technique to organize tables, eliminate redundancy, and ensure data dependencies make sense — achieved through normal forms (1NF through BCNF).

Interviewer expects: Clear definition plus mention of anomalies (insert, update, delete).

Q2. Explain 1NF with an example.

Answer: 1NF requires atomic values and no repeating groups. Instead of columns phone1/phone2/phone3, create a separate phone table with one phone per row.

Interviewer expects: Before/after table example and understanding of atomicity.

Q3. What is a partial dependency?

Answer: When a non-key column depends on part of a composite primary key, not the whole key. Example: in (order_id, product_id, product_name), product_name depends only on product_id.

Interviewer expects: Composite key context and 2NF connection.

Q4. What is a transitive dependency?

Answer: Column A → Column B → Column C, where B and C are non-key. Example: emp_id → dept_id → dept_name. dept_name transitively depends on emp_id via dept_id.

Interviewer expects: Link to 3NF and ability to decompose tables.

Q5. Difference between 3NF and BCNF?

Answer: 3NF allows non-key determinants in edge cases; BCNF requires every determinant to be a candidate key. BCNF is stricter.

Interviewer expects: Awareness that most production schemas target 3NF.

Q6. When would you denormalize a database?

Answer: When read performance is critical and JOINs are expensive — reporting, data warehouses, caching. Accept redundancy with sync/ETL validation.

Interviewer expects: Trade-off awareness, not "always normalize."

Q7. How does normalization affect your test strategy?

Answer: More tables = more INSERT/UPDATE/DELETE validation points. Need multi-table test data setup, FK orphan checks, and cross-table consistency queries after API calls.

Interviewer expects: Practical QA perspective, not just theory.

Q8. What anomalies does normalization prevent?

Answer: Update anomaly (inconsistent copies), insert anomaly (cannot insert without unrelated data), delete anomaly (deleting one fact removes another).

Interviewer expects: All three named with a brief example each.

Q9. Is a fully normalized database always better?

Answer: No. Over-normalization causes excessive JOINs and hurts read performance. Balance based on workload — OLTP favors normalization, OLAP favors denormalization.

Interviewer expects: Pragmatic engineering judgment.

Q10. How do you test after a normalization migration?

Answer: Compare logical entity counts pre/post, validate no data loss via checksums, verify FK relationships, run application regression, and compare aggregated totals (sum of orders, revenue).

Interviewer expects: Concrete validation approach beyond "run tests."

Scenario 3: Payment Processing Validation

Requirement

  • Payment status flow: PENDING → PROCESSING → COMPLETED or FAILED
  • Payment amount must equal order total_amount
  • On COMPLETED: order_status → PAID
  • On FAILED: order_status remains PENDING; user can retry
  • Refunds: COMPLETED → REFUNDED; partial refunds supported

Database Schema

CREATE TABLE payments (
    payment_id     BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_id       BIGINT NOT NULL,
    amount         DECIMAL(12,2) NOT NULL,
    currency       CHAR(3) DEFAULT 'USD',
    payment_status VARCHAR(20) DEFAULT 'PENDING',
    gateway_ref    VARCHAR(100),
    failure_reason VARCHAR(500),
    created_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (order_id) REFERENCES orders(order_id)
);

CREATE TABLE refunds (
    refund_id   BIGINT PRIMARY KEY AUTO_INCREMENT,
    payment_id  BIGINT NOT NULL,
    amount      DECIMAL(12,2) NOT NULL,
    reason      VARCHAR(200),
    status      VARCHAR(20) DEFAULT 'PENDING',
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (payment_id) REFERENCES payments(payment_id)
);

SQL Queries for Validation

-- 1. Verify payment amount matches order total
SELECT p.payment_id, p.amount AS payment_amt, o.total_amount AS order_total,
       CASE WHEN p.amount = o.total_amount THEN 'PASS' ELSE 'FAIL' END AS amt_check
FROM payments p
JOIN orders o ON p.order_id = o.order_id
WHERE p.payment_id = @payment_id;

-- 2. Verify status transition sequence (audit log or updated_at)
SELECT payment_id, payment_status, updated_at
FROM payments WHERE order_id = @order_id
ORDER BY updated_at;

-- 3. Verify order status after successful payment
SELECT o.order_id, o.order_status, p.payment_status
FROM orders o
JOIN payments p ON o.order_id = p.order_id
WHERE p.payment_status = 'COMPLETED' AND o.order_id = @order_id;
-- Expected: order_status = 'PAID'

-- 4. Verify failed payment does NOT mark order PAID
SELECT COUNT(*) AS false_paid_count
FROM orders o
JOIN payments p ON o.order_id = p.order_id
WHERE p.payment_status = 'FAILED' AND o.order_status = 'PAID';
-- Expected: 0

-- 5. Verify no duplicate successful payments for same order
SELECT order_id, COUNT(*) AS completed_count
FROM payments
WHERE payment_status = 'COMPLETED'
GROUP BY order_id
HAVING COUNT(*) > 1;

-- 6. Refund validation: total refunds <= payment amount
SELECT p.payment_id, p.amount AS paid,
       COALESCE(SUM(r.amount), 0) AS total_refunded
FROM payments p
LEFT JOIN refunds r ON p.payment_id = r.payment_id AND r.status = 'COMPLETED'
WHERE p.payment_id = @payment_id
GROUP BY p.payment_id, p.amount;

-- 7. Invalid state: REFUNDED without COMPLETED payment
SELECT r.refund_id FROM refunds r
JOIN payments p ON r.payment_id = p.payment_id
WHERE p.payment_status NOT IN ('COMPLETED', 'REFUNDED');

  SELECT user_id, email,

  FROM users

Top 100 SQL Interview Questions by Difficulty

Beginner Level (30 Questions)

BQ1. What is SQL and what is it used for?

Answer: SQL (Structured Query Language) is the standard language for managing relational databases — querying, inserting, updating, deleting data, and defining schema.

Interviewer expects: Basic definition; mention RDBMS (MySQL, PostgreSQL, Oracle, SQL Server).

BQ2. Difference between DDL, DML, DCL, and TCL?

Answer: DDL defines structure (CREATE, ALTER, DROP). DML manipulates data (INSERT, UPDATE, DELETE, SELECT). DCL controls access (GRANT, REVOKE). TCL manages transactions (COMMIT, ROLLBACK, SAVEPOINT).

Interviewer expects: Correct categorization with at least 2 commands per type.

BQ3. Write a query to select all columns from the employees table.

Answer: SELECT * FROM employees;

Interviewer expects: Correct syntax; bonus if they mention avoiding * in production.

BQ4. How do you select specific columns?

Answer: SELECT first_name, last_name, salary FROM employees;

Interviewer expects: Column list comma-separated from table.

BQ5. What does the WHERE clause do?

Answer: Filters rows based on a condition before returning results. Example: SELECT * FROM employees WHERE dept_id = 10;

Interviewer expects: Filtering concept with example.

BQ6. Difference between WHERE and HAVING?

Answer: WHERE filters rows before grouping. HAVING filters groups after GROUP BY. Example: SELECT dept_id, AVG(salary) FROM employees GROUP BY dept_id HAVING AVG(salary) > 50000;

Interviewer expects: Pre-aggregation vs post-aggregation filter distinction.

BQ7. How do you sort results in ascending and descending order?

Answer: ORDER BY salary ASC (default, lowest first); ORDER BY salary DESC (highest first). Multi-column: ORDER BY dept_id ASC, salary DESC;

Interviewer expects: ASC/DESC keywords and multi-column sort.

BQ8. What is the DISTINCT keyword?

Answer: Removes duplicate rows from result set. SELECT DISTINCT dept_id FROM employees; returns unique department IDs.

Interviewer expects: Uniqueness concept; not same as UNIQUE constraint.

BQ9. How does LIKE work with wildcards?

Answer: % matches zero or more chars; _ matches exactly one. Examples: WHERE name LIKE 'J%' (starts with J), WHERE name LIKE '_ohn' (4-char names ending in ohn).

Interviewer expects: Both wildcards with examples.

BQ10. What is the difference between COUNT(*), COUNT(column), and COUNT(DISTINCT column)?

Answer: COUNT(*) counts all rows including NULLs. COUNT(column) counts non-NULL values only. COUNT(DISTINCT column) counts unique non-NULL values.

Interviewer expects: NULL handling difference — critical for QA count validation.

BQ11. How do you handle NULL values in WHERE clause?

Answer: Use IS NULL or IS NOT NULL. Never use = NULL (always unknown). Example: WHERE email IS NULL;

Interviewer expects: IS NULL syntax; knows = NULL fails.

BQ12. What does COALESCE do?

Answer: Returns first non-NULL value from arguments. SELECT COALESCE(phone, 'N/A') FROM users; replaces NULL phone with 'N/A'.

Interviewer expects: NULL replacement; mention ISNULL (SQL Server) / NVL (Oracle) equivalents.

BQ13. Write an INSERT statement with specific columns.

Answer: INSERT INTO employees (emp_id, first_name, salary) VALUES (101, 'John', 50000);

Interviewer expects: Column list + VALUES; knows order matters if columns specified.

BQ14. How do you update multiple columns?

Answer: UPDATE employees SET salary = 55000, dept_id = 20 WHERE emp_id = 101;

Interviewer expects: SET with comma-separated columns; always mention WHERE to avoid updating all rows.

BQ15. Difference between DELETE and TRUNCATE?

Answer: DELETE removes specific rows (can use WHERE), is DML, can rollback, fires triggers. TRUNCATE removes all rows, is DDL, faster, resets identity, cannot rollback in some DBs.

Interviewer expects: DML vs DDL, rollback, WHERE clause difference.

BQ16. What is a PRIMARY KEY?

Answer: Column(s) that uniquely identify each row. NOT NULL + UNIQUE. One per table. Automatically indexed.

Interviewer expects: Uniqueness + NOT NULL; one PK per table.

BQ17. What is a FOREIGN KEY?

Answer: Column that references PRIMARY KEY of another table. Enforces referential integrity — prevents orphan records.

Interviewer expects: Parent-child relationship; orphan prevention.

BQ18. What is the IN operator?

Answer: Shorthand for multiple OR conditions. WHERE dept_id IN (10, 20, 30); equivalent to dept_id = 10 OR dept_id = 20 OR dept_id = 30.

Interviewer expects: Syntax and equivalence to OR.

BQ19. What is BETWEEN used for?

Answer: Range filter inclusive of boundaries. WHERE salary BETWEEN 40000 AND 60000; same as salary >= 40000 AND salary <= 60000.

Interviewer expects: Inclusive range; date ranges too.

BQ20. How do you limit rows returned (TOP/LIMIT)?

Answer: SQL Server: SELECT TOP 10 * FROM employees ORDER BY salary DESC; MySQL/PostgreSQL: SELECT * FROM employees ORDER BY salary DESC LIMIT 10;

Interviewer expects: Dialect awareness; ORDER BY with LIMIT for meaningful top-N.

BQ21. What are aggregate functions?

Answer: Functions that compute a single value from multiple rows: COUNT, SUM, AVG, MIN, MAX.

Interviewer expects: Name all five with brief purpose.

BQ22. What is GROUP BY?

Answer: Groups rows with same values in specified columns; used with aggregates. SELECT dept_id, COUNT(*) FROM employees GROUP BY dept_id;

Interviewer expects: Every non-aggregated SELECT column must be in GROUP BY.

BQ23. Can you use column alias in WHERE clause?

Answer: No. WHERE is evaluated before SELECT, so aliases aren't available. Use HAVING for aggregate aliases, or repeat the expression.

Interviewer expects: SQL execution order awareness.

BQ24. What is an alias and why use it?

Answer: Temporary name for column or table. SELECT first_name AS fname FROM employees e; Improves readability and shortens queries.

Interviewer expects: AS keyword (optional) for columns and tables.

BQ25. Difference between CHAR and VARCHAR?

Answer: CHAR is fixed-length (padded with spaces). VARCHAR is variable-length (stores only actual chars + length). Use CHAR for fixed codes (state abbreviations); VARCHAR for names, emails.

Interviewer expects: Storage difference; padding behavior of CHAR.

BQ26. What is a VIEW?

Answer: Virtual table based on a SELECT query. No stored data (usually). Simplifies complex queries; provides security by hiding columns.

Interviewer expects: Virtual table concept; not same as physical table.

BQ27. What happens if you omit WHERE in UPDATE or DELETE?

Answer: All rows in the table are updated or deleted. Dangerous in production — always verify with SELECT first.

Interviewer expects: Safety awareness; QA habit of SELECT before UPDATE.

BQ28. What is a UNIQUE constraint?

Answer: Ensures all values in column(s) are distinct. Allows one NULL (in most DBs). Unlike PK, multiple UNIQUE constraints per table allowed.

Interviewer expects: Difference from PRIMARY KEY.

BQ29. Write a query to find employees with NULL manager_id.

Answer: SELECT * FROM employees WHERE manager_id IS NULL;

Interviewer expects: IS NULL, not = NULL.

BQ30. What is the default JOIN type if you write just JOIN?

Answer: INNER JOIN. Returns only matching rows from both tables.

Interviewer expects: INNER JOIN as default; knows other types exist.

Intermediate Level (35 Questions)

IQ1. Explain INNER JOIN with example.

Answer: Returns rows where join condition matches in both tables. SELECT e.name, d.dept_name FROM employees e INNER JOIN departments d ON e.dept_id = d.dept_id;

Interviewer expects: Matching rows only; employees without dept excluded.

IQ2. Explain LEFT JOIN and when to use it.

Answer: Returns all rows from left table + matching from right; NULLs where no match. Use for orphan detection: SELECT e.* FROM employees e LEFT JOIN departments d ON e.dept_id = d.dept_id WHERE d.dept_id IS NULL;

Interviewer expects: All left rows preserved; QA orphan check pattern.

IQ3. Difference between LEFT JOIN and RIGHT JOIN?

Answer: LEFT keeps all left table rows; RIGHT keeps all right table rows. A RIGHT JOIN B = B LEFT JOIN A.

Interviewer expects: Symmetry understanding; prefer LEFT for readability.

IQ4. What is a FULL OUTER JOIN?

Answer: Returns all rows from both tables; NULLs where no match. MySQL doesn't support natively; use UNION of LEFT and RIGHT.

Interviewer expects: All rows both sides; dialect limitation awareness.

IQ5. What is a self join?

Answer: Joining a table to itself. Example: employee-manager hierarchy: SELECT e.name, m.name AS manager FROM employees e JOIN employees m ON e.manager_id = m.emp_id;

Interviewer expects: Aliases required; hierarchy use case.

IQ6. What is a subquery?

Answer: Query nested inside another query. Can be in WHERE, FROM, SELECT. Example: SELECT * FROM employees WHERE salary > (SELECT AVG(salary) FROM employees);

Interviewer expects: Nested SELECT; scalar vs table subquery.

IQ7. Difference between subquery and JOIN?

Answer: Both can achieve similar results. JOINs often perform better for large datasets. Subqueries are readable for filtering by aggregate. Optimizer may rewrite subqueries to joins.

Interviewer expects: Practical trade-offs, not dogmatic preference.

IQ8. What is a correlated subquery?

Answer: Inner query references outer query columns. Runs once per outer row. Example: employees earning above their department average.

Interviewer expects: Row-by-row execution concept; performance caution.

IQ9. Write query to find second highest salary.

Answer: SELECT MAX(salary) FROM employees WHERE salary < (SELECT MAX(salary) FROM employees); Or: SELECT DISTINCT salary FROM employees ORDER BY salary DESC LIMIT 1 OFFSET 1;

Interviewer expects: Multiple approaches; OFFSET or subquery.

IQ10. What is EXISTS and how differs from IN?

Answer: EXISTS returns true if subquery returns any row — stops at first match. Often faster than IN for large subqueries. WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.user_id)

Interviewer expects: Semi-join concept; SELECT 1 convention.

IQ11. Find duplicate emails in users table.

Answer: SELECT email, COUNT(*) FROM users GROUP BY email HAVING COUNT(*) > 1;

Interviewer expects: GROUP BY + HAVING pattern — daily QA query.

IQ12. What is UNION vs UNION ALL?

Answer: UNION combines results and removes duplicates (sorts). UNION ALL combines without dedup — faster. Column count and types must match.

Interviewer expects: Dedup difference; performance implication.

IQ13. Explain CASE expression with example.

Answer: Conditional logic in SQL. SELECT name, CASE WHEN salary > 80000 THEN 'Senior' WHEN salary > 50000 THEN 'Mid' ELSE 'Junior' END AS level FROM employees;

Interviewer expects: Searched CASE; useful for test data categorization.

IQ14. What are indexes and why use them?

Answer: Data structures (B-tree) speeding up SELECT/WHERE/JOIN. Trade-off: slower INSERT/UPDATE. Create on frequently filtered/joined columns.

Interviewer expects: Read vs write trade-off; not index everything.

IQ15. Difference between clustered and non-clustered index?

Answer: Clustered index defines physical row order (one per table, usually PK). Non-clustered is separate structure pointing to rows. Multiple non-clustered allowed.

Interviewer expects: One clustered per table; SQL Server specific but commonly asked.

IQ16. What is a composite key?

Answer: Primary key made of multiple columns together. Example: (order_id, product_id) in order_items — neither alone is unique, together they are.

Interviewer expects: Multi-column uniqueness example.

IQ17. What is referential integrity?

Answer: FK ensures child records reference valid parent records. Prevents orphan rows. ON DELETE CASCADE/SET NULL/RESTRICT controls parent deletion behavior.

Interviewer expects: FK + cascade options.

IQ18. Write query to find orphan order_items (no matching order).

Answer: SELECT oi.* FROM order_items oi LEFT JOIN orders o ON oi.order_id = o.order_id WHERE o.order_id IS NULL;

Interviewer expects: LEFT JOIN + IS NULL — core QA pattern.

IQ19. What is normalization in one sentence?

Answer: Organizing tables to minimize redundancy by splitting data and linking via keys through normal forms (1NF–3NF).

Interviewer expects: Redundancy reduction; link to Section 13 depth if probed.

IQ20. How do you find employees without any orders?

Answer: SELECT e.* FROM employees e LEFT JOIN orders o ON e.emp_id = o.emp_id WHERE o.order_id IS NULL; Or: WHERE NOT EXISTS (SELECT 1 FROM orders o WHERE o.emp_id = e.emp_id);

Interviewer expects: Anti-join pattern; two valid approaches.

IQ21. What is a CTE (Common Table Expression)?

Answer: Named temporary result set with WITH clause. WITH dept_avg AS (SELECT dept_id, AVG(salary) AS avg_sal FROM employees GROUP BY dept_id) SELECT e.name, e.salary, d.avg_sal FROM employees e JOIN dept_avg d ON e.dept_id = d.dept_id WHERE e.salary > d.avg_sal;

Interviewer expects: WITH syntax; readability over nested subqueries.

IQ22. Difference between DELETE with FK and TRUNCATE parent table?

Answer: DELETE respects FK constraints — fails if children exist (unless CASCADE). TRUNCATE fails on referenced parent table. Must delete/truncate children first or use CASCADE.

Interviewer expects: FK blocks both; order of deletion matters.

IQ23. What is a CHECK constraint?

Answer: Validates column values against a condition. CHECK (salary > 0) or CHECK (status IN ('ACTIVE','INACTIVE')). Test by attempting invalid INSERT.

Interviewer expects: QA tests boundary values against CHECK.

IQ24. How do you concatenate strings in SQL?

Answer: MySQL: CONCAT(first_name, ' ', last_name). SQL Server: first_name + ' ' + last_name or CONCAT (2012+). PostgreSQL: || operator.

Interviewer expects: Dialect differences; CONCAT handles NULL in MySQL.

IQ25. Find departments with more than 5 employees.

Interviewer expects: HAVING not WHERE for aggregate filter.

IQ26. What is a cross join?

Interviewer expects: M × N rows; test data generation use case.

IQ27. How to get current date/time in SQL?

Interviewer expects: Know their target DB function names.

IQ29. What is an AUTO_INCREMENT / IDENTITY column?

IQ30. How do you find mismatched data between two tables?

IQ31. What is the difference between CHAR length and byte storage?

IQ32. Explain ON DELETE CASCADE with example.

Advanced Level (35 Questions)

AQ1. Explain ROW_NUMBER, RANK, and DENSE_RANK.

Answer: ROW_NUMBER: unique sequential number per partition. RANK: same rank for ties, skips next (1,2,2,4). DENSE_RANK: same rank for ties, no skip (1,2,2,3). ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC)

Interviewer expects: Tie behavior difference; PARTITION BY + ORDER BY.

AQ2. What are window functions?

Answer: Perform calculations across related rows without collapsing like GROUP BY. Use OVER clause. Examples: ROW_NUMBER, SUM() OVER, LAG, LEAD.

Interviewer expects: Difference from aggregates — rows preserved.

AQ3. Explain LAG and LEAD with use case.

Answer: LAG accesses previous row; LEAD accesses next row. LAG(salary, 1) OVER (ORDER BY hire_date) compares current salary to previous employee's. Use for period-over-period analysis.

Interviewer expects: Offset parameter; QA use for audit trail comparison.

AQ4. What is a recursive CTE?

Answer: CTE that references itself. Anchor member + recursive member united by UNION ALL. Use for org hierarchies, bill of materials. Must have termination condition.

Interviewer expects: Anchor/recursive parts; avoid infinite loops.

AQ5. Explain transaction ACID properties.

Answer: Atomicity (all or nothing), Consistency (valid state to valid state), Isolation (concurrent transactions don't interfere), Durability (committed data survives crash).

Interviewer expects: All four defined; link to COMMIT/ROLLBACK.

AQ6. What are isolation levels?

Answer: READ UNCOMMITTED, READ COMMITTED (default most DBs), REPEATABLE READ, SERIALIZABLE. Higher isolation = less concurrency, fewer anomalies (dirty read, non-repeatable read, phantom read).

Interviewer expects: Name levels; dirty read vs phantom read.

AQ7. What is a deadlock and how do you detect it?

Answer: Two transactions waiting for each other's locks. DB kills one victim. Detect via error message or monitoring. Prevent by consistent lock ordering.

Interviewer expects: Practical awareness; QA may hit deadlocks in parallel tests.

AQ8. Write a query using SUM() OVER for running total.

Interviewer expects: OVER ORDER BY for cumulative sum.

AQ9. How do you optimize a slow query?

Answer: EXPLAIN/EXPLAIN ANALYZE, add indexes on WHERE/JOIN columns, avoid SELECT *, rewrite subqueries as JOINs, update statistics, avoid functions on indexed columns in WHERE.

Interviewer expects: EXPLAIN plan mention; index strategy.

AQ10. What is EXPLAIN PLAN used for?

Answer: Shows how optimizer executes query — scan type (index vs full table), join method, estimated rows. Essential for performance testing and index validation.

Interviewer expects: Has used EXPLAIN in real debugging.

AQ11. Difference between WHERE and ON in JOIN?

Answer: ON defines join condition (before outer join fill). WHERE filters after join. For LEFT JOIN, condition in ON preserves unmatched left rows; in WHERE converts to INNER behavior.

Interviewer expects: LEFT JOIN filter placement — common trick question.

AQ12. What is a materialized view?

Answer: View that stores physical results. Must refresh (FULL/INCREMENTAL). Faster reads; stale until refresh. Test refresh job and data accuracy.

Interviewer expects: Staleness testing for QA.

AQ13. Explain MERGE/UPSERT statement.

Answer: INSERT if not exists, UPDATE if exists. MySQL: INSERT ... ON DUPLICATE KEY UPDATE. PostgreSQL: INSERT ... ON CONFLICT DO UPDATE. SQL Server: MERGE.

Interviewer expects: Idempotent data load pattern.

AQ14. How do you find gaps in sequential IDs?

Answer: SELECT a.id + 1 AS gap_start FROM table a WHERE NOT EXISTS (SELECT 1 FROM table b WHERE b.id = a.id + 1) AND a.id < (SELECT MAX(id) FROM table);

Interviewer expects: Sequence integrity check for QA.

AQ15. What is query execution order?

Answer: FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT. Explains why alias fails in WHERE.

Interviewer expects: Logical order; not necessarily physical execution.

AQ16. Write query to delete duplicate rows keeping one.

Answer: DELETE FROM users WHERE user_id NOT IN (SELECT MIN(user_id) FROM users GROUP BY email); Or use ROW_NUMBER in CTE (safer).

Interviewer expects: Safe delete with backup/transaction mention.

AQ17. What is a covering index?

Answer: Index containing all columns needed by query — no table lookup required. INCLUDE columns (SQL Server) or composite index covering SELECT/WHERE.

Interviewer expects: Index-only scan in EXPLAIN.

AQ18. Explain CAP theorem briefly.

Answer: Distributed systems choose 2 of 3: Consistency, Availability, Partition tolerance. SQL RDBMS traditionally CP; NoSQL often AP. Relevant for microservices DB testing.

Interviewer expects: High-level; not deep distributed systems proof.

AQ19. What is sharding vs partitioning?

Answer: Partitioning splits one DB table into segments (same server). Sharding distributes across multiple servers. Both affect how QA validates data completeness.

Interviewer expects: Scale-out concept awareness.

AQ20. How do you test concurrent updates (lost update problem)?

Answer: Two sessions UPDATE same row without locking — last write wins. Test with optimistic locking (version column) or SELECT FOR UPDATE. Verify version increment and conflict error.

Interviewer expects: Race condition testing methodology.

AQ21. What is SELECT FOR UPDATE?

Answer: Locks selected rows for update until transaction commits. Prevents concurrent modification. SELECT * FROM accounts WHERE id = 1 FOR UPDATE;

Interviewer expects: Pessimistic locking for financial apps.

AQ22. Explain NTILE window function.

Answer: Divides rows into N roughly equal buckets. NTILE(4) OVER (ORDER BY salary) splits employees into 4 quartiles. Useful for sampling test data.

Interviewer expects: Practical use beyond definition.

AQ23. What is a hash join vs nested loop join?

Answer: Nested loop: for each outer row, scan inner (good for small tables). Hash join: builds hash table on smaller table (good for large equi-joins). Optimizer picks based on stats.

Interviewer expects: Read EXPLAIN join types.

AQ24. How do you compare two tables for data differences?

Answer: (SELECT * FROM source EXCEPT SELECT * FROM target) UNION ALL (SELECT * FROM target EXCEPT SELECT * FROM source); Or checksum: SELECT MD5(GROUP_CONCAT(row ORDER BY pk)) per table.

Interviewer expects: ETL/migration validation expertise.

AQ25. What is temporal table / system-versioned table?

Answer: SQL Server/PostgreSQL feature storing row history automatically. Current + history table. Query FOR SYSTEM_TIME AS OF '2024-01-01' for point-in-time.

Interviewer expects: Audit/history testing without app-level logging.

AQ26. Write a query to find customers who placed orders in 3 consecutive months.

Answer: Use LAG on distinct order months per customer; check month diff = 1 for two consecutive LAG values. Or self-join on month+1, month+2.

Interviewer expects: Complex analytical thinking; window functions preferred.

AQ27. What is the N+1 query problem?

Answer: App runs 1 query for list + N queries for each item's details. Fix with JOIN or batch IN query. Detect in testing via query log count.

Interviewer expects: API/DB integration performance awareness.

AQ28. Explain database connection pooling.

Answer: Reuses open connections instead of creating new ones per request. Test under load — pool exhaustion causes timeouts. Config: max pool size, idle timeout.

Interviewer expects: Performance/load test connection to DB.

AQ29. What is READ ONLY replica and how to test?

Answer: Copy of primary for read queries. Replication lag means stale reads. Test: write on primary, immediately read replica — may not see new data for seconds.

Interviewer expects: eventual consistency testing.

AQ30. How do you use SAVEPOINT?

Answer: BEGIN; INSERT ...; SAVEPOINT sp1; UPDATE ...; ROLLBACK TO sp1; COMMIT; Partial rollback within transaction. Test nested transaction behavior.

Interviewer expects: Granular rollback for complex test cleanup.

AQ31. What is an execution plan red flag?

Answer: Full table scan on large table, key lookups in millions, high cost percentage on sort/hash, estimated vs actual rows mismatch, missing index suggestions.

Top 10 JOIN Questions (Quick Revision)

  1. INNER JOIN — only matching rows from both tables.
  2. LEFT JOIN — all left rows + matches; use for orphan detection.
  3. RIGHT JOIN — mirror of LEFT; rarely used (swap tables instead).
  4. FULL OUTER JOIN — all rows both sides; NULL for gaps.
  5. Self join — same table twice with aliases (employee-manager).
  6. Cross join — Cartesian product; M × N rows.
  7. ON vs WHERE in LEFT JOIN — filter in ON preserves unmatched left rows.
  8. Multiple JOINs — chain: users → orders → order_items → products.
  9. JOIN vs subquery — JOIN usually faster; subquery more readable.
  10. Anti-join — LEFT JOIN IS NULL or NOT EXISTS for "rows without match."

Top 10 Window Function Questions

  1. ROW_NUMBER — unique sequential number; use for dedup and top-N per group.
  2. RANK — ties get same rank, gaps after (1,2,2,4).
  3. DENSE_RANK — ties same rank, no gaps (1,2,2,3).
  4. LEAD/LAG — access next/previous row for period comparison.
  5. SUM() OVER — running total without GROUP BY collapse.
  6. PARTITION BY — resets window per group (per dept, per customer).
  7. Window vs GROUP BY — window keeps all rows; GROUP BY collapses.
  8. NTILE — divide into N buckets for sampling/quartiles.
  9. FIRST_VALUE — first value in window frame.
  10. Top-N per group pattern — ROW_NUMBER + WHERE rn <= N in subquery/CTE.

Top 50 Quick Revision Points

  1. SQL = Structured Query Language for RDBMS.
  2. DDL: CREATE, ALTER, DROP, TRUNCATE.
  3. DML: INSERT, UPDATE, DELETE, SELECT.
  4. TCL: COMMIT, ROLLBACK, SAVEPOINT.
  5. PRIMARY KEY = unique + NOT NULL; one per table.
  6. FOREIGN KEY enforces referential integrity.
  7. UNIQUE allows one NULL (most DBs); multiple per table.
  8. CHECK validates column values against condition.
  9. DEFAULT auto-fills omitted columns on INSERT.
  10. Never use = NULL; use IS NULL.
  11. COUNT(*) counts all rows including NULLs.
  12. WHERE filters before grouping; HAVING after.
  13. Every non-aggregated SELECT column must be in GROUP BY.
  14. INNER JOIN returns only matching rows.
  15. LEFT JOIN preserves all left table rows.
  16. Orphan check = LEFT JOIN + WHERE parent.key IS NULL.
  17. Self join requires table aliases.
  18. Subquery in WHERE must return scalar for comparison operators.
  19. Correlated subquery runs once per outer row.
  20. EXISTS returns TRUE if subquery has any row.
  21. UNION removes duplicates; UNION ALL does not.
  22. CTE: WITH name AS (SELECT ...) SELECT ... FROM name.
  23. ROW_NUMBER always unique; RANK/DENSE_RANK handle ties differently.
  24. Window functions use OVER (PARTITION BY ... ORDER BY ...).
  25. DELETE can use WHERE; TRUNCATE removes all rows.
  26. TRUNCATE is DDL; may not rollback; resets identity.
  27. Index speeds reads, slows writes; don't over-index.
  28. EXPLAIN shows query execution plan.
  29. ACID: Atomicity, Consistency, Isolation, Durability.
  30. PreparedStatement prevents SQL injection.
  31. Normalization reduces redundancy (1NF→3NF).
  32. Denormalization improves read performance in DWH.
  33. 1NF: atomic values, no repeating groups.
  34. 2NF: no partial dependency on composite key.
  35. 3NF: no transitive dependency.
  36. Soft delete: is_deleted flag; row still in DB.
  37. Hard delete: row physically removed.
  38. API POST → verify INSERT in DB.
  39. API PUT → verify UPDATE in DB.
  40. API DELETE → verify DELETE or soft delete flag.
  41. ETL: validate source count = target count.
  42. Migration: compare checksums pre and post.
  43. DECIMAL for money; never FLOAT.
  44. utf8mb4 for emoji and international chars.
  45. Test isolation: unique data per test run.
  46. Async: poll DB with retry, don't assert immediately.
  47. Read replica may lag behind primary.
  48. ON DELETE CASCADE: deleting parent deletes children.
  49. Chopstick testing: Create → Read → Update → Delete.
  50. Always SELECT before UPDATE/DELETE in manual testing.

Common Mistakes to Avoid in SQL Interviews

  • Using = NULL instead of IS NULL — always wrong.
  • Forgetting WHERE in UPDATE/DELETE — updates/deletes all rows.
  • Using WHERE instead of HAVING for aggregate filters.
  • Not knowing LEFT JOIN for orphan detection — most common QA pattern.
  • Confusing RANK with ROW_NUMBER — ties behave differently.
  • Putting filter on LEFT JOIN in WHERE — accidentally converts to INNER JOIN.
  • Using SELECT * in production/testing — specify columns explicitly.
  • Comparing FLOAT/DECIMAL without tolerance — rounding failures.
  • Ignoring NULL in COUNT/GROUP BY — wrong counts.
  • Not mentioning PreparedStatement when discussing SQL in automation.
  • Saying "DELETE removes table" — that's DROP TABLE.
  • Unable to write second highest salary — prepare 2-3 approaches.
  • No QA context — always tie SQL to validation, not just syntax.
  • Ignoring transaction/testing rollback — show atomicity awareness.
  • Not asking which database dialect — TOP vs LIMIT vs ROWNUM matters.
