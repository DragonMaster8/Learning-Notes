GROUP BY and HAVING are foundational for reporting, data quality audits, and backend validation in e-commerce QA. Every SDET validating order totals, duplicate registrations, or error trends across environments relies on these clauses. This section uses a realistic schema: customers, orders, order_items, products, payments, and application_errors.

GROUP BY Fundamentals

GROUP BY collapses rows that share the same value(s) in the specified column(s) into a single group. Aggregate functions (COUNT, SUM, AVG, MIN, MAX) then operate on each group independently.

Golden Rule for QA: Every column in the SELECT list must either appear in GROUP BY or be wrapped in an aggregate function. Violating this causes SQL errors in strict databases (PostgreSQL, SQL Server) or unpredictable results in lenient ones (MySQL with ONLY_FULL_GROUP_BY disabled). 
-- Count orders per customer
SELECT customer_id, COUNT(*) AS order_count
FROM orders
WHERE status NOT IN ('CANCELLED', 'FAILED')
GROUP BY customer_id
ORDER BY order_count DESC;
-- Total revenue per product category (join required)
SELECT p.category_id,
       COUNT(DISTINCT oi.order_id) AS orders_with_category,
       SUM(oi.quantity * oi.unit_price) AS category_revenue
FROM order_items oi
JOIN products p ON oi.product_id = p.product_id
JOIN orders o ON oi.order_id = o.order_id
WHERE o.status = 'COMPLETED'
GROUP BY p.category_id
ORDER BY category_revenue DESC;

FROM orders

-- Average order value (AOV) by payment method
SELECT payment_method,
       ROUND(AVG(o.total_amount), 2) AS avg_order_value,
       MIN(o.total_amount) AS min_order,

FROM orders o

HAVING vs WHERE — Critical Distinction

WHERE filters individual rows before grouping. HAVING filters groups after aggregation. You cannot use aggregate functions in WHERE (with rare exceptions like subqueries).

AspectWHEREHAVING
When appliedBefore GROUP BYAfter GROUP BY
FiltersIndividual rowsGrouped results
Aggregate functionsNot allowed (directly)Allowed
Can use non-aggregated columns not in GROUP BYYesOnly if in GROUP BY
Example useWHERE status = 'COMPLETED'HAVING COUNT(*) > 5
PerformanceReduces rows early — preferred for row filtersProcesses all groups first — use only when needed
 
-- WRONG: aggregate in WHERE
SELECT customer_id, COUNT(*) AS cnt
FROM orders
WHERE COUNT(*) > 10          -- ERROR in all major DBs
GROUP BY customer_id;

-- CORRECT: filter rows with WHERE, filter groups with HAVING
SELECT customer_id, COUNT(*) AS order_count
FROM orders
WHERE order_date >= '2025-01-01'    -- row filter first
GROUP BY customer_id
HAVING COUNT(*) > 10;               -- group filter after aggregation
-- Customers who spent more than $5000 total in Q1 2025
SELECT o.customer_id,
       c.email,
       SUM(o.total_amount) AS total_spent
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
WHERE o.order_date BETWEEN '2025-01-01' AND '2025-03-31'
  AND o.status = 'COMPLETED'
GROUP BY o.customer_id, c.email
HAVING SUM(o.total_amount) > 5000
ORDER BY total_spent DESC;

Duplicate Detection Queries

Duplicate detection is one of the highest-value QA queries. Run these after data imports, API bulk operations, or migration cutovers.

Finding Duplicate Emails

-- Step 1: Find emails appearing more than once
SELECT LOWER(TRIM(email)) AS normalized_email,
       COUNT(*) AS duplicate_count
FROM customers
GROUP BY LOWER(TRIM(email))
HAVING COUNT(*) > 1
ORDER BY duplicate_count DESC;

-- Step 2: Drill into full records for remediation
SELECT customer_id, email, created_at, status
FROM customers
WHERE LOWER(TRIM(email)) IN (
    SELECT LOWER(TRIM(email))
    FROM customers
    GROUP BY LOWER(TRIM(email))
    HAVING COUNT(*) > 1
)
ORDER BY email, created_at;

Finding Duplicate Orders (Same Customer, Same Amount, Same Day)

SELECT customer_id,
       total_amount,
       DATE(order_date) AS order_day,
       COUNT(*) AS duplicate_order_count,
       STRING_AGG(CAST(order_id AS VARCHAR), ', ') AS order_ids
FROM orders
WHERE status NOT IN ('CANCELLED')
GROUP BY customer_id, total_amount, DATE(order_date)
HAVING COUNT(*) > 1;

Complex GROUP BY with Multiple Columns

-- Sales by category and region per month
SELECT DATE_TRUNC('month', o.order_date) AS sales_month,
       p.category_id,
       c.region,
       SUM(oi.quantity) AS units_sold,
       SUM(oi.quantity * oi.unit_price) AS revenue
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
JOIN customers c ON o.customer_id = c.customer_id
WHERE o.status = 'COMPLETED'
GROUP BY DATE_TRUNC('month', o.order_date), p.category_id, c.region
ORDER BY sales_month, category_id, region;
-- GROUP BY with ROLLUP for subtotals (see next section)
SELECT category_id, region, SUM(revenue) AS total_revenue
FROM (
    SELECT p.category_id, c.region, oi.quantity * oi.unit_price AS revenue
    FROM order_items oi
    JOIN products p ON oi.product_id = p.product_id
    JOIN orders o ON oi.order_id = o.order_id
    JOIN customers c ON o.customer_id = c.customer_id
    WHERE o.status = 'COMPLETED'
) sales
GROUP BY ROLLUP (category_id, region);

ROLLUP and CUBE — Brief Overview

ROLLUP generates subtotal and grand total rows. CUBE generates all possible grouping combinations — powerful for ad-hoc QA dashboards but expensive on large datasets.

-- ROLLUP: subtotals per category, then grand total
SELECT category_id,
       SUM(quantity * unit_price) AS revenue
FROM order_items oi
JOIN products p ON oi.product_id = p.product_id
GROUP BY ROLLUP (category_id);
-- Result includes: per-category rows + one row with category_id = NULL (grand total)

-- CUBE: all combinations of category and payment method
SELECT p.category_id, pay.payment_method, SUM(o.total_amount) AS revenue
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
JOIN payments pay ON o.order_id = pay.order_id
WHERE o.status = 'COMPLETED'
GROUP BY CUBE (p.category_id, pay.payment_method);

Interview Questions & Answers (12 Questions)

  1. What is the difference between GROUP BY and DISTINCT?
    DISTINCT removes duplicate row combinations. GROUP BY groups rows for aggregate calculations. SELECT DISTINCT customer_id FROM orders lists unique customers; SELECT customer_id, COUNT(*) FROM orders GROUP BY customer_id counts orders per customer.
  2. Can you use WHERE and HAVING together?
    Yes. WHERE filters rows before grouping; HAVING filters groups after. Example: find customers with more than 5 completed orders in 2025 — filter completed orders in WHERE, then HAVING COUNT(*) > 5.
  3. Why can't you write HAVING without GROUP BY?
    In standard SQL, HAVING without GROUP BY treats the entire result as one group (valid but rare). Most real queries pair HAVING with GROUP BY. Some databases allow HAVING COUNT(*) > 0 without explicit GROUP BY.
  4. What happens if you GROUP BY a nullable column?
    All NULL values form a single group. QA implication: if region is NULL for many customers, you get one group with NULL — verify whether NULL means "unknown" or a data defect.
  5. COUNT(*) vs COUNT(column) vs COUNT(DISTINCT column)?
    COUNT(*) counts all rows in the group including NULLs. COUNT(column) counts non-NULL values only. COUNT(DISTINCT column) counts unique non-NULL values — use for unique customer counts per day.
  6. How do you find duplicate records in SQL?
    GROUP BY the business key, HAVING COUNT(*) > 1. For emails: normalize with LOWER(TRIM(email)) first. Always drill into full rows for remediation.
  7. Write a query to find customers who placed orders in every month of 2025.
    SELECT customer_id
FROM orders
WHERE order_date >= '2025-01-01' AND order_date < '2026-01-01'
GROUP BY customer_id
HAVING COUNT(DISTINCT DATE_TRUNC('month', order_date)) = 12;
  8. What is the execution order of SQL clauses?
    FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. This explains why WHERE cannot reference SELECT aliases or aggregates.
  9. How would you validate monthly sales report accuracy as QA?
    Reconcile report totals against raw SUM(total_amount) with same filters. Cross-check order count. Spot-check random months manually. Compare staging vs production reports for same date range after deployment.
  10. GROUP BY with JOINs — common pitfall?
    Joining before GROUP BY can inflate counts if the join is one-to-many. Example: joining orders to order_items then COUNT(*) counts line items, not orders. Use COUNT(DISTINCT o.order_id) instead.
  11. What is ROLLUP used for in testing?
    Quick subtotal validation — verify category subtotals sum to grand total in financial reports. If subtotals don't reconcile, aggregation logic or data duplication in joins is suspect.
  12. How do you test for orphan records after a migration?
    Run LEFT JOIN queries from child to parent; assert zero rows where parent key IS NULL. Automate as post-migration smoke tests. Example: order_items LEFT JOIN orders WHERE orders.order_id IS NULL.

Section 8: CTEs & Window Functions

Common Table Expressions (CTEs) and window functions are among the most frequently tested SQL topics in SDET interviews. They enable readable complex queries, pagination, deduplication, running totals, and trend analysis — all critical for API-DB validation and data quality automation.

CTE (Common Table Expression) — Syntax and Advantages

A CTE is a temporary named result set defined with the WITH clause, referenced in the main query. It improves readability and allows query reuse within a single statement.

-- Basic CTE syntax
WITH completed_orders AS (
    SELECT order_id, customer_id, total_amount, order_date
    FROM orders
    WHERE status = 'COMPLETED'
)
SELECT customer_id,
       COUNT(*) AS order_count,
       SUM(total_amount) AS lifetime_value
FROM completed_orders
GROUP BY customer_id
HAVING SUM(total_amount) > 1000;

CTEs vs Subqueries — Comparison

AspectCTESubquery
ReadabilityTop-down, named, easy to debugNested, harder to read when deep
ReusabilityReference same CTE multiple times in one queryMust duplicate or use temp table
RecursionSupports WITH RECURSIVENot directly recursive
PerformanceMay be materialized (PostgreSQL 12+) or inlinedOften inlined; varies by optimizer

-- Multiple CTEs chained
WITH active_customers AS (
    SELECT customer_id, email FROM customers WHERE status = 'ACTIVE'
),
recent_orders AS (
    SELECT customer_id, SUM(total_amount) AS spend_30d
    FROM orders
    WHERE order_date >= CURRENT_DATE - INTERVAL '30 days'
      AND status = 'COMPLETED'
    GROUP BY customer_id
)
SELECT ac.customer_id, ac.email, COALESCE(ro.spend_30d, 0) AS spend_30d
FROM active_customers ac
LEFT JOIN recent_orders ro ON ac.customer_id = ro.customer_id
WHERE COALESCE(ro.spend_30d, 0) = 0;  -- active but no recent purchase

Recursive CTE — Hierarchical Data

Recursive CTEs traverse tree structures: org charts, category trees, bill-of-materials. Essential for validating nested product categories or manager hierarchies.

-- Category tree: find all subcategories under 'Electronics' (category_id = 1)
WITH RECURSIVE category_tree AS (
    -- Anchor: root category
    SELECT category_id, name, parent_category_id, 0 AS depth
    FROM product_categories
    WHERE category_id = 1

    UNION ALL

    -- Recursive: children
    SELECT pc.category_id, pc.name, pc.parent_category_id, ct.depth + 1
    FROM product_categories pc
    JOIN category_tree ct ON pc.parent_category_id = ct.category_id
)
SELECT category_id, name, depth
FROM category_tree
ORDER BY depth, name;
-- Org chart: all reports under manager_id = 100
WITH RECURSIVE org_hierarchy AS (
    SELECT employee_id, name, manager_id, 1 AS level
    FROM employees
    WHERE manager_id = 100

    UNION ALL

    SELECT e.employee_id, e.name, e.manager_id, oh.level + 1
    FROM employees e
    JOIN org_hierarchy oh ON e.manager_id = oh.employee_id
)
SELECT * FROM org_hierarchy ORDER BY level, name;
QA Scenario — Circular Reference Detection: After importing category data, run recursive CTE with a depth limit. If any row exceeds expected max depth (e.g., 10), investigate circular parent-child references that would break the UI navigation tree.

ROW_NUMBER() — Pagination, Deduplication, Ranking

-- Pagination: page 3, 20 rows per page (rows 41-60)
SELECT order_id, customer_id, total_amount, order_date
FROM (
    SELECT order_id, customer_id, total_amount, order_date,
           ROW_NUMBER() OVER (ORDER BY order_date DESC) AS rn
    FROM orders
    WHERE status = 'COMPLETED'
) paginated
WHERE rn BETWEEN 41 AND 60;
-- Deduplication: keep most recent record per email
WITH ranked AS (
    SELECT customer_id, email, created_at,
           ROW_NUMBER() OVER (
               PARTITION BY LOWER(TRIM(email))
               ORDER BY created_at DESC
           ) AS rn
    FROM customers
)
SELECT customer_id, email, created_at
FROM ranked
WHERE rn = 1;
-- Ranking within groups: top 3 products by revenue per category
WITH product_revenue AS (
    SELECT p.category_id, p.product_id, p.name,
           SUM(oi.quantity * oi.unit_price) AS revenue,
           ROW_NUMBER() OVER (
               PARTITION BY p.category_id
               ORDER BY SUM(oi.quantity * oi.unit_price) DESC
           ) AS rank_in_category
    FROM order_items oi
    JOIN products p ON oi.product_id = p.product_id
    GROUP BY p.category_id, p.product_id, p.name
)
SELECT * FROM product_revenue WHERE rank_in_category <= 3;

When two employees share the same salary, RANK() skips the next rank (1, 2, 2, 4). DENSE_RANK() does not skip (1, 2, 2, 3).

       DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_rank_no_gaps,

NTILE() — Dividing Data into Buckets

-- Split customers into 4 spending quartiles for A/B test segmentation
WITH customer_spend AS (

LEAD() and LAG() — Comparing Adjacent Rows

-- Compare each day's order count with previous and next day
WITH daily_orders AS (
    SELECT DATE(order_date) AS order_day, COUNT(*) AS order_count
    FROM orders
    GROUP BY DATE(order_date)
)
SELECT order_day,
       order_count,
       LAG(order_count)  OVER (ORDER BY order_day) AS prev_day_count,
       LEAD(order_count) OVER (ORDER BY order_day) AS next_day_count,
       order_count - LAG(order_count) OVER (ORDER BY order_day) AS day_over_day_change
FROM daily_orders
ORDER BY order_day;
-- Detect price changes between consecutive order line items for same product
SELECT product_id, order_id, unit_price, order_date,
       LAG(unit_price) OVER (PARTITION BY product_id ORDER BY order_date) AS prev_price,
       unit_price - LAG(unit_price) OVER (PARTITION BY product_id ORDER BY order_date) AS price_change
FROM order_items oi
JOIN orders o ON oi.order_id = o.order_id;

Running Totals and Moving Averages — SUM/AVG/COUNT OVER

-- Running total of daily revenue
SELECT DATE(order_date) AS order_day,
       SUM(total_amount) AS daily_revenue,
       SUM(SUM(total_amount)) OVER (ORDER BY DATE(order_date)) AS running_total
FROM orders
WHERE status = 'COMPLETED'
GROUP BY DATE(order_date)
ORDER BY order_day;
-- 7-day moving average of daily orders
WITH daily AS (
    SELECT DATE(order_date) AS order_day, COUNT(*) AS order_count
    FROM orders GROUP BY DATE(order_date)
)
SELECT order_day, order_count,
       ROUND(AVG(order_count) OVER (
           ORDER BY order_day
           ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
       ), 2) AS moving_avg_7d
FROM daily
ORDER BY order_day;

PARTITION BY — Detailed Explanation

PARTITION BY divides rows into independent groups for window function calculation — like GROUP BY but without collapsing rows. Each row retains its identity while getting a computed value within its partition.

Without PARTITION BY:     one window for ALL rows
  ROW_NUMBER() → 1, 2, 3, 4, 5, 6, 7, 8 ...

With PARTITION BY category_id:
  Category A: 1, 2, 3        (resets per category)
  Category B: 1, 2, 3, 4
  Category C: 1, 2

-- Running balance per customer (cumulative spend)
SELECT order_id, customer_id, order_date, total_amount,
       SUM(total_amount) OVER (
           PARTITION BY customer_id
           ORDER BY order_date
           ROWS UNBOUNDED PRECEDING
       ) AS running_balance
FROM orders
WHERE status = 'COMPLETED'
ORDER BY customer_id, order_date;

Year-over-Year (YoY) Revenue Comparison

WITH monthly_revenue AS (
    SELECT DATE_TRUNC('month', order_date) AS month,
           SUM(total_amount) AS revenue
    FROM orders WHERE status = 'COMPLETED'
    GROUP BY DATE_TRUNC('month', order_date)

       revenue,
       LAG(revenue, 12) OVER (ORDER BY month) AS revenue_same_month_last_year,
       ROUND(

ORDER BY month;

Interview Questions & Answers (16 Questions)

  1. What is a CTE and why use it over a subquery?
    A named temporary result set via WITH. Improves readability, allows multiple references, supports recursion. Subqueries work for simple cases; CTEs scale better for complex QA validation pipelines.
  2. Can a CTE be referenced multiple times?
    Yes — one major advantage. Example: compare current month revenue CTE against prior month CTE without duplicating logic.
  3. What is the difference between ROW_NUMBER, RANK, and DENSE_RANK?
    All assign ranks; ROW_NUMBER is always unique (needs tiebreaker). RANK skips ranks after ties. DENSE_RANK does not skip. Use ROW_NUMBER for deduplication; RANK/DENSE_RANK for leaderboard reporting.
  4. Write a query to find the 3rd highest order value.
    SELECT total_amount FROM (
    SELECT total_amount, DENSE_RANK() OVER (ORDER BY total_amount DESC) AS dr
    FROM orders
) t WHERE dr = 3;
  5. What does PARTITION BY do?
    Creates independent windows per group value. Resets window calculations for each partition. Like GROUP BY but keeps all rows.
  6. LEAD vs LAG — which looks forward?
    LEAD accesses the next row; LAG accesses the previous row. Both require ORDER BY in the OVER clause.
  7. How do you implement pagination with ROW_NUMBER?
    Wrap query in subquery with ROW_NUMBER, filter WHERE rn BETWEEN start AND end. Page N with size S: start = (N-1)*S+1, end = N*S.
  8. What is a running total query pattern?
    SUM(amount) OVER (ORDER BY date ROWS UNBOUNDED PRECEDING). Add PARTITION BY for per-customer running totals.
  9. How do you remove duplicates keeping the latest record?
    ROW_NUMBER partitioned by business key, ordered by timestamp DESC, filter rn = 1.
  10. What is NTILE used for in testing?
    Split data into equal buckets for load testing (simulate 10% traffic), stratified sampling, or quartile-based test case selection.
  11. Recursive CTE — what are anchor and recursive members?
    Anchor is the base case (non-recursive SELECT). Recursive member references the CTE itself via UNION ALL. Terminates when recursive member returns no rows.
  12. Window function vs GROUP BY aggregate?
    GROUP BY collapses rows into one per group. Window functions compute aggregates while preserving each row — essential for running totals and row-level comparisons.
  13. How to find employees earning more than their department average?
    SELECT employee_id, salary, department_id,
       AVG(salary) OVER (PARTITION BY department_id) AS dept_avg
FROM employees
-- filter in outer query: WHERE salary > dept_avg
  14. What is ROWS vs RANGE in window frame?
    ROWS counts physical rows (6 PRECEDING = 6 prior rows). RANGE uses logical value range (same date values). Use ROWS for moving averages by row count.
  15. How would you validate YoY dashboard data as SDET?
    Run independent queries for each year-month, compare against dashboard API response. Use LAG(12) query as source of truth. Assert growth percentage calculation handles division by zero.
  16. Performance concern with window functions?
    Full table scans without indexes on PARTITION BY / ORDER BY columns. For large tables, filter with WHERE first, then apply window function in subquery.

Section 9: Database Testing

Database testing validates that data stored, transformed, and retrieved by application layers is accurate, consistent, secure, and performant. For a 4-year SDET, this is not optional — UI tests prove the screen works; database tests prove the system of record is correct.

Database Testing Fundamentals

What Is Database Testing?

Verifying the data layer directly using SQL queries, schema comparison tools, and stored procedure execution — independent of or in complement to UI/API tests. It confirms CRUD operations persist correctly, constraints hold, and reports match source data.

Why Database Testing Matters

  • UI can display cached or computed values that mask DB corruption
  • API responses may be assembled from multiple tables — partial failures go undetected in UI-only tests
  • Financial/regulatory systems require audit trail validation at the DB level
  • ETL and migration defects surface only when comparing source vs target data

When to Perform Database Testing

SDET Principle: Every critical API test should have a corresponding DB assertion for at least one "source of truth" field — not the entire payload, but the fields that matter (order status, payment amount, inventory count).

Types of Database Testing

Structural Testing (Schema Testing)

  • Verify tables, columns, data types, nullability, defaults exist as designed
  • Validate primary keys, foreign keys, unique constraints, check constraints
  • Compare staging schema vs production after migration scripts run
  • Verify indexes exist on expected columns
-- PostgreSQL: list foreign keys on orders table
SELECT tc.constraint_name, kcu.column_name,
       ccu.table_name AS foreign_table, ccu.column_name AS foreign_column
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
  ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.constraint_column_usage ccu
  ON ccu.constraint_name = tc.constraint_name
WHERE tc.constraint_type = 'FOREIGN KEY'
  AND tc.table_name = 'orders';

Functional Testing

  • CRUD operations persist expected values
  • Stored procedures return correct results for valid/invalid inputs
  • Triggers fire on INSERT/UPDATE/DELETE and modify related tables
  • Business rules enforced via constraints (e.g., order total > 0)

Non-Functional Testing

  • Performance: Query response time under load; identify missing indexes
  • Security: Role-based access; no plaintext passwords; SQL injection resistance
  • Scalability: Table growth, partition strategy, archive policies
  • Recovery: Backup restore validation; point-in-time recovery

Backend Validation — Verifying Data After UI Actions

Pattern: perform UI action → capture entity ID from UI/API → query DB → assert.

-- After UI checkout: verify order header

Data Integrity Testing

Entity Integrity

Primary keys are unique and not NULL. No duplicate primary keys exist.

SELECT order_id, COUNT(*) FROM orders GROUP BY order_id HAVING COUNT(*) > 1;

Referential Integrity

Every foreign key value references an existing parent row.

-- Orphan order_items (no parent order)
SELECT oi.* FROM order_items oi
LEFT JOIN orders o ON oi.order_id = o.order_id
WHERE o.order_id IS NULL;

Domain Integrity

Values fall within allowed ranges and types. Status fields contain only valid enum values.

SELECT order_id, status FROM orders
WHERE status NOT IN ('PENDING','PAID','SHIPPED','DELIVERED','CANCELLED','REFUNDED');

ETL Testing — Extract, Transform, Load

ETL Process Overview

SOURCE (OLTP DB / Files)  →  STAGING  →  TRANSFORM  →  TARGET (Data Warehouse)
     Extract                  Load         Rules           Load

ETL Validation Approach

  1. Row count reconciliation: source count = target count (± expected filters)
  2. Column mapping: every target column has a defined source and transformation
  3. Null handling: source NULLs map correctly (default value or remain NULL)
  4. Duplicate handling: dedup logic verified with known duplicate source rows
  5. Incremental load: only new/changed rows loaded after initial full load
  6. Reject/quarantine: invalid rows logged in error table, not silently dropped
-- Source vs target row count
SELECT 'source' AS layer, COUNT(*) FROM source_db.orders WHERE order_date = '2025-06-01'
UNION ALL
SELECT 'target', COUNT(*) FROM dw.fact_orders WHERE order_date_key = 20250601;

-- Checksum comparison on aggregated amounts
SELECT SUM(total_amount) FROM source_db.orders WHERE order_date = '2025-06-01';
SELECT SUM(order_amount) FROM dw.fact_orders WHERE order_date_key = 20250601;

Common ETL Issues

  • Timezone conversion errors (order_date off by one day)
  • Truncation (VARCHAR(50) cuts product names)
  • Character encoding (UTF-8 special characters corrupted)
  • Duplicate key violations on re-run without idempotent logic
  • Orphan dimension keys (fact references missing dimension row)

Data Migration Testing

Pre-Migration

  • Baseline row counts and checksums per table
  • Document known data quality issues in source
  • Validate migration scripts in lower environment with production-like volume

During Migration

  • Monitor error logs and rejected record counts
  • Verify application read-only mode if applicable
  • Check migration job checkpoints and resume capability

Post-Migration

-- Row count match
SELECT 'orders' AS tbl,
       (SELECT COUNT(*) FROM legacy.orders) AS legacy_cnt,
       (SELECT COUNT(*) FROM new_schema.orders) AS new_cnt;

-- Sample record comparison (hash or field-by-field)
SELECT l.order_id, l.total_amount AS legacy_amount, n.total_amount AS new_amount
FROM legacy.orders l
JOIN new_schema.orders n ON l.order_id = n.order_id
WHERE l.total_amount <> n.total_amount
   OR l.status <> n.status
LIMIT 100;

Data Reconciliation — Source vs Target

-- Records in source but missing in target
SELECT s.order_id FROM source.orders s
LEFT JOIN target.orders t ON s.order_id = t.order_id
WHERE t.order_id IS NULL;

-- Records in target but not in source (unexpected inserts)
SELECT t.order_id FROM target.orders t
LEFT JOIN source.orders s ON t.order_id = s.order_id
WHERE s.order_id IS NULL;

-- Field-level mismatch report
SELECT s.order_id,
       s.total_amount AS src_amount, t.total_amount AS tgt_amount,
       s.status AS src_status, t.status AS tgt_status
FROM source.orders s
JOIN target.orders t ON s.order_id = t.order_id
WHERE s.total_amount <> t.total_amount OR s.status <> t.status;

Stored Procedure Testing

  1. Identify input parameters and expected output (result set or OUT params)
  2. Test happy path with valid inputs
  3. Test boundary values (zero quantity, max discount)
  4. Test invalid inputs (negative amount, non-existent customer_id)
  5. Verify side effects (related tables updated, audit log written)
  6. Compare procedure output with equivalent manual SQL for same inputs
-- Example: test sp_create_order
CALL sp_create_order(
    p_customer_id => 1001,
    p_product_ids => ARRAY[101, 102],
    p_quantities  => ARRAY[1, 2],
    p_order_id    => NULL  -- OUT parameter
);
-- Then verify:
SELECT * FROM orders WHERE order_id = :returned_order_id;
SELECT * FROM order_items WHERE order_id = :returned_order_id;
SELECT stock_quantity FROM inventory WHERE product_id IN (101, 102);

Trigger Testing

-- Trigger: after INSERT on orders, write to order_audit_log
-- Test: insert order, verify audit row
INSERT INTO orders (customer_id, total_amount, status)
VALUES (1001, 299.99, 'PENDING');

SELECT * FROM order_audit_log
WHERE order_id = (SELECT MAX(order_id) FROM orders)
  AND action = 'INSERT'
  AND changed_at >= CURRENT_TIMESTAMP - INTERVAL '1 minute';

SELECT column_name, data_type, is_nullable, column_default
FROM information_schema.columns
WHERE table_name = 'orders'
ORDER BY ordinal_position;

Test Data Management Strategies

  • Dedicated test schema: Isolated from production; refreshed nightly from masked snapshot
  • Factory pattern: Programmatic test data creation with unique identifiers (timestamp suffix)
  • Cleanup hooks: @AfterEach / @AfterClass deletes test-created rows by tag column or ID prefix
  • Idempotent seeds: SQL scripts use MERGE/UPSERT so re-runs don't fail
  • PII masking: Never use real customer data in lower environments
  • Referential order: Delete child rows before parent rows in cleanup scripts

Interview Questions & Answers (16 Questions)

  1. What is the difference between frontend and database testing?
    Frontend validates presentation and user flows. Database testing validates persisted state, constraints, and data transformations directly — catches bugs UI may hide due to caching or computed views.
  2. How do you test referential integrity?
    Attempt to insert child row with invalid FK (expect failure). Query for orphan child rows via LEFT JOIN. Verify ON DELETE CASCADE behavior if applicable.
  3. What checks do you run after a data migration?
    Row counts, checksums on numeric columns, sample field-by-field comparison, FK constraint validation, application smoke tests, orphan detection queries.
  4. How do you test a stored procedure?
    Execute with known inputs, assert output parameters/result sets, verify side effects on related tables, test invalid inputs for expected error handling.
  5. What is ETL testing?
    Validating data movement from source to target through transformation rules — row counts, data types, business rule application, incremental load correctness, error handling for bad records.
  6. How do you validate CRUD at the database level?
    Pre-query confirms state, perform operation via API/UI, post-query asserts expected changes. Include negative cases (duplicate insert, update non-existent row).
  7. What is data reconciliation?
    Comparing source and target datasets to find missing, extra, or mismatched records — essential after ETL runs and migrations.
  8. How do you test triggers?
    Perform triggering action (INSERT/UPDATE/DELETE), query affected/audit tables, verify trigger logic including edge cases (bulk operations, NULL values).
  9. What tools do you use for database testing?
    DBeaver/pgAdmin for ad-hoc queries; JDBC in automation frameworks; schema compare tools (Liquibase, Flyway) for migration validation; CI pipelines running SQL assertion scripts.
  10. How do you manage test data in automation?
    Create data via API/factory before test, tag with unique identifier, assert, cleanup in teardown. Use transactions with rollback where supported for isolation.
  11. Structural vs functional database testing?
    Structural = schema, constraints, indexes. Functional = CRUD correctness, business rules, procedure/trigger behavior.
  12. How do you test soft delete?
    After DELETE API call, verify row still exists with is_deleted = true and deleted_at populated. Confirm row excluded from normal SELECT queries but present in admin/audit queries.
  13. What is domain integrity?
    Column values conform to defined domain — data type, range, allowed enum values. Test by querying for out-of-domain values.
  14. How do you automate database assertions in CI?
    JDBC/Node pg client runs SQL after API tests; assert row counts and field values; fail build on mismatch. Store expected values in test data files or compute dynamically.
  15. Common data quality issues you check for?
    Duplicates, NULLs in required fields, orphan FKs, amount mismatches between header and line items, future-dated records, inactive users with recent orders.
  16. How do you test without affecting production data?
    Dedicated test environment, test schema isolation, transaction rollback pattern, never run destructive queries against shared/staging without approval.

Section 10: API + Database Validation

Modern SDET workflows require end-to-end validation: the API response tells you what the service claims happened; the database tells you what actually happened. This section covers the complete workflow, SQL validation queries, and automation patterns with REST Assured, JDBC, Postman, and UI frameworks.

End-to-End API-DB Testing Workflow

1. Setup test data (API or direct SQL seed)
2. Capture baseline DB state (optional SELECT)
3. Execute API call (POST/PUT/DELETE/GET)
4. Assert API response (status, body, headers)
5. Query database with entity ID from response
6. Assert DB values match API response fields
7. Cleanup test data (API delete or SQL teardown)
Best Practice: Map each API response field to a specific DB column in your test design document. Unmapped fields are untested persistence — a common gap in API test suites.

Scenario 1: POST API Creates Order → Verify in Database

API: POST /api/v1/orders with cart items → returns 201 with orderId, status, totalAmount.

-- Step 1: Verify order header persisted
SELECT order_id, customer_id, status, total_amount, created_at
FROM orders
WHERE order_id = :orderIdFromResponse;

-- Expected: status = 'PENDING', total_amount matches API response

-- Step 2: Verify order line items
SELECT product_id, quantity, unit_price, quantity * unit_price AS line_total
FROM order_items
WHERE order_id = :orderIdFromResponse
ORDER BY product_id;

-- Step 3: Verify line items sum equals header total
SELECT o.order_id, o.total_amount,
       SUM(oi.quantity * oi.unit_price) AS calculated_total
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
WHERE o.order_id = :orderIdFromResponse
GROUP BY o.order_id, o.total_amount;

-- Step 4: Verify payment record created
SELECT payment_id, order_id, amount, status, payment_method
FROM payments
WHERE order_id = :orderIdFromResponse;

-- Step 5: Verify inventory decremented
SELECT i.product_id, i.stock_quantity
FROM inventory i
WHERE i.product_id IN (:productIdsFromRequest);

Scenario 2: PUT API Updates User → Verify Update in DB

API: PUT /api/v1/customers/{id} updates email and phone.

-- Pre-update baseline
SELECT customer_id, email, phone, updated_at
FROM customers WHERE customer_id = :customerId;

-- Post-update verification
SELECT customer_id, email, phone, updated_at
FROM customers WHERE customer_id = :customerId;
-- Assert: email and phone match request body
-- Assert: updated_at > pre-update timestamp

-- Verify audit trail if applicable
SELECT customer_id, field_name, old_value, new_value, changed_at
FROM customer_audit_log
WHERE customer_id = :customerId
ORDER BY changed_at DESC
LIMIT 5;

Scenario 3: DELETE API → Verify Soft Delete vs Hard Delete

Soft Delete Pattern

-- After DELETE /api/v1/customers/{id}
SELECT customer_id, email, is_deleted, deleted_at
FROM customers
WHERE customer_id = :customerId;
-- Expect: is_deleted = true, deleted_at IS NOT NULL, row still exists

-- Verify customer excluded from active listing query (same as app uses)
SELECT COUNT(*) FROM customers
WHERE customer_id = :customerId AND is_deleted = false;
-- Expect: 0

Hard Delete Pattern

-- After DELETE /api/v1/cart/items/{itemId}
SELECT COUNT(*) FROM cart_items WHERE cart_item_id = :itemId;
-- Expect: 0

-- Verify cascade: related records also removed if designed
SELECT COUNT(*) FROM cart_item_options WHERE cart_item_id = :itemId;
-- Expect: 0

Automation Strategy — REST Assured + JDBC

JDBC Connection Setup (Java)

import java.sql.*;

public class DatabaseHelper {
    private static final String URL =
        "jdbc:postgresql://localhost:5432/ecommerce_test";
    private static final String USER = "test_user";
    private static final String PASS = "test_pass";

    public static Connection getConnection() throws SQLException {
        return DriverManager.getConnection(URL, USER, PASS);
    }

    public static void closeQuietly(AutoCloseable resource) {
        if (resource != null) {
            try { resource.close(); }
            catch (Exception e) { /* log */ }
        }
    }
}

Execute Query After API Call (Java)

public Map<String, Object> getOrderById(long orderId) throws SQLException {
    String sql = "SELECT order_id, customer_id, status, total_amount " +
                 "FROM orders WHERE order_id = ?";
    try (Connection conn = DatabaseHelper.getConnection();
         PreparedStatement ps = conn.prepareStatement(sql)) {
        ps.setLong(1, orderId);
        try (ResultSet rs = ps.executeQuery()) {
            if (!rs.next()) return null;
            Map<String, Object> row = new HashMap<>();
            row.put("order_id", rs.getLong("order_id"));
            row.put("customer_id", rs.getLong("customer_id"));
            row.put("status", rs.getString("status"));
            row.put("total_amount", rs.getBigDecimal("total_amount"));
            return row;
        }
    }
}

Assert DB Values Match API Response (Java + REST Assured + TestNG)

@Test
public void createOrder_shouldPersistToDatabase() throws SQLException {
    OrderRequest payload = OrderRequest.builder()
        .customerId(1001L)
        .items(List.of(new Item(101, 2), new Item(205, 1)))
        .build();

    Response response = given()
        .contentType(ContentType.JSON)
        .body(payload)
        .when()
        .post("/api/v1/orders")
        .then()
        .statusCode(201)
        .extract().response();

    long orderId = response.jsonPath().getLong("orderId");
    String apiStatus = response.jsonPath().getString("status");
    double apiTotal = response.jsonPath().getDouble("totalAmount");

    Map<String, Object> dbOrder = getOrderById(orderId);
    assertNotNull(dbOrder, "Order not found in database");
    assertEquals(dbOrder.get("status"), apiStatus);
    assertEquals(((BigDecimal) dbOrder.get("total_amount")).doubleValue(),
                 apiTotal, 0.01);

    // Verify line item count
    int dbItemCount = getOrderItemCount(orderId);
    assertEquals(dbItemCount, payload.getItems().size());
}

Automation with Postman + Database (Newman Approach)

  1. Create Postman collection with API requests and test scripts asserting response fields
  2. Use Postman's built-in SQL request type (Postman v10+) or pre-request script calling external DB helper
  3. Alternative: Newman runs collection in CI; separate Node.js script queries DB and compares JSON output
  4. Store orderId in Postman environment variable from test script: pm.environment.set("orderId", json.orderId)
// Postman test script — capture ID for downstream DB validation
pm.test("Order created", function () {
    pm.response.to.have.status(201);
    var json = pm.response.json();
    pm.expect(json.orderId).to.be.a('number');
    pm.environment.set("orderId", json.orderId);
    pm.environment.set("expectedTotal", json.totalAmount);
});

Playwright / Selenium + DB Validation Patterns

  • UI triggers action, API ID captured from network: Intercept POST response in Playwright, extract orderId, query DB via JDBC helper
  • UI shows value, DB confirms source: Read displayed order total from DOM, compare with SQL query result
  • Async persistence: Poll DB with retry (order may be written asynchronously via message queue)
// Playwright: intercept API response, then validate DB
const [response] = await Promise.all([
    page.waitForResponse(r => r.url().includes('/api/v1/orders') && r.status() === 201),
    page.click('[data-testid="place-order-btn"]')
]);
const { orderId, totalAmount } = await response.json();
const dbRow = await dbHelper.query(
    'SELECT total_amount FROM orders WHERE order_id = $1', [orderId]
);
expect(Number(dbRow.total_amount)).toBeCloseTo(totalAmount, 2);

Data Cleanup Strategies for Test Automation

  • API cleanup: Call DELETE endpoint in @AfterMethod — mirrors production delete path
  • SQL cleanup: Delete child rows first, then parent (respect FK order)
  • Tagged data: Insert with source = 'AUTOMATION' column; bulk delete by tag
  • Transaction rollback: Wrap test in transaction, rollback after assertions (not viable if API uses separate connection pool)
  • Dedicated test customer: Reuse customer_id = 999999; reset state before each test
  • Idempotent teardown: DELETE FROM order_items WHERE order_id IN (SELECT order_id FROM orders WHERE customer_id = 999999)
-- Cleanup script (run in @AfterClass)
DELETE FROM payments WHERE order_id IN (
    SELECT order_id FROM orders WHERE customer_id = 999999
);
DELETE FROM order_items WHERE order_id IN (
    SELECT order_id FROM orders WHERE customer_id = 999999
);
DELETE FROM orders WHERE customer_id = 999999;

Interview Questions & Answers (12 Questions)

  1. Why validate API response AND database?
    API may return success while async persistence fails, or response may be computed/cached without matching stored state. DB is the system of record.
  2. How do you get the entity ID for DB validation after POST?
    Extract from API response body (orderId, id field). Never hardcode IDs — they are auto-generated.
  3. How do you handle async DB writes in tests?
    Poll DB with retry (e.g., Awaitility: wait up to 10s, poll every 500ms) until expected row appears or timeout fails test.
  4. REST Assured + JDBC integration approach?
    Separate DB helper class with PreparedStatement queries. Test method: API call → extract ID → JDBC query → assert fields match response.
  5. How do you compare decimal amounts (API vs DB)?
    Use delta comparison: assertEquals(apiAmount, dbAmount, 0.01). Never compare doubles with ==.
  6. Soft delete vs hard delete — how to verify each?
    Soft: row exists, is_deleted = true. Hard: SELECT COUNT(*) = 0. Also verify related records per cascade rules.
  7. What if API returns 201 but DB has no row?
    Bug in persistence layer, transaction rolled back after response sent, or writing to wrong schema/environment. Log API timestamp and check DB connection config in test.
  8. How do you avoid test data pollution?
    Unique identifiers per run, cleanup in teardown, dedicated test accounts, never use production DB for automation.
  9. Can Postman directly query a database?
    Native SQL requests in newer Postman versions; otherwise use Newman + external script or pre-request hooks calling a DB microservice.
  10. How do you validate nested API arrays against DB?
    Count rows in child table = array length. Sort both by key, compare field-by-field. Example: order_items rows vs items[] in response.
  11. UI + DB validation — when to use?
    When UI displays computed/formatted values — verify display matches DB source. Intercept network call for entity ID rather than parsing DOM for IDs.
  12. How do you test PUT partial update?
    Capture pre-update DB snapshot. Send PATCH with one field changed. Assert changed field updated, all other fields unchanged, updated_at incremented.

Section 11: Indexing & Performance

QA engineers don't just verify functional correctness — they identify slow queries that degrade user experience and catch missing indexes that cause production incidents. Understanding indexing and execution plans helps you file actionable performance bugs with evidence.

What Is an Index?

An index is a separate data structure that speeds up row lookup — like a book's index pointing to page numbers instead of reading every page sequentially.

Without Index (Full Table Scan):     With Index (Seek):
  Row 1  ─┐                            Index: customer_id → row location
  Row 2   │ Scan ALL rows              Lookup: customer_id = 1001 → Row 847
  Row 3   │ O(n) time                  Read ONLY Row 847
  ...     │
  Row 1M ─┘                            O(log n) time with B-Tree
QA Analogy: Testing a search feature without an index is like verifying the search box works while the backend reads every product row — functionally correct but unusable at scale. Performance is a test requirement.

Clustered Index

  • Determines the physical order of data rows on disk
  • Only one clustered index per table (usually the primary key)
  • Data rows are stored in the index leaf pages — no separate heap lookup needed
  • Range queries on clustered key are fast (e.g., orders by order_date if clustered on date)
-- SQL Server: view clustered index
SELECT i.name AS index_name, i.type_desc
FROM sys.indexes i
WHERE i.object_id = OBJECT_ID('orders') AND i.type = 1;

-- PostgreSQL: tables are heap-organized by default;
-- CLUSTER command temporarily reorders by index (not maintained automatically)

Non-Clustered Index

  • Separate structure from data rows — contains key columns + pointer to row
  • Multiple non-clustered indexes allowed per table
  • Extra lookup step (key lookup / bookmark lookup) to fetch non-indexed columns
  • Ideal for frequently filtered/joined columns not in clustered key
-- Create non-clustered index on frequently searched column
CREATE INDEX idx_orders_customer_id ON orders (customer_id);
CREATE INDEX idx_orders_status ON orders (status);
CREATE INDEX idx_orders_order_date ON orders (order_date);

Composite Index — Column Order Matters

A composite index on (customer_id, order_date) supports queries filtering by customer_id alone or by both columns. It does not efficiently support queries filtering only by order_date (leftmost prefix rule).

CREATE INDEX idx_orders_customer_date ON orders (customer_id, order_date);

-- USES index efficiently:
SELECT * FROM orders WHERE customer_id = 1001;
SELECT * FROM orders WHERE customer_id = 1001 AND order_date > '2025-01-01';

-- May NOT use index efficiently (order_date alone):
SELECT * FROM orders WHERE order_date > '2025-01-01';

Covering Index

Includes all columns needed by a query in the index itself — avoids key lookup to the table (index-only scan).

-- Query always needs customer_id, status, total_amount
CREATE INDEX idx_orders_covering
ON orders (customer_id)
INCLUDE (status, total_amount);  -- SQL Server syntax

-- PostgreSQL equivalent: include columns in index definition
CREATE INDEX idx_orders_covering_pg
ON orders (customer_id) INCLUDE (status, total_amount);

Unique Index

Enforces uniqueness while providing lookup speed. Automatically created for PRIMARY KEY and UNIQUE constraints.

CREATE UNIQUE INDEX idx_customers_email ON customers (LOWER(email));

-- QA validation: attempt duplicate insert should fail
INSERT INTO customers (email, name) VALUES ('test@example.com', 'Test');
-- Second insert with same email → unique violation error

When to Create Indexes

  • Columns in WHERE, JOIN, ORDER BY clauses with high selectivity
  • Foreign key columns (speeds up joins and cascade operations)
  • Tables with high SELECT-to-INSERT ratio (product catalog, order history reads)
  • Columns used in reporting queries run frequently

When NOT to Create Indexes

  • Small tables (full scan is faster than index seek overhead)
  • Columns with low selectivity (e.g., boolean is_active with 50/50 split)
  • Tables with heavy INSERT/UPDATE/DELETE and few reads (write-heavy audit logs)
  • Columns rarely used in queries
  • Over-indexing: each index slows writes and consumes storage

N+1 Query Problem

Application loads a list of N orders, then executes 1 query per order to fetch items — total N+1 queries instead of 1 JOIN or 2 batch queries.

N+1 Pattern (Bad):                    Optimized (Good):
  SELECT * FROM orders; -- 1 query       SELECT o.*, oi.*
  foreach order:                          FROM orders o
    SELECT * FROM order_items             JOIN order_items oi ON ...
    WHERE order_id = ?;  -- N queries     WHERE o.customer_id = 1001;
  Total: N+1 queries                     Total: 1 query

QA Detection: Enable SQL query logging during UI test. Navigate to order history page. If log shows 1 query for orders + 20 identical item queries, file N+1 performance bug.

Interview Questions & Answers (12 Questions)

  1. What is a database index?
    A data structure (usually B-Tree) that speeds up row retrieval by key — trades storage and write speed for read speed.
  2. Clustered vs non-clustered index?
    Clustered defines physical row order; one per table. Non-clustered is separate structure with pointer to row; multiple allowed.
  3. Why does composite index column order matter?
    Leftmost prefix rule: index on (A, B) supports WHERE A and WHERE A,B but not efficiently WHERE B alone.
  4. What is a covering index?
    Index containing all columns a query needs — enables index-only scan without table lookup.
  5. When should you NOT add an index?
    Small tables, low-selectivity columns, write-heavy tables with rare reads, or when too many indexes degrade INSERT/UPDATE performance.
  6. What is EXPLAIN used for?
    Shows how the database executes a query — scan types, join methods, estimated cost. Essential for diagnosing slow queries.
  7. Seq Scan vs Index Scan vs Index Seek?
    Seq Scan reads all rows. Index Scan reads all index entries. Index Seek directly locates specific entries — fastest for point lookups.
  8. What is the N+1 query problem?
    Loading a collection then querying related data one row at a time. Fix with JOINs or batch IN queries. Detect via SQL logging during UI tests.
  9. What is a sargable query?
    Search ARGument ABLE — predicate written so index can be used. Avoid functions on indexed columns in WHERE clause.
  10. How does indexing affect INSERT performance?
    Each index must be updated on insert. More indexes = slower writes. Balance read vs write patterns.
  11. How would you test query performance as SDET?
    Define SLA thresholds, run EXPLAIN ANALYZE in CI for critical queries, compare plans across releases, monitor query logs during load tests.
  12. Unique index vs unique constraint?
    Unique constraint is a logical rule; unique index is the physical implementation enforcing it. Both prevent duplicates.

Section 12: Transactions & ACID

Transactions guarantee that multi-step database operations (checkout, payment, inventory update) either complete fully or leave no partial state. Every SDET testing e-commerce order flows, financial transfers, or concurrent user actions must understand ACID properties and isolation levels.

What Is a Transaction?

A transaction is a logical unit of work comprising one or more SQL statements that execute as a single indivisible operation. It begins with BEGIN (or implicit start), ends with COMMIT (persist changes) or ROLLBACK (undo all changes).

BEGIN;
    INSERT INTO orders (customer_id, total_amount, status)
    VALUES (1001, 599.98, 'PENDING');

    INSERT INTO order_items (order_id, product_id, quantity, unit_price)
    VALUES (currval('orders_order_id_seq'), 101, 2, 299.99);

    UPDATE inventory SET stock_quantity = stock_quantity - 2
    WHERE product_id = 101;

COMMIT;  -- all three statements persist together

ACID Properties — Detailed Explanation

A — Atomicity

All statements in a transaction succeed together or none persist. Partial completion is impossible from the caller's perspective.

Banking Transfer Example: Debit $100 from Account A, credit $100 to Account B. If credit fails after debit, atomicity requires both operations roll back — no lost money.

BEGIN;
    UPDATE accounts SET balance = balance - 100 WHERE account_id = 'A';
    UPDATE accounts SET balance = balance + 100 WHERE account_id = 'B';
    -- If any statement fails:
ROLLBACK;  -- both updates undone

C — Consistency

Transaction moves database from one valid state to another, enforcing all constraints (PK, FK, CHECK, UNIQUE). Invalid data never commits.

BEGIN;
    INSERT INTO orders (customer_id, total_amount) VALUES (99999, -50);
    -- CHECK constraint total_amount > 0 fails → entire transaction rolls back
ROLLBACK;

I — Isolation

Concurrent transactions don't interfere with each other's intermediate state. Each transaction appears to run in isolation — though physical execution may be interleaved.

Scenario: User A reads inventory = 5. User B purchases 3 (inventory = 2). User A still sees 5 unless isolation level permits dirty reads.

D — Durability

Once COMMIT succeeds, changes survive system crash, power failure, or restart. Database writes to persistent storage (WAL/redo log) before acknowledging commit.

QA Scenario — Checkout Failure: Simulate payment gateway timeout after order insert. Verify: either order + payment + inventory update all exist, or none exist. Query for orders with status PENDING and no payment record — indicates atomicity violation. 
-- Detect partial checkout state (atomicity bug)
SELECT o.order_id, o.status, p.payment_id
FROM orders o
LEFT JOIN payments p ON o.order_id = p.order_id
WHERE o.status = 'PENDING'
  AND o.created_at >= CURRENT_DATE - INTERVAL '1 day'
  AND p.payment_id IS NULL
  AND o.total_amount > 0;

SAVEPOINT — Partial Rollback

BEGIN;
    INSERT INTO orders (customer_id, total_amount, status)
    VALUES (1001, 199.99, 'PENDING');
    SAVEPOINT after_order;

    INSERT INTO order_items (order_id, product_id, quantity, unit_price)
    VALUES (currval('orders_order_id_seq'), 101, 1, 199.99);
    SAVEPOINT after_items;

    UPDATE inventory SET stock_quantity = stock_quantity - 1
    WHERE product_id = 101;
    -- Inventory update fails (insufficient stock):
    ROLLBACK TO SAVEPOINT after_items;
    -- Order and items remain; only inventory update undone

COMMIT;

Isolation Levels — Comparison Table

Isolation LevelDirty ReadNon-Repeatable ReadPhantom ReadPerformance
READ UNCOMMITTEDPossiblePossiblePossibleFastest
READ COMMITTEDPreventedPossiblePossibleDefault in PostgreSQL, Oracle, SQL Server

Phenomena Explained

  • Dirty Read: Reading uncommitted data from another transaction that may rollback
  • Non-Repeatable Read: Same query in one transaction returns different rows because another transaction committed an update
  • Phantom Read: Same query returns different number of rows because another transaction inserted/deleted matching rows
-- Set isolation level (PostgreSQL)
BEGIN ISOLATION LEVEL REPEATABLE READ;
    SELECT stock_quantity FROM inventory WHERE product_id = 101;  -- returns 10
    -- Another session updates stock to 7 and commits
    SELECT stock_quantity FROM inventory WHERE product_id = 101;
    -- READ COMMITTED: returns 7 (non-repeatable read)
    -- REPEATABLE READ: returns 10 (snapshot)
COMMIT;

Deadlocks — Detection and Prevention

A deadlock occurs when Transaction A holds lock on Row 1 waiting for Row 2, while Transaction B holds lock on Row 2 waiting for Row 1. Database detects and aborts one transaction.

Transaction A:  LOCK orders(1001)  →  WAIT order_items(5001)
Transaction B:  LOCK order_items(5001)  →  WAIT orders(1001)
                     ↓
              DEADLOCK detected → one transaction ROLLBACK

-- PostgreSQL: view recent deadlocks in log
-- Enable: log_lock_waits = on, deadlock_timeout = 1s

-- SQL Server: system health session extended events
-- MySQL: SHOW ENGINE INNODB STATUS; (look for LATEST DETECTED DEADLOCK)

Prevention Strategies

  • Access tables/rows in consistent order across all transactions
  • Keep transactions short — minimize lock hold time
  • Use appropriate isolation level (don't use SERIALIZABLE unless required)
  • Implement retry logic in application for deadlock victims
  • Use row-level locking hints only when necessary

Interview Questions & Answers (12 Questions)

  1. What is a database transaction?
    A group of SQL operations treated as a single unit — all succeed (COMMIT) or all fail (ROLLBACK).
  2. Explain ACID properties.
    Atomicity (all or nothing), Consistency (valid state to valid state), Isolation (concurrent transactions don't interfere), Durability (committed data survives crashes).
  3. What is the difference between COMMIT and ROLLBACK?
    COMMIT permanently saves changes. ROLLBACK undoes all changes since BEGIN.
  4. What is a SAVEPOINT?
    Marker within a transaction allowing partial rollback to that point without undoing earlier statements in the same transaction.
  5. Explain isolation levels.
    READ UNCOMMITTED (least strict) through SERIALIZABLE (strictest). Higher isolation prevents more concurrency anomalies but reduces performance and increases deadlock risk.
  6. What is a dirty read?
    Reading uncommitted data from another transaction that may still rollback — possible only at READ UNCOMMITTED level.
  7. Non-repeatable read vs phantom read?
    Non-repeatable read: same row returns different values (updated by another txn). Phantom read: same query returns different number of rows (inserts/deletes by another txn).
  8. What is a deadlock?
    Circular wait — two transactions each hold a lock the other needs. DB kills one transaction; application should retry.
  9. How do you test transaction rollback?
    Begin transaction, insert/update data, force failure, rollback, verify data absent or unchanged. Automate with JDBC connection.setAutoCommit(false).
  10. How do you test concurrent inventory updates?
    Parallel API calls purchasing same product with stock = 1. Assert exactly one succeeds, inventory = 0, no negative stock.
  11. Default isolation level in PostgreSQL vs MySQL?
    PostgreSQL: READ COMMITTED. MySQL InnoDB: REPEATABLE READ.
  12. How would you detect a partial transaction bug in production?
    Query for orphaned/inconsistent states: orders without payments, payments without orders, inventory not decremented after completed orders. Monitor counts after deployment.
