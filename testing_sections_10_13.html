Interview Questions & Answers (Section 10)

Q1: Explain the Jira issue hierarchy from Epic to Sub-task.

A: Epic is the largest container for a feature initiative. Stories (or User Stories) deliver user value and link to Epics. Tasks break stories into dev/QA work. Sub-tasks are the smallest units assigned to individuals. Bugs can link to Stories independently. In Jira Cloud, Epics can also be parents in next-gen projects.

Q2: What is the difference between Priority and Severity in Jira bugs?

A: Severity measures technical impact (Blocker = system down). Priority measures business urgency for fix order (P1 = fix immediately). A cosmetic UI bug might be Low Severity but High Priority if a key client demo is tomorrow.

Q3: How do you use Fix Versions in release management?

A: Fix Version tags which release includes a fix or feature. Before release, I run JQL to find open issues in that version. After deployment, I mark the version Released and generate release notes from resolved issues.

Q4: Describe a Jira workflow you configured or improved for QA.

A: I added a "Ready for QA" status between Code Review and In Test. Automation notifies QA on transition. Stories cannot move to Done without passing through Verified. This prevents devs from closing stories before QA sign-off.

Q5: What JQL query finds all open bugs blocking a release?

A: project = PROJ AND type = Bug AND fixVersion = "v2.4.0" AND status not in (Closed, Done, Rejected). Add AND priority in (P1, P2) for critical blockers only.

Q6: How do WIP limits help QA teams?

A: WIP limits on "In Test" column (e.g., 3) prevent dev from flooding QA with stories. When limit is reached, team swarms on testing before pulling new dev work — improving flow and reducing context switching.

Q7: Explain sprint velocity and how it guides planning.

A: Velocity is completed story points per sprint, averaged over 3+ sprints. If average is 40 points, we commit ~40 points next sprint. It's a planning guide, not a performance metric — teams shouldn't inflate estimates to increase velocity.

Q8: What happens to incomplete stories when a sprint closes?

A: Jira moves them back to the backlog (or next sprint if manually moved). Their story points don't count toward velocity. Sprint Report shows spillover — a signal for retrospective discussion on over-commitment or blockers.

Q9: How do you link a bug to the story it was found in?

A: Use "Linked Issues" → "relates to" or "is caused by" the Story. Also set the Epic Link to maintain traceability. Some teams use "Blocks" if the bug prevents story completion.

Q10: What automation would you set up for bug triage?

A: When Bug is created with Severity = Blocker, auto-assign to on-call dev, notify PO via Slack, and add label "needs-triage". When status stays Open > 24 hours, escalate to dev lead with email notification.

Q11: Difference between Scrum board and Kanban board in Jira?

A: Scrum boards are tied to Sprints with start/end dates and backlog grooming. Kanban boards are continuous flow without sprints, ideal for support/ops teams. Both support columns, WIP limits, and swimlanes.

Q12: How do Smart Commits integrate Git with Jira?

A: Commit messages like PROJ-123 #comment Fixed login #time 2h #resolve automatically update Jira issues — adding comments, logging time, and transitioning status. QA verifies these transitions match actual test results.

Interview Questions & Answers (Section 11)

Q1: Walk me through your typical day as QA using Jira.

A: I start by checking my "Ready for QA" filter and open bugs needing retest. I update the sprint board, execute tests linked to in-progress stories, log defects with full reproduction steps, and comment on stories with test progress. Q2: How do you ensure test coverage is traceable to requirements in Jira?

A: Each acceptance criterion maps to one or more test cases linked via Xray/Zephyr. I generate a traceability matrix report showing Story → Test Cases → Execution Result → Defects. Gaps (AC with no linked test) are flagged before testing starts.

Q3: How do you calculate and report defect density?

A: Defect Density = Total bugs found in sprint / Story points completed. If Sprint 12 had 8 bugs and 40 points, density = 0.2. I track trend over sprints — increasing density signals quality issues in new features or tech debt.

Q4: What JQL do you use daily as QA?

A: Key queries: (1) status = "Ready for QA" AND sprint in openSprints(), (2) type = Bug AND status = Fixed AND reporter = currentUser(), (3) sprint in openSprints() AND assignee = currentUser() AND status != Done.

Q5: How do you handle a bug that cannot be reproduced?

A: Add detailed logs, video recording, HAR file. Tag "intermittent" label. If dev cannot reproduce, move to Cannot Reproduce with comment documenting attempts. Reopen if seen again with additional evidence. Never close without team agreement.

Q6: Compare Zephyr Scale vs Xray for Jira test management.

A: Both are native Jira apps. Xray uses Jira issue types (Test, Test Plan) — good for teams wanting everything as Jira issues. Zephyr Scale has a dedicated test repository with versioning — better for large enterprise test libraries. Both support traceability and CI integration; choice depends on team size and BDD needs.

Q7: How do you track defect leakage in Jira?

A: Production bugs get label "prod-bug" and Component matching the feature. Leakage Rate = prod-bugs / (QA bugs + prod-bugs) × 100. Root cause analysis on each prod bug: missed AC, missing test case, environment gap, or regression.

Q8: What should a good bug report contain?

A: Clear summary, numbered reproduction steps, expected vs actual result, environment details (OS, browser, build), test data used, screenshots/logs attached, severity/priority, linked story, and Fix Version. A developer should reproduce without asking me questions.

Q9: How do you use Jira dashboards to communicate QA status?

A: My dashboard shows: open bugs by priority (pie), test execution progress (bar), bugs created vs resolved trend (line), and stories in "In Test" (filter results). I share the dashboard link in sprint reviews instead of building slides manually.

Q10: How do you manage test cases for regression in Jira?

A: Tag regression test cases with label "regression" and organize into a Test Set/Cycle per release. Each sprint, clone the cycle, execute against the staging build, and track pass/fail. Failed tests either log new bugs or reopen existing ones.

Q11: What is your approach to requirement clarification in Jira?

A: During Days 1-2, I review every story's AC and post structured questions as comments. I identify edge cases (empty input, max length, concurrent users) and propose additional AC if missing. I don't start test case writing until critical questions are resolved.

Q12: How do you prevent duplicate bug reports?

A: Before creating a bug, I search JQL: text ~ "OTP expired" AND project = PROJ AND status != Closed. If found, I add a comment with my additional evidence and link as "duplicates" rather than creating a new issue.

Automation ROI Calculation

ROI = (Manual Execution Cost × Runs per Year − Automation Maintenance Cost) / Automation Investment

Example: Regression suite = 40 manual hours × $50/hr = $2,000 per run × 26 sprints = $52,000/year manual cost.
Automation investment = 200 hours × $50 = $10,000. Maintenance = 40 hours/year × $50 = $2,000/year.
Automated run cost = 2 hours × $50 × 26 = $2,600/year.
ROI = ($52,000 − $2,600 − $2,000) / $10,000 = 4.74× return in Year 1.

Interview Questions & Answers (Section 12)

Q1: What is Agile Testing and how is it different from traditional testing?

A: Agile Testing is continuous and iterative — testing happens throughout the sprint alongside development, not as a separate phase after coding. Testers collaborate daily with dev and PO, adapt to changing requirements, and automate regression to keep pace with frequent releases.

Q2: Explain the four Agile Testing Quadrants.

A: Q1 (unit/component, tech, support team), Q2 (functional/story, business, support team), Q3 (exploratory/UAT, business, critique product), Q4 (performance/security, tech, critique product). All four quadrants are needed — not just Q2 functional testing.

Q3: What is Shift Left testing? Give a practical example.

A: Shift Left means testing earlier in the lifecycle. Example: During refinement, I review acceptance criteria and identify that "password must be strong" is vague. I ask PO to define rules before dev starts. I write API tests against the dev branch on Day 3 — finding a validation bug before UI is even built.

Q4: What is Shift Right testing?

A: Shift Right extends testing into production — monitoring real user behavior, canary deployments, feature flags, synthetic monitoring, and chaos engineering. It catches issues that lab environments miss: real traffic patterns, data edge cases, and infrastructure failures.

Q5: Describe QA gates in a CI/CD pipeline.

A: Each pipeline stage has pass/fail criteria that block progression. Unit tests must pass with ≥80% coverage before merge. Integration tests gate deploy to QA. Functional smoke/regression gates release candidate. Performance and security scans gate production deploy. Failures notify QA and dev via Slack/Jira.

Q6: Explain the Test Automation Pyramid.

A: Base = many fast unit tests (60-70%). Middle = integration/API tests (20-30%). Top = few E2E UI tests (5-10%). The pyramid shape reflects speed, stability, and cost — push tests as low as possible. E2E tests are slow and brittle; rely on them sparingly.

Q7: What tests should you NOT automate?

A: Exploratory tests, one-time investigations, frequently changing UI features, tests requiring human judgment (usability, visual design), unstable/flaky scenarios, and tests where automation maintenance exceeds manual execution cost.

Q8: How do you calculate automation ROI?

A: ROI = (Manual cost per year − Automated run cost − Maintenance cost) / Initial investment. If manual regression costs $52K/year and automation costs $10K to build + $4.6K/year to run/maintain, ROI = 3.4× in Year 1. Present this to justify automation sprint capacity.

Q9: What is BDD and how does Gherkin help?

A: BDD uses natural language (Gherkin: Given/When/Then) to define behavior collaboratively with PO, dev, and QA. Feature files become living documentation and automated test scripts via Cucumber step definitions. It bridges communication gaps between technical and business teams.

Q10: Compare TDD, BDD, and ATDD.

A: TDD is developer-driven unit testing before code (Red-Green-Refactor). BDD specifies behavior in Gherkin for the whole team. ATDD defines acceptance tests from AC before development starts — typically in "Three Amigos" sessions. They complement each other: TDD for code quality, ATDD for requirements, BDD for communication.

Q11: What is your Definition of Done for testing?

A: All AC verified, automated tests pass in CI, no open P1/P2 bugs, test cases executed and linked in Xray, regression ≥95% pass, cross-browser tested, and test data cleaned up. DoD is team-agreed and displayed in Confluence — not QA-only criteria.

Q12: How do you balance manual and automated testing in a sprint?

A: Automate regression and smoke tests for stable features. Manual effort focuses on new story testing (first-time validation), exploratory testing (Q3), and complex scenarios hard to automate. Each sprint I allocate ~60% execution on automation, ~40% on manual/exploratory for new features.

Q13: How does continuous testing differ from test automation?

A: Test automation is writing scripts to execute tests. Continuous testing is the practice of running the right tests automatically at every pipeline stage with quality gates. It includes automation plus CI integration, test selection, environment management, and result-driven deploy decisions.

Q14: What testing activities happen during backlog refinement?

A: QA reviews stories for testability, identifies missing AC, proposes edge cases, estimates testing effort, and flags dependencies. I ask: "How do we test this? What test data is needed? Are there NFR requirements?" This is Shift Left in action.

Q15: How do you handle testing when requirements change mid-sprint?

A: Assess impact on existing test cases and automation scripts. Update test cases in Jira/Xray. Communicate scope change to SM/PO. Reprioritize — drop lower-priority tests if needed. Update AC comments with agreed changes. Never silently test against old requirements.

Q16: What role does QA play in CI/CD?

A: QA defines quality gates, maintains automated test suites in the pipeline, triages test failures (bug vs flaky test), manages test environments and data, reviews deployment checklists, and performs smoke tests post-deploy. QA is a pipeline stakeholder, not just a consumer of builds.
