Section 14: Interview Preparation

Comprehensive Q&A bank for QA Automation Engineers (4+ years) targeting Senior QA / SDET roles. Every question includes a detailed answer and what the interviewer expects.

Software Testing — Top 100 Questions

Beginner: Testing Basics (Q1–Q10)

Q1. What is software testing?

Answer: Software testing is the process of evaluating a software product to determine whether it meets specified requirements and to identify defects before release. It involves executing the application under controlled conditions, comparing actual vs expected behavior, and reporting discrepancies. Testing provides stakeholders with information about quality, risk, and readiness for production. It is not just finding bugs—it validates that the system delivers intended business value safely and reliably.

Interviewer expects: A definition beyond "finding bugs"—show you understand testing as risk mitigation and quality assurance.

Q2. Why is software testing necessary?

Answer: Testing reduces the cost of defects by catching issues early when fixes are cheaper. It prevents production failures that damage reputation, revenue, and user trust. Testing validates compliance with requirements, regulations, and SLAs. It also builds confidence for releases, especially in complex systems with integrations. Without testing, teams rely on hope rather than evidence when shipping software.

Interviewer expects: Business impact reasoning—cost, risk, compliance—not just "to find bugs."

Q3. What is the difference between QA and QC?

Answer: Quality Assurance (QA) is process-oriented—it focuses on preventing defects by improving development processes, standards, and practices across the SDLC. Quality Control (QC) is product-oriented—it focuses on identifying defects in the actual product through inspection and testing. QA is proactive; QC is reactive. In practice, QA engineers often do both: they define processes and execute tests.

Interviewer expects: Clear distinction: QA = process/prevention, QC = product/detection.

Q4. What is SDLC?

Answer: Software Development Life Cycle (SDLC) is a structured framework describing phases of software development from inception to retirement. Common phases include requirements, design, development, testing, deployment, and maintenance. SDLC models (Waterfall, Agile, Spiral, etc.) define how these phases are sequenced and iterated. Testing activities are embedded differently depending on the model chosen.

Interviewer expects: Know SDLC phases and that testing is integrated throughout, not a single phase.

Q5. What is STLC?

Answer: Software Testing Life Cycle (STLC) defines the sequence of testing activities: requirement analysis, test planning, test case development, environment setup, test execution, and test cycle closure. Each phase has entry/exit criteria and deliverables (test plan, test cases, RTM, defect reports). STLC runs within SDLC and ensures testing is systematic rather than ad hoc.

Interviewer expects: List STLC phases with deliverables—shows structured testing knowledge.

Q6. What are entry and exit criteria in testing?

Answer: Entry criteria are conditions that must be met before testing can begin (e.g., requirements signed off, build deployed, test environment ready). Exit criteria define when testing can stop (e.g., all P1/P2 test cases passed, defect closure rate met, coverage targets achieved). They prevent premature testing and premature release. Both should be documented in the test plan and agreed with stakeholders.

Interviewer expects: Concrete examples of entry/exit criteria, not vague definitions.

Q7. What is a test plan and what does it contain?

Answer: A test plan is a master document outlining the scope, approach, resources, schedule, and deliverables for a testing effort. Key sections include test objectives, scope (in/out), test strategy, environment requirements, roles, schedule, risks, and entry/exit criteria. It serves as a contract between QA and stakeholders on what will be tested and how. For agile teams, it may be lighter but still exists as a sprint or release test strategy.

Interviewer expects: Mention key sections and link to stakeholder alignment.

Q8. What is a test case?

Answer: A test case is a documented set of preconditions, steps, test data, expected results, and postconditions used to verify a specific requirement or scenario. Good test cases are atomic, repeatable, traceable, and independent. They should include positive, negative, and edge scenarios. Test cases form the basis for manual execution and automation scripts.

Interviewer expects: Know test case components and qualities (traceable, repeatable, independent).

Q9. What is a Requirement Traceability Matrix (RTM)?

Answer: RTM is a document mapping requirements to test cases, ensuring every requirement has test coverage and every test traces back to a requirement. It helps identify gaps (untested requirements) and orphan tests. RTM is updated throughout the project and is critical for audit, compliance, and release sign-off. In agile, traceability may live in Jira links rather than a spreadsheet.

Interviewer expects: Explain bidirectional traceability and gap identification.

Q10. What is the difference between test scenario and test case?

Answer: A test scenario is a high-level description of what to test (e.g., "Verify user login with valid credentials"). A test case is the detailed, step-by-step executable document with specific data and expected results. One scenario may spawn multiple test cases (valid login, invalid password, locked account, etc.). Scenarios help brainstorming; test cases drive execution.

Interviewer expects: Hierarchy understanding—scenario is abstract, test case is concrete.

Beginner: Testing Types (Q11–Q20)

Q11. What is functional testing?

Answer: Functional testing validates that the software behaves according to specified functional requirements—what the system should do. It tests features, user workflows, API contracts, and business rules. Techniques include equivalence partitioning, boundary value analysis, and decision tables. Functional testing can be manual or automated and occurs at unit, integration, system, and acceptance levels.

Interviewer expects: Focus on "what the system does" vs non-functional "how well."

Q12. What is non-functional testing?

Answer: Non-functional testing evaluates attributes like performance, security, usability, reliability, scalability, and compatibility. It answers "how well" the system performs under various conditions rather than "what" it does. Examples include load testing, stress testing, accessibility testing, and penetration testing. NFRs are often overlooked but critical for production readiness.

Interviewer expects: Name specific NFR types and when they matter in release decisions.

Q13. What is smoke testing?

Answer: Smoke testing is a shallow, broad check of critical functionality on a new build to determine if it is stable enough for further testing. It is quick (often 15–30 minutes) and covers core paths like login, navigation, and primary workflows. If smoke fails, the build is rejected ("build is broken"). It is typically automated in CI pipelines as a gate before deeper test suites run.

Interviewer expects: Emphasize build acceptance gate—not deep testing.

Q14. What is sanity testing?

Answer: Sanity testing is a narrow, deep check focused on specific changed functionality after a bug fix or small change. It verifies the fix works and that related areas weren't broken. Unlike smoke, sanity is not build-wide—it targets the impacted module. It is often unscripted and performed by the developer or tester familiar with the change.

Interviewer expects: Contrast with smoke: narrow/deep vs broad/shallow.

Q15. What is regression testing?

Answer: Regression testing re-runs previously passed test cases to ensure new changes haven't broken existing functionality. It is essential after bug fixes, feature additions, or refactoring. Regression suites grow over time and are prime candidates for automation. Risk-based regression selects subsets based on change impact rather than running everything every time.

Interviewer expects: Mention automation, suite maintenance, and risk-based selection.

Q16. What is retesting?

Answer: Retesting executes a specific test case again after a reported defect has been fixed to confirm the fix works. It is targeted—only the failed test(s) related to the bug are re-run. Retesting is distinct from regression: retesting validates the fix; regression checks for side effects elsewhere. Both are often needed after a bug fix.

Interviewer expects: Clear retesting vs regression distinction with examples.

Q17. What is integration testing?

Answer: Integration testing verifies interactions between combined modules, services, or systems. Approaches include big-bang (all at once), top-down, bottom-up, and sandwich/hybrid. API testing is a common form of integration testing. It catches interface mismatches, data format issues, and contract violations that unit tests miss.

Interviewer expects: Know integration approaches and API testing as a practical example.

Q18. What is system testing?

Answer: System testing evaluates the complete, integrated application against specified requirements in an environment close to production. It covers functional, non-functional, and end-to-end workflows. System testing is typically performed by a dedicated QA team, not developers. It validates the system as a whole before UAT and release.

Interviewer expects: Whole-system validation in near-production environment.

Q19. What is UAT?

Answer: User Acceptance Testing (UAT) is performed by end users or business representatives to confirm the system meets business needs and is ready for production. It uses real-world scenarios and data. UAT sign-off is often a formal release gate. QA supports UAT by providing test environments, training, and triaging defects found during UAT.

Interviewer expects: Business-user perspective and release sign-off role.

Q20. What is exploratory testing?

Answer: Exploratory testing is simultaneous learning, test design, and test execution without pre-written scripts. Testers use charters, heuristics, and domain knowledge to uncover defects. It is valuable for new features, complex UI, and areas with poor documentation. Session-based exploratory testing adds time-boxing and structured reporting.

Interviewer expects: Show it complements scripted testing—not a replacement.

Beginner: Test Case Design & Planning (Q21–Q30)

Q21. What are the components of a good test case?

Answer: A good test case includes: unique ID, title, preconditions, test steps, test data, expected result, actual result (during execution), status, and priority. It should be clear, atomic (one objective), repeatable, and traceable to a requirement. Avoid ambiguous language and combine only logically related steps. Peer review improves test case quality.

Interviewer expects: List components and quality attributes confidently.

Q22. What is positive and negative testing?

Answer: Positive testing verifies the system works correctly with valid inputs and expected user behavior. Negative testing verifies the system handles invalid inputs, error conditions, and edge cases gracefully—showing proper error messages and not crashing. Both are essential; negative testing often reveals more critical defects. Security testing heavily relies on negative scenarios.

Interviewer expects: Balance both types—don't only test happy paths.

Q23. What is boundary value analysis (BVA)?

Answer: BVA tests values at the edges of input domains because defects cluster at boundaries. For a field accepting 1–100, test 0, 1, 2, 99, 100, 101. BVA complements equivalence partitioning by focusing on min, min+1, max-1, max, and just-outside values. It is efficient—few tests cover high-risk areas.

Interviewer expects: Give a numeric example with boundary values listed.

Q24. What is equivalence partitioning (EP)?

Answer: EP divides input data into partitions (classes) where all values in a class are expected to behave similarly. One representative value per partition is tested instead of every value. Valid and invalid partitions are identified separately. EP reduces test count while maintaining coverage. It works best for numeric ranges, dropdowns, and categorical inputs.

Interviewer expects: Explain partitions with a concrete example.

Q25. What is a decision table?

Answer: A decision table maps combinations of conditions to actions/outcomes in a tabular format. It is ideal for complex business rules with multiple inputs affecting outcomes (e.g., discount eligibility based on membership, cart value, and promo code). Decision tables ensure no combination is missed and make logic explicit for testers and developers.

Interviewer expects: Know when to use it—complex conditional logic scenarios.

Q26. What is test data and why is it important?

Answer: Test data is the input values used during test execution—user accounts, transactions, configurations, etc. Realistic, isolated, and refreshable test data is critical for reliable results. Poor test data causes false failures and blocks automation. Strategies include synthetic data generation, data masking, sub-setting production data, and dedicated test data management tools.

Interviewer expects: Mention data isolation, refresh, and automation impact.

Q27. What is a test environment?

Answer: A test environment is the hardware, software, network, and data configuration where testing occurs. It should mirror production as closely as feasible (OS, DB version, integrations, configs). Environment instability is a top cause of flaky tests and blocked testing. Environment management includes provisioning, configuration control, and access management.

Interviewer expects: Production parity and stability concerns.

Q28. What is defect reporting?

Answer: Defect reporting documents a discrepancy between expected and actual behavior with enough detail for developers to reproduce and fix. A good bug report includes: title, steps to reproduce, expected vs actual, environment, severity, priority, screenshots/logs, and build version. Clear, reproducible reports reduce back-and-forth and speed resolution.

Interviewer expects: Describe a well-written bug report structure.

Q29. What is test coverage?

Answer: Test coverage measures the extent to which requirements, code, or risk areas have been tested. Types include requirements coverage (RTM), code coverage (line, branch, path), and risk coverage. 100% coverage is rarely achievable or cost-effective; the goal is meaningful coverage of critical paths. Coverage metrics inform release decisions but don't guarantee quality alone.

Interviewer expects: Distinguish coverage types and note coverage ≠ quality.

Q30. What is the difference between verification and validation?

Answer: Verification asks "Are we building the product right?"—checking against specs through reviews, walkthroughs, and static analysis without executing code. Validation asks "Are we building the right product?"—confirming the product meets user needs through dynamic testing and UAT. Both are essential: verification catches spec issues early; validation confirms real-world fitness.

Interviewer expects: Classic "building it right vs building the right thing" answer.

Intermediate: Defect Management (Q31–Q45)

Q31. What is a defect lifecycle?

Answer: Defect lifecycle tracks a bug from discovery to closure: New → Assigned → Open/In Progress → Fixed → Retest → Verified/Closed (or Reopened if fix fails). Some organizations add Deferred, Duplicate, Won't Fix, or Cannot Reproduce states. Each transition should have clear ownership and criteria. Lifecycle visibility helps measure defect aging and team responsiveness.

Interviewer expects: Walk through states in order with transitions.

Q32. What is severity?

Answer: Severity measures the technical impact of a defect on the system—how badly it affects functionality. Levels typically include Critical (system crash, data loss), Major (major feature broken, no workaround), Minor (feature impaired, workaround exists), and Trivial (cosmetic). Severity is usually set by QA based on impact analysis. It remains relatively stable regardless of release timing.

Interviewer expects: Technical impact focus—set by tester, not business urgency.

Q33. What is priority?

Answer: Priority indicates how urgently a defect must be fixed from a business perspective. Levels include P1 (fix immediately), P2 (fix before release), P3 (fix in next release), P4 (fix when possible). Priority is set by product owner or project manager based on business impact, release deadlines, and customer exposure. A cosmetic bug on the homepage may be High priority but Low severity.

Interviewer expects: Business urgency—can differ from severity.

Q34. Give an example where severity and priority differ.

Answer: A typo in the company logo on the login page: Low severity (cosmetic, no functional impact) but High priority if a major product launch is tomorrow and executives will see it. Conversely, a backend API timeout under extreme load: High severity (service degradation) but Low priority if it only occurs at 10x normal traffic and the release is internal-only this sprint.

Interviewer expects: Concrete scenario showing both dimensions independently.

Q35. What is defect triage?

Answer: Defect triage is a regular meeting where QA, dev, and product review new/open defects to assign priority, severity, ownership, and target fix version. It prevents defect backlog from growing unmanaged. Triage decisions consider release impact, customer reports, and fix effort. Effective triage requires reproducible bug reports and stakeholder participation.

Interviewer expects: Cross-functional meeting with prioritization outcomes.

Q36. What is a blocked defect?

Answer: A blocked defect cannot be verified or progressed because an external dependency prevents testing—e.g., missing test data, environment down, or another bug blocking access to the feature. Blocked defects should be flagged with reason and owner to unblock. Tracking blocked defects highlights infrastructure and process bottlenecks.

Interviewer expects: Distinguish blocked from open—dependency awareness.

Q37. What is defect leakage?

Answer: Defect leakage is the percentage of defects found by customers or in production vs total defects (production + pre-release). Formula: (Defects found in production / Total defects) × 100. High leakage indicates gaps in test coverage, environment parity, or release criteria. It is a key quality metric for QA effectiveness.

Interviewer expects: Know the formula and what high leakage implies.

Q38. What is defect density?

Answer: Defect density measures defects per unit size of software—commonly defects per KLOC (thousand lines of code) or per story point. Formula: Total defects / Size of module. It helps compare quality across modules and predict testing effort. High density modules deserve more testing focus and code review.

Interviewer expects: Formula and practical use for risk targeting.

Q39. What is root cause analysis (RCA) in testing?

Answer: RCA investigates why a defect occurred and escaped to production—not just fixing the symptom. Techniques include 5 Whys, fishbone diagrams, and defect categorization (requirements gap, coding error, environment, test gap). RCA drives process improvements: better reviews, new test cases, or pipeline gates. Recurring defects signal systemic issues.

Interviewer expects: Process improvement mindset beyond bug fixing.

Q40. How do you handle a non-reproducible bug?

Answer: Document everything: environment, browser, timestamps, logs, screenshots, and user actions. Check for race conditions, timing issues, and environment-specific configs. Attempt reproduction on same build/environment. Add logging or monitoring if needed. Mark as "Cannot Reproduce" with evidence, but keep open if customer-reported. Intermittent bugs often indicate concurrency or infrastructure issues.

Interviewer expects: Systematic approach—not just closing as CNR.

Q41. What is a showstopper defect?

Answer: A showstopper (P1/Critical) prevents core functionality, causes data corruption, or makes the system unusable with no workaround. It blocks release until fixed. Examples: payment failure, login broken, data breach vulnerability. Release criteria typically mandate zero open showstoppers. Smoke test failures often reveal showstoppers early.

Interviewer expects: Release-blocking criteria with examples.

Q42. What is defect aging?

Answer: Defect aging tracks how long defects remain open from creation to closure. High aging indicates bottlenecks in dev capacity, unclear priorities, or complex fixes. SLA targets (e.g., P1 within 24 hours) help manage aging. Dashboards showing aging trends drive accountability and process improvements.

Interviewer expects: Metric awareness and SLA management.

Q43. What is the difference between defect and enhancement?

Answer: A defect is behavior that deviates from agreed requirements or specifications. An enhancement (improvement) is a request for new or changed functionality not originally specified. Misclassified enhancements clog defect backlogs. Clarify with product whether behavior matches spec (bug) or spec is insufficient (enhancement/story).

Interviewer expects: Requirements baseline as the deciding factor.

Q44. How do you prioritize testing when time is limited?

Answer: Apply risk-based testing: prioritize by business impact, likelihood of failure, and change frequency. Test critical user journeys, new/changed code, and historically defect-prone areas first. Defer low-risk, stable modules. Communicate coverage gaps and residual risk to stakeholders for informed release decisions. Never silently skip high-risk areas.

Interviewer expects: Risk-based approach with stakeholder communication.

Q45. What metrics do you track for defect management?

Answer: Key metrics: defect discovery rate, defect removal efficiency, open defect count by severity, defect aging, defect leakage, reopen rate, and defects by module/root cause. Trend analysis matters more than point-in-time numbers. Metrics should drive action, not blame. Pair quantitative metrics with qualitative release readiness assessment.

Interviewer expects: Name 4–5 metrics and explain why trends matter.

Intermediate: Test Design Techniques (Q46–Q55)

Q46. When do you use equivalence partitioning vs BVA?

Answer: Use EP first to identify valid and invalid input classes and select one representative per class. Use BVA on partitions with defined boundaries (numeric ranges, length limits) to test edge values. They are complementary: EP reduces test count; BVA targets high-defect boundary zones. Apply both together for input fields with clear ranges.

Interviewer expects: Complementary use—not either/or.

Q47. What is state transition testing?

Answer: State transition testing models system behavior as states and events that trigger transitions (e.g., order: New → Paid → Shipped → Delivered). Test valid transitions, invalid transitions, and boundary states. State diagrams guide test case design for workflows, authentication sessions, and protocol-driven systems.

Interviewer expects: Valid vs invalid transitions with workflow example.

Q48. What is pairwise testing?

Answer: Pairwise (all-pairs) testing covers all pairs of input parameter combinations using orthogonal arrays, dramatically reducing test count vs exhaustive combinations. Effective when parameters interact but full combinatorial testing is impractical. Tools generate pairwise test sets automatically. Common for configuration testing (browser × OS × locale).

Interviewer expects: Combinatorial reduction strategy awareness.

Q49. What is error guessing?

Answer: Error guessing leverages tester experience and intuition to anticipate where defects are likely—null inputs, division by zero, max-length strings, special characters, concurrent access. It is not systematic but highly effective when combined with formal techniques. Senior testers excel at error guessing based on past defect patterns.

Interviewer expects: Experience-based technique complementing formal methods.

Q50. What is use case testing?

Answer: Use case testing derives test cases from use case documents describing actor interactions with the system. Each use case flow (main, alternate, exception) becomes test scenarios. It ensures testing aligns with user goals and business processes. Use cases bridge requirements and test cases in use-case-driven development.

Interviewer expects: Main/alternate/exception flows coverage.

Q51. How do you test an API without UI?

Answer: Use tools like Postman, REST Assured, or HttpClient to send requests and validate responses. Test HTTP methods, status codes, response body schema, headers, authentication, error handling, and edge cases. Contract testing (Pact) validates consumer-provider agreements. API tests are faster, more stable, and ideal for CI integration.

Interviewer expects: Practical API testing approach with tools.

Q52. What is contract testing?

Answer: Contract testing verifies that services meet agreed API contracts between consumer and provider without full integration environments. Consumer tests define expected requests/responses; provider verifies it satisfies those contracts. Tools like Pact enable shift-left integration validation. It reduces brittle end-to-end tests and catches breaking changes early.

Interviewer expects: Consumer-provider model and tools like Pact.

Q53. What is risk-based testing?

Answer: Risk-based testing prioritizes testing effort based on risk = probability of failure × impact of failure. High-risk areas get more thorough testing; low-risk areas get lighter coverage. Risk assessment considers new code, complexity, past defects, and business criticality. It optimizes limited testing time for maximum risk reduction.

Interviewer expects: Risk formula and practical prioritization.

Q54. What is orthogonal array testing?

Answer: Orthogonal arrays are mathematical constructs enabling balanced coverage of factor combinations with minimal test cases. Used in pairwise and robustness testing for systems with many configuration parameters. Standard arrays (L9, L16) provide proven coverage levels. Reduces test effort while maintaining statistical coverage of interactions.

Interviewer expects: Statistical combinatorial reduction knowledge.

Q55. How do you design tests for a login feature?

Answer: Cover: valid login, invalid password, invalid username, empty fields, SQL injection, XSS in inputs, account lockout after N attempts, session timeout, remember-me, password reset flow, concurrent sessions, and API-level auth token validation. Apply EP/BVA for password length rules. Include negative security tests and performance under load.

Interviewer expects: Comprehensive scenario list including security.

Intermediate: Integration, System, E2E, UAT (Q56–Q65)

Q56. What are the levels of testing?

Answer: Four main levels: Unit testing (individual components, dev-owned), Integration testing (module interactions), System testing (complete application), and Acceptance testing (business validation). Each level has different scope, ownership, and environment requirements. The test pyramid recommends many unit tests, fewer integration, and minimal E2E tests.

Interviewer expects: Four levels with scope and ownership.

Q57. What is end-to-end (E2E) testing?

Answer: E2E testing validates complete user workflows across all system layers—from UI through backend to database and external integrations. It simulates real user journeys (e.g., browse → cart → checkout → payment). E2E tests are valuable but slow, brittle, and expensive to maintain. Limit E2E to critical business paths; prefer lower-level tests for coverage.

Interviewer expects: Full-stack user journey with pyramid caution.

Q58. What is the difference between integration and system testing?

Answer: Integration testing focuses on interfaces between modules or services—does data flow correctly between components? System testing evaluates the entire integrated application against requirements as a complete product. Integration may use stubs/mocks; system testing uses the full stack. Integration catches interface bugs; system catches holistic behavior issues.

Interviewer expects: Interface focus vs whole-system focus.

Q59. What is alpha and beta testing?

Answer: Alpha testing is done internally by employees in a controlled environment before external release—QA and internal users. Beta testing releases the product to a limited external user group to gather real-world feedback before general availability. Both are forms of acceptance testing. Beta feedback drives final fixes and UX improvements.

Interviewer expects: Internal vs external acceptance testing distinction.

Q60. What is compatibility testing?

Answer: Compatibility testing verifies the application works across browsers, OS versions, devices, screen resolutions, and network conditions. Use browser matrices (Chrome, Firefox, Safari, Edge) and device labs or cloud platforms (BrowserStack, Sauce Labs). Responsive design testing ensures UI adapts correctly. Critical for web and mobile applications.

Interviewer expects: Cross-browser/device matrix approach.

Q61. What is localization and internationalization testing?

Answer: Internationalization (i18n) ensures the application supports multiple languages/regions without code changes—Unicode, date formats, RTL layouts. Localization (l10n) verifies translated content, currency, and locale-specific behavior display correctly. Test for text expansion, character encoding, timezone handling, and cultural appropriateness.

Interviewer expects: i18n vs l10n distinction with test considerations.

Q62. What is accessibility testing?

Answer: Accessibility testing ensures software is usable by people with disabilities—vision, hearing, motor, cognitive. Standards include WCAG 2.1 (levels A, AA, AAA). Test with screen readers, keyboard navigation, color contrast, ARIA labels, and tools like axe or Lighthouse. Accessibility is often a legal requirement (ADA, Section 508).

Interviewer expects: WCAG awareness and practical test methods.

Q63. What is mutation testing?

Answer: Mutation testing evaluates test suite quality by introducing small code changes (mutations) and checking if tests fail. If tests pass despite mutations, the suite has gaps. Tools like PIT (Java) automate mutation testing. High mutation score indicates strong unit tests; low score reveals ineffective tests that need improvement.

Interviewer expects: Test-the-tests concept with tool awareness.

Q64. How do you approach testing microservices?

Answer: Test each service in isolation (unit + API tests), then contract tests between services, then integration in staging, and finally selective E2E for critical journeys. Use service virtualization for dependencies. Test resilience: timeouts, circuit breakers, retries. Monitor distributed tracing (Jaeger, Zipkin) for failure diagnosis. Independent deployability requires independent testability.

Interviewer expects: Layered testing strategy for distributed architecture.

Q65. What is the role of QA in UAT?

Answer: QA prepares UAT environment and test data, trains business users, provides test scripts or charters, triages defects found during UAT, and verifies fixes. QA does not perform UAT itself—that is the business user's role. QA ensures UAT runs smoothly and defects are properly logged and prioritized. UAT sign-off is a key release gate.

Interviewer expects: Support role—not executing UAT.

Advanced: Agile Testing, Shift-Left, CI/CD, Test Pyramid (Q66–Q80)

Q66. What is agile testing?

Answer: Agile testing embeds testing throughout the sprint—not as a separate phase after development. Testers collaborate from story refinement, write acceptance criteria, automate during development, and provide continuous feedback. Testing adapts to changing requirements with each iteration. The whole team owns quality; QA is a quality coach, not a gatekeeper at the end.

Interviewer expects: Continuous collaboration—not phase-gate testing.

Q67. What is shift-left testing?

Answer: Shift-left moves testing activities earlier in the SDLC—requirements review, static analysis, unit tests, and API tests before UI/E2E. Catching defects in design or code costs far less than in production. Practices include TDD, BDD, code reviews, and CI pipeline gates. Shift-left reduces defect leakage and accelerates delivery.

Interviewer expects: Early detection philosophy with concrete practices.

Q68. What is shift-right testing?

Answer: Shift-right extends testing into production through monitoring, canary releases, A/B testing, chaos engineering, and synthetic monitoring. It validates real user behavior and catches issues impossible to reproduce in test environments. Shift-right complements shift-left—together they cover pre-release and post-release quality. Observability tools (Datadog, New Relic) enable shift-right.

Interviewer expects: Production monitoring and experimentation awareness.

Q69. What is the test pyramid?

Answer: The test pyramid (Mike Cohn) recommends many fast, cheap unit tests at the base; fewer integration/API tests in the middle; and minimal slow, expensive E2E/UI tests at the top. This balance optimizes feedback speed, maintenance cost, and coverage. An inverted pyramid (ice cream cone) with mostly UI tests leads to slow pipelines and flaky suites.

Interviewer expects: Draw or describe pyramid with ratio reasoning.

Q70. What is the testing quadrants model?

Answer: Brian Marick's quadrants map tests by technology-facing vs business-facing and supporting development vs critiquing product. Q1: unit/component (tech, support dev). Q2: functional/API (business, support dev). Q3: exploratory/UAT (business, critique). Q4: performance/security (tech, critique). All quadrants are needed for comprehensive quality.

Interviewer expects: Four quadrants with examples in each.

Q71. How does testing fit in CI/CD?

Answer: CI/CD pipelines run automated tests at every commit/build: unit → integration → smoke → regression → performance gates. Failed tests block deployment. Pipelines provide fast feedback (minutes, not days). QA defines which tests run at each stage, maintains test stability, and monitors pipeline health. CD requires high confidence from automated test suites.

Interviewer expects: Pipeline stages and quality gates.

Q72. What is continuous testing?

Answer: Continuous testing executes automated tests throughout the delivery pipeline whenever code changes—on commit, merge, deploy. It provides immediate feedback on quality risks. Requires reliable test infrastructure, stable tests, and parallel execution. Continuous testing is the enabler for continuous delivery and DevOps culture.

Interviewer expects: Automated feedback loop in pipeline context.

Q73. What is BDD?

Answer: Behavior-Driven Development uses Gherkin syntax (Given/When/Then) to write executable specifications that serve as living documentation and automated tests. Tools: Cucumber, SpecFlow, Behave. BDD bridges business, QA, and dev with shared language. Scenarios become acceptance criteria and automation scripts simultaneously.

Interviewer expects: Gherkin format and collaboration benefit.

Q74. What is TDD and how does it relate to testing?

Answer: Test-Driven Development: write a failing unit test first, write minimal code to pass, refactor. Red-Green-Refactor cycle. TDD is a development practice that produces unit tests as a byproduct. QA benefits from higher code testability and fewer integration defects. QA collaborates on acceptance tests; dev owns unit TDD.

Interviewer expects: Red-Green-Refactor and QA's collaborative role.

Q75. What is a Definition of Done (DoD)?

Answer: DoD is the team's agreed checklist for when a story is complete: code reviewed, unit tests pass, integration tests pass, documentation updated, deployed to staging, QA verified, no open defects. DoD ensures consistent quality standards across stories. QA input is critical—testing criteria must be in DoD.

Interviewer expects: Team agreement with testing criteria included.

Q76. What is a Definition of Ready (DoR)?

Answer: DoR defines when a story is ready to be pulled into a sprint: clear acceptance criteria, estimated, dependencies identified, testable, and sized appropriately. DoR prevents sprint disruption from unclear stories. QA validates testability during backlog refinement—if you can't write a test, the story isn't ready.

Interviewer expects: Testability as a readiness criterion.

Q77. How do you test in a DevOps culture?

Answer: QA integrates into cross-functional teams, automates tests in CI/CD, participates in infrastructure-as-code reviews, uses containerized test environments, monitors production, and practices shift-left and shift-right. "Quality is everyone's responsibility." QA skills expand to include pipeline management, observability, and release engineering collaboration.

Interviewer expects: DevOps mindset—not siloed QA team.

Q78. What is test automation in agile sprints?

Answer: Automation happens within the sprint alongside feature development—not after release. Automate acceptance criteria as stories complete. Maintain regression suite incrementally. Sprint capacity includes automation effort. Definition of Done includes automated test coverage for new features. Automate stable, repeatable tests; keep exploratory testing manual.

Interviewer expects: In-sprint automation—not big-bang after release.

Q79. What is the difference between CI and CD?

Answer: Continuous Integration (CI) merges code frequently with automated build and test on each integration. Continuous Delivery (CD) extends CI to automatically deploy to staging/production after all gates pass. Continuous Deployment deploys every passing build to production automatically. CI is the foundation; CD requires mature test automation and confidence.

Interviewer expects: CI → CD → Continuous Deployment progression.

Q80. How do you handle testing in micro-release cycles?

Answer: Rely on fast automated smoke and regression suites in CI. Use feature flags to deploy incomplete features safely. Implement canary/blue-green deployments with production monitoring. Risk-based test selection based on change impact analysis. Maintain strong contract tests between services. Manual exploratory testing focuses on new/changed areas only.

Interviewer expects: Feature flags, canary, and smart test selection.

Advanced: Test Automation Strategy & Framework Design (Q81–Q90)

Q81. What is a test automation framework?

Answer: A test automation framework is a structured set of guidelines, coding standards, libraries, and utilities that provide an environment for creating and executing automated tests efficiently. Components include driver management, page objects, reporting, config management, and test data handling. Good frameworks promote reusability, maintainability, and scalability across teams.

Interviewer expects: Structure and components—not just "a tool."

Q82. What is Page Object Model (POM)?

Answer: POM encapsulates page elements and actions in dedicated classes, separating test logic from UI locators. When UI changes, update one page class—not every test. POM improves maintainability and readability. Extended patterns include Page Factory, Screenplay pattern, and component-based models for complex SPAs.

Interviewer expects: Separation of concerns and maintenance benefit.

Q83. What tests should you automate?

Answer: Automate: repetitive regression tests, smoke/sanity suites, data-driven scenarios, API contract tests, and stable critical paths. Do NOT automate: one-time tests, exploratory tests, frequently changing UI, tests requiring human judgment (visual design), or tests cheaper to run manually. ROI drives automation decisions.

Interviewer expects: ROI-based selection criteria.

Q84. What is the Screenplay pattern?

Answer: Screenplay models tests as actors performing tasks and asking questions about the application. It separates who (Actor), what (Task/Action), and how (Interaction/Ability). More scalable than POM for complex workflows. Used in Serenity BDD. Promotes reusable, readable tests that read like business scenarios.

Interviewer expects: Actor-Task-Ability model awareness.

Q85. How do you handle flaky tests?

Answer: Identify root causes: timing issues (add explicit waits), test data conflicts (isolate data), environment instability (fix infra), order dependencies (ensure independence), and locators (use stable selectors). Quarantine flaky tests from CI until fixed. Track flakiness metrics. Never ignore flakes—they erode trust in the entire suite.

Interviewer expects: Root cause analysis—not retry hacks.

Q86. What is a hybrid automation framework?

Answer: Hybrid frameworks combine data-driven, keyword-driven, and modular approaches. Test data lives in external files (Excel, JSON), keywords map to reusable functions, and modules organize page objects/utilities. This flexibility suits teams with mixed skill levels—manual testers can write data/keyword tests while engineers maintain framework code.

Interviewer expects: Combined approach benefits for team scalability.

Q87. What is parallel test execution?

Answer: Running tests concurrently across threads, processes, or machines to reduce total execution time. Requires test isolation (independent data, no shared state). Frameworks like TestNG, pytest-xdist, and CI matrix builds support parallelism. Cloud grids (Selenium Grid, BrowserStack) enable cross-browser parallel runs. Essential for large regression suites in CI.

Interviewer expects: Isolation requirements and CI integration.

Q88. What is self-healing automation?

Answer: Self-healing uses AI/ML to automatically update broken locators when UI changes—reducing maintenance burden. Tools like Healenium, Testim, and Mabl offer this capability. It helps with dynamic UIs but shouldn't replace good locator strategies (data-testid, accessibility IDs). Use as a safety net, not primary strategy.

Interviewer expects: Balanced view—helpful but not a silver bullet.

Q89. How do you measure automation ROI?

Answer: ROI = (Manual execution cost × runs per year − automation cost) / automation investment. Track: execution time saved, defect detection rate, release frequency improvement, and team capacity freed. Also measure cost of maintenance (flaky tests, broken scripts). Present ROI to stakeholders for continued investment in automation infrastructure.

Interviewer expects: Quantifiable metrics—not just "saves time."

Q90. What is a test automation maturity model?

Answer: Maturity progresses: Level 1 (ad hoc scripts) → Level 2 (basic framework) → Level 3 (CI integrated, reporting) → Level 4 (parallel, data-driven, cross-browser) → Level 5 (self-service, metrics-driven, AI-assisted). Assess current level, identify gaps, and roadmap improvements. Maturity models guide investment and hiring decisions.

Interviewer expects: Progressive maturity stages with self-assessment.

Advanced: Risk-Based Testing, Metrics, Estimation (Q91–Q100)

Q91. How do you perform risk analysis for testing?

Answer: Identify features/modules, assess probability of failure (complexity, change frequency, past defects) and impact (revenue, users, compliance). Plot on risk matrix (High/Medium/Low). Allocate testing effort proportionally—more tests for high-risk areas. Reassess each sprint/release as code and requirements change. Document residual risk for release decisions.

Interviewer expects: Probability × impact matrix with actionable output.

Q92. What testing metrics matter to leadership?

Answer: Leadership cares about: defect leakage to production, mean time to detect/fix defects, test automation coverage of critical paths, release frequency and success rate, escaped defect cost, and customer-reported issues. Translate QA metrics into business language—risk reduced, revenue protected, faster time-to-market enabled by automation.

Interviewer expects: Business-aligned metrics—not vanity counts.

Q93. What is test estimation?

Answer: Test estimation predicts effort, time, and resources needed for testing activities. Techniques: work breakdown structure, three-point estimation (optimistic/likely/pessimistic), analogy (similar past projects), and percentage of dev effort (30–40% for thorough testing). In agile, estimate per story during planning. Include automation, environment setup, and regression in estimates.

Interviewer expects: Multiple techniques and agile story-level estimation.

Q94. What is velocity-based test planning in agile?

Answer: Use team's historical velocity (story points completed per sprint) to forecast how many stories can be tested per sprint. Track testing-specific capacity separately if QA is a bottleneck. If dev velocity exceeds QA capacity, advocate for more automation, shift-left, or additional QA resources. Balance prevents untested stories from accumulating.

Interviewer expects: Capacity planning awareness in agile context.

Q95. What is MTTR and MTTD in testing context?

Answer: Mean Time To Detect (MTTD) measures average time from defect introduction to discovery. Mean Time To Repair (MTTR) measures average time from discovery to fix deployment. Lower MTTD via better monitoring and shift-left; lower MTTR via efficient triage and CI/CD. Both are key DevOps/SRE quality metrics.

Interviewer expects: Definitions and improvement strategies.

Q96. How do you report test progress?

Answer: Daily: test execution status (passed/failed/blocked/not run). Sprint: burndown of test tasks, defect trends, automation coverage delta. Release: test summary report with coverage, open defects by severity, risk assessment, and go/no-go recommendation. Use dashboards (Jira, TestRail, Allure) for real-time visibility. Tailor report audience—technical for dev, summary for management.

Interviewer expects: Multi-level reporting for different stakeholders.

Q97. What is a test strategy document?

Answer: Test strategy is a high-level document defining testing approach for the project/product: scope, test levels, tools, environments, automation strategy, entry/exit criteria, risk approach, and team structure. It is stable across releases; test plans vary per release/sprint. Strategy aligns QA with organizational goals and architecture decisions.

Interviewer expects: Strategy vs plan distinction.

Q98. How do you test AI/ML features?

Answer: ML testing differs from traditional: test data quality and bias, model accuracy metrics (precision, recall, F1), drift detection in production, A/B testing of model versions, edge cases in input features, and explainability. Automate data validation pipelines. Monitor model performance post-deployment. Combine ML-specific testing with standard functional/API tests for ML-powered features.

Interviewer expects: Awareness of ML-specific quality dimensions.

Q99. What is chaos engineering and how does QA relate?

Answer: Chaos engineering intentionally injects failures (server crashes, network latency, dependency outages) in production/staging to verify system resilience. Tools: Chaos Monkey, Gremlin, Litmus. QA collaborates on defining steady-state hypotheses, designing experiments, and validating recovery behavior. It extends testing beyond functional correctness to operational resilience.

Interviewer expects: Resilience testing beyond functional scope.

Q100. How do you stay current as a senior QA/SDET?

Answer: Follow testing communities (Ministry of Testing, Test Automation University), contribute to open source, experiment with new tools (Playwright, k6, Pact), learn cloud/DevOps (AWS, Docker, K8s), understand architecture patterns (microservices, event-driven), and develop soft skills (mentoring, stakeholder communication). Certifications (ISTQB Advanced, AWS) validate breadth. Regularly review and refactor your own automation frameworks.

Interviewer expects: Continuous learning mindset with specific examples.

Agile — Top 50 Questions

Agile Fundamentals (Q1–Q20)

Q1. What is Agile?

Answer: Agile is an iterative approach to software development emphasizing collaboration, customer feedback, and small, frequent releases. It responds to change over following a fixed plan. Agile values working software, individuals and interactions, customer collaboration, and responding to change. Frameworks like Scrum, Kanban, and XP implement agile principles.

Interviewer expects: Mindset and values—not just "sprints."

Q2. What is the Agile Manifesto?

Answer: The Agile Manifesto (2001) declares four values: Individuals and interactions over processes and tools; Working software over comprehensive documentation; Customer collaboration over contract negotiation; Responding to change over following a plan. It also lists 12 supporting principles. The manifesto doesn't reject the right-side items—it prioritizes the left when trade-offs arise.

Interviewer expects: Recite four values accurately.

Q3. What are the 12 Agile principles?

Answer: Key principles include: satisfy customers through early delivery; welcome changing requirements; deliver working software frequently (weeks not months); business and dev collaborate daily; build projects around motivated individuals; face-to-face conversation; working software is primary progress measure; sustainable pace; technical excellence; simplicity; self-organizing teams; regular reflection and adjustment.

Interviewer expects: Know at least 5–6 principles with examples.

Q4. Agile vs Waterfall—key differences?

Answer: Waterfall is sequential (requirements → design → build → test → deploy) with fixed scope; Agile is iterative with evolving scope. Waterfall has late testing; Agile tests continuously. Waterfall suits stable, well-defined projects; Agile suits dynamic requirements and fast feedback needs. Waterfall documentation is heavy; Agile favors working software and light docs.

Interviewer expects: Structured comparison—not "Agile is better."

Q5. What is iterative development?

Answer: Iterative development builds software in repeated cycles (iterations/sprints), each producing a potentially shippable increment. Each iteration includes planning, development, testing, and review. Feedback from each iteration informs the next. This reduces risk by validating assumptions early and often rather than betting everything on a single big release.

Interviewer expects: Incremental delivery with feedback loops.

Q6. What is incremental delivery?

Answer: Incremental delivery releases functional pieces of the product over time, each adding value. Iteration 1 might ship login; iteration 2 adds search; iteration 3 adds checkout. Customers get usable features early rather than waiting for the complete product. Each increment is fully tested and deployable.

Interviewer expects: Value delivery in chunks—not partial/broken features.

Q7. What frameworks implement Agile?

Answer: Scrum (most popular—sprints, roles, ceremonies), Kanban (visual workflow, WIP limits), Extreme Programming/XP (TDD, pair programming), Lean Software Development, SAFe (scaled agile for enterprises), and LeSS (large-scale Scrum). Teams often blend practices. Choice depends on team size, organization culture, and project complexity.

Interviewer expects: Name 3+ frameworks with distinguishing features.

Q8. What is Kanban vs Scrum?

Answer: Scrum uses fixed-length sprints with defined roles and ceremonies. Kanban is continuous flow with WIP limits and no prescribed sprints or roles. Scrum plans sprint commitments; Kanban pulls work as capacity allows. Kanban suits support/maintenance teams; Scrum suits feature development teams. Both are agile—choose based on workflow needs.

Interviewer expects: Flow-based vs time-boxed distinction.

Q9. What is technical debt in agile?

Answer: Technical debt is the implied cost of shortcuts taken during development—skipped tests, poor design, missing documentation—that must be repaid later with interest (slower development, more bugs). Agile teams allocate sprint capacity (10–20%) for debt reduction. Ignoring debt eventually cripples velocity and quality.

Interviewer expects: Business metaphor understanding and sprint allocation.

Q10. What is a user story?

Answer: A user story is a lightweight requirement format: "As a [role], I want [feature], so that [benefit]." It includes acceptance criteria defining done. Stories are estimated, prioritized, and fit in one sprint. They encourage conversation over detailed specs. INVEST criteria ensure story quality (Independent, Negotiable, Valuable, Estimable, Small, Testable).

Interviewer expects: Standard format plus INVEST acronym.

Q11. What are acceptance criteria?

Answer: Acceptance criteria are conditions a story must meet to be accepted by the product owner. Written as bullet points or Given/When/Then scenarios. They define testable boundaries of the story. QA uses them to design test cases; dev uses them to know when done. Clear AC prevents scope creep and ambiguous "done."

Interviewer expects: Testable, specific, agreed before development.

Q12. What is story pointing?

Answer: Story points estimate relative effort/complexity using a Fibonacci-like scale (1, 2, 3, 5, 8, 13). Points consider complexity, uncertainty, and effort—not hours. Team velocity (points per sprint) emerges over time and aids forecasting. Planning poker ensures team consensus. Points are team-specific—not comparable across teams.

Interviewer expects: Relative estimation—not time-based hours.

Q13. What is velocity?

Answer: Velocity is the average story points completed per sprint over 3–5 sprints. It helps forecast how much work a team can take in future sprints. Velocity is a planning tool, not a performance metric—comparing velocities across teams is meaningless. Stable velocity indicates predictable delivery; declining velocity signals problems.

Interviewer expects: Planning tool—not productivity comparison.

Q14. What is a sprint?

Answer: A sprint is a fixed time-box (usually 2 weeks) during which the team completes a set of prioritized stories. Sprints have consistent duration. At the end, a potentially shippable increment is demonstrated. No scope changes mid-sprint (except critical defects). Sprint cadence creates rhythm and predictability.

Interviewer expects: Time-boxed iteration with shippable increment.

Q15. What is a product backlog?

Answer: The product backlog is an ordered list of all desired work—features, bugs, tech debt, spikes—maintained by the Product Owner. Highest priority items sit at the top. It is dynamic, constantly refined. The backlog is the single source of truth for what to build next. Items have descriptions, estimates, and acceptance criteria.

Interviewer expects: PO-owned, prioritized, living document.

Q16. What is backlog refinement (grooming)?

Answer: Backlog refinement is ongoing activity to prepare upcoming stories: clarify requirements, add acceptance criteria, estimate, split large stories, and identify dependencies. Typically 5–10% of sprint capacity. Well-refined backlog makes sprint planning efficient. QA ensures stories are testable during refinement.

Interviewer expects: Continuous preparation—not a one-time activity.

Q17. What is a spike in agile?

Answer: A spike is a time-boxed research/investigation story to reduce uncertainty before committing to implementation. Example: "Spike: evaluate API testing tools (3 days)." Output is knowledge (document, prototype, recommendation)—not production code. Spikes prevent teams from estimating blind on unknown technology.

Interviewer expects: Research time-box with concrete deliverable.

Q18. What is WIP limit?

Answer: Work-In-Progress limits cap how many items can be in a given workflow stage simultaneously. WIP limits prevent multitasking, expose bottlenecks, and improve flow. Core Kanban practice also useful in Scrum boards. If QA column has WIP 3 and 3 items stuck, it signals a testing bottleneck needing attention.

Interviewer expects: Flow improvement and bottleneck visibility.

Q19. What is SAFe?

Answer: Scaled Agile Framework (SAFe) applies agile practices at enterprise scale with multiple teams. Levels: Team (Scrum), Program (ART—Agile Release Train), Portfolio. Adds roles like Release Train Engineer, PI Planning, and system-level coordination. QA participates in PI planning, system demo, and cross-team test strategy. Used in large organizations (50+ teams).

Interviewer expects: Enterprise scaling awareness if relevant to target company.

Q20. What is the QA role in agile?

Answer: QA is a full sprint participant—not a downstream gatekeeper. Activities: refine stories for testability, write acceptance criteria, automate tests during development, pair with devs, perform exploratory testing, and advocate quality in DoD. QA shifts from "test at end" to "build quality in." Senior QA also mentors, defines test strategy, and improves CI pipelines.

Interviewer expects: Embedded team member with shift-left mindset.

Sprint Lifecycle, Ceremonies, Artifacts (Q21–Q35)

Q21. What happens in sprint planning?

Answer: Team selects stories from backlog matching velocity capacity. Stories are broken into tasks. Team commits to sprint goal. QA estimates testing and automation tasks. Dependencies and risks are identified. Output: sprint backlog with committed stories and tasks. Duration: max 2 hours per week of sprint (4 hours for 2-week sprint).

Interviewer expects: Commitment, task breakdown, QA involvement.

Q22. What is a daily standup?

Answer: A 15-minute daily sync where each member shares: what I did yesterday, what I'll do today, any blockers. Not a status report to manager—a coordination meeting for the team. Blockers are addressed after standup. QA shares test progress, automation status, and defects blocking stories. Standups keep the team aligned and surface impediments early.

Interviewer expects: Three questions format—blocker focus.

Q23. What is a sprint review (demo)?

Answer: At sprint end, the team demonstrates completed work to stakeholders. Only done (DoD met) items are shown. Stakeholders provide feedback that may create new backlog items. It's a working demo—not slides. QA may demo test coverage or automation achievements. Duration: 1 hour per week of sprint.

Interviewer expects: Working software demo with stakeholder feedback.

Q24. What is a sprint retrospective?

Answer: Team reflects on the sprint: what went well, what didn't, what to improve. Action items are created and tracked next sprint. Safe environment for honest feedback. Formats: Start/Stop/Continue, 4Ls, sailboat. QA raises testing process improvements—flaky tests, environment issues, DoD gaps. Retros drive continuous team improvement.

Interviewer expects: Continuous improvement with actionable items.

Q25. What is a sprint backlog?

Answer: Sprint backlog contains stories committed for the current sprint plus their broken-down tasks. Owned by the dev team. Updated daily during standup. Visible on the board (To Do → In Progress → In Review → Done). QA tasks (write tests, execute, automate) appear alongside dev tasks on the same board.

Interviewer expects: Team-owned, visible, daily updated.

Q26. What is a burndown chart?

Answer: Burndown chart plots remaining work (story points or tasks) against sprint days. Ideal line shows steady progress to zero by sprint end. Actual line above ideal indicates behind schedule. Helps team self-correct mid-sprint. Burnup charts (showing completed work) are an alternative that handles scope additions better.

Interviewer expects: Visual progress tracking with interpretation.

Q27. What is an increment?

Answer: An increment is the sum of all completed product backlog items at the end of a sprint plus all previous increments. Must be in usable condition and meet DoD. Each increment is potentially shippable. Multiple increments compose the evolving product. QA verifies each increment meets quality standards before demo.

Interviewer expects: Cumulative, shippable, DoD-compliant product slice.

Q28. What is sprint goal?

Answer: Sprint goal is a short statement describing the sprint's objective (e.g., "Enable user registration and email verification"). Provides focus when team faces decisions during the sprint. All committed stories should contribute to the goal. If goal becomes unachievable, PO and team renegotiate scope.

Interviewer expects: Unifying objective beyond individual stories.

Q29. What artifacts does Scrum define?

Answer: Three artifacts: Product Backlog (ordered work list), Sprint Backlog (current sprint work), and Increment (done product slice). Each has a commitment: Product Goal, Sprint Goal, and Definition of Done respectively. Artifacts provide transparency. QA ensures DoD includes testing criteria for every increment.

Interviewer expects: Three artifacts with their commitments.

Q30. How long should agile ceremonies take?

Answer: For a 2-week sprint: Sprint Planning (4 hrs), Daily Standup (15 min × 10 = 2.5 hrs), Sprint Review (2 hrs), Retrospective (1.5 hrs). Backlog refinement (~2 hrs/week). Total ceremony overhead ≈ 10–15% of sprint time. Time-boxes prevent meetings from consuming development capacity.

Interviewer expects: Time-box awareness—ceremonies serve the team.

Q31. What is the role of QA in sprint planning?

Answer: QA reviews story acceptance criteria for testability, estimates testing/automation effort, identifies test dependencies and environment needs, and flags stories too large to test in one sprint. QA commits to specific testing tasks in the sprint backlog. Advocate for including automation and regression in capacity planning.

Interviewer expects: Active participant—not passive recipient of stories.

Q32. What is a release in agile?

Answer: A release bundles one or more sprints' increments for deployment to production. Release frequency varies—some teams deploy every sprint (continuous delivery), others batch releases monthly. Release planning aligns backlog priorities with business milestones. QA ensures regression suites and release criteria are met before deployment.

Interviewer expects: Flexible release cadence tied to business needs.

Q33. What is a task board?

Answer: A task board (physical or Jira) visualizes workflow columns—typically To Do, In Progress, In Review/Testing, Done. Cards represent stories or tasks. WIP limits may apply per column. QA column ("In Testing") makes testing status visible. Boards enable at-a-glance progress tracking for the whole team.

Interviewer expects: Visual workflow with QA column visibility.

Q34. What is epic vs story vs task?

Answer: Epic is a large body of work spanning multiple sprints (e.g., "Payment System"). Story is a sprint-sized user-facing requirement (e.g., "Pay with credit card"). Task is a dev/QA work item within a story (e.g., "Write API tests for payment endpoint"). Hierarchy: Epic → Story → Task/Sub-task.

Interviewer expects: Clear hierarchy with sizing context.

Q35. What is capacity planning in a sprint?

Answer: Capacity = available team hours minus meetings, PTO, and support duties. Compare capacity to story point commitment. If QA capacity is 30% of team, ensure testing tasks don't exceed that. Account for automation maintenance and regression in QA capacity. Overcommitment leads to untested stories and sprint failure.

Interviewer expects: Realistic commitment including QA bandwidth.

Agile Scenario Questions (Q36–Q50)

Q36. What happens when a sprint fails (team doesn't complete committed work)?

Answer: Incomplete stories return to the backlog—NOT carried over automatically. Team discusses root causes in retrospective: overcommitment, blockers, unclear requirements, dependencies. Adjust next sprint capacity accordingly. Never inflate velocity to "catch up." Failed sprint is a learning opportunity, not a punishment. PO re-prioritizes returned stories.

Interviewer expects: No carry-over guilt—inspect and adapt.

Q37. How do you handle changing requirements mid-sprint?

Answer: Scrum principle: no scope changes mid-sprint. If change is critical, PO evaluates trade-off—swap out equal-sized story or cancel sprint and replan. Minor clarifications to acceptance criteria are acceptable. QA assesses test impact of any change. Escalate to PO/SM rather than silently absorbing scope creep.

Interviewer expects: Protect sprint commitment—change via PO.

Q38. Dev finishes stories on last day of sprint—how should QA handle this?

Answer: This is a process problem to address in retro. Short-term: prioritize testing by risk, communicate untested stories to PO for go/no-go. Long-term fixes: enforce DoR/DoD requiring earlier dev completion, implement shift-left (QA starts test prep during dev), use feature flags, and split stories smaller. "Mini-waterfall within sprint" must be eliminated.

Interviewer expects: Process fix—not heroic last-minute testing.

Q39. How do you manage testing when dev velocity exceeds QA capacity?

Answer: Short-term: risk-based test prioritization, extend testing into next sprint with PO agreement. Medium-term: increase automation, shift-left (dev writes unit/integration tests), pair QA with dev. Long-term: add QA headcount, implement test-driven culture, improve DoD. Escalate capacity mismatch to leadership with data—velocity charts showing the gap.

Interviewer expects: Multi-horizon solution with data-driven escalation.

Q40. PO wants to skip testing to meet a deadline—your response?

Answer: Present risk assessment: what could break, customer impact, cost of production defects vs delay. Propose alternatives: reduce scope (MVP), deploy with feature flag off, run abbreviated risk-based suite. Never silently skip testing. Document residual risk and get written acceptance from PO/stakeholders. Protect the team from blame if untested release fails.

Interviewer expects: Risk-based pushback with alternatives—not flat refusal.

Q41. How do you introduce automation in a team new to agile?

Answer: Start small: automate smoke tests for critical paths. Demonstrate value with fast CI feedback. Train team incrementally. Integrate automation tasks into sprint planning. Don't boil the ocean—one module at a time. Show ROI metrics (time saved, defects caught). Get dev buy-in by making tests part of DoD. Celebrate early wins.

Interviewer expects: Incremental adoption with demonstrated value.

Q42. Two stories have a dependency—how do you test?

Answer: Identify dependency during refinement—not during sprint. Options: sequence stories across sprints, use mocks/stubs for dependent service, deploy Story A to test env before Story B starts. QA creates integration test plan covering the dependency. If dependency discovered mid-sprint, escalate to SM for replanning.

Interviewer expects: Early identification and mock/stub strategies.

Q43. How do you handle production hotfixes in agile?

Answer: Hotfix branch from production, fix, test (smoke + targeted regression), deploy, then merge back to main. Some teams allocate hotfix capacity per sprint (e.g., 10%). Post-hotfix: add regression test to prevent recurrence, conduct lightweight RCA. Track hotfix frequency—frequent hotfixes indicate quality process gaps.

Interviewer expects: Structured hotfix process with prevention follow-up.

Q44. Team wants to skip retrospective—your view?

Answer: Retrospectives are non-negotiable—they drive continuous improvement. Without them, teams repeat mistakes (flaky tests, env issues, communication gaps). If team feels retros are useless, change the format, ensure action items are tracked, and make improvements visible. SM facilitates; QA should actively contribute quality-related improvement items.

Interviewer expects: Advocate for retros with format improvement suggestions.

Q45. How do you test across multiple agile teams?

Answer: Define integration test strategy at program level. Contract tests between team services. Shared staging environment with coordinated deployment. System-level regression suite owned by a central QA or rotating team. Cross-team sync meetings for dependency testing. PI planning in SAFe coordinates cross-team test activities.

Interviewer expects: Cross-team coordination mechanisms.

Q46. Stakeholder bypasses PO and asks QA to test extra features—what do you do?

Answer: Redirect to PO for prioritization—QA doesn't accept ad hoc work outside sprint backlog. Explain that unplanned work jeopardizes sprint commitment. If urgent, PO swaps equivalent work out. Document the request in backlog for future sprint. Protecting process prevents burnout and maintains transparency.

Interviewer expects: Process integrity—redirect to PO.

Q47. How do you measure quality in agile without slowing delivery?

Answer: Embed quality in DoD and CI pipeline—fast automated gates don't slow delivery, they accelerate it. Track: escaped defects, automation coverage of critical paths, build stability, MTTR. Use feature flags for safe deployment. Quality metrics in sprint review dashboards. Balance speed with sustainable quality—not speed vs quality as opposites.

Interviewer expects: Quality embedded in pipeline—not separate phase.

Q48. New team member joins mid-sprint—how do you onboard QA?

Answer: Pair with experienced QA on current stories. Provide environment setup guide and framework walkthrough. Start with manual testing of small stories before automation. Assign a "buddy" for the sprint. Don't expect full velocity immediately—ramp over 2–3 sprints. Include onboarding tasks in sprint capacity planning.

Interviewer expects: Structured onboarding with realistic expectations.

Q49. How do you handle stories without clear acceptance criteria?

Answer: Do not start testing—send back to PO via DoR violation. Offer to help write AC during refinement. Document assumptions if forced to proceed (with PO sign-off). Missing AC leads to scope disputes and untestable stories. QA should champion DoR enforcement as a quality gate at refinement, not at sprint end.

Interviewer expects: DoR enforcement with collaborative AC writing.

Q50. How would you improve agile testing maturity in a team?

Answer: Assess current state: test automation %, CI integration, DoD compliance, defect leakage. Prioritize: (1) define/enforce DoD with test criteria, (2) automate smoke in CI, (3) shift-left via refinement participation, (4) add API tests, (5) introduce exploratory testing charters, (6) track metrics and retro on them. Incremental roadmap over 3–6 months with measurable milestones.

Interviewer expects: Structured maturity roadmap with measurable steps.

Scrum — Top 50 Questions

Scrum Roles (Q1–Q15)

Q1. What are the three Scrum roles?

Answer: Product Owner (maximizes product value, manages backlog), Scrum Master (facilitates process, removes impediments, coaches team), and Developers (cross-functional team delivering the increment—includes QA, devs, designers). No sub-teams in Scrum. Each role has distinct accountability. Combined accountability ensures delivery without external dependencies.

Interviewer expects: Three roles with clear accountability.

Q2. What does a Product Owner do?

Answer: PO defines product vision, owns and prioritizes the backlog, writes/clarifies stories and acceptance criteria, accepts or rejects completed work, and represents stakeholder interests. Single person—not a committee. Available to the team for daily questions. PO decides scope and priority; the team decides how to implement.

Interviewer expects: Single accountable person for "what" and "why."

Q3. What does a Scrum Master do?

Answer: SM facilitates Scrum events, coaches team on agile practices, removes impediments, protects team from external disruption, and fosters self-organization. SM is a servant leader—not a project manager or task assigner. SM helps QA by removing environment blockers, facilitating retro quality discussions, and coaching on agile testing practices.

Interviewer expects: Servant leader—not boss or PM.

Q4. What is the Developer role in Scrum (includes QA)?

Answer: "Developers" in Scrum means all team members who create the increment—developers, testers, designers, ops. QA is a Developer in Scrum terms. The team is cross-functional and self-managing. QA contributes to sprint commitment, DoD, and increment delivery equally with coders. No "QA phase"—testing is integrated.

Interviewer expects: QA is part of Developers—cross-functional team.

Q5. Can the Scrum Master also be the Product Owner?

Answer: No—combining SM and PO creates conflict of interest. PO pushes for more features; SM protects team capacity. Same person can't effectively serve both masters. Similarly, SM shouldn't be a manager with HR authority over the team. Role clarity is essential for Scrum to work.

Interviewer expects: Role separation rationale.

Q6. What is the QA's relationship with the Product Owner?

Answer: QA collaborates with PO on acceptance criteria clarity and testability during refinement. QA provides quality perspective on story priority (risk-based). PO accepts/rejects stories based on AC—QA verifies AC are met. QA escalates quality concerns to PO for scope/priority decisions but doesn't override PO on business priorities.

Interviewer expects: Collaborative but PO decides acceptance.

Q7. Who owns quality in Scrum?

Answer: The entire Scrum team owns quality—it is not delegated to QA alone. DoD is a team agreement. Developers write unit tests; QA writes integration/E2E tests; PO defines acceptance criteria. If quality fails, the team fails—not QA alone. This shared ownership is fundamental to agile quality culture.

Interviewer expects: Whole team ownership—not QA silo.

Q8. What is a self-organizing team?

Answer: Self-organizing teams decide internally how to accomplish sprint goals—who works on what, task breakdown, technical approach. No external manager assigns tasks. Team collectively has skills to deliver the increment. QA participates in task selection and may pair with devs on testable code design.

Interviewer expects: Team autonomy on "how" not "what."

Q9. What is the difference between Scrum Master and Project Manager?

Answer: PM controls scope, schedule, budget, and assigns tasks (command-and-control). SM facilitates, coaches, and removes impediments (servant leadership). PM is accountable for delivery; SM is accountable for Scrum process. In agile organizations, PM role often dissolves into PO + SM + self-organizing team.

Interviewer expects: Facilitator vs controller distinction.

Q10. How should QA interact with the Scrum Master?

Answer: Escalate impediments (environment down, blocked defects, missing requirements) to SM. Request retro time for quality topics. Ask SM to coach team on DoD compliance and shift-left. SM helps QA by shielding from scope creep and ensuring testing tasks are visible on the board. Regular sync on team health metrics.

Interviewer expects: Impediment escalation and process coaching.

Q11. What happens if PO is unavailable?

Answer: Team is blocked on decisions—stories can't be clarified, priorities can't change, acceptance can't happen. SM escalates to organization to ensure PO availability (at least daily for questions). Proxy PO is a anti-pattern unless empowered to decide. QA should flag stories with assumptions due to PO unavailability.

Interviewer expects: Blocker identification and escalation.

Q12. Can QA be shared across multiple Scrum teams?

Answer: Discouraged—QA should be dedicated to one team for full sprint commitment and context. Shared QA creates context-switching, overcommitment, and bottlenecks. If unavoidable, cap at 2 teams and allocate sprint capacity explicitly. Better solution: grow QA skills in dev team (TDD) and add headcount.

Interviewer expects: Dedicated team member preference with pragmatic alternatives.

Q13. What is a feature team vs component team?

Answer: Feature teams own end-to-end delivery of user-facing features (preferred in Scrum). Component teams own layers (UI team, API team, DB team)—creates handoffs and integration delays. QA in feature teams tests complete user journeys. Component teams require extra integration testing coordination.

Interviewer expects: Feature team preference for agile.

Q14. What is the PO's role in defect prioritization?

Answer: PO decides defect priority based on business impact and includes critical bugs in sprint backlog alongside features. PO balances new feature work vs defect fixes each sprint. QA provides severity and impact analysis; PO makes the priority call. Unprioritized defects accumulate as technical debt.

Interviewer expects: PO prioritizes with QA input on impact.

Q15. How does QA contribute to the Scrum team beyond testing?

Answer: QA contributes to refinement (testability), DoD definition, sprint planning estimates, code reviews (testability perspective), CI/CD pipeline maintenance, mentoring devs on testing, documentation of test strategy, and stakeholder communication on quality status. Senior QA is a quality coach for the entire team.

Interviewer expects: Quality coach role beyond test execution.

Scrum Ceremonies (Q16–Q30)

Q16. Describe sprint planning in detail.

Answer: Part 1: PO presents prioritized backlog items; team discusses and selects items matching velocity. Part 2: Team breaks stories into tasks and estimates hours. QA adds tasks: write test cases, execute tests, automate, regression. Output: Sprint Goal + Sprint Backlog. Who attends: entire Scrum team + PO. QA must actively participate—not receive assignments passively.

Interviewer expects: Two-part structure with QA task creation.

Q17. What makes a good daily standup for QA?

Answer: QA shares: tests completed yesterday, tests planned today, blockers (env issues, unclear AC, blocked defects). Keep it to 15 minutes. Don't solve problems in standup—take offline. Highlight stories ready for testing or waiting for fixes. Standup is sync, not report-to-manager.

Interviewer expects: Concise blocker-focused updates.

Q18. What is backlog refinement and QA's role?

Answer: Ongoing activity (5–10% sprint capacity) to prepare future stories. QA ensures: AC are testable and complete, dependencies identified, test data needs documented, stories split to sprint size. Well-refined stories enable smooth sprint planning. QA asks "how will I test this?" during refinement.

Interviewer expects: Testability validation during refinement.

Q19. How do you demo testing work in sprint review?

Answer: Show working features validated by tests—not test reports. Demo: "Here's the payment flow working correctly, validated by 15 automated tests and exploratory testing for edge cases." Optionally show CI pipeline green, coverage metrics. Focus on business value delivered with quality confidence, not testing mechanics.

Interviewer expects: Business value focus—not testing navel-gazing.

Q20. What QA topics belong in retrospective?

Answer: Flaky test trends, environment stability, DoD compliance gaps, dev-QA handoff timing, automation debt, defect escape patterns, test data issues, and tooling improvements. Bring data (metrics) not just opinions. Propose specific action items with owners and deadlines. Track previous retro actions related to quality.

Interviewer expects: Data-driven quality improvement proposals.

Q21. How often should backlog refinement happen?

Answer: Continuously, but typically 1–2 dedicated sessions per week (1–2 hours each). Goal: 1–2 sprints of refined stories ready at all times. Ad hoc refinement happens when PO adds new items. QA attends regularly—not just before sprint planning. Insufficient refinement causes sprint planning delays and untestable stories.

Interviewer expects: Regular cadence with sprint-ahead readiness.

Q22. What is the sprint time-box?

Answer: Sprints are fixed length (1–4 weeks, commonly 2). Duration never changes mid-project. All events fit within the sprint calendar. Shorter sprints = faster feedback but more ceremony overhead. QA benefits from 2-week sprints—enough time to test and automate without excessive planning overhead.

Interviewer expects: Fixed duration with rationale for 2-week default.

Q23. Can you extend a sprint if testing isn't done?

Answer: No—sprint duration is fixed. Incomplete stories go back to backlog. Extending sprints destroys cadence and predictability. Root cause: overcommitment, late dev delivery, or insufficient QA capacity. Fix in retro, not by extending. PO decides whether to re-prioritize incomplete stories next sprint.

Interviewer expects: Fixed time-box—fix process not extend sprint.

Q24. What is Scrum of Scrums?

Answer: When multiple Scrum teams work on one product, representatives from each team meet regularly (2–3×/week) to sync dependencies, integration points, and impediments. QA representatives share cross-team testing needs. Helps coordinate integration testing and shared environments across teams.

Interviewer expects: Multi-team coordination mechanism.

Q25. What is the purpose of sprint goal?

Answer: Sprint goal unifies the team around a shared objective beyond individual stories. When trade-offs arise mid-sprint, the goal guides decisions. Example: "Launch MVP checkout"—team may defer nice-to-have stories to achieve the goal. QA aligns testing priority with sprint goal.

Interviewer expects: Decision-making guide during sprint.

Q26. How do you handle bugs found during the sprint?

Answer: Bugs in current sprint stories: fix immediately within the sprint (part of DoD). Bugs in previously completed work: PO prioritizes against backlog—may add to current sprint if critical. Production bugs: hotfix process. QA logs bugs with severity/priority; PO decides when to fix. Don't silently fix without PO awareness.

Interviewer expects: In-sprint vs backlog bug handling distinction.

Q27. What is the cancellation of a sprint?

Answer: Only PO can cancel a sprint (rare)—usually due to major pivot in business direction. Team completes nearly-done work, PO re-evaluates backlog, new sprint planning occurs. QA documents test status of in-progress stories for continuity. Sprint cancellation is exceptional, not a workaround for failure.

Interviewer expects: Rare PO decision—not sprint failure remedy.

Q28. How do ceremonies differ for remote teams?

Answer: Use video conferencing with cameras on. Digital boards (Jira) replace physical boards. Async updates in Slack for timezone gaps. Keep time-boxes strict—remote meetings drag longer. Record sprint reviews for absent stakeholders. QA ensures test environments accessible remotely. Over-communicate blockers in chat channels.

Interviewer expects: Remote adaptations with async considerations.

Q29. What is a scrum board vs kanban board?

Answer: Scrum board is reset each sprint with committed stories. Kanban board is continuous—items flow without sprint boundaries. Scrum board columns typically: To Do → In Progress → In Review → In Testing → Done. Both provide visual workflow. QA benefits from explicit "In Testing" column for visibility.

Interviewer expects: Sprint-scoped vs continuous flow.

Q30. How do you prevent standups from becoming status reports?

Answer: Focus on coordination and blockers, not detailed task reports. SM redirects off-topic discussions offline. Use board to show status visually—standup discusses exceptions and dependencies. Rotate who starts. QA raises blockers, not lists every test case run. 15-minute time-box enforced strictly.

Interviewer expects: Coordination focus with SM facilitation.

Scrum Artifacts, Burndown, Velocity (Q31–Q40)

Q31. Explain product backlog ordering.

Answer: PO orders backlog by business value, risk, dependencies, and learning needs—highest value at top. Not strictly by priority number—relative ordering matters. QA input on risk influences ordering. Top items should be refined and ready. Ordering changes as market conditions and feedback evolve.

Interviewer expects: Value-driven ordering with risk input.

Q32. What makes a good Definition of Done?

Answer: DoD checklist: code reviewed, unit tests pass (>80% coverage), integration tests pass, no open P1/P2 bugs, deployed to staging, QA verified, documentation updated, acceptance criteria met, automated tests added to CI. DoD is team agreement—QA must ensure testing criteria are included and enforced.

Interviewer expects: Concrete checklist including test criteria.

Q33. How is velocity calculated?

Answer: Sum story points of all completed (DoD met) stories in a sprint. Average over 3–5 sprints for planning velocity. Incomplete stories contribute zero points—even if 90% done. Velocity stabilizes as team matures. Use for forecasting, not performance comparison between teams or individuals.

Interviewer expects: Only done stories count—forecasting tool.

Q34. What is a release burndown vs sprint burndown?

Answer: Sprint burndown tracks remaining work within one sprint (days on X-axis). Release burndown tracks remaining story points across all sprints until release milestone. Release burndown helps PO forecast release dates. QA workload should be visible in both—testing tasks are part of remaining work.

Interviewer expects: Sprint vs release scope distinction.

Q35. What is story splitting?

Answer: Breaking large epics/stories into sprint-sized pieces. Techniques: by workflow step, by business rule, by data variation, by interface (API then UI), by CRUD operation. Each split must deliver testable value. QA helps split by identifying independently testable slices. Stories should fit one sprint including testing.

Interviewer expects: Splitting techniques with testable increments.

Q36. What is the increment in practical terms?

Answer: At end of Sprint 3, the increment = all Done work from Sprints 1 + 2 + 3. It's the current shippable product state. Each sprint adds to it. QA verifies the entire increment meets DoD—not just new stories. Regression testing validates previous increments still work.

Interviewer expects: Cumulative shippable product understanding.

Q37. How do you track QA-specific work on the board?

Answer: QA tasks appear as sub-tasks under stories or as separate tasks in "In Testing" column. Visible metrics: stories in testing, blocked tests, automation tasks remaining. Use labels (automation, manual, regression) for filtering. Transparency prevents QA from being an invisible bottleneck.

Interviewer expects: Visible QA work—not hidden bottleneck.

Q38. What is a product goal?

Answer: Product goal is the long-term objective for the product backlog (new in Scrum Guide 2020). Provides context for backlog ordering. Example: "Become the leading mobile payment app in Q3." Sprint goals align toward product goal. QA aligns test strategy with product goal priorities.

Interviewer expects: Long-term vision guiding backlog.

Q39. What is the difference between commitment and forecast in Scrum?

Answer: Modern Scrum uses "forecast" not "commitment"—team forecasts what they can deliver based on velocity, acknowledging uncertainty. Sprint goal provides commitment to an objective, not a fixed scope guarantee. QA forecasts testing capacity honestly—overforecasting leads to untested releases.

Interviewer expects: Forecast humility vs rigid commitment.

Q40. How do you use velocity to plan QA automation work?

Answer: Allocate 20–30% of QA sprint capacity to automation maintenance and new automation. Track automation story points separately. If automation debt grows, negotiate with PO to allocate full sprints to automation via tech debt stories. Velocity should account for testing AND automation—not just feature testing.

Interviewer expects: Explicit automation capacity in planning.

Scrum Scenario Questions (Q41–Q50)

Q41. Team conflict: dev says "QA is the bottleneck"—how do you handle it?

Answer: Bring data to retro: stories delivered to QA on which day, QA cycle time, WIP in testing column. Often root cause is late dev delivery, not slow QA. Propose: earlier story completion, smaller stories, dev writes unit tests, pair testing. Focus on system fix, not blame. SM facilitates constructive discussion.

Interviewer expects: Data-driven retro discussion—not defensive.

Q42. Unfinished stories at sprint end—what happens?

Answer: Incomplete stories revert to backlog at zero points—work is not credited. PO re-prioritizes for next sprint. Team analyzes in retro: why incomplete? Common causes: overcommitment, blockers, scope creep, underestimated complexity. Do NOT carry over automatically or work overtime to finish.

Interviewer expects: Back to backlog—inspect in retro.

Q43. Blocking issue: test environment down for 2 days—what do you do?

Answer: Escalate immediately to SM as impediment. Work on non-environment tasks: write test cases, update automation, review AC for upcoming stories, refactor test code. Communicate impact in standup: "X stories blocked, Y stories can proceed." SM escalates to infra team. Document environment SLA breach for retro.

Interviewer expects: Immediate escalation with productive alternative work.

Q44. PO adds urgent story mid-sprint—how do you respond?

Answer: SM facilitates impact assessment: swap out equal-sized story or accept reduced sprint delivery. QA estimates test impact. If swapped story was partially tested, document status. Never silently add scope. If PO insists without swap, team communicates reduced delivery forecast to stakeholders.

Interviewer expects: Scope swap negotiation—not silent absorption.

Q45. Two team members disagree on DoD compliance—resolve how?

Answer: Refer to written DoD checklist—objective criteria, not opinions. If DoD is ambiguous, update it in retro for clarity. SM facilitates discussion. Example: "Automated tests pass"—which suite? Define explicitly. QA often drives DoD clarity because testing criteria are most commonly disputed.

Interviewer expects: Written DoD as arbiter—update if ambiguous.

Q46. Sprint review stakeholder rejects a "done" story—now what?

Answer: PO determines if rejection is new requirement (new story) or DoD/AC failure (reopened story). If AC were met but stakeholder changed mind, create new story. If AC weren't met, story re-enters sprint. QA re-tests after fix. Misalignment between AC and stakeholder expectations indicates refinement gap.

Interviewer expects: Distinguish new requirement vs defect.

Q47. How do you handle a team member not participating in ceremonies?

Answer: SM addresses individually—understand root cause (timezone, disengagement, workload). Reinforce ceremony value in retro. If QA member skips standup, testing blockers go unreported. Team accountability: ceremonies are team agreements. Chronic absence is an SM coaching issue, not QA's to solve alone.

Interviewer expects: SM responsibility—team accountability.

Q48. Velocity dropped 40% over 3 sprints—concerns?

Answer: Investigate: team composition changes, increased tech debt, environment instability, unclear requirements, holiday periods, or increased bug load. Review in retro with data. Don't pressure team to " increase velocity." Fix root causes. QA contributes data on test cycle time and defect trends. Temporary dips are normal.

Interviewer expects: Root cause analysis—not pressure tactics.

Q49. New PO resets entire backlog priority—impact on QA?

Answer: QA adapts test plans to new priorities. In-progress test work on deprioritized stories pauses. Regression suite priorities may shift. Request refinement sessions for newly prioritized items. Communicate test coverage gaps on deprioritized areas to PO for risk acceptance. Flexibility is key in agile.

Interviewer expects: Adapt with risk communication.

Q50. How would you coach a team transitioning from Waterfall QA to Scrum?

Answer: Phase 1: Embed QA in team, attend all ceremonies. Phase 2: Define DoD with testing criteria. Phase 3: Shift-left—QA in refinement. Phase 4: Automate smoke in CI. Phase 5: Dev-QA pairing on stories. Address cultural resistance: "QA is not a gate." SM and QA lead jointly. Expect 2–3 sprints for adjustment.

Interviewer expects: Phased transition plan with cultural awareness.

Jira — Top 50 Questions

Jira Basics (Q1–Q15)

Q1. What is Jira?

Answer: Jira is Atlassian's project management and issue tracking tool widely used in agile software development. It manages stories, bugs, tasks, epics, and sprints. Features include boards, backlogs, workflows, JQL search, dashboards, and integrations with CI/CD and test tools. Jira is the central hub for tracking work from backlog to release.

Interviewer expects: Agile issue tracking—not just "bug tracker."

Q2. What are Jira issue types?

Answer: Common types: Epic (large initiative), Story/User Story (user-facing feature), Task (general work), Bug/Defect (defect report), Sub-task (child of any issue), Spike (research). Types are configurable per project. QA primarily works with Stories, Bugs, Tasks, and Sub-tasks. Each type can have different workflows and fields.

Interviewer expects: Hierarchy and QA-relevant types.

Q3. What is a Jira project?

Answer: A Jira project is a container for issues belonging to one product or team. Project types: Scrum (sprints, backlog), Kanban (continuous flow), or Bug Tracking (simple). Each project has its own workflow, issue types, fields, and board. QA configures test-related custom fields at project level.

Interviewer expects: Project as team/product container.

Q4. What is a Jira board?

Answer: A board visualizes workflow columns (To Do, In Progress, In Testing, Done). Scrum boards are sprint-based with backlog; Kanban boards are continuous. Filters control which issues appear. QA benefits from dedicated "In Testing" column. Quick filters highlight bugs, blocked items, or assigned-to-me issues.

Interviewer expects: Visual workflow with QA column.

Q5. What is a Jira workflow?

Answer: Workflow defines issue lifecycle: statuses (Open, In Progress, In Review, In Testing, Done) and transitions between them. Conditions, validators, and post-functions control transitions. Bug workflow: Open → In Progress → Fixed → In Testing → Verified → Closed. QA typically owns the In Testing → Verified transition.

Interviewer expects: Statuses, transitions, and QA ownership.

Q6. What are Jira components?

Answer: Components group issues by module or feature area (e.g., Login, Payment, API). Useful for filtering, reporting, and assigning default owners. QA uses components to filter regression scope and analyze defect density by module. Components appear in JQL queries and dashboard filters.

Interviewer expects: Module-based organization for reporting.

Q7. What are Jira labels vs components?

Answer: Components are structured, project-defined categories (one per issue typically). Labels are free-form tags (multiple per issue)—e.g., "regression," "automation," "flaky." Labels are flexible but can become messy without governance. QA uses labels to tag test-related issues for filtering.

Interviewer expects: Structured vs free-form tagging distinction.

Q8. What is the Jira backlog?

Answer: Backlog is the ordered list of unscheduled work in Scrum projects. PO drags items to prioritize. Sprint planning pulls from top of backlog. Backlog grooming/refinement happens here. QA reviews upcoming items for testability. Backlog filter can show only unestimated or unrefined items needing attention.

Interviewer expects: Prioritized unscheduled work list.

Q9. What are Jira sprints?

Answer: Sprints are time-boxed iterations in Scrum projects. Created in backlog view with start/end dates and sprint goal. Issues are dragged into sprints during planning. Active sprint appears on the board. QA tracks testing progress via sprint scope. Completed sprints archive for velocity reporting.

Interviewer expects: Time-boxed iteration container.

Q10. What are custom fields in Jira?

Answer: Custom fields extend issue data: Severity, Priority, Environment, Test Case Link, Automation Status, Found In Version, Fixed In Version. QA commonly uses: Severity, Steps to Reproduce, Environment, Affected Version. Fields appear on issue screens, filters, and JQL queries. Configure per project/issue type.

Interviewer expects: QA-relevant custom fields examples.

Q11. What is issue linking in Jira?

Answer: Links connect related issues: "blocks/is blocked by," "is cloned by," "relates to," "is tested by," "duplicates." QA links bugs to stories they affect and test cases that cover them. Links enable traceability and impact analysis. "Blocks" links prevent premature transitions.

Interviewer expects: Traceability via links—blocks, tested by.

Q12. What are Jira versions/releases?

Answer: Versions represent product releases (v1.0, v2.1). Issues are assigned fix versions (when fixed) and affected versions (where found). Release page shows progress: done vs remaining. QA uses versions to scope regression testing and verify fix deployment. Release notes auto-generate from completed issues.

Interviewer expects: Version-based release tracking.

Q13. What is a Jira filter?

Answer: Filters are saved JQL queries reusable across boards, dashboards, and subscriptions. Examples: "My Open Bugs," "Sprint Testing Queue," "Critical Unresolved." QA creates filters for daily work queues. Filters can be shared with team and subscribed to for email notifications.

Interviewer expects: Saved JQL for reusable queries.

Q14. What are Jira permissions?

Answer: Permission schemes control who can create, edit, transition, assign, and delete issues. QA typically needs: create bugs, transition through testing statuses, add comments/attachments, link issues. Restrict delete and admin permissions. Project roles (Administrators, Developers, Users) map to permission sets.

Interviewer expects: QA-appropriate permission scope.

Q15. What is the difference between Jira Cloud and Server/Data Center?

Answer: Cloud is SaaS hosted by Atlassian—auto updates, less admin overhead. Server (deprecated) and Data Center are self-hosted with full control, custom plugins, and data residency options. Cloud is standard for new deployments. QA workflows are similar; admin capabilities differ. Data Center for enterprise scale (500+ users).

Interviewer expects: Cloud vs self-hosted trade-offs.

JQL, Filters, Dashboards (Q16–Q30)

Q16. What is JQL?

Answer: Jira Query Language (JQL) is a structured query language to search issues. Syntax: field operator value (project = "APP" AND status = "Open" AND assignee = currentUser()). Supports AND, OR, NOT, ORDER BY, IN, WAS, CHANGED. JQL powers filters, boards, dashboards, and automation rules.

Interviewer expects: Basic syntax with AND/OR operators.

Q17. Write JQL to find all open bugs assigned to you.

Answer:

project = "MYAPP" AND issuetype = Bug AND status != Done AND assignee = currentUser() ORDER BY priority DESC

Add severity filter: AND severity in (Critical, Major). Use status in ("Open", "In Progress", "In Testing") for multiple statuses. Save as filter "My Open Bugs" for daily use.

Interviewer expects: Write syntactically correct JQL on the spot.

Q18. Write JQL to find bugs found in current sprint.

Answer:

project = "MYAPP" AND issuetype = Bug AND sprint in openSprints() AND created >= startOfSprint() ORDER BY created DESC

For specific sprint: sprint = "Sprint 23". Combine with component or label for module-specific defect tracking during sprint.

Interviewer expects: Sprint-scoped queries with date functions.

Q19. Write JQL to find untested stories in current sprint.

Answer:

project = "MYAPP" AND issuetype = Story AND sprint in openSprints() AND status = "Ready for Testing" ORDER BY priority DESC

Adjust status name to match your workflow. Alternative: status changed to "In Testing" AFTER startOfSprint() AND status != Done for stories currently in testing.

Interviewer expects: QA work queue query—practical and correct.

Q20. Write JQL for defect density by component.

Answer:

project = "MYAPP" AND issuetype = Bug AND component = "Payment" AND created >= -30d ORDER BY severity DESC

For comparison across components, create separate filters or use dashboard gadgets. Add AND status in (Open, "In Progress", Reopened) for open defect count per module.

Interviewer expects: Component-scoped defect analysis query.

Q21. What JQL functions are commonly used?

Answer: currentUser(), now(), startOfDay(), startOfWeek(), startOfSprint(), openSprints(), closedSprints(), membersOf("qa-team"), linkedIssues("PROJ-123"), issueFunction in linkedIssuesOf(). Functions enable dynamic queries that update automatically without manual date changes.

Interviewer expects: Name 5+ useful functions.

Q22. How do you create a Jira dashboard for QA?

Answer: Create dashboard with gadgets: Filter Results (open bugs by severity), Sprint Burndown, Pie Chart (bugs by component), Created vs Resolved chart, Two-Dimensional Filter Statistics (severity × priority). Add automation coverage custom field chart if tracked. Share with team. Refresh daily for standup visibility.

Interviewer expects: Specific gadgets with QA metrics focus.

Q23. What is a Jira gadget?

Answer: Gadgets are dashboard widgets displaying Jira data visually: filter results, charts, sprint health, burndown, activity stream. Configure each gadget with a saved filter and display options. QA dashboards combine bug trend charts, test queue lists, and sprint progress gadgets for at-a-glance status.

Interviewer expects: Dashboard building blocks.

Q24. How do you subscribe to a Jira filter?

Answer: Open saved filter → Details → Subscriptions → Add Subscription. Set schedule (daily at 8 AM) and email recipients. QA subscribes to "Critical Open Bugs" for daily alerts. Useful for release week monitoring. Avoid over-subscribing—filter noise reduces attention to important alerts.

Interviewer expects: Automated notification setup.

Q25. What is advanced JQL: WAS and CHANGED?

Answer: WAS finds issues that previously had a value: status WAS "In Testing". CHANGED finds issues that changed field values: status CHANGED FROM "In Testing" TO "Done" DURING (startOfWeek(), now()). Useful for QA metrics: how many stories tested this week, defect reopen rate.

Interviewer expects: Historical change tracking queries.

Q26. How do you bulk update issues in Jira?

Answer: Run JQL search → select issues (checkbox) → Tools → Bulk Change. Update fields, transition status, assign, add labels, or edit. Requires bulk change permission. QA uses bulk update to assign test cycle, add "regression" label, or transition verified bugs to Closed. Max 1000 issues per bulk operation.

Interviewer expects: Efficient mass update workflow.

Q27. What are Jira quick filters on boards?

Answer: Quick filters are one-click board filters: "Only My Issues," "Recently Updated," "Bugs Only," "Blocked." Configured in board settings. QA adds quick filters: "In Testing," "Needs Automation," "Critical Bugs." Speeds daily board navigation without writing JQL each time.

Interviewer expects: Board-level convenience filters.

Q28. How do you track sprint progress in Jira?

QA monitors: stories in "In Testing," blocked items, bug count created during sprint. Sprint report identifies spillover stories.

Interviewer expects: Multiple progress views—board, burndown, report.

Q29. What is Jira automation?

Answer: Jira Automation (built-in rules engine) triggers actions on events: when bug created → assign to QA lead, when story moves to Done → notify PO, when sprint starts → create test sub-task. Rules: trigger + conditions + actions. QA automates repetitive workflow tasks without coding. Available in Cloud and Data Center.

Interviewer expects: Rule-based workflow automation awareness.

Q30. Write JQL to find reopened bugs.

Answer:

Framework Design Decisions (Q1–Q15)

Q1. How do you choose between Selenium, Playwright, and Cypress?

Answer: Selenium: mature, multi-language, widest browser support, large community—but slower and flakier. Playwright: modern, auto-wait, multi-tab, fast, great debugging—best for new projects. Cypress: developer-friendly, time-travel debugging—but limited cross-browser and no multi-tab. Choose based on team skills, app architecture (SPA vs multi-page), and CI requirements. For 2025+ greenfield, Playwright is often the default.

Interviewer expects: Trade-off analysis—not "one is best."

Q2. How do you design a test automation framework from scratch?

Answer: Step 1: Define scope (UI, API, or both). Step 2: Choose language matching dev team (Java/Python/TS). Step 3: Select design pattern (POM, Screenplay). Step 4: Build core layers: config, driver management, page objects, utilities, reporting. Step 5: CI integration. Step 6: Documentation and coding standards. Start minimal—add complexity as needed. Deliver working smoke suite in sprint 1.

Interviewer expects: Layered architecture with incremental delivery.

Q3. Page Object Model vs Screenplay—which do you prefer?

Answer: POM is simpler—page classes encapsulate elements and actions. Good for small-medium projects. Screenplay scales better for complex workflows—actors, tasks, and abilities decouple further. For a 4-year engineer targeting Senior/SDET: start with POM, evolve to Screenplay when tests become unreadable or duplicated. Match pattern to team size and app complexity.

Interviewer expects: Pragmatic choice based on complexity.

Q4. How do you organize test project folder structure?

Answer: Typical structure: tests/ (test classes), pages/ (POM), utils/ (helpers, API clients), config/ (env configs), testdata/ (JSON/CSV), reports/, conftest/fixtures. Separate by test type: tests/api/, tests/ui/, tests/e2e/. Consistent naming conventions. README with setup instructions. Structure should be intuitive for new team members.

Interviewer expects: Clean separation of concerns with concrete folders.

Q5. How do you handle test configuration across environments?

Answer: Externalize config in JSON/YAML/properties files per environment (dev, staging, prod). Use environment variables for secrets (never commit credentials). Framework reads env flag (TEST_ENV=staging) and loads matching config. CI sets env vars per pipeline stage. Single codebase, multiple configs—no hardcoded URLs or credentials in tests.

Interviewer expects: Externalized config with secrets management.

Q6. What reporting tool do you use and why?

Answer: Allure Reports: rich HTML reports, history trends, categories, attachments (screenshots, logs). Extent Reports: customizable dashboards. Built-in TestNG/pytest HTML for simplicity. CI publishes reports as artifacts. Allure integrates with Jenkins, GitHub Actions, and Azure DevOps. Good reports reduce debugging time and provide stakeholder visibility.

Interviewer expects: Named tool with CI integration rationale.

Q7. How do you implement custom wait strategies?

Answer: Prefer explicit waits over implicit waits and Thread.sleep. Custom conditions: element visible, clickable, text present, page loaded, API response received. Wrapper methods: waitForElement(locator, timeout), waitForPageLoad(). Playwright auto-wait reduces need; Selenium requires disciplined explicit waits. Log wait timeouts with element locators for debugging.

Interviewer expects: Explicit waits—never Thread.sleep in production frameworks.

Q8. How do you design reusable test utilities?

Answer: Build utilities for: random data generation (Faker), date/time helpers, API client wrapper, database query helpers, screenshot capture, email verification, file upload/download. Keep utilities stateless and pure functions where possible. Document with examples. Avoid god-class utilities—group by domain (ApiUtils, DbUtils, StringUtils).

Interviewer expects: DRY principle with domain-grouped helpers.

Q9. How do you handle authentication in automated tests?

Answer: API-level: obtain token via login API, store in header/cookie for subsequent requests—fastest approach. UI-level: login once, save cookies/localStorage state, reuse in tests (Playwright storageState, Cypress cy.session). Never hardcode credentials—use vault/CI secrets. Separate test accounts per parallel thread to avoid session conflicts.

Interviewer expects: Token/session reuse—not login in every test.

Q10. How do you decide test granularity (UI vs API vs unit)?

Answer: Follow test pyramid: unit tests for business logic (dev-owned), API tests for service contracts and data validation (QA-owned), UI tests only for critical user journeys requiring visual verification. If an scenario can be tested at API level, don't automate at UI. Each UI test should justify its maintenance cost with business criticality.

Interviewer expects: Pyramid-driven granularity decisions.

Q11. How do you handle dynamic elements in UI automation?

Answer: Prefer stable locators: data-testid attributes, accessibility IDs, unique CSS selectors. Avoid XPath based on position or dynamic IDs. Playwright's getByRole, getByText are resilient. For truly dynamic content: partial text match, regex locators, or wait for specific state. Work with devs to add test-friendly attributes (data-testid).

Interviewer expects: Stable locator strategy with dev collaboration.

Q12. How do you implement data-driven testing?

Answer: Externalize test data in JSON, CSV, Excel, or YAML files. Framework reads data and generates test iterations via TestNG @DataProvider, pytest parametrize, or JUnit parameterized tests. Separate test logic from test data. Support environment-specific data sets. Version control test data alongside code. Faker generates dynamic data for unique constraints.

Interviewer expects: External data sources with parameterization.

Q13. How do you version-control and review test code?

Answer: Tests live in same repo as app (preferred) or dedicated automation repo linked via submodule. PR reviews for test code same as production code: coding standards, no hardcoded values, proper waits, meaningful assertions. Branch strategy matches dev team (feature branches). CI runs tests on PR before merge. Treat test code as production code.

Interviewer expects: Test code quality standards equal to app code.

Q14. How do you handle test dependencies between test cases?

Answer: Tests must be independent—no test should depend on another test's outcome. Use setup/teardown fixtures to create preconditions (API creates user, then UI test logs in). If order seems necessary, the test design is wrong—refactor into self-contained tests. TestNG dependsOnMethods is a last resort, not a pattern. Independence enables parallel execution.

Interviewer expects: Independent tests—fixtures for preconditions.

Q15. How would you migrate from manual testing to automation?

Answer: Phase 1: Automate top 10 smoke tests (highest ROI). Phase 2: API tests for backend coverage. Phase 3: Expand regression module by module. Phase 4: CI integration with nightly runs. Phase 5: Full regression in pipeline. Maintain manual exploratory testing for new features. Track automation coverage metrics. Don't automate everything—focus on stable, repetitive, high-value tests.

Interviewer expects: Phased roadmap with ROI focus.

Q16. How do you integrate tests into a Jenkins pipeline?

Answer: Pipeline stages: Checkout → Build → Unit Tests → Deploy to Test Env → Smoke Tests → Integration Tests → Report. Use Jenkinsfile (declarative pipeline). Publish Allure/JUnit reports. Set quality gates: fail pipeline if smoke fails or pass rate < 95%. Parallel stages for UI and API tests. Archive screenshots on failure. Trigger on PR and nightly schedule.

Interviewer expects: Pipeline stages with quality gates.

Q17. How do you integrate tests with GitHub Actions?

Answer: Workflow YAML: trigger on push/PR to main. Jobs: setup language runtime, install dependencies, run tests, upload artifacts (reports, screenshots). Matrix strategy for cross-browser testing. Interviewer expects: YAML workflow with PR gate.

Q18. A test passes locally but fails in CI—how do you debug?

Answer: Check: environment differences (URLs, configs), headless vs headed browser behavior, timing (CI is slower— increase waits), test data conflicts (parallel runs), missing dependencies, screen resolution, and timezone/locale. Enable video recording and screenshots in CI. Compare CI logs with local logs. Run CI Docker image locally to reproduce. Common fix: replace Thread.sleep with explicit waits.

Interviewer expects: Systematic CI-vs-local diff analysis.

Q19. How do you set up parallel test execution in CI?

Answer: TestNG: parallel="methods" thread-count="4". pytest: pytest -n 4 with pytest-xdist. CI matrix: split test suites across jobs. Requirements: test isolation (unique test data per thread), independent browser sessions, sufficient CI agents/cores. Monitor for shared state causing flakiness. Balance parallelism vs resource cost.

Interviewer expects: Parallel config with isolation requirements.

Q20. How do you implement quality gates in CI/CD?

Answer: Gates at each stage: (1) Unit test pass + coverage > 80%, (2) Smoke tests 100% pass, (3) Regression pass rate > 95%, (4) Zero P1 open bugs, (5) Performance within SLA. Pipeline fails and blocks deployment if gate fails. Override requires manager approval with documented risk. Gates evolve as maturity increases—start with smoke only.

Interviewer expects: Staged gates with measurable thresholds.

Q21. How do you run different test suites for different triggers?

Answer: Tag/label tests: @smoke, @regression, @api, @critical. CI triggers: PR → smoke + affected tests, nightly → full regression, release → regression + performance. TestNG groups, pytest markers, or JUnit tags filter execution. Jenkins parameters select suite. Optimize PR feedback to <15 minutes with smoke; full regression runs asynchronously.

Interviewer expects: Tag-based suite selection by trigger.

Q22. How do you handle test failures in CI without blocking everything?

Answer: Categorize: smoke failures block merge (hard gate). Regression failures: quarantine known flaky tests (mark @quarantine, exclude from gate), create bug, investigate async. Use retry for known transient failures (max 1 retry). Track flaky test rate as metric. Never ignore new failures. Separate pipeline stages so smoke failure doesn't waste regression execution time.

Interviewer expects: Tiered failure handling with quarantine discipline.

Q23. How do you integrate API tests in CI/CD?

Answer: API tests run early in pipeline—no browser needed, fast feedback (2–5 min). Deploy service to test env, run Postman/Newman, REST Assured, or pytest requests suite. Validate contracts, status codes, response schemas (JSON Schema validation). API tests gate before UI tests. Publish results to same reporting dashboard.

Interviewer expects: Early pipeline placement for speed.

Q24. How do you manage Docker for test environments in CI?

Answer: Docker Compose spins up app + database + dependencies for integration tests. CI pulls images, starts containers, runs tests, tears down. Ensures consistent environment across local and CI. Use test containers (Java Testcontainers, pytest-docker) for database dependencies. Dockerfile.test for test-specific configurations. Isolated containers prevent test pollution.

Interviewer expects: Containerized consistent environments.

Q25. How do you implement scheduled nightly test runs?

Answer: Jenkins cron trigger (H 2 * * *) or GitHub Actions schedule. Run full regression against staging environment. Email/Slack notification on failure with report link. Track pass rate trends over time. Nightly runs catch integration issues missed during sprint. Include performance baseline tests weekly.

Interviewer expects: Scheduled runs with alerting.

Q26. How do you integrate with Azure DevOps pipelines?

Answer: YAML pipeline with stages: Build, Test, Deploy. PublishTestResults@2 task for JUnit/NUnit XML. Docker@2 for containerized tests. Variable groups for secrets. Test plans integration for manual + automated tracking. Pipeline triggers on PR with branch policies requiring green build. Parallel jobs for test splitting.

Interviewer expects: Azure-specific task awareness if relevant.

Q27. CI pipeline takes 2 hours—how do you optimize?

Answer: Profile: identify slowest tests (usually UI E2E). Strategies: parallelize (4–8 threads), replace UI tests with API where possible, use headless browsers, cache dependencies, split suites across jobs, run only affected tests on PR (test impact analysis), optimize test data setup (API setup vs UI), and remove redundant tests. Target: PR feedback <15 min, full regression <45 min.

Q28. How do you notify team of test failures?

Separate secrets per environment. Mask secrets in pipeline output. Test accounts use dedicated service accounts with minimal permissions. Audit secret access periodically.

Q30. How do you implement deployment verification tests?

Validates: app is up, health check endpoints respond, critical login/payment flows work, database connectivity. Fail triggers automatic rollback (if configured). Synthetic monitoring in production continues verification. Keep smoke suite under 5 minutes for fast rollback decisions.

Test Data & Environment Management (Q31–Q40)

Q31. How do you manage test data for automation?

Never depend on shared mutable data. Each parallel test uses isolated data set. Refresh test data nightly via scheduled job.

Interviewer expects: Isolation and freshness strategies.

Q32. How do you handle database validation in tests?

Q33. How do you test with production-like data safely?

Answer: Never use raw production data in lower environments—GDPR/privacy risk. Use data masking/subsetting tools (Delphix, Tonic) to anonymize PII while preserving data relationships and volume characteristics. Synthetic data generation for most tests. Production-like data reserved for performance/staging tests with access controls.

Interviewer expects: Privacy-compliant data strategy.

Q34. How do you manage multiple test environments?

Answer: Maintain environment registry: URL, purpose, owner, refresh schedule, known issues. Config files map environment names to endpoints. CI deploys to dedicated test env per pipeline. Interviewer expects: Registry and provisioning discipline.

Q35. How do you handle third-party service dependencies in tests?

Interviewer expects: Mock/sandbox strategy for externals.

Q36. How do you implement test data cleanup?

Answer: Teardown hooks (@AfterMethod, pytest fixture yield, try/finally) delete created records via API or DB. Tag test data with prefix (TEST_) for bulk cleanup scripts. Interviewer expects: Automated cleanup with idempotent tests.

Q37. How do you test file upload/download scenarios?

Q38. How do you handle date/time sensitive tests?

Interviewer expects: Deterministic date handling.

Interviewer expects: Automated driver management.

Interviewer expects: Pragmatic approach with dev partnership.

Q44. How do you automate testing for a microservices architecture?

Avoid monolithic E2E covering all services—too slow and brittle.

Interviewer expects: Per-service testing with contract tests.

Q46. How do you test a feature behind a feature flag?

Coordinate with dev on flag lifecycle—remove flag and dual tests when feature is fully released. LaunchDarkly/Unleash SDKs allow programmatic flag control in tests.

Interviewer expects: Both states tested with flag control.

Q47. How do you handle visual regression testing?

Interviewer expects: Tool-aware with baseline management.

Section 16: 1-Day Interview Revision Notes

Condensed revision for the day before or morning of your interview. Read top-to-bottom in ~2 hours.

Top 25 Most Asked Testing Questions

  1. What is software testing? — Evaluating software against requirements to find defects and assess quality before release.
  2. SDLC vs STLC? — SDLC is full development lifecycle; STLC is the testing-specific phases within it.
  3. Smoke vs Sanity? — Smoke is broad/shallow on new build; sanity is narrow/deep on a specific fix.
  4. Regression vs Retesting? — Regression checks for side effects; retesting verifies a specific bug fix.
  5. Severity vs Priority? — Severity is technical impact; priority is business urgency—they can differ.
  6. Verification vs Validation? — Verification: building it right (reviews). Validation: building the right thing (testing).
  7. QA vs QC? — QA is process-oriented prevention; QC is product-oriented detection.
  8. What is RTM? — Requirement Traceability Matrix mapping requirements to test cases bidirectionally.
  9. Test plan contents? — Scope, strategy, resources, schedule, entry/exit criteria, risks, deliverables.
  10. What is exploratory testing? — Simultaneous learning, design, and execution without pre-written scripts.
  11. What is the test pyramid? — Many unit tests, fewer integration tests, minimal E2E/UI tests.
  12. What tests to automate? — Repetitive, stable, high-ROI tests: smoke, regression, API—not exploratory or one-time.
  13. What is BVA? — Testing values at input domain boundaries where defects cluster.
  14. What is equivalence partitioning? — Dividing inputs into classes; one representative test per class.
  15. What is defect lifecycle? — New → Assigned → Fixed → Retest → Verified → Closed (with Reopened path).
  16. What is shift-left testing? — Moving testing earlier in SDLC—reviews, unit tests, API tests before UI.
  17. What is CI/CD in testing? — Automated tests run in pipeline on every commit; failures block deployment.
  18. What is Page Object Model? — Design pattern separating page elements/actions from test logic for maintainability.
  19. How do you handle flaky tests? — Root cause analysis (timing, data, locators), quarantine, fix—never ignore.
  20. What is risk-based testing? — Prioritizing testing by probability × impact of failure.
  21. What is UAT? — Business users validate the system meets their needs before production release.
  22. What is API testing? — Validating endpoints, status codes, response schemas, auth, and error handling.
  23. What is BDD? — Given/When/Then scenarios that serve as specs and automated tests (Cucumber).
  24. What metrics do you track? — Defect leakage, test coverage, automation %, pass rate, MTTR, escaped defects.
  25. Static vs Dynamic testing? — Static: no code execution (reviews). Dynamic: executing code (running tests).

Top 15 Agile/Scrum Quick Q&A

  1. 4 Agile values? — Individuals/interactions, working software, customer collaboration, responding to change.
  2. 3 Scrum roles? — Product Owner, Scrum Master, Developers (includes QA).
  3. Sprint duration? — Fixed time-box, typically 2 weeks, never extended.
  4. What is velocity? — Average story points completed per sprint; used for forecasting, not comparison.
  5. What is DoD? — Team checklist for when a story is truly done—including test criteria.
  6. What is DoR? — Criteria for when a story is ready to enter a sprint—including testability.
  7. What is a user story? — As a [role], I want [feature], so that [benefit] + acceptance criteria.
  8. What is backlog refinement? — Ongoing preparation of upcoming stories with AC, estimates, and splits.
  9. What happens to unfinished stories? — Return to backlog at zero points; analyzed in retro.
  10. Agile vs Waterfall? — Iterative/evolving scope vs sequential/fixed scope; continuous vs phase-gate testing.
  11. Who owns quality in Scrum? — The entire team—not QA alone.
  12. What is an increment? — Sum of all Done work—potentially shippable product slice.
  13. Can scope change mid-sprint? — No, unless PO swaps equivalent work or cancels sprint.
  14. What is a sprint goal? — Unifying objective guiding team decisions during the sprint.
  15. QA role in agile? — Embedded team member: refinement, automation, testing, quality coaching.

Top 10 Jira Quick Q&A

  1. What is JQL? — Jira Query Language for searching issues: project = "X" AND status = "Open"
  2. Issue types? — Epic, Story, Task, Bug, Sub-task, Spike.
  3. What is a workflow? — Statuses and transitions defining issue lifecycle.
  4. Epic vs Story vs Task? — Epic spans sprints; Story fits one sprint; Task is work item within a story.
  5. How to find your open bugs? — issuetype = Bug AND assignee = currentUser() AND status != Done
  6. What is Xray/Zephyr? — Test management plugins for test cases, execution, and coverage in Jira.
  7. What are components? — Module/feature grouping for filtering and defect density reporting.
  8. What is a filter? — Saved JQL query reusable in dashboards and subscriptions.
  9. How to link bugs to stories? — Issue links: "blocks," "relates to," "is tested by."
  10. Jira + CI integration? — Smart commits, build status on issues, automated test result import.

  4. Test independence enables parallel execution and reliable results.
  5. Good bug reports are reproducible, specific, and include environment details.

  7. Explicit waits beat Thread.sleep in every scenario.
  8. data-testid attributes are the most stable UI locators.
  9. API tests are faster, more stable, and better suited for CI than UI tests.
  10. Contract tests catch breaking API changes without full integration environments.
  11. Flaky tests erode team trust in automation—invest in fixing, not retrying.
  12. Test data isolation prevents parallel execution conflicts.
  13. Page Object Model separates locators from test logic for maintainability.
  14. CI pipeline smoke tests should complete in under 5 minutes.
  15. Feature flags enable safe deployment of incomplete features.
  16. Shift-left: test during development, not after.
  17. Shift-right: monitor and experiment in production.
  18. Definition of Done must include test criteria to be effective.
  19. QA in agile is a quality coach, not a phase-gate checkpoint.
  20. Velocity is for forecasting, not performance evaluation.
  21. Sprint commitments are forecasts, not guarantees.
  22. Backlog refinement prevents sprint planning surprises.
  23. Retrospectives drive continuous improvement—never skip them.
  24. Risk-based testing optimizes limited time for maximum risk reduction.
  25. Defect leakage to production is the ultimate QA effectiveness metric.
  26. Severity is technical impact; priority is business urgency.
  27. Regression suites grow over time—regularly prune obsolete tests.
  28. Environment parity with production reduces "works on my machine" defects.
  29. Security testing includes input validation, auth, and OWASP Top 10.
  30. Performance testing validates response time, throughput, and resource usage under load.
  31. Accessibility is often legally required—WCAG 2.1 AA is the common standard.
  32. BDD scenarios serve as living documentation and automated tests.

  37. Never hardcode credentials in test code—use CI secrets management.

  40. Visual regression catches UI bugs functional tests miss.
  41. Service virtualization enables testing without real third-party dependencies.
  42. Test code deserves the same review standards as production code.
  43. Metrics should drive improvement, not blame individuals.
  44. ISTQB provides structured testing vocabulary for interviews.

Common Interview Mistakes to Avoid

  1. Defining testing as only "finding bugs"—show you understand risk mitigation and quality assurance.
  2. Saying "we automate everything"—demonstrate ROI-based selection instead.
  3. Confusing severity and priority—always explain they are independent dimensions.
  4. Not knowing the test pyramid—shows lack of automation strategy understanding.
  5. Blaming developers for bugs—focus on process improvement and collaboration.
  6. Unable to write basic JQL or SQL—technical skills matter for SDET roles.
  7. No STAR-format project examples—prepare 2–3 real stories with measurable outcomes.
  8. Dismissing manual/exploratory testing—automation complements, not replaces.
  9. Not asking the interviewer questions—shows lack of engagement and curiosity.
  10. Overstating skills—be honest about experience level; depth beats breadth of buzzwords.

  9. For system design questions: discuss test infrastructure, CI pipeline architecture, data strategy.
