Section 1: Software Testing Fundamentals

What is Software Testing?

Software testing is the process of evaluating and verifying that a software application or system does what it is supposed to do. It involves executing the software under controlled conditions, observing actual behavior, and comparing it against expected requirements. Testing is not merely about finding bugs — it is a structured activity that provides stakeholders with objective information about product quality, risk, and readiness for release.

At its core, testing answers three fundamental questions:

  • Does it work? — Functional correctness against requirements
  • Does it work well? — Performance, usability, security, reliability
  • Does it break under stress? — Edge cases, load, failure scenarios
Formal Definition (ISTQB): Software testing is the process consisting of all lifecycle activities, both static and dynamic, concerned with planning, preparation, and evaluation of software products and related work products to determine that they satisfy specified requirements, to demonstrate that they are fit for purpose, and to detect defects.

Real-World Example: E-Commerce Checkout

Consider an online retail platform launching a new checkout flow. A QA engineer would:

  1. Review the requirements document for the checkout feature (static testing)
  2. Design test cases covering valid payments, declined cards, expired coupons, and empty cart scenarios
  3. Set up a staging environment mirroring production payment gateways (sandbox mode)
  4. Execute manual and automated tests across Chrome, Firefox, Safari, and mobile browsers
  5. Log defects such as "Tax calculation incorrect for international shipping" with reproduction steps
  6. Retest after developer fixes and perform regression on related modules (cart, inventory)
  7. Sign off with a test summary report indicating pass/fail rates and residual risks

Without this testing, customers could be charged twice, orders could be lost, or payment data could be exposed — each carrying significant business and legal consequences.

Why Testing is Important?

Software defects are expensive. The cost of fixing a bug increases exponentially as it moves through the SDLC. A requirement defect caught during analysis might cost $100 to fix; the same defect found in production could cost $10,000 or more when factoring in emergency patches, customer support, reputation damage, and potential legal liability.

Real-World Failures Caused by Inadequate Testing

Ariane 5 Rocket (1996) — $370 Million Loss

On its maiden flight, the Ariane 5 rocket self-destructed 37 seconds after launch. The root cause was a software exception: a 64-bit floating-point value representing horizontal velocity was converted to a 16-bit signed integer, causing an overflow. This code was reused from Ariane 4 without adequate testing under the new flight conditions. The overflow triggered a cascade of system failures. Lesson: regression testing and boundary value analysis on reused components are critical in safety-critical systems.

Knight Capital (2012) — $440 Million in 45 Minutes

A deployment error reactivated dormant code that executed millions of erroneous stock trades in 45 minutes, bankrupting the firm. The old code path had not been properly removed or tested during deployment. Automated tests did not cover the legacy flag scenario. Lesson: deployment testing, feature flag validation, and smoke tests in production-like environments are essential for financial systems.

Healthcare.gov (2013) — National Launch Failure

The U.S. healthcare enrollment portal crashed on launch day under load from millions of users. Inadequate performance testing, insufficient integration testing between federal and state systems, and rushed timelines led to a site that could handle only 1,100 users when 250,000+ attempted to enroll. Lesson: load testing, end-to-end integration testing, and realistic capacity planning cannot be skipped for high-traffic public systems.

Key Takeaway for Senior QA: Testing is risk management. Your job is not to prove the software has zero bugs (impossible), but to provide evidence-based confidence that critical risks are mitigated before release.

STLC Overview — Software Testing Life Cycle

The STLC is a systematic process specific to testing activities. It defines what testers do, when, and what deliverables they produce. The six phases are sequential but often overlap in Agile environments.

Phase 1: Requirement Analysis

Testers review business requirements, user stories, acceptance criteria, and design documents to understand what to test. Ambiguities and untestable requirements are flagged early.

  • Activities: Review BRD/FRD/user stories; identify testable requirements; clarify ambiguities with BA/PO; identify test data needs; assess automation feasibility
  • Deliverables: Requirement traceability notes, list of clarifications/questions, initial risk assessment
  • Entry Criteria: Requirements document or user stories available and baselined (or sprint-ready)

Phase 2: Test Planning

Define the overall test strategy, scope, resources, schedule, and approach. This is the blueprint for all testing activities.

  • Activities: Define test scope (in/out); choose test approach (manual/automated/mixed); estimate effort; identify tools; define entry/exit criteria; assign roles; plan environments
  • Deliverables: Test Plan document, test strategy, resource allocation, schedule/milestones
  • Entry Criteria: Requirements analyzed; project timeline available

Phase 3: Test Case Design

Create detailed test cases, test scenarios, and test data based on requirements and test design techniques.

  • Activities: Write test scenarios and test cases; apply EP, BVA, decision tables; create RTM; design test data; peer review test cases; prepare automation scripts (if applicable)
  • Deliverables: Test cases, test scenarios, test data, RTM, automation scripts (draft)
  • Entry Criteria: Test plan approved; requirements stable (or sprint stories groomed)

Phase 4: Test Environment Setup

Prepare hardware, software, network, and data configurations needed to execute tests.

  • Activities: Configure test servers; install builds; set up databases with test data; configure third-party integrations (payment sandbox, email stub); verify environment smoke test
  • Deliverables: Test environment ready checklist, environment configuration document, smoke test results
  • Entry Criteria: Test cases ready; infrastructure available

Phase 5: Test Execution

Run test cases, log defects, retest fixes, and track progress against the test plan.

  • Activities: Execute test cases (manual + automated); log defects in tracking tool; perform exploratory testing; retest fixed defects; update test case status; daily status reporting
  • Deliverables: Test execution results, defect reports, daily/weekly status reports, updated RTM
  • Entry Criteria: Build deployed to test environment; entry criteria met; smoke test passed

Phase 6: Test Closure

Evaluate exit criteria, compile metrics, document lessons learned, and formally close the testing cycle.

  • Activities: Verify exit criteria met; calculate metrics (defect density, test coverage); archive test artifacts; conduct retrospective; prepare test summary report; sign-off recommendation
  • Deliverables: Test Summary Report (TSR), test metrics dashboard, archived test artifacts, lessons learned document
  • Entry Criteria: Testing complete or time-box expired; exit criteria evaluation ready

Interview Questions & Answers — Section 1

Q1: What is the difference between testing and debugging?

Answer: Testing is performed by QA/testers to find defects — it is an evaluation activity that compares actual vs expected behavior. Debugging is performed by developers to locate and fix the root cause of a defect. Testing asks "Is there a problem?"; debugging asks "Where exactly is the problem and how do I fix it?" Testing is verification/validation; debugging is diagnosis and repair.

Q2: Can testing show the absence of defects?

Answer: No. Testing can only prove the presence of defects, never their absence. This is a fundamental principle (Dijkstra's aphorism). Exhaustive testing is impossible for non-trivial software due to infinite input combinations. Instead, testing reduces risk by covering critical paths, high-risk areas, and representative scenarios. A "pass" means no defects were found in tested areas — not that zero defects exist.

Q3: What is the difference between SDLC and STLC?

Answer: SDLC encompasses the entire software development process (requirements, design, coding, testing, deployment, maintenance). STLC is a subset focused specifically on testing activities (requirement analysis for testing, test planning, test design, environment setup, execution, closure). STLC phases map to SDLC phases but are testing-specific. In Agile, both are compressed into sprint cycles.

Q4: Explain shift-left testing.

Answer: Shift-left testing moves testing activities earlier in the SDLC — starting at requirements and design rather than waiting for code completion. Activities include: requirements review for testability, TDD/BDD during development, static analysis in CI, unit tests written by developers, API contract testing before UI. Benefits: earlier defect detection (cheaper fixes), shared quality ownership, faster feedback loops. As a Senior QA, you champion shift-left by embedding QA in sprint planning, grooming, and CI pipeline design.

Q5: What is the difference between a defect, bug, error, and failure?

Answer: An error is a human mistake (developer writes wrong logic). A defect/bug is the manifestation of that error in code/documentation (the code doesn't handle null input). A failure is when the defect causes the system to behave incorrectly during execution (application crashes when user submits empty form). Error → Defect → Failure is the causal chain. Terms "bug" and "defect" are used interchangeably in industry.

Q6: What is entry criteria and exit criteria in testing?

Answer: Entry criteria are conditions that must be met before testing begins (e.g., build deployed, smoke test passed, test cases reviewed, environment ready). Exit criteria are conditions to stop testing (e.g., 95% test case pass rate, zero critical/open defects, all P1 test cases executed, test summary report approved). They prevent premature testing starts and define when sufficient testing has occurred.

Q7: How do you decide what NOT to test?

Answer: Use risk-based testing: assess features by business impact × probability of failure. Out-of-scope items include: third-party components already certified, deprecated features scheduled for removal, areas with no requirement changes in this release. Document exclusions in the test plan with justification. Senior QA communicates residual risk to stakeholders for informed release decisions.

Q8: What is the V-Model and how does it relate to testing levels?

Answer: The V-Model pairs each development phase with a corresponding test level: Requirements ↔ Acceptance Testing, System Design ↔ System Testing, Module Design ↔ Integration Testing, Coding ↔ Unit Testing. Test planning for each level begins when its corresponding design document is ready — enabling early test design. It ensures traceability but remains sequential unlike Agile.

Q9: Explain the test pyramid.

Answer: The test pyramid (Mike Cohn) recommends a layered automation strategy: many fast, cheap unit tests at the base; fewer integration/API tests in the middle; minimal slow, expensive UI/E2E tests at the top. Ratio guideline: 70% unit, 20% integration, 10% E2E. This optimizes feedback speed and maintenance cost. Anti-pattern: "ice cream cone" with too many UI tests causing flaky, slow pipelines.

Q10: What metrics do you track as a QA lead?

Answer: Key metrics: test case execution progress (% complete), pass/fail rate, defect density (defects/KLOC), defect leakage (prod defects missed by QA), defect rejection ratio (invalid bugs logged), test coverage (requirements, code), automation coverage %, mean time to detect/fix defects, escaped defect rate. Important: metrics drive behavior — use them for insight, not as individual performance weapons.

Q11: How does QA add value in an Agile team?

Answer: QA brings quality advocacy, risk identification, test automation, acceptance criteria refinement, exploratory testing for edge cases developers miss, CI/CD quality gates, and customer-perspective validation. In Agile, QA is not a "testing phase at the end" — it's continuous collaboration: pairing with devs, reviewing PRs, writing BDD scenarios, maintaining automation frameworks, and providing fast feedback every sprint.

Q12: What is the difference between QA and testing?

Answer: Testing is an activity — executing tests to find defects. QA is a discipline — a process-oriented approach to ensuring quality throughout the SDLC. All testing is part of QA, but QA also includes process improvement, standards, audits, metrics, and prevention activities. A tester executes tests; a QA engineer builds the quality system around testing.

Q13: What is risk-based testing?

Answer: Risk-based testing prioritizes testing effort based on assessed risk (Probability × Impact). High-risk areas (payment processing, authentication, new features) get deep coverage; low-risk stable modules get lighter regression. Steps: identify risks, assess probability/impact, prioritize test cases, allocate resources accordingly, report residual risk. Essential when time/budget constraints prevent full coverage.

Q14: Explain the concept of "test oracle."

Answer: A test oracle is a mechanism to determine expected results — the source of truth for pass/fail. Oracles include: requirements documents, user stories with acceptance criteria, comparable products, user expectations, legal/regulatory standards, mathematical models. Challenge: ambiguous requirements create "oracle problems" where expected behavior is unclear. Senior QA resolves oracle problems early through clarification with stakeholders.

Q15: What is the difference between positive and negative testing?

Answer: Positive testing verifies the system works with valid inputs (login with correct credentials → success). Negative testing verifies the system handles invalid inputs gracefully (login with wrong password → error message, no crash; SQL injection in search field → sanitized/rejected). Both are essential — positive proves functionality works; negative proves robustness and security. A system passing only positive tests is not production-ready.

Q16: How would you test a product with no requirements documentation?

Answer: Use exploratory testing with charter-based sessions; interview stakeholders/PO for implicit requirements; analyze competitor products for expected behavior; test against industry standards (WCAG, OWASP); review existing code/API docs; create requirements as you discover them (reverse-engineered RTM). Document assumptions and get stakeholder sign-off. This is common in legacy systems and poorly managed projects — Senior QA creates structure from ambiguity.

1. Smoke Testing

Definition: Smoke testing (Build Verification Testing) is a preliminary test suite executed on every new build to verify that critical functionalities work and the build is stable enough for further testing.

Objective: Catch showstopper defects early; reject unstable builds before investing in full regression; save QA time.

When to Perform: Immediately after every new build deployment to QA/staging; first step in CI/CD pipeline after deployment.

Real Example (E-Commerce): After deploying build v2.4.1, run smoke suite: homepage loads, user can login, product search returns results, item can be added to cart, checkout page renders. If login fails → reject build, notify dev team.

Comparison with Similar Type: Smoke testing is broader but shallow (critical paths only). Sanity testing is narrower and deeper on a specific changed area. See Smoke vs Sanity table below.

Interview Q&A:

Q: Who performs smoke testing? A: Typically QA, but in mature CI/CD teams, automated smoke tests run in pipeline before QA even receives the build. Developers may run smoke locally before committing.

Q: How many test cases in a smoke suite? A: Usually 15–30 critical path tests. Should complete in 15–30 minutes (automated) or 1–2 hours (manual).

2. Sanity Testing

Definition: Sanity testing is a focused, narrow regression on a specific module or functionality after a minor change/fix to verify that the change works and hasn't broken related features.

Objective: Quickly validate a specific fix or feature without running the full regression suite; confirm the bug fix works.

When to Perform: After a bug fix in a specific module; after minor feature enhancement; when full regression is not feasible due to time constraints.

Real Example (E-Commerce): Developer fixes "coupon code not applying discount." QA runs sanity on coupon module only: apply valid coupon, apply expired coupon, apply coupon + free shipping combo, verify order total recalculates — not the entire checkout/payment/inventory suite.

Comparison with Similar Type: See Smoke vs Sanity table below.

Interview Q&A:

Q: Is sanity testing documented? A: Often informal/ad-hoc — tester uses experience to select relevant cases. In mature teams, sanity checklists exist per module.

Q: Can sanity testing be automated? A: Yes — tag specific test cases as "sanity" in your framework and trigger them on fix deployments.

3. Regression Testing

Definition: Regression testing re-executes previously passed test cases to ensure that new code changes, bug fixes, or enhancements have not adversely affected existing functionality.

Objective: Detect unintended side effects (regressions) in previously working features; maintain confidence in overall system stability.

When to Perform: After every code change, bug fix, or new feature addition; before every release; continuously in CI/CD pipelines.

Real Example (E-Commerce): New "wishlist" feature added. Regression suite re-tests: login, search, cart, checkout, payment, order history, user profile — ensuring wishlist development didn't break existing flows.

Comparison with Similar Type: Regression is broader than sanity (full or partial suite). Retesting is narrower (only the failed test case re-run after fix). See Regression vs Retesting table below.

Interview Q&A:

Q: Full vs partial regression? A: Full regression runs entire suite (pre-release). Partial/selective regression runs tests related to changed modules (sprint-level). Risk-based regression prioritizes high-impact areas.

Q: How do you maintain regression suites? A: Regular grooming — remove obsolete tests, update for UI changes, tag by module/priority, monitor flaky tests, track execution time, use impact analysis to optimize.

4. Retesting (Confirmation Testing)

Definition: Retesting is executing a specific test case that previously failed, after the defect has been fixed, to confirm the fix works correctly.

Objective: Verify that a reported defect has been successfully resolved in the new build.

When to Perform: After developer marks defect as Fixed/Ready for Test; when a new build containing the fix is deployed.

Real Example (E-Commerce): Bug: "Adding 99+ items to cart shows negative total." After fix, retest ONLY that scenario (add 100 items, verify total is correct). If pass → mark Verified. Does NOT re-run entire cart regression unless the fix touched shared code.

Comparison with Similar Type: See Regression vs Retesting table below.

Interview Q&A:

Q: What if retest fails? A: Reopen the defect, add new evidence (screenshots, logs), assign back to developer with note "Fix did not resolve issue" or "Partial fix — still fails when X."

Q: Should you retest with same data? A: Use same data to confirm exact fix, then vary data (boundary, edge cases) to ensure robust fix.

5. Integration Testing

Definition: Integration testing verifies that different modules, services, or components work correctly when combined. It focuses on interfaces, data flow, and communication between integrated units.

Objective: Detect defects in module interactions, API contracts, data transformations, and third-party integrations.

When to Perform: After unit testing of individual modules; when connecting new services; before system testing; continuously in CI for microservices.

Real Example (E-Commerce): Test that Order Service correctly calls Inventory Service (stock deduction), Payment Gateway (charge processing), and Notification Service (confirmation email) when customer completes checkout. Verify data consistency across all services.

Integration Approaches

Big Bang Integration: All modules integrated simultaneously and tested as a whole. Simple but risky — hard to isolate failure source. Used rarely in modern development.

Incremental Integration: Modules integrated and tested one at a time. Two sub-approaches:

  • Top-Down: Start with top-level modules (UI/API gateway), use stubs for lower modules. Good for early UI validation; lower modules tested late.
  • Bottom-Up: Start with lower-level modules (database, services), use drivers for upper modules. Good for data layer validation; UI tested late.
  • Sandwich/Hybrid: Combines top-down and bottom-up, meeting in the middle. Most practical for large systems.

Comparison with Similar Type: Unit testing tests individual components in isolation. Integration testing tests component interactions. System testing tests the complete integrated system.

Interview Q&A:

Q: How do you test microservices integration? A: Contract testing (Pact), API integration tests, service virtualization for dependencies, test containers for databases, message queue verification, chaos engineering for failure scenarios.

Q: What are stubs and drivers? A: Stubs are dummy lower-level modules that return canned responses (used in top-down). Drivers are dummy upper-level modules that call lower modules (used in bottom-up).

6. System Testing

Definition: System testing evaluates the complete, fully integrated application against specified requirements in an environment that closely mirrors production.

Objective: Validate end-to-end system behavior; verify functional and non-functional requirements at the system level; ensure the application meets business requirements.

When to Perform: After integration testing completes; before UAT; typically in staging/pre-production environment.

Real Example (E-Commerce): Complete user journey: register account → browse categories → filter products → read reviews → add to cart → apply coupon → select shipping → pay via credit card → receive email confirmation → track order → initiate return. Test across web and mobile, multiple browsers, different user roles (guest, registered, admin).

Comparison with Similar Type: System testing is performed by QA team in test environment. UAT is performed by end users/business in production-like environment. E2E testing focuses on user workflows across integrated systems.

Interview Q&A:

Q: What environment for system testing? A: Staging/pre-prod that mirrors production: same OS, database version, third-party integrations (sandbox), similar data volume, network configuration.

7. End-to-End (E2E) Testing

Definition: E2E testing validates complete application workflows from start to finish, simulating real user scenarios across all integrated systems, databases, networks, and third-party services.

Objective: Ensure data integrity and process flow across multiple subsystems; validate real-world user journeys; catch integration issues missed by unit/integration tests.

When to Perform: Before major releases; after significant integration changes; as part of CI/CD for critical user paths; nightly automated E2E runs.

Real Example (E-Commerce): Automated E2E: User searches "laptop" → clicks product → adds to cart → proceeds to checkout → enters shipping address → pays with Stripe test card → verifies order in database → confirms email sent via Mailhog → checks inventory decremented → validates admin dashboard shows new order.

Comparison with Similar Type: System testing is broader (includes NFRs). E2E specifically tests cross-system user workflows. E2E is often a subset of system testing focused on critical paths.

Interview Q&A:

Q: E2E test flakiness — how to handle? A: Use explicit waits (not sleep), retry mechanisms with limits, isolate test data, mock external services where possible, run in dedicated CI agents, quarantine flaky tests, root-cause and fix (don't just retry forever).

Q: Cypress vs Selenium for E2E? A: Cypress — faster, easier debugging, auto-waiting, but limited to browser (no multi-tab). Selenium — cross-browser, multi-language, grid support, but slower and more setup. Playwright — modern alternative with auto-wait, multi-browser, API testing.

8. User Acceptance Testing (UAT)

Definition: UAT is the final testing phase where actual end users or business representatives validate that the system meets business requirements and is ready for production deployment.

Objective: Confirm the system solves business problems; validate workflows match business processes; obtain stakeholder sign-off for release.

When to Perform: After system testing passes; before production release; in production-like environment with business-realistic data.

Real Example (E-Commerce): Business users (merchandising team) test: creating promotional campaigns, managing product catalog, processing bulk orders, generating sales reports. Customer service team tests: handling returns, issuing refunds, managing customer complaints through the admin portal.

Comparison with Similar Type: System testing is QA-led, requirement-based. UAT is business-led, scenario-based. Alpha testing is UAT done internally; Beta testing is UAT done by external users.

Interview Q&A:

Q: Who writes UAT test cases? A: Business users with QA support. QA provides templates, training, and environment setup. Business users write scenarios based on real business processes.

Q: UAT fails — what happens? A: Defects logged, prioritized, fixed, re-tested. Release delayed if critical business scenarios fail. QA facilitates but business owns the go/no-go decision.

9. Exploratory Testing

Definition: Exploratory testing is simultaneous learning, test design, and test execution — an unscripted, experience-based approach where testers actively explore the application to discover defects that scripted tests miss.

Objective: Discover unknown defects; test areas not covered by scripted cases; leverage tester creativity and domain knowledge; find usability issues and edge cases.

When to Perform: New feature releases; when requirements are incomplete; alongside automated regression; before release as a "bug bash"; when scripted tests pass but quality "feels" wrong.

Real Example (E-Commerce): Charter: "Explore checkout with various payment methods for 90 minutes." Tester tries: multiple currencies, browser back button during payment, concurrent tabs, slow network (throttling), expired session mid-checkout, special characters in address fields, applying multiple coupons, gift card + credit card combo.

Comparison with Similar Type: Ad-hoc testing is unstructured with no documentation. Exploratory testing is structured by charters, time-boxed, and findings are documented. Scripted testing follows predefined test cases.

Interview Q&A:

Q: How do you make exploratory testing measurable? A: Session-based test management (SBTM): time-boxed charters, mission statements, note-taking, debrief sessions, metrics on areas covered and defects found per session.

10. Compatibility Testing

Definition: Compatibility testing verifies that the application works correctly across different browsers, operating systems, devices, screen resolutions, and network environments.

Objective: Ensure consistent user experience across the supported platform matrix; identify rendering, functional, or performance differences.

When to Perform: Before every major release; when UI changes are made; when new browser/OS versions are released; when expanding supported platform matrix.

Real Example (E-Commerce): Test checkout flow on: Chrome 120/Firefox 121/Safari 17/Edge 120 on Windows 11, macOS Sonoma, iOS 17 (Safari), Android 14 (Chrome). Verify responsive design at 320px, 768px, 1024px, 1440px widths. Test on 3G, 4G, WiFi network speeds.

Comparison with Similar Type: Cross-browser testing is a subset of compatibility testing focused on browsers. Cross-platform testing focuses on OS differences. Responsive testing focuses on screen sizes.

Interview Q&A:

Q: How do you manage browser testing at scale? A: Cloud platforms (BrowserStack, Sauce Labs, LambdaTest), Selenium Grid, Playwright with browser projects, visual regression tools (Percy, Applitools), prioritize based on analytics (most-used browser/OS combos).

11. Security Testing

Definition: Security testing identifies vulnerabilities, threats, and risks in the application to ensure data protection, authentication integrity, and defense against malicious attacks.

Objective: Protect sensitive data (PII, payment info); prevent unauthorized access; comply with regulations (PCI-DSS, GDPR, HIPAA); identify OWASP Top 10 vulnerabilities.

When to Perform: Every release (automated SAST/DAST in CI); before major releases (manual penetration testing); after security-related changes; annual third-party security audits.

Real Example (E-Commerce): Test: SQL injection in search field, XSS in product review comments, CSRF on checkout form, broken authentication (session hijacking), insecure direct object references (accessing other users' orders by changing URL ID), payment data exposure in API responses.

12. Performance Testing

Definition: Performance testing evaluates the speed, responsiveness, stability, scalability, and resource usage of an application under various workload conditions.

Objective: Ensure the application meets performance SLAs; identify bottlenecks before production; validate infrastructure capacity; prevent performance regressions.

When to Perform: Before major releases; after architecture changes; when user base grows; after database schema changes; as part of CI for performance regression.

Real Example (E-Commerce): Verify: homepage loads in < 2 seconds under 1000 concurrent users; search API responds in < 500ms at P95; checkout completes in < 5 seconds; system handles Black Friday traffic (10x normal load) without degradation.

Comparison with Similar Type: Performance is the umbrella term. Load, stress, volume, spike, endurance are types of performance testing with different objectives. See Load vs Stress table below.

Interview Q&A:

Q: Key performance metrics? A: Response time (avg, P90, P95, P99), throughput (requests/sec), error rate, CPU/memory utilization, database query time, concurrent users supported.

13. Load Testing

Definition: Load testing evaluates system behavior under expected normal and peak load conditions to verify it handles anticipated user traffic without performance degradation.

Objective: Validate system meets performance SLAs under expected load; determine capacity; identify performance bottlenecks under normal operating conditions.

When to Perform: Before production launch; before expected traffic events (sales, marketing campaigns); after infrastructure scaling changes.

Real Example (E-Commerce): Simulate 5,000 concurrent users browsing products, 500 users in checkout simultaneously during a flash sale. Verify response times stay within SLA and error rate < 0.1%.

Comparison with Similar Type: See Load vs Stress table below.

Interview Q&A:

Q: How do you design load test scenarios? A: Analyze production analytics for user behavior patterns, think time, ramp-up profile. Model realistic scenarios: 70% browse, 20% search, 10% checkout. Use JMeter/k6/Gatling with gradual ramp-up.

14. Stress Testing

Definition: Stress testing evaluates system behavior beyond normal capacity — pushing the application beyond its breaking point to determine maximum capacity and observe recovery behavior.

Objective: Find the system's breaking point; verify graceful degradation (not catastrophic failure); test recovery mechanisms; understand failure modes.

When to Perform: Before high-traffic events; capacity planning exercises; after major infrastructure changes; disaster recovery validation.

Real Example (E-Commerce): Gradually increase from 5,000 to 50,000 concurrent users until system fails. Observe: at what point do response times exceed 10s? Does the system crash or show "try again" messages? After load drops, does it recover automatically?

15. Volume Testing

Objective: Verify database queries perform with large datasets; ensure batch jobs complete within time limits; validate storage and memory handling with data growth.

When to Perform: Before go-live with expected data volumes; when migrating legacy data; after database schema changes; for data-heavy features (reporting, analytics).

Real Example (E-Commerce): Test product search with 10 million product records; order history page for user with 50,000 past orders; generate monthly sales report from 100 million transaction records; bulk import 500,000 products via admin CSV upload.

Comparison with Similar Type: Load testing focuses on concurrent users. Volume testing focuses on data size. A system can handle 10,000 users (load) but fail with 10 million records (volume).

Interview Q&A:

Q: How do you generate volume test data? A: Database seeding scripts, data generation tools (Faker, Mockaroo), production data anonymization (with privacy compliance), synthetic data factories in test frameworks.

16. Accessibility Testing (A11y)

Definition: Accessibility testing ensures the application is usable by people with disabilities — visual, auditory, motor, and cognitive impairments — complying with standards like WCAG.

Objective: Ensure inclusive user experience; comply with legal requirements (ADA, Section 508, EAA); expand market reach; improve overall UX for all users.

When to Perform: During development (shift-left with automated tools); before every release; during UI redesigns; for government/education sector products (mandatory compliance).

Real Example (E-Commerce): Test: screen reader (NVDA/VoiceOver) can navigate product catalog and complete checkout; all images have alt text; color contrast ratio ≥ 4.5:1 for text; all interactive elements keyboard-accessible (Tab, Enter, Escape); form fields have proper labels; focus indicators visible; video product demos have captions.

Interview Questions & Answers — Section 2

Q1: What is the difference between smoke and sanity testing?

Answer: Smoke testing is wide and shallow — verifies the entire build is stable enough for testing (critical paths across the app). Sanity testing is narrow and deep — verifies a specific fix or feature works correctly. Smoke runs on every build; sanity runs after specific changes. Both are subsets of regression testing. Smoke rejects the build; sanity validates a specific change.

Q2: When would you choose manual testing over automation?

Answer: Manual testing is preferred for: exploratory testing, usability/UX evaluation, one-time tests, frequently changing UI, tests requiring human judgment (visual design, accessibility with screen readers), new features where automation ROI is unclear, ad-hoc/sanity testing. Automation is preferred for: regression, smoke, data-driven tests, performance/load, repetitive stable tests, CI/CD integration.

Q3: Explain the testing levels pyramid.

Answer: Testing levels from bottom to top: Unit Testing (developers, individual functions) → Integration Testing (module interfaces) → System Testing (complete application) → Acceptance Testing (business validation). Each level has broader scope but fewer tests. Combined with the automation pyramid (many unit tests, fewer E2E), this ensures efficient defect detection at the appropriate level.

Q4: What is the difference between integration and system testing?

Answer: Integration testing focuses on interfaces between modules/services — does Module A correctly communicate with Module B? System testing evaluates the entire integrated application against requirements — does the complete system meet business needs? Integration can use stubs/drivers; system testing uses the full environment. Integration finds interface defects; system testing finds requirement gaps.

Q5: How do you prioritize which tests to automate?

Answer: Prioritize by: (1) Frequency of execution — regression, smoke run every build; (2) Stability — features unlikely to change; (3) Business criticality — payment, login, checkout; (4) Data-driven nature — many input combinations; (5) Manual effort — time-consuming repetitive tests; (6) ROI — automation cost vs manual execution cost over time. Don't automate: exploratory, one-time, frequently changing UI, tests requiring human judgment.

Q6: What is alpha and beta testing?

Answer: Alpha testing is UAT performed internally by employees in a controlled environment before external release. Beta testing is UAT performed by a limited external user group in real-world conditions. Alpha catches major issues internally; beta validates real-world usage patterns. E-commerce example: alpha = internal team shops on staging; beta = selected customers use pre-release production site.

Q7: Explain performance testing types.

Answer: Load testing — expected normal/peak load, verify SLAs. Stress testing — beyond capacity to find breaking point. Volume testing — large data sets. Spike testing — sudden traffic surge (flash sale). Endurance/soak testing — sustained load over hours/days for memory leaks. Scalability testing — adding resources improves performance linearly.

Q8: What is mutation testing?

Answer: Mutation testing evaluates the quality of your test suite by introducing small code changes (mutations) and checking if tests catch them. If tests pass despite mutations, your test suite has gaps. Tools: PIT (Java), Stryker (JS). Used to measure test effectiveness, not commonly asked but shows advanced knowledge for Senior/SDET roles.

Q9: What is contract testing?

Answer: Contract testing verifies that microservices adhere to agreed API contracts (request/response schemas) without needing all services running. Consumer defines expectations; provider verifies it meets them. Tool: Pact. Example: Order Service (consumer) expects Inventory Service (provider) to return stock count in specific JSON format. Enables independent deployment of microservices.

Q10: How do you test an API?

Answer: Verify: HTTP status codes (200, 201, 400, 401, 404, 500), response body schema/structure, response time, authentication/authorization, error handling, edge cases (empty body, invalid JSON, missing headers), data validation, rate limiting, CRUD operations, pagination, filtering. Tools: Postman, REST Assured, HTTPie. Automate with REST Assured + TestNG/JUnit in CI.

Q11: What is the difference between black-box and white-box testing?

Answer: Black-box testing examines functionality without knowledge of internal code structure — based on requirements (most QA testing). White-box testing examines internal code structure, logic paths, and branches — requires programming knowledge (unit testing, code coverage). Gray-box is hybrid — partial code knowledge (API testing with database validation).

Q12: What is chaos engineering?

Answer: Chaos engineering intentionally introduces failures (server crashes, network latency, disk failures) in production or staging to verify system resilience and recovery. Tools: Chaos Monkey (Netflix), Gremlin. Example: Kill a payment service pod during checkout — does the system retry gracefully or lose the order? Validates that monitoring, alerting, and failover mechanisms work.

Q13: How do you approach testing a microservices architecture?

Answer: Multi-layered approach: (1) Unit tests per service (developer); (2) Contract tests between services (Pact); (3) Integration tests with test containers; (4) API tests per service endpoint; (5) E2E tests for critical user journeys; (6) Performance tests per service + system-wide; (7) Chaos tests for resilience. Service virtualization for external dependencies. Independent deployability is key — each service tested in isolation and together.

Q14: What is visual regression testing?

Answer: Visual regression testing captures screenshots of UI pages and compares them against baseline images to detect unintended visual changes. Tools: Percy, Applitools, BackstopJS, Playwright visual comparisons. Catches CSS breaking, layout shifts, font changes that functional tests miss. Challenge: dynamic content (ads, timestamps) causes false positives — use ignore regions and stabilization techniques.

Q15: Explain shift-right testing.

Answer: Shift-right testing extends testing into production: canary releases, A/B testing, feature flags, production monitoring, synthetic monitoring, real user monitoring (RUM). Complements shift-left. Example: deploy new checkout to 5% of users, monitor error rates and conversion metrics, gradually increase if healthy. Tools: LaunchDarkly (feature flags), Datadog (monitoring), Selenium Grid in prod for synthetic checks.

Q16: What is the difference between functional and non-functional requirements?

Answer: Functional requirements define WHAT the system should do (user can add item to cart, system sends confirmation email). Non-functional requirements define HOW WELL the system should perform (page loads in 2 seconds, supports 10,000 concurrent users, 99.99% uptime, WCAG AA compliant). Both need testing — NFRs often neglected until late, causing release delays.

Q17: What is gorilla testing?

Answer: Gorilla testing (also called repetitive testing) focuses on repeatedly testing a single module or functionality to uncover defects that appear under sustained use. Example: repeatedly adding/removing items from cart 100 times to find memory leaks or race conditions. Related to monkey testing (random input testing without plan).

Q18: How do you test mobile applications differently from web?

Answer: Mobile-specific considerations: device fragmentation (screen sizes, OS versions), network conditions (offline, 3G, switching WiFi/mobile), battery consumption, app lifecycle (background/foreground, kill/restart), gestures (swipe, pinch, long press), push notifications, app store guidelines, installation/update testing, permissions (camera, location). Tools: Appium, Espresso (Android), XCUITest (iOS), BrowserStack for device cloud.

Q19: What is A/B testing and is it a testing type?

Answer: A/B testing compares two versions of a feature to determine which performs better against business metrics (conversion rate, click-through). It's experimentation, not QA testing — but QA ensures both variants function correctly before experiment launch. Example: Test red vs green "Buy Now" button — QA verifies both buttons work; product team measures conversion. QA role: validate functionality, monitor for defects during experiment.

Q20: What is the difference between test environment and production environment testing?

Answer: Test environment: controlled, synthetic data, sandbox integrations, may differ in scale/config from production. Production testing: real users, real data, real integrations — limited to monitoring, synthetic checks, canary releases, feature flags. Never run destructive/load tests in production. Staging should mirror production as closely as possible. Production testing (shift-right) complements but never replaces pre-production testing.

Q21: How would you test a payment gateway integration?

Answer: Use sandbox/test mode. Test: successful payment, declined card, expired card, insufficient funds, invalid CVV, 3D Secure authentication, refund processing, partial refund, currency conversion, webhook callbacks, timeout handling, duplicate payment prevention (idempotency), PCI compliance (no card data stored). Test with gateway-provided test card numbers (Stripe: 4242... for success, 4000...0002 for decline).

Q22: What is localization vs internationalization testing?

Answer: Internationalization (i18n) testing verifies the application supports multiple languages/regions without code changes — Unicode, date/number formats, RTL layouts. Localization (l10n) testing verifies translations, cultural appropriateness, and region-specific content for a specific locale. E-commerce: i18n = app supports € and date format DD/MM/YYYY; l10n = German product descriptions are accurate and culturally appropriate.

Section 3: Test Design Techniques

Test design techniques are systematic approaches to derive test cases from requirements. They help achieve optimal test coverage with minimal test cases — critical for Senior/SDET interviews where you must demonstrate analytical thinking, not just execution.

1. Equivalence Partitioning (EP)

Concept: Divide input data into partitions (equivalence classes) where all values in a partition are expected to be processed identically. Test one representative value from each partition instead of testing every possible input. Partitions can be valid (accepted) or invalid (rejected).

Detailed Example: Age Validation Field (Accepts Ages 18–65)

A registration form accepts user age where valid range is 18 to 65 inclusive. Values outside this range should be rejected with an error message.

Minimum Test Cases from EP: 7 (one per partition). Combined with BVA below, coverage is comprehensive.

Concept: Defects cluster at boundaries of input ranges. BVA tests values at, just below, and just above partition edges. For range [min, max], test: min-1, min, min+1, max-1, max, max+1.

6. Error Guessing

Concept: Experience-based technique where testers use intuition, domain knowledge, and historical defect data to guess where errors are likely to occur. Not systematic — complements formal techniques.

Common Error Areas

  • Null/Empty inputs: Empty strings, null values, whitespace-only input
  • Boundary values: Zero, negative numbers, MAX_INT, empty collections
  • Special characters: SQL injection (' OR 1=1 --), XSS (<script>alert(1)</script>), Unicode (emoji, RTL text)
  • Concurrency: Double-click submit, simultaneous edits, race conditions
  • Session/State: Browser back button, session timeout mid-transaction, cookie deletion
  • File handling: Oversized files, wrong formats, corrupted files, path traversal (../../etc/passwd)
  • Date/Time: Leap years, timezone changes, daylight saving transitions, Feb 29
  • Network: Slow connection, timeout, disconnect mid-request

Interview Questions & Answers — Section 3

Q1: When would you use EP vs BVA vs Decision Tables?

Answer: Use EP for any input field to identify valid/invalid classes. Add BVA when inputs have numeric/date ranges — always combine EP + BVA. Use Decision Tables when business logic depends on multiple condition combinations (e.g., discount rules, permission checks). They complement each other — use EP+BVA for individual fields, Decision Tables for business rule combinations.

Q2: How many test cases for a field accepting 1–1000?

Answer: EP: 3 partitions (invalid low, valid, invalid high) = 3 cases minimum. BVA (robust): 0, 1, 2, 999, 1000, 1001 = 6 boundary cases. Combined EP+BVA: ~8-9 cases. Plus invalid types (non-numeric, empty, negative) = ~12 total. Not 1000 (exhaustive).

Q3: What is orthogonal array testing?

Answer: Orthogonal array testing is the mathematical foundation of pairwise testing. It uses orthogonal arrays — structured tables where each pair of columns contains all possible pairs equally often. Used when pairwise tools aren't available; provides statistically balanced coverage for multi-parameter testing.

Q4: Explain cause-effect graphing.

Answer: Cause-effect graphing is a intermediate step between requirements and decision tables. "Causes" are input conditions; "effects" are output actions. Graph shows logical relationships (AND, OR, NOT) between causes and effects. The graph is then converted to a decision table. Useful when requirements have complex logical dependencies. Rarely used directly today — decision tables are more practical.

Q5: How do you apply test design techniques to API testing?

Answer: EP: valid/invalid request body fields, auth tokens, HTTP methods. BVA: pagination limits (page=0, page=1, page=max, page=max+1), string length limits. Decision Tables: different auth roles × HTTP methods × resource states. State Transition: order status API (pending → confirmed → shipped → delivered). Error Guessing: missing headers, malformed JSON, SQL injection in query params.

Q6: What is combinatorial testing?

Answer: Combinatorial testing is the umbrella term for testing techniques that cover interactions between parameters. Pairwise (2-way) is most common. 3-way covers triple interactions for safety-critical systems. NIST research shows ~70% of defects triggered by single parameters, ~20% by pairs, ~10% by higher interactions. Pairwise provides optimal ROI for most applications.

Q7: How do you derive test cases from user stories?

Answer: Extract acceptance criteria as test cases (Given-When-Then). Apply EP/BVA to each input mentioned. Use Decision Tables for complex acceptance rules. Identify alternative flows (what if payment fails?). Apply Error Guessing for unstated edge cases. Example: "As a user, I want to reset my password" → test valid email, invalid email, expired token, already-used token, empty email, SQL injection in email field.

Q8: What is the difference between test design and test execution?

Answer: Test design is planning — analyzing requirements, selecting techniques, writing test cases, preparing test data. Test execution is running — executing test cases, comparing actual vs expected, logging defects. Design is intellectual/analytical; execution is operational. Senior QA spends more time on design; junior QA may focus more on execution. SDET automates both.

Q9: How do you know when you have enough test cases?

Answer: Criteria: (1) All requirements traced in RTM; (2) All equivalence partitions covered; (3) All boundaries tested; (4) All decision table rules covered; (5) All use case flows (main + alternative) covered; (6) Risk-based — high-risk areas have deeper coverage; (7) Historical defect analysis — areas with past bugs get extra attention. "Enough" is risk-based, not exhaustive.

Q10: Apply BVA to a password field (8–20 characters).

Answer: Test: 7 chars (min-1, invalid), 8 chars (min, valid), 9 chars (min+1, valid), 19 chars (max-1, valid), 20 chars (max, valid), 21 chars (max+1, invalid). Also: empty (0 chars), exactly 8 chars with special characters, 20 chars with Unicode emoji (may count differently depending on encoding).

Q11: What is domain testing?

Answer: Domain testing is the theoretical foundation behind EP and BVA. It treats the input space as domains (ranges) and selects test points from each domain based on how the program treats values within that domain. EP and BVA are practical applications of domain testing theory. Understanding this shows depth in Senior/SDET interviews.

Q12: How would you test a feature flag system using decision tables?

Answer: Conditions: Feature Flag ON/OFF × User in Beta Group YES/NO × Environment (Staging/Production). Actions: Feature visible/hidden, correct API endpoint called, fallback behavior. Example: Flag ON + Beta User + Staging → New feature visible. Flag OFF + Any User + Production → Old feature visible. Ensures all flag combinations behave correctly without unintended exposure.

Interview Questions & Answers — Section 4

Q1: What makes a good test case?

Answer: A good test case is: (1) Traceable to a requirement; (2) Reproducible — anyone can follow steps and get same result; (3) Independent — doesn't depend on other test cases; (4) Clear expected result — no ambiguity; (5) Concise — tests one thing; (6) Maintainable — easy to update when UI changes; (7) Prioritized — P1/P2/P3 assigned; (8) Automatable — structured for automation conversion.

Q2: How many test cases are enough for a feature?

Answer: No fixed number — depends on complexity, risk, and test design techniques applied. A login feature might need 15–25 test cases (valid, invalid, boundary, security, session). A simple CRUD API might need 10–15. Use RTM to ensure requirement coverage, not arbitrary counts. Risk-based approach: more test cases for high-risk/complex features.

Q3: What is the difference between test plan and test strategy?

Answer: Test Strategy is the high-level organizational document defining testing approach, tools, standards, and philosophy — created once per project. Test Plan is the release-specific document with scope, schedule, resources, and deliverables — created per release/sprint. Strategy answers "how we test in general"; Plan answers "what we test this release."

Q4: How do you maintain test cases when requirements change frequently (Agile)?

Answer: Integrate test case updates into sprint workflow: update test cases during story grooming/refinement; link test cases to user stories in Jira/TestRail; automate regression for stable features; use BDD (Cucumber) so test cases ARE living documentation; groom test suite each sprint — remove obsolete, update changed, add new; version control automated tests in Git alongside application code.

Q5: What is BDD and how does it relate to test cases?

Answer: Behavior-Driven Development uses Gherkin syntax (Given-When-Then) to write test scenarios in plain language understood by business and QA. Example: "Given user is on login page, When user enters valid credentials, Then dashboard is displayed." Tools: Cucumber, SpecFlow, Behave. BDD scenarios serve as both requirements and test cases — living documentation that stays in sync with code.

Q6: How do you organize test cases in TestRail/Jira?

Answer: Structure: Project → Test Suite (by module) → Test Sections (by feature) → Test Cases. Tag by priority, type (smoke/regression), automation status. Link to requirements/user stories. Use custom fields for environment, browser, test data references. Create test runs per sprint/release. Track execution history for trend analysis.

Q7: What is a test charter?

Answer: A test charter is a mission statement for an exploratory testing session. Format: "Explore [target] with [resources] to discover [information]." Example: "Explore checkout payment flow with Chrome DevTools network throttling to discover timeout and error handling issues." Includes: time box (60–90 min), scope, tester name, findings template. Used in Session-Based Test Management (SBTM).

Q8: How do you handle test cases for features behind feature flags?

Answer: Create separate test cases for flag ON and flag OFF states. Tag test cases with feature flag name. In automation, parameterize flag state. Test: flag ON → new feature works; flag OFF → old feature works (no regression); flag toggled mid-session → graceful handling. Include in RTM with flag condition noted in preconditions.

Q9: What information should a test summary report contain?

Answer: Executive summary (go/no-go recommendation); test scope (what was tested); test environment details; test execution metrics (total, passed, failed, blocked, not executed); defect summary (by severity/priority/status); test coverage (RTM coverage %); risks and residual issues; lessons learned; appendices (detailed results, open defects list).

Q10: How do you estimate testing effort?

Answer: Break down by activity: test case design (2–4 hours per feature), environment setup (varies), test execution (15–30 min per manual test case), automation development (2–8 hours per test case depending on complexity), regression (automated suite runtime + manual exploratory), defect retesting (30 min per defect). Add buffer for unknowns (20–30%). In Agile, estimate per story during sprint planning using story points or hours.

Q11: What is the difference between test data and test oracle?

Answer: Test data is the input values used during test execution (email: john@test.com, password: Pass@123). Test oracle is the mechanism to determine the expected/correct result (requirement document, acceptance criteria, comparable product behavior). You need both: data to input and oracle to judge pass/fail. Ambiguous oracles (unclear requirements) are a major testing challenge.

Q12: How do you write test cases for API testing?

Answer: Include: HTTP method, endpoint URL, request headers (Authorization, Content-Type), request body (JSON payload), expected status code (200, 201, 400, 401, 404, 500), expected response body schema/fields, response time threshold. Example: POST /api/login, body: {"email":"john@test.com","password":"Pass@123"}, expected: 200, body contains "token" field, response time < 500ms.

Root Cause Analysis (RCA)

Root Cause Analysis is a systematic process to identify the underlying cause of a defect (not just the symptom) to prevent recurrence. Essential for Senior/SDET roles where you contribute to quality improvement, not just defect reporting.

5 Whys Technique — Example

Problem: Customer charged twice for the same order.

  1. Why? Payment API was called twice for the same order.
  2. Why? User double-clicked "Place Order" button and no debounce protection existed.
  3. Why? Frontend did not disable the button after first click.
  4. Why? No requirement specified button disable behavior on form submission.
  5. Why? Requirements template does not include UX interaction standards for form submissions.

Root Cause: Missing UX standards in requirements template for form submission behavior.

Corrective Action: (1) Immediate fix: disable button on click + backend idempotency key. (2) Process fix: add "form submission UX standards" to requirements template. (3) Prevention: add automated test for double-click scenario in regression suite.

Fishbone (Ishikawa) Diagram — Concept

The Fishbone diagram categorizes potential causes of a problem into branches:

  • People: Insufficient training, communication gaps, skill gaps
  • Process: Missing review steps, no coding standards, inadequate test coverage criteria
  • Technology/Tools: Outdated libraries, missing CI checks, inadequate monitoring
  • Environment: Config differences between staging and production, missing test data
  • Requirements: Ambiguous specs, missing NFRs, late requirement changes
  • Management: Unrealistic timelines, insufficient resources, scope creep without impact analysis

Use Fishbone for complex recurring defects where 5 Whys doesn't reveal the full picture. Gather cross-functional team (dev, QA, ops, BA) for brainstorming.

Defect Metrics

Defect Density

Formula: Defect Density = Total Defects / Size of Module (KLOC or Function Points)

Example: Checkout module has 23 defects and 5,000 lines of code → Density = 23/5 = 4.6 defects per KLOC.

Usage: Compare quality across modules; identify high-risk areas needing refactoring; track improvement over releases. Industry average: 1–5 defects/KLOC for tested commercial software.

Defect Leakage (Escaped Defects)

Formula: Defect Leakage = (Defects Found in Production / Total Defects) × 100

Example: 80 defects found in QA, 5 found in production → Leakage = 5/(80+5) × 100 = 5.9%.

Usage: Measures QA effectiveness; target < 5% for mature teams. High leakage indicates gaps in test coverage, inadequate test environments, or missing test types (performance, security).

Defect Rejection Ratio

Formula: Defect Rejection Ratio = (Rejected + Duplicate Defects / Total Defects Logged) × 100

Example: 100 defects logged, 12 rejected (not a bug), 8 duplicates → Rejection = (12+8)/100 × 100 = 20%.

Usage: Measures bug report quality. Target < 10%. High rejection indicates: poor triage before logging, unclear requirements, insufficient reproduction steps, environment issues mistaken as bugs.

Best Practices for Writing Good Bug Reports

  1. One defect per report: Don't combine multiple unrelated issues. Split "Login broken AND search broken" into two reports.
  2. Reproducible steps: Developer must reproduce without asking "what did you do?" Include exact URLs, exact data, exact sequence.
  3. Specific title: "Checkout total incorrect when 2+ coupons applied" not "Checkout issue" or "Bug in checkout."
  4. Include environment details: Browser, OS, app version, test environment, test account used.
  5. Attach evidence: Screenshots with annotations (red circles/arrows), screen recordings for complex flows, console logs, network HAR files.
  6. Compare expected vs actual: Always include both. Reference requirement ID or acceptance criteria when available.
  7. Assess severity accurately: Don't mark everything Critical. Use the severity definitions consistently.
  8. Check for duplicates first: Search existing defects before logging. Link related defects.
  9. Verify it's reproducible: Reproduce at least twice before logging. Note if intermittent — include reproduction rate (e.g., "3 out of 5 attempts").
  10. Isolate the problem: Narrow down to minimum steps. Remove unnecessary steps that aren't required to reproduce.
  11. Include test data: Exact credentials, product IDs, coupon codes used — or note "available on request" for sensitive data.
  12. Log early, log often: Don't wait until end of day to log 10 defects from memory. Log immediately while context is fresh.
Bad Bug Report Example: Title: "Login not working." Steps: "Try to login." Expected: "Should work." Actual: "Doesn't work." — This wastes everyone's time. Developer cannot reproduce, tester gets frustrated, defect gets rejected.

Interview Questions & Answers — Section 5

Q1: What is the difference between severity and priority?

Answer: Severity is the technical impact of a defect on system functionality — set by QA (Critical = system crash/data loss, Low = cosmetic). Priority is the business urgency to fix — set by PM/PO (P1 = fix immediately, P4 = fix when convenient). They are independent: a cosmetic typo on the CEO demo page = Low Severity + P1 Priority. A data corruption bug in an unused admin report = Critical Severity + P4 Priority.

Q2: What would you do if a developer rejects your bug as "Works as Designed"?

Answer: (1) Re-read the requirement/acceptance criteria — is the behavior actually specified? (2) Discuss with developer to understand their perspective. (3) If requirement supports your report, provide evidence and re-open with requirement reference. (4) If requirement is ambiguous, escalate to BA/PO for clarification — this is a requirements defect, not a code defect. (5) Document the decision. Never take rejection personally — focus on product quality.

Q3: How do you handle a critical bug found near release deadline?

Answer: (1) Log immediately with Critical severity, P1 priority. (2) Notify QA Lead, Dev Lead, and PM immediately — don't wait for daily standup. (3) Provide clear reproduction steps and impact assessment. (4) Assess workaround availability. (5) Participate in go/no-go meeting with evidence. (6) If release proceeds despite critical bug, document the risk acceptance decision. (7) Ensure fix is in top of retest queue post-release if deferred.

Q4: Explain defect triage meeting.

Answer: Defect triage is a regular meeting (daily or every 2 days) with QA Lead, Dev Lead, and PM to review new defects. Activities: validate defect validity (reject invalid/duplicate), assign severity and priority, assign to developer, discuss deferred defects, identify trends. Goals: ensure critical defects are addressed promptly, prevent backlog buildup, maintain defect report quality. QA presents defects; PM prioritizes; Dev Lead assigns.

Q5: What is defect clustering?

Answer: Defect clustering (Pareto principle applied to testing) states that a small number of modules contain most of the defects. Typically 80% of defects are found in 20% of modules. Implication: focus testing effort on historically defect-prone modules; apply more rigorous testing, code reviews, and refactoring to high-defect areas. Track defect distribution by module over releases to identify clusters.

Q6: How do you prevent duplicate bug reports?

Answer: (1) Search existing defects before logging — use keywords from the title. (2) Check if already logged by teammate in daily standup. (3) Use consistent naming conventions for similar defects. (4) Link related defects instead of creating new ones. (5) QA Lead reviews new defects during triage for duplicates. (6) In automation, use test failure deduplication in CI reporting tools.

Q7: What is the difference between reopened and retest?

Answer: Retest is the standard state when a fix is deployed and awaiting QA verification — expected happy path. Reopened means QA verified the fix and it FAILED — the defect persists or the fix introduced a regression. Reopened defects typically get higher priority and are assigned back to the same developer with additional evidence about why the fix didn't work.

Q8: How do you measure QA team effectiveness?

Answer: Key metrics: defect leakage rate (< 5% target), defect rejection ratio (< 10%), test coverage (requirements + code), automation coverage, mean time to detect, customer-reported vs QA-found defect ratio. Important: never use defect count alone — finding more defects means better testing, not worse quality. Combine quantitative metrics with qualitative assessment (test case quality, collaboration, process improvement contributions).

Q9: What is a showstopper defect?

Answer: A showstopper (release blocker) is a Critical severity defect that prevents release because it: crashes the system, causes data loss/corruption, has no workaround, affects all users, or violates compliance/security requirements. Example: payment processing completely broken on release candidate. Release cannot proceed until showstopper is fixed and verified. QA Lead has authority to block release based on showstoppers.

Q10: How do you write a bug report for an intermittent defect?

Answer: Document: (1) Reproduction rate — "Occurs approximately 3 out of 10 attempts"; (2) Conditions when it occurs — "Only after page has been idle for 5+ minutes"; (3) Conditions when it does NOT occur — "Never occurs on first page load"; (4) Browser console logs from both success and failure attempts; (5) Network conditions, timing factors; (6) Screen recording capturing the failure. Tag as "Intermittent" so triage team understands reproduction difficulty.

Q11: What is defect seeding (error seeding)?

Answer: Defect seeding is intentionally inserting known defects into code by the development team, then checking if QA finds them during testing. Used to evaluate test suite effectiveness — if seeded defects aren't found, test coverage has gaps. Formula: Efficiency = (Defects Found / Defects Seeded) × 100. Rarely used in industry due to ethical concerns and artificial nature, but understanding the concept shows testing maturity in Senior/SDET interviews.

Q12: How do you handle defects found in production?

Answer: (1) Log immediately with Critical/High severity and P1 priority. (2) Provide production evidence (screenshots, user reports, logs). (3) Assess impact — how many users affected? Data integrity at risk? (4) Coordinate with on-call dev team for hotfix assessment. (5) Perform root cause analysis — why did QA miss it? (6) Add test case to regression suite to prevent recurrence. (7) Update defect leakage metrics. (8) Conduct blameless post-mortem for Critical production defects.

Q13: What is the difference between deferred and rejected defects?

Answer: Rejected = NOT a valid defect. Reasons: works as designed, cannot reproduce, environment issue, duplicate, out of scope. No fix needed. Deferred = IS a valid defect but fix is postponed to a future release/sprint. Reasons: low priority, workaround exists, requires major refactoring, planned for future feature. Deferred defects remain in backlog and must be re-evaluated before each release.

Q14: How do you use defect metrics to improve quality?

Answer: High leakage → analyze escaped defects for test coverage gaps, add test cases, improve test environments. High rejection ratio → improve bug reporting training, add pre-logging checklist, verify reproducibility. High density in one module → recommend refactoring, increase code review rigor, add unit tests. Increasing defect count across releases → investigate process issues, technical debt, team capacity. Metrics drive process improvements, not blame.
