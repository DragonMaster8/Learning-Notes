Section 6: Agile Methodology

What is Agile?

Agile is an iterative and incremental approach to software development that emphasizes collaboration, customer feedback, and the ability to respond to change. Rather than delivering an entire product at the end of a long development cycle, Agile teams deliver working software in small, frequent increments — typically every one to four weeks.

Definition

Agile is a mindset and set of practices defined by the Agile Manifesto (2001) that prioritizes individuals and interactions, working software, customer collaboration, and responding to change over rigid processes, comprehensive documentation, contract negotiation, and following a fixed plan.

History

In the late 1990s, software projects were failing at alarming rates under the Waterfall model. The 2001 gathering at Snowbird, Utah brought together 17 software practitioners — including Kent Beck, Martin Fowler, and Jeff Sutherland — who authored the Agile Manifesto. Frameworks like Scrum (1995), Extreme Programming (XP), and Kanban soon followed, giving teams concrete practices to implement Agile values.

Why Agile Replaced Waterfall

Waterfall failed because: requirements changed after months of development; testing happened too late to fix architectural issues; customers saw the product only at the end; and long release cycles meant competitors shipped first. Agile addresses each failure mode by shortening feedback loops, integrating QA from sprint one, and delivering value continuously.

  • Changing requirements: Waterfall locked requirements at the start; Agile welcomes change even late in development.
  • Late testing: Waterfall deferred QA to the final phase; Agile embeds testing in every sprint.
  • Customer disconnect: Waterfall had sign-off gates; Agile involves customers every sprint review.
  • High risk: Waterfall discovered problems at integration; Agile surfaces risk early through working increments.

Agile Manifesto: The Four Values

The Agile Manifesto states four value pairs. The items on the left are valued more — but the items on the right still matter.

1. Individuals and Interactions over Processes and Tools

Meaning: Effective communication between developers, QA, PO, and stakeholders drives quality more than any tool chain.

QA Focus: As a QA engineer, you participate in standups, refinement, and pairing sessions — not just ticket updates. When you find a defect, walk to the developer or hop on a quick call rather than filing a vague bug and waiting three days. Tools like Jira and Selenium support your work; they don't replace the conversation about edge cases, test data, or acceptance criteria ambiguity.

2. Working Software over Comprehensive Documentation

Meaning: Delivering tested, deployable software each sprint matters more than 200-page requirement specs.

QA Focus: Your test cases, automation scripts, and CI results are the living documentation. Focus on executable specifications (BDD Gherkin, automated regression suites) over Word documents that go stale. Document enough for audit and onboarding — not so much that maintenance becomes a second job.

3. Customer Collaboration over Contract Negotiation

Meaning: Ongoing partnership with the business beats arguing over what the contract "really meant."

QA Focus: Attend sprint reviews and demo prep. When acceptance criteria are unclear, ask the PO or customer proxy directly: "If the user enters zero quantity, should we block or warn?" Your questions during refinement prevent contract-style disputes later.

4. Responding to Change over Following a Plan

Meaning: Plans are useful; rigidity is not. Priorities shift — teams adapt.

QA Focus: Build flexible test automation (Page Object Model, API abstraction layers) so when UI changes mid-sprint, you update one layer — not 400 tests. Maintain a regression suite that runs fast enough to give confidence on every change, not just planned releases.

SAFe Overview

What is SAFe?

Scaled Agile Framework (SAFe) is an enterprise-scale Agile methodology that aligns multiple Scrum/Kanban teams toward common program and portfolio objectives. It adds layers — Team, Program, Large Solution, Portfolio — so that 10 teams shipping independently still integrate into a cohesive release train.

PI Planning (Program Increment Planning)

PI Planning is a two-day, face-to-face (or virtual) event held every 8–12 weeks where all teams on an Agile Release Train (ART) plan the next Program Increment together. Teams break down features into stories, identify dependencies, and commit to PI Objectives. QA participates by estimating test effort, flagging integration test needs, and identifying cross-team test dependencies (e.g., Team A's API change breaks Team B's UI tests).

Agile Release Trains (ARTs)

An ART is a long-lived team of Agile teams (typically 50–125 people) that plans, commits, and executes together. Each ART delivers a continuous flow of value on a shared cadence. QA engineers belong to a specific team within the ART but coordinate system-level testing during the PI.

QA Role in SAFe

  • Team level: Same as Scrum — refine stories, automate, test within sprint, meet Definition of Done.
  • Program level: Participate in System Demo; execute end-to-end scenarios spanning teams; maintain shared regression suite.
  • PI Planning: Estimate testing capacity; negotiate test environment availability; identify NFR (performance, security) test stories.
  • Inspect & Adapt: Contribute to problem-solving workshops on quality metrics, escaped defects, automation coverage gaps.

Advantages of Agile for QA

Early Involvement: QA joins refinement before development starts — catching ambiguous acceptance criteria when they're cheap to fix, not after code is merged.

  • Continuous Feedback: Daily standups and sprint reviews create rapid defect discovery and resolution cycles.
  • Shift-Left Testing: Unit tests, API tests, and contract tests run in CI before QA manual pass — reducing escaped defects.
  • Reduced Test Debt: Testing every sprint prevents the "test everything at the end" avalanche.
  • Direct PO Access: QA clarifies requirements with the Product Owner in real time.
  • Automation ROI: Frequent releases justify automation investment; each regression run saves manual hours.
  • Quality as Team Responsibility: Developers write unit tests; QA focuses on integration, E2E, and exploratory testing.
  • Visible Progress: Burndown charts and CI dashboards show test completion transparently.
  • Retrospective Improvement: QA proposes process fixes (better test data, stable environments) every sprint.

Challenges of Agile for QA

  • Incomplete Requirements: Stories may lack edge-case AC; QA must ask probing questions under time pressure.
  • Time Pressure: "Done" at sprint end means testing competes with last-minute dev commits — risk of shallow testing.
  • Regression Overhead: Each sprint adds features; full regression grows unless automation keeps pace.
  • Changing Scope: Mid-sprint scope creep (even when discouraged) forces re-prioritizing test effort.
  • Environment Instability: Shared QA environments break frequently; blocked testing stalls sprint goals.
  • Flaky Automation: CI failures from unstable tests erode trust in the pipeline.
  • Documentation Gaps: Less formal specs mean QA relies on tribal knowledge — risky for onboarding.
  • Sprint Commitment Pressure: Teams may mark stories "done" without meeting full DoD to hit velocity targets.

Interview Questions & Answers — Agile Methodology

Q1: What is Agile, and how is it different from Scrum?

A: Agile is a philosophy and set of values/principles for iterative development. Scrum is a specific framework that implements Agile through defined roles, ceremonies, and artifacts. You can be Agile without Scrum (e.g., Kanban, XP), but Scrum is the most common Agile framework.

Q2: Explain the Agile Manifesto values and how they apply to QA.

A: The four values prioritize people, working software, collaboration, and adaptability. For QA: collaborate in refinement (interactions over tools), rely on automated tests as living docs (software over documents), clarify AC with PO directly (collaboration over contracts), and refactor tests when requirements change (change over plan).

Q3: Why did Agile emerge? What problems does it solve for testing?

A: Waterfall caused late defect discovery, requirement drift, and QA bottlenecks at project end. Agile integrates testing every sprint, enabling shift-left, continuous feedback, and cheaper defect fixes.

Q4: What are the 12 Agile Principles? Name five and explain QA relevance.

A: Examples: (1) Early delivery — test each increment; (7) Working software as progress — DoD requires tested code; (9) Technical excellence — invest in automation frameworks; (11) Self-organizing teams — QA estimates and owns test tasks; (12) Reflect and adapt — retros improve test processes.

Q5: Compare Agile and Waterfall from a QA perspective.

A: Waterfall: QA starts after dev completes; big test phase; late defects expensive. Agile: QA from refinement; continuous testing; defects found within days; regression grows each sprint requiring automation discipline.

Q6: What is the difference between Scrum, Kanban, and SAFe?

A: Scrum uses time-boxed sprints with fixed roles and ceremonies. Kanban uses continuous flow with WIP limits — no sprints. SAFe scales Agile to enterprises with ARTs, PI Planning, and alignment across dozens of teams.

Q7: What is PI Planning, and what does QA do there?

A: PI Planning is a 2-day event every 8–12 weeks where ART teams plan the next Program Increment. QA estimates test effort, identifies cross-team dependencies, plans integration/regression scope, and commits to PI objectives.

Q8: What are the main advantages of Agile for QA?

A: Early involvement in requirements, continuous feedback, shift-left, direct PO access, justified automation ROI, quality as shared responsibility, and sprint retros for process improvement.

Q9: What challenges does Agile create for QA?

A: Incomplete requirements, sprint-end time pressure, growing regression scope, environment instability, flaky automation, reduced formal documentation, and temptation to skip DoD to meet velocity.

Q10: How does QA "shift left" in Agile?

A: Participate in refinement and Three Amigos; write test scenarios before dev; encourage TDD/BDD; run API and contract tests in CI; review PRs for testability; automate smoke tests on every build.

Q11: What is an increment in Agile?

A: A potentially shippable piece of product at the end of an iteration, meeting Definition of Done — including tested, integrated, and documented (as agreed) functionality.

Q12: How do you handle changing requirements mid-sprint as QA?

A: Discuss impact with PO and SM; if scope change is essential, swap equal-effort stories out; update test cases and automation; re-run targeted regression; never silently absorb extra scope — it hides quality risk.

Q13: What metrics does QA track in Agile?

A: Sprint burndown (test tasks), test case execution rate, automation coverage, escaped defect rate, CI pass rate, defect density per story, cycle time from In Progress to Done, flaky test count.

Q14: How is quality everyone's responsibility in Agile?

A: Developers write unit/integration tests and fix bugs they introduce. PO writes testable AC. QA provides quality coaching, exploratory testing, and automation — but doesn't own quality alone. DoD is a team agreement.

Q15: Describe a situation where Agile practices helped you catch a critical bug early.

A: (Use STAR format) During refinement for a payment story, I asked about currency rounding. The AC didn't specify banker's rounding — we updated it before dev started. My API test written on day 2 caught a rounding bug that would have caused financial discrepancies in production.

Q16: Can Agile work without automation?

A: For small products early on, yes — manual regression may suffice briefly. As sprints accumulate features, manual regression becomes the bottleneck. Sustainable Agile QA requires automation for regression, smoke, and CI gates.

Q17: What is the QA engineer's role in sprint review?

A: Support the demo by confirming tested scenarios, present test coverage summary if asked, note known open defects and workarounds, and gather stakeholder feedback that becomes new backlog items.

Q18: How do you prevent testing from becoming a sprint bottleneck?

A: Test alongside development (not after), automate regression, use risk-based testing for new features, ensure stories are small and testable (INVEST), maintain stable test environments, and include test tasks in sprint planning with realistic capacity.

Q19: What is the difference between Agile and Lean?

A: Agile focuses on iterative software delivery and customer collaboration. Lean (originally from manufacturing) focuses on eliminating waste and optimizing flow. Kanban applies Lean principles to knowledge work. Both complement each other — Agile teams often use Lean thinking to identify waste in testing (waiting for environments, redundant manual tests).

Q20: How do you measure Agile maturity from a QA standpoint?

A: Indicators: QA involved in refinement (>80% of stories), automated regression in CI (<30 min), DoD consistently enforced, escaped defect rate decreasing, cycle time from Ready for QA to Done stable, test tasks visible on sprint board, retros produce actionable quality improvements.

Agile Anti-Patterns QA Should Watch For

  • Water-Scrum-Fall: Sprints for dev, but testing and release happen in big batches afterward — negates Agile benefits
  • Dark Scrum: Velocity worship; stories marked Done without meeting DoD — QA must push back
  • Fragile Agile: No automation investment; manual regression grows until team collapses under test debt
  • Zombie Standups: Status reports to manager, no blockers raised — QA should model raising real impediments
  • PO Absent: No one to clarify AC — QA documents assumptions and escalates via SM

Section 7: Scrum Framework

Scrum Overview

Scrum is the most widely adopted Agile framework. It organizes work into fixed-length sprints, defines three accountabilities (Product Owner, Scrum Master, Developers), five events, and three artifacts. For QA engineers, Scrum provides predictable cadence, clear entry points for quality activities, and transparent progress tracking.

Scrum Roles

Product Owner (PO)

The Product Owner maximizes product value by owning and ordering the Product Backlog. They represent stakeholders and customers, decide what gets built and in what order, and accept or reject completed work.

PO Interactions with QA: QA helps PO write testable AC during refinement. QA asks "what if" questions that expose gaps. PO clarifies business rules QA needs for test data (e.g., valid discount codes). At sprint end, PO relies on QA's sign-off that DoD is met before formal acceptance.

AC updated: "Points may cover up to 80% of order total; remaining must be paid by card." QA writes test cases from clarified AC — prevents rework.

The Scrum Master serves the team by ensuring Scrum is understood and enacted, removing impediments, and fostering an environment where the team can be productive. They are a servant-leader — leading through service, not authority.

Servant-Leader Concept: The SM doesn't assign tasks or dictate solutions. They ask: "What's blocking you?" and "What does the team need?" For QA, this means the SM advocates for test environment stability, realistic sprint capacity including test effort, and team agreement on Definition of Done — without being QA's manager.

Development Team (Developers)

In Scrum, "Developers" means everyone who creates the increment — not just coders. This includes frontend, backend, QA/automation engineers, and sometimes DevOps. The team is cross-functional (all skills to deliver increment) and self-organizing (decides how to accomplish sprint goal).

Scrum Artifacts

Product Backlog

Ordered list of everything needed in the product. PO owns ordering. Items include user stories, bugs, tech debt, spikes, and enablers.

Prioritization Techniques:

  • MoSCoW: Must have, Should have, Could have, Won't have — QA flags Must-have quality items (e.g., security fix)
  • WSJF (Weighted Shortest Job First): Cost of Delay / Job Size — used in SAFe; QA input on "job size" includes test complexity
  • Value vs Effort: Quick wins vs strategic bets — QA highlights hidden test effort on "small" UI changes

Sprint Backlog

Subset of Product Backlog selected for the sprint plus plan to deliver it. Includes dev tasks AND test tasks.

Increment

Sum of all Product Backlog items completed during the sprint, meeting Definition of Done. Each increment is additive to prior work.

Definition of Done — QA Perspective:

  • All AC verified (manual and/or automated)
  • No open Critical/High bugs on the story
  • Automation added/updated for regression paths (per team agreement)
  • Code merged to main/develop; CI pipeline green
  • Test results documented in test management tool or Jira
  • Peer review of test coverage for complex stories
  • Performance/security checks if applicable NFRs exist

Velocity

Calculation: Sum of story points completed (per DoD) over the last 3–5 sprints, averaged. Example: Sprints 8–10 completed 34, 38, 36 points → velocity ≈ 36 points/sprint.

How QA Uses Velocity:

  • Capacity planning: if velocity is 36 and each story averages 20% QA effort, plan ~7 QA-days of test work
  • Push back on over-commitment: "We averaged 36 pts; committing to 50 risks untested stories"
  • Track quality-adjusted velocity: some teams track "accepted points" vs "committed" to reflect DoD compliance

Sprint Duration Best Practices

  • 2 weeks is the industry sweet spot — long enough to deliver meaningful work, short enough for frequent feedback
  • 1 week suits mature teams with strong CI/CD; overhead of ceremonies proportionally higher
  • 3–4 weeks acceptable for hardware-dependent or compliance-heavy work; risk of late discovery increases
  • Keep duration consistent — changing sprint length breaks velocity comparison
  • QA needs at least 20–30% of sprint capacity reserved for regression and automation maintenance
  • Align sprint end with release cadence if continuous deployment isn't mature

Interview Questions & Answers — Scrum Framework

Q1: What are the three Scrum roles and their responsibilities?

A: Product Owner — maximizes value, owns backlog, sets priorities. Scrum Master — serves team, removes impediments, facilitates Scrum. Developers — cross-functional, self-organizing team that delivers the increment (includes QA).

Q2: What are the five Scrum events?

A: Sprint (container), Sprint Planning, Daily Scrum, Sprint Review, Sprint Retrospective. Backlog Refinement is a complementary practice, not an official event but essential.

Q3: How does QA participate in Sprint Planning?

A: Review AC for testability, break out test tasks, contribute to estimation, flag environment dependencies, and ensure sprint capacity includes adequate test effort and regression time.

Q4: What should QA say in Daily Standup?

A: Yesterday's testing completed, bugs found/fixed, automation progress. Today's test plan. Blockers like environment issues, missing test data, or waiting on bug fixes.

Q5: What is the QA role in Backlog Refinement?

A: Ask clarifying questions, define test scenarios, suggest Given-When-Then AC, identify stories too large to test in one sprint, and estimate testing complexity.

Q6: Explain Definition of Done from a QA perspective.

A: All AC verified, no critical bugs open, automation updated, CI green, test results recorded, code merged. DoD is a team contract — story isn't Done without it.

Q7: What is Definition of Ready and why does QA care?

A: Criteria a story must meet before sprint commitment: clear AC, estimated, testable, dependencies known. QA cares because starting on vague stories causes rework, missed edge cases, and sprint-end crunch.

Q8: How is velocity calculated and used?

A: Average story points completed over 3–5 sprints. Used for capacity planning — not performance comparison. QA uses it to ensure committed work is testable within sprint.

Q9: What is a burndown chart and how does QA use it?

A: Graph of remaining work vs sprint days. QA tracks test task burndown to identify if testing is falling behind development, triggering early intervention (swarming, scope negotiation).

Q10: What is the Sprint Goal?

A: Single objective for the sprint providing focus and flexibility in scope. Example: "Enable mobile checkout." QA aligns test priorities to stories supporting that goal.

Q11: Can the PO change sprint scope mid-sprint?

A: Scrum discourages it. If scope changes dramatically, PO and team renegotiate — swap equal-effort items. QA assesses test impact of any change before agreement.

Q12: What is a servant-leader Scrum Master?

A: SM leads by serving — removing blockers, coaching, facilitating — not by assigning tasks or micromanaging. For QA: SM escalates environment issues and protects DoD standards.

Q13: How do you avoid QA being a bottleneck at sprint end?

A: Test continuously from Day 1; automate regression; enforce DoR; keep stories small; include test tasks in sprint backlog with time estimates; devs merge early; swarm on testing Days 8–9 if needed.

Q14: What happens in Sprint Retrospective relevant to QA?

A: Team reflects on process. QA raises quality concerns: flaky tests, bad AC, environment issues. Action items with owners improve next sprint's quality workflow.

Q15: Difference between Sprint Review and Retrospective?

A: Review = inspect product increment with stakeholders, gather feedback. Retro = inspect team process internally, identify improvements. QA demos quality metrics in Review; proposes process fixes in Retro.

Q16: What are the three Scrum artifacts?

A: Product Backlog, Sprint Backlog, Increment. Each includes transparency; Product Backlog is ordered by PO; Increment must meet DoD.

Q17: How do you estimate story points as QA?

A: Use relative sizing (Fibonacci), considering dev + test + integration complexity. Reference past similar stories. Planning Poker — discuss outliers. QA voice ensures test effort isn't ignored.

Q18: What is the "Increment" in Scrum?

A: All completed backlog items at end of sprint that meet DoD — potentially shippable. QA ensures increment quality through testing, not just "dev complete" code.

Q19: Describe Three Amigos and QA's role.

A: Session with PO (business), Developer (implementation), QA (testing perspective) before development. QA asks edge-case questions, defines AC, identifies test scenarios — prevents defects from ambiguity.

Q20: Ideal sprint length and why?

A: Two weeks for most teams — balances delivery speed with enough time for dev, test, and automation. Consistent length enables reliable velocity. Adjust based on release maturity and regulatory needs.

Section 8: User Stories

User Story Format

A user story is a lightweight requirement written from the user's perspective. The standard template:

As a [type of user/persona],
I want [functionality/capability],
So that [benefit/business value].

Example: "As a registered customer, I want to save items to a wishlist, so that I can purchase them later without searching again."

Stories are intentionally brief — details live in Acceptance Criteria, conversations, and mockups. QA derives test cases from the story + AC, not from the story title alone.

INVEST Principles

PrincipleExplanationExample
I — IndependentStory can be developed and tested without depending on other storiesBad: "Checkout with loyalty points" depends on unreleased loyalty system. Good: Split — "Display points balance" ships independently first
N — NegotiableStory is a conversation starter, not a fixed contractPO says "filter products"; team negotiates: price only in v1, brand filter in v2
V — ValuableDelivers value to user or businessBad: "Refactor database schema." Good (as enabler): "Reduce checkout load time by 50%" with tech tasks linked
E — EstimableTeam can reasonably estimate sizeBad: "Make search better" (vague). Good: "Search by SKU with autocomplete" — team estimates 5 points
S — SmallCompletable within one sprintBad: "Build entire payment module" (epic). Good: "Apply single coupon code at checkout" — 3 points
T — TestableClear pass/fail criteria existBad: "Fast page load." Good: "Search results return in <2 seconds for 10K product catalog"

Acceptance Criteria — Given-When-Then (BDD Style)

Acceptance Criteria define the conditions a story must satisfy to be accepted. BDD format makes them directly translatable to automated tests.

Scenario: Successful price filter
  Given I am on the product listing page
  And the catalog contains products from $10 to $500
  When I set the minimum price to $50 and maximum to $200
  And I click "Apply Filter"
  Then I see only products priced between $50 and $200
  And the filter badge shows "$50 - $200"

Scenario: Invalid price range
  Given I am on the product listing page
  When I set minimum price to $300 and maximum to $100
  Then I see error message "Minimum price cannot exceed maximum price"
  And no filter is applied

Scenario: Empty results
  Given I am on the product listing page
  When I set minimum price to $9999
  And I click "Apply Filter"
  Then I see message "No products match your filters"
  And I see a "Clear Filters" button

Story Points

Fibonacci Scale

Teams use modified Fibonacci: 1, 2, 3, 5, 8, 13, 21. Gaps increase with uncertainty — larger stories are harder to estimate precisely.

Relative Estimation

Story points measure relative effort/complexity, not hours. A 5-point story is roughly twice as complex as a 2-point story. QA complexity (integration, test data, environments) counts in the estimate.

Planning Poker

Each team member selects a card privately, reveals simultaneously. Outliers explain reasoning — QA might say "I voted 8 because this touches 3 microservices and needs contract tests." Team converges on consensus.

How QA Writes Test Cases from User Stories — Step-by-Step

  1. Read the story — understand persona, goal, and business value
  2. Review Acceptance Criteria — each AC should map to at least one test case
  3. Identify test scenarios — happy path, negative, boundary, integration, NFR (performance, security, accessibility)
  4. Apply equivalence partitioning and boundary value analysis — derive data sets from input rules
  5. Write test cases with: ID, title, preconditions, steps, expected result, priority, traceability to story/AC
  6. Review with PO/Dev — Three Amigos or async review for missed edge cases
  7. Prioritize — P1 for AC coverage, P2 for edge cases, P3 for exploratory charters
  8. Automate — convert Given-When-Then AC to Cucumber/Cypress/Playwright specs where ROI is high
  9. Execute and update — log results in Jira/Zephyr; update when AC changes

Interview Questions & Answers — User Stories

Q1: What is a user story and how is it formatted?

A: A brief requirement from the user's perspective: "As a [user], I want [functionality], so that [benefit]." Details are in AC and conversations.

Q2: Explain INVEST principles.

A: Independent, Negotiable, Valuable, Estimable, Small, Testable — criteria for well-formed stories. QA especially cares about Testable and Small.

Q3: What are Acceptance Criteria and how do you write them?

A: Conditions of satisfaction for a story. Use Given-When-Then for clarity and automation. Each AC should be verifiable pass/fail. QA collaborates with PO to write them during refinement.

Q4: What are story points and how do they differ from hours?

A: Relative measure of effort/complexity using Fibonacci scale. Not time — a 5-point story is ~2.5x a 2-point story. Accounts for dev, test, and integration complexity.

Q5: What is Planning Poker?

A: Estimation technique where team members vote privately with cards, reveal together, discuss outliers, and converge. QA ensures test complexity is considered.

Q6: Epic vs Feature vs Story vs Task — explain hierarchy.

A: Epic = large initiative spanning sprints. Feature = cohesive capability within epic. Story = single sprint-sized user value. Task = breakdown work item for dev/QA within a story.

Q7: How do you derive test cases from a user story?

A: Map each AC to test cases; add negative, boundary, integration, and NFR scenarios; apply test design techniques; trace test IDs to story; prioritize and automate high-value paths.

Q8: What makes a user story "testable"?

A: Clear, measurable AC with defined inputs and expected outputs. Avoid subjective terms ("fast," "easy"). Specific thresholds and error messages enable pass/fail determination.

Q9: Give an example of splitting a large story.

A: "User registration" epic splits into: (1) Register with email/password, (2) Email verification, (3) Social login, (4) Profile setup. Each independently deliverable and testable.

Q10: What is the difference between a user story and a use case?

A: User story is brief, user-value focused, with details deferred to conversation. Use case is detailed, step-by-step interaction between actor and system — more formal and comprehensive.

Q11: How do you handle a story with no acceptance criteria?

A: Do not accept into sprint. Flag in refinement; work with PO to define AC before sprint planning. Testing without AC leads to ambiguity disputes and missed defects.

Q12: What is a spike in relation to user stories?

A: Time-boxed research story to answer questions (e.g., "Spike: evaluate payment gateway APIs"). Outcome is knowledge, not shippable feature — informs future stories.

Q13: How does BDD relate to user stories?

A: BDD uses Given-When-Then Gherkin syntax for AC, bridging business language and automated tests. Stories provide context; scenarios provide executable specifications.

Q14: Who writes user stories?

A: Primarily the Product Owner, with input from team. QA contributes testability feedback and AC during refinement — doesn't usually own story authorship.

Q15: How do you estimate a story that is mostly testing effort?

A: Story points cover whole-team effort. If integration testing dominates, vote higher (8 vs 5). Compare to similar past stories. Discuss in Planning Poker — don't estimate in isolation.

Q16: What is story mapping and how does QA use it?

A: Story mapping is a visual technique arranging user activities (backbone) and stories (below) to show user journey and release slices. QA uses it to identify integration test paths across the journey, spot missing edge-case stories, and plan end-to-end test scenarios before sprint commitment.

Q17: How do technical stories differ from user stories?

A: Technical stories (enablers) may use format "As a developer, I want to..." or be written as Tasks. They don't deliver direct user value but enable future stories (e.g., "Set up test data service"). QA validates enablers by verifying downstream stories become testable.

Section 9: Jira Complete Guide

What is Jira?

Jira is Atlassian's issue tracking and project management tool used by Agile teams to plan, track, and release software. It provides boards, backlogs, workflows, dashboards, reporting, and JQL search — making work visible and traceable from idea to production.

Purpose

  • Centralize requirements, bugs, tasks, and test-related work
  • Visualize sprint progress via Scrum/Kanban boards
  • Enforce workflow states (e.g., Ready for QA → Testing → Done)
  • Generate reports: velocity, burndown, cumulative flow
  • Integrate with CI/CD, test management, and collaboration tools

Why Industry Standard

Jira dominates because it is highly configurable (custom workflows, fields, issue types), integrates with the Atlassian ecosystem (Confluence, Bitbucket) and 3000+ marketplace apps (Zephyr, Xray, Jenkins), and scales from startup teams to enterprise SAFe programs. Interview familiarity is expected for Senior QA/SDET roles.

JQL (Jira Query Language)

Basic Queries

project = "CHECKOUT"
status = "Ready for QA"
assignee = currentUser()
priority = Highest

Advanced Queries

project = CHECKOUT AND created >= -7d AND labels = regression
component = Payment AND status in ("Testing", "Ready for QA")
fixVersion = "2026-Q3-Release" AND type = Bug AND priority in (Highest, High)

Jira Dashboards

Gadgets Useful for QA:

  • Filter Results: Display saved JQL — e.g., "Ready for QA" queue
  • Sprint Burndown: Track remaining work in current sprint
  • Sprint Health: Flags scope creep, at-risk items
  • Pie Chart: Bugs by priority, component, or status
  • Created vs Resolved: Bug trend — are we finding more than fixing?
  • Two-Dimensional Filter Statistics: Matrix of component vs priority
  • Activity Stream: Recent changes on watched issues

Best Practices for QA Using Jira

  • Log bugs immediately with complete reproduction steps, screenshots, environment, and severity
  • Link bugs to stories using "blocks" or "relates to" — traceability for release decisions
  • Use subtasks for test work — visible on sprint board, tracked in burndown
  • Apply consistent labels — regression, smoke, automation, flaky — enables powerful JQL filters
  • Transition statuses promptly — don't leave items in "Testing" after completion
  • Comment with test results — "12/12 AC pass, automation PR #445 merged, moving to Done"
  • Create saved filters for daily queues: Ready for QA, My Bugs, Retest Needed
  • Watch critical issues — get notified on status changes without manual checking
  • Don't use Jira as chat — short comments with facts; long discussions in Slack/Confluence with link
  • Keep story points on stories only — not bugs or subtasks (team convention)
  • Review dashboard daily — sprint health, open P1 bugs, items in Testing column
  • Use Zephyr/Xray if available — avoid duplicate test tracking in spreadsheets

Interview Questions & Answers — Jira

Q1: What is Jira and why is it widely used?

A: Atlassian's issue tracking and Agile project management tool. Industry standard due to configurability, Atlassian ecosystem integration, extensive marketplace apps, and support for Scrum, Kanban, and SAFe at scale.

Q2: Difference between Scrum board and Kanban board in Jira?

A: Scrum board uses sprints with sprint backlog and velocity metrics. Kanban board uses continuous flow with WIP limits and cycle time metrics. QA uses Scrum for planned feature work, Kanban for support/production bugs.

Q3: Explain common Jira issue types.

A: Epic (large initiative), Story (user value), Task (technical work), Bug (defect), Spike (research), Subtask (breakdown). QA primarily tests stories, logs bugs, and creates test subtasks.

Q4: What fields are important when logging a bug?

A: Summary, steps to reproduce, expected vs actual, environment (browser/OS/build), severity, priority, screenshots/logs, link to story, assignee.

Q5: What is JQL and give three useful queries for QA.

A: Jira Query Language for searching issues. Examples: status = "Ready for QA", type = Bug AND priority = Highest AND status != Done, sprint in openSprints() AND assignee = currentUser().

Q6: Describe a QA-friendly workflow in Jira.

A: To Do → In Progress → In Review → Ready for QA → Testing → Done. Dev moves to Ready for QA after merge; QA picks up, tests, logs bugs if needed, moves to Done when AC pass.

Q7: What is the difference between Priority and Severity?

A: Severity = technical impact of the bug (Critical to Cosmetic) — usually set by QA. Priority = business urgency for fixing (Highest to Lowest) — usually set by PO/team lead. A cosmetic typo can be Low severity but High priority if on homepage during launch.

Q8: How do you use Jira dashboards as QA?

A: Create personal dashboard with Filter Results (Ready for QA queue), Sprint Burndown, Pie Chart (bugs by priority), and Created vs Resolved trend. Review daily for workload and quality trends.

Q9: What reports help QA identify bottlenecks?

A: Cumulative Flow Diagram — widening "Testing" column indicates QA bottleneck. Burndown — test tasks not decreasing signals falling behind. Sprint Report — scope changes affecting test plan.

Q10: How does Jira integrate with test management tools?

A: Zephyr Scale and Xray are Jira plugins enabling test case creation, linking tests to stories, execution tracking, and coverage reports — all within Jira without separate tools.

Q11: What is a Jira filter and why save them?

A: Saved JQL query reusable for boards, dashboards, and subscriptions. QA saves filters like "Retest Needed," "My Open Bugs," "Sprint Test Queue" for daily efficiency.

Q12: How do you link a bug to a user story in Jira?

A: Use "Link Issue" → "blocks" (bug blocks story completion) or "relates to." Ensures traceability — release decisions consider open blocking bugs on stories.

Q13: What are Components and Labels — when to use each?

A: Components = structured module taxonomy (Checkout, Auth) — one per issue, project-defined. Labels = flexible tags (regression, smoke) — multiple per issue, anyone can add. Both enable JQL filtering.

Q14: How do you track test execution in Jira without Xray/Zephyr?

A: Use subtasks ("Manual test STORY-123"), transition parent story status, comment test results, attach evidence. Less ideal than dedicated test management but workable for small teams.

Q15: What is Fix Version used for?

A: Target release for a bug fix or feature. QA filters fixVersion = "2.5" AND type = Bug AND status != Done to assess release readiness.

Q16: Explain smart commits in Jira.

A: Git commit messages with Jira issue keys trigger automatic actions. Example: git commit -m "CHK-892 Fix tax calculation #done" transitions issue to Done when pushed via Bitbucket/GitHub integration. QA verifies transitions match actual test status — don't auto-close without testing.

Q17: How do you bulk-update issues in Jira?

A: Run a JQL filter → select issues (Bulk Change) → transition, edit fields, or add labels in batch. Useful after regression: label all failed areas regression-sprint-14 or bulk-move verified fixes to Done.

Common Jira Mistakes QA Should Avoid

  • Logging bugs without reproduction steps — causes "Cannot Reproduce" churn
  • Leaving resolved bugs unverified — always retest before closing
  • Creating duplicate bugs — search JQL before filing
  • Not linking bugs to parent story — breaks release traceability
  • Using comments for test evidence without attaching logs/screenshots
  • Transitioning to Done when AC partially met — erodes trust in quality gates
