Section 16: Cucumber & BDD Introduction

What is BDD? (Behavior Driven Development)

Behavior Driven Development (BDD) is an agile software development methodology that encourages collaboration among developers, QA engineers, and business stakeholders by describing application behavior in plain, business-readable language before writing code or tests.

Core Idea: BDD bridges the gap between business requirements and technical implementation. Instead of writing tests in programming language syntax, teams describe what the system should do in a shared vocabulary that everyone understands — POs, BAs, developers, and testers alike.

Key Principles of BDD

  • Outside-in development — Start from user behavior and work inward to implementation.
  • Ubiquitous language — Use the same terms in requirements, tests, and code.
  • Examples over abstractions — Concrete scenarios clarify ambiguous requirements.
  • Living documentation — Feature files stay synchronized with actual system behavior.
  • Collaboration — Three Amigos (PO, Dev, QA) define behavior together before automation.

How BDD Bridges Business and Technology

Traditional testing often creates a translation gap: business writes requirements in Word/Jira, developers write code in Java, and QA writes test scripts in Selenium. BDD eliminates this gap by making Gherkin feature files the single source of truth that all three groups can read, review, and approve.

What is Cucumber?

Cucumber is an open-source, Java-based BDD framework that executes plain-text specifications written in Gherkin syntax and maps them to executable step definitions written in Java (or other JVM languages like Kotlin, Scala).

  • Originally created by Aslak Hellesøy in 2008; now maintained by the Cucumber open-source community.
  • Supports multiple languages: Java, JavaScript, Ruby, Python (.NET via SpecFlow).
  • Integrates with TestNG, JUnit, Maven, Gradle, Selenium, RestAssured, and CI/CD pipelines.
  • Current stable version: Cucumber 7.x (JUnit Platform / TestNG integration).

Why Cucumber?

Business-Readable Tests

Feature files use plain English (or any natural language). A Product Owner can read Given the user is on the login page without knowing Java.

Living Documentation

Feature files describe current system behavior. When behavior changes, scenarios are updated — documentation never goes stale like Word documents.

Collaboration

Three Amigos sessions produce feature files together. Everyone agrees on behavior before coding starts, reducing rework and miscommunication.

Traceability

Each scenario maps to a user story. Tags like @JIRA-1234 link tests to tickets for audit and coverage tracking.

Advantages of Cucumber (10+ Points)

  • Business-readable specifications — Non-technical stakeholders can review and approve test scenarios.
  • Living documentation — Feature files always reflect current system behavior.
  • Improved collaboration — Three Amigos sessions align PO, Dev, and QA early.
  • Reusable step definitions — Same step text reused across multiple scenarios.
  • Tag-based execution — Run smoke, regression, or sprint-specific subsets selectively.
  • Multiple report formats — HTML, JSON, JUnit XML, Allure integration out of the box.
  • Data-driven testing — Scenario Outline with Examples tables for parameterized tests.
  • CI/CD friendly — Integrates with Jenkins, GitHub Actions via Maven/Gradle commands.
  • Multi-language support — Same Gherkin syntax works across Java, JS, Ruby, Python.
  • Parallel execution — TestNG parallel DataProvider runs scenarios concurrently.
  • Hooks support — @Before/@After for setup, teardown, screenshots, and conditional hooks by tag.
  • Requirement traceability — Tags link scenarios to Jira tickets and user stories.

Limitations of Cucumber (5+ Points)

  • Abstraction overhead — Maintaining feature files AND step definitions doubles maintenance effort.
  • Learning curve — Teams must learn Gherkin syntax, Cucumber architecture, and DI patterns.
  • Performance — Slower than direct TestNG tests due to Gherkin parsing and step matching.
  • Over-engineering risk — Simple unit tests don't need BDD; forcing Cucumber adds unnecessary complexity.
  • Step definition sprawl — Large projects accumulate hundreds of steps; ambiguous/duplicate steps cause failures.
  • False sense of collaboration — If only QA writes feature files, BDD benefits are lost.
  • Debugging difficulty — Failures span feature file → step definition → framework code; stack traces are longer.

Interview Q&A — Cucumber & BDD Introduction

Q1. What is BDD and how is it different from TDD?

Answer: BDD focuses on describing system behavior in business-readable language (Gherkin) for acceptance testing with stakeholder collaboration. TDD focuses on writing unit tests in programming language before code, driven by developers for design quality. BDD operates at a higher abstraction level and often uses TDD internally for step definition implementation.

Q2. What problem does Cucumber solve?

Answer: Cucumber bridges the communication gap between business and technical teams. It allows requirements to be written as executable tests in plain language, ensuring everyone agrees on behavior before development and that tests serve as always-current documentation.

Q3. What is Gherkin?

Answer: Gherkin is a structured plain-text language with keywords (Feature, Scenario, Given, When, Then, And, But) used to describe test scenarios in a business-readable format. Cucumber parses Gherkin and executes matching step definitions.

Q4. What are the main components of Cucumber framework?

Answer: Feature files (Gherkin scenarios), step definitions (Java glue code), runner class (@CucumberOptions entry point), hooks (@Before/@After), and dependency injection (PicoContainer) for state sharing.

Q5. Can Cucumber be used without Selenium?

Answer: Yes. Cucumber is a BDD test runner, not a browser automation tool. Step definitions can call REST APIs (RestAssured), database queries, unit test assertions, or any Java code. Selenium is commonly used for UI steps but is not required.

Q6. What is the role of a feature file?

Answer: A feature file describes a feature's behavior through scenarios written in Gherkin. It serves as both an executable test specification and living documentation that business stakeholders can read and approve.

Q7. What is glue code in Cucumber?

Answer: Glue code refers to step definition classes that connect Gherkin steps to executable Java methods. The glue attribute in @CucumberOptions specifies the package(s) containing these classes.

Q8. Is BDD suitable for all projects?

Answer: No. BDD adds value when business stakeholders actively participate and acceptance tests need to be readable. For API-only backends, performance testing, or QA-only teams, traditional TestNG/JUnit frameworks are more efficient.

Q9. What is living documentation?

Answer: Living documentation is test specifications that stay synchronized with the application. When behavior changes, feature files are updated alongside code. Unlike static Word documents, living docs are always accurate because they must pass to be considered valid.

Q10. How does Cucumber integrate with TestNG?

Answer: Create a runner class extending AbstractTestNGCucumberTests with @CucumberOptions. TestNG treats each scenario as a test method. Override the scenarios() DataProvider for parallel execution.

Q11. What is the difference between Cucumber and SpecFlow?

Answer: Cucumber is the original BDD framework supporting Java, JS, Ruby, Python. SpecFlow is the .NET port providing identical Gherkin syntax for C# projects. Both follow the same BDD principles and Gherkin specification.

Q12. What are hooks in Cucumber?

Answer: Hooks are methods annotated with @Before, @After, @BeforeStep, @AfterStep that run at defined points in the scenario lifecycle. They handle setup, teardown, screenshots, and can be conditional using tag expressions like @Before("@ui").

Three Amigos Meeting

The Three Amigos is a structured collaboration session involving three perspectives before development begins:

RoleContributionQuestions They Ask
Product Owner (PO)Business rules, priorities, acceptance criteria"What should happen when the user enters an expired coupon?"
Developer (Dev)Technical feasibility, edge cases, implementation approach"Can we validate the coupon against an external API or only our DB?"
QA / SDETTest scenarios, negative cases, data requirements"What happens with special characters in the coupon code field?"
Outcome: A shared understanding documented as Gherkin scenarios that all three roles agree represent the intended behavior. No surprises during sprint review.

Discovery → Formulation → Automation Pattern

1. Discovery

  • Explore the user story through conversation and concrete examples.
  • Ask "What if?" questions to uncover edge cases and business rules.
  • Do NOT write Gherkin yet — focus on understanding.
  • Output: agreed examples on a whiteboard or Example Mapping cards.

2. Formulation

  • Convert discovered examples into formal Gherkin scenarios.
  • One scenario = one behavior. Keep scenarios independent.
  • Use Scenario Outline when multiple data combinations test the same behavior.
  • Review with PO for sign-off before automation begins.

3. Automation

  • Implement step definitions mapping to Gherkin steps.
  • Run with dryRun = true first to verify step coverage.
  • Integrate with CI pipeline and tag appropriately.
  • Refactor step definitions for reusability — avoid duplication.

Living Documentation Concept

Living documentation means your test specifications ARE your documentation. Feature files describe what the system does in business language, and they must pass to be valid — so they can never drift out of date.

  • Traditional docs: Written once, outdated within weeks, nobody reads them.
  • Living docs: Updated with every behavior change, executable, always accurate.
  • Published reports: Cucumber HTML reports and Allure dashboards serve as browsable documentation for stakeholders.
  • Cucumber Reports (cloud): Upload JSON results to reports.cucumber.io for team-wide visibility.

BDD Anti-Patterns (What NOT to Do)

  • QA-only BDD — Only QA writes feature files without PO/Dev involvement. Defeats the collaboration purpose.
  • Technical Gherkin — Writing Given I click element #login-btn instead of Given the user is on the login page.
  • Scenario bloat — One scenario with 20 steps testing multiple behaviors. Split into independent scenarios.
  • Imperative steps — When I enter "john" in field 1 and "pass" in field 2 and click submit — too UI-specific.
  • Duplicate step definitions — Multiple Java methods matching the same Gherkin pattern causing ambiguous step errors.
  • Skipping dryRun — Running full execution before verifying all steps have definitions wastes CI time.
  • Feature file as test script — Treating Gherkin like a manual test case with every click documented.
  • No tag strategy — Running all scenarios every time instead of using @smoke, @regression tags.
  • Static test data in features — Hardcoding production-like data instead of using fixtures or factories in step definitions.
  • Ignoring failures — Tagging broken scenarios as @skip permanently instead of fixing or removing them.

Interview Q&A — BDD Framework Lifecycle

Q1. What are the phases of the BDD lifecycle?

Answer: Requirement → User Story → Feature File → Scenario → Step Definition → Execution → Report. Before automation, teams go through Discovery (understand behavior), Formulation (write Gherkin), and Automation (implement step definitions).

Q2. What is a Three Amigos meeting?

Answer: A collaboration session with Product Owner, Developer, and QA/Tester to discuss a user story, explore examples, identify edge cases, and agree on acceptance criteria before writing feature files or code.

Q3. What is Example Mapping?

Answer: A structured technique using colored cards (rules, examples, questions, assumptions) during Three Amigos sessions to break down stories into concrete examples that become Gherkin scenarios.

Q4. Explain Discovery, Formulation, Automation.

Answer: Discovery is conversational exploration of behavior with examples. Formulation converts examples into Gherkin scenarios. Automation implements step definitions and integrates with CI. Each phase has clear entry/exit criteria.

Q5. How does BDD fit in Agile?

Answer: Feature files are created early in the sprint from user stories. Scenarios serve as acceptance criteria, definition of done, and demo evidence. CI runs tagged suites on every commit for continuous validation.

Q6. What is living documentation?

Answer: Executable specifications that stay current with the application. Feature files must pass to be valid, so they cannot become outdated like traditional documentation. Reports provide browsable behavior catalogs.

Q7. What are common BDD anti-patterns?

Answer: QA-only feature writing, UI-specific Gherkin, bloated scenarios, duplicate step definitions, no tag strategy, and treating feature files as manual test scripts instead of behavior descriptions.

Q8. Who should write feature files?

Answer: Ideally the whole Three Amigos team collaboratively. PO provides business rules, QA provides test perspective, Dev confirms feasibility. In practice, QA/SDET often drafts them but PO must review and approve.

Q9. When should you NOT use BDD?

Answer: When there is no business stakeholder involvement, for unit-level testing, performance/load testing, or when the team lacks Gherkin discipline and feature files become unmaintainable UI scripts.

Q10. How do you handle changing requirements in BDD?

Answer: Update the feature file first during Three Amigos review, then update step definitions. The feature file change serves as the specification change request. Version control tracks all modifications.

Section 18: Gherkin Language

Gherkin is the structured plain-text language used by Cucumber to describe software behavior. It uses a defined set of keywords to create readable, executable specifications.

Gherkin Keywords Reference

Feature

Syntax: Feature: <short description> — High-level description of functionality being tested. One Feature keyword per .feature file.

Usage Rules:

  • First keyword in every feature file (after optional comments/tags).
  • Followed by an optional multi-line description explaining business value.
  • Groups related scenarios under one functional area.
Feature: User Login
  As a registered user
  I want to log in to my account
  So that I can access personalized content

  Scenario: Successful login with valid credentials
    ...

Background

Syntax: Background: followed by Given/When/Then steps — Common preconditions executed before every scenario in the feature.

Usage Rules:

  • Runs before EACH scenario in the feature (not once per feature).
  • Use for shared setup like navigating to a page or logging in.
  • Keep minimal — heavy setup slows every scenario.
  • Only one Background per feature file.
Background:
  Given the user is on the login page
  And the login service is available

Scenario

Syntax: Scenario: <description> — A specific test case with Given/When/Then steps.

Usage Rules:

  • One scenario = one behavior/outcome.
  • Scenarios must be independent — no execution order dependency.
  • Use descriptive names: "Successful login with valid credentials" not "Test 1".
Scenario: Reject login with invalid password
  Given the user is on the login page
  When the user enters username "john@example.com"
  And the user enters password "wrongpassword"
  And the user clicks the login button
  Then an error message "Invalid credentials" is displayed

Scenario Outline

Syntax: Scenario Outline: with <placeholder> variables and an Examples: table.

Usage Rules:

  • Template for data-driven scenarios — same steps, different data.
  • Placeholders in angle brackets: <username>, <password>.
  • Cucumber runs one scenario per row in the Examples table.
Scenario Outline: Login with various credentials
  Given the user is on the login page
  When the user enters username "<username>"
  And the user enters password "<password>"
  And the user clicks the login button
  Then the user should see "<expected_result>"

  Examples:
    | username          | password    | expected_result      |
    | john@example.com  | Pass@123    | Welcome dashboard    |
    | invalid@test.com  | wrong       | Invalid credentials  |

Examples

Syntax: Examples: followed by a data table with column headers matching Scenario Outline placeholders.

Usage Rules:

  • Must follow a Scenario Outline (not a regular Scenario).
  • Column headers become placeholder values (without angle brackets).
  • Multiple Examples tables allowed in one Scenario Outline (e.g., grouped by browser).
  • First row is always the header row.

Given

Syntax: Given <precondition> — Sets up the initial context/state.

Usage Rules:

  • Describes state, not actions: "the user is logged in" not "the user logs in".
  • First step in most scenarios (after Background).
  • Use business language: "Given a premium subscription exists" not "Given DB row inserted".
Given the user is on the login page
Given the cart contains 3 items
Given the user has a valid authentication token

When

Syntax: When <action> — Describes the action or event being tested.

Usage Rules:

  • Describes a single key action or event.
  • Use active voice: "When the user submits the order".
  • Avoid UI specifics: "When the user searches for 'laptop'" not "When the user clicks #search-btn".
When the user clicks the login button
When the user searches for "wireless headphones"
When the user applies coupon code "SAVE20"

Then

Syntax: Then <expected outcome> — Asserts the expected result.

Usage Rules:

  • Must be verifiable and observable.
  • One primary outcome per scenario (additional assertions use And).
  • Describe what the user/system sees: "Then the order confirmation is displayed".
Then the user should be redirected to the dashboard
Then the error message "Out of stock" is displayed
Then the order total should be "$89.99"

And

Syntax: And <additional step> — Continues the previous step type (Given, When, or Then).

Usage Rules:

  • Inherits the type of the preceding step (Given/When/Then).
  • Improves readability over repeating Given/When/Then.
  • Can appear multiple times in a scenario.
Given the user is on the checkout page
And the cart contains 2 items
And the user is logged in
When the user selects express shipping
And the user clicks "Place Order"
Then the order confirmation is displayed
And the confirmation email is sent

But

Syntax: But <negative additional step> — Negative continuation of the previous step type.

Usage Rules:

  • Same inheritance rule as And, but for negative assertions or exclusions.
  • Used less frequently than And.
  • Common in Then clauses: "But the discount should not be applied".
Then the search results should contain "laptop"
But should not contain "refurbished" items

Data Tables

Syntax: Pipe-delimited tables passed as arguments to steps.

Usage Rules:

  • Used for structured data within a single step (not Scenario Outline).
  • First row can be headers (passed to DataTable.asMaps()).
  • Ideal for form submissions, menu selections, or multi-field validation.
When the user fills in the registration form:
  | Field           | Value              |
  | First Name      | John               |
  | Last Name       | Doe                |
  | Email           | john@example.com   |
  | Password        | SecurePass@123     |
Then the registration should be successful

Doc Strings

Syntax: Triple quotes """ for multi-line text passed to a step.

Usage Rules:

  • Used for long text: JSON payloads, email bodies, SQL queries, API request bodies.
  • Content between """ delimiters is passed as a single String parameter.
  • Can use content type hint: """json for JSON doc strings.
When the user submits the following JSON payload:
  """
  {
    "productId": "SKU-12345",
    "quantity": 2,
    "couponCode": "SAVE10"
  }
  """
Then the API should return status code 200

Tags

Syntax: @tagname on the line before Feature, Scenario, or Scenario Outline.

Usage Rules:

  • Multiple tags per scenario: @smoke @login @JIRA-1234.
  • Feature-level tags inherit to all scenarios in the feature.
  • Used for selective execution, hooks, and traceability.
  • Common: @smoke, @regression, @wip, @skip, @critical.
@smoke @login
Scenario: Successful login with valid credentials
  Given the user is on the login page
  ...

Gherkin Best Practices (10+ Points)

  • Write in business language — Use domain terms stakeholders understand, not CSS selectors or XPath.
  • One scenario = one behavior — Test a single outcome. Split complex flows into multiple scenarios.
  • Keep scenarios independent — Each scenario must run in isolation without depending on another scenario's state.
  • Avoid UI-specific language — "When the user searches for laptops" not "When the user clicks the search icon and types in #search-input".
  • Use Background sparingly — Only for truly shared preconditions. Heavy Backgrounds slow every scenario.
  • Descriptive scenario names — Names should explain the behavior: "Reject expired coupon at checkout".
  • Limit steps per scenario — Aim for 3–7 steps. More indicates the scenario tests too much.
  • Use Scenario Outline for data variations — Same behavior with different data belongs in Scenario Outline, not duplicate scenarios.
  • Tag consistently — Establish team conventions for @smoke, @regression, @wip and enforce them.
  • Don't test implementation details — Focus on behavior and outcomes, not internal API calls or database states.
  • Review with PO before automation — Feature files are specifications. Get sign-off before writing step definitions.
  • Avoid conjunctive steps — "When the user enters username and password and clicks login" should be three separate steps.
  • Keep feature files focused — One feature file per functional area. Don't combine login and checkout in one file.

Interview Q&A — Gherkin Language

Q1. What are the primary Gherkin keywords?

Answer: Feature, Background, Scenario, Scenario Outline, Examples, Given, When, Then, And, But. Given sets context, When describes action, Then asserts outcome. And/But continue the preceding step type.

Q2. What is the difference between Scenario and Scenario Outline?

Answer: Scenario is a single test case with fixed data. Scenario Outline is a template with placeholders filled by an Examples table, creating multiple scenario executions from one template.

Q3. What is the purpose of Background?

Answer: Background defines steps that run before every scenario in a feature file. It eliminates duplication of common preconditions like navigating to a page or ensuring a service is available.

Q4. How do Data Tables differ from Examples tables?

Answer: Data Tables pass structured data to a single step within a scenario. Examples tables drive Scenario Outline placeholders, creating multiple scenario executions. Data Tables are for one step; Examples are for entire scenarios.

Q5. What are Doc Strings used for?

Answer: Doc Strings (triple quotes) pass multi-line text to a step definition — commonly JSON payloads, XML, email bodies, or long text content that doesn't fit on one line.

Q6. Can you have multiple Background sections?

Answer: No. Only one Background is allowed per feature file. If you need different setups, use separate feature files or handle setup in individual scenario Given steps.

Q7. What happens if a scenario has no Then step?

Answer: Cucumber allows it, but it's a bad practice. Every scenario should have at least one Then step asserting an observable outcome. Scenarios without assertions don't validate behavior.

Q8. How do tags work at the Feature level?

Answer: Tags placed above the Feature keyword are inherited by all scenarios and scenario outlines in that file. Individual scenarios can have additional tags. Feature tags + scenario tags are combined for filtering.

Q9. What is the difference between And and But?

Answer: Both continue the preceding step type (Given/When/Then). And is for additional steps of the same polarity. But is for negative/additional exclusion assertions, typically after Then.

Q10. How many Feature keywords can a .feature file have?

Answer: Exactly one. Each .feature file describes one feature. Multiple features require multiple .feature files.

Q11. Can Gherkin steps be written in languages other than English?

Answer: Yes. Cucumber supports 70+ languages. Use # language: fr as the first line for French keywords (Fonctionnalité, Étant donné, etc.). The step definition annotations remain in English.

Q12. What makes a good Gherkin scenario?

Answer: Independent, focused on one behavior, written in business language, 3–7 steps, descriptive name, clear Given/When/Then structure, and verifiable Then assertion. Reviewed and approved by PO before automation.

Background Usage: Shared Preconditions

Background steps execute before every scenario in the feature. Use when all scenarios share the same starting context.

When to use Background: Navigation to a page, ensuring a service is up, setting up test data that ALL scenarios need. Do NOT use Background for scenario-specific setup — use Given steps instead.

Scenario Independence Principle

Each scenario must be executable in any order without depending on another scenario's side effects.

  • Do: Set up required state in Given steps or Background.
  • Don't: Assume Scenario 2 runs after Scenario 1 created data.
  • Do: Clean up in @After hooks if scenarios modify shared resources.
  • Don't: Use static variables to pass data between scenarios.

Best Practices for Maintainable Feature Files

  • Keep feature files under 300 lines — split large features into sub-features.
  • One Feature keyword per file — never combine unrelated features.
  • Use consistent step phrasing across all feature files in the project.
  • Tag every scenario with at least @smoke or @regression for selective execution.
  • Link to user stories via @JIRA-XXXX tags for traceability.
  • Review feature files in pull requests — treat them as production code.
  • Avoid hardcoded environment-specific data (URLs, credentials) — use config in step definitions.
  • Delete or update scenarios when features are removed — don't leave orphaned tests.

Interview Q&A — Feature Files

Q1. Where should feature files be located in a Maven project?

Answer: src/test/resources/features/ is the standard location. The path is specified in the runner's @CucumberOptions features attribute. Subdirectories organize by module.

Q2. Can one feature file contain multiple Feature keywords?

Answer: No. Each .feature file must have exactly one Feature keyword. Multiple features require multiple files.

Q3. What is the scenario independence principle?

Answer: Each scenario must run independently in any order. Setup belongs in Given steps or Background, not in dependencies on other scenarios' execution or side effects.

Q4. How do you organize feature files in a large project?

Answer: By application feature/module in subdirectories under features/. Tag with @module tags. Separate API and UI features. Link to Jira stories with @JIRA tags for traceability.

Q5. When should you use Background vs Given?

Answer: Background for preconditions shared by ALL scenarios in a feature. Given for scenario-specific setup. If only 2 of 5 scenarios need a precondition, use Given in those scenarios, not Background.

Q6. How do Data Tables work in feature files?

Answer: Pipe-delimited tables passed as DataTable objects to step definitions. Use asMaps() for header-based key-value rows, asLists() for single-column lists, asList() for flat lists.

Q7. What naming convention do you follow for feature files?

Answer: Lowercase with underscores matching the Feature name: "User Login" → user_login.feature. No spaces. Organized in module subdirectories.

Q8. How do you handle test data in feature files?

Answer: Use Scenario Outline Examples for data variations. Data Tables for form data. Avoid hardcoded production data — step definitions should create test data via APIs or factories. Sensitive data goes in config files, not feature files.

Section 20: Step Definitions

What are Step Definitions?

Step definitions are Java methods annotated with Cucumber annotations (@Given, @When, @Then) that map Gherkin steps from feature files to executable code. Each Gherkin step in a scenario must have exactly one matching step definition method.

Glue Code: Step definitions are often called "glue code" because they glue the human-readable Gherkin specifications to the automation framework (Selenium, RestAssured, etc.).

Step Definition Class Structure

package stepdefinitions;

import io.cucumber.java.en.*;
import pages.LoginPage;
import org.testng.Assert;

public class LoginSteps {

    private LoginPage loginPage;
    private TestContext testContext;

    public LoginSteps(TestContext testContext) {
        this.testContext = testContext;
        this.loginPage = testContext.getLoginPage();
    }

    @Given("the user is on the login page")
    public void userIsOnLoginPage() {
        loginPage.navigateTo("/login");
    }

    @When("the user enters email {string}")
    public void userEntersEmail(String email) {
        loginPage.enterEmail(email);
        testContext.setEmail(email);
    }

    @Then("the user should be redirected to {string}")
    public void userShouldBeRedirected(String expectedUrl) {
        Assert.assertTrue(loginPage.getCurrentUrl().contains(expectedUrl));
    }
}

    private final SearchPage searchPage;
    private List<String> searchResults;

    public SearchSteps(TestContext context) {
        this.searchPage = context.getSearchPage();
    }

    @Given("the user is on the homepage")
    public void userOnHomepage() {
        searchPage.navigateTo("/");
    }

    @Given("the product catalog contains {int} items")
    public void catalogContainsItems(int count) {
        Assert.assertTrue(searchPage.getCatalogCount() >= count);
    }

    @When("the user searches for {string}")
    public void userSearches(String keyword) {
        searchPage.enterSearchTerm(keyword);
        searchPage.clickSearchButton();
        searchResults = searchPage.getResultTitles();
    }

    @When("the user applies price filter from {string} to {string}")
    public void applyPriceFilter(String min, String max) {
        searchPage.setPriceRange(min, max);
        searchResults = searchPage.getResultTitles();
    }

    @Then("the search results should contain at least {int} product")
    public void verifyMinResults(int minCount) {
        Assert.assertTrue(searchResults.size() >= minCount);
    }

    @Then("each result should contain {string} in the title")
    public void verifyResultsContainKeyword(String keyword) {
        searchResults.forEach(title ->
            Assert.assertTrue(title.toLowerCase().contains(keyword.toLowerCase()))
        );
    }

    @Then("the message {string} should be displayed")
    public void verifyMessage(String expectedMessage) {
        Assert.assertEquals(searchPage.getNoResultsMessage(), expectedMessage);
    }

    @Then("all displayed products should be priced between ${int} and ${int}")
    public void verifyPriceRange(int min, int max) {
        searchPage.getResultPrices().forEach(price -> {
            Assert.assertTrue(price >= min && price <= max);
        });
    }
}

import io.cucumber.datatable.DataTable;
import java.util.List;
import java.util.Map;

public class RegistrationSteps {

    // Header row becomes map keys
    @When("the user fills in the registration form:")
    public void fillForm(DataTable dataTable) {
        List<Map<String, String>> rows = dataTable.asMaps();
        for (Map<String, String> row : rows) {
            String field = row.get("Field");
            String value = row.get("Value");
            registrationPage.enterField(field, value);
        }
    }

    // Two-column table as single map
    @When("the user enters shipping address:")
    public void enterAddress(DataTable dataTable) {
        Map<String, String> address = dataTable.asMap(String.class, String.class);
        checkoutPage.fillAddress(
            address.get("Street"),
            address.get("City"),
            address.get("Zip")
        );
    }

    // Single column as list
    @When("the user selects the following categories:")
    public void selectCategories(DataTable dataTable) {
        List<String> categories = dataTable.asList();
        categories.forEach(cat -> filterPage.selectCategory(cat));
    }

    // Raw rows without header treatment
    @Given("the following products are in stock:")
    public void productsInStock(DataTable dataTable) {
        List<List<String>> rows = dataTable.asLists();
        rows.remove(0); // skip header
        rows.forEach(row -> inventoryService.ensureStock(
            row.get(0), row.get(1), Integer.parseInt(row.get(3))

Step Definition Best Practices

  • Reuse steps — Write generic steps like the user clicks the {string} button instead of one step per button.
  • Keep step definitions thin — Delegate to Page Objects or service classes; don't put Selenium code directly in steps.
  • One responsibility — Each step definition method should do one thing.
  • Consistent phrasing — Agree on step wording across the team and document in a step catalog.
  • Use DI for state — PicoContainer or ScenarioContext, never static variables.
  • Avoid assertions in When steps — When steps perform actions; Then steps perform assertions.
  • Parameter types over regex — Prefer Cucumber Expressions for readability.

Interview Q&A — Step Definitions

Q1. What are step definitions?

Answer: Java methods annotated with @Given, @When, @Then that map Gherkin steps to executable code. They form the glue between feature files and the automation framework.

Q2. What is glue code?

Answer: Another name for step definitions. The glue package (specified in @CucumberOptions) contains all step definition and hook classes that Cucumber scans for matching step patterns.

Q3. Cucumber Expressions vs Regex — which do you prefer?

Answer: Cucumber Expressions ({string}, {int}, {word}) since Cucumber 4. They are readable, self-documenting, and less error-prone. Regex is legacy but still supported for complex patterns.

Q4. How do you share data between step definition classes?

Answer: PicoContainer dependency injection — add cucumber-picocontainer, create a TestContext class, inject it via constructor into all step classes. Each scenario gets a fresh instance.

Q5. How do you handle Data Tables in step definitions?

Answer: Accept DataTable as method parameter. Use asMaps() for header-based rows, asMap() for two-column key-value, asList() for single column, asLists() for raw rows.

Q6. What causes "Duplicate step definition" error?

Answer: Two or more methods in glue packages match the same Gherkin step pattern. Cucumber cannot determine which to invoke. Remove or differentiate the conflicting patterns.

Q7. What is dryRun in Cucumber?

Answer: When dryRun=true, Cucumber verifies all Gherkin steps have matching step definitions without executing them. Useful for validating coverage before full test runs.

Q8. Should step definitions contain Selenium code?

Answer: No. Step definitions should be thin — delegate to Page Object Model classes or service layers. This keeps steps maintainable and reusable across UI changes.

Q9. How do you handle optional parameters in steps?

Answer: Cucumber Expressions support optional text: the user(s) matches "the user" or "the users". For complex optionality, create separate step definitions.

Q10. Can one step definition match multiple Gherkin steps?

Answer: Yes, using alternative text in Cucumber Expressions: @When("the user clicks/taps the {string} button"). Or use multiple annotations on the same method.

Q11. What happens if a step definition throws an exception?

Answer: The scenario fails immediately. Cucumber reports the failing step, exception message, and stack trace in the test report. Subsequent steps in the scenario are skipped.

Q12. How do you organize step definition classes?

Answer: One class per feature domain: LoginSteps, SearchSteps, CheckoutSteps. All in the glue package. Shared utilities in TestContext or helper classes, not duplicated across step classes.

Interview Q&A — Cucumber Runner

Q1. What is the purpose of the Cucumber runner class?

Answer: Entry point that configures Cucumber execution — feature file paths, glue packages, tag filters, report plugins, and integration with TestNG/JUnit test engines.

Q2. How do you run Cucumber tests in parallel with TestNG?

Answer: Override the scenarios() method with @DataProvider(parallel = true) in a class extending AbstractTestNGCucumberTests. Configure thread count in TestNG suite XML or Maven surefire plugin.

Q3. What does dryRun=true do?

Answer: Cucumber parses feature files and verifies every Gherkin step has a matching step definition without executing any test code. Reports undefined and ambiguous steps.

Q4. How do you override tags at runtime?

Answer: Pass -Dcucumber.filter.tags="@smoke" on the Maven command line. Runtime tag filter overrides the tags attribute in @CucumberOptions.

Q5. What plugins do you use for reporting?

Answer: pretty for console, html for stakeholder reports, json for Allure/custom dashboards, junit for CI parsing. Allure plugin (io.qameta.allure.cucumber7jvm.AllureCucumber7Jvm) for rich reports with screenshots.

Q6. Can you have multiple runner classes?

Answer: Yes. Common pattern: SmokeTestRunner, RegressionTestRunner, and feature-specific runners. Each with different tags and plugin outputs. Run selectively via -Dtest=ClassName.

Q7. What is the glue attribute?

Answer: Specifies package(s) where Cucumber scans for step definitions and hooks. Multiple packages as array: glue = {"stepdefinitions", "hooks", "utils"}.

Q8. How do you integrate Cucumber with Jenkins?

Answer: Maven command with tag filter parameter, publish HTML report as Jenkins artifact, parse JUnit XML for test results trend, optionally use Cucumber Reports plugin for cucumber.json.

Q9. What is monochrome=true?

Answer: Disables ANSI color codes in console output. Useful for CI environments that don't render colors properly, producing cleaner log files.

Q10. What happens when strict mode finds undefined steps?

Answer: In Cucumber 6 and earlier, strict=true caused build failure on undefined/pending steps. In Cucumber 7+, undefined steps always fail by default; strict is deprecated.

Q11. How do you run only failed scenarios?

Answer: Cucumber doesn't natively support rerun of failures. Workarounds: use @rerun tag manually, third-party plugins, or TestNG retry analyzer on the runner class.

Q12. What is the difference between features path as file vs directory?

Answer: Directory path runs all .feature files recursively. File path runs only that file. Use directory for full suite, file path for focused testing during development.

What is Scenario Outline?

A Scenario Outline is a template for data-driven scenarios. Instead of writing separate scenarios for each data combination, you write one template with placeholders and provide data in an Examples table. Cucumber executes one scenario per row in the Examples table.

Key Concept: Scenario Outline = Scenario + Examples table. The outline defines steps with <placeholder> variables. The Examples table provides values for each execution.

Scenario Outline: <title with placeholders>
  Given ...

  Then ... "<expected>" ...

  Examples:
    | placeholder | expected |
    | value1      | result1  |
    | value2      | result2  |

Interview Q&A — Scenario Outline & Data-Driven BDD

Q1. What is Scenario Outline?

Answer: A parameterized scenario template with placeholders (<name>) filled by an Examples table. Cucumber runs one scenario execution per row, enabling data-driven BDD testing.

Q2. How do placeholders work?

Answer: Placeholders in angle brackets in the Scenario Outline match column headers in the Examples table. Each row provides values substituted into all steps before execution.

Q3. Can a Scenario Outline have multiple Examples tables?

Answer: Yes. Each named Examples section adds rows. Useful for grouping data by browser, environment, or test category. All rows execute as separate scenarios.

Q4. Scenario Outline vs Data Tables — when to use which?

Answer: Scenario Outline when the entire scenario repeats with different data (login attempts). Data Tables when structured data is passed to a single step (form fields, product lists).

Q5. How many Examples rows is too many?

Answer: Beyond 50 rows, consider external data sources (CSV, database) with TestNG DataProvider. Inline Examples are best for 5–30 business-meaningful combinations reviewable by PO.

Q6. Do Examples table values support expressions?

Answer: No. Examples values are literal strings substituted into steps. Calculations and dynamic data should happen in step definitions, not in the Examples table.

Q7. How are Scenario Outline rows reported?

Answer: Each row appears as a separate scenario in Cucumber reports: "Login attempt with various credentials — Example #1.1". Failures pinpoint the exact data row.

Q8. Can placeholders appear in Scenario Outline title?

Answer: Yes. Scenario Outline: Login with <email> substitutes the value in reports, making failures easier to identify.

Q9. How do you skip specific Examples rows?

Answer: Tag individual Examples sections (not rows). To skip rows, remove them or comment out. Cucumber doesn't support row-level tags within Examples.

Q10. Scenario Outline with TestNG parallel execution?

Answer: Override scenarios() with @DataProvider(parallel = true). Each Examples row runs as an independent parallel test. Configure thread count in suite XML or surefire plugin.

Q11. What is the difference between Scenario Outline and repeating Scenarios?

Answer: Scenario Outline avoids duplication — one step template, multiple data rows. Separate scenarios duplicate steps, increasing maintenance when steps change. Outline is DRY; separate scenarios are more readable for vastly different flows.

Q12. Can Examples data come from external files?

Answer: Not natively in Gherkin. Examples must be inline in the feature file. For external data, use TestNG DataProvider or custom Cucumber plugins. Some teams generate feature files from CSV in build pipelines.

What are Tags?

Tags are labels attached to Feature files and Scenarios that serve as metadata for organizing, filtering, and controlling test execution. Tags enable selective running of test subsets without modifying feature files or step definitions.

Syntax: Tags start with @ and appear on the line immediately before Feature, Scenario, or Scenario Outline. Multiple tags are space-separated: @smoke @login @JIRA-1234.

    ...

Tag-Based Hooks

Hooks can be conditional on tags, running only for scenarios matching the tag expression:

public class Hooks {

    @Before("@ui")
    public void setUpBrowser() {
        DriverFactory.initBrowser("chrome");
    }

    @Before("@api")
    public void setUpApiContext() {
        RestAssured.baseURI = Config.getBaseUrl();
    }

    @After("@ui")
    public void tearDownBrowser(Scenario scenario) {
        if (scenario.isFailed()) {
            Screenshots.capture(scenario);
        }
        DriverFactory.quitDriver();
    }

    @Before("@db")
    public void setUpDatabase() {
        DatabaseHelper.resetTestData();
    }

    @Before("@login")
    public void ensureLoginPageReady() {
        // Runs only for scenarios tagged @login
        loginPage.navigateTo("/login");
    }
}

Tag Best Practices

  • Define a tag catalog — Document all approved tags and their meanings in the project wiki or README.
  • Every scenario gets a suite tag — At minimum @smoke or @regression on every scenario.
  • Don't over-tag — 2–4 tags per scenario is sufficient. Too many tags make filtering confusing.
  • Remove @wip promptly — WIP tags left after completion cause scenarios to be excluded from CI.
  • @broken requires a Jira ticket — Never tag as broken without a linked defect.
  • Consistent naming — lowercase, hyphen-separated: @user-login not @UserLogin or @user_login.
  • Feature tags for broad filtering — @checkout on Feature level avoids tagging every scenario individually.

Interview Q&A — Cucumber Tags

Q1. What are Cucumber tags?

Answer: Metadata labels (@tagname) attached to features and scenarios for organizing, filtering, and controlling test execution. They enable selective running of test subsets via tag expressions in the runner or command line.

Q2. How does tag inheritance work?

Answer: Tags on a Feature are inherited by all scenarios within that feature. Scenario-level tags are additive. A scenario tagged @smoke in a Feature tagged @regression effectively has both tags.

Q3. Explain tag expression operators.

Answer: AND (intersection — both tags required), OR (union — either tag), NOT (exclusion). Precedence: not > and > or. Parentheses for grouping: (@smoke or @sanity) and not @skip.

Q4. How do you run specific tags from the command line?

Answer: mvn test -Dcucumber.filter.tags="@smoke and not @skip". This overrides the tags attribute in @CucumberOptions at runtime.

Q5. What is the difference between @skip and @wip?

Answer: @wip marks scenarios under active development — not yet ready. @skip marks stable scenarios temporarily disabled (deprecated feature, blocked dependency). Both exclude from default CI runs via not @skip and not @wip filters.

Q6. How do tag-based hooks work?

Answer: @Before("@ui") runs before scenarios tagged @ui. @After("@api") runs after API scenarios. Enables conditional setup/teardown without affecting all scenarios.

Q7. Can you tag individual rows in Examples tables?

Answer: No. Tags apply to Feature, Scenario, or Scenario Outline level. Individual Examples sections can be named but not tagged separately. To skip rows, remove them from the table.

Q8. How do you design a tag strategy for CI/CD?

Answer: @smoke on every commit (5–15 min), @regression nightly (full suite), @critical before release, not @skip and not @wip and not @broken on all CI runs. Jenkins parameterized with tag choice for on-demand runs.

Q9. What tags do you use for traceability?

Answer: @JIRA-XXXX linking to tickets, @sprint-XX for sprint tracking, @US-XXX for user story IDs. Enables filtering tests by requirement and verifying coverage per story.

Q10. How many tags should a scenario have?

Answer: 2–4 tags typically: one suite tag (@smoke/@regression), one feature tag (@login), one traceability tag (@JIRA-XXX), optionally one layer tag (@ui/@api). More tags create filtering complexity without added value.
