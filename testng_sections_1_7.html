Section 1: Automation Testing Overview

What is Test Automation?

Test automation is the practice of using specialized tools, scripts, and frameworks to execute predefined test cases, compare actual results against expected results, and report outcomes — with minimal human intervention during execution.

Definition

Automation testing converts manual test steps into reusable scripts that can run repeatedly across environments, browsers, and builds. The scripts interact with the application under test (AUT) through APIs, UI drivers, or direct code calls, then validate behavior using assertions.

Purpose

  • Repeatability — Run the same tests hundreds of times without human fatigue or inconsistency.
  • Speed — Execute large regression suites in minutes instead of days.
  • Early feedback — Integrate with CI/CD pipelines to catch defects immediately after code commits.
  • Coverage expansion — Free QA engineers from repetitive checks to focus on exploratory and edge-case testing.
  • Documentation — Well-written automation scripts serve as living documentation of expected system behavior.

When to Automate

  • Tests are stable — requirements and UI are not changing daily.
  • Tests are repetitive — run on every build, release, or sprint.
  • Tests are data-intensive — many input combinations need validation.
  • Tests require cross-browser / cross-platform execution.
  • Tests are part of a regression suite that must run overnight or on every PR.
  • ROI is positive — automation cost is justified by frequency and criticality of execution.
Interview Tip: Automation is not a replacement for manual testing — it is a force multiplier. The best teams automate the boring, repeatable checks and invest manual effort in exploratory, usability, and ad-hoc testing.

Types of Automation Frameworks

1. Linear (Record & Playback) Framework

Each test is a standalone script recorded sequentially. No reusability between tests.

  • Pros: Quick to create, no programming required, good for POC
  • Cons: Zero reusability, hard to maintain, breaks easily on UI changes, not scalable

2. Modular Framework

Application is divided into independent modules. Each module has its own test script; a master script calls modules in sequence.

  • Pros: Reusable modules, easier maintenance than linear, good separation of concerns
  • Cons: Master script changes needed when flow changes, data still hardcoded in modules

3. Data-Driven Framework

Test logic is separated from test data. Data is stored externally in Excel, CSV, JSON, or databases and fed into test scripts via parameters.

  • Pros: Same script runs with multiple data sets, reduces code duplication, easy to add new test cases by adding data rows
  • Cons: More complex setup, data management overhead, requires data validation strategy

4. Keyword-Driven Framework

Actions are defined as keywords (e.g., ClickButton, EnterText, VerifyTitle). Test cases are written as tables of keywords + arguments, interpreted by an engine.

  • Pros: Non-technical stakeholders can write tests, highly reusable action library, decouples test design from implementation
  • Cons: High initial development cost, keyword engine complexity, debugging is harder

5. Hybrid Framework

Combines modular, data-driven, and keyword-driven approaches. Most common in industry — e.g., Page Object Model + DataProvider + TestNG groups.

  • Pros: Best of all worlds, scalable, maintainable, industry-proven pattern
  • Cons: Requires experienced engineers, longer initial setup, team must follow conventions

6. BDD (Behavior-Driven Development) Framework

Tests written in business-readable Gherkin syntax (Given/When/Then). Tools: Cucumber (Java), SpecFlow (.NET), Behave (Python).

  • Pros: Bridges business and tech, living documentation, stakeholder-readable scenarios
  • Cons: Step definition maintenance, can become slow if over-engineered, abstraction overhead

Interview Q&A — Automation Testing Overview

Q1: What is test automation and why do companies invest in it?

Test automation uses scripts and tools to execute tests automatically. Companies invest because it provides faster feedback in CI/CD, reduces regression effort, improves consistency, and allows QA to focus on high-value exploratory testing.

Q2: Can automation replace manual testing entirely?

No. Automation validates known expected behavior. Manual testing is essential for exploratory testing, usability evaluation, ad-hoc scenarios, and new features where requirements are still evolving.

Q3: What is the Automation Testing Life Cycle?

Feasibility Analysis → Tool Selection → Framework Design → Script Development → Execution → Maintenance. Maintenance is continuous and often loops back to script development.

Q4: What is a Hybrid framework and why is it most popular?

A Hybrid framework combines modular structure, data-driven inputs, and often Page Object Model. It balances reusability, maintainability, and scalability — making it the industry standard for Selenium-based projects.

Q5: What is Page Object Model (POM)?

POM is a design pattern where each web page is represented by a Java class containing locators and interaction methods. Tests call page object methods instead of interacting with WebDriver directly, reducing duplication and easing maintenance.

Q6: How do you calculate ROI for automation?

ROI = (Manual execution cost × Number of executions − Automation development + maintenance cost) / Automation cost. Consider frequency, team size, defect escape rate reduction, and time-to-release improvements.

Q7: What tests should you automate first?

Start with smoke tests and high-risk regression paths that run frequently — login, checkout, payment, core CRUD operations. These give the highest immediate ROI.

Q8: What is the difference between Data-Driven and Keyword-Driven frameworks?

Data-Driven separates test data from test logic — same script, different inputs. Keyword-Driven separates test actions into reusable keywords — test cases are tables of keyword sequences independent of code.

Q9: What are common challenges in test automation?

Flaky tests, high maintenance when UI changes, tool selection mismatches, lack of skilled engineers, poor framework design, insufficient test data management, and unrealistic automation coverage expectations.

Q10: What is BDD and how does it differ from TDD?

BDD (Behavior-Driven Development) focuses on system behavior described in business language (Gherkin). TDD (Test-Driven Development) is a developer practice of writing unit tests before code. BDD operates at acceptance/integration level; TDD at unit level.

Q11: What is a flaky test and how do you handle it?

A flaky test passes and fails intermittently without code changes — usually due to timing issues, environment instability, or poor locators. Fix with explicit waits, stable locators, retry mechanisms (with caution), and root-cause analysis rather than just re-running.

Q12: How do you decide which automation tool to use?

Evaluate against: application type (web/mobile/API), tech stack, team skills, budget, CI/CD compatibility, community support, reporting needs, and cross-browser requirements. POC on 2–3 candidates before committing.

Q13: What is the difference between smoke, sanity, and regression testing?

Smoke: Broad shallow check — does the build work at all? (15–30 min). Sanity: Narrow deep check on a specific module after a fix. Regression: Full suite validating existing functionality after changes (hours).

Q14: What role does automation play in Agile/CI-CD?

Automation gates every commit/PR with smoke tests, runs regression nightly, provides immediate feedback to developers, and supports continuous delivery by ensuring quality at speed.

Section 2: TestNG Introduction

What is TestNG?

TestNG stands for Testing Next Generation. It is an open-source Java testing framework created by Cédric Beust in 2004, inspired by JUnit and NUnit. TestNG was designed to cover all categories of tests — unit, functional, integration, and end-to-end — with powerful configuration and reporting capabilities.

Unlike JUnit (originally a unit-testing framework), TestNG was built from the ground up for integration and system-level testing, making it the de facto standard for Selenium WebDriver automation in Java projects.

Key Fact for Interviews: TestNG is not a replacement for JUnit at the unit level — many projects use JUnit for unit tests and TestNG for integration/UI automation suites.

TestNG Features

  • Annotations — Rich lifecycle hooks at suite, test, class, and method levels
  • Parallel Execution — Run tests concurrently at method, class, or test level
  • Data-Driven Testing — @DataProvider feeds multiple data sets to a single @Test
  • Grouping — Tag tests (smoke, regression) and run subsets via XML or Maven
  • Dependency Management — Chain tests; skip dependents when parent fails
  • Listeners — Hook into test lifecycle for custom reporting, screenshots, retry
  • Reporting — Built-in HTML reports with pass/fail/skip counts and stack traces
  • XML Configuration — Externalize suite composition without recompiling code
  • Flexible Test Configuration — Parameters, groups, inclusion/exclusion, parallel modes

Interview Q&A — TestNG Introduction

Q1: What is TestNG and who created it?

TestNG (Testing Next Generation) is a Java testing framework created by Cédric Beust in 2004, inspired by JUnit and NUnit. It supports unit, functional, and integration testing with advanced features like grouping, parallel execution, and XML configuration.

Q2: Why is TestNG preferred over JUnit for Selenium automation?

TestNG offers native parallel execution, test groups, dependency management, DataProvider, XML suite configuration, detailed HTML reporting, and listeners — all critical for large-scale UI automation that JUnit 4 lacked natively.

Q3: What is testng.xml?

testng.xml is TestNG's suite configuration file. It defines which tests to run, group inclusion/exclusion, parallel mode, thread count, parameters, and listener classes — all without recompiling Java code.

Q4: What is the TestNG execution hierarchy?

Suite → Test → Class → Method. A suite contains one or more <test> tags. Each test tag lists classes. Each class contains @Test annotated methods.

Q5: How do you add TestNG to a Maven project?

Add the org.testng:testng dependency with test scope in pom.xml. Configure maven-surefire-plugin to point to testng.xml via <suiteXmlFiles>.

Q6: Can TestNG and JUnit coexist in the same project?

Yes. Many projects use JUnit for unit tests and TestNG for integration/UI tests. Maven Surefire can be configured to run both with separate plugin executions.

Q7: What reports does TestNG generate by default?

TestNG generates reports in test-output/ folder: index.html (main report), emailable-report.html (summary for email), testng-results.xml, and per-suite HTML pages with pass/fail/skip details.

Q8: What is the difference between TestNG and Selenium?

Selenium is a browser automation library (WebDriver API). TestNG is a test framework that orchestrates, groups, parallelizes, and reports on tests. Selenium drives the browser; TestNG manages the test lifecycle.

Q9: What are TestNG listeners?

Listeners are interfaces (ITestListener, ISuiteListener, IAnnotationTransformer, IRetryAnalyzer) that hook into test events — onTestStart, onTestFailure, onTestSuccess — enabling custom reporting, screenshots, and retry logic.

Q10: Is TestNG only for UI testing?

No. TestNG can run any Java test — unit tests, API tests (with REST Assured), database tests, integration tests. It is widely used for UI because of its Selenium ecosystem integration, but it is not limited to UI.

Q11: What is the latest stable TestNG version?

As of 2024–2025, TestNG 7.x is current (7.10.x). Always check Maven Central for the latest. Version 7 added Java module support and improved parallel execution.

Q12: How does TestNG integrate with CI/CD?

Run via Maven (mvn test), Gradle, or direct TestNG CLI. Jenkins/GitHub Actions trigger on commit, pass group parameters (-Dgroups=smoke), archive test-output reports, and fail the build on test failures.

Section 3: TestNG Annotations (Detailed)

TestNG annotations define the test lifecycle. Understanding when each annotation runs is the #1 most asked TestNG interview topic.

Complete Execution Flow

@BeforeSuite
    └── @BeforeTest
            └── @BeforeClass
                    └── @BeforeMethod → @Test → @AfterMethod
                    └── @BeforeMethod → @Test → @AfterMethod  (repeat per method)
            └── @AfterClass
    └── @AfterTest
└── @AfterSuite
Memory Aid: Before = setup (outside → inside). After = teardown (inside → outside). Suite is outermost; Method is innermost.

1. @BeforeSuite

Purpose: Runs once before any tests in the entire suite begin.

When it runs: Before @BeforeTest, @BeforeClass, @BeforeMethod, and @Test — exactly once per suite execution.

Real framework usage: Initialize database connections, load global configuration properties, set up ExtentReports instance, create shared test data directories.

@BeforeSuite
public void globalSetup() {
    ConfigReader.load("config.properties");
    DBConnection.initialize();
    ExtentReportManager.initReport();
    System.out.println("@BeforeSuite — Global setup complete");
}

Interview tip: Only one @BeforeSuite method should exist per suite (across all classes). If multiple classes define it, all will run but order is not guaranteed unless using dependencies.

2. @BeforeTest

Purpose: Runs before each <test> tag defined in testng.xml.

When it runs: After @BeforeSuite, before @BeforeClass. Runs once per <test> block.

Real framework usage: Launch browser instance for a test block, set test-level parameters, initialize WebDriver for a specific environment.

@BeforeTest
@Parameters({"browser", "url"})
public void launchBrowser(String browser, String url) {
    driver = DriverFactory.createDriver(browser);
    driver.get(url);
    System.out.println("@BeforeTest — Browser launched for test block");
}

Interview tip: @BeforeTest is tied to the XML <test> tag, NOT to a Java @Test method. This confuses many candidates.

3. @BeforeClass

Purpose: Runs once before the first @Test method in the current class.

When it runs: After @BeforeTest, before any @BeforeMethod in that class.

Real framework usage: Initialize Page Object instances, create test data specific to the class, open a shared resource.

@BeforeClass
public void initPageObjects() {
    loginPage = new LoginPage(driver);
    homePage = new HomePage(driver);
    System.out.println("@BeforeClass — Page objects initialized");
}

Interview tip: @BeforeClass methods must be static if the class is run with parallel="classes" — otherwise TestNG cannot instantiate the class properly for parallel execution.

4. @BeforeMethod

Purpose: Runs before each @Test method in the class.

When it runs: After @BeforeClass, immediately before every @Test method execution.

Real framework usage: Navigate to starting URL, perform login, reset application state, clear cookies, start test timer.

@BeforeMethod
public void navigateToLogin() {
    driver.get(baseUrl + "/login");
    System.out.println("@BeforeMethod — Navigated to login page");
}

Interview tip: This is the most commonly used setup annotation in Selenium frameworks — it ensures each test starts from a clean, known state.

5. @Test

Purpose: Marks an actual test method that TestNG will execute and report on.

When it runs: After all applicable @Before* methods for that test method.

Real framework usage: Contains test logic — interact with UI, call APIs, validate results with assertions.

@Test(description = "Verify successful login with valid credentials",
      priority = 1,
      groups = {"smoke", "login"})
public void testValidLogin() {
    loginPage.enterCredentials("user@test.com", "Pass123");
    loginPage.clickLogin();
    Assert.assertTrue(homePage.isWelcomeMessageDisplayed());
    System.out.println("@Test — testValidLogin executed");
}

Interview tip: A method without @Test is NOT executed by TestNG, even if it is public. Only @Test (or @Factory) methods are test methods.

6. @AfterMethod

Purpose: Runs after each @Test method completes (pass or fail).

When it runs: Immediately after every @Test method, before the next @BeforeMethod.

Real framework usage: Capture screenshot on failure, log test result, clear test-specific data, attach logs to report.

@AfterMethod
public void captureScreenshotOnFailure(ITestResult result) {
    if (result.getStatus() == ITestResult.FAILURE) {
        ScreenshotUtil.capture(driver, result.getName());
    }
    System.out.println("@AfterMethod — Cleanup after: " + result.getName());
}

Interview tip: @AfterMethod always runs even if @Test throws an exception — use it for guaranteed cleanup.

7. @AfterClass

Purpose: Runs once after all @Test methods in the class have finished.

When it runs: After the last @AfterMethod in the class.

Real framework usage: Close class-level resources, delete test data created for the class, reset shared state.

@AfterClass
public void classCleanup() {
    testDataHelper.deleteTestRecords();
    System.out.println("@AfterClass — Class-level cleanup done");
}

8. @AfterTest

Purpose: Runs after all classes in the current <test> tag have finished.

When it runs: After all @AfterClass methods for classes in that test block.

Real framework usage: Quit browser, flush test-level logs, send partial report for the test block.

@AfterTest
public void closeBrowser() {
    if (driver != null) {
        driver.quit();
    }
    System.out.println("@AfterTest — Browser closed for test block");
}

9. @AfterSuite

Purpose: Runs once after all tests in the entire suite have completed.

When it runs: After all @AfterTest methods — the very last hook.

Real framework usage: Generate final ExtentReport, close DB connections, send email summary, archive test artifacts.

@AfterSuite
public void generateFinalReport() {
    ExtentReportManager.flushReport();
    DBConnection.close();
    EmailUtil.sendReport("test-output/emailable-report.html");
    System.out.println("@AfterSuite — Final report generated");
}

10. @BeforeGroups / @AfterGroups

Purpose: Run setup/teardown before and after the first/last test method in a specific group.

When it runs: @BeforeGroups runs before the first method in the specified group; @AfterGroups runs after the last method in that group.

@BeforeGroups("checkout")
public void setupCheckoutData() {
    cartHelper.addDefaultItems();
    System.out.println("@BeforeGroups — Checkout data prepared");
}

@AfterGroups("checkout")
public void cleanupCheckoutData() {
    cartHelper.clearCart();
    System.out.println("@AfterGroups — Checkout data cleaned");
}

@Test(groups = "checkout")
public void testPlaceOrder() {
    checkoutPage.placeOrder();
    Assert.assertTrue(confirmationPage.isOrderConfirmed());
}

Interview tip: Groups hooks run within the normal Before/After Method lifecycle — they are additional, not replacements.

Interview Q&A — TestNG Annotations

Q1: What is the execution order of TestNG annotations?

@BeforeSuite → @BeforeTest → @BeforeClass → @BeforeMethod → @Test → @AfterMethod → @AfterClass → @AfterTest → @AfterSuite. For multiple @Test methods, @BeforeMethod → @Test → @AfterMethod repeats.

Q2: What is the difference between @BeforeTest and @BeforeMethod?

@BeforeTest runs once per <test> tag in testng.xml (before all classes in that test block). @BeforeMethod runs before every individual @Test method. @BeforeTest is coarser-grained; @BeforeMethod is per-test-method.

Q3: What is the difference between @BeforeSuite and @BeforeClass?

@BeforeSuite runs once for the entire XML suite (all test tags). @BeforeClass runs once per Java class before its first @Test method. Suite is global; Class is per Java file.

Q4: Does @AfterMethod run if @Test fails?

Yes. @AfterMethod always runs after @Test completes, regardless of pass, fail, or skip status. This makes it ideal for screenshots and cleanup.

Q5: Can you have multiple @BeforeMethod methods in one class?

Yes. All @BeforeMethod methods run before each @Test, but their order among themselves is not guaranteed unless you use priority (not available on @BeforeMethod) or dependsOnMethods pattern.

Q6: What happens if @BeforeSuite throws an exception?

All tests in the suite are skipped. @BeforeSuite failure is catastrophic — no tests run. Always handle exceptions carefully in suite-level setup.

Q7: Must @BeforeClass be static?

Not always, but it MUST be static when running with parallel="classes" because TestNG creates separate instances per class in parallel mode and static methods don't require instance state.

Q8: What is @BeforeGroups used for?

It runs once before the first test method in the specified group executes. Useful for group-specific setup like preparing checkout test data only when checkout group tests run.

Q9: What is the difference between @Test(enabled=false) and commenting out a test?

enabled=false keeps the test in reports as "Skipped" and preserves the code for future use. Commenting removes it entirely from TestNG's discovery. enabled=false is the professional approach.

Q10: What does @Test(timeOut = 5000) do?

Fails the test if it runs longer than 5000 milliseconds (5 seconds). Useful for catching hung tests, infinite loops, or missing waits in Selenium scripts.

Q11: What is expectedExceptions in @Test?

@Test(expectedExceptions = IOException.class) means the test PASSES if an IOException is thrown, and FAILS if no exception or a different exception occurs. Used for negative testing.

Q12: What is invocationCount?

@Test(invocationCount = 5) runs the same test method 5 times. Useful for stability testing, load smoke checks, or verifying intermittent issues.

Q13: How does @Test(priority = -1) work?

Negative priorities are allowed. Lower numbers run first, so priority = -1 runs before priority = 0 (default) which runs before priority = 1.

Q14: What annotations can receive ITestResult as a parameter?

@AfterMethod and @BeforeMethod can accept ITestResult (or Method, ITestContext) as parameters. TestNG injects them automatically — useful for getting test name, status, and parameters in hooks.

Q15: What is the difference between @Factory and @Test?

@Test marks a single test method. @Factory marks a method that returns an array of test class instances, allowing dynamic test creation — e.g., run the same test class with different browsers.

Q16: If a class has no @Test methods but has @BeforeClass, what happens?

TestNG skips the class entirely if no @Test methods are present (or all are enabled=false). Configuration methods alone do not trigger class execution.

Interview Q&A — TestNG Execution Flow

Q1: Explain the TestNG execution hierarchy.

Suite (testng.xml) contains Test (<test> tags) which contain Classes (Java test classes) which contain Methods (@Test annotated methods). Configuration annotations run at each level accordingly.

Q2: In what order are test classes executed by default?

Alphabetically by fully qualified class name, unless preserve-order is set or dependencies/priorities override the order.

Q3: How do you run only specific test methods from a class?

In testng.xml, use <methods><include name="methodName"/></methods> inside the <class> tag. Alternatively, use groups to filter.

Q4: What happens when a test with dependsOnMethods fails?

All dependent tests are SKIPPED — they do not run and are not marked as failed. They appear as skipped in the report.

Q5: What is the difference between parallel="methods" and parallel="classes"?

parallel="methods" runs individual @Test methods concurrently (maximum parallelism). parallel="classes" runs entire classes concurrently — methods within a class still run sequentially.

Q6: Can you run multiple testng.xml files in one Maven execution?

Yes. Configure maven-surefire-plugin with multiple <suiteXmlFile> entries, or use a suite-of-suites XML that references other XML files.

Q7: What is thread-count in testng.xml?

thread-count specifies the number of threads for parallel execution. With parallel="methods" thread-count="5", up to 5 test methods run simultaneously.

Q8: How do you skip a test class entirely in testng.xml?

Remove the class from the XML, or use <class name="..."><methods><exclude name=".*"/></methods></class>, or set enabled=false on all @Test methods in the class.

Q9: What is the difference between include/exclude in testng.xml vs groups?

include/exclude operates on specific method names — surgical selection. Groups operate on tagged categories — run all tests marked "smoke" regardless of class or method name.

Q10: How does TestNG handle tests in multiple <test> tags?

Each <test> tag is an independent block with its own @BeforeTest/@AfterTest lifecycle. Tests in different <test> tags can run in parallel if parallel="tests" is set.

Section 5: TestNG Assertions

Assertions validate that actual results match expected results. TestNG provides hard assertions via the Assert class and soft assertions via the SoftAssert class.

Hard Assertions

Hard assertions stop test execution immediately on failure. Subsequent lines and assertions are not executed.

Assert.assertEquals(actual, expected)

Verifies two values are equal. Works with objects, primitives, strings, arrays.

@Test
public void testProductPrice() {
    String actualPrice = productPage.getPrice();
    Assert.assertEquals(actualPrice, "$29.99");
}

Assert.assertEquals(actual, expected, message)

Same as above but with a custom failure message for clearer debugging.

Assert.assertEquals(actualCount, 5,
    "Cart item count mismatch — expected 5 items after adding products");

Assert.assertTrue(condition)

Verifies a boolean condition is true. Most common for UI visibility checks.

Assert.assertTrue(loginPage.isErrorMessageDisplayed(),
    "Error message should appear for invalid credentials");

Assert.assertFalse(condition)

Verifies a boolean condition is false.

Assert.assertFalse(checkoutPage.isPlaceOrderButtonEnabled(),
    "Place Order button should be disabled with empty cart");

Assert.assertNull(object)

Verifies an object reference is null.

Assert.assertNull(sessionManager.getActiveSession(),
    "Session should be null after logout");

Assert.assertNotNull(object)

Verifies an object reference is not null.

Assert.assertNotNull(driver.findElement(By.id("submitBtn")),
    "Submit button must be present on the page");

Assert.assertNotEquals(actual, expected)

Verifies two values are NOT equal.

Assert.assertNotEquals(apiResponse.getStatusCode(), 500,
    "API should not return server error");

Assert.assertSame / Assert.assertNotSame

Compares object references (identity), not values. assertSame passes if both refer to the exact same object in memory.

Object singleton = ConfigManager.getInstance();
Object sameRef = ConfigManager.getInstance();
Assert.assertSame(singleton, sameRef, "ConfigManager should be singleton");

Assert.assertNotSame(new ArrayList<>(), new ArrayList<>(),
    "Two new ArrayLists are different objects");

Assert.fail(message)

Forces test failure with a custom message. Used when a condition should never be reached.

@Test
public void testUnsupportedBrowser() {
    if (browser.equals("IE")) {
        Assert.fail("IE browser is no longer supported — update test configuration");
    }
}

Soft Assertions

Soft assertions collect all failures during test execution and report them together at the end via assertAll(). The test continues executing even after a soft assertion failure.

import org.testng.asserts.SoftAssert;

@Test
public void testUserProfilePage() {
    SoftAssert softAssert = new SoftAssert();

    softAssert.assertEquals(profilePage.getName(), "John Doe", "Name mismatch");
    softAssert.assertEquals(profilePage.getEmail(), "john@test.com", "Email mismatch");
    softAssert.assertTrue(profilePage.isAvatarDisplayed(), "Avatar not displayed");
    softAssert.assertEquals(profilePage.getPhone(), "+1-555-0100", "Phone mismatch");
    softAssert.assertFalse(profilePage.isEditButtonEnabled(), "Edit should be disabled");
    softAssert.assertNotNull(profilePage.getMemberSinceDate(), "Member date is null");

    softAssert.assertAll(); // Reports ALL failures at once
}
Critical: Always call assertAll() at the end. Without it, soft assertion failures are silently ignored and the test reports as PASSED.

Create project-specific assertion methods to improve readability and reduce duplication across the framework.

Interview Q&A — TestNG Assertions

Q1: What is the difference between hard and soft assertions?

Hard assertions (Assert class) stop test execution on first failure. Soft assertions (SoftAssert class) collect failures and report all at once via assertAll(). Use hard for critical path checks; soft for multi-field validation.

Q2: What happens if you forget to call assertAll()?

The test will report as PASSED even if soft assertions failed. assertAll() is mandatory — it throws AssertionError with all collected failures.

Q3: What is the parameter order for assertEquals?

Assert.assertEquals(actual, expected) — actual value first, expected second. TestNG also supports assertEquals(expected, actual) in some overloads, but the standard is actual first.

Q4: When would you use Assert.fail()?

When a code path should never be reached — unsupported configurations, unexpected switch cases, or when try-catch catches an exception that should not occur.

Q5: Can you mix hard and soft assertions in one test?

Yes. Common pattern: hard assert for critical preconditions (page loaded, user logged in), then soft assert for multiple field validations on the resulting page.

Q6: What is the difference between assertEquals and assertSame?

assertEquals compares values (calls equals()). assertSame compares object references (=== in memory). Two String objects with same value pass assertEquals but fail assertSame.

Q7: How do you create custom assertions?

Create a utility class with static methods that wrap Assert calls with domain-specific logic and meaningful messages — e.g., assertPageTitle, assertApiResponse.

Q8: Does TestNG have assertAll like JUnit 5?

TestNG uses SoftAssert.assertAll() for grouped soft assertions. JUnit 5 uses assertAll() as a static method. The concept is the same; the API differs.

Q9: What assertion should you use after clicking a button that opens a modal?

Hard assert — Assert.assertTrue(modal.isDisplayed()) — because if the modal didn't open, all subsequent modal validations are meaningless.

Q10: How do assertions work with TestNG listeners?

When an assertion fails, TestNG marks the test as FAILED and triggers onTestFailure in ITestListener. Soft assertion failures only trigger this at assertAll() time.

Q11: Can SoftAssert be reused across multiple test methods?

No. Create a new SoftAssert instance per test method. Reusing across methods can carry over failures from previous tests.

Q12: What is assertNotEquals used for?

Verifying a value has changed — e.g., after an update operation, assertNotEquals(oldValue, newValue) confirms the value was modified.

Section 6: TestNG Priority and Dependency

Priority

The priority attribute controls the order in which @Test methods execute within a class. Lower numbers run first.

Rules

  • Default priority is 0 for all @Test methods without explicit priority.
  • Negative priorities are allowed: @Test(priority = -1) runs before priority 0.
  • Methods with the same priority run in alphabetical order by method name.
  • Priority only affects order within the same class — not across classes.
  • Priority does NOT guarantee execution if combined with dependsOnMethods — dependencies take precedence.

Complete Example — 5 Test Methods

public class PriorityDemo {

    @Test(priority = 3)
    public void verifyCheckout() {
        System.out.println("Priority 3 — verifyCheckout (runs 4th)");
    }

    @Test(priority = 1)
    public void verifyLogin() {
        System.out.println("Priority 1 — verifyLogin (runs 2nd)");
    }

    @Test // default priority = 0
    public void verifyHomePage() {
        System.out.println("Priority 0 — verifyHomePage (runs 1st alphabetically among p=0)");
    }

    @Test(priority = 0)
    public void verifyNavigation() {
        System.out.println("Priority 0 — verifyNavigation (runs 2nd alphabetically among p=0)");
    }

    @Test(priority = -1)
    public void setupTestData() {
        System.out.println("Priority -1 — setupTestData (runs FIRST)");
    }

    @Test(priority = 2)
    public void verifySearch() {
        System.out.println("Priority 2 — verifySearch (runs 3rd)");
    }
}

3. verifyNavigation    (priority = 0)
4. verifyLogin         (priority = 1)
5. verifySearch        (priority = 2)
6. verifyCheckout      (priority = 3)
*/

Dependency

Dependency ensures tests run in a logical sequence where downstream tests depend on upstream tests passing. If an upstream test fails, downstream tests are SKIPPED.

dependsOnMethods

@Test
public void login() {
    loginPage.login("user@test.com", "pass123");
    Assert.assertTrue(homePage.isDisplayed());
}

@Test(dependsOnMethods = {"login"})
public void searchProduct() {
    searchPage.search("laptop");
    Assert.assertTrue(searchPage.getResultCount() > 0);
}

@Test(dependsOnMethods = {"searchProduct"})
public void addToCart() {
    searchPage.clickFirstResult();
    productPage.addToCart();
    Assert.assertEquals(cartPage.getItemCount(), 1);
}

@Test(dependsOnMethods = {"addToCart"})
public void checkout() {
    cartPage.proceedToCheckout();
    checkoutPage.completeOrder();
    Assert.assertTrue(confirmationPage.isOrderConfirmed());
}

dependsOnGroups

@Test(groups = "setup")
public void createTestAccount() {
    apiHelper.createUser("testuser");
}

@Test(groups = "setup")
public void loadTestProducts() {
    apiHelper.seedProducts();
}

@Test(dependsOnGroups = {"setup"})
public void testPurchaseFlow() {
    // Runs only if ALL methods in "setup" group passed
    purchasePage.completePurchase();
}

alwaysRun = true

@Test(dependsOnMethods = {"login"}, alwaysRun = true)
public void captureLogsOnFailure() {
    // Runs even if login() failed — useful for cleanup/diagnostics
    logCollector.saveLogs();
}

Common Mistakes

  • Using priority for dependent flows — Priority doesn't skip tests on failure. If login fails but checkout has priority 2, checkout still runs and fails confusingly.
  • Cross-class dependsOnMethods without same test tag — Dependencies work best within the same <test> tag. Cross-test dependencies are unreliable.
  • Circular dependencies — testA depends on testB, testB depends on testA → TestNG throws an error at startup.
  • Typo in dependsOnMethods name — TestNG skips the dependent test silently if the method name doesn't match exactly.
  • Combining priority with dependency — Dependencies take precedence. Priority only matters among independent tests.

Interview Q&A — Priority and Dependency

Q1: What is the default priority of a @Test method?

0. Methods without an explicit priority attribute default to priority 0 and run alphabetically among themselves.

Q2: Can priority be negative?

Yes. @Test(priority = -1) runs before priority 0. Useful for setup-like tests without using @BeforeMethod.

Q3: What happens when two tests have the same priority?

TestNG executes them in alphabetical order by method name.

Q4: What is dependsOnMethods?

An attribute that makes a test dependent on another test method passing. If the parent test fails or is skipped, the dependent test is SKIPPED.

Q5: What is the difference between dependsOnMethods and dependsOnGroups?

dependsOnMethods depends on a specific method passing. dependsOnGroups depends on ALL methods in the specified group passing.

Q6: What does alwaysRun = true do?

The test runs even if its dependency failed. Used for cleanup, log capture, or diagnostic tests that should execute regardless of upstream status.

Q7: Can dependsOnMethods reference a method in another class?

Only if both classes are in the same <test> tag in testng.xml. Cross-class dependencies within the same test block work; across test blocks they do not.

Q8: Priority vs Dependency — which should I use for a login → checkout flow?

Dependency. If login fails, checkout should be skipped — not run and fail independently. Priority would still run checkout even after login failure.

Q9: What happens with circular dependencies?

TestNG detects circular dependencies at initialization and throws a TestNGException — the suite fails to start.

Q10: Does priority work across different test classes?

No. Priority only controls order within the same class. For cross-class ordering, use testng.xml class order or dependencies.

Q11: If testA (priority=1) fails and testB (priority=2) has no dependency, does testB run?

Yes. Priority does not skip tests on failure. Only dependsOnMethods/dependsOnGroups causes skipping.

Q12: How do you make a test run last?

Assign the highest priority number: @Test(priority = 999). Or use dependsOnMethods to depend on all other tests (less common).

Section 7: TestNG Groups

What are Groups?

Groups are TestNG's mechanism for categorizing tests and running subsets selectively. Instead of running the entire suite, you can run only "smoke" tests after a deployment or only "regression" tests nightly.

Groups are assigned via the groups attribute on @Test and configured for execution in testng.xml or via Maven parameters.

Assigning Groups to Tests

// Single group
@Test(groups = "smoke")
public void testLoginPageLoads() {
    Assert.assertTrue(loginPage.isDisplayed());
}

// Multiple groups — test belongs to both
@Test(groups = {"regression", "ui"})
public void testProductSearch() {
    searchPage.search("phone");
    Assert.assertTrue(searchPage.hasResults());
}

// Method in three groups
@Test(groups = {"smoke", "regression", "checkout"})
public void testQuickCheckout() {
    checkoutPage.quickOrder();
    Assert.assertTrue(confirmationPage.isDisplayed());
}
Key Point: A test can belong to multiple groups. Running any one of those groups will include the test.

testng.xml Group Configuration

<suite name="Smoke Suite">
    <test name="Smoke Tests">

            <class name="com.tests.LoginTest"/>
            <class name="com.tests.HomePageTest"/>
            <class name="com.tests.CheckoutTest"/>
        </classes>
    </test>
</suite>

        </groups>
        <packages>
            <package name="com.tests.*"/>
        </packages>
    </test>

Interview Q&A — TestNG Groups

Q1: What are TestNG groups?

Groups are tags assigned to @Test methods that allow selective execution. You can run only "smoke" tests, exclude "broken" tests, or combine groups in testng.xml.

Q2: Can a test belong to multiple groups?

Yes. @Test(groups = {"smoke", "regression", "ui"}) belongs to all three. It will run when any of those groups is included.

Q3: How do you exclude a group in testng.xml?

Use <exclude name="groupName"/> inside the <run> block: <groups><run><include name="regression"/><exclude name="broken"/></run></groups>

Q4: What are MetaGroups?

MetaGroups are groups of groups defined in testng.xml using <define>. They let you create aliases like "full-regression" that include smoke + regression + api groups.

Q5: How do you run a specific group from the command line?

Create a testng.xml with the group include/exclude configuration and run: mvn test -DsuiteXmlFile=smoke.xml. Or use TestNG CLI with group parameters.

Q6: What is the difference between groups and priority?

Groups categorize tests for selective execution (run subset). Priority controls execution order within a run. They serve completely different purposes.

Q7: How do groups integrate with Jenkins?

Jenkins pipeline passes the suite XML or group name as a Maven parameter. Different pipeline stages run different group configurations — smoke on deploy, regression nightly.

Q8: What is the "broken" group pattern?

Tests with known failures are tagged "broken" and excluded from CI via <exclude name="broken"/>. This keeps CI green while tracking known issues. Remove the tag once fixed.

Q9: Can @BeforeGroups replace @BeforeMethod?

No. @BeforeGroups runs once before the first test in a group; @BeforeMethod runs before every test method. They serve different scopes and complement each other.

Q10: How do groups work with parallel execution?

Groups filter which tests run but don't affect parallel behavior. parallel="methods" with group filtering runs matching methods in parallel.

Q11: What happens if a test has no group assigned?

It runs only when no group filtering is applied (full suite) or when explicitly included by class/method name in testng.xml. Ungrouped tests are excluded when group filtering is active.

Q12: How do you run tests that belong to groupA OR groupB?

Include both groups in testng.xml: <include name="groupA"/><include name="groupB"/>. TestNG runs tests belonging to either group (union, not intersection).

Q13: What is the difference between include/exclude groups and include/exclude methods?

Group include/exclude filters by group tag — dynamic, tag-based. Method include/exclude filters by exact method name — static, name-based. Groups are more maintainable for large suites.

Q14: How would you design groups for a microservices application?

Group by service (user-service, order-service, payment-service) AND by test type (smoke, regression, api). Combine: @Test(groups = {"smoke", "order-service"}) for targeted runs per microservice.
