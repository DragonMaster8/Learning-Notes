Section 8: Parameterization

8.1 @Parameters Annotation — Reading from testng.xml

TestNG's @Parameters annotation injects values defined in testng.xml directly into test method parameters at runtime. This is ideal for environment-specific configuration such as browser name, base URL, or credentials that change per suite but remain static for a given run.

Syntax

@Test
@Parameters({"paramName1", "paramName2"})
public void myTest(String param1, int param2) {
    // use param1, param2
}

Single Parameter

// testng.xml
<parameter name="browser" value="chrome"/>

// Test class
@Test
@Parameters("browser")
public void launchBrowser(String browser) {
    System.out.println("Running on: " + browser);
}

Multiple Parameters

// testng.xml
<parameter name="username" value="admin"/>
<parameter name="password" value="secret123"/>

// Test class
@Test
@Parameters({"username", "password"})
public void testLogin(String username, String password) {
    loginPage.login(username, password);
    Assert.assertTrue(dashboardPage.isDisplayed());
}

testng.xml <parameter> Tag

Parameters can be declared at suite level (inherited by all tests) or test level (overrides suite-level values for that test block).

<!DOCTYPE suite SYSTEM "https://testng.org/testng-1.0.dtd">
<suite name="Login Suite">
    <parameter name="baseUrl" value="https://qa.example.com"/>
    <test name="Login Tests">
        <parameter name="username" value="testuser"/>
        <parameter name="password" value="Pass@123"/>
        <classes>
            <class name="com.example.tests.LoginTest"/>
        </classes>
    </test>
</suite>

@DataProvider supplies two-dimensional arrays of objects to test methods, enabling true data-driven testing with multiple iterations per test method.

Return Type: Object[][]

8.6 Interview Q&A — Parameterization

Q1: What is the difference between @Parameters and @DataProvider?

Answer: @Parameters reads static string values from testng.xml and injects them as method arguments — one execution per test. @DataProvider is a Java method returning Object[][] that supplies multiple data sets, causing the test to run once per row. @Parameters suits configuration; @DataProvider suits data-driven testing.

Interviewer expects: Clear distinction between static XML config and dynamic Java-based data injection.

Q2: Can @Parameters pass non-String values?

Answer: XML values are always strings, but TestNG auto-converts them to matching parameter types (int, boolean, etc.) if the conversion is supported. Complex objects cannot be passed via @Parameters — use @DataProvider instead.

Interviewer expects: Awareness of type conversion limits and when to switch approaches.

Q3: What is the return type of a DataProvider method?

Answer: Object[][] — a two-dimensional array where each inner array represents one iteration's arguments. Alternatively, Iterator<Object[]> can be used for lazy/large data sets to avoid loading everything into memory at once.

Interviewer expects: Knowledge of Iterator alternative for memory efficiency.

Q4: How do you use a DataProvider from another class?

Answer: Define the @DataProvider in a separate class and reference it with dataProviderClass: @Test(dataProvider = "name", dataProviderClass = OtherClass.class). The DataProvider method can be static or instance method in the other class.

Interviewer expects: Practical knowledge of separating data from test logic.

Q5: Can a DataProvider method depend on another DataProvider?

Answer: Not directly via annotation chaining, but a DataProvider method can call another method or utility to build its data. You can also use @Factory with DataProvider for dynamic test class instantiation.

Interviewer expects: Understanding of @Factory as an advanced alternative.

Q6: How does TestNG match DataProvider columns to test method parameters?

Answer: By position — the first element in each Object[] maps to the first method parameter, second to second, and so on. The number of elements per row must match the number of test method parameters or TestNG throws an exception.

Interviewer expects: Positional mapping awareness and error handling knowledge.

Q7: Can you run DataProvider iterations in parallel?

Answer: Yes. Set parallel="methods" in testng.xml with an appropriate thread-count. Each DataProvider iteration is treated as a separate method invocation and can run concurrently if thread-safe.

Interviewer expects: Connection to parallel execution and thread safety concerns.

Q8: What happens if a DataProvider throws an exception?

Answer: TestNG marks all dependent tests as SKIPPED and reports the DataProvider failure in the test results. The test method itself never executes. Fix the data source or exception in the provider method.

Interviewer expects: Understanding of TestNG failure propagation from providers.

Q9: How do you pass parameters at suite level vs test level?

Answer: Declare <parameter> inside <suite> for global values inherited by all tests. Declare inside <test> to override or scope values to that test block only. Test-level parameters take precedence over suite-level.

Interviewer expects: XML hierarchy and override behavior knowledge.

Q10: Can one test method use multiple DataProviders?

Answer: No — a test method can reference only one DataProvider via the dataProvider attribute. To combine data sources, merge them inside a single DataProvider method or use a utility that aggregates multiple sources into one Object[][].

Interviewer expects: Practical workaround for combining data sources.

Q11: What is the purpose of the name attribute in @DataProvider?

Answer: It assigns a logical name used by @Test(dataProvider = "name") to link the provider to the test. If omitted, the method name becomes the default provider name. Explicit naming improves readability in large frameworks.

Interviewer expects: Naming conventions in enterprise frameworks.

Q12: How do you read test data from Excel in a DataProvider?

Answer: Use Apache POI to open the workbook, iterate rows/columns, and populate Object[][]. Encapsulate reading logic in an ExcelReader utility class. Keep the DataProvider thin — delegate file I/O to the utility for maintainability.

Interviewer expects: Apache POI familiarity and separation of concerns.

Q13: What is Java Faker and why use it in test automation?

Answer: Java Faker generates realistic random data (names, emails, addresses) programmatically. It eliminates hardcoded test data, prevents data collisions in parallel runs, and supports testing with varied inputs without maintaining large static data files.

Interviewer expects: Modern dynamic data generation approach.

Q14: Can @Parameters and @DataProvider be used together on the same test?

Answer: Yes. @Parameters values are injected first, followed by DataProvider values. The combined parameter list must match the test method signature order: @Parameters args first, then DataProvider columns.

Interviewer expects: Parameter ordering when combining both mechanisms.

9.11 Interview Q&A — TestNG XML

Q1: What is the purpose of testng.xml?

Answer: testng.xml is the TestNG suite configuration file that defines which test classes/methods execute, execution order, parallelism settings, parameters, group filters, and listeners. It decouples test code from execution configuration, enabling different runs without code changes.

Interviewer expects: Configuration vs code separation principle.

Q2: Difference between <suite> and <test> tags?

Answer: <suite> is the root container holding all configuration and one or more <test> blocks. Each <test> represents a logical test unit with its own classes, groups, and parameters. A suite can have multiple test blocks for different scenarios (smoke, regression, cross-browser).

Interviewer expects: Hierarchical XML structure understanding.

Q3: How do you run a specific test method from testng.xml?

Answer: Use <methods><include name="methodName"/></methods> inside the <class> tag. Only listed methods execute. Use <exclude> to run all except specified methods. Method names are case-sensitive and must match exactly.

Interviewer expects: Practical selective execution knowledge.

Q4: Can you have multiple testng.xml files in one project?

Answer: Yes — common pattern: smoke-suite.xml, regression-suite.xml, cross-browser-suite.xml. Maven Surefire plugin accepts suiteXmlFile configuration. CI pipelines select the appropriate XML per job/stage.

Interviewer expects: CI/CD integration with multiple suite files.

Q5: What does preserve-order="true" do?

Answer: When true, TestNG maintains the order of test methods as defined in the class or XML. When false (default in parallel mode), methods may execute out of order. Important for dependent tests or ordered workflows.

Interviewer expects: Interaction with parallel execution settings.

Q6: How do you exclude a group in testng.xml?

Answer: Inside <groups><run>, add <exclude name="groupName"/>. Tests annotated with @Test(groups = "groupName") are skipped. Combine with <include> to run specific groups while excluding others like "wip" or "broken".

Interviewer expects: Group filtering for CI pipeline stages.

Q7: What is the verbose attribute?

Answer: Controls TestNG logging detail on console. 0 = silent, 1 = failures only, 2 = standard (recommended), 10 = maximum debug output. Higher values help diagnose configuration and listener issues during development.

Interviewer expects: Debugging configuration knowledge.

Q8: How do suite-level parameters differ from test-level?

Answer: Suite-level parameters apply to all test blocks unless overridden. Test-level parameters scope values to that specific <test> block. Test-level takes precedence when both define the same parameter name.

Interviewer expects: Parameter inheritance and override behavior.

Q9: Can you reference packages instead of individual classes?

Answer: Yes — use <packages><package name="com.example.tests"/></packages>. TestNG discovers all @Test classes in that package. More maintainable than listing each class individually as the suite grows.

Interviewer expects: Scalable suite configuration approach.

Q10: How do you configure Maven to run TestNG with a specific XML?

Interviewer expects: Maven + TestNG integration hands-on experience.

Answer: Determines behavior when a @BeforeSuite, @BeforeTest, or @BeforeClass configuration method fails. "skip" (default) skips all tests in that scope. "continue" attempts to run remaining tests despite config failure.

Interviewer expects: Failure handling strategy awareness.

Section 10: Parallel Execution

10.1 Why Parallel Execution?

Sequential execution of a 500-test suite at 15 seconds per test takes ~2 hours. With parallel="methods" and thread-count="10", the same suite completes in ~20–30 minutes — a 4–6x speedup. Parallel execution is essential for CI/CD pipelines with tight feedback loops.

Real-world impact: A regression suite running sequentially for 2 hours blocks the CI pipeline. Parallel execution with 8 threads reduces this to ~25 minutes, enabling faster deployments and quicker defect detection.

10.2 Parallel Modes

parallel="methods" — Methods Run in Parallel

Each @Test method runs in its own thread. Best for large test classes with independent methods. DataProvider iterations also run as separate parallel invocations.

<suite name="Method Parallel Suite" parallel="methods" thread-count="5">
    <test name="All Tests">
        <classes>
            <class name="com.example.tests.LoginTest"/>
            <class name="com.example.tests.SearchTest"/>
        </classes>
    </test>
</suite>

parallel="classes" — Classes Run in Parallel

Each test class runs in its own thread; methods within a class run sequentially. Good when methods in a class share setup state.

<suite name="Class Parallel Suite" parallel="classes" thread-count="3">
    <test name="Regression">
        <packages><package name="com.example.tests"/></packages>
    </test>
</suite>

parallel="tests" — <test> Tags Run in Parallel

Each <test> block in XML runs concurrently. Ideal for cross-browser testing where each browser is a separate test block.

<suite name="Test Parallel Suite" parallel="tests" thread-count="3">
    <test name="Chrome"><parameter name="browser" value="chrome"/>...</test>
    <test name="Firefox"><parameter name="browser" value="firefox"/>...</test>
    <test name="Edge"><parameter name="browser" value="edge"/>...</test>
</suite>

parallel="instances" — Instances Run in Parallel

When @Factory creates multiple instances of a test class, each instance runs in parallel. Used with data-driven class instantiation.

<suite name="Instance Parallel Suite" parallel="instances" thread-count="4">
    <test name="Factory Tests">
        <classes><class name="com.example.tests.BrowserFactoryTest"/></classes>
    </test>
</suite>

10.3 thread-count Attribute

Controls the maximum number of concurrent threads. Set at suite or test level. Rule of thumb: thread-count = number of CPU cores for CPU-bound tests; 2x cores for I/O-bound Selenium tests. Monitor for resource exhaustion.

ThreadLocal<WebDriver> Solution

public class DriverManager {
    private static ThreadLocal<WebDriver> driver = new ThreadLocal<>();

    public static WebDriver getDriver() {
        if (driver.get() == null) {
            String browser = System.getProperty("browser", "chrome");
            switch (browser.toLowerCase()) {
                case "firefox":
                    driver.set(new FirefoxDriver());
                    break;
                case "edge":
                    driver.set(new EdgeDriver());
                    break;
                default:
                    driver.set(new ChromeDriver());
            }
            driver.get().manage().window().maximize();
            driver.get().manage().timeouts().implicitlyWait(Duration.ofSeconds(10));
        }
        return driver.get();
    }

    public static void quitDriver() {
        if (driver.get() != null) {
            driver.get().quit();
            driver.remove(); // critical: prevent memory leaks
        }
    }
}

    }

10.8 Best Practices for Parallel Execution

  1. Always use ThreadLocal for WebDriver — never static shared drivers
  2. Make tests independent — no shared mutable state between tests
  3. Use unique test data per thread to avoid data collision
  4. Set realistic thread-count (CPU cores × 2 for I/O-bound tests)
  5. Avoid @DependsOnMethods in parallel suites — dependencies break parallelism
  6. Use parallel="tests" for cross-browser; parallel="methods" for speed
  7. Implement proper @AfterMethod cleanup even on failure
  8. Monitor Grid node health and session limits
  9. Use CI pipeline parallelism matching your thread-count
  10. Profile and tune — too many threads cause contention and flaky results

10.9 Interview Q&A — Parallel Execution

Q1: What parallel modes does TestNG support?

Answer: Four modes: methods (each @Test in its own thread), classes (each class in its own thread), tests (each <test> XML block in parallel), and instances (each @Factory-created instance in parallel). Configured via parallel attribute in testng.xml.

Interviewer expects: All four modes with use cases.

Q2: Why is static WebDriver dangerous in parallel tests?

Answer: Static fields are shared across all threads. One thread quitting or navigating the shared driver affects all other threads simultaneously, causing random failures, session errors, and unpredictable test behavior.

Interviewer expects: ThreadLocal solution knowledge.

Q3: Explain ThreadLocal WebDriver pattern.

Answer: ThreadLocal creates a per-thread copy of WebDriver. Each thread gets its own browser instance via driver.get(). Always call driver.remove() in @AfterMethod to prevent memory leaks in thread pools.

Interviewer expects: Complete implementation with cleanup.

Q4: What is the optimal thread-count?

Answer: Depends on test type. CPU-bound: match CPU core count. I/O-bound (Selenium): 2× cores. Grid: match available node capacity. Start conservative (4–5), monitor, and increase until failures or resource exhaustion appear.

Interviewer expects: Practical tuning approach, not a fixed number.

Q5: Can @DependsOnMethods work with parallel execution?

Answer: Technically yes with preserve-order="true", but it largely defeats parallelism because dependent methods must wait. Best practice: design independent tests and avoid dependencies in parallel suites.

Interviewer expects: Design recommendation against dependencies in parallel.

Q6: Difference between parallel="tests" and parallel="classes"?

Answer: parallel="tests" runs each <test> XML block concurrently — ideal for cross-browser where each browser is a test block. parallel="classes" runs each Java test class concurrently — methods within a class remain sequential.

Interviewer expects: XML structure mapping to parallel behavior.

Q7: How does Selenium Grid help parallel execution?

Answer: Grid distributes browser sessions across multiple nodes (machines/containers). Tests send commands to the Hub, which routes them to available Nodes. Enables scaling beyond local machine capacity.

Interviewer expects: Hub-Node architecture understanding.

Q8: What thread safety issues exist beyond WebDriver?

Answer: Shared test data (same user login), static counters/flags, shared file writes, non-thread-safe collections, and shared database records. Each thread needs isolated data and resources.

Interviewer expects: Comprehensive thread safety awareness.

Q9: How do DataProvider iterations behave in parallel mode?

Answer: Each DataProvider row creates a separate method invocation that can run in parallel when parallel="methods" is set. Ensure each iteration uses independent data and WebDriver instances.

Interviewer expects: Data isolation per iteration.

Q10: Can you mix parallel modes?

Answer: Yes — suite can have parallel="tests" while individual <test> blocks override with parallel="methods". The more specific (test-level) setting takes precedence over suite-level for that test block.

Interviewer expects: Override hierarchy knowledge.

Q11: What is data-provider-thread-count?

Answer: A suite-level attribute controlling how many threads run DataProvider-fed test invocations concurrently. Separate from thread-count which controls overall parallelism. Useful for limiting DataProvider concurrency independently.

Interviewer expects: Fine-grained thread control awareness.

Q12: How do you debug flaky parallel tests?

Answer: Run failing tests sequentially first to isolate parallelism issues. Check for shared state, static variables, and data collisions. Add thread name logging. Reduce thread-count to identify resource contention. Use ThreadLocal consistently.

Interviewer expects: Systematic debugging methodology.

import org.openqa.selenium.WebDriver;

import java.io.File;
import java.io.IOException;
import java.text.SimpleDateFormat;
import java.util.Date;

public class ScreenshotListener implements ITestListener {

    private static final String SCREENSHOT_DIR = "reports/screenshots/";

    @Override
    public void onTestFailure(ITestResult result) {
        captureScreenshot(result);
    }

    private void captureScreenshot(ITestResult result) {
        WebDriver driver = DriverManager.getDriver();
        if (driver instanceof TakesScreenshot) {
            File src = ((TakesScreenshot) driver).getScreenshotAs(OutputType.FILE);
            String methodName = result.getMethod().getMethodName();
            String timestamp = new SimpleDateFormat("yyyyMMdd_HHmmss").format(new Date());
            String dest = SCREENSHOT_DIR + methodName + "_" + timestamp + ".png";
            try {
                FileUtils.copyFile(src, new File(dest));
                result.setAttribute("screenshot", dest);
                System.out.println("Screenshot saved: " + dest);
            } catch (IOException e) {
                System.err.println("Screenshot capture failed: " + e.getMessage());
            }
        }
    }
}

11.11 Listener Execution Order

When multiple listeners are registered, TestNG invokes them in registration order for each event. ISuiteListener.onStart fires before ITestListener.onStart. IExecutionListener wraps the entire run. If one listener throws an exception, subsequent listeners for that event may not execute.

Section 12: Retry Mechanism

12.1 Why Retry?

Flaky tests fail intermittently due to network latency, environment instability, timing issues, or resource contention — not actual defects. Retry mechanisms re-execute failed tests automatically, reducing false negatives in CI/CD pipelines and improving suite reliability metrics.

When retry helps: Transient network timeouts, slow page loads, Grid node restarts, and environment warmup delays. When retry hurts: Genuine application bugs that pass on retry mask real failures and erode trust in test results.

12.2 IRetryAnalyzer Interface

package com.example.utils;

import org.testng.IRetryAnalyzer;
import org.testng.ITestResult;

public class RetryAnalyzer implements IRetryAnalyzer {
    private int retryCount = 0;
    private static final int MAX_RETRY = 2; // total attempts = MAX_RETRY + 1

    @Override
    public boolean retry(ITestResult result) {
        if (retryCount < MAX_RETRY) {
            retryCount++;
            System.out.println("Retrying " + result.getName()
                + " — attempt " + (retryCount + 1));
            return true;  // retry the test
        }
        return false; // no more retries
    }
}

12.9 Interview Q&A — Retry Mechanism

Q1: What is IRetryAnalyzer in TestNG?

Answer: An interface with a single retry(ITestResult) method returning boolean. Return true to re-execute the failed test, false to stop. Controls how many times a failed test is retried before final failure is reported.

Interviewer expects: Interface method signature and return logic.

Q2: How do you apply retry to all tests globally?

Answer: Implement IAnnotationTransformer, set RetryAnalyzer class on every ITestAnnotation in the transform method, and register the transformer as a listener in testng.xml. Avoids annotating each test individually.

Interviewer expects: IAnnotationTransformer global pattern.

Q3: Why must RetryAnalyzer manage instance state carefully?

Answer: TestNG may reuse the same RetryAnalyzer instance across tests. The retryCount field must reset per test or use ITestResult to track attempts. Otherwise, retry count from one test affects another.

Interviewer expects: Instance state isolation awareness.

Q4: What is the difference between retry and @Test(invocationCount)?

Answer: invocationCount runs the test N times regardless of pass/fail — used for load testing. Retry re-executes only on failure up to a limit — used for handling flakiness. Completely different purposes.

Interviewer expects: Clear distinction between the two mechanisms.

Q5: How do you configure retry count externally?

Answer: Read retry.count from config.properties or system properties inside the RetryAnalyzer constructor. Allows CI pipelines to adjust retry behavior without code changes via -Dretry.count=3.

Interviewer expects: Externalized configuration approach.

Q6: Can retry mask real bugs?

Answer: Yes — if a test fails due to an actual defect but passes on retry (e.g., race condition in the app), retry hides the bug. Monitor retry success rates; high retry-then-pass rates indicate instability needing investigation.

Interviewer expects: Critical thinking about retry misuse.

Q7: How does TestNG report retried tests?

Answer: TestNG shows the final result after all retry attempts. Intermediate failures appear in verbose logs. ExtentReports and Allure can log each attempt separately if the listener tracks invocation count via getCurrentInvocationCount().

Interviewer expects: Reporting integration for retry attempts.

Q8: Should you retry data setup failures?

Answer: No — if @BeforeMethod or DataProvider fails, the root cause is infrastructure or data issue. Retrying the test won't fix it. Fix the setup logic or data source instead of applying retry.

Interviewer expects: Judgment on appropriate retry scope.

Q9: How do retry and listeners interact?

Answer: onTestFailure fires on each failed attempt including intermediate retries. onTestSuccess fires when retry eventually passes. Listeners should use getCurrentInvocationCount() to distinguish attempts and avoid duplicate actions.

Interviewer expects: Event firing behavior across retry attempts.

package com.example.listeners;

import com.aventstack.extentreports.*;
import com.aventstack.extentreports.reporter.ExtentSparkReporter;
import com.aventstack.extentreports.reporter.configuration.Theme;
import org.testng.*;

public class ExtentReportListener implements ITestListener, ISuiteListener {
    private static ExtentReports extent;
    private static ThreadLocal<ExtentTest> test = new ThreadLocal<>();

    @Override
    public void onStart(ISuite suite) {
        ExtentSparkReporter spark = new ExtentSparkReporter("reports/ExtentReport.html");
        spark.config().setTheme(Theme.STANDARD);
        spark.config().setDocumentTitle("Test Report");
        spark.config().setReportName(suite.getName());
        extent = new ExtentReports();
        extent.attachReporter(spark);
        extent.setSystemInfo("Tester", System.getProperty("user.name"));
    }

    @Override
    public void onTestStart(ITestResult result) {
        ExtentTest extentTest = extent.createTest(
            result.getMethod().getMethodName(),
            result.getMethod().getDescription());
        test.set(extentTest);
    }

    @Override
    public void onTestSuccess(ITestResult result) {
        test.get().pass("Test passed");
    }

    @Override
    public void onTestFailure(ITestResult result) {
        test.get().fail(result.getThrowable());
        String screenshot = (String) result.getAttribute("screenshot");
        if (screenshot != null) {
            test.get().addScreenCaptureFromPath(screenshot);
        }
    }

    @Override
    public void onTestSkipped(ITestResult result) {
        test.get().skip(result.getThrowable());
    }

    @Override
    public void onFinish(ISuite suite) {
        if (extent != null) extent.flush();
    }
}

14.6 ConfigReader Utility

package com.example.utils;

import java.io.IOException;
import java.io.InputStream;
import java.util.Properties;

public class ConfigReader {
    private Properties properties = new Properties();

    public ConfigReader() {
        String env = System.getProperty("env", "qa");
        String fileName = "config-" + env + ".properties";
        try (InputStream is = getClass().getClassLoader()
                .getResourceAsStream(fileName)) {
            if (is == null) {
                is = getClass().getClassLoader().getResourceAsStream("config.properties");
            }
            properties.load(is);
        } catch (IOException e) {
            throw new RuntimeException("Failed to load config: " + fileName, e);
        }
    }

    public String get(String key) {
        String sysProp = System.getProperty(key);
        return sysProp != null ? sysProp : properties.getProperty(key);
    }

    public int getInt(String key) {
        return Integer.parseInt(get(key));
    }

    public boolean getBoolean(String key) {
        return Boolean.parseBoolean(get(key));
    }
}

14.7 WebDriver Factory — Complete Code

package com.example.utils;

import io.github.bonigarcia.wdm.WebDriverManager;
import org.openqa.selenium.WebDriver;
import org.openqa.selenium.chrome.ChromeDriver;
import org.openqa.selenium.chrome.ChromeOptions;
import org.openqa.selenium.edge.EdgeDriver;
import org.openqa.selenium.firefox.FirefoxDriver;
import org.openqa.selenium.firefox.FirefoxOptions;
import java.time.Duration;

public class DriverManager {
    private static ThreadLocal<WebDriver> tlDriver = new ThreadLocal<>();

    public static WebDriver initDriver(String browser) {
        WebDriver driver;
        switch (browser.toLowerCase()) {
            case "firefox":
                WebDriverManager.firefoxdriver().setup();
                driver = new FirefoxDriver();
                break;
            case "firefox-headless":
                WebDriverManager.firefoxdriver().setup();
                FirefoxOptions fOpts = new FirefoxOptions();
                fOpts.addArguments("-headless");
                driver = new FirefoxDriver(fOpts);
                break;
            case "edge":
                WebDriverManager.edgedriver().setup();
                driver = new EdgeDriver();
                break;
            case "chrome-headless":
                WebDriverManager.chromedriver().setup();
                ChromeOptions cOpts = new ChromeOptions();
                cOpts.addArguments("--headless=new", "--window-size=1920,1080");
                driver = new ChromeDriver(cOpts);
                break;
            default:
                WebDriverManager.chromedriver().setup();
                driver = new ChromeDriver();
        }
        driver.manage().window().maximize();
        driver.manage().timeouts().implicitlyWait(Duration.ofSeconds(10));
        driver.manage().timeouts().pageLoadTimeout(Duration.ofSeconds(30));
        tlDriver.set(driver);
        return driver;
    }

    public static WebDriver getDriver() { return tlDriver.get(); }

    public static void quitDriver() {
        if (tlDriver.get() != null) {
            tlDriver.get().quit();
            tlDriver.remove();
        }
    }
}

14.8 Framework Design Principles

  • DRY (Don't Repeat Yourself): Common logic in utilities and base classes — never duplicate WebDriver setup or config reading
  • Single Responsibility: Pages handle UI, tests handle assertions, utils handle infrastructure
  • Loose Coupling: Tests depend on page interfaces, not Selenium internals — swap WebDriver without changing tests
  • Configuration Externalization: URLs, credentials, browser in properties — not hardcoded in tests
  • Data Separation: Test data in Excel/JSON files via DataProviders — not embedded in test methods

Q1: What is TestNG?

Answer: TestNG (Test Next Generation) is a Java testing framework inspired by JUnit and NUnit. It provides annotations, flexible test configuration via XML, parallel execution, data-driven testing, grouping, dependencies, and rich reporting — designed for all test levels from unit to integration.

Interviewer expects: Beyond "testing framework" — mention key differentiators from JUnit.

Q3: What annotations does TestNG provide?

Answer: Core annotations: @Test, @BeforeSuite, @AfterSuite, @BeforeTest, @AfterTest, @BeforeClass, @AfterClass, @BeforeMethod, @AfterMethod, @BeforeGroups, @AfterGroups, @DataProvider, @Parameters, @Factory, @Listeners. Each controls test lifecycle or data injection.

Interviewer expects: At least 8-10 annotations with purposes.

Q4: What is the difference between @BeforeClass and @BeforeMethod?

Answer: @BeforeClass runs once before all methods in a class — used for expensive setup like DB connection. @BeforeMethod runs before each @Test method — used for per-test setup like browser launch. @BeforeClass is more efficient but shared state risks exist.

Interviewer expects: Execution frequency and use case distinction.

Q5: What is the TestNG test execution order?

Answer: @BeforeSuite → @BeforeTest → @BeforeClass → @BeforeMethod → @Test → @AfterMethod → @AfterClass → @AfterTest → @AfterSuite. @BeforeGroups/@AfterGroups run before/after groups. Multiple classes follow alphabetical order unless prioritized.

Interviewer expects: Complete lifecycle sequence.

Q6: How do you skip a test in TestNG?

Answer: Set enabled=false on @Test annotation: @Test(enabled = false). Or use IAnnotationTransformer to disable dynamically. Or exclude via testng.xml groups/methods. Disabled tests appear as SKIPPED in reports.

Interviewer expects: Multiple skip mechanisms.

Q7: What is the purpose of @Test(priority)?

Answer: Controls execution order of test methods within a class. Lower priority number runs first (priority=1 before priority=2). Default priority is 0. Only affects order within same class in non-parallel mode.

Interviewer expects: Lower number = higher priority.

Q8: What assertions does TestNG provide?

Answer: org.testng.Assert class: assertTrue, assertFalse, assertEquals, assertNotEquals, assertNull, assertNotNull, assertSame, assertNotSame, fail. SoftAssert (org.testng.asserts.SoftAssert) collects all failures before reporting.

Interviewer expects: SoftAssert awareness for multiple checks.

Q9: What is SoftAssert?

Answer: SoftAssert collects assertion failures without stopping execution. Call assertAll() at the end to report all failures together. Useful when validating multiple fields on a page — one failure doesn't prevent checking others.

Interviewer expects: assertAll() requirement understanding.

Q10: What are TestNG groups?

Answer: Tags assigned to tests via @Test(groups = {"smoke", "regression"}). Enable selective execution in testng.xml using include/exclude. A test can belong to multiple groups. Groups organize tests by type, feature, or pipeline stage.

Interviewer expects: XML group filtering knowledge.

Q11: How do you run tests belonging to a specific group?

Interviewer expects: Both XML and Maven approaches.

Q12: Can a test belong to multiple groups?

Interviewer expects: Multi-group assignment pattern.

Q13: What is @BeforeGroups used for?

Answer: Runs once before the first test method in the specified group executes. Used for group-specific setup like preparing test data for "database" group tests. @AfterGroups runs after the last test in that group.

Interviewer expects: Group-scoped setup/teardown.

Q14: What is the difference between @BeforeTest and @BeforeClass?

Answer: @BeforeTest runs once per <test> tag in XML before any classes in that test block. @BeforeClass runs once per test class. A suite with two <test> tags triggers @BeforeTest twice but @BeforeClass once per class.

Interviewer expects: XML test tag vs Java class scope.

Q15: What is @AfterMethod alwaysRun?

Answer: @AfterMethod(alwaysRun = true) executes even when the test fails or is skipped. Without it, @AfterMethod may not run if @BeforeMethod fails. Critical for cleanup like WebDriver.quit() and resource release.

Interviewer expects: Negative testing use case.

Q27: Can TestNG run JUnit tests?

Q29: How do you ignore a test temporarily?

15.2 Intermediate Level (35 Questions)

Q31: Explain @DataProvider in detail.

Answer: @DataProvider marks a method returning Object[][] or Iterator<Object[]>. Each inner array supplies arguments for one test invocation. Linked via @Test(dataProvider = "name"). Enables data-driven testing with multiple data sets per test method.

Interviewer expects: Return type and invocation model.

Q32: @Parameters vs @DataProvider?

Answer: @Parameters reads static strings from testng.xml — one execution, config values. @DataProvider is a Java method returning Object[][] — multiple executions, any object type, can read external files. Parameters for config, DataProvider for test data.

Interviewer expects: Clear use case for each.

Q33: How do you read Excel data in DataProvider?

Answer: Use Apache POI: open workbook, select sheet, iterate rows/columns into Object[][]. Encapsulate in ExcelReader utility. DataProvider calls the utility. Keep file path in config.properties for environment flexibility.

Interviewer expects: Apache POI and utility pattern.

Q34: What is parallel="methods"?

Answer: TestNG parallel mode where each @Test method runs in its own thread. Configured in testng.xml: parallel="methods" thread-count="5". DataProvider iterations also run as separate parallel invocations. Requires thread-safe test design.

Interviewer expects: Thread safety implications.

Q35: Explain all parallel modes in TestNG.

Answer: methods (each @Test in own thread), classes (each class in own thread), tests (each <test> XML block in parallel), instances (each @Factory instance in parallel). Choose based on granularity needed — methods for speed, tests for cross-browser.

Interviewer expects: All four with use cases.

Q36: What is thread-count in testng.xml?

Answer: Maximum concurrent threads for parallel execution. Set at suite or test level. Rule of thumb: CPU cores for CPU-bound, 2× cores for I/O-bound Selenium tests. Too high causes resource contention and flakiness.

Interviewer expects: Tuning guidance.

Q37: How do you configure TestNG in Maven?

Answer: maven-surefire-plugin with suiteXmlFiles pointing to testng.xml. Add testng dependency. Configure parallel and threadCount in plugin config. Use profiles for different suites: mvn test -Pregression.

Interviewer expects: Surefire plugin configuration.

Q38: What are TestNG listeners?

Answer: Event-driven interfaces hooking into test lifecycle: ITestListener (test events), ISuiteListener (suite events), IReporter (custom reports), IAnnotationTransformer (modify annotations). Enable logging, screenshots, and reporting without changing test code.

Interviewer expects: Cross-cutting concern pattern.

Q39: How do you register listeners?

Answer: Three ways: @Listeners on test class, <listeners> in testng.xml (preferred for global), ServiceLoader via META-INF/services. XML registration is most visible and maintainable in enterprise frameworks.

Interviewer expects: XML as enterprise standard.

Q40: What is @Test(dependsOnMethods)?

Answer: @Test(dependsOnMethods = {"testLogin"}) runs only if testLogin passes. Failed dependency skips the dependent test. Can depend on multiple methods. Avoid in parallel suites — breaks parallelism and creates fragile test chains.

Interviewer expects: Parallel incompatibility warning.

Q41: What is @Test(dependsOnGroups)?

Answer: Depends on all tests in a group completing successfully: dependsOnGroups = {"setup"}. The dependent test runs only if every test in the "setup" group passed. Broader dependency than dependsOnMethods.

Interviewer expects: Group-level dependency scope.

Q42: What happens when a dependent test's dependency fails?

Answer: TestNG marks the dependent test as SKIPPED with reason "Dependant methods not run." Skipped tests appear in reports separately from failures. Fix the root dependency failure first.

Interviewer expects: SKIP status and reporting.

Q43: How do you include/exclude methods in testng.xml?

Answer: Inside <class>: <methods><include name="testLogin"/><exclude name="testPayment"/></methods>. Include runs only listed methods. Exclude runs all except listed. Method names are case-sensitive.

Interviewer expects: Selective execution for CI stages.

Q44: How do you run tests from multiple packages?

Interviewer expects: Package-based discovery.

Q45: What is preserve-order in testng.xml?

Interviewer expects: Order vs speed trade-off.

Q46: Explain ITestListener methods.

Q48: What is ISuiteListener?

Q51: How do you implement cross-browser testing?

Interviewer expects: XML structure + factory pattern.

Q52: What is ThreadLocal WebDriver?

Answer: ThreadLocal<WebDriver> provides each thread its own browser instance. get() returns thread-specific driver. remove() in cleanup prevents memory leaks. Essential for parallel test execution safety.

Interviewer expects: Why static WebDriver fails.

Q53: Why is static WebDriver dangerous?

Answer: Static fields are shared across all threads. One thread quitting the driver kills sessions for all threads. Causes random failures, stale elements, and session not found errors in parallel execution.

Interviewer expects: ThreadLocal as the solution.

Q54: How do you handle test data in parallel execution?

Interviewer expects: Data isolation strategy.

Q55: What is @Test(successPercentage)?

Interviewer expects: Load testing tolerance pattern.

Q56: How do you integrate TestNG with Selenium Grid?

Q57: What is verbose in testng.xml?

Interviewer expects: Debugging use case.

Q58: How do you run a single test method?

Interviewer expects: XML include as CI-compatible approach.

Q59: What is dataProviderClass attribute?

Interviewer expects: Separation of data and tests.

Q60: Can DataProvider return Iterator?

Interviewer expects: Memory efficiency for large data.

Q61: How do you share data between test methods?
